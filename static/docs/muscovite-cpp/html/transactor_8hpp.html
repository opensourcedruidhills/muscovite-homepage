Execute a database operation with optional retry support.

The callable is invoked with a fresh pqxx::work transaction. If the callable completes without throwing, the transaction is committed. If an exception is thrown, the transaction is rolled back.

With a RetryPolicy, transient errors (serialization failures, deadlocks) cause automatic retry with exponential backoff.

pqxx::params{name}

return id;

Execute a deadline-aware database operation.

Propagates gRPC deadline to PostgreSQL via SET LOCAL statement_timeout. If the deadline has already expired, returns DeadlineExceeded immediately. The timeout is clamped to max_statement_timeout and reduced by deadline_buffer to ensure PostgreSQL cancels before the gRPC deadline fires.

Execute a database operation using an EXISTING transaction.

This overload is designed for testing - it uses an existing transaction and does NOT commit. The caller controls when (or if) to commit/rollback. This enables transaction-per-test isolation patterns where all test operations are wrapped in a transaction that's rolled back at the end.

Template Parameters
FCallable type
RReturn type (deduced)
Parameters
txExisting transaction reference (caller manages commit/rollback)
fnCallable taking pqxx::work& and returning R
Returns
Result on success, Error on failure

Example (test isolation):

TEST_CASE("Repository test with isolation", "[db]") {
TestTransactionFixture fixture(pool); // Auto-rollback at end
auto result = execute(fixture.transaction(), [&](pqxx::work& tx) {
return tx.query_value<int>("SELECT 1");
});
REQUIRE(result.has_value());
// Transaction rolled back by fixture destructor - no data persisted
}

Execute a database operation using UnitOfWork for domain event support.

Creates a fresh UnitOfWork on each attempt. On retry (serialization failure, deadlock), the UnitOfWork is reconstructed — pending events are discarded, preventing duplicate domain events. The UnitOfWork's commit_or_throw() persists events and commits the transaction atomically.
