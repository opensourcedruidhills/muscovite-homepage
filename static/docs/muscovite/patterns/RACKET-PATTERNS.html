Racket Pattern Catalog for Muscovite Generators

Racket Pattern Catalog for Muscovite Generators

Purpose: Constraint document for AI agents and human developers. Every Racket pattern in Muscovite generators must use the canonical form. This document is the single source of truth for "how to write Racket in this project."

Enforcement: ADR 20260226-racket-idiomatic-style-and-byte-compilation.md, ADR 20260226-mandatory-template-dsl-adoption.md

Scope: All .rkt files in dslgen/. Parser, model, generators, migration, validation, tests.

How to Use This Document

Each section has three tiers:

Tier Meaning
✅ CANONICAL The required pattern. Use this in all new and modified code.
✅ ACCEPTABLE Permitted in specific circumstances (documented).
❌ ANTI-PATTERN Forbidden in new code. Convert when touching existing files.

When an AI agent generates Racket code for Muscovite, it MUST use only CANONICAL or ACCEPTABLE patterns. When an AI agent modifies an existing file, it MUST convert any anti-patterns in the modified function to canonical form.

1. Iterator Patterns

✅ CANONICAL: in-list on all for over lists

;; Always annotate the sequence type
(for/list ([ctx (in-list contexts)])
  (generate-context ctx))

(for ([field (in-list (entity-fields entity))])
  (validate-field field))

(for/fold ([acc '()])
          ([ctx (in-list contexts)])
  (append acc (generate-context ctx)))

Why: in-list eliminates generic sequence dispatch at runtime. Without it, Racket checks at every iteration whether the value is a list, vector, hash, string, etc. 5–10% speedup on iteration-heavy generators. More importantly, it communicates intent and catches type errors earlier.

✅ CANONICAL: map / filter / filter-map for simple transforms

;; Pure transform: prefer map
(map entity-name (bounded-context-entities ctx))

;; Filter: prefer filter/filter-map
(filter entity-event-sourced? entities)
(filter-map entity-doc entities)  ;; drops #f values

;; Parallel map over multiple lists
(map (λ (name type) (format "~a ~a;" type name))
     field-names field-types)

Why: map/filter are declarative — the reader knows instantly that the output is a same-length (or shorter) list with no side effects. for/list is for when the body needs define, multiple expressions, or complex control flow.

✅ ACCEPTABLE: for/list when body needs bindings or control flow

;; Body needs define — for/list is appropriate
(for/list ([fld (in-list fields)])
  (define cpp-type (field->cpp-type fld))
  (define cpp-name (field->cpp-member fld))
  (format "    ~a ~a;" cpp-type cpp-name))

❌ ANTI-PATTERN: Bare for without sequence annotation

;; FORBIDDEN — generic sequence dispatch, no type safety
(for/list ([x xs]) (f x))
(for ([x xs]) (do-something x))

;; FORBIDDEN — struct accessor returns list, needs in-list
(for/list ([m (ddd:repository-methods repo)]) ...)

Found in codebase: dslgen/generators/qt/saga-widgets.rkt, dslgen/viewer.rkt, dslgen/generators/qt/state-machine-widgets.rkt, dslgen/generators/search/document-mapper.rkt

❌ ANTI-PATTERN: for/list where map suffices

;; FORBIDDEN — map is simpler and clearer
(for/list ([x (in-list xs)]) (f x))
;; USE:
(map f xs)

;; FORBIDDEN — nested for/list for a pure transform
(string-join
 (for/list ([ev (in-list events)])
   (format "case ~a:" ev))
 "\n")
;; USE:
(string-join (map (λ (ev) (format "case ~a:" ev)) events) "\n")

2. Accumulation Patterns (for/fold vs set!)

✅ CANONICAL: for/fold for building results

;; Build a list by folding
(define results
  (for/fold ([acc '()])
            ([ctx (in-list contexts)])
    (if (context-has-entities? ctx)
        (cons (generate-context ctx) acc)
        acc)))

;; Build a hash by folding
(define entity-map
  (for/fold ([h (hash)])
            ([ent (in-list entities)])
    (hash-set h (entity-uuid ent) ent)))

;; Accumulate multiple values
(define-values (tables views)
  (for/fold ([tables '()] [views '()])
            ([proj (in-list projections)])
    (if (projection-materialized? proj)
        (values tables (cons proj views))
        (values (cons proj tables) views))))

✅ CANONICAL: for/list with #:when for filtered collection

;; Collect only matching items
(for/list ([fld (in-list fields)]
           #:when (field-required? fld))
  (field-name fld))

✅ CANONICAL: for/append for flat-mapping

;; Flatten nested lists
(define all-fields
  (for/append ([ent (in-list entities)])
    (entity-fields ent)))

✅ ACCEPTABLE: let loop for stateful iteration

;; Parser token consumption (inherently stateful)
(let loop ([tokens remaining])
  (cond
    [(null? tokens) '()]
    [(token-keyword? (car tokens) "END")
     (cdr tokens)]
    [else
     (cons (parse-item (car tokens))
           (loop (cdr tokens)))]))

❌ ANTI-PATTERN: set! accumulator loop

;; FORBIDDEN
(define issues '())
(for ([repo (in-list (bounded-context-repositories ctx))])
  (define for-agg (repository-for-aggregate repo))
  (unless (set-member? agg-names for-agg)
    (set! issues
          (cons (validation-issue 'error 'reference ...) issues))))
issues

;; USE for/fold:
(for/fold ([issues '()])
          ([repo (in-list (bounded-context-repositories ctx))])
  (define for-agg (repository-for-aggregate repo))
  (if (set-member? agg-names for-agg)
      issues
      (cons (validation-issue 'error 'reference ...) issues)))

Found in codebase: dslgen/validator.rkt (12 instances), dslgen/verification/verify-types.rkt (4 instances), dslgen/verification/verify-schema.rkt (3 instances)

❌ ANTI-PATTERN: set! counter

;; FORBIDDEN
(define count 0)
(for ([pair (in-list searchable-pairs)])
  (generate-file pair)
  (set! count (add1 count)))
(printf "Total: ~a files\n" count)

;; USE:
(define count (length searchable-pairs))
(for ([pair (in-list searchable-pairs)])
  (generate-file pair))
(printf "Total: ~a files\n" count)

;; OR if count depends on a condition:
(define count
  (for/sum ([pair (in-list searchable-pairs)]
            #:when (generate-file pair))
    1))

3. Hash / Map Patterns

✅ CANONICAL: Immutable hash construction

;; Literal hash
(define type-map
  (hash 'UUID "boost::uuids::uuid"
        'String "std::string"
        'Integer "std::int32_t"))

;; Hash from iteration
(define entity-map
  (for/hash ([ent (in-list entities)])
    (values (entity-uuid ent) ent)))

;; Hash from fold (when multiple sources feed in)
(define uuid-map
  (for*/fold ([h (hash)])
             ([ctx (in-list contexts)]
              [ent (in-list (all-entities-in-context ctx))])
    (hash-set h (entity-uuid ent) (list ent ctx))))

;; Updating an immutable hash
(define h2 (hash-set h 'new-key new-value))
(define h3 (hash-update h 'count add1 0))

✅ ACCEPTABLE: make-hash for module-level registries

;; WHY: Module-level registry populated at load time, read-only thereafter
(define generators (make-hash))

(define (register-generator name proc)
  (hash-set! generators name proc))

(define (get-generator name)
  (hash-ref generators name
             (λ () (error 'get-generator "Unknown generator: ~a" name))))

When acceptable: Only for module-scoped registries that are write-once-read-many. Must include a WHY: comment.

✅ ACCEPTABLE: make-hash for performance-critical paths

;; WHY: Performance — >1000 insertions in tight loop; immutable hash would
;; create O(n) intermediate copies
(define h (make-hash))
(for ([row (in-list large-dataset)])
  (hash-set! h (row-key row) (row-value row)))
;; Freeze: use only hash-ref after this point

When acceptable: Only when profiling shows immutable hash is a bottleneck (>1000 insertions). Must include a WHY: performance comment.

❌ ANTI-PATTERN: Mutable hash for build-then-read

;; FORBIDDEN — accumulation into mutable hash
(define h (make-hash))
(for ([ctx (in-list contexts)])
  (for ([agg (in-list (bounded-context-aggregates ctx))])
    (hash-set! h (entity-uuid (aggregate-root agg))
               (list (aggregate-root agg) ctx (aggregate-uuid agg)))))
h

;; USE for*/fold:
(for*/fold ([h (hash)])
           ([ctx (in-list contexts)]
            [agg (in-list (bounded-context-aggregates ctx))])
  (hash-set h (entity-uuid (aggregate-root agg))
            (list (aggregate-root agg) ctx (aggregate-uuid agg))))

Found in codebase: dslgen/migration/diff-model.rkt (entities->uuid-map, relationships->uuid-map), dslgen/incremental/incremental-gen.rkt (11 hash-set! calls building uuid->ctx-name map)

4. Match / Dispatch Patterns

✅ CANONICAL: match on structs

;; Dispatch on struct type
(match node
  [(template-node 'text str) str]
  [(template-node 'line str) (string-append str "\n")]
  [(template-node 'block (cons level items))
   (apply string-append (map (curry emit-node level) items))]
  [_ (error 'emit "Unknown template node: ~a" node)])

;; Destructure struct fields
(match field
  [(struct field (uuid name type constraints doc req? uniq? _ _ _))
   (format "~a ~a~a"
           (type->sql type)
           (symbol->sql-id name)
           (if req? " NOT NULL" ""))])

✅ CANONICAL: match on token types (parser)

;; Token dispatch — already used correctly in parser.rkt
(match (current-token ps)
  [(token 'keyword "ENTITY" _ _)  (parse-entity ps)]
  [(token 'keyword "AGGREGATE" _ _) (parse-aggregate ps)]
  [(token 'keyword "SAGA" _ _)    (parse-saga ps)]
  [tok (parse-error ps (format "Unexpected token: ~a" tok))])

✅ ACCEPTABLE: case for symbol/string dispatch

;; case for fixed symbol dispatch
(case (type-spec-name type)
  [(UUID uuid) "UUID"]
  [(String TEXT Text text) "TEXT"]
  [(Integer INT int) "INTEGER"]
  [(Boolean BOOL bool) "BOOLEAN"]
  [else (error 'type->sql "Unknown type: ~a" (type-spec-name type))])

❌ ANTI-PATTERN: cond + equal? chains

;; FORBIDDEN — use case or match instead
(cond
  [(equal? key "ENTITY") (parse-entity ps)]
  [(equal? key "AGGREGATE") (parse-aggregate ps)]
  [(equal? key "SAGA") (parse-saga ps)]
  [else (error "Unknown keyword")])

;; USE case:
(case key
  [("ENTITY") (parse-entity ps)]
  [("AGGREGATE") (parse-aggregate ps)]
  [("SAGA") (parse-saga ps)]
  [else (error "Unknown keyword")])

5. String Output Patterns (Generators)

✅ CANONICAL: Template DSL for output construction

(require "../template-dsl.rkt")

(define-template generate-entity-header
  (entity ctx)
  (block 0
    (text (copyright-header))
    (line "#pragma once")
    (line "")
    (line (format "#include <~a>" (context->include-path ctx)))
    (line "")
    (line (format "namespace ~a::dba {" (symbol->snake-case (bounded-context-name ctx))))
    (line "")
    (line (format "/// @brief ~a" (or (entity-doc entity) (symbol->string (entity-name entity)))))
    (line (format "struct ~a final {" (symbol->cpp-class (entity-name entity))))
    (map-items
     (λ (fld)
       (indent 1
         (line (format "~a ~a{};"
                       (field->cpp-type fld)
                       (field->cpp-member fld)))))
     (entity-fields entity))
    (line "};")
    (line "")
    (line (format "} // namespace ~a::dba" (symbol->snake-case (bounded-context-name ctx))))))

;; Usage
(define output (emit (generate-entity-header my-entity my-ctx)))

Why: Template DSL provides structured composition, automatic indentation, conditional sections via when-true, list mapping via map-items, and source-location tracking in errors. See ADR 20260226-mandatory-template-dsl-adoption.

✅ ACCEPTABLE: format + string-append in utility functions

;; Utility functions computing names/paths (not output construction) — format is fine
(define (entity->include-path entity ctx)
  (format "~a/dba/~a/entities/~a.hpp"
          (symbol->snake-case (bounded-context-name ctx))
          (symbol->snake-case (bounded-context-name ctx))
          (symbol->cpp-filename (entity-name entity))))

;; Short single-line formatting — format is fine
(define (field->sql-column fld)
  (format "~a ~a~a"
          (symbol->sql-id (field-name fld))
          (type->sql (field-type fld))
          (if (field-required? fld) " NOT NULL" "")))

When acceptable: Functions that compute intermediate string values (names, paths, single-line SQL column definitions) that are consumed by template nodes. The distinction is: template DSL for output structure, format for value computation.

❌ ANTI-PATTERN: Large string-append blocks for output

;; FORBIDDEN — 40+ line string-append chains
(define (generate-service-proto agg ctx)
  (string-append
   (proto-header package-name)
   "import \"" proto-path-prefix (symbol->proto-filename agg-name) "\";\n"
   "import \"" proto-path-prefix "types.proto\";\n"
   "import \"google/protobuf/empty.proto\";\n"
   (if spatial-enabled?
       "import \"muscovite/spatial.proto\";\n"
       "")
   "\n"
   "service " service-name " {\n"
   ;; ... 25 more lines
   "}\n"))

Found in codebase: dslgen/generators/grpc/service.rkt (55+ line string-append), dslgen/generators/qt/state-machine-widgets.rkt (75-line string-append)

❌ ANTI-PATTERN: (if cond (format ...) "") for conditionals

;; FORBIDDEN — empty string hack for conditional output
(string-append
 (if has-events? (format "// Events: ~a\n" event-count) "")
 (if has-sagas? (format "// Sagas: ~a\n" saga-count) ""))

;; USE when-true:
(block 0
  (when-true has-events?
    (line (format "// Events: ~a" event-count)))
  (when-true has-sagas?
    (line (format "// Sagas: ~a" saga-count))))

6. Contract / Module Boundary Patterns

✅ CANONICAL: contract-out on public APIs

(provide
 (contract-out
  ;; Struct with typed fields
  [struct entity ([uuid string?]
                  [name symbol?]
                  [id-field field?]
                  [fields (listof field?)]
                  [invariants (listof invariant?)]
                  [commands (listof command?)]
                  [doc (or/c string? #f)])]
  ;; Functions with contracts
  [parse-file (-> path-string? (listof bounded-context?))]
  [symbol->sql-id (-> symbol? string?)]
  [symbol->cpp-class (-> symbol? string?)]))

Priority targets for enforcement (in order):

  1. model/ddd-types.rkt — 70 structs, the IR core
  2. model/type-registry.rkt — type mapping functions
  3. generators/headers.rkt — license header generation
  4. parser/parser.rkt — parser entry points

✅ ACCEPTABLE: Explicit provide list without contracts (migration period)

;; Acceptable during migration — explicit list without contract-out
(provide parse
         parse-file
         parse-file-with-map
         parse-string)

When acceptable: During the incremental migration period. Files without contracts should be converted when touched. This is strictly better than all-defined-out because it documents the public API.

❌ ANTI-PATTERN: (provide (all-defined-out))

;; FORBIDDEN in new files — exports everything, no API boundary
(provide (all-defined-out))

Found in codebase: dslgen/model/ddd-types.rkt, most generator files. Convert when touching these files.

7. Error Handling Patterns

✅ CANONICAL: error with component name and context

;; Always include the component/function name and actionable context
(error 'parse-entity
       "Entity ~a references undefined parent ~a at ~a:~a"
       (entity-name entity) parent-name
       (token-line tok) (token-col tok))

(error 'type->sql
       "Unknown DSL type: ~a (known types: ~a)"
       (type-spec-name type)
       (string-join (map symbol->string (hash-keys KNOWN-TYPES)) ", "))

✅ CANONICAL: raise with structured exception (for recoverable errors)

;; Define custom exceptions where callers need to catch specific failures
(struct exn:fail:parse exn:fail
  (line col token-type token-value)
  #:transparent)

(define (parse-error ps message)
  (define tok (current-token ps))
  (raise (exn:fail:parse
          (format "Parse error: ~a at ~a:~a"
                  message (token-line tok) (token-col tok))
          (current-continuation-marks)
          (token-line tok) (token-col tok)
          (token-type tok) (token-value tok))))

When to use structured exceptions: When callers need to catch and handle specific error types (e.g., parse errors that should produce user-friendly messages vs internal generator errors that indicate bugs).

✅ ACCEPTABLE: Plain error for internal generator failures

;; Internal assertion — should never happen if IR is valid
(error 'generate-entity "Entity has no id-field — IR invariant violated")

When acceptable: For internal consistency checks that indicate bugs in the generator or parser, not user input errors. These should crash with a clear message.

❌ ANTI-PATTERN: error without component name

;; FORBIDDEN — no component name, unclear origin
(error "Something went wrong")
(error (format "Unknown type: ~a" type))

;; FORBIDDEN — string "error" prefix (error already prints "error")
(error 'foo "Error: unknown type ~a" type)

  2. SPDX license header

  6. Public API section with clear separator
  7. Internal helpers section
  8. module+ test at bottom

✅ CANONICAL: #lang racket/base (not #lang racket)

;; REQUIRED for all new files
#lang racket/base

;; Import only what you need
(require racket/list    ;; for filter-map, append-map, etc.
         racket/match   ;; for match
         racket/string  ;; for string-join, string-split, etc.
         racket/format) ;; for ~a formatting

Why: #lang racket re-exports ~600 bindings. #lang racket/base re-exports ~200. Smaller namespace = faster module expansion = better byte compilation. Most generator files need only racket/list, racket/match, racket/string.

✅ CANONICAL: module+ test for inline tests

(module+ test
  (require rackunit)

  (test-case "symbol->sql-id converts PascalCase to snake_case"
    (check-equal? (symbol->sql-id 'PatientContext) "patient_context")
    (check-equal? (symbol->sql-id 'order-item) "order_item"))

  (test-case "entity with no fields raises error"
    (check-exn exn:fail?
      (λ () (validate-entity (make-entity 'Empty))))))

Why: module+ test submodules are only executed when explicitly invoked (raco test file.rkt or racket -l rackunit/text-ui file.rkt). They don't slow down normal compilation. They enable per-module testing (ADR 20260223).

❌ ANTI-PATTERN: #lang racket for generators

;; FORBIDDEN — imports 400+ unnecessary bindings, slower compilation
#lang racket

9. Template DSL Authoring Patterns

✅ CANONICAL: Template structure

(require "template-dsl.rkt")

;; Template for a C++ header file
(define-template entity-header
  (entity ctx)
  (block 0
    (text (copyright-header))              ;; Raw text: no trailing newline
    (line "#pragma once")                   ;; Line: text + newline
    (line "")                               ;; Blank line
    ;; Conditional section
    (when-true (entity-has-relationships? entity)
      (block 0
        (line "// Forward declarations")
        (map-items
         (λ (rel)
           (line (format "class ~a;" (symbol->cpp-class (relationship-target-entity rel)))))
         (entity-relationships entity))
        (line "")))
    ;; Main content
    (line (format "struct ~a final {" (symbol->cpp-class (entity-name entity))))
    ;; Map over fields with indentation
    (map-items
     (λ (fld)
       (indent 1
         (line (format "~a ~a{};" (field->cpp-type fld) (field->cpp-member fld)))))
     (cons (entity-id-field entity) (entity-fields entity)))
    (line "};")))

✅ CANONICAL: Template composition

;; Shared templates in generators/shared/templates/cpp-templates.rkt
(define-template cpp-doc-comment
  (text)
  (block 0
    (line "/**")
    (line (format " * @brief ~a" text))
    (line " */")))

;; Compose in a larger template
(define-template entity-header
  (entity ctx)
  (block 0
    (text (emit (cpp-doc-comment (or (entity-doc entity) "Entity"))))
    (line (format "struct ~a final {" (symbol->cpp-class (entity-name entity))))
    ;; ...
    (line "};")))

✅ CANONICAL: Emit to string

;; emit returns a string — write it to file
(define content (emit (entity-header entity ctx)))
(display-to-file content output-path #:exists 'replace)

;; Or in a generator function that returns string
(define (generate-entity-hpp entity ctx)
  (emit (entity-header entity ctx)))

Key template node reference

Node Purpose Example
(text str) Raw text, no newline (text (copyright-header))
(line str) Text + newline (line "#pragma once")
(line "") Blank line Separator between sections
(block n items ...) Indented block (block 0 (line "class {") ...)
(indent n str) Indent string by n×4 spaces (indent 1 (line "int x;"))
(join sep items) Join with separator (join ", " field-strings)
(when-true cond tmpl) Conditional (when-true has-doc? (line doc))
(map-items fn lst) Map to templates (map-items field->line fields)

10. Command-Line Configuration Pattern

✅ CANONICAL: Config struct from command-line

;; Define an immutable config struct
(struct gen-config
  (project-path cpp-dba? grpc? qt-frontend fresh? dry-run?)
  #:transparent)

;; Build config from command-line, then resolve dependencies purely
(define raw-config
  (command-line
   #:program "generate"
   #:once-each
   [("--project") path "Project file" (set! project-path path)]
   [("--cpp-dba") "Generate C++ DBA" (set! cpp-dba? #t)]
   [("--grpc") "Generate gRPC" (set! grpc? #t)]
   ;; ...
   ))

(define (resolve-dependencies config)
  ;; WHY: Pure function — dependencies resolved without mutation
  (let* ([config (if (gen-config-qt-frontend config)
                     (struct-copy gen-config config [grpc? #t])
                     config)]
         [config (if (gen-config-grpc? config)
                     (struct-copy gen-config config [cpp-dba? #t])
                     config)])
    config))

Note: set! inside command-line flag handlers is acceptable — Racket's command-line macro specifically uses this pattern. The key is to capture the result into an immutable struct immediately and resolve dependencies via a pure function.

Quick Reference Card

Category ✅ Use ❌ Avoid
Iteration (for/list ([x (in-list xs)]) ...) (for/list ([x xs]) ...)

Accumulation (for/fold ([acc '()]) ...) (define acc '()) (set! acc ...)
Flat-map (for/append ([x ...]) ...) (apply append (map ...))
Hash build (for/hash ([x ...]) (values k v)) (make-hash) + hash-set!
Hash literal (hash 'a 1 'b 2) (define h (make-hash)) (hash-set! h ...)
Dispatch (match x [(pat) ...]) or (case x ...) (cond [(equal? x ...) ...] ...)
Output (define-template name ...) + (emit ...) (string-append (format ...) ...)

grep -rc "string-append\|format " dslgen/generators/ --include="*.rkt" | awk -F: '{s+=$2} END {print s}'

grep -rl "^#lang racket$" dslgen/ --include="*.rkt" | wc -l
