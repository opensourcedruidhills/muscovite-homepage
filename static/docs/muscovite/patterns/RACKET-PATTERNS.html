Racket Pattern Catalog for Muscovite Generators

Racket Pattern Catalog for Muscovite Generators

Purpose: Constraint document for AI agents and human developers. Every Racket pattern in Muscovite generators must use the canonical form. This document is the single source of truth for "how to write Racket in this project."

Enforcement: ADR 20260226-racket-idiomatic-style-and-byte-compilation.md, ADR 20260226-mandatory-template-dsl-adoption.md

Scope: All .rkt files in dslgen/. Parser, model, generators, migration, validation, tests.

How to Use This Document

Each section has three tiers:

Tier Meaning
✅ CANONICAL The required pattern. Use this in all new and modified code.
✅ ACCEPTABLE Permitted in specific circumstances (documented).
❌ ANTI-PATTERN Forbidden in new code. Convert when touching existing files.

When an AI agent generates Racket code for Muscovite, it MUST use only CANONICAL or ACCEPTABLE patterns. When an AI agent modifies an existing file, it MUST convert any anti-patterns in the modified function to canonical form.

1. Iterator Patterns

✅ CANONICAL: in-list on all for over lists

;; Always annotate the sequence type
(for/list ([ctx (in-list contexts)])
  (generate-context ctx))

(for ([field (in-list (entity-fields entity))])
  (validate-field field))

(for/fold ([acc '()])
          ([ctx (in-list contexts)])
  (append acc (generate-context ctx)))

Why: in-list eliminates generic sequence dispatch at runtime. Without it, Racket checks at every iteration whether the value is a list, vector, hash, string, etc. 5–10% speedup on iteration-heavy generators. More importantly, it communicates intent and catches type errors earlier.

✅ CANONICAL: map / filter / filter-map for simple transforms

;; Pure transform: prefer map
(map entity-name (bounded-context-entities ctx))

;; Filter: prefer filter/filter-map
(filter entity-event-sourced? entities)
(filter-map entity-doc entities)  ;; drops #f values

;; Parallel map over multiple lists
(map (λ (name type) (format "~a ~a;" type name))
     field-names field-types)

Why: map/filter are declarative — the reader knows instantly that the output is a same-length (or shorter) list with no side effects. for/list is for when the body needs define, multiple expressions, or complex control flow.

❌ ANTI-PATTERN: filter-map + lambda with (if pred transform #f)

;; FORBIDDEN — use for/list + #:when
(filter-map (λ (x) (if (pred x) (transform x) #f)) xs)
;; USE:
(for/list ([x (in-list xs)] #:when (pred x))
  (transform x))

❌ ANTI-PATTERN: filter-map + lambda with (and pred transform) (where and filters)

;; FORBIDDEN — use for/list + #:when
(filter-map (λ (x) (and (pred1 x) (pred2 x) (transform x))) xs)
;; USE:
(for/list ([x (in-list xs)] #:when (and (pred1 x) (pred2 x)))
  (transform x))

❌ ANTI-PATTERN: when inside filter-map (BUG — returns # not #f)

;; FORBIDDEN — use for/list + #:when
(filter-map (λ (x) (when (pred x) (transform x))) xs)
;; USE:
(for/list ([x (in-list xs)] #:when (pred x))
  (transform x))

✅ ACCEPTABLE: filter-map with accessor or hash-ref (no conditional lambda)

;; OK — drops missing values
(filter-map entity-doc entities)
(filter-map (λ (x) (hash-ref table x #f)) keys)
(filter-map (λ (x) (findf pred list)) items)

✅ ACCEPTABLE: for/list when body needs bindings or control flow

;; Body needs define — for/list is appropriate
(for/list ([fld (in-list fields)])
  (define cpp-type (field->cpp-type fld))
  (define cpp-name (field->cpp-member fld))
  (format "    ~a ~a;" cpp-type cpp-name))

❌ ANTI-PATTERN: Bare for without sequence annotation

;; FORBIDDEN — generic sequence dispatch, no type safety
(for/list ([x xs]) (f x))
(for ([x xs]) (do-something x))

;; FORBIDDEN — struct accessor returns list, needs in-list
(for/list ([m (ddd:repository-methods repo)]) ...)

Found in codebase: 140 out of 253 generator files (55%) are missing in-list annotations. Concentrated in dslgen/generators/qt/, dslgen/generators/postgres/, and dslgen/generators/cpp/. Run grep -rL "in-list" dslgen/generators/ --include="*.rkt" | wc -l to track progress.

❌ ANTI-PATTERN: for/list where map suffices

;; FORBIDDEN — map is simpler and clearer
(for/list ([x (in-list xs)]) (f x))
;; USE:
(map f xs)

;; FORBIDDEN — nested for/list for a pure transform
(string-join
 (for/list ([ev (in-list events)])
   (format "case ~a:" ev))
 "\n")
;; USE:
(string-join (map (λ (ev) (format "case ~a:" ev)) events) "\n")

2. Accumulation Patterns (for/fold vs set!)

✅ CANONICAL: for/fold for building results

;; Build a list by folding
(define results
  (for/fold ([acc '()])
            ([ctx (in-list contexts)])
    (if (context-has-entities? ctx)
        (cons (generate-context ctx) acc)
        acc)))

;; Build a hash by folding
(define entity-map
  (for/fold ([h (hash)])
            ([ent (in-list entities)])
    (hash-set h (entity-uuid ent) ent)))

;; Accumulate multiple values
(define-values (tables views)
  (for/fold ([tables '()] [views '()])
            ([proj (in-list projections)])
    (if (projection-materialized? proj)
        (values tables (cons proj views))
        (values (cons proj tables) views))))

✅ CANONICAL: for/list with #:when for filtered collection

;; Collect only matching items
(for/list ([fld (in-list fields)]
           #:when (field-required? fld))
  (field-name fld))

✅ CANONICAL: for/list with #:when + #:do for filtered collection with bindings

Use #:do to introduce bindings inside the loop when the predicate or transform needs an intermediate value.

;; Transform requires an intermediate binding — use #:do [define ...]
(for/list ([fld (in-list fields)]
           #:do [(define cpp-type (dsl-type->cpp fld))]
           #:when cpp-type)
  (format "~a ~a;" cpp-type (field->cpp-member fld)))

;; Multiple intermediate bindings before the guard
(for/list ([cmd (in-list commands)]
           #:do [(define sig (command-signature cmd))
                 (define grpc-name (symbol->grpc-method (command-name cmd)))]
           #:when (command-exported? cmd))
  (format-rpc grpc-name sig))

When to use: The body needs a binding that is also used in the #:when guard, or when computing the binding once is more efficient than computing it in both the guard and the body. Do NOT use #:do merely to avoid let — prefer let/define in the body if the binding is only used after the guard.

✅ CANONICAL: for/append for flat-mapping

;; Flatten nested lists
(define all-fields
  (for/append ([ent (in-list entities)])
    (entity-fields ent)))

✅ ACCEPTABLE: let loop for stateful iteration

;; Parser token consumption (inherently stateful)
(let loop ([tokens remaining])
  (cond
    [(null? tokens) '()]
    [(token-keyword? (car tokens) "END")
     (cdr tokens)]
    [else
     (cons (parse-item (car tokens))
           (loop (cdr tokens)))]))

❌ ANTI-PATTERN: set! accumulator loop

;; FORBIDDEN
(define issues '())
(for ([repo (in-list (bounded-context-repositories ctx))])
  (define for-agg (repository-for-aggregate repo))
  (unless (set-member? agg-names for-agg)
    (set! issues
          (cons (validation-issue 'error 'reference ...) issues))))
issues

;; USE for/fold:
(for/fold ([issues '()])
          ([repo (in-list (bounded-context-repositories ctx))])
  (define for-agg (repository-for-aggregate repo))
  (if (set-member? agg-names for-agg)
      issues
      (cons (validation-issue 'error 'reference ...) issues)))

Found in codebase: dslgen/validator.rkt (18 instances), dslgen/verification/verify-schema.rkt (8 instances), dslgen/verification/verify-types.rkt (4 instances). Total set! in generators: 56 occurrences.

❌ ANTI-PATTERN: set! counter

;; FORBIDDEN
(define count 0)
(for ([pair (in-list searchable-pairs)])
  (generate-file pair)
  (set! count (add1 count)))
(printf "Total: ~a files\n" count)

;; USE:
(define count (length searchable-pairs))
(for ([pair (in-list searchable-pairs)])
  (generate-file pair))
(printf "Total: ~a files\n" count)

;; OR if count depends on a condition:
(define count
  (for/sum ([pair (in-list searchable-pairs)]
            #:when (generate-file pair))
    1))

❌ ANTI-PATTERN: set! counter in for loop

;; FORBIDDEN — use for/fold
(define count 0)
(for ([item (in-list items)])
  (process item)
  (set! count (add1 count)))

;; CORRECT:
(define count
  (for/fold ([n 0]) ([item (in-list items)])
    (process item)
    (add1 n)))

❌ ANTI-PATTERN: set! accumulator in for loop

;; FORBIDDEN — use for/fold
(define results '())
(for ([item (in-list items)])
  (when (valid? item)
    (set! results (cons (transform item) results))))

;; CORRECT:
(define results
  (for/fold ([acc '()]) ([item (in-list items)]
                          #:when (valid? item))
    (cons (transform item) acc)))

✅ ACCEPTABLE: Mutable hash for registries

;; OK — global registries are inherently stateful
(define registry (make-hash))
(define (register! name proc)
  (hash-set! registry name proc))

✅ ACCEPTABLE: Box for stateful closures

;; OK — ID generator with encapsulated state
(define (make-id-generator)
  (define counter (box 0))
  (lambda (prefix)
    (define n (add1 (unbox counter)))
    (set-box! counter n)
    (format "~a~a" prefix n)))

3. Hash / Map Patterns

✅ CANONICAL: Named constants for C++ type strings (FTR-522)

C++ type names appear 150+ times across generator files. Inline strings create silent disagreements (e.g., Integer mapping to std::int32_t in one file and std::int64_t in another). Named constants make disagreements impossible and enable IDE-supported rename refactoring.

;; Import type constants — never use raw strings
(require "../type-registry.rkt")  ; cpp:string, cpp:int32, cpp:uuid, etc.

;; Use constants, not string literals
(define (my-field-type spec)
  (case (type-spec-name spec)
    [(String TEXT) cpp:string]
    [(UUID) cpp:uuid]
    [(Integer) cpp:int32]
    [else (error 'my-field-type "Unknown type: ~a" (type-spec-name spec))]))

Available constants: cpp:string, cpp:string-view, cpp:bool, cpp:uuid, cpp:int32, cpp:int64, cpp:uint64, cpp:float, cpp:double, cpp:decimal, cpp:timestamp, cpp:date, cpp:time-of-day, cpp:point, cpp:polygon, cpp:linestring

❌ ANTI-PATTERN: Inline C++ type string literals

;; FORBIDDEN — use cpp:uuid constant
(format "    boost::uuids::uuid ~a{};" id-name)

;; FORBIDDEN — use cpp:string constant
(define (type->cpp type)
  (case type
    [(String) "std::string"]    ; should be cpp:string
    [(UUID) "boost::uuids::uuid"] ; should be cpp:uuid
    ...))

❌ ANTI-PATTERN: Parallel type mapping functions

;; FORBIDDEN — there is ONE type registry (generators/type-registry.rkt)
;; Do not create inline case/cond type mappings in individual generators
(define (type-spec->cpp-type ts)
  (case (type-spec-name ts)
    [(UUID) "std::string"]    ;; WRONG — disagrees with registry
    [(Integer) "int64_t"]     ;; WRONG — missing std:: prefix
    [else "std::string"]))

;; CORRECT — delegate to the registry
(define (type-spec->cpp-type ts)
  (define entry (registry-lookup (type-spec-name ts)))
  (if entry
      (cpp-type-entry-cpp-type entry)
      "std::string"))

✅ CANONICAL: Immutable hash construction

;; Literal hash
(define type-map
  (hash 'UUID "boost::uuids::uuid"
        'String "std::string"
        'Integer "std::int32_t"))

;; Hash from iteration
(define entity-map
  (for/hash ([ent (in-list entities)])
    (values (entity-uuid ent) ent)))

;; Hash from fold (when multiple sources feed in)
(define uuid-map
  (for*/fold ([h (hash)])
             ([ctx (in-list contexts)]
              [ent (in-list (all-entities-in-context ctx))])
    (hash-set h (entity-uuid ent) (list ent ctx))))

;; Updating an immutable hash
(define h2 (hash-set h 'new-key new-value))
(define h3 (hash-update h 'count add1 0))

✅ ACCEPTABLE: make-hash for module-level registries

;; WHY: Module-level registry populated at load time, read-only thereafter
(define generators (make-hash))

(define (register-generator name proc)
  (hash-set! generators name proc))

(define (get-generator name)
  (hash-ref generators name
             (λ () (error 'get-generator "Unknown generator: ~a" name))))

When acceptable: Only for module-scoped registries that are write-once-read-many. Must include a WHY: comment.

✅ ACCEPTABLE: make-hash for performance-critical paths

;; WHY: Performance — >1000 insertions in tight loop; immutable hash would
;; create O(n) intermediate copies
(define h (make-hash))
(for ([row (in-list large-dataset)])
  (hash-set! h (row-key row) (row-value row)))
;; Freeze: use only hash-ref after this point

When acceptable: Only when profiling shows immutable hash is a bottleneck (>1000 insertions). Must include a WHY: performance comment.

❌ ANTI-PATTERN: Mutable hash for build-then-read

;; FORBIDDEN — accumulation into mutable hash
(define h (make-hash))
(for ([ctx (in-list contexts)])
  (for ([agg (in-list (bounded-context-aggregates ctx))])
    (hash-set! h (entity-uuid (aggregate-root agg))
               (list (aggregate-root agg) ctx (aggregate-uuid agg)))))
h

;; USE for*/fold:
(for*/fold ([h (hash)])
           ([ctx (in-list contexts)]
            [agg (in-list (bounded-context-aggregates ctx))])
  (hash-set h (entity-uuid (aggregate-root agg))
            (list (aggregate-root agg) ctx (aggregate-uuid agg))))

Found in codebase: dslgen/migration/diff-model.rkt (entities->uuid-map, relationships->uuid-map), dslgen/incremental/incremental-gen.rkt (11 hash-set! calls building uuid->ctx-name map)

4. Match / Dispatch Patterns

✅ CANONICAL: match on structs

;; Dispatch on struct type
(match node
  [(template-node 'text str) str]
  [(template-node 'line str) (string-append str "\n")]
  [(template-node 'block (cons level items))
   (apply string-append (map (curry emit-node level) items))]
  [_ (error 'emit "Unknown template node: ~a" node)])

;; Destructure struct fields
(match field
  [(struct field (uuid name type constraints doc req? uniq? _ _ _))
   (format "~a ~a~a"
           (type->sql type)
           (symbol->sql-id name)
           (if req? " NOT NULL" ""))])

✅ CANONICAL: match on token types (parser)

;; Token dispatch — already used correctly in parser.rkt
(match (current-token ps)
  [(token 'keyword "ENTITY" _ _)  (parse-entity ps)]
  [(token 'keyword "AGGREGATE" _ _) (parse-aggregate ps)]
  [(token 'keyword "SAGA" _ _)    (parse-saga ps)]
  [tok (parse-error ps (format "Unexpected token: ~a" tok))])

✅ ACCEPTABLE: case for symbol/string dispatch

;; case for fixed symbol dispatch
(case (type-spec-name type)
  [(UUID uuid) "UUID"]
  [(String TEXT Text text) "TEXT"]
  [(Integer INT int) "INTEGER"]
  [(Boolean BOOL bool) "BOOLEAN"]
  [else (error 'type->sql "Unknown type: ~a" (type-spec-name type))])

❌ ANTI-PATTERN: cond + equal? chains

;; FORBIDDEN — use case or match instead
(cond
  [(equal? key "ENTITY") (parse-entity ps)]
  [(equal? key "AGGREGATE") (parse-aggregate ps)]
  [(equal? key "SAGA") (parse-saga ps)]
  [else (error "Unknown keyword")])

;; USE case:
(case key
  [("ENTITY") (parse-entity ps)]
  [("AGGREGATE") (parse-aggregate ps)]
  [("SAGA") (parse-saga ps)]
  [else (error "Unknown keyword")])

✅ CANONICAL: Guard clause chain → cond

When nested if forms all return the same fallback value ("guard chain"), use cond:

;; BEFORE — depth-4 nested ifs, all returning rel-issues as fallback
(if (not meta) rel-issues
  (if (not target-pk-type) rel-issues
    (if (not source-field) rel-issues
      (if (eq? fk-type target-pk-type) rel-issues
        (cons (make-issue ...) rel-issues)))))

;; AFTER — cond with guard clauses
(cond
  [(not meta) rel-issues]
  [(not target-pk-type) rel-issues]
  [(not source-field) rel-issues]
  [(eq? fk-type target-pk-type) rel-issues]
  [else (cons (make-issue ...) rel-issues)])

When to use: Any chain of 3+ nested ifs where the "no" branch returns the same value. The cond makes it clear these are sequential guard checks.

✅ CANONICAL: Decision tree → cond with compound predicates

When nested if forms create a decision tree with different actions per branch:

;; BEFORE — depth-3 nested ifs with different branches
(if has-optimistic?
    (if version-already-declared?
        (for/list ([col (in-list cols)])
          (if (equal? col version-field-name)
              (format "~a = ~a + 1" col col)
              (format "~a = $~a" col (next-param!))))
        (append
         (for/list ([col (in-list cols)])
           (format "~a = $~a" col (next-param!)))
         (list (format "~a = ~a + 1" version-field-name version-field-name))))
    raw-set-clauses)

;; AFTER — cond with compound predicates
(cond
  [(and has-optimistic? version-already-declared?)
   (for/list ([col (in-list cols)])
     (if (equal? col version-field-name)
         (format "~a = ~a + 1" col col)
         (format "~a = $~a" col (next-param!))))]
  [has-optimistic?
   (append
    (for/list ([col (in-list cols)])
      (format "~a = $~a" col (next-param!)))
    (list (format "~a = ~a + 1" version-field-name version-field-name)))]
  [else raw-set-clauses])

When to use: Any nested if tree where different branches have different actions. Flatten using and/or compound predicates in the cond clauses.

❌ ANTI-PATTERN: Nested if chains (depth 3+)

;; FORBIDDEN — use cond instead
(if has-optimistic?
    (if version-declared?
        (if (equal? col version-field)
            ...
            ...)
        ...)
    ...)

Applies to: Any if nesting of 3+ levels. Convert to cond with compound predicates or extract a helper function.

✅ ACCEPTABLE: Simple ternary if

;; Two-branch if is fine — no nesting needed
(if required? " NOT NULL" "")
(if (null? items) default-val (car items))
(if condition result-a result-b)

When to use: Only for simple two-branch conditionals with no nesting.

❌ ANTI-PATTERN: cond without else

;; FORBIDDEN — returns #<void> on unmatched case, which is almost always a bug
(cond
  [(add-context-op? op) (hash-set! ...)]
  [(add-entity-op? op) (hash-set! ...)]
  [(add-field-op? op) (hash-set! ...)])
;; What happens for remove-entity-op? Silent void.

;; CORRECT — explicit else, even if it's just void or an error:
(cond
  [(add-context-op? op) (hash-set! ...)]
  [(add-entity-op? op) (hash-set! ...)]
  [(add-field-op? op) (hash-set! ...)]
  [else (void)])  ; Intentionally ignored — document the intent

Why: Makes intent explicit. If a case should do nothing, [else (void)] documents that it's intentional, not forgotten.

4A. Local Binding Patterns

✅ CANONICAL: Flat define sequences for 3+ bindings

Use flat define sequences at function top-level for pipelines with 3 or more bindings.

;; CANONICAL — flat defines for complex pipelines
(define (generate-entity-table ent ctx)
  (define name        (entity-name ent))
  (define fields      (entity-fields ent))
  (define class       (symbol->cpp-class name))
  (define table       (symbol->sql-id name))
  (define schema      (or (ctx-schema ctx) "public"))
  (define id-field    (entity-id-field ent))
  (define constraints (entity-constraints ent))
  (format "CREATE TABLE ~a.~a ..." schema table))

Why: Flat define sequences are more readable than deeply nested let* for multi-step transformations. Each binding is clearly visible, alignment is consistent, and adding/removing bindings doesn't require reindenting the entire block.

✅ CANONICAL: match-define for list/pair destructuring

;; Destructure lists and pairs with match-define
(match-define (cons key value) (car result))
(match-define (list mode interval unit) (parse-refresh-mode ps))
(match-define (list timestamp level message) log-entry)

Why: More expressive than car/cdr chains, catches arity mismatches at runtime, documents expected structure.

✅ ACCEPTABLE: let* for 1-2 bindings

;; ACCEPTABLE — let* is fine for short sequences
(let* ([cfg (make-project-config ...)]
       [result (generate-cmake-presets cfg '())])
  (check-equal? (length result) 1))

Why: No readability penalty for 1-2 bindings. The nesting is shallow and the scope is clear.

❌ ANTI-PATTERN: Deep let* with 3+ bindings

;; FORBIDDEN — use flat define instead
(let* ([proj-name (symbol->string (project-config-name project-config))]
       [version (project-config-version project-config)]
       [gen-opts (project-config-generate-options project-config)]
       [enable-grpc (hash-ref gen-opts "grpc" #f)]
       [enable-qt (or (hash-ref gen-opts "qt_widgets" #f)
                     (hash-ref gen-opts "qt_qml" #f))]
       [test-framework "doctest"])
  (list ...))

Why: Deep let* nesting harms readability, especially when bindings have complex expressions. Flat define sequences make dependencies explicit and are easier to reorder or modify.

❌ ANTI-PATTERN: cadr/caddr/cadddr chains

;; FORBIDDEN — use second/third/fourth or match-define
(cadr result)    ; Use: (second result)
(caddr result)   ; Use: (third result)
(cadddr result)  ; Use: (fourth result)

;; Or better, use match-define:
(match-define (list _ key value extra) result)

Why: cadr/caddr/cadddr are opaque to readers unfamiliar with Lisp ancestry. second/third/fourth are self-documenting. match-define is even better when the structure is known.

5. String Output Patterns (Generators)

✅ CANONICAL: Template DSL for output construction

(require "../template-dsl.rkt")

(define-template generate-entity-header
  (entity ctx)
  (block 0
    (text (copyright-header))
    (line "#pragma once")
    (line "")
    (line (format "#include <~a>" (context->include-path ctx)))
    (line "")
    (line (format "namespace ~a::dba {" (symbol->snake-case (bounded-context-name ctx))))
    (line "")
    (line (format "/// @brief ~a" (or (entity-doc entity) (symbol->string (entity-name entity)))))
    (line (format "struct ~a final {" (symbol->cpp-class (entity-name entity))))
    (map-items
     (λ (fld)
       (indent 1
         (line (format "~a ~a{};"
                       (field->cpp-type fld)
                       (field->cpp-member fld)))))
     (entity-fields entity))
    (line "};")
    (line "")
    (line (format "} // namespace ~a::dba" (symbol->snake-case (bounded-context-name ctx))))))

;; Usage
(define output (emit (generate-entity-header my-entity my-ctx)))

Why: Template DSL provides structured composition, automatic indentation, conditional sections via when-true, list mapping via map-items, and source-location tracking in errors. See ADR 20260226-mandatory-template-dsl-adoption.

✅ ACCEPTABLE: format + string-append in utility functions

;; Utility functions computing names/paths (not output construction) — format is fine
(define (entity->include-path entity ctx)
  (format "~a/dba/~a/entities/~a.hpp"
          (symbol->snake-case (bounded-context-name ctx))
          (symbol->snake-case (bounded-context-name ctx))
          (symbol->cpp-filename (entity-name entity))))

;; Short single-line formatting — format is fine
(define (field->sql-column fld)
  (format "~a ~a~a"
          (symbol->sql-id (field-name fld))
          (type->sql (field-type fld))
          (if (field-required? fld) " NOT NULL" "")))

When acceptable: Functions that compute intermediate string values (names, paths, single-line SQL column definitions) that are consumed by template nodes. The distinction is: template DSL for output structure, format for value computation.

❌ ANTI-PATTERN: Large string-append blocks for output

;; FORBIDDEN — 40+ line string-append chains
(define (generate-service-proto agg ctx)
  (string-append
   (proto-header package-name)
   "import \"" proto-path-prefix (symbol->proto-filename agg-name) "\";\n"
   "import \"" proto-path-prefix "types.proto\";\n"
   "import \"google/protobuf/empty.proto\";\n"
   (if spatial-enabled?
       "import \"muscovite/spatial.proto\";\n"
       "")
   "\n"
   "service " service-name " {\n"
   ;; ... 25 more lines
   "}\n"))

Found in codebase: dslgen/generators/grpc/service.rkt (55+ line string-append), dslgen/generators/qt/state-machine-widgets.rkt (75-line string-append). Template DSL adoption: 22 out of 253 generator files (9%) use define-template. Remaining: 6,451 format/string-append calls across generators.

❌ ANTI-PATTERN: (if cond (format ...) "") for conditionals

;; FORBIDDEN — empty string hack for conditional output
(string-append
 (if has-events? (format "// Events: ~a\n" event-count) "")
 (if has-sagas? (format "// Sagas: ~a\n" saga-count) ""))

;; USE when-true:
(block 0
  (when-true has-events?
    (line (format "// Events: ~a" event-count)))

  • For multi-value return, use (values type1? type2?) or any if complex

❌ ANTI-PATTERN: Hand-rolled assertion frameworks

9. Template DSL Authoring Patterns

✅ CANONICAL: Template structure

(require "template-dsl.rkt")

;; Template for a C++ header file
(define-template entity-header
  (entity ctx)
  (block 0
    (text (copyright-header))              ;; Raw text: no trailing newline
    (line "#pragma once")                   ;; Line: text + newline
    (line "")                               ;; Blank line
    ;; Conditional section
    (when-true (entity-has-relationships? entity)
      (block 0
        (line "// Forward declarations")
        (map-items
         (λ (rel)
           (line (format "class ~a;" (symbol->cpp-class (relationship-target-entity rel)))))
         (entity-relationships entity))
        (line "")))
    ;; Main content
    (line (format "struct ~a final {" (symbol->cpp-class (entity-name entity))))
    ;; Map over fields with indentation (indent wraps a string, inside line)
    (map-items
     (λ (fld)
       (line (indent 1
         (format "~a ~a{};" (field->cpp-type fld) (field->cpp-member fld)))))
     (cons (entity-id-field entity) (entity-fields entity)))
    (line "};")))

✅ CANONICAL: Template composition

;; Shared templates in generators/shared/templates/cpp-templates.rkt

  (text)
  (block 0
    (line "/**")
    (line (format " * @brief ~a" text))
    (line " */")))

;; Compose in a larger template
(define-template entity-header
  (entity ctx)
  (block 0
    (text (emit (cpp-doc-comment (or (entity-doc entity) "Entity"))))
    (line (format "struct ~a final {" (symbol->cpp-class (entity-name entity))))
    ;; ...
    (line "};")))

❌ ANTI-PATTERN: indent on template nodes

THE #1 source of agent errors. indent returns a string, not a template node. It cannot wrap block, line, or other template nodes.

;; ❌ WRONG — indent takes a string, not a template node
(indent 2 (block 0 (line "foo")))           ;; ERROR: contract violation
(indent 1 (line "int x;"))                  ;; ERROR: line returns template-node, not string

;; ✅ CORRECT — indent wraps a string, use inside (line ...)
(line (indent 1 "int x;"))                  ;; → "    int x;\n"
(line (indent 2 (format "~a ~a;" type name))) ;; → "        int age;\n"

;; ✅ CORRECT — use indent-block for indenting template sub-trees
(indent-block 1
  (line "int x;")
  (line "int y;"))                          ;; All lines indented 4 spaces

;; ✅ CORRECT — embed padding in format strings for statement compilers
(define pad (make-string (* 4 depth) #\space))
(format "~aconst auto ~a = ~a;\n" pad name value)

      (line (format "        const ~a& params) const {" params-struct))

      (text body)                            ;; ← raw string from compiler
      (line "    return decision;")

             (expression->cpp (let-stmt-expr stmt) "entity"))]
    ;; ... more statement types
    ))

9A. Common Template DSL Migration Pitfalls (FTR-558)

The v1.77.0 migration from format/string-append to the template DSL introduced 5 recurring bug patterns. All are now covered by regression tests in test-migration-regression.rkt and test-template-dsl-conformance.rkt.

Pitfall 1: when-true with 3 arguments (eager else-branch)

when-true takes exactly 2 arguments: condition and body. Passing a third "else" argument causes an arity error. Unlike Racket's if, there is no else branch — use when-false or choose instead.

;; ❌ WRONG — arity violation, crashes at call site
(when-true has-doc?
  (line (format "/// ~a" doc))
  (line "/// No documentation"))

;; ✅ CORRECT — use when-true + when-false pair
(when-true has-doc?
  (line (format "/// ~a" doc)))
(when-false has-doc?
  (line "/// No documentation"))

;; ✅ CORRECT — use choose for multi-way conditional
(choose
  (cons has-doc? (line (format "/// ~a" doc)))
  (cons #t       (line "/// No documentation")))

Pitfall 2: map-items argument order reversed

map-items takes (proc list), not (list proc). This mirrors map but is easy to confuse when migrating from for/list which puts the list in the header.

;; ❌ WRONG — list first, proc second → contract violation (list is not procedure?)
(map-items (entity-fields entity)
  (λ (fld) (line (format "~a ~a;" (field-type fld) (field-name fld)))))

;; ✅ CORRECT — proc first, list second (like map)
(map-items
  (λ (fld) (line (format "~a ~a;" (field-type fld) (field-name fld))))
  (entity-fields entity))

Pitfall 3: text wrapping a template node instead of a string

text has contract (-> string? any/c). Passing a template-node (the return value of another template constructor) triggers a contract violation. Use emit to convert a template node to a string first, or restructure to avoid nesting.

;; ❌ WRONG — text gets a template-node, not a string → contract violation
(text (line "hello"))
(text (when-true condition (line "x")))

;; ✅ CORRECT — use emit to convert template to string, then wrap in text
(text (emit (line "hello")))

;; ✅ BETTER — restructure to avoid the nesting entirely
(line "hello")

Pitfall 4: Spaces in proto message names

String interpolation bugs during migration can produce proto message names with embedded spaces (e.g., message UpdatePatient Request). This is syntactically invalid in protobuf.

;; ❌ WRONG — space between entity name and suffix
(line (format "message ~a ~a {" (entity-name ent) "Request"))

;; ✅ CORRECT — concatenate into a single identifier
(line (format "message ~aRequest {" (symbol->PascalCase (entity-name ent))))

Pitfall 5: when-true with eagerly-evaluated else body in CMake emitter

In the CMake emitter, when-true was sometimes used with an else clause in a block context. Because block evaluates all its children eagerly (they are template nodes constructed at call time), the "else" body would crash even when the condition was true, since the else-branch expression was evaluated during node construction.

;; ❌ WRONG — else-body is evaluated eagerly during block construction
(block 0
  (when-true enable-grpc
    (line "find_package(gRPC REQUIRED)"))
  ;; This next line runs regardless of enable-grpc:
  (when-true (not enable-grpc)
    (line (format "# gRPC disabled for ~a" (some-fn-that-crashes)))))

;; ✅ CORRECT — guard expensive computations behind the condition
(block 0
  (when-true enable-grpc
    (line "find_package(gRPC REQUIRED)"))
  (when-false enable-grpc
    (line "# gRPC disabled")))

Key insight: All template constructors are eagerly evaluated at call time — they build data structures. The condition in when-true only controls which data structure template-eval later emits. If constructing the body itself crashes (e.g., calling a function on #f), the crash happens before when-true can check the condition.

10. Command-Line Configuration Pattern

✅ CANONICAL: Config struct from command-line

The generator CLI (generate.rkt) uses command-line to build an immutable config struct, then resolves flag dependencies via a pure function. As of v1.65, the CLI provides ~25 flags:

Category Flags
Required --project <path>
Generation targets --cpp-dba, --cpp-dba-test, --grpc, --grpc-test, --qt-frontend <widgets|qml|both>, --qt-test, --doc, --fact-base, --messaging, --tlaplus
Lifecycle --full (force regen), --fresh (delete output first), --force-fresh (fresh + ignore Sqitch), --force-regen-stubs (write .new sidecar files)
Validation --lint-only (parse + validate, no codegen), --format <text|json>
Debug --emit-ir (write IR to debug-ir.sexp), --dry-run (preview without writing)
Output -o <dir>, --output <dir> (override output directory)
Deprecated --qt (use --qt-frontend widgets) 
;; Define an immutable config struct
(struct gen-config
  (project-path cpp-dba? grpc? qt-frontend fresh? dry-run?)
  #:transparent)

;; Build config from command-line, then resolve dependencies purely
(define raw-config
  (command-line
   #:program "generate"
   #:once-each
   [("--project") path "Project file" (set! project-path path)]
   [("--cpp-dba") "Generate C++ DBA" (set! cpp-dba? #t)]
   [("--grpc") "Generate gRPC" (set! grpc? #t)]
   ;; ...
   ))

(define (resolve-dependencies config)
  ;; WHY: Pure function — dependencies resolved without mutation
  (let* ([config (if (gen-config-qt-frontend config)
                     (struct-copy gen-config config [grpc? #t])
                     config)]
         [config (if (gen-config-grpc? config)
                     (struct-copy gen-config config [cpp-dba? #t])
                     config)])
    config))

Dependency chain: --qt-frontend → implies --grpc → implies --cpp-dba. The resolve-dependencies function handles this transitively.

Note: set! inside command-line flag handlers is acceptable — Racket's command-line macro specifically uses this pattern. The key is to capture the result into an immutable struct immediately and resolve dependencies via a pure function.

11. New Module Checklist

Run these steps every time you create a new .rkt module. Missing items from provide cause silent runtime failures ("unbound identifier") that are only caught during integration testing. This was learned from FTR-472.

Step 1 — Load test

Verify the module loads without error:

racket -e '(require "dslgen/path/to/new-file.rkt")'

A clean exit (no output, exit 0) means the file parses and all its dependencies resolve.

Step 2 — Symbol spot-check

Verify every exported accessor is actually reachable:

racket -e '(require "dslgen/path/to/new-file.rkt") (display (list accessor1 accessor2 accessor3))'

If any name is missing from provide, this crashes with unbound identifier — catching the bug before integration tests.

Step 3 — Provide completeness check

Diff your (provide ...) list against all defined public symbols:

# List what you're providing
grep -A50 '(provide' dslgen/path/to/new-file.rkt | grep -v '^;'

# List all top-level defines (public API candidates)
grep '^(define ' dslgen/path/to/new-file.rkt

Every defined function/struct that callers will need must appear in provide. Struct accessor functions (e.g. my-struct-field) are generated automatically but must be explicitly listed in provide — they are NOT exported by default.

Step 4 — Byte-compile

cd dslgen && raco make path/to/new-file.rkt

This catches syntax errors and unresolved requires that racket -e may not surface.

✅ Checklist Summary

[ ] racket -e '(require "...")'                        — loads without error
[ ] racket -e '(require "...") (display (list ...))'   — all accessors reachable
[ ] provide list matches all public defines/accessors  — no omissions
[ ] raco make path/to/new-file.rkt                     — byte-compiles cleanly
