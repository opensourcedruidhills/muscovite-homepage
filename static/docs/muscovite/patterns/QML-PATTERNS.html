QML Generator Patterns

Purpose: Canonical pattern catalog for AI agents working on QML/Qt Quick generators. Companion to: ADR 20260227 — QML Idiomatic Output Quality Sibling to: POSTGRES-PATTERNS.md (PostgreSQL patterns), RACKET-PATTERNS.md (Racket coding constraints) Qt 6 References: Best Practices | Coding Conventions | Performance

Rating System

Rating Meaning
CANONICAL The only acceptable pattern for new and modified code
ACCEPTABLE Tolerated in existing code; convert when touching the file
ANTI-PATTERN Must never appear in new code; fix immediately when found

1. Internationalization (qsTr)

CANONICAL: ALL user-facing strings in generated QML MUST be wrapped in qsTr().

// ✅ CANONICAL
Label { text: qsTr("Family Name *:") }
Dialog { title: viewModel.id.length > 0 ? qsTr("Edit Patient") : qsTr("New Patient") }
TextField { placeholderText: qsTr("Enter email") }
ToolTip.text: qsTr("Enter date as YYYY-MM-DD")

// ❌ ANTI-PATTERN: unwrapped strings
Label { text: "Family Name *:" }
Dialog { title: "Edit Patient" }
TextField { placeholderText: "Enter email" }

Qt Reference: "Make User-Facing Strings Translatable" (Best Practices)

Rationale: Without qsTr(), the string is invisible to Qt Linguist and lupdate. Generated apps cannot be localized. This blocks internationalization for all downstream users.

Status: ✅ Implemented — all generated QML now wraps user-facing strings in qsTr(). Dialog labels, titles, placeholders, tooltips, and child section headers all comply.

Exception: qsTr() is NOT needed for:

  • Property bindings to ViewModel values (already localized upstream)
  • id strings (internal identifiers, not visible)
  • Format patterns like "yyyy-MM-dd" (locale-independent)

2. Type-Safe Required Properties

CANONICAL: All required property declarations MUST use the ViewModel's actual type.

// ✅ CANONICAL — typed property (qml-saga.cpp/.hpp L212 is the reference)
required property PatientViewModel viewModel

// ❌ ANTI-PATTERN — untyped var
required property var viewModel

Qt Reference: "Type Safety — ALWAYS use actual types (string, int, MyMenu) instead of var — var hides errors, prevents static analysis" (Best Practices)

Rationale: property var defeats QML's type system. The QML engine cannot validate property access at compile time, IDE completion stops working, and AOT compilation cannot optimize the binding. Using the actual ViewModel type enables static analysis by qmllint and the QML compiler.

Status: ✅ Implemented — all generators now emit typed required property {Type}ViewModel viewModel (e.g., qml-dialog.cpp/.hpp L165). No property var remains in QML generators.

3. Versionless Qt 6 Imports

CANONICAL: All QML import statements MUST use versionless form.

// ✅ CANONICAL (Qt 6)
import QtQuick
import QtQuick.Controls
import QtQuick.Layouts

// ❌ ANTI-PATTERN (Qt 5 style)
import QtQuick 2.15
import QtQuick.Controls 2.15
import QtQuick.Layouts 1.15

Qt Reference: Qt 6 migration guide; qt_add_qml_module documentation; AOT compilation requirements.

Rationale: Versioned imports lock modules to Qt 5 semantics. Qt 6's qt_add_qml_module and QML AOT compiler work with versionless imports. Versioned imports prevent access to Qt 6-only features and types.

Status: ✅ Implemented — common.cpp/.hpp (L208-210) provides centralized versionless import QtQuick / import QtQuick.Controls / import QtQuick.Layouts block used by all QML generators.

4. QAbstractListModel for ListView Data

CANONICAL: ViewModel itemModel property MUST expose an EntityListModel* (QAbstractListModel subclass), not QVariantList.

Status: Implemented (FTR-170, v1.44.0)

// ✅ CANONICAL (C++ ViewModel header — generated by qml-viewmodel-header.cpp/.hpp)
#include <muscovite/qt/models/entity_list_model.hpp>

Q_PROPERTY(muscovite::qt::models::EntityListModel* itemModel READ itemModel CONSTANT)
muscovite::qt::models::EntityListModel* itemModel_{nullptr};

// Constructor initializes: itemModel_ = new EntityListModel(this);
// Load replaces items:     itemModel_->replaceItems(items);
// ✅ CANONICAL (QML ListView — generated by qml-list.cpp/.hpp)
ListView {
    model: viewModel.itemModel    // EntityListModel* (CONSTANT, no signal needed)
    delegate: ItemDelegate {
        required property int index
        required property string id
        required property string display
        Label { text: model.title ?? "" }      // Direct role access
        Button { onClicked: viewModel.load(model.id) }
    }
}
// ❌ ANTI-PATTERN
Q_PROPERTY(QVariantList items READ items NOTIFY itemsChanged)
QVariantList items_;  // Forces full replacement, no animations, no role binding
// ❌ ANTI-PATTERN
model: viewModel.items                        // QVariantList → copies all items to JS
Label { text: model.modelData.title || "" }   // QVariantMap index access, slow

Qt Reference: "Models/Views — use proper model for incremental updates, animations, and delegate recycling" (Performance)

Rationale: QVariantList forces complete model replacement on every change. This destroys ListView animations, breaks delegate recycling, converts every item to QVariant → QVariantMap → JavaScript object (expensive), and prevents role-based data access.

Framework class: muscovite::qt::models::EntityListModel (framework/qt/include/muscovite/qt/models/entity_list_model.hpp)

  • Extends QAbstractListModel with auto-derived roles from QVariantMap keys
  • replaceItems() / appendItem() / removeItem() / clear()
  • Qt::DisplayRole maps to "display" key (or first key as fallback)
  • FK ComboBox models also use EntityListModel* (property: {rel}ItemModel)
  • Child ViewModels use EntityListModel* for child entity lists

Generator files (all updated for FTR-170):

  • qml-viewmodel-header.cpp/.hpp — EntityListModel* property + include + member
  • qml-viewmodel.cpp/.hpp — constructor initializes EntityListModel*
  • qml-viewmodel-grpc.cpp/.hpp — replaceItems() in load/FK-load methods
  • qml-list.cpp/.hpp — viewModel.itemModel, model.x role access
  • qml-dialog.cpp/.hpp — FK ComboBox model uses {rel}ItemModel
  • qml-dialog-children.cpp/.hpp — child Repeater uses model.x role access
  • qml-child-viewmodel.cpp/.hpp — child ViewModel EntityListModel* property

Test enforcement: Rule 7 in test-qml-patterns.cpp/.hpp (4 test cases)

5. Loader for Dynamic Dialog Loading

CANONICAL: Dynamic dialog instantiation MUST use Loader, not Qt.createComponent().

// ✅ CANONICAL — Loader with asynchronous loading
Loader {
    id: dialogLoader
    active: false
    asynchronous: true
    source: "../dialogs/patient_dialog.qml"
    onLoaded: item.open()
}

Button {
    onClicked: {
        viewModel.createNew()
        dialogLoader.active = true
    }
}

// ❌ ANTI-PATTERN (qml-list.cpp/.hpp L83-92)
function openEditDialog() {
    var component = Qt.createComponent("../dialogs/entity_dialog.qml")
    if (component.status === Component.Ready) {
        var dialog = component.createObject(root, { viewModel: viewModel })
        if (dialog) dialog.open()
    }
}

Qt Reference: "Lazy Initialization — use Loader with asynchronous: true to defer creation costs" (Performance)

Rationale: Qt.createComponent() is synchronous — it blocks the UI thread while parsing and compiling the QML file. Loader with asynchronous: true defers compilation to a background thread.

Status: ✅ Implemented (FTR-171) — qml-dialog-children.cpp/.hpp uses Loader with asynchronous: true for all child dialog instantiation. No Qt.createComponent() remains in QML generators.

6. SpinBox: onValueModified vs onValueChanged

CANONICAL: SpinBox user interaction MUST use onValueModified.

// ✅ CANONICAL — fires only on user interaction
SpinBox {
    value: viewModel.age
    onValueModified: { viewModel.setAge(value); dialog.dirty = true }
}

// ❌ ANTI-PATTERN — fires on programmatic changes too
SpinBox {
    value: viewModel.age
    onValueChanged: { viewModel.setAge(value); dialog.dirty = true }
}

Qt Reference: "Prefer explicit interaction signals over value-changed signals" (Best Practices)

Rationale: onValueChanged fires when the ViewModel sets the SpinBox value programmatically (e.g., loading an entity). This causes a feedback loop: load → set value → onValueChanged fires → dirty flag set.

Remaining violations:

  • qml-dialog.cpp/.hpp — ✅ Fixed: now uses onValueModified (L66)
  • view.cpp/.hpp — L113: still uses onValueChanged (legacy Widgets path)

7. ListView Clipping

CANONICAL: ListView MUST NOT have clip: true.

// ✅ CANONICAL
ListView {
    Layout.fillWidth: true
    Layout.fillHeight: true
    model: viewModel.items
}

// ❌ ANTI-PATTERN (qml-list.cpp/.hpp L136)
ListView {
    clip: true
}

Qt Reference: "Clipping — disabled by default, increases renderer complexity. Especially detrimental in delegates." (Performance)

Rationale: ListView already handles delegate visibility through its internal viewport management. Adding clip: true forces the GPU to set up a scissor rectangle.

Note: clip: true on ScrollView is acceptable — it is designed to clip viewport content.

Remaining violations (6 occurrences across 5 files):

  • qml-saga.cpp/.hpp — L246, L277 (two ListView instances)
  • qml-search.cpp/.hpp — L328
  • view.cpp/.hpp — L191
  • qml-app.cpp/.hpp — L171
  • qml-dialog.cpp/.hpp — L215

8. Theme Constants (No Magic Numbers)

CANONICAL: Layout dimensions MUST reference theme constants, not hardcoded pixel values.

// ✅ CANONICAL — theme property
Label { text: qsTr("Name:"); Layout.preferredWidth: Theme.formLabelWidth }

// ❌ ANTI-PATTERN (qml-dialog.cpp/.hpp L81, L85, L90, L95, L112, L134)
Label { text: "Name:"; Layout.preferredWidth: 140 }

Qt Reference: "Scalable UIs — avoid explicit pixel sizes; use anchors, layouts, and scalable measurements" (Best Practices)

Status: ✅ Implemented — generators use MuscoviteTheme.formLabelWidth and other theme constants (sidebarColor, textOnPrimaryColor, borderRadius, errorBackgroundColor, etc.). Theme file generated by qml-app.cpp/.hpp.

9. QML Property Ordering

CANONICAL: Properties MUST follow Qt convention order.

// ✅ CANONICAL order
Dialog {
    id: patientDialog                                    // 1. id

    required property PatientViewModel viewModel          // 2. required properties
    property string parentEntityId: ""                   // 3. property declarations
    property bool dirty: false

    signal saved()                                        // 4. signal declarations

    function validate() { ... }                          // 5. JavaScript functions

    title: qsTr("Edit Patient")                          // 6. object properties
    width: 560
    modal: true

    contentItem: ColumnLayout { ... }                    // 7. child objects
}

Qt Reference: "id → property declarations → signal declarations → JavaScript functions → object properties → child objects" (Coding Conventions)

Status: ✅ Implemented — qml-dialog.cpp/.hpp emits properties in correct order: required property (L165) → property string (L166) → property bool (L168) → title: (L170) → width: (L171) → modal: (L173) → child objects.

10. Grouped Anchor Notation

CANONICAL: When setting 3+ properties from the same group, use block notation.

// ✅ CANONICAL — group notation for 3+ properties
anchors {
    fill: parent
    leftMargin: 8
    rightMargin: 8
}

// ✅ ACCEPTABLE — dot notation for 1-2 properties
anchors.fill: parent

// ❌ ANTI-PATTERN — dot notation for 3+ properties
anchors.fill: parent
anchors.leftMargin: 8
anchors.rightMargin: 8

Qt Reference: "Use group notation when setting several properties from one group" (Coding Conventions)

Status: ✅ Implemented — delegate layouts use grouped notation where 3+ properties apply.

11. Type Dispatch Consolidation

CANONICAL: All QML type dispatch MUST go through common.cpp/.hpp functions.

;; ✅ CANONICAL — centralized in common.cpp/.hpp
(define (dsl-type->qml-read-expr type-sym field-name)
  (case type-sym
    [(UUID Email Phone) (format "~a().value()" field-name)]
    [(Date) (format "muscovite::grpc::to_date(~a())" field-name)]
    ...))

;; ❌ ANTI-PATTERN — duplicated case block in each generator file
;; qml-viewmodel-mapping.cpp/.hpp:  (case type-sym [(UUID) ...] [(Date) ...] ...)
;; qml-viewmodel-grpc.cpp/.hpp:     (case type-sym [(UUID) ...] [(Date) ...] ...)
;; qml-child-viewmodel.cpp/.hpp:    (case type-sym [(UUID) ...] [(Date) ...] ...)
;; display-helper-codegen.cpp/.hpp:  (case type-sym [(UUID) ...] [(Date) ...] ...)

Rationale: Four separate type-dispatch blocks drift independently. Adding a new type requires updating all four.

Status: ✅ Implemented (FTR-174) — type dispatch centralized in common.cpp/.hpp. Individual generator files delegate to shared functions.

12. Module Import Pattern (except-in Resolution)

CANONICAL: Qt generators MUST import through common.cpp/.hpp, not directly from ddd-types.cpp/.hpp with exclusions.

;; ✅ CANONICAL — import curated exports from common.cpp/.hpp
(require "common.cpp/.hpp")

;; ❌ ANTI-PATTERN — fragile exclusion (21 files)
(require (except-in "../../model/ddd-types.cpp/.hpp" dsl-type->cpp-type dsl-type->proto-type))

Rationale: 21 files exclude the same 2 identifiers from ddd-types.cpp/.hpp. If ddd-types.cpp/.hpp adds another conflicting export, all 21 files break simultaneously.

Affected files: 21 generator files (see ADR for complete list)

13. Template DSL Compliance

CANONICAL: All QML generator functions MUST use define-template per ADR 20260226.

See doc/RACKET-PATTERNS.md Section 9 and doc/TEMPLATE-DSL.md for authoring patterns.

Current status: 3 out of 16 QML generator files use define-template (qml-validators.cpp/.hpp, plugins/qt-circuit-breaker.cpp/.hpp, plugins/qt-remote-objects.cpp/.hpp). The remaining 13 files still use format/string-append chains.

Note: Template DSL migration is tracked per ADR 20260226. QML generators are in Wave 2b (higher-risk migration). The patterns in this document (1–12) can be fixed independently of the template DSL migration.
