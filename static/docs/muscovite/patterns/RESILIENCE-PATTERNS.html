Resilience Sandwich Pattern Catalog

Resilience Sandwich Pattern Catalog

Comprehensive reference for Muscovite's deadline propagation, retry, and error classification architecture.

ADRs: 20260228-grpc-deadline-to-statement-timeout, 20260228-grpc-resilience-sandwich-full-architecture

Features: FTR-225, FTR-230, FTR-231, FTR-232, FTR-233, FTR-235

1. Concept: The Resilience Sandwich

The Resilience Sandwich is a layered timeout and retry architecture that propagates client deadlines through every system boundary: gRPC → business logic → PostgreSQL.

flowchart TD
    L1["Client Deadline · gRPC metadata\n← outermost boundary"]
    L2["Server Deadline · per-aggregate clamp\n← GRPC_DEADLINE"]
    L3["Transactor Retry Budget\n← RetryPolicy.deadline"]
    L4["PostgreSQL statement_timeout\n← innermost boundary"]

    L1 --> L2 --> L3 --> L4

Invariant (must always hold):

statement_timeout < remaining - buffer < server_deadline ≤ client_deadline

Each layer's timeout is strictly shorter than its parent's, ensuring:

  1. PostgreSQL times out before the server does → server can return a meaningful error
  2. Server times out before the client does → client can retry or fail gracefully
  3. Retry budget prevents wasting attempts near deadline expiry

2. Timeout Hierarchy

2.1 Client Deadline (Outer Layer)

Set by the gRPC client as a metadata deadline. This is the absolute upper bound for the entire operation. The server reads it via context->deadline().

2.2 Server Deadline — clamp_deadline() (Middle Layer)

Per-aggregate SLA enforcement. The DSL GRPC_DEADLINE clause sets a maximum duration for an aggregate's operations:

AGGREGATE PatientRecord @uuid "..." ROOT PatientRecord
    GRPC_DEADLINE "5s"
    DOC "Latency-sensitive patient lookup";

Generated code calls clamp_deadline() to enforce:

// From framework/server/include/muscovite/server/deadline.hpp
auto deadline = muscovite::server::clamp_deadline(
    context, std::chrono::milliseconds{5000});

clamp_deadline() returns min(now + max_duration, client_deadline) — the server never waits longer than the client allows.

2.3 Retry Budget (Transactor Layer)

The transactor's RetryPolicy struct includes deadline-awareness:

Field Default Purpose
max_retries 3 Maximum retry attempts
initial_delay 100ms First backoff delay
backoff_multiplier 2.0 Exponential backoff factor
max_delay 5000ms Backoff ceiling
deadline max() Retry budget boundary (FTR-235)
min_remaining_for_retry 200ms Minimum time to allow another attempt
retry_on_serialization_failure true Retry PostgreSQL 40001
retry_on_deadlock true Retry PostgreSQL 40P01
retry_on_connection_error false Retry connection failures

Budget check — before each retry:

bool has_retry_budget(const RetryPolicy& policy) {
    if (policy.deadline == time_point::max()) return true;  // no deadline → unlimited
    auto remaining = policy.deadline - now();
    return remaining > policy.min_remaining_for_retry;
}

Backoff capping — when a deadline is set, the delay is capped at remaining / 2 to avoid sleeping past the deadline.

2.4 PostgreSQL statement_timeout (Inner Layer)

The deadline-aware execute() injects SET LOCAL statement_timeout into each transaction:

auto result = muscovite::dba::execute(pool_, deadline, [&](pqxx::work& tx) {
    // statement_timeout = min(remaining - buffer, max_statement_timeout)
    // ... business logic ...
}, max_statement_timeout, deadline_buffer);

If the query exceeds statement_timeout, PostgreSQL cancels it with SQLSTATE 57014, which the transactor maps to DbError::DeadlineExceeded.

3. Error Classification

3.1 gRPC Status Code Classification

From framework/grpc/include/muscovite/grpc/retry_classification.hpp:

gRPC Status Classification Retry? Notes
OK — No Success
CANCELLED — No Client-initiated
UNKNOWN — No Ambiguous
INVALID_ARGUMENT Permanent No Fix the request
DEADLINE_EXCEEDED Retryable ¹ Conditional Only if budget remains
NOT_FOUND Permanent No Resource does not exist
ALREADY_EXISTS Permanent No Duplicate key
PERMISSION_DENIED Permanent No Authorization failure
RESOURCE_EXHAUSTED Retryable Yes Backpressure / rate limit
FAILED_PRECONDITION Permanent No Invalid state
ABORTED Retryable Yes Serialization conflict
UNIMPLEMENTED Permanent No Wrong server version
INTERNAL Neither No Ambiguous — log and alert
UNAVAILABLE Retryable Yes Server temporarily down
UNAUTHENTICATED Permanent No Bad/expired credentials
DATA_LOSS — No Unrecoverable

¹ DEADLINE_EXCEEDED is retryable only when has_retry_budget() returns true and the failure was from a downstream timeout, not the client's own deadline.

4. DSL Configuration Reference

4.1 SERVER Block (Project DSL — .mus)

Configures global gRPC deadline defaults:

SERVER {
    GRPC_DEFAULT_DEADLINE "5s";
    GRPC_MAX_DEADLINE "120s";
}
Directive Default Purpose
GRPC_DEFAULT_DEADLINE 5000ms Default deadline for aggregates without GRPC_DEADLINE
GRPC_MAX_DEADLINE 30000ms Upper bound on any deadline (safety clamp)

Generated into the server configuration INI file under [deadlines].

4.2 GRPC_DEADLINE on AGGREGATE (Domain DSL — .ddd)

Overrides the server default for specific aggregates:

AGGREGATE DataImport @uuid "a1b2c3d4-..."
    ROOT ImportJob
    GRPC_DEADLINE "60s"
    DOC "Long-running batch import — needs more time than default";

Duration format: "Ns" (seconds) or "Nms" (milliseconds). Examples: "5s", "100ms", "60s".

Effect: Generated service methods call clamp_deadline(context, milliseconds{N}) instead of context->deadline().

4.3 Retry Policy (C++ — compile-time configuration)

Planned: FTR-413 (v1.66.0) will add DATABASE { RETRY { ... } } and TIMEOUT { ... } blocks to the project DSL, allowing retry/timeout policies to be declared in .mus files and generated into C++ as constexpr RetryPolicy defaults. Until then, policies are configured in C++ code.

The RetryPolicy struct is configured in C++ application code:

muscovite::dba::RetryPolicy policy{
    .max_retries = 5,
    .initial_delay = std::chrono::milliseconds{50},
    .backoff_multiplier = 2.0,
    .max_delay = std::chrono::milliseconds{2000},
    .retry_on_serialization_failure = true,
    .retry_on_deadlock = true,
    .retry_on_connection_error = false,
    // .deadline and .min_remaining_for_retry are set automatically
    // by the deadline-aware execute() overload
};

The deadline field is populated automatically when using the deadline-aware execute() overload — it threads the gRPC deadline into the retry budget mechanism.

5. Anti-Patterns

❌ Ignoring the Client Deadline

// WRONG: Uses a fixed 30s deadline regardless of what the client set
auto deadline = system_clock::now() + 30s;
auto result = muscovite::dba::execute(pool_, deadline, fn, ...);

Why it's wrong: The client may have set a 2s deadline. Using a fixed 30s deadline wastes server resources on a request the client has already abandoned.

Fix: Always derive the server deadline from the client's:

auto deadline = muscovite::server::clamp_deadline(context, 30s);

❌ Retrying DEADLINE_EXCEEDED Without Budget Check

// WRONG: Retries DEADLINE_EXCEEDED blindly
if (is_retryable(status.error_code())) {
    retry();  // Will retry DEADLINE_EXCEEDED even with 0ms remaining
}

Why it's wrong: The retry will immediately fail again (or worse, succeed after the client has timed out and disconnected).

Fix: Check remaining budget before retry:

if (is_retryable(status.error_code())) {
    if (is_deadline_exceeded(status.error_code()) && !has_retry_budget(policy)) {
        return status;  // No point retrying
    }
    retry();
}

❌ Setting statement_timeout Equal to gRPC Deadline

// WRONG: No buffer for response marshaling
auto timeout = remaining;
tx.exec0(format("SET LOCAL statement_timeout = '{}'", timeout.count()));

Why it's wrong: The query may finish just in time, but the server needs time to marshal the response, commit the transaction, and send the gRPC reply. Without a buffer, the response arrives after the client has timed out.

Fix: Subtract a buffer:

auto timeout = min(remaining - deadline_buffer, max_statement_timeout);

❌ Sleeping Past the Deadline

// WRONG: Exponential backoff may exceed remaining time
auto delay = initial_delay * pow(backoff_multiplier, attempt);
sleep(delay);  // May sleep for 8s when only 500ms remains

Why it's wrong: The retry attempt after the sleep will fail immediately because the deadline has expired.

Fix: Cap backoff at remaining / 2:

auto delay = min(computed_delay, remaining / 2);

❌ Using Runtime Queries for Retry Configuration

// WRONG: Runtime query for framework constants
auto max_retries = db.query("SELECT value FROM config WHERE key = 'max_retries'");

Why it's wrong: Per ADR 20260212, framework constants use compile-time values (UUIDv5 + constexpr). Runtime queries add latency and failure modes to the retry path itself.

Fix: Use RetryPolicy struct with compile-time defaults.

6. Troubleshooting

"gRPC deadline already expired"

The server received a request whose deadline was already in the past. Causes:

  • Client set a very short deadline
  • Network latency consumed the entire budget in transit
  • Clock skew between client and server

Action: Check client-side deadline configuration. Consider increasing the deadline or using GRPC_DEADLINE on the aggregate to enforce a minimum.

"Insufficient time remaining for database operation"

Deadline minus buffer is ≤ 0. The request arrived with less remaining time than the deadline buffer requires.

Action: Reduce deadline_buffer_ms in the server config, or increase the client deadline.

"PostgreSQL statement_timeout exceeded (propagated from gRPC deadline)"

The database query ran longer than the allocated time slice. This is the innermost timeout firing.

Action: Optimize the query, add appropriate indexes, or increase the aggregate's GRPC_DEADLINE.

"Retry budget exhausted"

The transactor detected insufficient remaining time (< min_remaining_for_retry) and skipped a retry.

Action: This is correct behavior — the system prevented a wasteful retry. If the operation legitimately needs more retries, increase the client deadline.

7. Generated Code Reference

Service Method (with GRPC_DEADLINE)

grpc::Status PatientServiceImpl::CreateImpl(
    grpc::ServerContext* context, ...) {
    auto deadline = muscovite::server::clamp_deadline(
        context, std::chrono::milliseconds{5000});
    auto result = muscovite::dba::execute(
        pool_, deadline,
        [&](pqxx::work& tx) { /* ... */ },
        max_statement_timeout_, deadline_buffer_);
    // ...
}

Service Method (default — no GRPC_DEADLINE)

grpc::Status LabResultServiceImpl::CreateImpl(
    grpc::ServerContext* context, ...) {
    auto deadline = context->deadline();
    auto result = muscovite::dba::execute(
        pool_, deadline,
        [&](pqxx::work& tx) { /* ... */ },
        max_statement_timeout_, deadline_buffer_);
    // ...
}

The only difference is the deadline extraction line. Everything downstream is identical.
