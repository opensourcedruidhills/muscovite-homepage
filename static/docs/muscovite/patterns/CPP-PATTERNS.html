C++ Pattern Catalog for Muscovite

C++ Pattern Catalog for Muscovite

Purpose: Constraint document for AI agents and human developers. Every C++ pattern in Muscovite framework code (Layer 1) and generated output (Layer 2) must use the canonical form. This document is the single source of truth for "how to write C++ in this project."

Scope: All .hpp/.cpp files in framework/ (Layer 1) and all C++ generator output in dslgen/generators/cpp/ and dslgen/generators/grpc/ (Layer 0 → Layer 2). Qt-specific patterns are in QT-WIDGETS-PATTERNS.md and QML-PATTERNS.md.

Sibling to: RACKET-PATTERNS.md (Racket coding constraints), POSTGRES-PATTERNS.md (PostgreSQL patterns), QML-PATTERNS.md (QML output patterns), QT-WIDGETS-PATTERNS.md (Qt Widgets output patterns)

Companion ADR: TBD — Modern C++23 Style Guide (planned Phase 1 of C++ modernization)

Standard: C++23 (-std=c++23). Compiler: GCC 14+ / Clang 18+.

Rating System

Rating Meaning
CANONICAL The only acceptable pattern for new and modified code
ACCEPTABLE Tolerated in specific circumstances (documented)
ANTI-PATTERN Forbidden in new code; convert when touching the file

When an AI agent generates or modifies C++ code for Muscovite, it MUST use only CANONICAL or ACCEPTABLE patterns. When an AI agent modifies an existing file, it MUST convert any anti-patterns in the modified function to canonical form.

1. Error Handling (std::expected)

CANONICAL: Use std::expected<T, E> for fallible operations. Muscovite already grades A on this — maintain it.

// ✅ CANONICAL — std::expected for fallible operations
[[nodiscard]] auto find_by_id(const boost::uuids::uuid& id)
    -> std::expected<Patient, muscovite::dba::Error>;

// ✅ CANONICAL — monadic chaining (C++23)
auto result = find_by_id(id)
    .and_then([](auto&& p) { return validate(p); })
    .transform([](auto&& p) { return to_response(p); });

// ✅ ACCEPTABLE — exceptions for truly exceptional conditions (connection loss, OOM)
throw muscovite::dba::ConnectionError("PostgreSQL unreachable");
// ❌ ANTI-PATTERN — error codes or output parameters
int find_by_id(const uuid& id, Patient* out);

// ❌ ANTI-PATTERN — exceptions for expected failures (not-found, validation)
throw EntityNotFoundException(id);  // Use std::expected<T, NotFound> instead

// ❌ ANTI-PATTERN — std::optional for errors (loses error information)
std::optional<Patient> find_by_id(const uuid& id);  // Why did it fail?

Generator files: dba/unified.rkt, grpc/cpp-service-v2.rkt

2. Attribute Annotations

2a. [[nodiscard]]

CANONICAL: ALL functions returning values MUST be [[nodiscard]]. Muscovite grades A — maintain it.

// ✅ CANONICAL
[[nodiscard]] auto find_all() -> std::vector<Patient>;
[[nodiscard]] auto to_string() const -> std::string;
[[nodiscard]] constexpr auto is_valid() const noexcept -> bool;
// ❌ ANTI-PATTERN — missing [[nodiscard]] on value-returning functions
auto find_all() -> std::vector<Patient>;
std::string to_string() const;

2b. noexcept

CANONICAL: Mark noexcept on: move constructors, move assignment, destructors, trivial getters, converters, swap, _opt variants that return std::optional.

// ✅ CANONICAL — noexcept on moves
Patient(Patient&&) noexcept = default;
Patient& operator=(Patient&&) noexcept = default;

// ✅ CANONICAL — noexcept on trivial getting/conversion
[[nodiscard]] constexpr auto id() const noexcept -> const boost::uuids::uuid&;
[[nodiscard]] auto to_uuid_string() const noexcept -> std::string;

// ✅ CANONICAL — noexcept on _opt variants
[[nodiscard]] auto find_by_id_opt(const uuid& id) noexcept
    -> std::optional<Patient>;
// ❌ ANTI-PATTERN — missing noexcept on move operations
Patient(Patient&&) = default;  // Add noexcept

// ❌ ANTI-PATTERN — missing noexcept on trivial converters
auto to_string() const -> std::string;  // If it can't throw, mark noexcept

Audit: Current grade D — ~30 move operations and ~50 converters missing noexcept. Generator files: dba/unified.rkt (entity structs), grpc/cpp-service-v2.rkt

3. Ranges and Views (std::ranges)

CANONICAL: Use std::ranges algorithms and std::views pipelines instead of raw loops. This is the single highest-impact modernization item (~30 occurrences).

// ✅ CANONICAL — ranges::to for container construction (C++23)
auto names = patients
    | std::views::transform([](const auto& p) { return p.family_name(); })
    | std::ranges::to<std::vector>();

// ✅ CANONICAL — ranges algorithm for simple transforms
std::ranges::sort(patients, {}, &Patient::family_name);
std::ranges::for_each(events, [&](const auto& e) { emit(e); });

// ✅ CANONICAL — views pipeline for filtering + transforming
auto active_names = patients
    | std::views::filter(&Patient::is_active)
    | std::views::transform(&Patient::display_name)
    | std::ranges::to<std::vector>();

// ✅ CANONICAL — views::join for flattening
auto all_fields = contexts
    | std::views::transform(&Context::entities)
    | std::views::join
    | std::views::transform(&Entity::fields)
    | std::views::join
    | std::ranges::to<std::vector>();
// ❌ ANTI-PATTERN — reserve + push_back loop (P0 item, ~30 occurrences)
std::vector<std::string> names;
names.reserve(patients.size());
for (const auto& p : patients) {
    names.push_back(p.family_name());
}

// ❌ ANTI-PATTERN — std::transform with back_inserter
std::vector<std::string> names;
std::transform(patients.begin(), patients.end(),
    std::back_inserter(names),
    [](const auto& p) { return p.family_name(); });

// ❌ ANTI-PATTERN — manual filter loop
std::vector<Patient> active;
for (const auto& p : patients) {
    if (p.is_active()) active.push_back(p);
}

Audit: Current grade F — zero std::ranges usage in entire codebase. ~30 reserve+push_back loops. Generator files: dba/unified.rkt (query result mapping), grpc/cpp-service-v2.rkt (response building)

4. Structured Bindings

CANONICAL: Use structured bindings for multi-value returns, pair/tuple, and map iteration.

// ✅ CANONICAL — structured bindings for row fields
for (const auto& [id, name, active] : rows) {
    result.emplace_back(Patient{id, name, active});
}

// ✅ CANONICAL — map iteration
for (const auto& [key, value] : metadata) {
    process(key, value);
}

// ✅ CANONICAL — std::tie replacement
auto [iter, inserted] = cache.emplace(key, value);
// ❌ ANTI-PATTERN — indexed tuple access
for (const auto& row : rows) {
    result.emplace_back(Patient{
        std::get<0>(row), std::get<1>(row), std::get<2>(row)
    });
}

// ❌ ANTI-PATTERN — .first / .second
for (const auto& entry : metadata) {
    process(entry.first, entry.second);
}

Audit: Current grade F — zero structured bindings in generated or framework code. Generator files: dba/unified.rkt (row mapping in repositories)

5. Concepts and Constraints

CANONICAL: Use C++20 concepts to constrain template parameters. Define project-specific concepts for domain patterns.

// ✅ CANONICAL — DbEntity concept for repository templates
template<typename T>
concept DbEntity = requires(T t) {
    { t.id() } -> std::convertible_to<boost::uuids::uuid>;
    { t.table_name() } -> std::convertible_to<std::string_view>;
};

// ✅ CANONICAL — constrained template
template<DbEntity T>
[[nodiscard]] auto find_by_id(const boost::uuids::uuid& id)
    -> std::expected<T, Error>;

// ✅ CANONICAL — constrain to_param<T>
template<typename T>
    requires std::convertible_to<T, std::string> || std::integral<T>
auto to_param(const T& value) -> pqxx::params;
// ❌ ANTI-PATTERN — unconstrained template
template<typename T>
auto find_by_id(const boost::uuids::uuid& id) -> std::expected<T, Error>;

// ❌ ANTI-PATTERN — SFINAE instead of concepts
template<typename T, typename = std::enable_if_t<...>>
auto process(const T& value);

Audit: Current grade F — zero concepts defined. to_param<T> is unconstrained. Generator files: dba/unified.rkt, framework dba/repository.hpp

6. Designated Initializers

CANONICAL: Use designated initializers for struct construction. Already graded A — maintain it.

// ✅ CANONICAL — designated initializers for entity construction
auto patient = Patient{
    .id = uuid,
    .mrn = row["mrn"].as<std::string>(),
    .family_name = row["family_name"].as<std::string>(),
    .active = row["active"].as<bool>()
};

// ✅ CANONICAL — designated initializers for request structs
auto req = CreatePatientRequest{
    .mrn = "MRN-001",
    .family_name = "Smith",
    .given_name = "John"
};
// ❌ ANTI-PATTERN — positional construction (fragile, order-dependent)
auto patient = Patient{uuid, "MRN-001", "Smith", true};

Generator files: dba/unified.rkt (entity struct construction from DB rows)

7. String Handling

7a. std::string_view for Non-Owning References

CANONICAL: Use std::string_view for function parameters that don't need ownership.

// ✅ CANONICAL — string_view for non-owning read access
[[nodiscard]] auto find_by_code(std::string_view code)
    -> std::expected<LookupEntry, Error>;

void log_event(std::string_view event_type, std::string_view payload);

// ✅ CANONICAL — static constexpr for SQL query strings
static constexpr auto kInsertSql = pqxx::zview{
    "INSERT INTO patient.patient (id, mrn, family_name) VALUES ($1, $2, $3)"
};
// ❌ ANTI-PATTERN — const std::string& for non-owning read access
auto find_by_code(const std::string& code) -> std::expected<LookupEntry, Error>;

// ❌ ANTI-PATTERN — duplicated or inline SQL strings
auto result = txn.exec_params(
    "INSERT INTO patient.patient (id, mrn) VALUES ($1, $2)", id, mrn);

Audit: Current grade D for string_view, B- for SQL extraction. Generator files: dba/unified.rkt (repository methods), grpc/cpp-service-v2.rkt

7b. std::format over fmt::format

CANONICAL: Use std::format (C++20) for string formatting. Already graded A- — maintain.

// ✅ CANONICAL
auto msg = std::format("Patient {} not found", boost::uuids::to_string(id));

// ❌ ANTI-PATTERN — fmt::format (unnecessary dependency when std::format available)
auto msg = fmt::format("Patient {} not found", boost::uuids::to_string(id));

// ❌ ANTI-PATTERN — string concatenation
auto msg = "Patient " + boost::uuids::to_string(id) + " not found";

8. Const Correctness

CANONICAL: All query methods, getters, and non-mutating functions MUST be const.

// ✅ CANONICAL — const on query service methods
class PatientQueryService {
public:
    [[nodiscard]] auto find_by_id(const boost::uuids::uuid& id) const
        -> std::expected<Patient, Error>;
    [[nodiscard]] auto find_all(const PageRequest& page) const
        -> std::vector<Patient>;
};

// ✅ CANONICAL — const on entity getters
struct Patient {
    [[nodiscard]] auto id() const noexcept -> const boost::uuids::uuid&;
    [[nodiscard]] auto display_name() const -> std::string;
};
// ❌ ANTI-PATTERN — missing const on query methods
auto find_by_id(const boost::uuids::uuid& id) -> std::expected<Patient, Error>;

Audit: Current grade B — query service generators sometimes omit const. Generator files: dba/unified.rkt, grpc/cpp-service-v2.rkt

9. Forward Declarations

CANONICAL: Use forward declarations in headers to minimize includes. Include the full header only in .cpp files.

// ✅ CANONICAL — forward declare in header
namespace muscovite::dba {
class ConnectionPool;
}  // namespace muscovite::dba

class PatientCommandService {
public:
    explicit PatientCommandService(muscovite::dba::ConnectionPool& pool);
    // ...
private:
    muscovite::dba::ConnectionPool& pool_;
};

// In .cpp: #include <muscovite/dba/connection_pool.hpp>
// ❌ ANTI-PATTERN — full include in header when forward declaration suffices
#include <muscovite/dba/connection_pool.hpp>  // Only need reference/pointer

Audit: Current grade D+ in Qt framework (barely used), B in DBA framework. Generator files: dba/unified.rkt (header generation), grpc/cpp-service-v2.rkt

10. Header/Source Separation

CANONICAL: Headers contain declarations only. Implementation goes in .cpp files. Exception: templates, constexpr, and trivial one-line getters.

// ✅ CANONICAL — header: declaration only
// patient_command_repository.hpp
class PatientCommandRepository {
public:
    [[nodiscard]] auto insert(pqxx::work& txn, const CreatePatientRequest& req)
        -> std::expected<Patient, Error>;
    // ...
};

// patient_command_repository.cpp
auto PatientCommandRepository::insert(pqxx::work& txn, const CreatePatientRequest& req)
    -> std::expected<Patient, Error>
{
    // Full implementation here
}
// ✅ ACCEPTABLE — inline in header for trivial getters and constexpr
[[nodiscard]] constexpr auto id() const noexcept -> const uuid& { return id_; }

// ✅ ACCEPTABLE — inline in header for templates
template<DbEntity T>
[[nodiscard]] auto find_by_id(const uuid& id) -> std::expected<T, Error> { ... }
// ❌ ANTI-PATTERN — substantial implementation in header
// entity_validator.hpp — 526 lines of implementation inline (actual framework file)
class EntityValidator {
    void validate(const QVariantMap& data) {
        // ... 200 lines of validation logic ...
    }
};

Audit: Framework grades D+ — ~3,200 lines of implementation in headers. Worst offenders: entity_validator.hpp (526 lines), entity_dialog_state_machine.hpp (469 lines), validation_result.hpp (337 lines), field_metadata.hpp (378 lines). Generator files: dba/unified.rkt (generates proper split), framework qt/ needs manual retrofit.

11. Namespace Structure

CANONICAL: Use nested namespaces (C++17). Follow project conventions strictly.

// ✅ CANONICAL — nested namespace declaration
namespace muscovite::dba::patient {

struct Patient final { /* ... */ };

}  // namespace muscovite::dba::patient

// ✅ CANONICAL — anonymous namespace for file-local helpers in .cpp
namespace {
constexpr auto kMaxRetries = 3;
}  // namespace
// ❌ ANTI-PATTERN — separate namespace blocks
namespace muscovite {
namespace dba {
namespace patient {
// ...
}}}

Audit: Current grade A — maintain.

12. Memory and Ownership

CANONICAL: Use value semantics by default. Smart pointers only when ownership transfer is needed. No raw new/delete.

// ✅ CANONICAL — value semantics (default)
auto patient = Patient{.id = uuid, .mrn = "MRN-001"};

// ✅ CANONICAL — std::unique_ptr for exclusive ownership
auto service = std::make_unique<PatientCommandService>(pool);

// ✅ CANONICAL — std::shared_ptr only when genuinely shared
auto pool = std::make_shared<ConnectionPool>(config);
// ❌ ANTI-PATTERN — raw new/delete
auto* patient = new Patient();
// ... use ...
delete patient;

// ❌ ANTI-PATTERN — C-style memory management
auto* buf = (char*)malloc(size);
free(buf);

Generator files: All DBA generators use value semantics — maintain.

13. C Legacy Elimination

CANONICAL: Replace all C standard library functions with C++ equivalents.

// ✅ CANONICAL — C++ equivalents
#include <cmath>
constexpr auto pi = std::numbers::pi;
auto distance = std::sqrt(dx * dx + dy * dy);
auto n = std::stoi(str);
auto d = std::stod(str);

// ✅ CANONICAL — from_chars for performance-critical parsing (C++17)
double value{};
auto [ptr, ec] = std::from_chars(str.data(), str.data() + str.size(), value);
// ❌ ANTI-PATTERN — C math constants
#define _USE_MATH_DEFINES
#include <cmath>
auto circumference = 2.0 * M_PI * radius;  // Use std::numbers::pi

// ❌ ANTI-PATTERN — C string functions
auto len = strlen(str);      // Use str.size()
auto cmp = strcmp(a, b);     // Use a == b or a <=> b
sprintf(buf, "%d", value);   // Use std::format

// ❌ ANTI-PATTERN — atoi/atof (no error reporting)
auto n = atoi(str);  // Use std::stoi or std::from_chars

Audit: Current grade A for most code, but proto_type_converters.hpp (416 lines) uses C89-style function signatures and manual buffer construction — needs full rewrite. Generator files: grpc/cpp-service-v2.rkt (proto converter generation)

14. SQL Query Management

CANONICAL: Extract SQL queries to static constexpr pqxx::zview constants at class scope or file scope. Never inline SQL in method bodies.

// ✅ CANONICAL — extracted SQL constant
class PatientCommandRepository {
    static constexpr auto kInsertSql = pqxx::zview{
        "INSERT INTO patient.patient (id, mrn, family_name, active) "
        "VALUES ($1, $2, $3, $4) "
        "RETURNING id"
    };

public:
    [[nodiscard]] auto insert(pqxx::work& txn, const CreatePatientRequest& req)
        -> std::expected<Patient, Error>
    {
        auto result = txn.exec_params(kInsertSql, req.id, req.mrn, req.family_name, req.active);
        // ...
    }
};
// ❌ ANTI-PATTERN — inline SQL strings in method bodies
auto result = txn.exec_params(
    "INSERT INTO patient.patient (id, mrn, family_name, active) "
    "VALUES ($1, $2, $3, $4) RETURNING id",
    req.id, req.mrn, req.family_name, req.active);

Audit: Current grade B- — many queries still inline. Generator files: dba/unified.rkt (repository SQL generation)

15. Duplicate Code Elimination

CANONICAL: Factor out utility functions to shared locations. No copy-paste across generators.

// ✅ CANONICAL — single uuid_to_string utility
// In muscovite/dba/conversions.hpp
[[nodiscard]] inline auto uuid_to_string(const boost::uuids::uuid& id) noexcept
    -> std::string
{
    return boost::uuids::to_string(id);
}

// ✅ CANONICAL — shared proto conversion utilities
// In muscovite/grpc/type_mapping.hpp
[[nodiscard]] auto from_proto_uuid(const muscovite::common::UUID& proto)
    -> boost::uuids::uuid;
[[nodiscard]] auto to_proto_uuid(const boost::uuids::uuid& id)
    -> muscovite::common::UUID;
// ❌ ANTI-PATTERN — duplicated uuid_to_string across files
// patient_service.cpp
auto str = boost::uuids::to_string(id);
// encounter_service.cpp — same code repeated
auto str = boost::uuids::to_string(id);

Audit: uuid_to_string duplicated in multiple generated service files. Proto type converters duplicated across contexts instead of shared. Generator files: grpc/cpp-service-v2.rkt, dba/unified.rkt

16. Include Guards

CANONICAL: Use #pragma once for all headers. Muscovite already uses this consistently — maintain.

// ✅ CANONICAL
#pragma once

#include <string>
// ...

// ❌ ANTI-PATTERN — traditional include guards
#ifndef MUSCOVITE_DBA_PATIENT_HPP
#define MUSCOVITE_DBA_PATIENT_HPP
// ...
#endif

Audit: Current grade B- — consistent #pragma once but #include ordering is inconsistent.

CANONICAL include order (enforced by clang-format):

  1. Corresponding header (for .cpp files)
  2. Project headers (muscovite/...)
  3. Third-party headers (boost/..., pqxx/..., grpc++/...)
  4. Standard library headers (<string>, <vector>, ...)

17. NATS/Messaging Patterns (Framework)

CANONICAL: RAII wrappers for NATS resources. std::expected for messaging errors.

// ✅ CANONICAL — RAII wrapper for NATS connection
class NatsConnection {
public:
    explicit NatsConnection(const NatsConfig& config);
    ~NatsConnection();  // Calls natsConnection_Destroy

    NatsConnection(const NatsConnection&) = delete;
    NatsConnection& operator=(const NatsConnection&) = delete;
    NatsConnection(NatsConnection&&) noexcept;
    NatsConnection& operator=(NatsConnection&&) noexcept;

    [[nodiscard]] auto publish(std::string_view subject, std::string_view data)
        -> std::expected<void, NatsError>;
};
// ❌ ANTI-PATTERN — raw NATS handles without RAII
natsConnection* conn = nullptr;
natsConnection_ConnectTo(&conn, url);
// ... use ...
natsConnection_Destroy(conn);  // Leaked if exception thrown before this line

Audit: Framework NATS code uses raw handles — needs RAII retrofit. Framework files: framework/dba/include/muscovite/dba/nats_*.hpp

18. constexpr Where Possible

CANONICAL: Use constexpr for compile-time evaluable functions, lookup table UUIDs, and configuration constants.

// ✅ CANONICAL — constexpr lookup constants (ADR 20260212)
namespace lookup::saga_status {
    inline constexpr auto pending = boost::uuids::uuid{/* UUIDv5 bytes */};
    inline constexpr auto completed = boost::uuids::uuid{/* UUIDv5 bytes */};
    inline constexpr auto failed = boost::uuids::uuid{/* UUIDv5 bytes */};
}

// ✅ CANONICAL — constexpr for simple utilities
[[nodiscard]] constexpr auto is_nullable(FieldSpec spec) noexcept -> bool {
    return spec.nullable;
}
// ❌ ANTI-PATTERN — runtime constants that could be constexpr
const auto max_page_size = 100;      // Should be constexpr
const std::string table_name = "patient";  // Should be constexpr string_view

// ❌ ANTI-PATTERN — runtime lookup for framework constants (ADR 20260212)
auto status_id = query("SELECT id FROM saga_status WHERE code = 'completed'");

Audit: Current grade B- — constexpr header generation works, but not applied everywhere. Generator files: cpp/lookup-constants.rkt

19. Testing Patterns (doctest)

CANONICAL: All new C++ tests MUST use doctest BDD macros (ADR 20260212). Catch2 for existing tests only.

// ✅ CANONICAL — doctest BDD for new tests
#include <doctest/doctest.h>

TEST_SUITE("dba::PatientRepository") {

SCENARIO("creating a patient") {
    GIVEN("a valid create request") {
        auto req = CreatePatientRequest{
            .mrn = "MRN-001",
            .family_name = "Smith"
        };

        WHEN("insert is called") {
            auto result = repo.insert(txn, req);

            THEN("a patient with a UUID is returned") {
                REQUIRE(result.has_value());
                CHECK(result->id() != boost::uuids::nil_uuid());
                CHECK(result->mrn() == "MRN-001");
            }
        }
    }
}

}  // TEST_SUITE
// ❌ ANTI-PATTERN — Catch2 for new tests (existing tests may keep Catch2)
#include <catch2/catch_all.hpp>
TEST_CASE("PatientRepository", "[patient][dba]") { ... }

// ❌ ANTI-PATTERN — Google Test
TEST(PatientRepository, CreatePatient) { ... }

20. pqxx API Usage

CANONICAL: Use modern pqxx 7.x API with pqxx::params.

// ✅ CANONICAL — pqxx::params (ADR 20251229)
auto params = pqxx::params{id, mrn, family_name};
auto result = txn.exec(kInsertSql, params);

// ✅ CANONICAL — transaction types
pqxx::work txn{conn};              // Read-write transaction (commands)
pqxx::nontransaction ntxn{conn};   // Read-only (queries)
// ❌ ANTI-PATTERN — deprecated exec_params (ADR 20251229)
auto result = txn.exec_params(sql, id, mrn, family_name);

// ❌ ANTI-PATTERN — string interpolation in SQL
auto result = txn.exec("SELECT * FROM patient WHERE id = '" + id_str + "'");

Audit: exec_params fully eliminated (FTR-047 resolved). Maintain compliance. Generator files: dba/unified.rkt
