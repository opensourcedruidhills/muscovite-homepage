PostgreSQL Generator Patterns

Purpose: Canonical pattern catalog for AI agents working on PostgreSQL Sqitch generators. Companion to: ADR 20260227 — PostgreSQL Idiomatic Output Quality Sibling to: RACKET-PATTERNS.md (Racket coding constraints)

Rating System

Rating Meaning
CANONICAL The only acceptable pattern for new and modified code
ACCEPTABLE Tolerated in existing code; convert when touching the file
ANTI-PATTERN Must never appear in new code; fix immediately when found

1. UUID Default Functions

CANONICAL: Use (uuid-default-fn) from sqitch-common.cpp/.hpp (returns "uuidv7()").

-- ✅ CANONICAL
id UUID PRIMARY KEY DEFAULT uuidv7()

-- ❌ ANTI-PATTERN: hardcoded gen_random_uuid()
id UUID PRIMARY KEY DEFAULT gen_random_uuid()

Rationale: UUIDv7 provides time-ordered UUIDs that improve B-tree index locality and enable temporal ordering without a separate created_at column. sqitch-common.cpp/.hpp L61-63 defines the single source of truth.

Affected files (current violations):

  • sqitch-event-changes.cpp/.hpp — 4 occurrences (L38, L159, L180, L200)
  • sqitch-state-machine.cpp/.hpp — 2 occurrences (L121, L220)
  • sqitch-saga.cpp/.hpp — 3 occurrences (L132, L151, L166)
  • sqitch-events.cpp/.hpp — 2 occurrences (L46, L150)
  • migration-sql-deploy.cpp/.hpp — 1 occurrence (L174)
  • sqitch-bitemporal.cpp/.hpp — 1 occurrence (system trigger)

2. Foreign Key Constraints

CANONICAL: Named constraints via ALTER TABLE ... ADD CONSTRAINT.

-- ✅ CANONICAL
ALTER TABLE encounter.encounter
    ADD CONSTRAINT fk_encounter_patient_id
    FOREIGN KEY (patient_id) REFERENCES patient.patient(id);

-- ❌ ANTI-PATTERN: inline REFERENCES in CREATE TABLE
CREATE TABLE encounter.encounter (
    patient_id UUID NOT NULL REFERENCES patient.patient(id)  -- unnamed, inline
);

Rationale: Named constraints appear in pg_constraint, error messages, and migration scripts. Inline REFERENCES produces auto-generated names like encounter_patient_id_fkey that vary across PostgreSQL versions.

Reference implementation: migration-sql-deploy.cpp/.hpp L127 already uses named FK constraints.

Affected files (current violations):

  • sqitch-entity.cpp/.hpp — L66, L111
  • sqitch-entity-columns.cpp/.hpp — L93, L112, L146
  • sqitch-state-machine.cpp/.hpp — L113, L122-126, L221-223
  • sqitch-event-changes.cpp/.hpp — L181, L201

FTR: FTR-160

3. FK Column Indexes

CANONICAL: Every FK column gets a B-tree index.

-- ✅ CANONICAL: index immediately after FK constraint
ALTER TABLE encounter.encounter
    ADD CONSTRAINT fk_encounter_patient_id
    FOREIGN KEY (patient_id) REFERENCES patient.patient(id);

CREATE INDEX idx_encounter_patient_id
    ON encounter.encounter (patient_id);

-- ❌ ANTI-PATTERN: FK without index
-- (causes sequential scan on JOIN/CASCADE DELETE)

Rationale: PostgreSQL does NOT auto-create indexes on FK columns (unlike MySQL InnoDB). Without indexes, JOIN through FK and CASCADE DELETE from parent trigger sequential scans.

Existing infrastructure: indexes-triggers.cpp/.hpp L58 generates metadata-driven indexes but does not systematically cover all FK columns.

FTR: FTR-161

4. Column and Object Documentation (COMMENT ON)

CANONICAL: Every database object gets a COMMENT ON when the description is derivable.

Current coverage (✅ = fully generated, ❌ = gap):

Object Type Status Generator
TABLE ✅ entity, saga, projection, state-machine
COLUMN ✅ sqitch-entity-columns.cpp/.hpp:490-525
CONSTRAINT (FK) ✅ sqitch-entity-columns.cpp/.hpp:526-543
CONSTRAINT (CHECK) ✅ sqitch-entity-columns.cpp/.hpp:545-557
TRIGGER ✅ audit triggers

DOMAIN ✅ sqitch-entity-domains.cpp/.hpp:78-79

-- ✅ CANONICAL: column comment from DSL field-level DOC
COMMENT ON COLUMN patient.patient.family_name
    IS 'The patient''s legal family name as recorded in official documents';

-- ✅ CANONICAL: auto-derived fallback when no explicit DOC
COMMENT ON COLUMN patient.patient.given_name
    IS 'Given name of patient';

-- ✅ CANONICAL: audit column comments (always generated)
COMMENT ON COLUMN patient.patient.created_at IS 'Row creation timestamp';
COMMENT ON COLUMN patient.patient.updated_at IS 'Last modification timestamp';

-- ✅ CANONICAL: table-level comment
COMMENT ON TABLE patient.patient
    IS 'A person receiving healthcare services';

-- ✅ CANONICAL: FK constraint comment
COMMENT ON CONSTRAINT fk_encounter_patient_id ON encounter.encounter
    IS 'FK patient -> patient.patient (to-one, required)';

-- ✅ CANONICAL: CHECK constraint comment
COMMENT ON CONSTRAINT chk_patient_birth_date ON patient.patient
    IS 'Birth date must not be in the future';

Implementation: The sql-comment-on helper in sqitch-common.cpp/.hpp:279-288 handles all object types uniformly. It accepts an optional #:source keyword for provenance tracking. Column comments use field-level DOC from the DSL (grammar: FieldDocClause, parser: parse-doc, IR: field.doc) when present, with auto-derivation from field names as fallback.

Rationale: Per ADR 20260217, all database objects that support COMMENT ON must have generated comments. Comments appear in \d+ output, pgAdmin, and BI tool metadata discovery.

FTR: FTR-162 (initial), FTR-344 (field-level DOC), FTR-422 (INDEX/UNIQUE/EXCLUDE gap)

5. String Quoting in Seeds

CANONICAL: Standard SQL single quotes with doubled escaping.

-- ✅ CANONICAL
INSERT INTO basedata.priority (id, code, display)
VALUES ('...', 'high', 'High Priority');

-- ✅ CANONICAL: escaped single quote
VALUES ('...', 'o''brien', 'O''Brien');

-- ❌ ANTI-PATTERN: E-prefix string literals
VALUES (E'...', E'high', E'High Priority');

Rationale: E'...' is PostgreSQL-specific escape string syntax. Standard '...' quoting with '' for embedded quotes is SQL-standard and portable. No seed data requires backslash escapes.

Affected file: sqitch-unified-seeds.cpp/.hpp sql-escape-value (L68-85).

FTR: FTR-163

6. Idempotent Seed Data

CANONICAL: INSERT ... ON CONFLICT (id) DO UPDATE SET.

-- ✅ CANONICAL: idempotent upsert
INSERT INTO basedata.priority (id, code, display, sort_order, is_active)
VALUES
    ('...', 'low', 'Low', 1, true),
    ('...', 'medium', 'Medium', 2, true),
    ('...', 'high', 'High', 3, true)
ON CONFLICT (id) DO UPDATE SET
    code = EXCLUDED.code,
    display = EXCLUDED.display,
    sort_order = EXCLUDED.sort_order,
    is_active = EXCLUDED.is_active;

-- ❌ ANTI-PATTERN: bare INSERT (fails on re-deploy)
INSERT INTO basedata.priority (id, code, display)
VALUES ('...', 'low', 'Low');

Rationale: Sqitch deploy may run seeds multiple times during development. Bare INSERT fails with duplicate key violation on second run. ON CONFLICT ... DO UPDATE makes deploys idempotent.

Additional improvement: Multi-row VALUES instead of one INSERT per row (reduces round-trips).

Affected file: sqitch-unified-seeds.cpp/.hpp INSERT generation (L136-141).

FTR: FTR-163

7. SQL Object Naming Conventions

CANONICAL: Consistent prefixed names for all SQL objects.

Object Prefix Pattern Example
Foreign key fk_ fk_{table}_{column} fk_encounter_patient_id
B-tree index idx_ idx_{table}_{column} idx_encounter_patient_id
GiST index idx_ idx_{table}_{column} idx_observation_location
Trigger trg_ trg_{table}_{purpose} trg_patient_audit
Check constraint chk_ chk_{table}_{rule} chk_patient_birth_date_not_future
Unique constraint uq_ uq_{table}_{columns} uq_patient_mrn
Exclusion (GIST) excl_ excl_{table}_{purpose} excl_patient_valid_period
Primary key pk_ pk_{table} pk_patient
Sequence seq_ seq_{table}_{column} seq_order_number 
-- ✅ CANONICAL
ALTER TABLE patient.patient
    ADD CONSTRAINT fk_patient_status_id
    FOREIGN KEY (status_id) REFERENCES basedata.status(id);

CREATE INDEX idx_patient_status_id
    ON patient.patient (status_id);

-- ❌ ANTI-PATTERN: inconsistent or missing prefixes
ALTER TABLE patient.patient
    ADD CONSTRAINT patient_status_fkey ...  -- PostgreSQL auto-name

8. Verify Script Depth

CANONICAL: Verify scripts check structure, not just existence.

-- ✅ CANONICAL: deep verify
DO $$
BEGIN
    -- Verify table exists with correct columns
    ASSERT (
        SELECT count(*) FROM information_schema.columns
        WHERE table_schema = 'patient'
          AND table_name = 'patient'
          AND column_name IN ('id', 'mrn', 'family_name', 'given_name')
    ) = 4, 'patient.patient missing expected columns';

    -- Verify FK constraint exists
    ASSERT EXISTS (
        SELECT 1 FROM pg_constraint
        WHERE conname = 'fk_patient_status_id'
          AND contype = 'f'
    ), 'FK constraint fk_patient_status_id missing';

    -- Verify index exists
    ASSERT EXISTS (
        SELECT 1 FROM pg_indexes
        WHERE indexname = 'idx_patient_status_id'
    ), 'Index idx_patient_status_id missing';
END $$;

-- ❌ ANTI-PATTERN: shallow verify
SELECT 1 FROM patient.patient WHERE FALSE;

Rationale: Shallow verify only checks table existence. A table could exist with wrong columns, missing constraints, or missing indexes and shallow verify would pass. Deep verify catches deployment drift.

FTR: FTR-164

9. JSONB Exception for Infrastructure Tables

CANONICAL: JSONB is allowed for event infrastructure tables only.

-- ✅ CANONICAL: JSONB in domain_events (infrastructure)
CREATE TABLE patient.domain_events (
    id              UUID PRIMARY KEY DEFAULT uuidv7(),
    aggregate_type  TEXT NOT NULL,
    aggregate_id    UUID NOT NULL,
    event_type      TEXT NOT NULL,
    payload         JSONB NOT NULL,
    occurred_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- ❌ ANTI-PATTERN: JSONB in domain entity (parser rejects this)
CREATE TABLE patient.patient (
    metadata JSONB  -- NEVER in domain entities
);

Rationale: The parser-level JSONB prohibition applies to domain model fields in .ddd files. Infrastructure tables (domain_events, saga_state, event_store.*) have fixed schemas defined by generators and serialize typed C++ structs to JSON for messaging. They are never queried by payload field.

Exempt tables: {schema}.domain_events, {schema}.domain_events_dlq, event_store.*, {schema}.saga_*, {schema}.outbox

10. Schema Qualification

CANONICAL: All table references in generated SQL are schema-qualified.

-- ✅ CANONICAL
CREATE TABLE patient.patient (...);
ALTER TABLE patient.encounter ADD CONSTRAINT ... REFERENCES patient.patient(id);
COMMENT ON TABLE patient.patient IS '...';

-- ❌ ANTI-PATTERN: unqualified references
CREATE TABLE patient (...);  -- which schema?

Rationale: Schema qualification prevents ambiguity when search_path is not set. All generated SQL must work regardless of connection search_path settings.

11. Transaction Safety

CANONICAL: DDL operations should be safe for Sqitch's single-transaction deploy.

-- ✅ CANONICAL: DDL is transactional in PostgreSQL
BEGIN;
CREATE TABLE patient.patient (...);
ALTER TABLE patient.patient ADD CONSTRAINT ...;
CREATE INDEX idx_patient_status_id ON patient.patient (status_id);
COMMIT;

-- ⚠️ ACCEPTABLE: CREATE INDEX CONCURRENTLY (cannot run in transaction)
-- Use only when explicitly needed for zero-downtime migrations

Rationale: PostgreSQL DDL is transactional (unlike MySQL). Sqitch deploys each change in a transaction. CREATE INDEX CONCURRENTLY is the exception — it cannot run inside a transaction block.

12. Enforcement Test

CANONICAL: A Racket test scans generated SQL output for pattern violations.

;; test-postgres-patterns.cpp/.hpp
;; Scans generated SQL files and asserts:
;; - No gen_random_uuid() (except documented exceptions)
;; - All FK constraints have fk_ prefix
;; - All FK columns have idx_ indexes
;; - No E'...' string literals in seed files
;; - All seed INSERTs have ON CONFLICT clause
;; - All verify scripts use DO $$ ... END $$ structure

Rationale: Patterns enforced by tests cannot regress. This is the PostgreSQL equivalent of test-grammar-conformance.cpp/.hpp for DSL syntax.

13. Outbox Table Infrastructure

CANONICAL: Per-context outbox table with partial index on unprocessed rows.

-- ✅ CANONICAL: outbox table (one per bounded context)
CREATE TABLE patient.outbox (
    id            UUID PRIMARY KEY DEFAULT uuidv7(),
    effect_type   TEXT NOT NULL,
    payload       JSONB NOT NULL,
    aggregate_id  UUID NOT NULL,
    created_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    processed_at  TIMESTAMPTZ,
    retry_count   INTEGER NOT NULL DEFAULT 0
);

-- ✅ CANONICAL: partial index for polling unprocessed entries
CREATE INDEX idx_outbox_unprocessed
    ON patient.outbox (created_at)
    WHERE processed_at IS NULL;

Rationale: The transactional outbox pattern (ADR 20260307) writes domain effects to the outbox table within the same transaction as the aggregate mutation. A background poller reads unprocessed rows using the partial index (WHERE processed_at IS NULL), publishes them, and marks them processed. The partial index ensures the poller only scans the working set.

JSONB is permitted here because the outbox serializes typed C++ effect structs — it is infrastructure, not domain data (see §9).

Generator: sqitch-outbox.cpp/.hpp:34-56

14. CQRS Projection Views

CANONICAL: Projections generate as MATERIALIZED VIEW with a UNIQUE INDEX on id.

-- ✅ CANONICAL: materialized view projection
CREATE MATERIALIZED VIEW patient.patient_summary AS
    SELECT p.id, p.family_name, p.given_name, s.display AS status
    FROM patient.patient p
    JOIN basedata.status s ON p.status_id = s.id;

COMMENT ON MATERIALIZED VIEW patient.patient_summary
    IS 'Read-optimized view of patient with resolved status';

-- ✅ CANONICAL: unique index required for REFRESH CONCURRENTLY
CREATE UNIQUE INDEX idx_patient_summary_id
    ON patient.patient_summary (id);

Rationale: REFRESH MATERIALIZED VIEW CONCURRENTLY requires a unique index to perform an incremental refresh without locking readers. Without it, PostgreSQL forces an exclusive lock during refresh, blocking all read queries.

ON_EVENT projections generate as regular tables with explicit COMMENT ON TABLE.

Generator: sqitch-projections.cpp/.hpp:146-198

15. Bitemporal Data & GiST Indexes

CANONICAL: Bitemporal entities use EXCLUDE USING GIST to prevent overlapping valid-time periods and btree_gist extension for mixed-type exclusion.

-- ✅ CANONICAL: btree_gist required for EXCLUDE with non-range columns
CREATE EXTENSION IF NOT EXISTS btree_gist;

-- ✅ CANONICAL: system-time columns (SYSTEM_TIME AUTO)
CREATE TABLE patient.patient (
    id          UUID PRIMARY KEY DEFAULT uuidv7(),
    -- ... domain fields ...
    valid_from  TIMESTAMPTZ NOT NULL,
    valid_to    TIMESTAMPTZ NOT NULL DEFAULT 'infinity'::TIMESTAMPTZ,
    system_from TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    system_to   TIMESTAMPTZ NOT NULL DEFAULT 'infinity'::TIMESTAMPTZ,

    -- ✅ CANONICAL: EXCLUDE prevents overlapping periods per entity
    CONSTRAINT excl_patient_valid_period
        EXCLUDE USING GIST (
            id WITH =,
            tstzrange(valid_from, valid_to, '[)') WITH &&
        )
);

-- ✅ CANONICAL: system-time trigger for automatic timestamping
CREATE TRIGGER trg_patient_system_time
    BEFORE INSERT OR UPDATE ON patient.patient
    FOR EACH ROW
    EXECUTE FUNCTION _system.set_system_time_columns();

-- ✅ CANONICAL: GiST index for spatial/geometry columns
CREATE INDEX idx_observation_location
    ON observation.observation USING GIST(location);

Rationale: The EXCLUDE USING GIST constraint enforces the temporal invariant at the database level: no two rows for the same entity (id WITH =) may have overlapping valid-time ranges (tstzrange(...) WITH &&). The [) range semantics (closed-open) ensures contiguous periods are allowed. btree_gist is required because id is UUID (B-tree type) but GIST indexes natively support range types only.

GiST indexes for geometry columns enable efficient spatial queries (ST_Contains, ST_Intersects, etc.) on PostGIS types.

Generator: sqitch-entity-columns.cpp/.hpp:388-426 (bitemporal), sqitch-entity-columns.cpp/.hpp:470-476 (spatial GiST)

ADR: 20260221 (bitemporal design), FTR-107/108

16. PostgreSQL DOMAIN Strong Types

CANONICAL: Value objects with constraints generate as PostgreSQL DOMAIN types.

-- ✅ CANONICAL: domain with CHECK constraint
CREATE DOMAIN patient.email_address AS TEXT
    CONSTRAINT chk_email_address CHECK (VALUE ~ '^[^@]+@[^@]+\.[^@]+$');

COMMENT ON DOMAIN patient.email_address IS 'RFC-compliant email address';

-- ✅ CANONICAL: domain with default
CREATE DOMAIN patient.priority_level AS INTEGER
    DEFAULT 0
    CONSTRAINT chk_priority_level CHECK (VALUE >= 0 AND VALUE <= 10);

Rationale: DOMAIN types push validation to the database layer, ensuring constraints are enforced even for raw SQL inserts outside the application. They appear in information_schema.domains and provide self-documenting column types in \d+ output.

Generator: sqitch-entity-domains.cpp/.hpp:31-79

ADR: 20260119

17. UUIDv5-Seeded Lookup Tables

CANONICAL: Framework and saga lookup tables use deterministic UUIDv5 identifiers seeded from a namespace UUID and the code string.

-- ✅ CANONICAL: lookup table with deterministic IDs
CREATE TABLE patient.saga_status (
    id       UUID PRIMARY KEY,
    code     TEXT NOT NULL UNIQUE,
    display  TEXT NOT NULL,
    doc      TEXT
);

COMMENT ON TABLE patient.saga_status IS 'Saga status (framework lookup)';

-- ✅ CANONICAL: UUIDv5-seeded inserts (deterministic across environments)
INSERT INTO patient.saga_status (id, code, display, doc) VALUES
    ('...uuid5...', 'not_started',  'Not Started',  'Saga created but not yet started'),
    ('...uuid5...', 'in_progress',  'In Progress',  'Saga is actively executing steps'),
    ('...uuid5...', 'compensating', 'Compensating', 'Saga is rolling back completed steps'),
    ('...uuid5...', 'completed',    'Completed',    'Saga completed successfully'),
    ('...uuid5...', 'failed',       'Failed',       'Saga failed after compensation or error'),
    ('...uuid5...', 'cancelled',    'Cancelled',    'Saga was cancelled before completion')
ON CONFLICT (id) DO UPDATE SET
    code = EXCLUDED.code, display = EXCLUDED.display, doc = EXCLUDED.doc;

Rationale: UUIDv5 ensures identical IDs across dev/staging/prod environments — no environment-specific seed data. The C++ framework generates constexpr headers with these IDs (ADR 20260212), enabling compile-time lookup:

// Generated — never SELECT at runtime for framework constants
namespace lookup::saga_status {
    inline constexpr auto completed = "...uuid5...";
}

Framework lookup codes: saga_status (6 codes), saga_step_result (3 codes), plus per-saga {saga_name}_step tables with one entry per named step.

Generator: saga-lookup-tables.cpp/.hpp:48-75 (codes), saga-lookup-tables.cpp/.hpp:200-240 (SQL)

ADR: 20260212 (compile-time lookup constants)
