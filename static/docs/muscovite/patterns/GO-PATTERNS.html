Go Patterns — `muscovite-go-extractor` Coding Guide

Go Patterns — muscovite-go-extractor Coding Guide

Applies to: muscovite-go-extractor repository and tools/go-extractor/ subtree ADR: 20260310-brownfield-extraction-architecture Severity levels: 🔴 FORBIDDEN (PR-blocking) | 🟡 REQUIRED (review-blocking) | 🟢 PREFERRED

1. Language Version and Module Setup

🟡 Go 1.22+ — use latest stable Go. Module path: github.com/opensourcedruidhills/muscovite-go-extractor.

🔴 FORBIDDEN: replace directives in go.mod for released code. Development-only replace must be removed before merge.

2. Zero External Dependencies for Core Logic

🟡 Core extraction logic uses Go stdlib only: go/ast, go/parser, go/types, go/token, os, path/filepath, strings, fmt, encoding/json, text/template.

🟢 Allowed external dependencies (minimal surface):

  • YAML parser: gopkg.in/yaml.v3 (config files only)
  • PEG generator: build-time only, not a runtime dependency

🔴 FORBIDDEN: ORM libraries, database drivers, HTTP frameworks, logging frameworks (use log/slog).

3. Package Layout

🟡 Follow the standard Go project layout:

cmd/go-extractor/main.go    # CLI entry, flag parsing, orchestration
pkg/ast/                     # Go AST walking, type resolution
pkg/orm/xorm/                # XORM-specific tag parsing
pkg/orm/gorm/                # GORM-specific tag parsing (future)
pkg/relationship/            # Relationship inference from pointer fields
pkg/behavioral/              # Service layer analysis (commands, events, state machines)
pkg/ui/                      # Template + Swagger analysis
pkg/confidence/              # Multi-source confidence scoring
pkg/emitter/                 # .ddd file generation
pkg/validator/               # PEG-based .ddd validation
pkg/config/                  # YAML configuration

🔴 FORBIDDEN: Circular imports between packages. Each package has a single responsibility.

4. Error Handling

🟡 Return errors, don't panic. Use fmt.Errorf("context: %w", err) for wrapping.

🟡 Errors must include source location: file path, line number, struct/field name.

// GOOD
return fmt.Errorf("parsing struct %s in %s:%d: %w", name, file, line, err)

// BAD
return err

🔴 FORBIDDEN: panic() outside of test code. log.Fatal() outside of main().

5. AST Walking Patterns

🟡 Use ast.Inspect for simple traversals, ast.Walk with a visitor for complex ones.

🟡 Resolve types via go/types — don't rely on string matching for type names.

// GOOD: type-checked resolution
info := &types.Info{Types: make(map[ast.Expr]types.TypeAndValue)}
conf := types.Config{Importer: importer.ForCompiler(fset, "source", nil)}
pkg, err := conf.Check("path", fset, files, info)

// BAD: string matching
if fieldType == "*repo_model.Repository" { ... }

🟢 Cache parsed packages — don't reparse the same package for different analysis passes.

6. Struct Tag Parsing

🟡 Use reflect.StructTag.Lookup() for standard Go struct tags. For XORM's non-standard format (space-separated, not key:"value"), use a custom tokenizer.

🟡 XORM tag tokenizer must handle all 15 keywords documented in FTR-350: pk, autoincr, INDEX, UNIQUE(name), NOT NULL, DEFAULT x, TEXT, LONGTEXT, VARCHAR(n), created, updated, JSON, -, name, 'column'.

🔴 FORBIDDEN: Regex-only tag parsing. XORM tags have context-dependent semantics (e.g., UNIQUE vs UNIQUE(group_name)).

7. Confidence Scoring

🟡 Every extracted construct carries a confidence score (0.0–1.0) and an evidence list.

type Evidence struct {
    Source     string  // "schema", "service", "template", "api", "constraint"
    File       string  // source file path
    Line       int     // source line number
    Detail     string  // human-readable description
}

type Construct struct {
    // ...
    Confidence float64
    Evidence   []Evidence
}

🟡 Score thresholds are configurable via extractor-config.yaml, not hardcoded.

8. .ddd Output Generation

🟡 Use text/template for .ddd output — never raw fmt.Sprintf concatenation.

🟡 Templates live in pkg/emitter/templates/ — one template per .ddd construct type.

🟡 Every output file must include the SPDX license header and generation metadata comment.

🟡 Validate output before writing: call the Go PEG validator on the rendered .ddd string. If validation fails, log the error and skip the file (don't write invalid .ddd).

// GOOD
rendered := renderDDD(entity)
if err := validator.Validate(rendered); err != nil {
    slog.Error("invalid .ddd output", "entity", entity.Name, "err", err)
    return err  // skip, don't write broken output
}
os.WriteFile(path, []byte(rendered), 0644)

🔴 FORBIDDEN: Writing .ddd files that haven't passed PEG validation.

9. Testing

🟡 Table-driven tests for all tag parsing, relationship detection, and type mapping.

🟡 Golden file tests for .ddd output: testdata/golden/ contains expected .ddd output, tests compare actual output against golden files. Update goldens with -update flag.

🟡 Use testdata/ directory for test fixtures — vendored Go model subsets from real projects.

🟢 Fuzz testing for XORM tag parser: func FuzzXORMTags(f *testing.F).

// GOOD: table-driven
func TestXORMTagParser(t *testing.T) {
    tests := []struct {
        name     string
        tag      string
        wantPK   bool
        wantNull bool
    }{
        {"primary key", "pk autoincr", true, false},
        {"nullable text", "TEXT", false, true},
        {"unique group", "UNIQUE(repo_index)", false, false},
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) { ... })
    }
}

🔴 FORBIDDEN: Tests that require network access or external services.

10. Naming Conventions

🟡 Follow standard Go naming:

  • Exported types: PascalCase (EntityModel, RelationshipDetector)
  • Unexported: camelCase (parseStructTag, resolvePointerField)
  • Acronyms: ID not Id, URL not Url, ORM not Orm
  • Interfaces: verb-based (Parser, Detector, Emitter), not IParser

🟡 DDD-specific names: Use Muscovite terminology consistently:

  • Entity (not Table or Model — those are source concepts)
  • Field (not Column — that's the legacy name)
  • Relationship (not ForeignKey or Reference)
  • BoundedContext (not Module or Package)

11. Logging

🟡 Use log/slog (Go 1.21+ structured logging). No third-party loggers.

🟡 Log levels:

  • slog.Info: Extraction progress (per-context, per-pass)
  • slog.Warn: Low-confidence constructs, ambiguous patterns
  • slog.Error: Extraction failures, validation errors
  • slog.Debug: Per-struct, per-field details (verbose)
slog.Info("extraction pass complete",
    "pass", "schema",
    "entities", len(entities),
    "relationships", len(rels),
    "duration", elapsed)

12. CLI Interface

🟡 Use flag package (stdlib). No cobra/urfave unless complexity demands it later.

🟡 Required flags: --config (YAML path), --output (output directory). 🟢 Optional flags: --passes (comma-separated: schema,behavioral,ui), --min-confidence, --validate-only (dry run), --verbose.

🟡 Exit codes: 0=success, 1=extraction error, 2=validation error, 3=config error.

13. Concurrency

🟢 Parallelize per-file AST parsing — files within a package are independent.

🟡 Don't parallelize cross-package type resolution — type checker needs sequential package loading.

🟡 Triangulation is single-threaded — merging evidence across passes requires consistent ordering.

14. Grammar Synchronization

🟡 The PEG grammar (grammar/muscovite.peg) must be synced from grammar/muscovite.peg at build time. A go generate directive or Makefile target handles this.

🟡 The Go PEG validator is regenerated when the grammar changes. CI enforces that the generated validator matches the current grammar.

🔴 FORBIDDEN: Modifying the grammar copy in the go-extractor — the toolchain/muscomp/grammar/ copy is the single source of truth (ADR 20260211).
