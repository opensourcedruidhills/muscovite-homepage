1. System Purpose

Muscovite is a Racket-based DDD code generation framework that transforms declarative domain models (.ddd files) into a complete full-stack application.

2. Architecture & DSL Governance

Three-Layer Hierarchy

Layer Location Purpose
Layer 0 — Brain dslgen/ (Racket) DSL parser, IR model, all generators
Layer 1 — Framework framework/ (C++) Reusable base classes (Repository, UnitOfWork, errors)
Layer 2 — Generated examples/{project}/ 100% generated from .ddd models. Never manually edited.

DSL Governance Chain

PEG Grammar (spec) → Parser (impl) → Grammar Conformance Test (parity check)
       ↓                    ↓                        ↓
  Syntax rules      IR struct instances      Keyword drift detection
       ↓                    ↓                        ↓
  doc/DSL.md (auto)   Generators (emit)       doc/CONTEXT.md (metrics)

Rule: Grammar changes precede parser changes (ADR 20260211). Zero opaque stubs.

12. User Profile

User Profile: Jolo

AI agents: Read this to understand communication style and working patterns.

Summary

Role: Staff Engineer / Technical Lead / Former Department Head Experience: 10+ years systems architecture, DDD, multi-language stacks (C++, C#, Python, Racket, SQL, QML) Key Trait: Pragmatic — values working code over theory, documentation over talk, tests over intuition

Communication Style

Preferred format: ## Status: [Feature] → Completed ✅ / Blocking (file:line) / Next Action (test command).

Do: Bullet points, tables, code snippets. File:line references. Test output as proof. Executable next steps.

Don't: ❌ "This might work..." | ❌ "Usually this should..." | ❌ Long explanations without actions | ❌ Workarounds instead of root-cause fixes.

What Frustrates

Anti-Pattern Why
"Let's add X because it's cool" No domain justification
"Quick fix now, proper later" Technical debt
"Just edit the generated code" Violates DSL-first
Suggesting manual SQL Bypasses migration pipeline

What Builds Trust

Pattern Why
"Here's a failing test proving the bug" Test-driven
"This affects files X, Y, Z" Holistic awareness
"Per ADR 20260120, use domain services" Respects decisions
Direct line references Precise, actionable

Technical Philosophy

  • Generator-first: Repeating code → Racket template
  • Fail fast: Catch errors at generation, not runtime
  • Single source of truth: DSL drives everything
  • Automation over discipline: Tools enforce conventions
  • Integration over unit tests: Meaningful > numerous
  • Vertical feature completeness: Full stack or nothing — no partial implementations
  • Production-grade or nothing: Rejects toy/lightweight tools for core infrastructure

How Agents Should Adapt

  1. Lead with executable actions, not explanations
  2. Cite specific files/lines
  3. Show test output to validate
  4. Use tables/bullets for decisions
  5. Never suggest editing generated code
  6. Reference ADRs for architectural changes
  7. Identify the correct layer (parser? generator? framework?) — wrong-layer suggestions erode trust immediately
  8. Think about downstream impact across all 8 generator targets
  9. Check license compatibility against AGPL-3.0 + Commercial dual licensing
  10. Suggest production-grade solutions sized to the domain
