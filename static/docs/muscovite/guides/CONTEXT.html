1. Project Summary

Muscovite is a Racket-based DDD code generation framework that transforms declarative domain models (.ddd files) into:

  • PostgreSQL schemas (Sqitch migrations, DOMAINs, constraints, seed data)
  • C++ backend (domain services, repositories via pqxx, event emission, event sourcing)
  • gRPC/Protobuf services with pagination and FK resolution
  • Qt Widgets desktop clients (master-detail, state machine validation)
  • Qt Quick/QML clients (ViewModel pattern, all-fields Q_PROPERTY)
  • CMake/Conan build system for standalone generated projects

3. Three-Layer Architecture

Layer Name Location Purpose
Layer 0 The Brain (Generator/IR) dslgen/ (Racket) DSL parser, IR model, all generators
Layer 1 The Framework framework/ (C++) Reusable base classes (Repository, UnitOfWork, errors). No domain logic.
Layer 2 Generated Applications examples/*/generated/ 100% generated from .ddd models. Never manually edited.

Critical rule: Changes flow Layer 0 → Layer 1 → Layer 2. Never edit Layer 2 directly.

Entity Example

    id          UUID        NOT_NULL @uuid "...",
    mrn         TEXT        NOT_NULL @unique @uuid "...",
    birth_date  DATE        @uuid "...",
    created_at  TIMESTAMPTZ NOT_NULL @uuid "..."
)
CHECK birth_date_not_future @uuid "...": birth_date <= CURRENT_DATE
DOC "A person receiving care"
SYNONYMS "client", "subject";

Relationships (Explicit Only)

ENTITY Observation @uuid "..." IDENTIFIED_BY id (
    id          UUID NOT_NULL,
    patient_id  UUID NOT_NULL  -- Just a UUID field, NOT auto-FK
)
RELATIONSHIPS {
    patient -> Patient REQUIRED ONE;          -- Same context
    status -> Basedata.Status REQUIRED ONE;   -- Cross-context
    encounter -> Encounter OPTIONAL ONE @embedded;
}

Saga Example

SAGA Fulfillment @uuid "..." TRIGGERED_BY OrderPlaced CORRELATE_BY order_id {
    STEP ReserveInventory
        INVOKE ReserveItems(order_id = $order_id)
        ON SUCCESS InventoryReserved -> ProcessPayment
        ON FAILURE InventoryNotAvailable -> COMPENSATE -> FAILED
        TIMEOUT 5m;

    STEP ProcessPayment
        INVOKE ChargePayment(order_id = $order_id)
        ON SUCCESS PaymentCharged -> COMPLETED
        ON FAILURE PaymentDeclined -> COMPENSATE -> FAILED
        COMPENSATES_WITH ReleaseInventory(order_id = $order_id)
        TIMEOUT 30s;
}
ON COMPLETED EMIT OrderFulfilled(order_id = $order_id)
ON FAILED EMIT OrderFailed(order_id = $order_id)
DOC "Order fulfillment workflow";

9. Architecture Patterns

Domain Services as Aggregate Boundary (ADR 20260120)

All aggregate access through CommandService/QueryService. No stored procedures.

gRPC → CommandService → CommandRepository (pqxx) → PostgreSQL
         └── emit event to domain_events table
Type Transaction Methods
CommandRepository pqxx::work insert(), update(), remove()
QueryRepository pqxx::nontransaction find_by_id(), find_all()

Event Sourcing (ADR 20260204)

Opt-in per aggregate with EVENT_SOURCED. Generates:

  • Typed event tables (no JSON) in event_store schema
  • Snapshot tables for current state
  • Dual-write: event + snapshot + entity table in single transaction

CQRS (ADR 20260120)

  • Command/query separation with separate repositories
  • Domain event emission on every write
  • Projection support (VIEWs + MATERIALIZED VIEWs)

Multi-Frontend Generation (ADR 20260206)

Frontend Flag Transport
Qt Widgets --qt-frontend widgets Direct C++ DBA
Qt Quick/QML --qt-frontend qml Direct C++ DBA

10. Build Workflow

Generate from DSL

cd dslgen && racket generate.rkt --project ../examples/fhir-health/project.mus

Options: --cpp-dba, --cpp-dba-test, --grpc, --qt-frontend widgets|qml|both, --test, --full, --dry-run, --fresh

Build C++ Backend

conan install --profile conan_profiles/debug .
cmake --preset debug
cmake --build --preset debug

Presets: debug, release, asan, tsan, coverage

Run Tests

# Racket tests
cd dslgen && racket tests/test-dsl-integration.rkt

# C++ tests
ctest --preset debug --output-on-failure
ctest --preset unit    # No database needed

Database Setup

cmake --build --preset debug --target docker-start      # Start PostgreSQL
cmake --build --preset debug --target db-setup-fhir      # Full setup
cmake --build --preset debug --target db-fresh-fhir      # Drop + recreate
cmake --build --preset debug --target db-help            # List all targets

Install mold + ccache for 6× faster rebuilds (auto-detected, ON by default).

16. Strategic Direction

Muscovite validates a 25-year thesis: a declarative domain model should be the single source of truth for an entire application stack.

Target Independence via IR

The IR supports arbitrary retargeting:

Current Alternative
PostgreSQL Oracle, SQL Server, CockroachDB
C++ / pqxx .NET / EF Core, Java / Spring, Rust / SQLx
gRPC / Protobuf REST / OpenAPI, GraphQL
Qt Widgets / QML React, Blazor

Adding a target = writing a generator (~1K–5K lines of Racket).

Legacy Modernization Pipeline (Long-Term Vision)

Brownfield System → AI Extraction (LLM + grammar) → .ddd Model → Human Review → New Stack

The .ddd file is the review artifact between AI extraction and human validation.
