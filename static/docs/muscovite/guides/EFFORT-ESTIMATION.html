This document estimates the effort required to produce the Muscovite platform and the projects built with it, across three scenarios:

  • Scenario A — Manual: all code written by hand, no code generation
  • Scenario B — Muscovite: .ddd domain models written manually, all generated code produced by the platform
  • Scenario C — Actual: the real effort spent building Muscovite and all example projects (moonlighting since November 2025, occasional weekends)

1. Platform Investment

The Muscovite platform itself (generators, framework, tooling) is a fixed cost that enables all downstream projects. This cost exists in Scenario B and Scenario C but not in Scenario A.

2. Per-Project Effort Estimation

Consumer repos (fhir-health, geosciences, hello-world, etc.) are independent projects that manage their own lifecycle. Each consumer tracks its own DSL input and generated output. Refer to the individual consumer repo's metrics for project-specific estimates.

3. Aggregate Effort Summary

Per-project generated output metrics are no longer collected in this report (FTR-449). Consumer repos publish their own metrics. The platform investment (Section 1) remains the primary effort metric tracked here.

4. Per-Line Productivity Context

These estimates use ~200 lines/day for Scenario A (manual C++ with tests). This rate assumes:

  • Production-quality code with proper error handling
  • Unit and integration tests (90%+ coverage)
  • Code review, documentation
  • Build system integration (CMake, Conan)
  • Cross-platform considerations (Qt Widgets + QML)

For generated code (Scenarios B/C), the rate is irrelevant — the generator produces all output in seconds.

5. Key Insight: Domain Modeling is the Bottleneck

In Scenario B, the dominant cost is domain modeling — understanding the business domain well enough to express it as .ddd declarations. The generator then amplifies this understanding into production code.

The generator amplification rate depends on the complexity of the domain model. Consumer repos track their own amplification metrics. The value proposition: domain expertise, not programming labor, drives the output.

6. Methodology

This document is auto-generated during the release ceremony by workflow/scripts/generate-effort-estimation.sh. Metrics are derived from live codebase counts (find + wc -l). Effort estimates use the following model:

  • Manual C++ productivity: 200 lines/day (industry standard for reviewed, tested C++)
  • Domain modeling productivity: ~500–1,000 lines/week for .ddd (depends on domain familiarity)
  • Platform development: Based on actual git history (first commit to current release)
  • Calendar months → person-months: Moonlighting at ~10–15 hrs/week ≈ 0.3 FTE

Effort estimates are conservative lower bounds for Scenario A and empirical for Scenario C.
