4. Per-Line Productivity Context

These estimates use ~200 lines/day for Scenario A (manual C++ with tests). This rate assumes:

  • Production-quality code with proper error handling
  • Unit and integration tests (90%+ coverage)
  • Code review, documentation
  • Build system integration (CMake, Conan)
  • Cross-platform considerations (Qt Widgets + QML)

For generated code (Scenarios B/C), the rate is irrelevant — the generator produces all output in seconds.

5. Key Insight: Domain Modeling is the Bottleneck

In Scenario B, the dominant cost is domain modeling — understanding the business domain well enough to express it as .ddd declarations. The generator then amplifies this understanding by 95× into production code.

The largest project (geosciences, 9607 lines of .ddd) produces 709197 lines of output. The domain expertise required to write those 9607 lines is the real investment — not programming skill.

This is the value proposition: domain expertise, not programming labor, drives the output.

6. Methodology

This document is auto-generated during the release ceremony by workflow/scripts/generate-effort-estimation.sh. Metrics are derived from live codebase counts (find + wc -l). Effort estimates use the following model:

  • Manual C++ productivity: 200 lines/day (industry standard for reviewed, tested C++)
  • Domain modeling productivity: ~500–1,000 lines/week for .ddd (depends on domain familiarity)
  • Platform development: Based on actual git history (first commit to current release)
  • Calendar months → person-months: Moonlighting at ~10–15 hrs/week ≈ 0.3 FTE

Effort estimates are conservative lower bounds for Scenario A and empirical for Scenario C.
