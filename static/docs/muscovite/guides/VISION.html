Muscovite Vision Document

Muscovite Vision Document

A fully declarative, DDD-first full-stack code generation platform

Executive Summary

Muscovite is a model-driven Domain-Driven Design framework that transforms declarative domain models (.ddd files) into a complete, production-ready full-stack application. While the system currently prioritizes a high-performance C++ and PostgreSQL stack, these targets are treated as implementation details optimized for a specific set of requirements: battle-proven stability, extreme runtime efficiency, and maximum climate/CO2 efficiency.

By minimizing the computational overhead of the generated system, Muscovite ensures that enterprise-scale logic runs with the smallest possible carbon footprint.

  • PostgreSQL schemas with Sqitch migrations, constraints, and seed data
  • gRPC/Protocol Buffer services providing strongly typed, high-efficiency APIs
  • Qt Widgets & QML frontends driven by shared domain logic
  • C++ Backend optimized for performance-per-watt and long-term maintainability
  • Security Infrastructure with authentication, authorization, and audit logging
  • AI-Native Workflow using documentation and ADRs as persistent context for autonomous agents
  • Schema Evolution with UUID-based semantic migrations
  • Legacy Modernization via a formal grammar-based ingestion pipeline

This document captures the architectural vision and serves as the north star for Muscovite development.

The Domain-Driven Design Language

Muscovite's DSL captures strategic and tactical DDD patterns in a readable, declarative syntax.

Bounded Contexts

Organize your domain into cohesive modules with explicit boundaries:

CONTEXT Patient @uuid "a1b2c3d4-..." {
    -- Everything about patient identity and demographics
}

CONTEXT Observation @uuid "e5f6a7b8-..." {
    -- Clinical observations and measurements
}

CONTEXT Encounter @uuid "c9d0e1f2-..." {
    -- Healthcare visits and episodes of care
}

Each context becomes a modular shared library (OSLPC — One Shared Library Per Context), allowing for granular build targets and link-time isolation:

  • A PostgreSQL schema (patient, observation, encounter)
  • A gRPC service library (PatientService, ObservationService, EncounterService)
  • A C++ DBA library (patient_dba, observation_dba, encounter_dba)
  • A Qt widget library (patient-qt-widgets, observation-qt-widgets)

The OSLPC pattern means that changing one bounded context only triggers recompilation of that context's library and its direct dependents — not the entire project. For large domain models (26+ contexts), this drastically reduces incremental build times and build-cycle energy consumption.

Entities with Rich Field Types

Define entities with type-safe fields, constraints, and documentation:

ENTITY Patient @uuid "..." IDENTIFIED_BY id (

Efficiency as a First-Class Citizen

Climate-Efficient Computation

The choice of C++ as the primary backend target is a strategic decision for climate efficiency. In an era of massive cloud computing costs and CO2 emissions, Muscovite generates code that requires fewer CPU cycles and less memory than traditional managed-runtime stacks. By compiling down to native machine code, the generated systems achieve higher "work-per-watt" ratios, directly reducing the CO2 footprint of the final product.

This is not premature optimization — it is a fundamental architectural commitment. When a domain model generates 300,000+ lines of C++ across 26 bounded contexts, the efficiency gains compound: lower memory pressure means fewer servers, faster request processing means lower energy per transaction, and native compilation means no JIT warmup or GC pause overhead.

The War on Build Times

The flip side of C++ efficiency at runtime is compilation cost at development time. To maintain developer and AI agent velocity, the build system (Layer 2) is aggressively optimized:

Technique Effect
Precompiled Headers (PCH) Eliminate redundant parsing of heavy libraries (Qt, gRPC, Boost)
Unity (Jumbo) Builds Batch entity compilation within each context, reducing process overhead
Mold Linker High-parallelism linking (auto-detected, 6× faster than ld)
ccache Compiler cache for unchanged translation units
OSLPC Isolation Per-context libraries limit recompilation blast radius
Parallelism Directive (Cores − 2) Prevent VM memory thrashing and "swap death spirals" on constrained machines

With these optimizations active, a clean build of the FHIR Health project (157K lines generated C++) completes in under 30 seconds. Incremental builds after single-context changes take under 10 seconds.

AI Persistence & ADR Governance

Muscovite is built to be developed by AI, for AI.

Documentation as External Context

Because LLM context windows are volatile, the project maintains over 141 Architecture Decision Records (ADRs). These serve as a persistent "transactional log of architectural intent":

  • ADRs are not just history — they are context persistence for AI agents
  • AI agents are required to ingest relevant ADRs before any generation or code modification
  • This prevents "architectural drift" and ensures the 95% AI-generated codebase remains strictly consistent
  • The .github/copilot-instructions.md file cross-references critical ADRs so every agent session starts with the right constraints

Concurrent Agent Coordination

As AI-first development matures, sequential single-agent workflows become a bottleneck. Muscovite supports concurrent agent sessions — a planning agent analyzing the codebase, an implementation agent executing the current iteration, and a hotfix agent resolving release issues, all operating simultaneously.

Coordination evolved from file-based claims and git worktrees (ADR 20260306) to a NATS-based coordinator daemon (v1.62.1) — a Python state machine that manages FTR assignment, quality gate enforcement, sync barriers, and merge queuing via JetStream messaging. The coordinator's state machine is formally verified with TLA+ (ADR 20260311): 8 safety invariants checked across 417,965 states with zero violations, proving properties like exclusive resource locking, one-worker-one-FTR assignment, and merge queue single-threading that are invisible to conventional testing. This is one of the few AI-agent coordination systems with model-checked correctness guarantees.

Grammar-First Evolution

Every syntax change starts with a PEG grammar edit (ADR 20260211). This ensures that the DSL remains a formal language, providing a deterministic constraint for agents and preventing the "vibe-coding" that leads to legacy debt.

The governance chain:

PEG Grammar (spec) → Parser (impl) → Grammar Conformance Test (parity check)
       ↓                    ↓                        ↓
  Syntax rules      IR struct instances      Keyword drift detection
       ↓                    ↓                        ↓
  doc/DSL.md (auto)   Generators (emit)       doc/CONTEXT.md (metrics)

The grammar conformance test (test-grammar-conformance.cpp/.hpp) enforces 98%+ keyword parity between grammar and parser, with zero opaque stub rules permitted.

STRONG_TYPEs (PostgreSQL DOMAINs)

Define validated data types with CHECK constraints enforced at the database level:

CONTEXT Basedata @uuid "..." {
    STRONG_TYPE Email BASED_ON TEXT
        CHECK email_format: VALUE ~ '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$'
        DOC "Email address with format validation";

    STRONG_TYPE Phone BASED_ON TEXT
        CHECK phone_format: VALUE ~ '^\+?[0-9\s\-\(\)]{7,20}$'
        DOC "Phone number with format validation";

    STRONG_TYPE PositiveDecimal BASED_ON DECIMAL(18,4)
        CHECK positive: VALUE > 0
        DOC "Positive decimal number";

    STRONG_TYPE Latitude BASED_ON DECIMAL(9,6)
        CHECK valid_range: VALUE BETWEEN -90 AND 90
        DOC "Geographic latitude";

    STRONG_TYPE Longitude BASED_ON DECIMAL(10,6)
        CHECK valid_range: VALUE BETWEEN -180 AND 180
        DOC "Geographic longitude";
}

Generated PostgreSQL:

CREATE DOMAIN basedata.email AS TEXT
    CONSTRAINT email_format CHECK (VALUE ~ '^[A-Za-z0-9._%+-]+@[A-Za-z0-9.-]+\.[A-Za-z]{2,}$');
COMMENT ON DOMAIN basedata.email IS 'Email address with format validation';

CREATE DOMAIN basedata.phone AS TEXT
    CONSTRAINT phone_format CHECK (VALUE ~ '^\+?[0-9\s\-\(\)]{7,20}$');

Field Constraint Annotations

Declare validation constraints directly on entity fields using @ annotations:

ENTITY Observation
    @uuid("...")
{
    temperature   Decimal   @min(-273.15) @max(1000.0);
    recorded_at   DateTime  @not_in_future;
    label         String    @not_empty @max(200);
    icd_code      String    @pattern("[A-Z][0-9]{2}\\.[0-9]{1,2}");
    start_date    Date;
    end_date      Date      @after(start_date);
};

Constraints generate enforcement code across all layers:

Annotation PostgreSQL C++ Strong Type Qt Widgets QML
@min(N) / @max(N) CHECK (col >= N) BoundedInteger<N,M> QSpinBox::setMinimum IntValidator{bottom:N}
@not_empty CHECK (length(col) > 0) BoundedString<1,unbounded> Input validator Non-empty check
@not_in_future — ConstrainedDateTime<PastOrPresent> setMaximumDate(today) Date validator
@pattern("re") CHECK (col ~ 're') PatternString<"re"> QRegularExpressionValidator RegularExpressionValidator
@after(field) — EntityValidator rule Cross-field validation ViewModel::validate()

Three-Tier Validation Architecture

Field constraints feed a three-tier validation pipeline orchestrated by DomainService:

Tier What Lifecycle Example
1. Strong Types Per-field construction validation Regenerated every run BoundedInteger<0,200>::try_from(val)
2. EntityValidator Cross-field invariants Regenerated every run end_date > start_date
3. PluginValidator User business rules Sidecar (user-owned) External service calls, authorization

All three tiers accumulate errors via ValidationErrors — clients receive all failures in one response. The DomainService sits above CommandService/QueryService, orchestrating validation before persistence. gRPC handlers become thin controllers: deserialize → DomainService → serialize.

Behavioral Sovereignty (v1.60.0–v1.62.0)

Muscovite's third sovereignty tier — moving business logic from manual C++ stubs into the DSL.

The problem: COMMANDs currently declare what changes state (parameters, events) but not how the decision is made. Actual business logic lives in sidecar stubs or DomainService implementations that Muscovite cannot parse, validate, test, or document.

The solution: Evolve COMMAND with a pure functional DECIDE block:

EFFECT SendDischargeNotification(email: String, reason: String);

COMMAND DischargePatient ON Patient(discharge_date: Date, reason: String)
    REQUIRES active_admission: patient.status = 'admitted'
    REQUIRES valid_date: discharge_date <= NOW()
    DECIDE {
        LET final_date = IF discharge_date IS NULL THEN NOW() ELSE discharge_date;
        EMIT PatientDischarged(final_date, reason);
        PLAN SendDischargeNotification(patient.email, reason);
    }
    ENSURES patient_discharged: patient.status = 'discharged'
EMITS PatientDischarged;

Key design decisions (7 ADRs, 20260307-*):

Principle Implementation
No CEL — use Muscovite's own expression AST Non-Turing complete, fully walkable by generators
EMIT vs PLAN — separate internal from external EMIT → uow.add_event() (same txn), PLAN → outbox (async)
EFFECT catalog — formal side-effect declarations Parse-time validation of PLAN references
ENSURES — postconditions for safety + testing Transaction rollback on failure, property-based test generation
DomainService host — not a separate "pure class" DECIDE compiles into decide_* const methods

Expression language tiers (each independently useful):

Tier Version Capability Example
1 — Collection Predicates v1.60.0 NOT IN, IS EMPTY, INTERSECTS, SUBSET OF REQUIRES: items IS NOT EMPTY
2 — Computation v1.61.0 IF/THEN/ELSE, LET/IN, MATCH/CASE LET discount = IF pax >= 10 THEN 0.15 ELSE 0.00
3 — List Comprehensions v1.62.0 MAP, FILTER, REDUCE, SUM, COUNT LET total = SUM(MAP(items, i -> i.price * i.qty))

Generated outputs from DECIDE blocks:

Target What's generated
C++ DomainService decide_* const method (pure function) + orchestration
PostgreSQL Outbox table schema, typed effect payloads
Docs/Mermaid Decision flow diagrams, Business Logic chapter
Property tests Fuzz inputs → verify ENSURES postconditions hold

This transforms Muscovite from an anemic domain model generator (structure only) into a rich domain model generator (structure + behavior).

Specifications (Reusable Query Predicates)

SPECIFICATION ActivePatient FOR Patient

SPECIFICATION PatientsByGender FOR Patient (gender_code TEXT)

SPECIFICATION OverdueAppointments FOR Appointment (threshold_days INTEGER)

Arc Constraints (N-of-M Field Validation)

Enforce that exactly/at least/at most N of M fields are populated—essential for polymorphic types:

-- FHIR Observation: Exactly 1 of 4 value types must be set
ENTITY Observation @uuid "..." IDENTIFIED_BY id (
    id              UUID NOT_NULL,
    value_quantity  DECIMAL,
    value_string    TEXT,
    value_boolean   BOOLEAN,
    value_date      DATE
)
CHECK value_arc @uuid "...":
    ((value_quantity IS NOT NULL)::int + 
     (value_string IS NOT NULL)::int + 
     (value_boolean IS NOT NULL)::int + 
     (value_date IS NOT NULL)::int) = 1
DOC "Clinical observation with polymorphic value";

Generated PostgreSQL:

CONSTRAINT chk_observation_value_arc CHECK (
    ((value_quantity IS NOT NULL)::INT + 
     (value_string IS NOT NULL)::INT + 
     (value_boolean IS NOT NULL)::INT + 
     (value_date IS NOT NULL)::INT) = 1
)

PRESENTATION Hints (Qt Display Customization)

Control how entities appear in the Qt client:

ENTITY Patient @uuid "..." IDENTIFIED_BY id (

    SHORT: "{family_name}, {given_name}";
    FULL: "{family_name}, {given_name} (MRN: {mrn})";

Schema Evolution: Safe, Semantic Migrations

Muscovite tracks every DSL element by UUID, enabling semantic diff between versions:

# Compare git versions and generate migration
racket migrate.cpp/.hpp --from v1.0.0 --to HEAD --output migrations/

The migration system detects:

Change Type Example Generated SQL
Add field New middle_name on Patient ALTER TABLE ADD COLUMN
Rename field surname → family_name ALTER TABLE RENAME COLUMN
Add entity New Appointment entity CREATE TABLE + API functions
Remove entity Delete LegacyCode DROP TABLE (with confirmation)
Change type INTEGER → BIGINT ALTER COLUMN TYPE
Add relationship New FK to Organization ALTER TABLE ADD CONSTRAINT

UUIDs Enable Rename Detection:

-- v1.0.0
ENTITY Patient (
    surname TEXT NOT_NULL @uuid "abc123..."
)

-- v2.0.0 (same UUID, different name = RENAME)

    family_name TEXT NOT_NULL @uuid "abc123..."

-- revert/v2.0.0-rename-patient-surname.sql  
ALTER TABLE patient.patient RENAME COLUMN family_name TO surname;

Viability Assessment

This vision is technically feasible and largely implemented:

Strengths:

  • Single source of truth for all layers
  • Strong alignment with DDD tactical patterns
  • Type safety propagated across DB, API, and UI
  • Semantic migrations preserve domain intent

Challenges:

  • Complex business logic beyond CRUD still requires manual coding
  • UI customization may need developer extensions
  • Large models require modular generation strategies

Bottom line: Muscovite represents the transition from "writing code" to "governing intent." By treating targets as implementation details, the system remains future-proof. Whether the target is C++, Rust, or a future WASM-based runtime, the domain asset remains preserved in the .ddd model.

Muscovite: One model. Consistent stack. Maximum efficiency. Zero drift.

Strategic Context

Origin and Purpose

Muscovite is the distillation of 25 years of consulting experience in enterprise systems architecture, DDD, and multi-language stacks. The core thesis: a declarative domain model should be the single source of truth for an entire application stack, and AI-assisted development now makes building such a generator economically viable for a single engineer.

The project currently serves as a proof of concept validating that thesis. It is not (yet) a product, a framework for teams, or a commercial offering. It is a spike exploring breadth—8 generator targets, 2 complex domain examples, event sourcing, CQRS, spatial capabilities—to stress-test whether the abstraction holds under real-world complexity.

Target Independence via IR

The current generator targets (PostgreSQL, C++, gRPC, Qt) are just one materialization of the intermediate representation. Because the DSL compiles to a well-defined semantic IR (with UUID-tracked elements, typed fields, explicit relationships, and aggregate boundaries), the outputs are arbitrarily retargetable:

Current Target Equivalent Alternative
PostgreSQL Oracle, SQL Server, CockroachDB
C++ / pqxx .NET / EF Core, Java / Spring, Rust / SQLx
gRPC / Protobuf REST / OpenAPI, GraphQL
Qt Widgets / QML React, Blazor
Sqitch migrations Flyway, Liquibase, EF Migrations

Adding a new target is a generator-writing exercise (~1,000–5,000 lines of Racket per target), not an architecture change. The IR is the asset; the generators are replaceable.

Formal Grammar Governance (Achieved)

The DSL is formally specified by a 208-rule PEG grammar (grammar/muscovite.peg) that serves three roles simultaneously:

  1. Constraint for AI generation — LLMs follow the formal grammar to produce syntactically valid .ddd files, enabling automated domain extraction from brownfield systems.
  2. Specification for parser evolution — Grammar changes drive parser changes (not the reverse), ensuring the language evolves intentionally. Parser changes without a preceding grammar edit are forbidden (ADR 20260211).
  3. Living documentation — The grammar generates doc/DSL.md and doc/PROJECT-DSL.md automatically, eliminating the gap between specification and implementation.

A grammar conformance test enforces 98%+ keyword parity between grammar and parser, with zero opaque stub rules. This is the foundation that prevents "vibe-coding" — every syntax change is intentional, traceable, and deterministic.

Legacy Modernization Pipeline

The long-term vision positions Muscovite as a building block for AI-assisted legacy system modernization — turning "vibe-coded" systems held together by implicit knowledge into formally modeled, climate-efficient applications.

The pipeline has four stages:

  1. Grasping — Multi-source extraction: analyzing ORM model structs, service layer functions, UI templates, API specs, and notification systems to extract structural AND behavioral domain knowledge with confidence scoring
  2. Transformation — Emitting .ddd files directly from extractors, constrained by the PEG grammar, with confidence annotations guiding human review
  3. Generation — Materializing a clean, high-performance, climate-efficient C++ backend and modern frontends
  4. Data Migration — Generating Sqitch-based mapping layers to move legacy data into the new, validated UUID-based schema
Brownfield System
    ├── ORM model structs (entities, fields, relationships, enums)
    ├── Service layer (commands, events, state machines, workflows)
    ├── UI templates (domain-visible fields, user actions, dropdowns)
    ├── API specs / Swagger (endpoints, request/response schemas)
    └── Notification system (event types, publishers)
            ↓
    Multi-Source Extractor (5-pass: schema → behavioral → UI/API → triangulate → emit)
            ↓
    Confidence-Annotated .ddd Files (direct output, PEG-validated)
      ├── ≥0.8: Full construct (high confidence)
      ├── 0.4–0.8: REVIEW marker (needs human verification)
      └── <0.4: TODO comment (needs human authoring)
            ↓
    Human Review (domain expert + consultant curates .ddd)
            ↓
    Muscovite IR → New Stack + Data Migration

What multi-source extraction discovers (three confidence tiers):

Tier 1 — High confidence (signatures, always extractable):

  • ORM model structs → Entities with typed fields, relationships, constraints
  • Create*/Update*/Delete* service functions → COMMAND declarations
  • Notification types and *Event structs → DOMAIN_EVENT declarations
  • Go interface definitions → EXTENSION_POINT / plugin contracts
  • Status const/iota blocks → STATE_MACHINE states
  • SearchOptions filter structs → SPECIFICATION candidates

Tier 2 — Medium confidence (structure + heuristics):

  • Methods mutating status fields → STATE_MACHINE transitions
  • Event handler registrations → POLICY declarations
  • Multi-step service functions with rollback → SAGA candidates

Tier 3 — Low confidence (deep analysis, needs human review):

  • Conditional branches in command handlers → DECIDE block structure
  • Cross-package external calls → EFFECT catalog candidates
  • defer blocks and error-path cleanup → Compensation patterns

What requires human judgment (domain expertise):

  • Aggregate boundaries (which entities form consistency units)
  • Bounded context partitioning (where to draw service boundaries)
  • Ubiquitous language alignment (naming that reflects domain intent)
  • Value object vs. entity distinction (identity semantics)
  • Which business rules are essential vs. accidental
  • Full DECIDE block logic (domain semantics, not just branch structure)

The .ddd file sits at exactly the right abstraction level to be the review artifact between multi-source extraction and human validation. It is small enough to read (the FHIR example is ~10K lines for 9 bounded contexts), semantically rich enough to capture domain intent, and machine-parseable enough to generate from.

Brownfield Extraction: Current Capabilities

The brownfield extraction pipeline is no longer purely theoretical. Muscovite v1.31+ includes concrete DSL and generator support, and v1.64.0 adds a complete multi-source extraction toolchain:

MAPPING blocks (FTR-080, ADR 20260224-002) — A dedicated DSL construct for declaring field-level transformations between legacy schema structures and Muscovite target entities:

MAPPING LegacyCustomer @uuid "..." {
    FROM "legacy_customers" INTO Customer;
    FIELD customer_id    -> id           COPY_UUID;
    FIELD fname          -> given_name   @required COPY;
    FIELD lname          -> family_name  @required COPY;
    FIELD customer_email -> email        @unique VALIDATE_EMAIL;
    FIELD birth_dt       -> birth_date   COERCE_DATE format="MM/dd/yyyy";
    FIELD is_active      -> active       DEFAULT true COPY_BOOL;
    FIELD created_ts     -> created_at   COERCE_TIMESTAMP;
    DOC "Maps legacy customer table to Patient entity";
}

Each MAPPING block generates type-safe extractor and transformer components, bridging the gap between "AI extracted a candidate model" and "we can actually move the data."

Multi-source Go extractor (muscovite-go-extractor, ADR 20260310) — A generic Go application analyzer that performs five extraction passes (schema → behavioral → UI/API → triangulation → emit) and outputs confidence-annotated .ddd files directly. Key features:

  • Analyzes ORM models, service functions, Go templates, and Swagger specs
  • Extracts structural (entities, relationships) AND behavioral (commands, events, state machines) constructs
  • Confidence scoring via multi-source triangulation (evidence from schema + code + UI + API)
  • Dual conformance validation: Go-side PEG validator + Racket parse-back gate
  • System-agnostic: works on any Go/XORM application (Gitea, Gogs, Drone, etc.)
  • Integrated as git subtree at tools/go-extractor/

Gitea brownfield test (examples/brownfield-gitea-test/) — A comprehensive proof-of-concept extracting the full Gitea v1.21.0 schema (107 tables, 973 columns, 12 bounded contexts, zero foreign keys) into a Muscovite domain model. Demonstrates the complete flow: Go source analysis → multi-source extraction → .ddd generation → human curation → full stack generation with migration artifacts.

Near-term brownfield goals:

  1. Complete Gitea extraction — 107 tables across 12 contexts with behavioral constructs (commands, events, state machines, specifications)
  2. FHIR extraction tool — Python script parsing FHIR R4 StructureDefinition JSON → .ddd files directly
  3. MAPPING validation — Generator-side enforcement that every source field in a MAPPING block corresponds to a real column in the legacy schema (via schema introspection)
  4. Data migration Sqitch scripts — Generate Sqitch change sets that INSERT INTO ... SELECT from legacy tables through the MAPPING transformations
  5. Future extractors — Python (Django/SQLAlchemy), Java (JPA/Hibernate), Ruby (ActiveRecord) — all outputting .ddd directly using the same multi-source triangulation pattern

Potential Future Directions

Personal force-multiplier: Continue as a consultant toolkit—rapid prototyping of domain models, generation of working full-stack spikes, schema evolution management for client engagements.

Small team methodology: Formalize the brownfield extraction pipeline into a repeatable consulting engagement: extract → curate → generate → iterate. Multi-source extraction (schema + service layer + UI templates + API specs) produces confidence-annotated .ddd stubs; domain experts review and enrich them. The .ddd files become the shared language between technical and domain stakeholders.

Additional generator targets: A stub generator for a second backend stack (e.g., TypeScript/Express, Kotlin/Spring) would concretely demonstrate target independence with ~500–1,000 lines of Racket.

Architect's Intent — Why This Order, What I'd Cut

Added March 2026, extracted from structured architect interview.

The project's nature

Muscovite is a labbook — documenting experiments, observations, conclusions, successes, and failures. It is not a product roadmap driven by market demand. Features are chosen for their learning value: does this experiment stress-test whether the abstraction holds?

Commercialization is not in focus. "Depends on who pays me first."

Spike exit criteria

Two concrete, testable milestones define when the spike has succeeded:

  1. Complex domain front-to-back with TLA+-verified states — proving the DSL can express full behavioral sovereignty with formal verification of state machines and decision logic.

  2. Brownfield reversal of a Gitea-scale project — proving the extraction pipeline can analyze a real-world legacy system (≥107 tables, 12 contexts), generate .ddd models, and produce a working replacement stack. Sidecar implementations may be left as an exercise to the reader.

When both are achieved, productization can begin.

Roadmap ordering rationale

The roadmap follows a dependency chain, not a priority list:

Foundation (types, constraints, relationships)                    → DONE
Domain services (aggregate boundaries, CQRS)                      → DONE
Behavioral sovereignty (DECIDE/ENSURES, collection predicates)    → Tiers 1-2 DONE
List comprehensions + property-based testing                      → PLANNED (Tier 3-4)
Brownfield extraction (legacy modernization)                      → IN PROGRESS
Second backend target (retargetability proof)                     → NOT STARTED
Productization                                                    → NOT STARTED

Each layer depends on the one below. You can't extract brownfield business rules without a DSL expressive enough to represent them. You can't prove retargetability without a mature IR. You can't productize without proving the abstraction works at scale.

Second backend sequence

If building additional backend targets: Rust → Go → Java → C#. In this sequence. The order reflects philosophical alignment (performance, type safety, longevity) over market size. TypeScript is notably absent — the performance thesis doesn't accommodate managed runtimes as primary targets.

What I'd cut

Features are cut when they don't stress-test the abstraction, not when they don't generate revenue. Four cut patterns:

  1. Time constraint — Moved via move-feature.sh to next iteration.
  2. Dependency failure — Prerequisite not ready. Defer.
  3. Indefinite deferral — Too complex or low-ROI for current phase.
  4. Design direction failure — The approach doesn't work. Revert entirely. Design is fluid and adaptable. No attachment to failed approaches.

What NEVER causes a cut: market pressure, competitive features, sunk cost.

Velocity target

The deceleration from 8-10 FTRs to 2-4 FTRs per release was involuntary — the price paid for the NATS coordinator experiment. 8-10 FTRs per release is the recovery objective.

Autonomous agents

Still absolutely a goal. Blocked by tooling ecosystem maturity — wrapping Mistral SDK, GitHub Copilot SDK, or similar as autonomous agents is currently cumbersome. Will resume when better tools become available. The NATS failure was a rejection of the approach, not the goal.

Operating model

Single architect amplified by AI agents. The architect's irreducible function is direction — which experiments to run, which results to keep, which to discard. AI agents handle execution. Documentation is sufficient for execution; direction requires human judgment.

The biggest agent failure mode: optimizing for simplicity when the architect optimizes for correctness. "I want correct, not simple."
