Muscovite Ground Truth Reference

Muscovite Ground Truth Reference

Purpose: Canonical reference for AI agents to prevent hallucinations. Documents the three-layer architecture, exact Racket structures, naming conventions, and type mappings.

For the principles and red lines behind these conventions, see ARCHITECTURAL-PHILOSOPHY.md. For trade-off guidance when facing ambiguous choices, see DECISION-HEURISTICS.md.

0. Three-Layer Architecture (Non-Negotiable)

Muscovite enforces a strict three-layer hierarchy. Every change must respect this separation.

Layer Name Location Purpose
Layer 0 The Brain (Generator/IR) toolchain/muscomp/ (Racket) Defines the DSL IR, owns all templates and code generation
Layer 1 The Framework framework/ (C++ base libraries) Reusable base infrastructure (Repository, UnitOfWork, errors, connections, messaging). No domain logic.
Layer 2 Generated Applications examples/*/generated/ 100% generated from .ddd models. Never manually edited.

Lockstep Development (Mandatory)

When adding a DSL feature (Layer 0), you must:

  1. Update Layer 1 framework if new base behavior is needed
  2. Update Layer 0 templates to generate Layer 2 code that uses it
  3. Regenerate samples (Layer 2)

Layer Mapping: IR → Framework → Generated Code

IR Struct Layer 1 Base Layer 2 Generated
entity Repository base (framework/dba/.../repository.hpp) Entity struct (e.g., patient.hpp)
aggregate UnitOfWork (framework/dba/.../unit_of_work.hpp), ValidationErrors (framework/validation/.../validation_errors.hpp) *CommandService / *QueryService / *DomainService classes
repository RepositoryBase *CommandRepository / *QueryRepository
domain-event DomainEvent struct in UnitOfWork, Publisher (framework/messaging/.../publisher.hpp) domain_events table + emission in services
sourced-event (no framework base) Generated EventStore per aggregate
state-machine (no framework base) *StateMachine C++ class, state lookup tables, transition log, gRPC RPCs, Qt state-aware UI
entity + display-format (no framework base) QML ViewModels with Q_PROPERTY

Accessor/Mutator Patterns (gRPC)

Reading from proto:

(define (type->proto-accessor-pattern type-sym)
  (case type-sym
    [(String Integer Boolean) 'direct]          ; req.field_name()

V. Domain Services as Aggregate Boundary (ADR 20260120)

MANDATORY Architecture

All aggregate access through domain services. No stored procedures.

gRPC → DomainService → [Tier1: try_from] → [Tier2: EntityValidator] → [Tier3: PluginValidator] → CommandService → CommandRepository (pqxx) → PostgreSQL
                                                                                                       └── emit event to domain_events table
gRPC → DomainService → QueryService → QueryRepository (pqxx) → PostgreSQL  (queries bypass validation)

Generated Services

VI. CQRS Event Emission (ADR 20260120)

Every aggregate gets a {schema}.domain_events table (id, aggregate_type, aggregate_id, event_type, payload TEXT, occurred_at, version). Events emitted inline in CommandService within the same transaction. The payload column stores pre-serialized JSON from protobuf MessageToJsonString() (ADR 20260306).

VII. Event Sourcing (ADR 20260204)

Opt-in per aggregate with EVENT_SOURCED. Typed event tables (no JSON), per-event payload tables, snapshot tables. Dual-write strategy: single transaction appends typed event + updates snapshot + updates entity table. No replay needed (snapshot = current state).

DSL: SOURCED_EVENT Name @uuid "..." (fields) APPLIES_TO AggregateRoot

X. INSTRUCT Blocks (MEP - Muscovite Embedded Prompts)

Platform-Agnostic AI Guidance

Per ADR 20260205: INSTRUCT blocks embed domain-specific AI prompts in the DSL.

(struct instruct-spec
  (uuid      ; String: stable UUID
   name      ; Symbol: instruction name
   intent    ; String: natural language intent
   inputs    ; List of string: inputs
   output    ; Symbol or #f: output type/code
   strategy  ; Symbol: 'deterministic, etc.
   doc)      ; String or #f
  #:transparent)

DSL Syntax (Proposed)

ENTITY Patient (...) {
    INSTRUCT calculate_bmi @uuid "..." (
        "Calculate Body Mass Index from height and weight"
        INPUTS: height_cm, weight_kg
        OUTPUT: Decimal
        STRATEGY: deterministic
    )
    DOC "Derived metric for clinical assessment";
}

Multi-Target Generation

INSTRUCT blocks generate:

  1. C++ method stubs with intent comment
  2. SQL CHECK constraints (if deterministic)
  3. QML validators for client-side logic

Key constraint: Must be platform-agnostic (no C++-specific or SQL-specific logic in DSL).

XII. cpp-peglib AST Patterns (muscomp)

muscomp uses cpp-peglib to parse PEG grammars. Understanding how the AST maps to PEG constructs is essential for writing correct ir_builder.cpp code.

Key AST Properties

Property Type Meaning
name std::string Grammar rule name (e.g. "BaseType", "EntityClause")
token std::string_view Matched text — empty for non-terminal nodes (is_token == false)
nodes vector<shared_ptr<Ast>> Child AST nodes
choice unsigned int 0-indexed index of the matched alternative in a PEG prioritized choice (/)
is_token bool true for terminal tokens, false for non-terminal rules

PEG Literal Keywords Create No Child Nodes

When a PEG rule matches a literal keyword (e.g. 'DOMAIN', 'UUID'), cpp-peglib does not create a child AST node for it. The matched literal is consumed but invisible in the AST. This means you cannot check child->name == "DOMAIN" — there is no such child.

Pattern 1: Prioritized Choice → ast.choice with Lookup Array

For PEG rules using the / (prioritized choice) operator, use peg::Ast::choice to determine which alternative matched. The idiomatic pattern is a static constexpr array indexed by the choice value:

// PEG grammar:
//   BaseType <- 'UUID' / 'STRING' / 'VARCHAR' / 'TEXT' / 'INTEGER' / ...
//
// Each literal creates NO child node and NO token — only choice is set.

static constexpr std::array<const char*, 22> base_type_names = {
    "UUID", "STRING", "VARCHAR", "TEXT",          // choices 0–3
    "INTEGER", "INT", "BIGINT", "SMALLINT",       // choices 4–7
    "DECIMAL", "FLOAT", "BOOLEAN", "BOOL",        // choices 8–11
    "DATE", "DATETIME", "TIMESTAMPTZ", "TIMESTAMP" // choices 12–15
    // ...
};
if (node.choice < base_type_names.size()) {
    ts.name = base_type_names[node.choice];
}

See: ir_builder.cpp — build_type_spec() (~line 700)

Pattern 2: Choice with switch for Different Behavior

When each alternative requires different handling, use switch (ast.choice):

// PEG grammar:
//   PolicyAction <- 'EMIT' IDENTIFIER ActionParams?      (choice 0)
//                 / 'INVOKE' IDENTIFIER ActionParams?     (choice 1)
//                 / 'NOTIFY' IDENTIFIER ActionParams?     (choice 2)
//                 / IDENTIFIER ('.' IDENTIFIER)? ActionParams?  (choice 3)

switch (ast.choice) {
    case 0: action.type = "emit-event"; break;
    case 1: action.type = "invoke-command"; break;
    case 2: action.type = "notify"; break;
    default: action.type = "call-service"; break;
}

See: ir_builder.cpp — build_policy_action() (~line 1572)

Pattern 3: Choice with Keyword + Payload

When each alternative starts with a literal keyword followed by a value, the keyword is invisible but the payload nodes remain as children:

// PEG grammar:
//   GroundingField <- 'DOMAIN' STRING                  (choice 0)
//                   / 'KEY_CONCEPTS' StringList         (choice 1)
//                   / 'DIALECT' IDENTIFIER              (choice 2)
//                   / ...

switch (field->choice) {
case 0: // DOMAIN STRING — keyword consumed, STRING child remains
    for (const auto& child : field->nodes) {
        if (child->name == "STRING") {
            grounding.domain = strip_quotes(sv(child->token));
            break;
        }
    }
    break;
case 1: // KEY_CONCEPTS StringList — keyword consumed, StringList child remains
    // ...
}

See: ir_builder.cpp — build_grounding() (~line 2683)

Pattern 4: Sequences → Positional Child Analysis

For PEG sequences (no / operator), children appear in order. Use child->name to match expected nodes positionally:

// PEG grammar:
//   EntityClause <- RelationshipsBlock / DocClause / PresentationBlock / ...

for (const auto& clause : child->nodes) {
    if (clause->name == "RelationshipsBlock") {
        entity.metadata.relationships = build_relationships_block(*clause);
    } else if (clause->name == "DocClause") {
        entity.doc = extract_doc(*child);
    }
    // ...
}

See: ir_builder.cpp — build_entity() (~line 470)

Common Pitfalls

  1. Don't check token on non-terminals — it will be empty. Use choice or inspect nodes.
  2. Array order must match PEG grammar order — choice == 0 is the first alternative, etc. If the grammar is reordered, update the array.
  3. PEG is greedy — the first matching alternative wins. Place longer/more-specific alternatives before shorter ones (e.g. 'TIMESTAMPTZ' before 'TIMESTAMP').
  4. Helpers: Use find_first_identifier(), find_child(), find_children(), extract_string() from ir_builder.hpp instead of raw positional indexing when possible.
