End-to-End Testing Strategy

End-to-End Testing Strategy

  • Status: accepted
  • Date: 2026-02-23
  • Tags: testing, e2e, grpc, qt, qml, ctest, cmake, generators
  • Deciders: Johannes Lochmann

Technical Story: Generated applications must be verifiable against a live stack (PostgreSQL + NATS + OpenSearch + gRPC server) through all layers up to and including the Qt ViewModel and QML property bindings. This ADR establishes the three-tier e2e strategy, the CMake orchestration model, the Racket generator responsibilities, and the Racket file-size discipline required.

Context and Problem Statement

ADR 20260212 (doctest + TDD) covers unit and integration tests at the framework level. It does not address full-stack verification of generated applications. Today there are zero tests that:

  • Start a generated gRPC server binary against a live database
  • Call that server via the generated stub client libraries
  • Verify that Qt ViewModel Q_PROPERTY values are populated correctly from a live gRPC response
  • Verify that QML property bindings observe the correct signals

The 19× DSL amplification ratio means one generator defect silently corrupts every entity in every generated project. A broken Q_PROPERTY type mapping or a wrong proto field name is not caught until a human runs the application.

Decision Drivers

  • CTest fixture infrastructure already exists (Testing.cmake FIXTURES_SETUP)
  • Three infrastructure services already in docker-compose.yml (postgres, nats, opensearch)
  • Qt6::Test already linked in tests/CMakeLists.txt
  • ADR 20260223 (generator unit testing) mandates test emitters for all generators
  • Racket file-size discipline: 400 LOC per file is the hard limit; emitter modules must be split by concern, not collapsed into existing generator files

Considered Options

  1. Three-tier generated e2e — CMake orchestration + three generated test tiers driven by Racket emitters, registered in Racket test harness
  2. External test harness (Squish/Robot Framework) — GUI automation tool
  3. Hand-written e2e tests only — no generator involvement

Decision Outcome

Chosen option: Option 1 — Three-tier generated e2e. Options 2 and 3 are incompatible with the generator-first architecture: externally maintained test code drifts from generator output by definition. Only generated tests can track the 19× amplification surface.

Tier 1 — gRPC contract layer (doctest)

Pure C++ doctest SCENARIO/GIVEN/WHEN/THEN tests. Each generated aggregate gets one test file: {context}_e2e_grpc_test.cpp. Calls real gRPC stubs against a live server. Verifies: all CRUD RPCs return status.ok(), list RPCs paginate, FK fields resolve, FAILED_PRECONDITION from a bad state transition.

Requires CTest fixture e2e-grpc (see CMake section). Label: e2e;grpc

Tier 2 — Qt ViewModel layer (QTest + QSignalSpy)

QTEST_GUILESS_MAIN — headless, no display required. Each generated ViewModel gets one test file: {entity}_view_model_e2e_test.cpp. Tests:

  • load() populates all Q_PROPERTY values
  • QSignalSpy(&vm, &{Entity}ViewModel::{field}Changed) fires after load
  • QAbstractItemModelTester passed against the list model (Qt contract)
  • Null optional fields remain at Qt default (empty string / 0 / false)

Requires e2e-grpc fixture. Label: e2e;qt

Tier 3 — QML binding layer (Qt Quick Test, offscreen)

QT_QPA_PLATFORM=offscreen. Each entity gets {Entity}FormTest.qml using TestCase + SignalSpy. Tests: QML property round-trip, model delegate correctness, form submit signal emitted. Headless via cmake -DQTEST_FUNCTION_TIMEOUT=30000.

Label: e2e;qml

CMake Orchestration

Testing.cmake gains muscovite_register_e2e_fixture(NAME ... [REQUIRES ...]):

muscovite_register_e2e_fixture(e2e-infra)
# SETUP:   docker compose -f ${DOCKER_COMPOSE_FILE} up -d --wait
#          sqitch deploy <all registered projects>
#          seed lookup tables
# CLEANUP: docker compose down

muscovite_register_e2e_fixture(e2e-grpc REQUIRES e2e-infra)
# SETUP:   $<TARGET_FILE:grpc-server> &; write PID to tmp/grpc-server.pid
#          poll :50051 until open (max 20s)
# CLEANUP: kill $(cat tmp/grpc-server.pid)

All three test tiers set FIXTURES_REQUIRED "e2e-grpc".

CTest preset e2e runs only -L e2e. Default unit/debug presets exclude it.

Racket Generator Responsibilities

File discipline (HARD LIMIT: 400 LOC per file)

Each e2e test emitter is a separate Racket module — never merged into the existing generator file it accompanies. Naming convention:

Layer Generator file E2E emitter file
gRPC grpc/cpp-service-v2.rkt grpc/e2e-grpc-test.rkt
Qt Widgets ViewModel qt/widgets-views.rkt qt/e2e-viewmodel-test.rkt
QML ViewModel qt/quick-views.rkt qt/e2e-qml-test.rkt

If any emitter approaches 400 LOC, split further by aggregate vs entity vs lookup.

Racket test harness registration (MANDATORY)

Every new emitter module MUST be exercised in dslgen/tests/:

  • A companion test file dslgen/tests/test-{emitter-slug}-e2e-emitter.rkt verifying the emitter output (pattern: generate → inspect string → assert presence of SCENARIO, QSignalSpy, TestCase, etc.)
  • Registered in dslgen/tests/run-all-tests.rkt require list

FORBIDDEN: Emitter modules that are not represented in the Racket test harness.

Entry points

Each emitter exposes a single public function following the existing convention:

;; grpc/e2e-grpc-test.rkt
;; WHY: tier-1 e2e test emitter for a bounded context
;; ADR: 20260223-end-to-end-testing-strategy
(define (generate-grpc-e2e-tests context output-dir) ...)

Called from dslgen/generators/grpc/main.rkt when the --test flag is active.

Positive Consequences

  • Every generated aggregate ships with a full-stack test suite at no manual cost
  • Regression protection scales with the DSL model, not with developer time
  • ctest -L e2e gives a pass/fail verdict on a live generated application
  • Racket test harness covers the emitters themselves (two-level verification)

Negative Consequences

  • CI run time increases significantly for e2e label (infrastructure startup)
  • Requires Docker in CI
  • Racket emitter maintenance: each generator change may require emitter update

Mandatory Racket File-Size Discipline

The 400 LOC limit per Racket file is not a guideline — it is a hard constraint for this feature. Rationale: beyond 400 LOC, test emitters become untestable as units, require-chains grow circular, and the Racket test harness cannot isolate failures. If an emitter naturally grows beyond 400 LOC, split by:

  1. Fixture/helpers module (e2e-grpc-helpers.rkt)
  2. Per-operation modules (e2e-grpc-crud.rkt, e2e-grpc-pagination.rkt)
