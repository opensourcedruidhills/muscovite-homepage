ADR: Use nlohmann_json for JWT Token Serialization

ADR: Use nlohmann_json for JWT Token Serialization

Date: 2025-12-29
Status: Accepted
Deciders: Architecture Team
Affected Components: Security Module (JWT service, gRPC message serialization)

Context

The project uses jwt-cpp 0.7.1 for JWT token generation and validation. This library is header-only and template-based, requiring an explicit JSON traits parameter to specify how tokens are serialized/deserialized.

jwt-cpp supports multiple JSON libraries via traits:

  • picojson (lightweight, bundled) - but Conan disables it by default
  • nlohmann_json (comprehensive, widely-adopted) - requires explicit dependency
  • rapidjson (fast, DOM-based)
  • Custom traits (complex)

The security module needs to:

  1. Serialize claims to JSON (e.g., username, groups, privileges)
  2. Deserialize JWT payloads for validation
  3. Represent token data in application structs

Without an explicit JSON library, jwt-cpp templates fail to instantiate, causing linker errors or missing symbols.

Decision

Use nlohmann_json 3.11.3 as the JSON serialization backend for JWT operations.

This decision applies to:

  • JWT token creation: jwt::create<jwt::traits::nlohmann_json>()
  • JWT token verification: jwt::decode<jwt::traits::nlohmann_json>(token)
  • Token claim serialization: json(value) for strings, json::array() for vectors
  • Token claim deserialization: payload.get<type>() for scalars, as_array() for collections

Rationale

Criterion nlohmann_json picojson rapidjson
Adoption De facto standard (C++17+) Lightweight High-performance
Integration Conan 3.11.3 available Conan disables by default Conan available
Header-only Yes Yes Yes (DOM)
JWT-cpp compatibility Direct ::nlohmann_json trait Not available in Conan build Not tested
Ecosystem Works with gRPC JSON mapping Limited Stream-oriented
Maintenance Active (Niels Lohmann) Unmaintained Active (Tencent)

Selected: nlohmann_json because:

  1. Aligns with modern C++ practices (C++17 structured bindings, get<T>())
  2. Direct trait support in jwt-cpp
  3. Already available via Conan with all dependencies resolved
  4. Comprehensive documentation and examples
  5. Ecosystem integrations (spdlog, etc.)

Consequences

Positive

  • ✅ JWT tokens serialize cleanly: auto payload = json{{"username", principal.username}, {"groups", principal.groups}}
  • ✅ Type-safe claim extraction: const auto username = decoded.get_payload().get<std::string>("username")
  • ✅ No custom serialization code; nlohmann_json handles lists, strings, timestamps
  • ✅ Future-proof: can migrate to nlohmann_json universally if other modules adopt it

Negative

  • ⚠️ Hard dependency on nlohmann_json (3.11.3+)
  • ⚠️ JSON overhead vs. binary formats (not a concern for short-lived auth tokens)
  • ⚠️ If jwt-cpp drops nlohmann_json support, requires traits rewrite

Constraints

  • All JWT operations must use jwt::traits::nlohmann_json explicitly
  • Token claims must be JSON-serializable types (string, int, vector, nested objects)
  • No binary or custom type claims without explicit serialization functions

AI Agent Guidance

When this decision applies:

  • Modifying JWT service: token creation, validation, claim extraction
  • Adding new token claims: ensure they're JSON-serializable
  • Integrating JWT with gRPC: nlohmann_json provides JSON transcoding

How it's enforced:

  • Conan dependency: requires = ("nlohmann_json/3.11.3", ...)
  • CMakeLists.txt: Link target nlohmann_json::nlohmann_json
  • Code review: All jwt::create<>() and jwt::decode<>() must use jwt::traits::nlohmann_json

Key files:

  • conanfile.py - Declare nlohmann_json dependency
  • cmake/Dependencies.cmake - Find nlohmann_json package
  • src/muscframework/server/src/security/jwt_service.cpp - Implementation
  • src/muscframework/server/include/muscovite/server/security/jwt_service.hpp - Header
  • Tests: tests/integration/security/jwt_service_test.cpp

Alternatives Considered

A. Use picojson (rejected)

  • Reason: Conan explicitly disables it; would require custom build
  • Cost: High (fork Conan recipe or bundled in source)

B. Use rapidjson (rejected)

  • Reason: jwt-cpp has no built-in rapidjson trait; would require custom trait implementation
  • Cost: Medium (write custom traits, test compatibility)

C. Implement custom JSON adapter (rejected)

  • Reason: Reinventing the wheel; nlohmann_json already works
  • Cost: Maintenance burden, harder to debug
