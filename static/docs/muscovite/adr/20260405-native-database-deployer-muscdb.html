Native Database Deployer — muscdb

Native Database Deployer — muscdb

  • Status: proposed
  • Deciders: Johannes Lochmann
  • Date: 2026-04-05
  • Tags: postgresql, deployment, migration, muscdb, infrastructure
  • Supersedes: 20251223-deploy-database-via-sqitch-only

Technical Story: Replace Sqitch with a native C++ deployer (muscdb) that uses libpqxx and tracks state in a muscovite schema, eliminating the external Perl dependency.

Context and Problem Statement

Muscovite generates deploy/revert/verify SQL scripts and a sqitch.plan manifest. The actual deployment is delegated to Sqitch — a Perl-based tool that:

  1. Requires a Perl runtime + CPAN dependencies (App::Sqitch, DBD::Pg)
  2. Uses its own sqitch.* schema tables for state tracking
  3. Has its own plan file format (sqitch.plan) with bespoke dependency syntax
  4. Uses db:pg:// URIs instead of standard PostgreSQL postgresql:// URIs
  5. Is an external dependency we don't control

After v2.8.0 (FTR-536/537), the generator layer is already tool-agnostic: files are named pg-*, the core struct is db-change, and a deployer interface exists with Sqitch as a pluggable backend. The SQL generation has zero Sqitch coupling — only the output writer (sqitch-output.rkt, now sqitch_plan.cpp in muscomp) and the manifest format are Sqitch-specific.

Meanwhile, Muscovite already manages infrastructure deployment for NATS messaging, OpenSearch, Docker, and telemetry via DSL-declared configuration blocks. Database deployment is the only infrastructure concern still delegated to an external tool.

Decision Drivers

  • Eliminate external Perl/CPAN dependency (Sqitch is the last Perl dependency)
  • Use standard PostgreSQL connection mechanisms (libpq environment variables)
  • Make deployment state part of the Muscovite-managed database schema
  • Unify infrastructure deployment under the DSL (like MESSAGING, SEARCH, TELEMETRY)
  • Preserve deploy/revert/verify triplet structure (proven pattern)
  • Leverage existing deployer interface (FTR-537) — this is the second backend

Considered Options

  • Keep Sqitch — status quo, external dependency, works but alien to the toolchain
  • Adopt Flyway/Liquibase — Java dependency, even worse than Perl
  • Shell script deployer — fragile, no transaction safety, no state tracking
  • Native C++ deployer (muscdb) — libpqxx, zero external deps, full control

Decision Outcome

Chosen option: "Native C++ deployer (muscdb)", because it eliminates the last external runtime dependency, integrates with the existing libpqxx ecosystem (framework already uses it), and inherits PostgreSQL connection config via standard libpq environment variables (PGHOST, PGPORT, PGUSER, PGPASSWORD, PGDATABASE).

Positive Consequences

  • Zero external runtime dependencies (C++ binary, statically linked)
  • libpq env vars work automatically (PGHOST, PGPORT, etc.)
  • Deployment state is queryable via standard SQL (SELECT * FROM muscovite.deployment_log)
  • Advisory locks prevent concurrent deployments (PostgreSQL-native)
  • Safety policies from the migration engine can be enforced at deploy time
  • One fewer language runtime in CI/CD pipelines
  • muscdb ships alongside muscomp — same build system, same Conan deps

Negative Consequences

  • Must implement and maintain a deployer (Sqitch is battle-tested)
  • Consumer repos using sqitch deploy directly need migration path
  • New CLI to learn (mitigated: subcommands mirror Sqitch's)

Implementation

1. The muscovite Schema

muscdb bootstraps a muscovite schema on first deploy. This replaces Sqitch's sqitch.* tables.

CREATE SCHEMA IF NOT EXISTS muscovite;

CREATE TABLE IF NOT EXISTS muscovite.deployment_log (
    id              BIGINT GENERATED ALWAYS AS IDENTITY PRIMARY KEY,
    change_name     TEXT NOT NULL,
    project_name    TEXT NOT NULL,
    deployed_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
    deployed_by     TEXT NOT NULL DEFAULT current_user,
    deploy_hash     TEXT NOT NULL,
    note            TEXT,
    dependencies    TEXT[] NOT NULL DEFAULT '{}'
);

CREATE UNIQUE INDEX IF NOT EXISTS uq_deployment_change
    ON muscovite.deployment_log (project_name, change_name);

CREATE TABLE IF NOT EXISTS muscovite.deploy_lock (
    project_name    TEXT PRIMARY KEY,
    locked_at       TIMESTAMPTZ NOT NULL DEFAULT now(),
    locked_by       TEXT NOT NULL DEFAULT current_user,
    pid             INTEGER NOT NULL DEFAULT pg_backend_pid()
);

Design notes:

  • deploy_hash is SHA-256 of the deploy script content — detects tampering
  • dependencies array stores change names this change depends on
  • Advisory lock (pg_advisory_lock) preferred over the lock table for short-lived deploy sessions; table is for observability
  • BIGINT GENERATED ALWAYS AS IDENTITY — not UUID, because this is an append-only log where ordering matters

2. The muscdb CLI

Lives at toolchain/muscdb/ alongside toolchain/muscomp/. Shares Conan profile and libpqxx dependency.

muscdb deploy  --profile PROFILE [--dir DIR] [--dry-run] [--allow-destructive]
muscdb verify  --profile PROFILE [--dir DIR]
muscdb revert  --profile PROFILE [--dir DIR] [--to CHANGE]
muscdb status  --profile PROFILE
muscdb log     --profile PROFILE [--limit N]

Connection resolution (in priority order):

  1. Profile file: deployment-profiles/{profile}.env (see §2a below)
  2. libpq environment variables: PGHOST, PGPORT, PGUSER, PGPASSWORD, PGDATABASE
  3. --uri postgresql://user:pass@host:port/dbname (explicit override)
  4. ~/.pgpass (standard PostgreSQL)

This is the key advantage over Sqitch: libpqxx uses libpq under the hood, so all standard PostgreSQL connection mechanisms work without any adapter code.

2a. Deployment Profiles

Every project declares at least three deployment profiles: dev, qa, prod. Access credentials MUST NOT be committed to git. Instead, muscdb reads non-committed environment files that source libpq variables.

Generated by muscomp (once, gitignored, user-owned after creation):

generated/{project}/deployment-profiles/
├── dev.env              ← actual credentials (gitignored)
├── qa.env               ← actual credentials (gitignored)
├── prod.env             ← actual credentials (gitignored)
├── dev.env.example      ← committed template (no secrets)
├── qa.env.example       ← committed template (no secrets)
└── prod.env.example     ← committed template (no secrets)

Example dev.env (gitignored, never committed):

# Deployment profile: dev
# Source: muscovite DEPLOYMENT block
PGHOST=localhost
PGPORT=5555
PGUSER=postgres
PGPASSWORD=postgres
PGDATABASE=fhir_health
MUSCDB_PROJECT=fhirhealth
MUSCDB_SCHEMA=muscovite

Example dev.env.example (committed template):

# Deployment profile: dev
# Copy to dev.env and fill in credentials. NEVER commit dev.env.
PGHOST=localhost
PGPORT=5555
PGUSER=
PGPASSWORD=
PGDATABASE=
MUSCDB_PROJECT=fhirhealth
MUSCDB_SCHEMA=muscovite

Profile resolution:

  1. muscdb reads --profile dev → looks for deployment-profiles/dev.env
  2. File is sourced as environment variables (key=value, shell-compatible)
  3. Variables are set as libpq environment — libpqxx picks them up automatically
  4. Any variable already set in the environment takes precedence (12-factor friendly)

Gitignore rule (generated into project .gitignore):

deployment-profiles/*.env
!deployment-profiles/*.env.example

CI/CD integration: In CI, profiles are populated from pipeline secrets — no file needed:

env:
  PGHOST: ${{ secrets.QA_DB_HOST }}
  PGPASSWORD: ${{ secrets.QA_DB_PASSWORD }}

muscdb falls back to libpq env vars when no profile file exists.

Deploy algorithm:

  1. Acquire advisory lock (pg_advisory_lock(hashtext(project_name)))
  2. Read manifest.json — topological order of changes with dependencies
  3. Read muscovite.deployment_log — which changes are already deployed
  4. Compute delta: changes in manifest not yet in log
  5. For each pending change (in dependency order): a. BEGIN b. Execute deploy/{change_name}.sql c. INSERT INTO muscovite.deployment_log (...) d. COMMIT
  6. Release advisory lock

Revert algorithm:

  1. Acquire advisory lock
  2. Read deployment log — what's deployed
  3. Compute reverse dependency order
  4. For each change to revert (reverse order): a. BEGIN b. Execute revert/{change_name}.sql c. DELETE FROM muscovite.deployment_log WHERE change_name = ... d. COMMIT
  5. Release advisory lock

4. DSL Integration — DEPLOYMENT Block

Database deployment becomes a first-class project DSL concern, alongside MESSAGING, SEARCH, and TELEMETRY. The existing sqitch_project directive in the OUTPUT block is deprecated.

Before (Sqitch-era):

OUTPUT {
    directory "generated"
    sqitch_project "fhirhealth"
}

DATABASE development {
    uri "db:pg://postgres:postgres@localhost:5555/fhir_health"
}

After (native deployer):

DEPLOYMENT {
    project "fhirhealth"
    schema "muscovite"
    strategy sequential
    profiles dev qa prod
}

DATABASE development {
    uri "postgresql://postgres:postgres@localhost:5555/fhir_health"
    description "Local Docker PostgreSQL"
}

Grammar addition:

ProjectItem          <- ... / DeploymentBlock

DeploymentBlock      <- 'DEPLOYMENT' '{' DeploymentDirective* '}'
DeploymentDirective  <- DeploymentProject
                      / DeploymentSchema
                      / DeploymentStrategy
                      / DeploymentProfiles
DeploymentProject    <- 'PROJECT' STRING
DeploymentSchema     <- 'SCHEMA' STRING
DeploymentStrategy   <- 'STRATEGY' ('SEQUENTIAL' / 'TRANSACTIONAL')
DeploymentProfiles   <- 'PROFILES' IDENTIFIER+

Rationale for DSL over external config:

  • Muscovite already declares MESSAGING (NATS), SEARCH (OpenSearch), TELEMETRY (OpenTelemetry) as project-level infrastructure blocks
  • Database deployment is infrastructure — same category
  • The project name, schema name, and strategy are properties of the domain model, not of the deployment environment
  • muscomp needs this information at generation time to produce the manifest
  • Keeps all infrastructure declarations in one place (the .mus file)

Migration path for sqitch_project:

  • sqitch_project remains valid for one release cycle (backward compat)
  • muscomp emits a deprecation warning: "sqitch_project is deprecated; use DEPLOYMENT { project ... }"
  • If both sqitch_project and DEPLOYMENT are present, DEPLOYMENT wins

5. CMake Integration

The MUSCOVITE_DEPLOYER cache variable (FTR-537) gains a native option:

set(MUSCOVITE_DEPLOYER "native" CACHE STRING "Database deployer: native | sqitch")

if(MUSCOVITE_DEPLOYER STREQUAL "native")
    set(_deploy_cmd "${MUSCDB_EXECUTABLE} deploy --profile dev --dir ${_migration_dir}")
    set(_verify_cmd "${MUSCDB_EXECUTABLE} verify --profile dev --dir ${_migration_dir}")
    set(_revert_cmd "${MUSCDB_EXECUTABLE} revert --profile dev --dir ${_migration_dir}")
elseif(MUSCOVITE_DEPLOYER STREQUAL "sqitch")
    # Legacy fallback — removed after Sqitch sunset
    set(_deploy_cmd "sqitch deploy ${_db_uri}")
    set(_verify_cmd "sqitch verify ${_db_uri}")
    set(_revert_cmd "sqitch revert ${_db_uri} -y")
endif()

6. Sqitch Migration Path

For existing projects with Sqitch deployment history:

muscdb import-sqitch --uri URI --dir DIR

This reads Sqitch's sqitch.changes table and populates muscovite.deployment_log with equivalent entries, then generates manifest.json from the existing deploy/revert/verify directory structure. One-time migration, idempotent.

7. Sqitch Removal — Full Purge

Once muscdb is the default deployer and import-sqitch exists for migration, all Sqitch references must be removed from the project. This is a dedicated phase because Sqitch is deeply embedded across documentation, agent instructions, build scripts, metrics, and governance artifacts.

Removal scope (audit: 435 files, 1,724 reference lines):

When This Decision Applies

  • When generating database deployment artifacts (manifest, deploy/revert/verify SQL)
  • When implementing or modifying muscdb CLI
  • When adding DEPLOYMENT block support to parser/IR/generators
  • When writing CMake database targets (db-setup-*, db-deploy-*, etc.)

When This Decision Does NOT Apply

  • SQL generation logic itself (unchanged — deploy/revert/verify scripts are tool-agnostic)
  • Non-PostgreSQL targets
  • Consumer repos that choose to keep Sqitch (their choice via MUSCOVITE_DEPLOYER)

Enforcement

  • muscomp generates manifest.json alongside deploy/revert/verify directories
  • muscdb is the default deployer when MUSCOVITE_DEPLOYER=native
  • Grammar conformance tests verify DEPLOYMENT block parsing
  • tools/lint-ddd.sh warns on sqitch_project usage after deprecation
