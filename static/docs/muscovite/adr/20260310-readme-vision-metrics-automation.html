ADR: Automate README and VISION Metrics in Release Ceremony

ADR: Automate README and VISION Metrics in Release Ceremony

  • Date: 2026-03-10
  • Status: Accepted
  • Deciders: Johannes Lochmann

Context

README.md and VISION.md contain hardcoded project metrics (version, release count, ADR count, grammar rules, line counts, amplification ratios, example project tables, and feature status). These metrics drifted badly — README showed v1.30.0 when the project was at v1.61.0 — because no automation existed to keep them current.

The project already has a pattern for self-updating documentation (ADR 20260217): generate-context.sh, generate-system-brief.sh, and generate-dsl-reference.sh all run during release.sh and produce fresh docs from live project sources. README and VISION were the gap.

Decision

  1. Create tools/update-readme-metrics.sh — collects metrics using the same patterns as generate-context.sh (find + wc -l, grep PEG rules, git tag count) and stamps them into README.md using sed marker-based replacement.

  2. Hook into release.sh — runs after generate-context.sh, alongside other doc generators. Failure is a warning (not a release blocker) since README staleness is cosmetic, not functional.

  3. Marker-based replacement — README.md uses <!-- BEGIN METRICS --> / <!-- END METRICS --> and <!-- BEGIN EXAMPLES --> / <!-- END EXAMPLES --> comment markers. The script regenerates content between markers, preserving all surrounding prose.

  4. VISION.md staleness check (deferred) — release.sh SHOULD emit a warning if VISION's Implementation Status table contains 🔲 Planned items whose version tag already exists as a git tag. Not yet implemented; flagged for future work.

  5. Add to Rule 17 (self-updating docs) in copilot-instructions.md.

Consequences

  • README metrics will be current at every release with zero manual effort.
  • Example project metrics (contexts, DSL lines, generated lines, amplification) update automatically.
  • VISION staleness is flagged but not auto-fixed (requires editorial judgment for feature descriptions).
  • The marker comment pattern is invisible when rendered on GitHub but enables reliable sed replacement.
