Specification-Based Queries

Specification-Based Queries

  • Status: accepted
  • Date: 2026-01-23
  • Tags: ddd, specification, query, domain-services, dsl

Context and Problem Statement

Domain-Driven Design includes the Specification pattern for encapsulating query predicates as first-class domain concepts. Currently, Muscovite parses SPECIFICATION declarations in the DSL but does not generate corresponding query methods.

SPECIFICATION ActivePatient FOR Patient
    WHERE active = true
    DOC "Patients currently receiving care";

SPECIFICATION PatientsByGender FOR Patient (gender_code TEXT)
    WHERE gender_id = (SELECT id FROM basedata.administrative_gender WHERE code = gender_code)
    DOC "Filter patients by administrative gender";

Questions:

  1. How should specifications map to generated code?
  2. Should specifications generate SQL WHERE clauses, C++ methods, or both?
  3. How do parameterized specifications handle type safety?

Decision Drivers

  • DDD Compliance: Specifications are a tactical DDD pattern for reusable query predicates
  • Type Safety: Parameters should be validated at compile time
  • Composability: Specifications should be combinable (AND, OR, NOT)
  • Cross-Layer Consistency: Same predicate usable in DB, gRPC, and Qt
  • Ubiquitous Language: Specification names should appear in APIs and UI

Considered Options

  1. SQL-only generation: Generate SQL WHERE clause fragments
  2. C++ QueryService methods only: Generate find_by_spec() methods
  3. Full-stack generation: SQL + C++ + gRPC + Qt filter support (chosen)

Decision Outcome

Chosen option: "Full-stack generation", because specifications are domain concepts that should be available at all layers.

Generated Artifacts

1. C++ QueryService Methods:

class PatientQueryService {
public:
    // Simple specification (no parameters)
    std::vector<Patient> findActivePatients();
    
    // Parameterized specification
    std::vector<Patient> findPatientsByGender(const std::string& gender_code);
};

2. SQL Predicate (internal to QueryRepository):

std::string PatientQueryRepository::activePatientPredicate() const {
    return "active = true";
}

std::string PatientQueryRepository::patientsByGenderPredicate(const std::string& gender_code) const {
    return fmt::format(
        "gender_id = (SELECT id FROM basedata.administrative_gender WHERE code = {})",
        txn_.quote(gender_code)
    );
}

3. gRPC Service Methods:

service PatientService {
    rpc ListActivePatients(ListActivePatientsRequest) returns (stream PatientResponse);
    rpc ListPatientsByGender(ListPatientsByGenderRequest) returns (stream PatientResponse);
}

message ListPatientsByGenderRequest {
    string gender_code = 1;
}

4. Qt Filter Support:

// Filter combo in PatientListWidget
void PatientListWidget::applySpecification(const QString& specName, const QVariantMap& params) {
    if (specName == "ActivePatient") {
        model_->setFilter(PatientModel::Filter::ActivePatients);
    } else if (specName == "PatientsByGender") {
        model_->setFilter(PatientModel::Filter::ByGender, params["gender_code"]);
    }
}

Composition Support (Future)

SPECIFICATION HighPriorityActivePatient FOR Patient
    COMBINES ActivePatient AND HighPriority
    DOC "Active patients with high priority flag";

Positive Consequences

  • Specifications become true domain objects across all layers
  • Query logic encapsulated and reusable
  • gRPC API reflects domain language
  • Qt clients can offer predefined filters

Negative Consequences

  • More generated code per specification
  • Complex SQL predicates may not translate cleanly to Qt filters
  • Subqueries in specifications need careful handling

Implementation Notes

Generator Changes Required
