Work Orders as Coordinator Dispatch Artifacts

Work Orders as Coordinator Dispatch Artifacts

  • Status: superseded by 20260318-nats-agent-coordination-abandoned
  • Deciders: Johannes Lochmann
  • Date: 2026-03-13
  • Tags: coordinator, multi-agent, planning

Technical Story: Derived from v1.65.1 delegation experiment where 3 of 5 items were dispatched to sub-agents (GPT-4.1, Claude Sonnet 4.5) via ephemeral prompt instructions that died with the session. Zero rework was needed, but zero audit trail remained.

Context and Problem Statement

The coordinator assigns FTRs to workers. Today, the unit of assignment is the FTR itself — one feature, one worker. The worker receives the FTR spec (a human-readable document describing problem, proposal, and acceptance criteria) and must independently determine the mechanical implementation steps.

This creates three problems:

  1. No audit trail. When an orchestrating agent decomposes an FTR into sub-tasks and dispatches them to sub-agents, the mechanical instructions are ephemeral prompt text. If the output is wrong, there is no committed artifact showing what the agent was told.

  2. No model-aware decomposition. A complex FTR like FTR-402 (TLA+ + Python + docs) could be split into 3 parallel work units at different capability tiers, but the coordinator only knows how to assign whole FTRs.

  3. Human specs become unreadable. Embedding mechanical agent instructions (exact file:line edits, before/after code blocks) into FTR specs destroys their value as human-readable product documents.

Decision Drivers

  • FTR specs must remain human-readable product documents
  • Agent instructions must be auditable (committed, diffable, reviewable)
  • The coordinator must support 1 FTR → N work orders → N workers
  • Model/cost optimization: mechanical work orders go to cheap models, cross-layer to standard
  • Work orders should be generated at dispatch time, not pre-authored

Considered Options

Option 1: Work orders as coordinator-managed artifacts

Work orders are first-class in the coordinator. The assignment model changes from FTR → worker to work order → worker. An FTR is complete when all its work orders are complete. Work orders live in workflow/versions/vX.Y.Z/workorders/ and are committed for auditability.

Option 2: Work orders internal to orchestrating agent

The coordinator still assigns whole FTRs. The receiving agent internally decomposes into sub-tasks and dispatches sub-agents. Work orders are session-ephemeral.

Option 3: Enhanced FTR specs with agent sections

Add machine-readable sections to FTR specs. One document serves both audiences.

Decision Outcome

Chosen option: Option 1 — Work orders as coordinator-managed artifacts, because it provides auditability, enables model-aware cost optimization, and keeps the FTR spec clean. Option 2 loses the audit trail (the original problem). Option 3 degrades FTR readability for humans.

Schema

Work order files follow the naming convention {FTR-NNN}-{seq}.workorder.md and live in workflow/versions/vX.Y.Z/workorders/. Each work order has a fixed schema:

# Work Order: {FTR-NNN}-{seq} — {short title}

**FTR**: FTR-NNN
**Sequence**: {seq} (e.g., 001, 002, 003)
**Status**: pending | assigned | done | failed
**Model**: {model-id} (e.g., gpt-4.1, claude-sonnet-4.5, claude-opus-4.6)

### Complexity Tiers and Model Mapping

| Tier | Description | Recommended Models | Example |
|------|-------------|-------------------|---------|
| `mechanical` | Find-and-replace, config changes, single-file edits | gpt-4.1, claude-haiku-4.5 | FTR-380: REQUIRED→QUIET |
| `cross-layer` | Multiple files, must understand relationships | claude-sonnet-4.5, gpt-5.1 | FTR-379: framework + generator + golden |
| `architectural` | Ambiguous, iterative, design decisions | claude-opus-4.6, claude-sonnet-4.6 | BFX-008: regex redesign |

### Assignment Model Change

**Before:** `coordinator.assign(ftr, worker)` — 1:1, FTR-granular.

**After:** `coordinator.assign(work_order, worker)` — N:1 (N work orders per FTR),
each work order independently assignable. An FTR transitions to `pr_ready` when all
its work orders reach `done`.

The `depgraph` module gains intra-FTR dependency tracking: work order 002 can depend
on work order 001 within the same FTR.

### FTR Lifecycle Impact

| FTR State | Trigger |
|-----------|---------|
| `pending` | No work orders exist or all are `pending` |
| `assigned` | At least one work order is `assigned` |
| `progressing` | At least one work order is `assigned` and work has started |
| `pr_ready` | All work orders are `done`, PR created |
| `reviewing` | PR under review |
| `merged` | PR merged |
| `blocked_human` | Any work order escalated to human |

### Positive Consequences

- Auditable dispatch: committed artifacts show exactly what each agent was told
- Cost optimization: mechanical tasks go to cheap models (~10× cost reduction)
- Parallelism: N work orders from one FTR can execute concurrently on N workers
- Reproducibility: failed work order can be re-dispatched (same or stronger model)
- Reviewable: work orders are diffable in PRs like any other artifact

### Negative Consequences

- Coordinator complexity increases (work order state tracking, intra-FTR dependencies)
- TLA+ state space grows (new variables, actions, invariants)
- Work orders may go stale if they reference exact line numbers and code changes between
  generation and execution — mitigated by generating at dispatch time, not pre-authoring

## AI Agent Guidance

### When This Decision Applies

- When the coordinator assigns work to workers
- When an orchestrating agent decomposes an FTR into sub-tasks
- When dispatching work to sub-agents of any model tier

### When This Decision Does NOT Apply

- Single-person manual implementation (no coordinator involvement)
- FTRs small enough to be a single work order (still create one work order for auditability)

### Enforcement

- Coordinator Python code: `assign_next()` creates work orders before assignment
- Work order files committed to `workflow/versions/vX.Y.Z/workorders/`
- Template at `workflow/templates/workorder.md`
- Rule added to `.github/copilot-instructions.md`

### Key Files

- `agents/coordinator.py` — assignment logic, work order state tracking
- `agents/depgraph.py` — intra-FTR work order dependency resolution
- `agents/tlaplus/Coordinator.tla` — formal specification of work order actions
- `workflow/templates/workorder.md` — work order file template
- `doc/COORDINATOR.md` — protocol documentation
