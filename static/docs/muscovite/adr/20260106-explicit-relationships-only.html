Derive All Foreign Keys from Explicit DDD Relationships

Derive All Foreign Keys from Explicit DDD Relationships

  • Status: accepted
  • Deciders: Project maintainers
  • Date: 2026-01-06
  • Tags: dsl, relationships, foreign-keys, qt, generators

Context and Problem Statement

The Qt Widgets generator used two methods to detect foreign key fields for ComboBox generation:

  1. Explicit RELATIONSHIPS blocks in DDD files
  2. Implicit inference from naming convention (UUID fields ending with _id)

The inference approach caused inconsistent behavior: some FK fields got ComboBoxes while others got plain LineEdits, depending on whether fuzzy entity name matching succeeded. This created confusion and bugs that were hard to diagnose.

All other generators (SQL, PL/pgSQL API, pgTAP, C++ DBA, gRPC) already used only explicit RELATIONSHIPS, making Qt the inconsistent outlier.

Decision Drivers

  • Consistency across all generators (SQL, API, DBA, gRPC, Qt)
  • Predictable, deterministic output from DSL input
  • Clear error messages when relationships are missing
  • Single source of truth for FK definitions
  • Avoid "magic" naming conventions that silently affect behavior

Considered Options

  • Keep inference with improved fuzzy matching
  • Use only explicit RELATIONSHIPS (remove inference)
  • Add new syntax for FK-only fields without full relationships

Decision Outcome

Chosen option: "Use only explicit RELATIONSHIPS", because it provides deterministic behavior, matches all other generators, and enables early validation with clear error messages.

Positive Consequences

  • All FK fields are explicitly declared in DDD model
  • Generators produce identical FK handling across all targets
  • Missing relationships fail fast with clear error paths
  • No "magic" behavior based on field naming
  • Easier to audit and understand FK structure

Negative Consequences

  • More verbose DDD files (must declare RELATIONSHIPS for all FKs)
  • Migration effort for existing DDD files missing RELATIONSHIPS
  • Cannot rely on naming conventions as shortcuts

Implementation

Qt Generator Changes

Remove from src/dslgen/generators/qt/widgets-views.rkt:

  • infer-target-entity-name function
  • Fuzzy matching in find-entity-by-name
  • inferred-fk-fields block in build-fk-info

Keep only explicit-fk-fields derived from get-to-one-relationships.

DDD File Pattern

-- Field declaration (implementation detail - the actual column)
ENTITY Mineral IDENTIFIED_BY id (
    id UUID NOT NULL,
    crystal_system_id UUID,        -- FK column exists
    optical_type_id UUID NOT NULL, -- FK column exists
)
-- Relationship declaration (semantic meaning)
RELATIONSHIPS {
    crystal_system -> CrystalSystem OPTIONAL ONE;   -- Generates FK handling
    optical_type -> OpticalType REQUIRED ONE;       -- Generates FK handling
}

The field declares the column; the relationship declares the semantic reference. Both are required for FK fields.

AI Agent Guidance

When This Decision Applies

  • Adding new entities with FK fields to DDD files
  • Modifying Qt Widgets generator FK detection
  • Debugging missing ComboBoxes in Qt forms
  • Any generator that needs to identify FK relationships

When This Decision Does NOT Apply

  • Primary key fields (always id, no relationship needed)
  • Non-FK UUID fields (correlation IDs, external references)
  • Aggregate containment (handled implicitly by CONTAINS)

Enforcement

  • Qt generator only reads explicit RELATIONSHIPS
  • Validation terminates generation if relationship targets don't exist
  • All FK fields must have corresponding RELATIONSHIPS entry

Key Files

  • src/dslgen/generators/qt/widgets-views.rkt - Qt FK detection
  • src/dslgen/validate.rkt - Relationship target validation
  • src/dslgen/model/ddd-types.rkt - Relationship struct definition
