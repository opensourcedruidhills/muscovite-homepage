ADR-20260206: Build Profiling Instrumentation (Per-Phase Metrics + HTML Reports)

ADR-20260206: Build Profiling Instrumentation (Per-Phase Metrics + HTML Reports)

Status: Accepted
Date: 2026-02-06
Deciders: Jolo (Staff Engineer)

Context

With multi-frontend generation (ADR 20260206-multi-frontend-architecture), build times become critical:

  • Widgets generation ~900ms
  • QML generation ~X ms (new, unknown)
  • C++ compilation + linking still dominates total cycle

The user's iteration loop depends on fast feedback:

  1. Generate DSL → Racket phase
  2. Build C++ → CMake/Ninja phase
  3. Test → Run application phase

Currently, phases are not systematically measured. We need visibility into where time is spent and historical trending to validate that multi-frontend strategy doesn't regress build speed.

Problem Statement

Current State (v0.9.0):

  • Racket generator has time-phase macro (1ms precision)
  • Per-generator metrics stored in JSON: generated-{project}/metrics/current.json and .jsonl history
  • HTML dashboard exists: generated-{project}/metrics/report.html
  • But: C++ build (CMake configure + compilation + linking) is not measured
  • And: No trending analysis (compare build #1 vs. build #100)
  • And: No alerting on regressions (build suddenly 2x slower)

Information Gaps:

  • Is QML generation faster or slower than Widgets?
  • Which C++ targets are bottlenecks (proto compilation? MOC? linking)?
  • Did the last code change regress build time?

Decision

Comprehensive Build Profiling Strategy

Three layers of instrumentation:

  1. Racket DSL Generation Phase (already exists, enhance)

    • Measure: sqitch-unified, cpp-dba, cpp-dba-test, grpc, qt-widgets, qt-quick
    • Capture: phase name, duration (ms), timestamp, optional details
    • Output: JSON snapshot + JSONL history

  2. CMake Configuration Phase (new)

    • Measure: cmake --preset <name> time (configure step)
    • Capture: preset name, duration, timestamp
    • Hook: CMake message() at start/end of configuration

  3. C++ Compilation Phase (new, via Ninja stats)

    • Measure: Per-target compilation time (if available), total build time
    • Capture: Ninja -d stats output → JSON conversion
    • Output: JSON with per-object/per-target timings

Metrics Collection Points

Phase Tool Measurement Output
Racket Generation Racket time-phase Per-generator (ms) metrics/current.json
CMake Configure CMake message() hooks Total configure time metrics/cmake-configure.json
Proto Compilation Ninja -d stats Per-proto file time metrics/ninja-stats.json
Qt MOC Ninja -d stats Per-entity MOC time metrics/ninja-stats.json
Linking Ninja -d stats Per-target link time metrics/ninja-stats.json

CMake build (new integration):

cmake --build --preset debug --verbose 2>&1 | tee build.log

# Post-build: extract Ninja stats
python3 tools/extract-ninja-metrics.py build/debug/.ninja_log \
  --output examples/geosciences/generated/metrics/ninja-stats.json

# Merge all metrics into unified report
python3 tools/merge-build-metrics.py \
  --racket-metrics examples/geosciences/generated/metrics/current.json \
  --ninja-metrics examples/geosciences/generated/metrics/ninja-stats.json \
  --output examples/geosciences/generated/metrics/report.html

Build Workflow Integration

Step 1: Generate DSL

cd dslgen && time racket generate.rkt \
  --project ../examples/geosciences/project.ddd \
  --cpp-dba --grpc --qt-frontend both
# Outputs: examples/geosciences/generated/metrics/current.json

Step 2: Build C++

cmake --preset debug && time cmake --build --preset debug 2>&1 | tee /tmp/build.log
# Ninja stats: build/debug/.ninja_log

Step 3: Extract + Merge Metrics

python3 tools/extract-ninja-metrics.py build/debug/.ninja_log \
  --output examples/geosciences/generated/metrics/ninja-stats.json

python3 tools/merge-build-metrics.py \
  --racket-metrics examples/geosciences/generated/metrics/current.json \
  --ninja-metrics examples/geosciences/generated/metrics/ninja-stats.json \
  --output examples/geosciences/generated/metrics/report.html

Step 4: View Report

open examples/geosciences/generated/metrics/report.html
# or
xdg-open examples/geosciences/generated/metrics/report.html

Consequences

Positive

  1. Visibility: Know exactly where build time is spent (Racket vs. CMake vs. Ninja)
  2. Trending: Detect regressions early (build suddenly 2x slower)
  3. AI Agent Validation: Prove that multi-frontend strategy doesn't explode build times
  4. Baseline Capture: Establish reference metrics before optimizations
  5. Future Optimization: Identify bottlenecks (e.g., "proto compilation is 40% of Ninja time")

Negative

  1. Tooling Overhead: New Python scripts, HTML generation, metrics storage
  2. Disk Space: JSONL history accumulates; need cleanup strategy
  3. Parse Complexity: Ninja stats format is per-version; fragile parsing
  4. Latency: Post-build metric extraction adds ~5-10s per build

Neutral

  1. Reproducibility: Metrics are non-deterministic (machine-dependent), but trends are meaningful
  2. CI Integration: Can be extended to GitHub Actions, but not required for local iteration

AI Agent Guidance

When is this decision applicable?

  • Any change affecting build performance (Racket generator, CMake, C++ code)
  • Any iteration round where build time matters
  • Any optimization work (baseline + measurement required)

  • HTML report is mandatory output (not optional)
  • Historical data is preserved in .jsonl for trending
