ADR-20260206: Multi-Frontend Architecture (Platform Agnostic)

ADR-20260206: Multi-Frontend Architecture (Platform Agnostic)

Status: Accepted
Date: 2026-02-06
Deciders: Jolo (Staff Engineer)

Context

Muscovite is designed as a platform-agnostic code generation framework. The current implementation targets Qt Widgets as the sole frontend (ADR 20251231). However, this decision, while pragmatic for v0.9.0, was scoped to early development stages.

As the framework matures, we need to:

  1. Validate separation of concerns: Prove that business logic (C++ DBA, gRPC, validation) is truly independent of presentation
  2. Support multiple frontends: Desktop (Widgets/QML), Web (future: Qt WASM, React), Mobile (future: Qt for Android/iOS)
  3. Enable AI iteration efficiency: QML's declarative syntax + interpreted/JIT execution permits UI tweaks without C++ recompilation bottleneck
  4. Maintain single source of truth: One DBA/gRPC layer, multiple presentation layers consuming it

Problem Statement

Current State (v0.9.0):

  • ADR 20251231 mandates Widgets-only; QML is forbidden
  • Qt generation (qt-generator.rkt) produces unified MDI client in a single code path
  • Widgets require C++ recompilation for every UI tweak → 5-7 minute rebuild cycles

Iteration Pain Points:

  • UI layout change (padding, color, spacing) requires:
    1. Edit C++ dialog code
    2. Rebuild Widgets (recompile, MOC, link)
    3. Test in running application (~5 min cycle)
  • This is inefficient for AI-driven UI iteration (agent's "painting" the UI needs sub-second feedback)

SoC Validation:

  • Are DBA/gRPC layers truly decoupled from presentation?
  • Current single-frontend approach doesn't test this assumption
  • Multi-frontend proves: same DBA/gRPC, different presentations

Decision

Architecture: Shared DBA/gRPC + Multiple Presentation Layers

Domain Model (.ddd)
  ↓
Racket Generators
  ├─→ PostgreSQL + Sqitch (deploy/)
  ├─→ C++ DBA Layer (generated-{project}/dba/)          ← SHARED
  ├─→ gRPC API (generated-{project}/grpc/)              ← SHARED
  └─→ Presentation Layer (multiple choices)
      ├─→ Qt Widgets (generated-{project}/qt-widgets/)
      └─→ Qt Quick/QML (generated-{project}/qt-quick/)

Constraint: No Separate Client Libraries

  • ❌ FORBIDDEN: Separate libmuscovite-widgets-client and libmuscovite-qml-client
  • ✅ REQUIRED: Both Widgets and QML frontends link against identical generated-{project}/dba/ and generated-{project}/grpc/

Rationale:

  • Single source of truth for entities, validators, services
  • Bug fix in DBA layer propagates to all frontends immediately
  • No feature drift between frontends
  • Easier testing: validate DBA once, all frontends inherit that validation

Frontend Generation Strategy

Aspect Widgets QML Future
Language C++ (Qt6::Widgets) QML + C++ ViewModel JS/React, Kotlin, Swift
Generation Flag --qt-frontend widgets --qt-frontend qml --frontend js, --frontend mobile
Output Dir qt-widgets/ qt-quick/ js/, mobile/
Build System CMake + AUTOMOC CMake + qt_add_qml_module Custom per platform
Compilation Model Compiled C++ + Qt Widgets Compiled C++ (ViewModel) + Interpreted QML Platform-dependent
DBA/gRPC Link Shared Shared Shared

Multi-Target Generation

Racket generate.rkt flag: --qt-frontend [widgets|qml|both] (default: both)

# Generate Widgets only
racket generate.rkt --project ../examples/geosciences/project.ddd --qt-frontend widgets

# Generate QML only
racket generate.rkt --project ../examples/geosciences/project.ddd --qt-frontend qml

# Generate both (SoC validation mode)
racket generate.rkt --project ../examples/geosciences/project.ddd --qt-frontend both

Output (when --qt-frontend both):

examples/geosciences/generated/
├── dba/                    ← Shared (both frontends link here)
├── grpc/                   ← Shared (both frontends link here)
├── qt-widgets/             ← Widgets frontend only
│   ├── main-widgets.cpp
│   ├── CMakeLists.txt
│   └── ...existing widget generation...
└── qt-quick/               ← QML frontend only
    ├── main-qml.cpp
    ├── CMakeLists.txt
    ├── qml/
    │   ├── main.qml
    │   ├── theme/MuscoviteTheme.qml
    │   └── dialogs/
    └── src/
        └── viewmodels/     ← C++ ViewModel classes

Consequences

Positive

  1. SoC Validation: Multi-frontend approach proves DBA/gRPC truly encapsulates business logic
  2. Iteration Efficiency: QML permits UI iteration without C++ recompilation (major win for AI agents)
  3. Platform Agnostic: Framework remains neutral; future frontends (JS, mobile) coexist
  4. No Feature Drift: Single DBA/gRPC source prevents Widgets and QML diverging
  5. Future-Ready: Qt WASM port uses identical QML + DBA; only entry point differs
  6. Generator Simplicity: Common code generation (display helpers, type converters) shared across frontends

Negative

  1. CMake Complexity: Dual-frontend projects have conditional includes and multiple executable targets
  2. Testing Surface: Both frontends must be tested; Qt Widgets tests continue, QML tests added (new test patterns)
  3. Generation Time: Slightly longer generation phase when --qt-frontend both (two code paths)
  4. Future Maintenance: As frontends multiply, generator maintainability increases (new ADRs per frontend)

Neutral

  1. No Impact on Database Layer: PostgreSQL, Sqitch migrations unchanged
  2. No Impact on gRPC: Proto definitions, service stubs unchanged
  3. ADR Supersession: ADR 20251231 (Widgets-only) superseded by this decision; not a breaking change (Widgets remains supported)

Implementation Plan

Phase 1: Qt Widgets + Qt Quick (Desktop, Same Machine)

Timeline: 4–6 weeks

Deliverables:

  1. Refactor qt-generator.rkt → separate Widgets and QML code paths (ADR 20260206-qml-viewmodel-binding-pattern)
  2. Implement ViewModel pattern for QML (all entity fields as Q_PROPERTY)
  3. Generate QML templates (theme, dialogs, lists, main.qml)
  4. Update CMake to support dual-frontend targets
  5. Test on geosciences example: both frontends coexist, share DBA/gRPC
  6. Benchmark: measure generation time and build time (baseline for optimization)
  7. Document build instrumentation (ADR 20260206-build-profiling-instrumentation)

Phase 2+: Qt WASM, Mobile, JavaScript (Future, post-v1.0.0)

Timeline: 2027+

These will follow the same multi-frontend pattern, leveraging shared DBA/gRPC.

AI Agent Guidance

When is this decision applicable?

  • Any new frontend target (Widgets, QML, etc.)
  • Any DSL extension affecting presentation generation
  • Any build system change affecting multi-frontend coordination

How it's enforced:

  • Racket generator ensures DBA/gRPC output is identical regardless of --qt-frontend flag
  • CMake validates shared library linkage in dual-frontend projects (test: ldd both binaries)
  • Tests verify: same entity validation results across Widgets and QML

Key files affected:

  • dslgen/generators/qt/ → split into widgets.rkt, qml.rkt, common.rkt
  • dslgen/generate.rkt → add --qt-frontend CLI flag
  • CMake build system → conditional frontend includes
  • framework/qt/ → ViewModel base classes (new)
