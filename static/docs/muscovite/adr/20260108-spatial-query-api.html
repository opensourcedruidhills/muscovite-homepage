Spatial Query API for Geologic Map Features

Spatial Query API for Geologic Map Features

Date: 2026-01-08
Status: Accepted
Deciders: System Architect, Database Team, GIS Team
Tags: postgresql, postgis, api, spatial-queries, performance

Context and Problem Statement

Geologic mapping applications require common spatial queries: "features within map extent", "samples near fault", "deposits within 5km of volcano", "outcrops along transect line". How should we expose PostGIS spatial operations through the PL/pgSQL API layer in a way that:

  1. Provides convenient functions for common spatial operations
  2. Performs efficiently with spatial indexes
  3. Maintains API consistency with existing CRUD patterns
  4. Supports multiple coordinate systems (SRID transformation)

Decision Drivers

  • Common Patterns: 80% of spatial queries follow predictable patterns (within, near, intersecting)
  • Performance: Leverage GIST spatial indexes for sub-second response on large datasets
  • SRID Flexibility: Support queries with mixed coordinate systems
  • API Consistency: Follow existing <schema>_api.<entity>_<operation>() naming
  • Geometry Type Safety: Return typed geometries (Point, LineString, Polygon)
  • Export Formats: Support WKT, GeoJSON, KML for interoperability

Considered Options

Option 1: Direct PostGIS Function Calls (REJECTED)

-- Application makes raw PostGIS calls
SELECT * FROM structural_geology.fault
WHERE ST_Intersects(
    trace_geometry,
    ST_MakeEnvelope(-180, -90, 180, 90, 4326)
);

Pros:

  • Full PostGIS power available
  • No wrapper overhead

Cons:

  • Complex syntax: Application must know ST_* functions
  • No abstraction: PostGIS changes break application
  • No SRID handling: Application must manage transformations
  • Inconsistent: Doesn't match existing API patterns

Option 2: REST/gRPC Spatial Service (REJECTED)

service SpatialQueryService {
    rpc FeaturesWithinExtent(Extent) returns (FeatureCollection);
    rpc FeaturesNearPoint(PointQuery) returns (FeatureCollection);
}

Pros:

  • Language-agnostic
  • Clear service boundaries

Cons:

  • Network overhead: Extra hop vs. direct database query
  • Serialization cost: Geometry encoding/decoding
  • Deployment complexity: Separate service to maintain
  • Inconsistent: Other queries use PL/pgSQL API

Option 3: PL/pgSQL Spatial API Functions (ACCEPTED)

-- Features within map extent
CREATE OR REPLACE FUNCTION geologic_mapping_api.map_features_within_extent(
    p_map_id UUID
) RETURNS SETOF geologic_mapping.geologic_map_feature;

-- Features near point
CREATE OR REPLACE FUNCTION geologic_mapping_api.features_near_point(
    p_longitude DECIMAL,
    p_latitude DECIMAL,
    p_distance_meters DECIMAL
) RETURNS SETOF geologic_mapping.geologic_map_feature;

-- Features intersecting geometry
CREATE OR REPLACE FUNCTION geologic_mapping_api.features_intersecting(
    p_geometry_wkt TEXT,
    p_srid INTEGER DEFAULT 4326
) RETURNS SETOF geologic_mapping.geologic_map_feature;

Pros:

  • Consistent API: Matches existing <schema>_api.* pattern
  • SRID Handling: Automatic coordinate transformation
  • Performance: Direct GIST index usage
  • Abstraction: Hides PostGIS complexity
  • Type Safety: Returns table types, not generic records
  • Single Hop: No network overhead

Cons:

  • Must define functions for each common pattern
  • PL/pgSQL boilerplate for each operation

Decision Outcome

Chosen Option: PL/pgSQL Spatial API Functions (Option 3)

Spatial operations are exposed through typed API functions in <schema>_api schemas, following the same pattern as existing CRUD operations.

API Function Categories

1. Within/Contains Queries

Features Within Map Extent:

CREATE OR REPLACE FUNCTION geologic_mapping_api.map_features_within_extent(
    p_map_id UUID
) RETURNS SETOF geologic_mapping.geologic_map_feature AS $$
BEGIN
    RETURN QUERY
    SELECT mf.*
    FROM geologic_mapping.geologic_map_feature mf
    JOIN geologic_mapping.geologic_map m ON mf.map_id = m.id
    WHERE mf.map_id = p_map_id
      AND ST_Intersects(mf.geometry, m.map_extent);
END;
$$ LANGUAGE plpgsql STABLE;

Features Within Polygon (arbitrary AOI):

CREATE OR REPLACE FUNCTION geologic_mapping_api.features_within_polygon(
    p_polygon_wkt TEXT,
    p_srid INTEGER DEFAULT 4326
) RETURNS SETOF geologic_mapping.geologic_map_feature AS $$
DECLARE
    v_polygon GEOMETRY;
BEGIN
    v_polygon := ST_GeomFromText(p_polygon_wkt, p_srid);
    
    RETURN QUERY
    SELECT mf.*
    FROM geologic_mapping.geologic_map_feature mf
    WHERE ST_Within(mf.geometry, v_polygon);
END;
$$ LANGUAGE plpgsql STABLE;

2. Distance/Proximity Queries

Features Near Point:

CREATE OR REPLACE FUNCTION geologic_mapping_api.features_near_point(
    p_longitude DECIMAL,
    p_latitude DECIMAL,
    p_distance_meters DECIMAL,
    p_srid INTEGER DEFAULT 4326
) RETURNS TABLE (
    feature_id UUID,
    source_context TEXT,
    source_entity TEXT,
    distance_meters DECIMAL
) AS $$
BEGIN
    RETURN QUERY
    SELECT 
        mf.id,
        mf.source_context,
        mf.source_entity,
        ST_Distance(
            mf.geometry::geography,
            ST_SetSRID(ST_MakePoint(p_longitude, p_latitude), p_srid)::geography
        ) AS distance_meters
    FROM geologic_mapping.geologic_map_feature mf
    WHERE ST_DWithin(
        mf.geometry::geography,
        ST_SetSRID(ST_MakePoint(p_longitude, p_latitude), p_srid)::geography,
        p_distance_meters
    )
    ORDER BY distance_meters;
END;
$$ LANGUAGE plpgsql STABLE;

Features Along Transect (buffer around line):

CREATE OR REPLACE FUNCTION geologic_mapping_api.features_along_transect(
    p_line_wkt TEXT,
    p_buffer_meters DECIMAL,
    p_srid INTEGER DEFAULT 4326
) RETURNS SETOF geologic_mapping.geologic_map_feature AS $$
DECLARE
    v_line GEOMETRY;
    v_buffer GEOMETRY;
BEGIN
    v_line := ST_GeomFromText(p_line_wkt, p_srid);
    v_buffer := ST_Buffer(v_line::geography, p_buffer_meters)::geometry;
    
    RETURN QUERY
    SELECT mf.*
    FROM geologic_mapping.geologic_map_feature mf
    WHERE ST_Intersects(mf.geometry, v_buffer);
END;
$$ LANGUAGE plpgsql STABLE;

3. Intersection Queries

Features Intersecting Geometry:

CREATE OR REPLACE FUNCTION geologic_mapping_api.features_intersecting(
    p_geometry_wkt TEXT,
    p_srid INTEGER DEFAULT 4326
) RETURNS SETOF geologic_mapping.geologic_map_feature AS $$
DECLARE
    v_geometry GEOMETRY;
BEGIN
    v_geometry := ST_GeomFromText(p_geometry_wkt, p_srid);
    
    RETURN QUERY
    SELECT mf.*
    FROM geologic_mapping.geologic_map_feature mf
    WHERE ST_Intersects(mf.geometry, v_geometry);
END;
$$ LANGUAGE plpgsql STABLE;

Faults Crossing Polygon:

CREATE OR REPLACE FUNCTION structural_geology_api.faults_crossing_polygon(
    p_polygon_wkt TEXT,
    p_srid INTEGER DEFAULT 4326
) RETURNS SETOF structural_geology.fault AS $$
DECLARE
    v_polygon GEOMETRY;
BEGIN
    v_polygon := ST_GeomFromText(p_polygon_wkt, p_srid);
    
    RETURN QUERY
    SELECT f.*
    FROM structural_geology.fault f
    WHERE ST_Crosses(f.trace_geometry, v_polygon);
END;
$$ LANGUAGE plpgsql STABLE;

4. Export Format Functions

GeoJSON Export:

CREATE OR REPLACE FUNCTION geologic_mapping_api.map_features_as_geojson(
    p_map_id UUID
) RETURNS JSON AS $$
BEGIN
    RETURN (
        SELECT json_build_object(
            'type', 'FeatureCollection',
            'features', COALESCE(json_agg(
                json_build_object(
                    'type', 'Feature',
                    'id', mf.id,
                    'geometry', ST_AsGeoJSON(mf.geometry)::json,
                    'properties', json_build_object(
                        'source_context', mf.source_context,
                        'source_entity', mf.source_entity,
                        'label', mf.label,
                        'confidence', cl.code,
                        'symbol', ms.code
                    )
                )
            ), '[]'::json)
        )
        FROM geologic_mapping.geologic_map_feature mf
        JOIN basedata.confidence_level cl ON mf.confidence_id = cl.id
        JOIN geologic_mapping.map_symbol ms ON mf.symbol_id = ms.id
        WHERE mf.map_id = p_map_id
    );
END;
$$ LANGUAGE plpgsql STABLE;

WKT Export (for QGIS import):

CREATE OR REPLACE FUNCTION geologic_mapping_api.map_features_as_wkt(
    p_map_id UUID
) RETURNS TABLE (
    feature_id UUID,
    geometry_wkt TEXT,
    source_context TEXT,
    source_entity TEXT,
    label TEXT
) AS $$
BEGIN
    RETURN QUERY
    SELECT 
        mf.id,
        ST_AsText(mf.geometry),
        mf.source_context,
        mf.source_entity,
        mf.label
    FROM geologic_mapping.geologic_map_feature mf
    WHERE mf.map_id = p_map_id;
END;
$$ LANGUAGE plpgsql STABLE;

Performance Optimization Patterns

1. STABLE Functions

Mark all spatial query functions as STABLE:

  • Allows result caching within transaction
  • Enables query planner optimizations
  • Safe for read-only spatial queries
CREATE OR REPLACE FUNCTION schema_api.spatial_query(...)
RETURNS ... AS $$
...
$$ LANGUAGE plpgsql STABLE;  -- Not VOLATILE

2. GIST Index Usage

Verify index usage with EXPLAIN:

EXPLAIN (ANALYZE, BUFFERS)
SELECT geologic_mapping_api.features_within_polygon('POLYGON(...)');

-- Should show: Index Scan using idx_map_feature_geometry on geologic_map_feature

3. Geography vs Geometry

Use GEOGRAPHY for distance queries:

-- Accurate distance on Earth's surface (meters)
ST_Distance(geom1::geography, geom2::geography)

-- Fast planar distance (degrees or projected units)
ST_Distance(geom1, geom2)

Use GEOMETRY for containment/intersection:

-- Efficient GIST index usage
ST_Intersects(geom1, geom2)
ST_Within(geom1, geom2)

4. Bounding Box Pre-Filter

For complex queries, use && operator for fast GIST index scan:

WHERE geom1 && geom2  -- Bounding box overlap (GIST indexed)
  AND ST_Intersects(geom1, geom2);  -- Exact intersection

SRID Transformation Strategy

1. Store in WGS84 (4326) by default:

CREATE TABLE structural_geology.fault (
    trace_geometry GEOMETRY(LineString, 4326)
);

2. Transform on query if needed:

-- Query in UTM Zone 54N (32654)
SELECT * FROM structural_geology.fault
WHERE ST_Intersects(
    ST_Transform(trace_geometry, 32654),
    ST_GeomFromText('POLYGON(...)', 32654)
);

3. Per-map SRID storage:

-- Maps can define native CRS
SELECT crs_epsg FROM geologic_mapping.geologic_map WHERE id = '...';

-- Transform feature geometry to map's CRS on insert
INSERT INTO geologic_mapping.geologic_map_feature (geometry, ...)
VALUES (ST_Transform(source_geometry, target_map_srid), ...);

Implementation Rules

1. Naming Convention

Spatial API functions follow pattern:

<schema>_api.<entity>_<operation>_<spatial_relation>(parameters)

Examples:

  • geologic_mapping_api.features_within_extent()
  • geologic_mapping_api.features_near_point()
  • structural_geology_api.faults_crossing_polygon()

2. Function Signature Pattern

CREATE OR REPLACE FUNCTION <schema>_api.<function_name>(
    -- Geometry input as WKT (user-friendly)
    p_geometry_wkt TEXT,
    p_srid INTEGER DEFAULT 4326,
    
    -- Query-specific parameters
    p_distance_meters DECIMAL DEFAULT 1000,
    
    -- Filters (optional)
    p_source_context TEXT DEFAULT NULL
) RETURNS SETOF <schema>.<table> AS $$
DECLARE
    v_geometry GEOMETRY;
BEGIN
    -- Parse WKT
    v_geometry := ST_GeomFromText(p_geometry_wkt, p_srid);
    
    -- Spatial query with GIST index
    RETURN QUERY
    SELECT t.*
    FROM <schema>.<table> t
    WHERE ST_<SpatialFunction>(t.geometry_column, v_geometry)
      AND (p_source_context IS NULL OR t.source_context = p_source_context);
END;
$$ LANGUAGE plpgsql STABLE;

3. Error Handling

BEGIN
    -- Validate WKT
    BEGIN
        v_geometry := ST_GeomFromText(p_geometry_wkt, p_srid);
    EXCEPTION WHEN OTHERS THEN
        RAISE EXCEPTION 'Invalid WKT geometry: %', p_geometry_wkt
            USING ERRCODE = '22P02';  -- Invalid text representation
    END;
    
    -- Validate SRID
    IF p_srid NOT IN (SELECT srid FROM spatial_ref_sys) THEN
        RAISE EXCEPTION 'Invalid SRID: %', p_srid
            USING ERRCODE = 'P0001';
    END IF;
END;

4. Return Type Consistency

Option A: Return table row type (preferred for single entity):

RETURNS SETOF geologic_mapping.geologic_map_feature

Option B: Return custom TABLE type (for enriched queries):

RETURNS TABLE (
    feature_id UUID,
    distance_meters DECIMAL,
    label TEXT
)

Option C: Return JSON (for GeoJSON export):

RETURNS JSON

Consequences

Positive

  • Consistent API: Spatial queries follow same pattern as CRUD
  • Performance: Direct GIST index usage, no network overhead
  • Type Safety: Returns strongly-typed table rows
  • SRID Handling: Automatic coordinate transformation
  • Abstraction: Application doesn't need PostGIS knowledge
  • Caching: STABLE functions allow query planner optimizations

Negative

  • Boilerplate: Must define function for each spatial pattern
  • Limited Flexibility: Complex ad-hoc queries may need custom SQL
  • Version Coupling: Tied to PostgreSQL/PostGIS versions

Notes

Spatial query functions are read-only operations (no INSERT/UPDATE/DELETE). They should be marked STABLE to enable query planner optimizations and result caching within transactions.

For write operations with spatial validation (e.g., "ensure fault trace doesn't cross map boundary"), use CHECK constraints or BEFORE INSERT triggers with spatial predicates.

The GEOGRAPHY type uses spherical/geodetic calculations (accurate for global datasets), while GEOMETRY uses planar calculations (faster for localized datasets). Default to GEOMETRY for map-scale data unless computing true distances across large areas.
