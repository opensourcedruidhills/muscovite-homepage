Polymorphic GeologicMapFeature Bridge for Cross-Domain Feature Documentation

Polymorphic GeologicMapFeature Bridge for Cross-Domain Feature Documentation

Date: 2026-01-08
Status: Accepted
Deciders: System Architect, DDD Expert, GIS Team
Tags: ddd, bounded-context, polymorphism, geologic-mapping, fgdc

Context and Problem Statement

Geologic maps document observations of features from multiple domain contexts (faults from StructuralGeology, volcanoes from Volcanology, deposits from EconomicGeology). Each feature has domain-specific properties (kinematics, eruption date, ore grade), but the map layer needs a unified way to:

  1. Associate features with map symbols (FaultSymbol, VolcanicSymbol, etc.)
  2. Store observed geometries (may differ from canonical feature geometry)
  3. Track confidence level (certain/approximate/inferred)
  4. Query "all features on this map" without JOIN to every feature domain

How can we bridge feature domains and the GeologicMapping context without violating DDD bounded context principles (no cross-context foreign keys)?

Decision Drivers

  • DDD Bounded Context Integrity: Domain contexts own features, GeologicMapping context documents observations
  • No Cross-Context FKs: ADR-20260106 forbids foreign keys between bounded contexts
  • Polymorphic References: Map features can reference Fault, Volcano, Deposit, Fold, etc.
  • Spatial Queries: Need "features within extent" without domain-specific JOINs
  • FGDC Compliance: USGS standard requires symbol assignment and confidence tracking
  • Audit Trail: Track who observed/deduced/suspected each feature placement
  • Flexible Geometry: Map may show simplified/approximated geometry vs. canonical feature

Considered Options

Option 1: Direct Foreign Keys to All Feature Types (REJECTED)

CREATE TABLE geologic_map_feature (
    fault_id UUID REFERENCES structural_geology.fault(id),
    volcano_id UUID REFERENCES volcanology.volcano(id),
    deposit_id UUID REFERENCES economic_geology.mineral_deposit(id),
    -- ... 50+ feature types
);

Pros:

  • Referential integrity enforced by database

Cons:

  • Violates DDD: Cross-context FKs couple domains
  • Unmaintainable: 50+ nullable columns for each feature type
  • Schema explosion: Adding new feature type requires DDL migration
  • Complex queries: Must check 50 columns to find which type is set

Option 2: Single Table Inheritance with Feature Union (REJECTED)

CREATE TABLE map_feature (
    feature_type TEXT,
    fault_trace GEOMETRY,
    volcano_outline GEOMETRY,
    deposit_outline GEOMETRY,
    -- ... union of all feature properties
);

Pros:

  • Single table query

Cons:

  • Massive schema: Hundreds of columns for all feature types
  • NULL proliferation: 95% of columns NULL for any given row
  • Tight coupling: Map layer knows about all domain internals
  • No domain encapsulation: Feature logic duplicated

Option 3: Polymorphic Bridge with Source Context Tracking (ACCEPTED)

CREATE TABLE geologic_mapping.geologic_map_feature (
    id UUID PRIMARY KEY,
    map_id UUID NOT NULL REFERENCES geologic_mapping.geologic_map(id),
    
    -- Polymorphic source reference (NO FK constraint)
    source_context TEXT NOT NULL,      -- 'StructuralGeology'
    source_entity TEXT NOT NULL,       -- 'Fault'
    source_entity_id UUID NOT NULL,    -- ID in source context
    
    -- Map-specific data
    symbol_id UUID NOT NULL REFERENCES geologic_mapping.map_symbol(id),
    geometry GEOMETRY NOT NULL,        -- PostGIS
    confidence_id UUID NOT NULL REFERENCES basedata.confidence_level(id),
    
    -- Audit
    observed_by TEXT,
    observed_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_map_feature_source 
    ON geologic_mapping.geologic_map_feature(source_context, source_entity, source_entity_id);
CREATE INDEX idx_map_feature_geometry 
    ON geologic_mapping.geologic_map_feature USING GIST(geometry);

Pros:

  • DDD Compliant: No cross-context FKs
  • Extensible: Adding new feature types requires no schema changes
  • Single query: All map features in one table
  • Spatial indexes: Fast "features within extent" queries
  • Domain independence: Feature domains evolve without map layer changes
  • Flexible geometry: Map can store observed geometry separate from canonical

Cons:

  • No database-enforced referential integrity on source_entity_id
  • Requires application-layer validation of source references
  • JOINs to source tables require dynamic query construction

Decision Outcome

Chosen Option: Polymorphic bridge with source context tracking (Option 3)

The GeologicMapFeature entity acts as a weak reference bridge between the GeologicMapping context and all feature-owning domains, respecting bounded context boundaries while enabling unified spatial queries.

DSL Definition

GeologicMapping Context (src/examples/geosciences/geologic-mapping.ddd):

ENTITY GeologicMapFeature IDENTIFIED_BY id (
    id UUID NOT NULL,
    map_id UUID NOT NULL,
    
    -- Polymorphic source (no FK constraint)
    source_context TEXT NOT NULL,
    source_entity TEXT NOT NULL,
    source_entity_id UUID NOT NULL,
    
    -- Cartographic properties
    symbol_id UUID NOT NULL,
    geometry GEOMETRY NOT NULL,
    confidence_id UUID NOT NULL,
    
    -- Metadata
    label TEXT,
    notes TEXT,
    observed_by TEXT,
    observed_at TIMESTAMPTZ NOT NULL DEFAULT now()
)
RELATIONSHIPS {
    map -> GeologicMap REQUIRED ONE;
    symbol -> MapSymbol REQUIRED ONE;
    confidence -> ConfidenceLevel REQUIRED ONE;
}
DOC "Bridge entity linking domain features to map symbols with spatial geometry";

Polymorphic Reference Pattern

Source Context Values (controlled vocabulary):

  • StructuralGeology → Fault, Fold, Lineation, Foliation
  • Volcanology → Volcano, LavaFlow, PyroclasticDeposit
  • EconomicGeology → MineralDeposit, AlterationZone, Vein
  • Geomorphology → Landform, Glacier, Moraine
  • Stratigraphy → LithostratigraphicUnit, Facies
  • Petrology → Outcrop, RockUnit
  • FieldObservation → Station, Outcrop

Source Entity Values (entity type names):

  • Entity name from DSL: Fault, Volcano, MineralDeposit, etc.

Source Entity ID (UUID):

  • Primary key value from source entity table

Validation Strategy

Application-Layer Validation (API function):

CREATE OR REPLACE FUNCTION geologic_mapping_api.map_feature_insert(
    p_map_id UUID,
    p_source_context TEXT,
    p_source_entity TEXT,
    p_source_entity_id UUID,
    p_symbol_id UUID,
    p_geometry TEXT,  -- WKT
    p_confidence_id UUID,
    p_label TEXT DEFAULT NULL,
    p_notes TEXT DEFAULT NULL,
    p_observed_by TEXT DEFAULT current_user
) RETURNS geologic_mapping.geologic_map_feature AS $$
DECLARE
    v_result geologic_mapping.geologic_map_feature%ROWTYPE;
    v_source_exists BOOLEAN;
BEGIN
    -- Validate source context + entity combination
    IF p_source_context NOT IN (
        'StructuralGeology', 'Volcanology', 'EconomicGeology',
        'Geomorphology', 'Stratigraphy', 'Petrology', 'FieldObservation'
    ) THEN
        RAISE EXCEPTION 'Invalid source_context: %', p_source_context
            USING ERRCODE = 'P0001';
    END IF;
    
    -- Validate source entity exists (dynamic query)
    EXECUTE format(
        'SELECT EXISTS(SELECT 1 FROM %I.%I WHERE id = $1)',
        lower(regexp_replace(p_source_context, '([A-Z])', '_\1', 'g')),  -- CamelCase → snake_case
        lower(regexp_replace(p_source_entity, '([A-Z])', '_\1', 'g'))
    ) INTO v_source_exists USING p_source_entity_id;
    
    IF NOT v_source_exists THEN
        RAISE EXCEPTION 'Source entity not found: %.% with id %',
            p_source_context, p_source_entity, p_source_entity_id
            USING ERRCODE = '23503';  -- FK violation code
    END IF;
    
    -- Insert with geometry transform to map's SRID
    INSERT INTO geologic_mapping.geologic_map_feature (
        map_id, source_context, source_entity, source_entity_id,
        symbol_id, geometry, confidence_id,
        label, notes, observed_by, observed_at
    ) VALUES (
        p_map_id,
        p_source_context,
        p_source_entity,
        p_source_entity_id,
        p_symbol_id,
        ST_GeomFromText(p_geometry, 4326),
        p_confidence_id,
        p_label,
        p_notes,
        p_observed_by,
        now()
    )
    RETURNING * INTO v_result;
    
    RETURN v_result;
END;
$$ LANGUAGE plpgsql;

Query Patterns

1. All Features on Map (spatial query):

SELECT * FROM geologic_mapping.geologic_map_feature
WHERE map_id = '...'
  AND ST_Intersects(geometry, ST_MakeEnvelope(...));

2. Features by Type:

SELECT * FROM geologic_mapping.geologic_map_feature
WHERE map_id = '...'
  AND source_context = 'StructuralGeology'
  AND source_entity = 'Fault';

3. Join to Source Feature (enriched data):

-- Faults on map with kinematics
SELECT 
    mf.id AS map_feature_id,
    mf.geometry AS observed_geometry,
    mf.label,
    f.name AS fault_name,
    f.fault_type,
    f.sense_of_slip
FROM geologic_mapping.geologic_map_feature mf
JOIN structural_geology.fault f ON mf.source_entity_id = f.id
WHERE mf.map_id = '...'
  AND mf.source_context = 'StructuralGeology'
  AND mf.source_entity = 'Fault';

4. GeoJSON Export with Source Details:

SELECT json_build_object(
    'type', 'FeatureCollection',
    'features', json_agg(
        json_build_object(
            'type', 'Feature',
            'id', id,
            'geometry', ST_AsGeoJSON(geometry)::json,
            'properties', json_build_object(
                'source_context', source_context,
                'source_entity', source_entity,
                'source_id', source_entity_id,
                'symbol', symbol_code,
                'confidence', confidence_code,
                'label', label
            )
        )
    )
)
FROM geologic_mapping.geologic_map_feature
WHERE map_id = '...';

Implementation Rules

1. No Cross-Context FKs

GeologicMapFeature MUST NOT have foreign key constraints to feature tables in other contexts:

-- CORRECT
source_entity_id UUID NOT NULL

-- INCORRECT (violates bounded context integrity)
fault_id UUID REFERENCES structural_geology.fault(id)

2. Application-Layer Validation

API layer MUST validate source references exist before INSERT:

  • Use dynamic SQL: EXECUTE format('SELECT EXISTS...')
  • Raise exception with ERRCODE = '23503' (FK violation) if not found
  • Log validation failures for debugging

3. Context/Entity Controlled Vocabulary

Maintain canonical list of valid combinations:

  • Document in ADR (this file)
  • Validate in API function
  • Consider lookup table if list grows large (>20 contexts)

4. Audit Trail Required

All map features MUST track:

  • observed_by (who documented the feature)
  • observed_at (when)
  • Optional: confidence_id (observation certainty)

5. Geometry Independence

Map feature geometry CAN differ from canonical feature geometry:

  • Fault canonical: High-precision GPS trace (10m accuracy)
  • Map feature: Simplified for 1:50,000 scale display
  • Both are valid for their purpose

Consequences

Positive

  • DDD Compliant: Respects bounded context boundaries
  • Extensible: New feature types require no schema changes
  • Performance: Single-table spatial queries with GIST index
  • Audit Trail: Tracks observation metadata
  • Flexible: Supports observed vs. canonical geometry
  • FGDC Compliant: Symbol assignment and confidence tracking
  • Domain Independence: Feature domains evolve independently

Negative

  • No DB Referential Integrity: Source references not enforced by PostgreSQL
  • Dynamic Queries: JOINs to source tables require runtime query construction
  • Application Validation: Must validate source references in API layer
  • Orphan Risk: If source feature deleted, map feature becomes orphan (mitigated by audit trail)
  • Query Complexity: Enriched queries require dynamic SQL or UNION across feature types

Mitigation Strategies

Orphan Detection:

-- Find map features with missing source entities
SELECT DISTINCT source_context, source_entity
FROM geologic_mapping.geologic_map_feature mf
WHERE NOT EXISTS (
    SELECT 1 FROM information_schema.tables t
    WHERE t.table_schema = lower(regexp_replace(mf.source_context, '([A-Z])', '_\1', 'g'))
      AND t.table_name = lower(regexp_replace(mf.source_entity, '([A-Z])', '_\1', 'g'))
);

Cascade Delete Policy:

  • Feature domains SHOULD emit domain events on deletion
  • GeologicMapping context SHOULD subscribe to deletion events
  • Option 1: Mark map features as "source_deleted"
  • Option 2: Soft-delete map features
  • Option 3: Retain as historical record (RECOMMENDED for audit trail)

AI Agent Guidance

When This Decision Applies

  • Always: Linking features from domain contexts to GeologicMapping
  • Always: Cross-context associations without FKs
  • Always: Polymorphic references to multiple entity types

When This Decision Does NOT Apply

  • Intra-context relationships (use normal FKs)
  • References within same bounded context
  • References to Basedata shared kernel (use normal FKs)

How It's Enforced

  1. DSL Validation: Generator checks for cross-context FK attempts
  2. API Template: API functions include source validation logic
  3. Index Generation: BTREE index on (source_context, source_entity, source_entity_id)
  4. GIST Index: Spatial index on geometry column

Notes

This pattern is sometimes called "Weak Reference" or "Polymorphic Association" in DDD literature. It trades database referential integrity for bounded context autonomy—a fundamental DDD tradeoff.

The pattern enables the GeologicMapping context to fulfill its responsibility ("document observed features on maps") without becoming tightly coupled to every feature-owning domain. Each domain remains independently deployable and evolvable.

For feature-rich applications requiring frequent source data enrichment, consider implementing a View Service that provides pre-joined queries across contexts (read-only materialized views or caching layer).
