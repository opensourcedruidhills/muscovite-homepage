Flutter as Generated Frontend Target

Flutter as Generated Frontend Target

  • Status: superseded
  • Superseded by: Deferred to post-1.x — product focus is Qt Widgets + QML (2026-02-17)
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: frontend, flutter, dart, generator, multi-platform

Context and Problem Statement

Muscovite currently generates two frontend dialects from a single .ddd model:

  • Qt Widgets: C++ QWidget-based desktop dialogs
  • Qt Quick/QML: C++ ViewModel + QML declarative UI

Both are Qt-based and share the same C++ DBA/gRPC layer (ADR 20260206 multi-frontend architecture). However, Qt's mobile story (Qt for Android/iOS) carries significant licensing cost and friction. A Flutter target would provide:

  • Native mobile (Android, iOS) without Qt licensing
  • Desktop (Windows, Linux, macOS) as a bonus
  • Web (Flutter Web) with caveats
  • A Dart-based frontend that consumes the same .proto contracts

Question: Should Muscovite add Flutter/Dart as a third generated frontend target?

Decision Drivers

  • Mobile reach without Qt licensing burden
  • Reuse existing .proto service contracts (no new API surface)
  • Maintain single source of truth (.ddd → all layers)
  • Prove target independence of the IR (per VISION.md)
  • Generator effort proportional to value (~1,000–5,000 lines of Racket per VISION.md estimate)
  • Dart's type system is strong enough for end-to-end type safety

Considered Options

  • Option 1: Add Flutter as a generated frontend target
  • Option 2: Add React/TypeScript instead
  • Option 3: Stay Qt-only, use Qt for Android/iOS

Decision Outcome

Chosen option: Option 1 — Add Flutter as a generated frontend target, because it validates target independence with minimal infrastructure, covers mobile without Qt licensing, and Dart's generated gRPC stubs are first-class.

Generator Architecture

Domain Model (.ddd)
  ↓
Racket Generators
  ├─→ PostgreSQL + Sqitch          ← unchanged
  ├─→ C++ DBA Layer                ← unchanged
  ├─→ gRPC API (.proto)            ← unchanged (consumed by Dart codegen)
  ├─→ Qt Widgets                   ← unchanged
  ├─→ Qt Quick/QML                 ← unchanged
  └─→ Flutter/Dart (NEW)
      ├─→ Dart gRPC client stubs   ← from protoc --dart_out
      ├─→ Dart service wrappers    ← generated from IR
      ├─→ Dart entity models       ← generated from IR
      └─→ Flutter UI widgets       ← generated from IR

Generation Flag

racket generate.rkt --project ../examples/fhir-health/project.ddd --flutter

Output Structure

generated-{project}/
├── dba/                          ← shared (C++ backend)
├── grpc/                         ← shared (C++ backend)
├── qt-widgets/                   ← existing
├── qt-quick/                     ← existing
└── flutter/                      ← NEW
    ├── pubspec.yaml              ← generated, pins grpc + protobuf
    ├── lib/
    │   ├── generated/
    │   │   ├── *.pb.dart         ← protoc output
    │   │   └── *.pbgrpc.dart     ← protoc output
    │   ├── models/
    │   │   └── patient.dart      ← entity model (from IR)
    │   ├── services/
    │   │   └── patient_service.dart  ← service wrapper
    │   └── ui/
    │       ├── patient_list.dart
    │       └── patient_form.dart
    └── test/
        └── patient_service_test.dart

Relationship to Existing Frontends

Aspect Qt Widgets Qt Quick/QML Flutter
Platform Desktop Desktop + embedded Mobile + desktop + web
Language C++ C++ + QML Dart
Backend link Direct C++ Direct C++ gRPC (network)
Hot reload No Partial (QML) Yes (full)
Build system CMake CMake pub/flutter
DBA dependency Compiled in Compiled in None (gRPC only)

Critical difference: Flutter clients are network clients — they don't link C++ DBA. They consume gRPC exclusively. This is the first frontend that validates the gRPC API as a complete, standalone integration surface.

Positive Consequences

  • Proves IR target independence (VISION.md thesis)
  • Mobile reach without Qt licensing
  • First non-C++ frontend validates gRPC completeness
  • Dart codegen from .proto is mature (protoc_plugin package)
  • Flutter hot reload dramatically shortens UI iteration
  • Opens the door for other Dart/gRPC consumers

Negative Consequences

  • New Racket generator (~2,000–4,000 lines estimated)
  • New type mapping column in type-registry.rkt
  • Flutter build toolchain is a separate dependency (not Conan/CMake)
  • Testing: Flutter widget tests are a different paradigm from Catch2/QTest
  • No shared C++ validation — validation must be duplicated or deferred to server

Open Questions (Future ADRs)

  • Validation strategy: Client-side Dart validation vs. server-only? (Likely: generate Dart validators from same IR invariants)
  • Offline support: gRPC requires network; offline-first Flutter needs local cache strategy
  • State management: Provider, Riverpod, or Bloc? (Generator must pick one)

Implementation Plan

  1. Add 'Dart to type-registry.rkt type mapping functions
  2. Create dslgen/generators/flutter/ directory with:
    • entity.rkt — Dart entity models
    • service.rkt — Dart service wrappers
    • ui.rkt — Flutter widget generation
    • pubspec.rkt — pubspec.yaml generation
  3. Wire --flutter flag into generate.rkt
  4. Validate with fhir-health example (Patient CRUD end-to-end)

AI Agent Guidance

When This Decision Applies

  • Adding Flutter/Dart generation to the Muscovite generator
  • Extending type-registry.rkt with Dart type mappings
  • Creating new generator files under dslgen/generators/flutter/
  • Generating pubspec.yaml, Dart models, services, or UI widgets from IR

When This Decision Does NOT Apply

  • Qt Widgets or QML frontend generation (unchanged)
  • C++ DBA/gRPC layer (unchanged — Flutter consumes gRPC, doesn't touch DBA)
  • PostgreSQL schema generation (unchanged)
  • Proto file generation (unchanged — Flutter uses existing .proto)

Enforcement

  • Flutter targets MUST consume gRPC — no direct C++ integration
  • Generated Dart code goes to generated-{project}/flutter/ only
  • Dart type mappings must be registered in type-registry.rkt
  • Never manually edit generated Dart files — fix dslgen/generators/flutter/*.rkt

Key Files

  • dslgen/model/type-registry.rkt — add Dart column
  • dslgen/generators/flutter/ — new generator directory
  • dslgen/generate.rkt — --flutter flag
  • generated-{project}/flutter/ — output directory
