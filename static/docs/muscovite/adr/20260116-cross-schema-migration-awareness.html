Cross-Schema Migration Awareness

Cross-Schema Migration Awareness

  • Status: accepted
  • Deciders: Architecture Team
  • Date: 2026-01-16
  • Tags: migration, schema, sqitch

Context and Problem Statement

The semantic diff pipeline correctly detects changes across multiple bounded contexts, but the SQL generator (sqitch-generator.rkt) receives only a single schema-name parameter. When migrations span multiple schemas (e.g., adding lookup tables in basedata while modifying entities in condition), the generator produces incorrect SQL that targets the wrong schema.

Example of the problem:

-- WRONG: Lookup tables should be in basedata schema, not condition
CREATE TABLE condition.condition_clinical_status (...)

Decision Drivers

  • Multi-context migrations are common when refactoring to lookup tables
  • Each bounded context maps to a PostgreSQL schema
  • Operations must know their target schema to generate correct SQL
  • Sqitch migrations should be schema-aware without manual intervention

Considered Options

  1. Pass schema name as parameter to each operation
  2. Derive schema from context UUID at generation time
  3. Embed schema name directly in operation structs

Decision Outcome

Chosen option: Embed schema name directly in operation structs

Each operation struct that generates DDL will include a schema-name field derived from the bounded context name at diff time. This ensures:

  • Schema information travels with the operation
  • No additional lookups needed at generation time
  • Clear provenance of where each operation targets

Implementation

  1. Extend operation structs in operations.rkt:

    (struct add-entity-op
  (uuid ctx-uuid agg-uuid name ent schema-name safety)
  #:transparent)

  2. Update diff-model.rkt to populate schema-name from bounded-context

  3. Update sqitch-generator.rkt to use operation's schema-name instead of global parameter

Affected Operations

Operation Schema Source
add-entity-op Context name → snake_case
remove-entity-op Context name → snake_case
rename-entity-op Entity's context
add-field-op Entity's context
remove-field-op Entity's context
add-relationship-op Source entity's context
add-check-op Entity's context
remove-check-op Entity's context

Consequences

Positive

  • Correct SQL generation for multi-schema migrations
  • No manual schema mapping required
  • Operations are self-contained with all needed metadata

Negative

  • Slightly larger operation structs
  • Need to update all operation creation sites

AI Agent Guidance

  • When generating migration operations, always derive schema from the bounded context name
  • Schema name = (string->symbol (string-downcase (symbol->string context-name)))
  • The sqitch-generator should iterate operations grouped by schema, not assume single schema
