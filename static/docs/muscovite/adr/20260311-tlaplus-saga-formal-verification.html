ADR: TLA+ Formal Verification for Generated Saga Specifications

ADR: TLA+ Formal Verification for Generated Saga Specifications

  • Date: 2026-03-11
  • Status: accepted
  • Deciders: Johannes Lochmann
  • Category: verification, sagas, tooling

Context

Muscovite generates saga orchestrations from .ddd SAGA declarations: PostgreSQL state tables, C++ orchestrator/processor code, gRPC management RPCs, and Qt visualization. The generated code is structurally correct by construction — but three classes of bugs escape structural correctness:

  1. Compensation order violations — partial failure mid-chain leaves steps uncompensated
  2. Liveness failures — saga instances stuck in in_progress after timeout + redelivery storms
  3. Correlation isolation breaches — concurrent instances with same correlation_id cross-contaminate state under NATS at-least-once redelivery

These bugs are invisible to unit tests (which test single steps) and integration tests (which test happy paths). They manifest only under specific interleaving of concurrent events — the domain of formal verification.

Decision

Generate TLA+ specifications from saga IR structs and verify them with the TLC model checker as a release quality gate.

Architecture

Component Location Generated?
Saga spec (state machine) generated/{context}/tlaplus/{SagaName}.tla Yes — from saga IR
Property module generated/{context}/tlaplus/{SagaName}Props.tla Yes — from saga IR
TLC config generated/{context}/tlaplus/{SagaName}.cfg Yes — from saga IR
Event bus model dslgen/generators/tlaplus/lib/EventBus.tla No — hand-authored
TLC runner (CMake) cmake/TLAPlus.cmake No — framework tooling
tla2tools.jar tools/tla2tools.jar No — vendored binary

Properties Verified Per Saga

Property Type TLA+ Formula Pattern
Termination Liveness <>(\A i \in Instances: status[i] \in {"completed", "failed", "cancelled"})
Compensation order Safety [](compensating[i] => CompensationReverseOrder(i))
Correlation isolation Safety [](\A i, j \in Instances: i /= j /\ corr[i] /= corr[j] => Disjoint(steps[i], steps[j]))
No orphaned instances Liveness <>(\A i \in Instances: status[i] /= "in_progress")
Timeout progress Liveness [](step_timeout[i] => <>(status[i] \in {"compensating", "failed"}))

Event Bus Abstraction

The EventBus.tla module models NATS JetStream semantics at the right abstraction level:

  • At-least-once delivery: messages may be delivered multiple times
  • Unordered: no guaranteed ordering between subjects
  • Correlation routing: messages carry correlation_id, saga processor filters by it
  • NOT modeled: consumer groups, ack timeouts, backpressure (irrelevant to saga correctness)

This keeps the state space bounded while catching the bugs we care about.

Release Gate Integration

Gate 9: TLA+ model checking (conditional)
  - Scans generated/ for *.tla files
  - Runs: java -jar tools/tla2tools.jar -config {saga}.cfg {saga}.tla
  - fail if any property violated, skip if no .tla files exist
  - State space bound: configurable via MUSCOVITE_TLA_MAX_STATES (default: 1,000,000)

Consequences

Positive

  • Catches concurrency bugs before code generation — verification runs on the model, not the C++ runtime
  • Zero per-project effort — specs generated from the same .ddd developers already write
  • Executable documentation — TLA+ specs serve as formal semantics for saga behavior
  • Deterministic CI — TLC is deterministic (same spec → same result), unlike fuzz testing

Negative

  • Java dependency — TLC requires JRE 11+. Added to CI and developer environments.
  • State space limits — Sagas with 10+ steps and 3+ concurrent instances may exceed 1M states. Requires abstraction or symmetry reduction.
  • EventBus.tla maintenance — Hand-authored module must be updated if NATS semantics change (e.g., switching to exactly-once delivery).
  • Learning curve — Developers reading counterexample traces need basic TLA+ literacy. Mitigated by generating human-readable trace summaries.

Alternatives Considered

Alternative Why Rejected
Property-based testing (QuickCheck/Hypothesis) Tests interleavings randomly; TLA+ exhaustively enumerates them. PBT misses rare corner cases.
Alloy Better for structural modeling (ERDs, object graphs); weaker for temporal/liveness properties.
SPIN/Promela Equivalent power to TLA+, but less ecosystem support and no Lamport-style operator notation.
No formal verification Status quo. Acceptable for ≤3-step sagas; unacceptable for 9+ step sagas like MetamorphicPathway.
