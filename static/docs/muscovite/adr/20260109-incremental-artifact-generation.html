Incremental Artifact Generation via Semantic Diff

Incremental Artifact Generation via Semantic Diff

Technical Story: Leverage semantic diff output to regenerate only artifacts affected by actual model changes, reducing build times and unnecessary file writes.

Context and Problem Statement

Currently, all generated artifacts (DB scripts, C++ backend, gRPC code, Qt Widgets) are fully regenerated on each run, even if only a small subset of the model changed.

With thousands of generated files, this causes:

  • Long build times
  • High disk I/O
  • Unnecessary rebuilds of unchanged code
  • CMake/Ninja re-linking when source files are touched but unchanged

We already implement semantic diffing on the DDD model to detect incremental changes and derive migrations. We want to leverage the same information to regenerate only artifacts affected by actual model changes.

Decision Drivers

  • Reduce generation and build time
  • Avoid unnecessary file writes
  • Maintain deterministic, traceable generation
  • Preserve existing artifact mappings to semantic elements
  • Support cross-layer dependencies (DB, backend, gRPC, Qt)
  • Align with migration workflow (ADRs 3-6)

Considered Options

  1. Full regeneration (current approach)

    • Simple, always up-to-date
    • Too slow for large projects → rejected

  2. Timestamp / file change detection

    • Detect changes based on DSL file modification times
    • Does not account for semantic equivalence → can regenerate unnecessarily → rejected

  3. Git-based change detection

    • Use Git diff to detect changed files
    • Only detects textual changes → insufficient for semantic renames or moves → rejected

  4. Semantic diff-based incremental generation (accepted)

    • Leverage existing semantic diff / Migration IR
    • Map changed elements to artifacts
    • Regenerate only affected files

Decision Outcome

Use the semantic diff output to trigger incremental artifact generation:

  1. Compute semantic diff between old and new model IR → Migration IR / Change Set
  2. Map changed elements → generated artifacts
  3. Regenerate only affected artifacts
  4. Update manifest of artifact hashes and element mappings for next run
  5. Propagate changes transitively for dependent artifacts (e.g., value objects used by multiple entities)

Positive Consequences

  • Drastically reduced generation and build times
  • Deterministic and traceable artifact updates
  • Fully aligned with migration and semantic diff workflow
  • Supports multi-layer generation: DB, C++, gRPC, Qt
  • Minimizes CMake rebuild triggers

Negative Consequences

  • Requires maintaining artifact → semantic element mapping
  • Dependency propagation must be handled carefully (e.g., nested value objects)
  • Slightly more complex generation logic than full regeneration

Implementation

Phase 1: Artifact Manifest

Create a manifest file tracking all generated artifacts:

;; src/dslgen/manifest/artifact-manifest.rkt

(struct artifact-entry
  (path           ; String - relative path to generated file
   element-uuids  ; (Listof UUID) - contributing semantic elements
   content-hash   ; String - SHA-256 of generated content
   generator      ; Symbol - which generator produced this
   timestamp)     ; Integer - generation timestamp
  #:transparent)

(struct artifact-manifest
  (version        ; String - manifest format version
   model-hash     ; String - hash of source IR
   entries)       ; (HashTable String artifact-entry) - path -> entry
  #:transparent)

;; Persistence
(define (write-manifest manifest path)
  ...)

(define (read-manifest path)
  ...)

Define mappings from semantic elements to generated artifacts:

(define element->artifacts (make-hash))

Phase 3: Incremental Generation Algorithm

(define (generate-incremental old-ir new-ir output-dir)
  ;; 1. Load previous manifest (or create empty)
  (define manifest (read-manifest (build-path output-dir ".artifact-manifest")))
  
  ;; 2. Compute semantic diff
  (define change-set (diff-model old-ir new-ir))
  
  ;; 3. Identify affected elements (with transitive closure)
  (define affected-uuids (compute-affected-elements change-set))
  
  ;; 4. Map to affected artifacts
  (define affected-paths (uuids->artifact-paths affected-uuids manifest))
  
  ;; 5. Regenerate only affected artifacts
  (for ([path affected-paths])
    (define entry (hash-ref (artifact-manifest-entries manifest) path #f))
    (define new-content (regenerate-artifact path new-ir))
    (define new-hash (sha256 new-content))
    
    ;; Skip write if content unchanged (avoids rebuild triggers)
    (unless (and entry (equal? new-hash (artifact-entry-content-hash entry)))
      (write-file path new-content))
    
    ;; Update manifest entry
    (update-manifest! manifest path new-hash ...))
  
  ;; 6. Handle deletions (elements removed from model)
  (for ([deleted (change-set-deletions change-set)])
    (define paths (uuid->artifact-paths (deleted-uuid deleted)))
    (for ([path paths])
      (delete-file path)
      (remove-from-manifest! manifest path)))
  
  ;; 7. Save updated manifest
  (write-manifest manifest (build-path output-dir ".artifact-manifest")))

Phase 4: Transitive Dependency Propagation

When an element changes, all dependent artifacts must be regenerated:

(define (compute-affected-elements change-set)
  (define directly-affected 
    (append (map change-uuid (change-set-additions change-set))
            (map change-uuid (change-set-modifications change-set))
            (map change-uuid (change-set-renames change-set))))
  
  ;; Expand to include dependents
  (define all-affected (mutable-set))
  (for ([uuid directly-affected])
    (set-add! all-affected uuid)
    ;; Add all elements that reference this one
    (for ([dependent (get-dependents uuid)])
      (set-add! all-affected dependent)))
  
  (set->list all-affected))

Dependency examples:

  • Entity field type changed → regenerate entity + all referencing entities
  • Value object changed → regenerate all entities embedding it
  • Relationship changed → regenerate both sides + junction table

Phase 5: Layer-Specific Regeneration

Each generator layer handles incremental updates:

Layer Affected By Artifacts
Sqitch Entity/field/index/FK changes deploy/revert/verify SQL

gRPC Proto Entity/field/service changes .proto files
C++ Service Proto changes service impl .cpp/.hpp
C++ DBA Entity/field changes repository .cpp/.hpp
Qt Models Entity/field changes model .cpp/.hpp
Qt Widgets Entity/relationship changes widget .cpp/.hpp

CLI Integration

# Full regeneration (ignore manifest, regenerate everything)
racket generate.rkt --project project.ddd --full

# Incremental regeneration (default, uses manifest)
racket generate.rkt --project project.ddd

# Show what would be regenerated without doing it
racket generate.rkt --project project.ddd --dry-run

# Force regeneration of specific layers
racket generate.rkt --project project.ddd --force sqitch,api

AI Agent Guidance

When This Decision Applies

  • When implementing code generation optimizations
  • When adding new generator layers
  • When modifying semantic diff output
  • When debugging "why wasn't X regenerated" issues

When This Decision Does NOT Apply

  • First-time generation (no previous manifest)
  • When --full flag is specified
  • When manifest is corrupted or missing

Enforcement

  • All generators MUST register their element→artifact mappings
  • Artifact writes MUST go through manifest-aware writer
  • Content hash comparison MUST happen before file write
  • Deleted elements MUST trigger artifact cleanup

Acceptance Criteria

  2. Unchanged artifacts NOT rewritten (verified by timestamp)
  3. Changed elements trigger regeneration of all dependent artifacts
  4. Deleted elements trigger artifact cleanup
  5. --dry-run shows affected files without writing
  6. --full bypasses manifest and regenerates everything
  7. Generation time reduced by >50% for small model changes
