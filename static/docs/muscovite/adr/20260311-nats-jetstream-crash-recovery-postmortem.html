ADR: NATS JetStream Crash Recovery Postmortem

ADR: NATS JetStream Crash Recovery Postmortem

  • Status: accepted
  • Date: 2026-03-11
  • Deciders: Johannes Lochmann
  • Category: infrastructure

Context

On 2026-03-11, a VM crash during an active multi-agent iteration (v1.63.0) exposed five compounding bugs in the NATS JetStream coordinator infrastructure. The crash caused complete loss of coordinator state, resulting in duplicate FTR assignments (two workers both claimed FTR-350) and an open PR (#132) being invisible to the restarted coordinator.

Recovery took approximately 90 minutes of manual debugging and multiple failed NATS restart attempts.

Root Causes (5 compounding bugs)

Bug 1: JetStream store_dir in /tmp/ (CRITICAL)

# nats-server.conf (BEFORE)
jetstream {
  store_dir: /tmp/muscovite-nats-agent   # tmpfs — wiped on reboot/crash
}

/tmp/ on most Linux distributions (including openSUSE Tumbleweed) is either a tmpfs mount (RAM-backed) or cleaned by systemd-tmpfiles-clean.timer. A VM crash instantly destroyed all JetStream state: streams, KV buckets, consumer offsets.

The storage: file setting on streams and KV buckets was meaningless because the "files" lived on volatile storage.

Bug 2: ftr-claims KV bucket TTL of 4 hours

# setup-streams.sh (BEFORE)
create_or_update_kv "ftr-claims" "4h" "..."

Even if the store had survived, any FTR claim older than 4 hours would have expired silently. Long-running FTRs (like FTR-350, a Go extractor taking multiple hours) would lose their claims. The heartbeat mechanism was supposed to refresh TTL, but heartbeats don't survive process crashes.

Bug 3: Events published via nc.publish(), not js.publish()

# coordinator.py _flush_events() (BEFORE)
await self._nc.publish(subject, payload)  # fire-and-forget, NOT persisted

Events were published using regular NATS publish (fire-and-forget) instead of JetStream publish (persisted to stream). This meant the AGENT-EVENTS stream never captured any events — crash recovery replay had nothing to replay.

Bug 4: Stream subject mismatch

# setup-streams.sh (BEFORE)
create_or_update_stream "AGENT-EVENTS" "muscovite.agent.>" ...
# coordinator.py (actual subject used)
subject = f"coord.event.{event['type']}"  # e.g., coord.event.ftr.assigned

The AGENT-EVENTS stream listened on muscovite.agent.> but events were published to coord.event.*. Even if js.publish() had been used, the stream would not have captured the events due to subject mismatch.

Bug 5: No replay logic in coordinator startup

# coordinator.py start() (BEFORE)
self._state = IterationState(dg)
# ... immediately starts serving with empty state

The apply_event() replay method existed (FTR-372) but was never called during start(). The coordinator always started with a fresh IterationState, losing all worker registrations, FTR assignments, and progress history.

Compounding Effect

These five bugs created a cascade where each one would have independently prevented crash recovery, but together they made the failure completely silent:

  1. Store wiped (Bug 1) → no data to recover
  2. Even if store survived, claims would expire (Bug 2)
  3. Even if claims survived, no events were persisted (Bug 3)
  4. Even if js.publish was used, wrong subject (Bug 4)
  5. Even if events existed, replay was never called (Bug 5)

Additional Operational Issue: System Service vs User Service

The NATS server runs as a system service (nats-muscovite-agent.service) under the nats user, with config at /etc/nats/muscovite-agent.conf. The repo config at agents/nats/nats-server.conf must be manually copied via sudo cp after changes. This was not documented and caused multiple failed restart attempts:

  • Relative store_dir paths don't work (service CWD is not the repo)
  • The nats user cannot traverse /home/jolo/ (mode 700)
  • /var/lib/nats/muscovite-agent requires nats:nats ownership

Decision

Fixes Applied (commit 4e836b89)

  1. store_dir → /var/lib/nats/muscovite-agent (persistent, nats-owned)
  2. ftr-claims TTL → 0 (no expiry, explicit release only)
  3. Event publishing → js.publish() with fallback to nc.publish()
  4. Stream subject → coord.event.> (matches actual event subjects)
  5. Crash recovery → _replay_events_from_jetstream() called in start()

Operational Documentation Required

  • AGENTS.md: System service management, config copy workflow
  • doc/COORDINATOR.md: Crash recovery section, durability guarantees
  • Config sync: sudo cp agents/nats/nats-server.conf /etc/nats/muscovite-agent.conf

Consequences

Positive

  • Coordinator survives VM crashes and restarts with full state recovery
  • FTR claims persist indefinitely until explicitly released
  • All coordinator events are persisted in JetStream for audit trail
  • Replay reconstructs workers, assignments, progress, resource locks

Negative

  • Requires manual sudo cp to deploy config changes (no auto-sync)
  • /var/lib/nats/ is outside project root (operational dependency)

Risks

  • If NATS data directory fills up (1GB max_file), oldest events are discarded
  • Replay of very long event streams may slow startup (mitigated by 30-day retention)

Lessons Learned

  1. Never use /tmp/ for anything that must survive a reboot — this is Infrastructure 101 but was missed because the initial setup was "just for local dev"
  2. Test crash recovery by actually crashing — the replay code existed but was never exercised because no integration test simulated a crash
  3. Verify the full persistence chain end-to-end — having a stream, having events, and having replay are three independent things that must all work together
  4. Document operational topology — system service vs user service vs manual start creates confusion when things break at 9 PM
  5. TTL on ownership claims is dangerous — heartbeat-refreshed TTLs assume the refresher survives, which is exactly what fails during a crash

Addendum: Leader Worktree Isolation Failure (2026-03-12)

What Happened

During the crash recovery described above, the leader session committed 5 fixes directly to develop from the main checkout (/home/jolo/muscovite), bypassing the required branch → PR → merge workflow:

Commit Description
4e836b89 NATS durability fixes
6582d055 Postmortem documentation
d43deae0 FTR-383 plan
85580335 Amend-state implementation
f18a8522 ITERATION.md corruption fix + FTR-384 spec

Why It Happened

  1. No leader role in session roles table — AGENTS.md only defined planner, impl, hotfix
  2. "Emergency" mindset — the leader rationalized direct commits as urgent fixes
  3. Ambiguous rules — "Planning sessions MAY work directly on develop" was misinterpreted to also cover emergency governance work
  4. No pre-commit hook — nothing prevents git commit on develop in the main checkout

Impact

  • 5 commits pushed directly to develop without PR review (cannot be undone — force-push forbidden)
  • Dirty working tree in main checkout caused git pull failures
  • Worker file leaks (FTR-383-*.md junk files) appeared in main checkout because leader and workers shared it

Corrective Actions

  1. ✅ Added leader role to session roles table — agents/leader/worktree/ on leader/governance
  2. ✅ IRON RULE documented: ALL writing roles MUST use worktrees, including leader
  3. ✅ Main checkout is read-only — only used for git log, gh pr view, coordinator CLI
  4. TODO (FTR-384): Pre-commit hook in main checkout that rejects commits on develop
  5. TODO (FTR-384): agent-pr.sh validates worktree isolation before creating PR
