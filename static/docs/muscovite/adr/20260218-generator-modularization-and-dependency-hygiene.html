Generator Modularization and Dependency Hygiene

Generator Modularization and Dependency Hygiene

  • Status: accepted
  • Date: 2026-02-18
  • Tags: generators, modularization, ai-context, dependency-hygiene, architecture
  • Deciders: Johannes Lochmann

Technical Story: The Racket generator codebase (93 files, ~80K lines) has grown cross-target coupling and oversized files that degrade AI agent effectiveness. A "Muscovite 2.0" analysis proposed rewriting generators in Python/Jinja2 to solve this. This ADR rejects the rewrite and prescribes targeted modularization within the existing Racket architecture instead.

Context and Problem Statement

Muscovite's generators in dslgen/generators/ produce output for 6 targets (postgres, cpp, grpc, qt, cmake, docs). As the generator count grew, cross-cutting concerns leaked across target boundaries:

  • generators/grpc/main.rkt imports generators/cpp/common.rkt and generators/cpp/dba/cached-repository.rkt directly
  • Naming conventions (symbol->cpp-class, camel-to-snake) are duplicated or imported ad-hoc
  • generators/grpc/main.rkt is 1203 lines — exceeds any AI agent's effective context window for targeted edits
  • No automated test prevents cross-target coupling from returning after cleanup

A "Muscovite 2.0 Manifesto" proposed solving this by pivoting to a JSON IR + Python/Jinja2 generator architecture. Analysis showed this would:

  • Destroy 4 months of proven, shipping generators
  • Downgrade codegen primitives (Jinja2 < Racket's pattern matching + quasiquoting)
  • Add serialization/deserialization tax for zero user value
  • Require 3–6 months to return to current parity

The actual problem is organizational, not architectural. The Racket struct-based IR (in-memory ddd-model tree) already serves as a clean interface between parser and generators. The fix is dependency hygiene and file decomposition, not a technology pivot.

Decision Drivers

  • AI agents work best with files under 400 lines and clear module boundaries
  • Cross-target require statements create hidden coupling (changing cpp internals silently breaks grpc)
  • The existing Racket struct IR is superior to JSON for this use case: zero serialization cost, pattern matching, type-safe accessors
  • Template DSL work (FTR-069, planned for v1.21.0) changes how code is emitted; modularization changes how generators are organized — orthogonal concerns
  • Debug IR inspection is valuable, but as an audit tool alongside normal generation, not as the generation pipeline

Considered Options

  1. Targeted modularization — extract shared module, decompose large files, enforce dependency rules, add debug IR export
  2. Full rewrite to Python/Jinja2 — JSON IR as primary pipeline, Jinja2 templates for all targets
  3. No action — accept growing coupling, rely on documentation

Decision Outcome

Chosen option: Option 1 — Targeted modularization, because it addresses the root cause (dependency hygiene and file size) without abandoning the working generator architecture or its superior Racket codegen primitives.

Implementation Plan

Five concrete work items (each tracked as a separate feature):

# Work Item Priority Effort
1 Extract generators/shared/ module from cross-cutting dependencies must 1–2 days
2 Decompose all generator files exceeding 400 lines must 2–3 days
3 Automated dependency graph test (regression guard) must 0.5 days
4 Debug IR export via --emit-ir flag (S-expression format) should 1–2 days
5 Auto-generated per-target AI context files could 1 day

Dependency Order

Item 1 (shared/) → Item 2 (decompose) → Item 3 (dep graph test)
Item 4 (debug IR) — independent
Item 5 (context files) — independent, benefits from Items 1+2

Key Design Decisions

S-expressions for debug IR, not JSON: The --emit-ir flag emits the parsed ddd-model as readable S-expressions (.sexp). S-expressions are native to Racket, round-trip via (read), and avoid inventing a JSON schema for internal debugging. JSON remains an option for external tooling in the future.

400-line file size threshold: Aggressive but justified — AI agents perform best with focused, single-responsibility files. Files exceeding 400 lines must be split into sub-modules with documented responsibilities.

Automated dependency enforcement: A test (test-generator-dependency-graph.rkt) scans require statements and enforces the rule: all cross-cutting utilities flow through generators/shared/, no direct imports between target generators.

Positive Consequences

  • AI agents can work on individual generators without loading the entire codebase
  • Cross-target changes are safe — shared/ is the only coupling point
  • Debug IR enables model inspection without REPL access
  • Regression test prevents coupling from returning
  • Complements FTR-069 (template DSL) — modularization is organizational, template DSL is mechanical

Negative Consequences

  • One-time refactoring cost (~6–8 days total)
  • Byte-identical output verification required for each decomposition step
  • More files to navigate (mitigated by clear naming and doc headers)

Rejected Alternative: Python/Jinja2 Rewrite

The "Muscovite 2.0 Manifesto" proposed:

  • Racket emits JSON IR → Python dispatcher → Jinja2 templates
  • Rationale: "AI comfort zone", multi-language generators, brownfield ingestors

Rejected because:

  • Technology downgrade: Jinja2 is a string template engine. Racket has pattern matching, quasiquoting, hygienic macros. Complex generation logic (saga orchestration, semantic diffing, event sourcing) would be harder in Jinja2 + Python helpers.
  • Serialization tax: JSON IR adds encode/decode boundary for zero user value. The in-memory struct tree is faster and type-safe.
  • Parity cost: 3–6 months to reproduce current output, with zero new features during the transition.
  • Moving target: "AI context limits" are a 2024-era concern being solved by longer context windows and better tooling (CONTEXT.md, GROUND-TRUTH.md — which Muscovite already has).
  • Scope creep: The manifesto bundled brownfield ingestors (a separate product) into the pivot, guaranteeing it would never ship.

The good idea from the manifesto — debug IR for inspection — is preserved as Item 4 above.
