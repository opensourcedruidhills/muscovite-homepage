Racket-Based Target Configuration for UI Generation

Racket-Based Target Configuration for UI Generation

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: targets, configuration, racket, qt, qml, code-generation

Context and Problem Statement

Muscovite's Qt generators (widgets-views.rkt, qml-main.rkt) have hardcoded layout decisions:

  • Widget type mapping is a match expression in field-type->widget-type (widgets-views.rkt)
  • Aggregate layout is always QVBoxLayout → QFormLayout → QTabWidget (widgets) or ColumnLayout → GroupBox (QML)
  • No mechanism to configure how abstract layout concepts (from ADR abstract-layout-vocabulary) map to concrete widgets per target

When the LAYOUT block is implemented, generators need a data-driven way to resolve abstract concepts (DISPLAY table, STYLE multi_section) to target-specific widgets (QTableView vs. ListView).

Question: How should target-specific widget mappings be configured?

Decision Drivers

  • Same language as generators — Racket modules are testable, type-checked, version-controlled
  • No external tooling — YAML/JSON/XML requires parsers, schema validation, tooling
  • Inheritance — targets should inherit from base configurations (e.g., qt-qml-desktop extends qt-qml-base)
  • Grammar-first — configuration is code, tested in the Racket test suite
  • Scope control — start with 2 targets, not 10

Considered Options

Option 1: Racket Modules as Target Configuration

Target configs are .rkt files under dslgen/targets/. Each exports a target-config struct with widget mapping hashes. Configurations support inheritance via merge-configs.

Option 2: YAML/JSON Configuration Files

Target configs are .yaml or .json files loaded at generation time.

Option 3: Inline Configuration in Generators

Keep mappings inside each generator file (current approach).

Decision Outcome

Chosen option: Option 1 — Racket Modules, because:

  • Same language, same tooling, same test infrastructure
  • Struct-based: target-config is a Racket struct — field access errors are caught at load time
  • Testable: configuration is exercised by Racket test suite
  • Composable: merge-configs enables inheritance without custom merge logic
  • No new dependencies: no YAML parser, no JSON schema validator

Initial Scope: Two Targets Only

Target Config File Generator
Qt Widgets Desktop dslgen/targets/qt-widgets-desktop.rkt generators/qt/widgets-views.rkt
Qt QML Desktop dslgen/targets/qt-qml-desktop.rkt generators/qt/qml-main.rkt

Additional targets (mobile, tablet) are deferred until the abstraction is proven on these 2.

Directory Structure

dslgen/targets/
├── target-config.rkt         # Struct definition, merge-configs, resolve-layout-mapping
├── qt-widgets-desktop.rkt    # Documents current hardcoded Qt Widgets behavior
└── qt-qml-desktop.rkt        # Documents current hardcoded QML behavior

Target Config Struct

(struct target-config
  (name               ; Symbol: e.g., 'qt-widgets-desktop
   form-factor        ; Symbol: 'desktop (only desktop initially)
   language           ; Symbol: 'cpp or 'qml
   aggregate-mappings ; Hash: style-symbol → widget-string
   master-mappings    ; Hash: 'orientation → (Hash symbol → string), 'grouping → ...
   detail-mappings    ; Hash: 'display → (Hash symbol → string), 'actions → ...
   action-mappings    ; Hash: 'placement → (Hash symbol → string)
   constraints)       ; Hash: capability → value
  #:transparent)

Widget Mapping Example (Qt Widgets Desktop)

(define qt-widgets-desktop-config
  (target-config
   'qt-widgets-desktop
   'desktop
   'cpp
   ;; aggregate-mappings: style → container widget
   (hash 'single-page   "QScrollArea + QVBoxLayout"
         'multi-section "QTabWidget"
         'wizard        "QWizard"
         'split-view    "QSplitter")
   ;; master-mappings
   (hash 'orientation (hash 'vertical   "QFormLayout"
                            'horizontal "QHBoxLayout"
                            'grid       "QGridLayout")
         'grouping    (hash 'flat     "QGroupBox"
                            'sections "QTabWidget"
                            'tabs     "QTabBar + QStackedWidget"))
   ;; detail-mappings
   (hash 'display (hash 'table "QTableView"
                        'cards "QListView + custom delegate"
                        'list  "QListView"
                        'tree  "QTreeView")
         'actions (hash 'toolbar      "QToolBar"
                        'context-menu "right-click menu"
                        'inline       "per-row QPushButtons"
                        'fab          "QToolButton (floating)"))
   ;; action-mappings
   (hash 'placement (hash 'top     "QDialogButtonBox (top)"
                          'bottom  "QDialogButtonBox (bottom)"
                          'sidebar "QToolBar (side)"
                          'floating "QToolButton (floating)"))
   ;; constraints
   (hash 'max-fields-per-form  #f
         'supports-nested-tabs #t
         'supports-inline-edit #t
         'supports-drag-drop   #t)))

Positive Consequences

  • Widget mappings are data, not embedded in control flow — easier to review and modify
  • Adding a new target is a config file + generator integration, not a DSL grammar change
  • Existing generator behavior is documented as explicit configuration (currently implicit)
  • Inheritance enables target families (e.g., all QML targets share base mappings)

Negative Consequences

  • Another layer of indirection between DSL and generated output
  • Initial effort to extract hardcoded mappings from widgets-views.rkt and qml-main.rkt
  • Two targets may not stress-test the abstraction enough (but this is intentionally conservative)

AI Agent Guidance

When This Decision Applies

  • Adding new UI generation targets
  • Modifying widget mappings for existing targets
  • Implementing LAYOUT block support in generators
  • Creating files under dslgen/targets/

When This Decision Does NOT Apply

  • Non-UI generators (PostgreSQL, gRPC, C++ DBA)
  • DSL grammar changes (those go through grammar-first process)
  • Runtime configuration (this is compile-time/generation-time only)

Enforcement

  • Target configs live in dslgen/targets/ as .rkt files
  • Each config exports a target-config struct
  • Generators load config via load-target-config (not hardcoded hashes)
  • Test coverage: each target config has unit tests in dslgen/tests/test-target-configs.rkt
  • Do not create YAML/JSON/XML config files — all configuration is Racket

Key Files

  • dslgen/targets/target-config.rkt — struct definition, merge, resolve functions
  • dslgen/targets/qt-widgets-desktop.rkt — Qt Widgets Desktop configuration
  • dslgen/targets/qt-qml-desktop.rkt — Qt QML Desktop configuration
  • dslgen/generators/qt/widgets-views.rkt — consumer of Qt Widgets config
  • dslgen/generators/qt/qml-main.rkt — consumer of QML config
