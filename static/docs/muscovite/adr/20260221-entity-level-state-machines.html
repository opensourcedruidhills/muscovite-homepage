Entity-Level State Machines via DSL Statechart Construct

Entity-Level State Machines via DSL Statechart Construct

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-21
  • Tags: dsl, state-machine, statechart, behavior, fsm, safety, lifecycle

Technical Story: Muscovite currently models data (CRUD entities) and cross-aggregate orchestration (Sagas), but lacks formal intra-entity lifecycle modeling. Domains like vending machines, railway signaling, and financial order processing require explicit state-transition graphs with guards, actions, and safety guarantees (no invalid transitions, reachability analysis, deadlock detection).

Context and Problem Statement

Many real-world aggregates have lifecycle behavior that goes beyond simple status fields: orders flow through Draft→Submitted→Approved→Fulfilled, railway signals cycle through Clear→Approach→Stop with safety interlocks, vending machines track Idle→Selecting→Dispensing→Returning. Today, developers implement this as procedural if/else or switch(status) blocks — error-prone, untestable, and invisible to the domain model.

Sagas already implement step-based state machines for cross-aggregate orchestration (7 generators, lookup tables, compile-time constants). The infrastructure exists — but Sagas are the wrong abstraction for intra-entity lifecycle. A STATE_MACHINE construct would formalize entity behavior in the .ddd model and generate enforcement at every layer.

Question: How should Muscovite model formal state machines for entity lifecycles?

Decision Drivers

  • Grammar-first language evolution (ADR 20260211) — PEG grammar must change before parser
  • Saga infrastructure provides proven patterns for state/transition lookup tables and compile-time constants (ADR 20260212)
  • Domain Services as Aggregate Boundary (ADR 20260120) — transitions must go through CommandService
  • Event Sourcing compatibility (ADR 20260204) — state transitions should emit domain events
  • CQRS-Ready (ADR 20260120) — transition commands are writes, state queries are reads
  • Safety domains (railway, medical devices, vending) require formal verification of state graphs

Considered Options

Option 1: Entity-level clause (like CONCURRENCY)

Add a STATE_MACHINE { ... } block as a clause inside ENTITY, similar to how CONCURRENCY works today. States and transitions are defined inline within the entity.

Option 2: Context-level construct with FOR clause (like SAGA)

Add STATE_MACHINE Name FOR Entity { ... } as a top-level context construct, following the Saga precedent. The entity gains a current_state_id FK but the state graph lives separately.

Option 3: Embedded in AGGREGATE with EVENT_SOURCED

Extend the existing EVENT_SOURCED mechanism to include statechart definitions, coupling state transitions to sourced events.

Decision Outcome

Chosen option: Option 2 — Context-level construct with FOR clause, because:

  1. Saga precedent works: 7 generators already handle context-level constructs with lookup tables, compile-time constants, and cross-layer generation. The STATE_MACHINE pattern is structurally identical.
  2. Separation of data and behavior: Entities describe structure, state machines describe lifecycle. Mixing them increases entity complexity.
  3. Multiple machines per entity: Rare but valid (e.g., OrderFulfillmentState + PaymentState). Context-level allows this naturally.
  4. Reachability analysis: A standalone construct is easier to validate (graph analysis) than one embedded in entity parsing.

DSL Syntax

STATE_MACHINE OrderLifecycle @uuid "..." FOR Order {
    STATE Draft       DOC "Initial state, order being composed";
    STATE Submitted   DOC "Order submitted for processing";
    STATE Approved    DOC "Order approved by manager";
    STATE Rejected    DOC "Order rejected";
    STATE Fulfilled   DOC "Order shipped and delivered";
    STATE Cancelled   DOC "Order cancelled";

    INITIAL Draft;
    TERMINAL Fulfilled, Rejected, Cancelled;

    TRANSITION Draft -> Submitted ON OrderSubmitted;
    TRANSITION Submitted -> Approved ON ManagerApproved
        GUARD "total_amount < approval_threshold";
    TRANSITION Submitted -> Rejected ON ManagerRejected;
    TRANSITION Approved -> Fulfilled ON ShipmentDelivered;
    TRANSITION Submitted -> Cancelled ON OrderCancelled
        GUARD "NOT has_shipped";
    TRANSITION Draft -> Cancelled ON OrderCancelled;
}

IR Structs

(struct state-machine (uuid name for-entity states transitions initial-state terminal-states doc))
(struct sm-state (name entry-action exit-action doc))
(struct sm-transition (from-state to-state event guard action doc))

Generated Artifacts Per Layer

Layer Artifact
PostgreSQL {entity}_state lookup table (UUIDv5 seeds), current_state_id FK on entity, {entity}_state_transition_log table, EXCLUDE constraint preventing invalid transitions
C++ DBA {Entity}StateMachine class: can_transition(current, event), transition(current, event), compile-time state constants (lookup::{entity}_state::draft)
gRPC Transition{Entity}State RPC, Get{Entity}AllowedTransitions RPC
Qt Widgets State-aware button enable/disable from allowed_transitions()
Qt QML ViewModel Q_PROPERTY for currentState + allowedTransitions
Documentation Mermaid statechart diagram from transition graph

Verification (Layer 0)

The Racket generator MUST perform static analysis on the state graph:

  • Reachability: All states must be reachable from INITIAL
  • Terminal completeness: At least one TERMINAL state must be declared
  • No orphan transitions: Every FROM/TO state must be declared in STATE list
  • Determinism: No two transitions from the same state on the same event (unless guards are mutually exclusive)

Positive Consequences

  • Formal lifecycle modeling visible in .ddd — no hidden behavior in procedural code
  • Reuse of saga lookup-table and compile-time constant infrastructure (ADR 20260212)
  • Safety-critical domains get verifiable state graphs with generated documentation
  • Qt UI automatically reflects allowed transitions — no manual button logic

Negative Consequences

  • New grammar section + IR structs + 6 generators — significant implementation effort
  • Guard expressions need a safe evaluation model (SQL-compatible subset)
  • State machines with many transitions may produce verbose .ddd files
