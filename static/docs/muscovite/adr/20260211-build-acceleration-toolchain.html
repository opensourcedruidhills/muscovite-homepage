Build Acceleration Toolchain (Mold + ccache)

Build Acceleration Toolchain (Mold + ccache)

  • Status: accepted
  • Date: 2026-02-11
  • Tags: build, cmake, performance, toolchain

Context and Problem Statement

C++ build times for muscovite framework and generated projects are observed at ~2h wallclock on a 6-core/16GB VM. The project uses GCC with the default ld linker and no compiler caching. ADR 20260113 (Shared Libraries Only) ensures all libraries are .so, making link-time performance especially relevant. IDEA-002 (Incremental Code Generation, shipped v1.4.0) addressed Racket generation-side performance; this ADR addresses C++ compilation and linking.

Two mature, zero-risk tools can immediately reduce build times:

  1. Mold: A drop-in parallel linker, 3-10× faster than ld for shared libraries
  2. ccache: A compiler cache that makes warm rebuilds near-instant for unchanged translation units

Both are optional — builds succeed without them using default system tools.

Decision Drivers

  • Build times: 2h wallclock is the primary blocker for developer iteration speed
  • Three build scopes: Framework (S1), Racket generation (S2), generated projects (S3) — mold and ccache help S1 and S3
  • ADR 20260113: All shared libraries — mold's strength is exactly shared library linking
  • ccache sequencing: Deliberately deferred until IDEA-002 shipped (v1.4.0) to avoid shadowing timestamp/diffing issues. Now unblocked.
  • Zero risk: Both tools are optional, well-tested, widely adopted. No code changes, no generator changes.

Considered Options

  • Mold only: Fastest linker available, Linux-native
  • ccache only: Compiler caching for warm rebuilds
  • Both mold + ccache: Orthogonal — mold handles linking, ccache handles compilation
  • LLD (LLVM linker): Alternative to mold, but slower for shared libraries

Decision Outcome

Chosen option: "Both mold + ccache", because they are orthogonal optimizations (mold = link phase, ccache = compile phase) with zero interaction risk. Both configured as optional — builds gracefully degrade without them.

Implementation

cmake/BuildAcceleration.cmake (auto-detection module, included from CMakeLists.txt):

# Mold: detected via find_program(), applied to Debug builds only
option(MUSCOVITE_USE_MOLD "Use mold linker if available" ON)
find_program(MOLD_EXECUTABLE mold)
# Sets CMAKE_EXE_LINKER_FLAGS, CMAKE_SHARED_LINKER_FLAGS if found

# ccache: detected via find_program(), applied to all build types
option(MUSCOVITE_USE_CCACHE "Use ccache compiler launcher if available" ON)
find_program(CCACHE_EXECUTABLE ccache)
# Sets CMAKE_CXX_COMPILER_LAUNCHER if found

Both tools are auto-detected at configure time — no CMakePresets.json changes needed. Developers can disable with -DMUSCOVITE_USE_MOLD=OFF or -DMUSCOVITE_USE_CCACHE=OFF.

Developer prerequisites (documented in DEVELOPER-GUIDE.md):

# Optional but recommended — significant build speedup
sudo apt install mold ccache

Applies to:

  • Muscovite framework builds (S1)
  • Generated project builds (S3) — via inherited presets or direct cmake invocation
  • Does NOT affect Racket generation (S2) — that's pure Racket, no C++ involved

Fallback Behavior

Tool Installed Not Installed
mold Used as linker (debug only) System ld used — no error
ccache Used as compiler launcher GCC compiles directly — no error

Detection is via find_program() at CMake configure time. No hard-coded flags in presets.

Positive Consequences

  • Immediate link-time reduction for debug builds (mold)
  • Near-instant warm rebuilds for unchanged files (ccache)
  • Branch switching and --fresh regeneration benefit from ccache
  • No generator changes required — pure CMake preset configuration
  • Zero risk to existing builds

Negative Consequences

  • Two additional optional dependencies for developers
  • ccache disk usage (~1-5GB default cache size, configurable)
  • Mold is Linux-only (macOS/Windows contributors would need alternatives)

Measurement

Before/after metrics captured via FTR-010 (Build Profiling Tooling) and stored in metrics/build-history.jsonl. Each acceleration step (baseline → mold → ccache) recorded as a separate history entry for trend analysis.

AI Agent Guidance

When This Decision Applies

  • Modifying CMakePresets.json — preserve mold and ccache settings
  • Adding new CMake presets — consider whether mold/ccache should apply
  • Documenting build prerequisites — mention mold and ccache as optional

When This Decision Does NOT Apply

  • Racket generation performance — unrelated (use incremental gen, v1.4.0)
  • Generated CMakeLists.txt for example projects — they inherit from their own presets
  • CI/CD pipelines — may want controlled environments without ccache

Key Files

  • cmake/BuildAcceleration.cmake — mold and ccache auto-detection module
  • CMakeLists.txt — include(BuildAcceleration) after CompilerWarnings
  • doc/DEVELOPER-GUIDE.md — prerequisite documentation
  • tools/merge-build-metrics.py — trend analysis
  • metrics/build-history.jsonl — historical tracking

Enforcement

Manual review during CMakePresets.json changes. No automated enforcement needed — these are optional optimizations.
