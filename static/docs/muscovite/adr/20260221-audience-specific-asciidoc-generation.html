AI-Supported Audience-Specific AsciiDoc Generation

AI-Supported Audience-Specific AsciiDoc Generation

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-21
  • Tags: documentation, asciidoc, glossary, llm, persona, audience, ddd-literature

Technical Story: Different stakeholders interacting with a Muscovite-generated project (auditors, domain experts, C-level, IT/ops, architects, developers) require vastly different documentation views derived from the same .ddd source of truth.

Context and Problem Statement

Muscovite already generates AsciiDoc glossaries via dslgen/generators/docs/glossary.rkt and glossary-adoc.rkt, but these produce a single undifferentiated view of the IR. Different stakeholders need fundamentally different projections of the same domain model:

  • Auditors (GDPR, Fintech): Data lineage, PII identification, audit log guarantees.
  • Domain Experts: Ubiquitous Language verification, Aggregate boundaries, business processes (Sagas).
  • C-Level / Sponsors (CTO, CFO, COO, CIO, CSO): High-level architecture, compliance summaries.
  • IT / Ops Staff: Deployment topologies, infrastructure requirements, graceful degradation (e.g., NATS unavailability and recovery).
  • Architects / Team Leads: Bounded Contexts, Context Maps, service boundaries.
  • Developers (Users of Muscovite): API contracts, repository interfaces, frontend bindings.
  • Developers (Developing Muscovite): Internal generator architecture documentation.
  • Technical Managers: Team workflow understanding, system capability overview.

Maintaining the Ubiquitous Language across the entire organizational chart — not just among developers — is one of the hardest problems in DDD. Writing audience-specific documentation manually always drifts from implementation. Since Muscovite holds the complete semantic model, it can generate this documentation automatically.

A key architectural insight: writing complex Racket text-formatting code for 5+ audiences is brittle and hard to maintain. Treating an LLM as a compiler target — generating a structured Fact Base in Racket and synthesizing audience-specific prose via persona prompts — is simpler and more extensible.

Question: How should Muscovite produce audience-tailored documentation from the domain model while anchoring it in globally accepted DDD literature?

Decision Drivers

  • Aligns with ADR 20260217 (Documentation as Generated Artifact) — all DSL metadata must propagate to every generated layer
  • Existing glossary.rkt / glossary-adoc.rkt pipeline provides a working foundation to extend
  • CMake documentation targets (FTR-092, completed in v1.20.0) provide the build integration point
  • Audience-specific views are a force multiplier for enterprise adoption
  • DDD literature citations add credibility and educational value
  • Minimizing Racket code complexity for text synthesis

Considered Options

Option 1: Pure Racket AsciiDoc templates per audience

Extend the existing glossary-adoc.rkt pipeline with per-audience Racket template modules (e.g., auditor-adoc.rkt, domain-expert-adoc.rkt). Each module filters and formats the IR for its target audience using Racket format strings.

Option 2: Structured Fact Base + LLM persona synthesis

Extend the existing glossary pipeline to emit a structured, machine-readable Fact Base (JSON or dense Markdown). Create versioned persona prompts per audience. Use an LLM to synthesize the final AsciiDoc reports from the Fact Base + persona prompt.

Option 3: External documentation tool integration

Export the domain model to a format consumable by external tools (Structurizr, Antora, etc.) and delegate audience-specific rendering to those tools.

Decision Outcome

Chosen option: Option 2 — Structured Fact Base + LLM persona synthesis, because it minimizes Racket code complexity, is the most extensible approach for adding new audiences, and produces higher-quality prose than template-based formatting.

Architecture

  1. Racket Fact Base Generator (Layer 0): Extend the existing glossary.rkt pipeline to emit domain-fact-base.json — a structured file fusing framework concepts (Aggregates, Sagas, Events) with the user's concrete domain objects, DDD term definitions, and bibliography citations.
  2. Persona Prompts: Versioned system prompts in dslgen/resources/prompts/audiences/ (e.g., auditor.prompt, domain-expert.prompt). Each prompt instructs the LLM what to include, exclude, and emphasize.
  3. AI Generation Pipeline: A script (tools/generate-audience-reports.sh) that takes the Fact Base, iterates persona prompts, calls an LLM (API or local tool like ollama), and outputs per-audience AsciiDoc files.
  4. CMake Integration: Per-audience build targets (e.g., cmake --build --preset debug --target docs-auditor), building on the existing documentation target infrastructure from FTR-092.
  5. Glossary Merging: The Fact Base interleaves user domain terms with Muscovite DDD terms. Cross-references link concrete instances (e.g., Patient) to their DDD concept (e.g., Aggregate Root).
  6. Bibliography: Citations are injected for architectural patterns used in the generated code (e.g., Evans 2003, Vernon 2013, Fowler). Output AsciiDoc includes a [bibliography] section.

Positive Consequences

  • Living Documentation guaranteed to match the implementation
  • Out-of-the-box compliance/auditability reports reduce enterprise adoption friction
  • Shared Ubiquitous Language glossary across technical and non-technical stakeholders
  • Vastly simpler Racket code — text synthesis delegated to LLM
  • Adding a new audience requires only a new prompt file, not new Racket code

Negative Consequences

  • Introduces LLM dependency (cost, availability, potential non-determinism) in the documentation build phase
  • Requires managing and versioning persona prompts alongside the codebase
  • LLM output quality depends on Fact Base completeness — incomplete IR metadata produces weak reports

AI Agent Guidance

When This Decision Applies

  • When extending the glossary.rkt / glossary-adoc.rkt pipeline
  • When adding new documentation output targets
  • When adding audience-specific content filtering to generators
  • When working on the Fact Base JSON schema or persona prompts

When This Decision Does NOT Apply

  • Existing per-layer documentation propagation (SQL COMMENT ON, C++ Doxygen, proto comments) — those remain pure Racket generators per ADR 20260217
  • Internal developer documentation for Muscovite itself (CONTEXT.md, GROUND-TRUTH.md) — those have their own generation pipelines

Enforcement

  • Fact Base schema validated by Racket tests
  • Persona prompts versioned in repository alongside DSL sources
  • CMake targets fail loudly if Fact Base generation fails (LLM step is optional/degradable)
