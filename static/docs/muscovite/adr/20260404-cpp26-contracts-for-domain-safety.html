C++26 Contracts for Domain Safety

C++26 Contracts for Domain Safety

  • Status: proposed
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04
  • Tags: c++26, contracts, safety, generators, framework

Technical Story: Muscovite roadmap v3.0–v3.1 — mapping DSL constraints to C++26 contracts.

Context and Problem Statement

C++26 introduces language-level contracts: pre (preconditions), post (postconditions), and contract_assert (in-body assertions). These replace the need for Microsoft GSL (Expects/Ensures) and C assert() macros with a standardized, toolchain-integrated mechanism.

Muscovite's DSL already declares constraints (REQUIRED, unique fields, value ranges, foreign key relationships) that imply runtime invariants. Currently, these constraints are enforced only at the database level (NOT NULL, UNIQUE constraints) and partially in generated validation code. The C++ layer has no systematic contract enforcement — invalid data can propagate through gRPC services and repositories before hitting a database error.

Decision Drivers

  • DSL constraints are the canonical truth — contracts should be projections of the model
  • Contracts catch violations at the C++ API boundary, before database round-trips
  • Healthcare/geosciences domains require demonstrable safety guarantees
  • Contract violation semantics must be configurable (enforce vs. observe vs. ignore)
  • Generated code volume must not explode — contracts should be targeted, not exhaustive
  • Framework APIs (repository, service_base, type_converters) have implicit preconditions that are currently undocumented

Considered Options

  1. DSL-driven contracts only — Generate contracts from DSL constraints; no framework changes
  2. Framework-first contracts — Add contracts to framework APIs; no generator changes
  3. Both layers — Contracts in framework APIs (v3.0) + generated contracts from DSL (v3.1)
  4. Adopt-on-touch — Add contracts when touching code, no systematic pass

Decision Outcome

Chosen option: "Both layers" (option 3), phased:

Phase 1 (v3.0.0): Framework contracts — add pre/post to public framework APIs:

  • repository<T>::insert() — pre(!entity.id().is_nil())
  • repository<T>::find_by_id() — pre(!id.is_nil())
  • service_base — pre(context.is_authenticated())
  • type_converters — pre on nullable-to-value conversions

Phase 2 (v3.1.0): Generated contracts from DSL constraints:

  • REQUIRED fields → pre(field.has_value()) on insert/update methods
  • Value ranges (when DSL supports them) → pre(value >= min && value <= max)
  • Foreign key fields → pre(!fk_id.is_nil()) on relationship accessors
  • Entity invariants → contract_assert in constructors

Positive Consequences

  • DSL constraints enforced at every layer (DB + C++ API + contracts)
  • Invalid data caught before database round-trip (faster failure, clearer errors)
  • Systematic documentation of API preconditions (contracts are self-documenting)

Negative Consequences

  • Generated code grows (each contract is an additional line per function)
  • C++26 contract semantics are still debated in the committee (risk of C++29 changes)

AI Agent Guidance

When This Decision Applies

  • When adding or modifying public framework APIs (add pre/post)
  • When generating C++ code from DSL entities with REQUIRED fields or relationships

When This Decision Does NOT Apply

  • v2.x development (C++23 — no contracts syntax available)
  • Private/internal helper functions (contracts are for API boundaries)

Enforcement

  • Generator code emits contracts from DSL constraints
  • Framework headers have contracts on public APIs
  • clang-tidy: enable contract-related diagnostics when available
