ADR: NATS JetStream for Multi-Agent Coordination

ADR: NATS JetStream for Multi-Agent Coordination

  • Status: superseded by 20260318-nats-agent-coordination-abandoned
  • Date: 2026-03-09
  • Deciders: Johannes Lochmann
  • Supersedes: ADR 20260306 (Multi-Agent Session Protocol) — file-based mechanism only; session roles, claim semantics, and workspace isolation rules remain in force.

Context

The current multi-agent coordination protocol (ADR 20260306) uses files:

Mechanism File Limitation
Event log agents/iteration-log.jsonl Polled, no push notification, no filtering
Claims agents/claims/ftr-NNN.lock No TTL, stale locks require manual PID checks
Heartbeat 30-min rule in agent_log Social enforcement only — no automatic detection
Monitoring tail -f tmp/*.log Single stream, unstructured, single machine

These limitations compound as the project scales:

  1. Multi-device agents — within 1 month, additional AI workers on separate machines will join the workflow. File-based coordination is single-machine by definition.
  2. Operational blindness — tail -f on a log file provides no filtering, no replay, no liveness detection. Monitoring a release ceremony means watching raw text scroll.
  3. Stale state — lock files outlive crashed sessions. The PID-based staleness check (>4 hours, dead PID) is unreliable across machines and doesn't account for agent sessions that hang rather than crash.
  4. No structured queries — "what is every agent doing right now?" requires parsing JSONL, reading lock files, and correlating manually.

NATS JetStream is already the domain event bus for generated applications (ADR 20260202, outbox pattern, FTR-247 subject schema). Using it for agent coordination is dogfooding — operational issues surface here before end users encounter them.

Decision

Replace the file-based agent coordination protocol with NATS JetStream.

Install nats-server as a persistent systemd service on each development machine. This is a development environment prerequisite, not a Docker-ephemeral service.

Installation

# openSUSE Tumbleweed / any Linux
curl -sf https://binaries.nats.dev/nats-io/nats-server/v2@latest | sh
sudo mv nats-server /usr/local/bin/
sudo cp agents/nats/nats-server.service /etc/systemd/system/
sudo systemctl enable --now nats-server

# CLI tool
curl -sf https://binaries.nats.dev/nats-io/natscli/nats@latest | sh
sudo mv nats /usr/local/bin/

Configuration: agents/nats/nats-server.conf — JetStream enabled, file store at /var/lib/nats/muscovite, 1GB storage limit, monitoring on port 8222.

Subject Hierarchy

muscovite.agent.{role}.lifecycle        # birth / heartbeat / death
muscovite.agent.{role}.ftr.{N}.claim    # start / release
muscovite.agent.{role}.ftr.{N}.progress # milestone events
muscovite.agent.{role}.ftr.{N}.gate     # quality gate pass/fail + timing
muscovite.agent.{role}.ftr.{N}.blocked  # blocker announcements
muscovite.release.{ver}.step            # release ceremony phase events
muscovite.release.{ver}.gate            # release quality gate results
muscovite.metrics.build                 # build-snapshot.json payloads
muscovite.metrics.test                  # test-report.json payloads
muscovite.metrics.coverage              # coverage snapshots

All payloads are JSON with mandatory fields: ts (ISO-8601), agent (role string), detail (human-readable string). Additional fields vary by subject.

JetStream Streams

Stream Subjects Retention Max Age
AGENT-EVENTS muscovite.agent.> limits 30 days
RELEASE-EVENTS muscovite.release.> limits 365 days
METRICS muscovite.metrics.> limits 365 days

Release events are retained longer — they form the audit trail for governance.

KV Buckets (Live State)

Bucket Key Pattern TTL Purpose
agent-sessions {role} 5 min Heartbeat watchdog. Agent puts every 2 min. TTL expiry = session dead.
ftr-claims ftr-{N} 4 hours Replaces agents/claims/ftr-NNN.lock. Atomic CAS for contention.
release-lock current 2 hours Replaces agents/claims/release.lock. Only one release at a time.
develop-push current 10 min Replaces agents/claims/develop-push.lock. Serialize merges.

KV operations use CAS (Compare-And-Swap) revision numbers for contention resolution. If two agents race for the same FTR claim, only the first CAS succeeds.

Session Lifecycle

Session Start:
  1. nats kv put agent-sessions {role} '{"pid":$$,"host":"...","started":"..."}'
  2. Start heartbeat loop: every 120s, nats kv put agent-sessions {role} (refreshes TTL)
  3. nats pub muscovite.agent.{role}.lifecycle '{"event":"birth",...}'
  4. Read current state: nats kv ls ftr-claims + nats stream view AGENT-EVENTS --last=50

During Work:
  5. Claim FTR: nats kv create ftr-claims ftr-{N} '{"role":"...","branch":"..."}'
     (create = put-if-absent, atomic)
  6. Publish progress: nats pub muscovite.agent.{role}.ftr.{N}.progress '...'
  7. Heartbeat continues automatically (background loop)

Session End:
  8. Release claims: nats kv del ftr-claims ftr-{N}
  9. nats pub muscovite.agent.{role}.lifecycle '{"event":"death",...}'
  10. Stop heartbeat loop
  11. nats kv del agent-sessions {role}

Session Crash (unclean):
  - KV TTL expires in ≤5 minutes → session automatically marked dead
  - Dashboard detects missing heartbeat → alerts user
  - Stale FTR claims expire in ≤4 hours (or manually released after verification)

Copilot "Mission Control" Session

A Copilot session can act as an intelligent observer:

  1. On start: replay last 100 events from AGENT-EVENTS stream
  2. Subscribe to muscovite.> for live updates
  3. Maintain running state model of all agents, claims, gates
  4. User asks natural-language questions: "what's impl stuck on?", "how long has the release been running?", "any blockers?"
  5. Proactive alerts: "⚠️ impl hasn't heartbeated in 5 minutes"

Release Ceremony Integration

release.sh gains NATS instrumentation:

# Before each step
nats pub muscovite.release.${VERSION}.step \
  '{"step":"cpp-tests","status":"running","ts":"..."}'

# After each gate
nats pub muscovite.release.${VERSION}.gate \
  '{"gate":"cpp-tests","result":"pass","duration_s":178,"ts":"..."}'

# On completion
nats pub muscovite.release.${VERSION}.step \
  '{"step":"complete","status":"success","ts":"..."}'

This replaces tail -f tmp/release-*.log with:

nats sub "muscovite.release.v1.62.0.>"   # filtered live view

Migration Path

Phase Scope
1. Install nats-server systemd + nats CLI as dev prerequisites
2. Parallel run nats-agent.sh publishes to NATS AND writes iteration-log.jsonl
3. Dashboard agent-dashboard.sh proves NATS observability is superior
4. Deprecate files Stop writing iteration-log.jsonl, remove lock file checks
5. Multi-device Configure NATS cluster or leaf nodes for remote agents

Phase 2 ensures zero-risk migration — file fallback remains until NATS is proven.

Relation to Domain NATS

Concern Agent Coordination Domain Events
Server Persistent systemd service Docker ephemeral container
Subjects muscovite.agent.*, muscovite.release.* muscovite.{project}.{context}.events.*
Consumers Dashboard, Copilot sessions Generated application services
Lifecycle Always running on dev machine Started/stopped per project

The two NATS instances are separate. The agent coordination server runs at OS level. The domain event server runs in Docker per the existing docker-compose.yml setup. A future optimization could merge them with subject-based isolation, but separation is cleaner for now.

Consequences

Positive

  • Real-time observability: Push-based events replace polled file reads
  • Automatic liveness: KV TTL detects dead sessions in ≤5 minutes
  • Cross-machine coordination: NATS clustering enables multi-device agents
  • Structured queries: Filter by subject wildcard, replay from any point
  • Dogfooding: Operational experience with NATS before end users need it
  • Audit trail: JetStream retains release events for 365 days
  • Atomic claims: KV CAS prevents race conditions on FTR ownership

Negative

  • New prerequisite: nats-server + nats CLI must be installed on dev machines
  • Operational surface: Another service to monitor (mitigated: systemd auto-restart)
  • Migration cost: All three agent roles need session helper updates
  • Single point of failure: If NATS is down, agents can't coordinate (mitigated: file fallback during phase 2; systemd restart for steady state)

Neutral

  • agents/log.sh and agents/iteration-log.jsonl remain as deprecated fallback until phase 4. No data loss during transition.
  • The NATS subject hierarchy for agent coordination (muscovite.agent.*) is completely separate from the domain event hierarchy (muscovite.{project}.*). No namespace collision risk.

Target Iteration

Implementation scope TBD. Likely v1.64.0+ (after Behavioral Sovereignty Tier 2-3) unless multi-device agents arrive sooner, in which case this takes priority.
