Example Repos as Independent Consumers of muscovite-cpp

Example Repos as Independent Consumers of muscovite-cpp

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-12
  • Tags: release, decoupling, consumer-repo

Technical Story: FTR-407, FTR-408, FTR-409, FTR-410, FTR-411

Context and Problem Statement

Since v1.60.0 (FTR-338), release.sh generated code into every example submodule, committed, tagged, and pushed during the release ceremony. This created tight coupling between the muscovite release and four consumer repositories, causing:

  1. Release ceremony takes 15+ minutes for submodule generation + push
  2. A single submodule push failure aborts the entire release
  3. Consumer repos cannot release independently (they wait for muscovite tags)
  4. The TLA+ quality gate scanned examples/*/generated/*/tlaplus/ — generated artifacts that only exist after a full generation pass
  5. Build profiling collected metrics from example build directories

Decision Drivers

  • Release ceremony reliability (single-purpose, fewer failure modes)
  • Consumer repo autonomy (independent generation, testing, release cycles)
  • Separation of concerns (muscovite ships the toolchain, consumers use it)
  • Reduced release wall-clock time

Considered Options

  • Option A: Keep submodule-based release with better error handling
  • Option B: Decouple completely — consumer repos own their lifecycle

Decision Outcome

Chosen option: Option B — Full decoupling, because it cleanly separates the toolchain (muscovite) from its consumers (example repos) and enables independent release cycles.

Implementation

FTR Change
FTR-407 Remove example generation loop, submodule commit+tag+push from release.sh
FTR-408 Relocate TLA+ gate to scan agents/tlaplus/ (internal, always present)
FTR-409 Remove --generated-ninja and --generation-metrics from build profiling
FTR-410 Provide tools/github-actions/dsl-lint.yml for consumer CI
FTR-411 Update governance docs, Rule 2, ADR

Consumer Repo Contract

Consumer repos:

  1. Read MUSCOVITE_CPP_VERSION from muscovite repo (or pin manually)
  2. Use FetchContent(GIT_TAG vX.Y.Z) to pull muscovite-cpp
  3. Run racket generate.rkt --project project.mus --full in their own CI
  4. Use tools/github-actions/dsl-lint.yml for PR-time DSL validation
  5. Manage their own release tags and changelogs

Positive Consequences

  • Release ceremony is ~10 minutes faster (no submodule generation/push)
  • Single submodule failure cannot break muscovite release
  • Consumer repos can adopt new muscovite versions at their own pace
  • TLA+ gate is deterministic (fixed internal specs, not generated output)
  • Build profiling is framework-only (stable, reproducible)

Negative Consequences

  • Consumer repos must set up their own CI (mitigated by FTR-410 template)
  • MUSCOVITE_CPP_VERSION must be communicated (file at repo root, already exists)
  • Submodule LOC metrics may be stale if not initialized (graceful degradation added)

AI Agent Guidance

When This Decision Applies

  • Any modification to release.sh that would re-introduce example submodule writes
  • Any generator change that assumes examples/*/generated/ is the deployment target
  • CI/CD setup for consumer repos using muscovite-cpp

When This Decision Does NOT Apply

  • muscovite-cpp sync (framework library push) — this remains in release.sh
  • Local development where developers run generation into examples/ for testing

Enforcement

  • release.sh no longer contains example generation or submodule push blocks
  • generate-system-brief.sh and generate-effort-estimation.sh gracefully degrade
  • Rule 2 in copilot-instructions.md updated to reflect consumer-repo model

Key Files

  • workflow/scripts/release.sh — release ceremony (submodule blocks removed)
  • tools/github-actions/dsl-lint.yml — consumer repo CI template
  • MUSCOVITE_CPP_VERSION — single source of truth for consumer FetchContent tag
  • .github/copilot-instructions.md — Rule 2 updated
  • AGENTS.md — release ceremony section updated
