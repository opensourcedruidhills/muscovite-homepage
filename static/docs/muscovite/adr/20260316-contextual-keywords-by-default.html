Contextual Keywords by Default

Contextual Keywords by Default

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-16
  • Tags: parser, dsl, keywords

Technical Story: v1.67.0 release blocker — 8 brownfield keywords (CLASSIFICATION, BODY, PARAMS, RETURNS, REFERENCES, CALLS, INOUT, OUT) collided with domain identifiers in existing .ddd files.

Context and Problem Statement

When adding new DSL keywords to the Muscovite lexer, the default approach has been to add them to the global keyword list in parser/lexer.rkt. The lexer performs case-insensitive matching (string-upcase before lookup), so any global keyword prevents that identifier from being used as an entity name, field name, or enum value anywhere in the language.

In v1.67.0, FTR-427 added 8 keywords for LEGACY_VIEW and LEGACY_PROCEDURE support. These collided with identifiers already in use: classification (pedology.ddd field), Classification (remote-sensing.ddd entity), body (todo.ddd field), out (multiple fields). The collision was not caught until the full Racket test suite ran during the release ceremony, costing ~40 min and ~50K tokens to diagnose and fix.

Decision Drivers

  • Keyword collisions silently break existing models (no error, just wrong parse)
  • The lexer's case-insensitive approach means even lowercase identifiers collide
  • Brownfield/legacy constructs (LEGACY_VIEW, LEGACY_PROCEDURE) use many common English words that are likely to appear as domain identifiers
  • Contextual keywords are the standard approach in modern language design (C# async, await, var; SQL REPLACE, KEY; Kotlin field, value)

Considered Options

  1. Contextual by default — New keywords are contextual unless they introduce a top-level construct. Collision check required before promoting to global.
  2. Global by default with collision check — Keep current approach but add a lint pass to detect collisions before committing.
  3. Status quo — Add keywords to global list, fix collisions when discovered.

Decision Outcome

Chosen option: "Contextual by default", because it prevents collisions at the design level rather than catching them after the fact. Combined with a mandatory collision check (FTR-447) for the rare cases where global keywords are needed.

Classification of Keywords

Category When to Use Examples
Global Introduces a top-level construct or is a structural keyword used across all contexts ENTITY, CONTEXT, ENUM, FIELD, LEGACY_VIEW, LEGACY_PROCEDURE
Contextual Used only inside a specific construct; common English word; likely to appear as an identifier CLASSIFICATION, BODY, PARAMS, RETURNS, REFERENCES, CALLS, INOUT, OUT

Rules

  1. New keywords MUST be contextual unless they meet ALL of:
    • Introduces or terminates a top-level block (e.g., ENTITY, CONTEXT)
    • Not a common English word likely used as an identifier
    • Grep of all .ddd files shows zero collisions
  2. Before adding ANY global keyword, run: grep -ri '<keyword>' examples/**/*.ddd
  3. Contextual keywords use match-contextual-keyword? in parser.rkt
  4. The contextual-keywords list in parser.rkt must include all contextual keywords so expect-identifier-or-contextual! allows them as identifiers elsewhere

Positive Consequences

  • Eliminates keyword collision as a class of release blockers
  • Domain modelers can freely use common words as identifiers
  • Aligns with modern language design practice

Negative Consequences

  • Slightly more complex parser code (contextual matching vs. token type check)
  • Parser functions for contextual keywords must handle the identifier token type

Pros and Cons of the Options

Option 2: Global by default with collision check

  • Good, because simpler parser code (just check token type)
  • Good, because lint catch would work
  • Bad, because collision check is reactive (catches after commit, not before design)
  • Bad, because adds CI overhead for something preventable by design

Option 3: Status quo

  • Good, because no process change
  • Bad, because v1.67.0 proves this costs real money and time
  • Bad, because brownfield pipeline will add many more common-word keywords

AI Agent Guidance

When This Decision Applies

  • When adding ANY new keyword to dslgen/parser/lexer.rkt
  • When implementing new DSL constructs that introduce sub-block keywords
  • When reviewing grammar changes in dslgen/grammar/muscovite.peg

When This Decision Does NOT Apply

  • Existing global keywords (no retroactive change required)
  • Keywords that are structurally impossible as identifiers (e.g., ENTITY — never appears inside a field list)

Enforcement

  • Code review: new keywords in the global keyword list require explicit justification and a grep -ri collision check across all .ddd files
  • FTR-447 (planned): automated lint check in tools/lint-ddd.sh
  • Parser test: test-grammar-conformance.rkt tracks contextual keywords in grammar-only-expected set

Key Files

  • dslgen/parser/lexer.rkt — global keyword list (lines ~80-130)
