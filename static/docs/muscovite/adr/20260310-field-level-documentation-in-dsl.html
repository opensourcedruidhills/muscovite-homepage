ADR: Field-Level Documentation in DSL

ADR: Field-Level Documentation in DSL

  • Date: 2026-03-10
  • Status: accepted
  • Deciders: Johannes Lochmann

Context

The Muscovite DSL supports entity-level DOC clauses that generate documentation across all 8 targets (PostgreSQL COMMENT ON TABLE, C++ Doxygen, Proto comments, system manuals). However, field-level documentation is currently impossible despite the infrastructure being fully ready:

  • IR: field struct has a doc slot (ddd-types.rkt:345) — always #f
  • Parser: parse-field hardcodes #f for doc (parser.rkt:453)
  • Grammar: Field rule has no DocClause? (muscovite.peg:124)
  • Generators: Already emit field docs when present:
    • PostgreSQL: COMMENT ON COLUMN with auto-fallback (sqitch-entity-columns.rkt:495-506, FTR-162)
    • C++ Doxygen: /// on struct members (unified-entity.rkt:178-191, FTR-073)
    • C++ DBA: /// @brief on getters (domain-helpers.rkt:31-40, FTR-089)
    • Proto: // above field declarations (proto-fields.rkt:35-40, FTR-074)

The sample projects demonstrate the gap:

  • FHIR Health: 1,692 .ddd lines, 0 field-level docs, 0 SPECIFICATIONs, 0 SAGAs in clinical contexts
  • Geosciences: ~9,750 .ddd lines, 0 field-level docs despite 429 entities with rich domain knowledge

Real-world domain standards (FHIR R4, GeoSciML) define field-level descriptions, constraints, and terminology bindings that should be captured in the DSL.

Decision

1. Enable Field-Level DOC Syntax

Extend the PEG grammar to allow optional DOC clauses on fields:

Field <- FieldName TypeSpec DefaultClause? FieldAnnotation* FieldDocClause?
FieldDocClause <- 'DOC' STRING

Parser changes: call parse-doc instead of hardcoding #f in parse-field.

This is a backward-compatible grammar extension — existing .ddd files without field DOC remain valid.

2. Create FHIR R4 Extraction Tool

Build a Python script that reads FHIR R4 StructureDefinition JSON bundles and generates .ddd context stubs with:

  • All fields with correct types
  • DOC strings from FHIR element definitions
  • CHECK constraints from FHIR invariants
  • SYNONYMS from common clinical abbreviations
  • Cardinality as @REQUIRED annotations

Human curation follows generation — the tool produces drafts, not final models.

3. Enrich Sample Projects Systematically

  • FHIR Health: Expand from 7 modeled resources to ~20 core clinical resources. Add field-level DOC, CHECKs, SPECIFICATIONs, SAGAs for clinical workflows.
  • Geosciences: Add field-level DOC to existing 429 entities using GeoSciML/FGDC documentation. Tighten scientific range constraints.

Consequences

Positive

  • Field-level documentation flows to all 8 generator targets with zero generator changes
  • .ddd lines → generated lines ratio improves (richer input → richer output)
  • Sample projects become credible domain showcases, not toy examples
  • FHIR extraction tool is reusable for any FHIR-based project
  • Generated system manuals contain field-level documentation

Negative

  • .ddd files grow larger with DOC annotations (acceptable — documentation is content)
  • FHIR extraction tool requires Python dependency (already in project for other tools)
  • One-time manual curation effort for generated stubs

Risks

  • FHIR R4 StructureDefinition format may have edge cases not covered by initial extractor
  • Geosciences enrichment requires domain knowledge verification
