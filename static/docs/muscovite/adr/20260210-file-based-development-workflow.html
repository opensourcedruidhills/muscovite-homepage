File-Based Development Workflow

File-Based Development Workflow

  • Status: accepted
  • Date: 2026-02-10
  • Tags: workflow, ai-agents, release-management

Context and Problem Statement

AI agents working on Muscovite struggle to conclude roadmap items predictably. The current workflow uses ROADMAP.md for planning and ADRs for decisions, but lacks:

  1. Structured feature lifecycle: No formal path from idea → evaluation → planning → implementation → verification → release
  2. Machine-readable task specs: Agents need structured acceptance criteria, not prose descriptions
  3. Release ceremony automation: Documentation updates, changelog, tagging, and pushing are manual and inconsistent

The workflow must live entirely in the repository with no cloud service dependencies.

Decision Drivers

  • AI agents must parse task specifications reliably
  • All workflow state must be git-tracked (auditability)
  • Release ceremony must be scriptable (no manual steps)
  • Must support solo and small-team workflows
  • No external tooling beyond git and shell

Considered Options

  1. File-based workflow — Directory structure with markdown files + shell scripts
  2. git-bug — Embedded issue tracker in git refs
  3. Fossil VCS — Built-in wiki, tickets, forum
  4. Org-mode / plain text TODO — Single-file task tracking

Decision Outcome

Chosen: File-based workflow (Option 1)

Markdown files in a workflow/ directory with naming conventions and shell scripts for lifecycle transitions.

Rationale

  • AI agents read markdown files natively — no CLI wrapping needed
  • Git tracks all state transitions as diffs
  • Shell scripts automate the release ceremony
  • Templates enforce consistent structure
  • Features have structured fields (status, acceptance criteria, test commands) that agents can parse and act on

Structure

workflow/
├── ideas/              # Intake: raw ideas
├── accepted/           # Evaluated + approved
├── versions/vX.Y.Z/   # Iteration planning + assigned features
├── completed/vX.Y.Z/  # Archive after release
├── templates/          # Standardized formats
└── scripts/            # Automation (new-idea, accept, plan, release)

Feature Lifecycle

idea → accepted → planned (assigned to version) → in-progress → done → released (archived)

Key Files

File Purpose
workflow/templates/idea.md Idea intake template
workflow/templates/feature.md Feature spec with acceptance criteria
workflow/templates/iteration.md Version planning with quality gateway
workflow/templates/release-checklist.md Release ceremony checklist
workflow/scripts/release.sh Automated release ceremony
workflow/scripts/generate-system-brief.sh Builds SYSTEM-BRIEF.md
SYSTEM-BRIEF.md Comprehensive project description for sparring sessions

Slash Commands

Command Prompt File Script
/new-idea new-idea.prompt.md new-idea.sh
/accept-idea accept-idea.prompt.md accept-idea.sh
/plan-iteration plan-iteration.prompt.md plan-feature.sh
/release release.prompt.md release.sh
/system-brief system-brief.prompt.md generate-system-brief.sh
/workflow-status workflow-status.prompt.md — (agent reads files directly)

Consequences

Positive

  • AI agents get structured, parseable task definitions
  • Release ceremony is a single script invocation
  • SYSTEM-BRIEF.md provides a comprehensive sparring file for any session
  • All workflow history is in git (diffs show state transitions)
  • Templates enforce consistent quality across features

Negative

  • More files in repository (mitigated by clear directory structure)
  • Shell scripts require bash (acceptable for Linux/macOS development)
  • Manual discipline needed to follow the workflow (mitigated by slash commands guiding agents)

Neutral

  • Replaces the freeform ROADMAP.md with structured iteration files
  • pitch.md content subsumed by SYSTEM-BRIEF.md

AI Agent Guidance

When This Applies

  • Any time you're asked to implement a feature, check workflow/versions/ for the active iteration
  • Before starting work, verify the feature file exists with acceptance criteria
  • After completing work, update the feature status and check off acceptance criteria

How It's Enforced

  • Slash commands guide agents through the workflow
  • Release script runs quality gates before allowing release
  • Feature files have structured fields that agents can parse

Key Files

  • workflow/README.md — Complete workflow documentation
  • .github/prompts/*.prompt.md — Slash command definitions
  • workflow/scripts/*.sh — Automation scripts
