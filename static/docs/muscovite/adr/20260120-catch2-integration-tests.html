Catch2 Integration Tests Replace pgTAP

Catch2 Integration Tests Replace pgTAP

  • Status: accepted
  • Date: 2026-01-20
  • Tags: testing, catch2, pgtap, integration-tests
  • Supersedes: 20251223-api-only-testing-with-ddd-seeding.md

Context and Problem Statement

Muscovite currently generates pgTAP tests that validate PL/pgSQL stored procedures. With the removal of stored procedures (see ADR: Remove PL/pgSQL API Layer), these tests become obsolete. We need a new testing strategy that validates business logic at the correct layer: C++ domain services.

Decision Drivers

  • Test the right layer: Business logic lives in C++ domain services, not stored procedures
  • End-to-end validation: Tests should exercise full path from service to database
  • Familiar tooling: Catch2 is the existing C++ test framework in Muscovite
  • Real database: Tests must use actual PostgreSQL to validate constraints/FKs
  • CI/CD integration: Tests must run in automated pipelines

Considered Options

  1. Keep pgTAP, mock stored procedures: Doesn't test actual business logic
  2. Unit tests with mocked repositories: Fast but misses integration issues
  3. Catch2 integration tests against real PostgreSQL: Full validation (chosen)
  4. Hybrid: unit + integration tests: More comprehensive but more maintenance

Decision Outcome

Chosen option: "Catch2 integration tests against real PostgreSQL", because:

  • Tests validate actual business logic in domain services
  • Real database catches constraint violations, FK errors, type mismatches
  • Catch2 is already used in Muscovite (consistent tooling)
  • Docker/testcontainers make test database setup reliable

Test Categories

Category Purpose Example
Command Tests Validate write operations Create, update, delete, child operations
Query Tests Validate read operations find_by_id, find_all, specifications
Constraint Tests Validate DB constraints FK violations, unique constraints
Transaction Tests Validate rollback behavior Partial failure scenarios
Event Tests Validate event emission Check domain_events table after commands

Generated Test Structure

// Generated: test/domain/patient_service_test.cpp

#include <catch2/catch_all.hpp>
#include "fhir/domain/patient_command_service.hpp"
#include "fhir/domain/patient_query_service.hpp"
#include "test_fixtures.hpp"

namespace fhir::test {

TEST_CASE("PatientCommandService", "[domain][patient][command]") {
    auto pool = TestFixtures::connection_pool();
    PatientCommandRepository cmd_repo;
    PatientQueryRepository query_repo;
    PatientCommandService cmd_svc(cmd_repo, pool);
    PatientQueryService query_svc(query_repo, pool);

    SECTION("create inserts patient and emits event") {
        CreatePatientRequest req{.name = "John Doe", .birth_date = "1990-01-15"};
        
        auto patient = cmd_svc.create(req);
        
        REQUIRE(patient.id != boost::uuids::nil_uuid());
        REQUIRE(patient.name == "John Doe");
        
        // Verify event emitted
        auto events = TestFixtures::get_events(pool, "Patient", patient.id);
        REQUIRE(events.size() == 1);
        REQUIRE(events[0].event_type == "PatientCreated");
    }

    SECTION("create with invalid FK fails") {
        CreatePatientRequest req{
            .name = "John Doe",
            .primary_provider_id = boost::uuids::random_generator()()  // Non-existent
        };
        
        REQUIRE_THROWS_AS(cmd_svc.create(req), pqxx::foreign_key_violation);
    }

    SECTION("delete removes patient") {
        auto patient = cmd_svc.create({.name = "To Delete"});
        
        cmd_svc.remove(patient.id);
        
        auto found = query_svc.find_by_id(patient.id);
        REQUIRE_FALSE(found.has_value());
    }
}

TEST_CASE("PatientQueryService", "[domain][patient][query]") {

    // ... query tests
}

} // namespace fhir::test

Positive Consequences

  • Tests validate actual business logic location (C++ services)
  • Real database catches constraint violations early
  • Familiar Catch2 patterns for C++ developers
  • CTest integration for CI/CD pipelines
  • Tests serve as documentation for service behavior

Negative Consequences

  • Slower than unit tests (database I/O)
    • Mitigation: Use transaction rollback instead of truncate
    • Mitigation: Parallel test execution
  • Requires PostgreSQL running for tests
    • Mitigation: Docker/testcontainers in CI
  • More test setup code than pgTAP
    • Mitigation: Generate fixtures from DSL model

Migration from pgTAP

  1. Remove pgTAP test generation from pgtap.rkt
  2. Add Catch2 test generation to new test-generator.rkt
  3. Update CMake to build and run Catch2 tests
  4. Update CI/CD to use Catch2 instead of pg_prove

AI Agent Guidance

When This Decision Applies

  • When generating test code for domain services
  • When setting up test infrastructure for new projects
  • When validating aggregate behavior
  • When testing constraint violations

When This Decision Does NOT Apply

  • Schema validation (use Sqitch verify scripts)
  • Performance testing (use dedicated benchmarks)
  • UI testing (separate concern)

Enforcement

  • pgTAP generation is removed from generators
  • Catch2 tests generated for each aggregate
  • CI/CD pipeline runs Catch2 tests, not pg_prove
  • Test coverage required for all command operations
