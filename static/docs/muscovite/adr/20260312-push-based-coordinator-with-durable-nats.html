ADR: Push-Based Coordinator with Durable NATS Transport

ADR: Push-Based Coordinator with Durable NATS Transport

  • Date: 2026-03-12
  • Status: superseded by 20260318-nats-agent-coordination-abandoned
  • Supersedes: Partial supersession of 20260309-nats-jetstream-agent-coordination (polling model)
  • Related: 20260311-nats-jetstream-crash-recovery-postmortem

Context

The current coordinator uses a request/reply polling model: workers call worker poll to pull the next FTR assignment, and the leader calls leader status to discover state changes. NATS is used as an RPC transport, not as a message bus.

During the v1.63.0 crash recovery incident (see ADR 20260311 postmortem), this architecture caused cascading failures:

  1. Stale assignments: After crash, workers polled and received already-completed FTRs because assign_next() walks the dependency graph sequentially — there is no way to skip past completed work without walking each FTR through the state machine.

  2. State corruption: The leader walked 13 unimplemented FTRs through progress → pr-ready → approve to skip past them, corrupting ITERATION.md and coordinator state. This required an emergency amend-state command.

  3. Lost events: The fallback from js.publish() to nc.publish() (line 1498) means events can silently vanish from the durable log.

  4. Unused infrastructure: KV bucket constants (ftr-state, iteration-state, worker-registry) are declared and provisioned but never used in Python code. State exists only in memory, reconstructed from event replay.

With 4 workers planned for v1.64.0, these problems would compound quadratically.

Decision

Refactor the coordinator to a push-based architecture using NATS JetStream durable consumers and KV buckets for state persistence.

Core Changes

1. Push-Based Assignment (No Polling)

Workers subscribe to a per-worker durable JetStream consumer on subject coord.assign.<worker-name>. When the coordinator decides to assign an FTR, it publishes to that subject. The worker ACKs after accepting.

Before: Worker → poll → Coordinator → reply {ftr} → Worker starts
After:  Coordinator → push coord.assign.worker-1 → Worker receives → ACK

If a worker crashes without ACK, JetStream redelivers after ack_wait timeout. The coordinator can then reassign to another worker.

2. Push-Based Leader Notifications

The leader subscribes to a durable consumer on coord.notify.leader.>:

  • coord.notify.leader.pr-ready — worker has a PR for review
  • coord.notify.leader.gate-failed — quality gate failed
  • coord.notify.leader.sync-needed — infra-impactful merge requires sync broadcast

No more polling leader status — events arrive as they happen.

3. KV-Backed State Snapshots

FTR state and worker state are written to KV buckets on every mutation:

  • ftr-state KV: key FTR-NNN → JSON {state, worker, pr, updated}
  • worker-registry KV: key worker-1 → JSON {current_ftr, registered_at}
  • iteration-state KV: key current → full serialized IterationState

Crash recovery becomes: read KV snapshot (instant), then replay only events after the snapshot timestamp. No full event log replay needed.

4. Eliminate Core NATS Fallback

All publishes go through JetStream. No exceptions. The nc.publish() fallback at line 1498 is removed. If JetStream is unavailable, the operation fails loudly.

5. Durable Storage Everywhere

Every NATS resource uses --storage file with the persistent store at /var/lib/nats/muscovite-agent. No in-memory streams, no TTL-based expiry for critical state. Specifically:

Resource Storage TTL Rationale
AGENT-EVENTS stream file 720h Event log for audit + replay
ftr-state KV file 0 FTR state survives crashes
worker-registry KV file 0 Worker state survives crashes
iteration-state KV file 0 Full snapshot for fast recovery
agent-sessions KV file 5m Heartbeat watchdog (TTL intentional)
ftr-claims KV file 0 Ownership claims (explicit release)

What Does NOT Change

  • Domain layer: IterationState, FtrState, FtrRecord, WorkerRecord structs are unchanged. The state machine is sound — only the transport was broken.
  • CLI interface: worker register, worker progress, worker pr-ready, leader review, leader status — all commands remain. Implementation changes.
  • Event sourcing: Events are still published to JetStream for audit and observer. KV snapshots supplement but do not replace the event log.
  • Dependency graph: DEPENDENCIES.toml parsing and assign_next() logic unchanged.

Worker Lifecycle (New)

1. worker register → coordinator creates durable consumer coord.assign.<name>
2. coordinator calls assign_next() → publishes to coord.assign.<name>
3. worker receives assignment → ACK → starts work
4. worker publishes progress to coord.event.worker.progress (JetStream)
5. worker publishes pr-ready to coord.event.worker.pr-ready (JetStream)
6. leader receives notification → reviews → approves/rejects
7. coordinator publishes result to coord.result.<name> (JetStream)
8. worker ACKs result → ready for next assignment
9. coordinator repeats from step 2 if more FTRs available

Crash Recovery (New)

1. Read iteration-state KV → instant state restoration
2. Read worker-registry KV → know which workers were active
3. Check unACKed messages on coord.assign.<name> consumers → detect in-flight assignments
4. Resume: re-publish assignments for unACKed FTRs, push new work to idle workers

No event replay needed for state recovery. Event log is retained for audit only.

Consequences

Positive

  • Zero polling: Workers and leader receive notifications instantly
  • Crash-resilient: KV state survives crashes, no full replay needed
  • 4-worker safe: No sequential polling races, no assignment contention
  • No more amend-state hacks: State lives in KV, can be corrected with a KV put
  • Simpler worker code: Subscribe once, receive work — no poll/sleep loop

Negative

  • More NATS infrastructure: Per-worker durable consumers must be managed
  • Migration effort: ~400 lines of coordinator.py transport layer rewrite
  • Worker CLI changes: Workers need a long-running subscriber mode, not one-shot poll

Risks

  • Per-worker consumers create cleanup obligation (unregister must delete consumer)
  • JetStream ACK semantics require careful timeout tuning (ack_wait)
  • KV snapshot + event log dual-write must be atomic (or at least ordered)
