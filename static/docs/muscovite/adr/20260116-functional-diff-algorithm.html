Functional Style for Diff Algorithm

Functional Style for Diff Algorithm

  • Status: accepted
  • Deciders: Architecture Team
  • Date: 2026-01-16
  • Tags: code-quality, racket, functional-programming

Context and Problem Statement

The current diff-model.rkt uses imperative mutation style with set!:

(define operations '())
(for ([(uuid ctx) (in-hash new-map)])
  (unless (hash-has-key? old-map uuid)
    (set! operations (cons (add-context-op ...) operations))))

This is non-idiomatic Racket and has several problems:

  • Harder to reason about (mutation at a distance)
  • Difficult to test individual detection functions
  • Cannot easily parallelize
  • Prone to ordering bugs

Decision Drivers

  • Racket is a functional-first language
  • Diff operations are pure computations (no side effects needed)
  • Code maintainability and testability
  • Idiomatic Racket style

Decision Outcome

Chosen option: Refactor to pure functional style

Before (Imperative)

(define (diff-contexts contexts-old contexts-new)
  (define old-map (contexts->uuid-map contexts-old))
  (define new-map (contexts->uuid-map contexts-new))
  (define operations '())
  
  ;; Additions
  (for ([(uuid ctx) (in-hash new-map)])
    (unless (hash-has-key? old-map uuid)
      (set! operations (cons (add-context-op ...) operations))))
  
  ;; Removals
  (for ([(uuid ctx) (in-hash old-map)])
    (unless (hash-has-key? new-map uuid)
      (set! operations (cons (remove-context-op ...) operations))))
  
  operations)

After (Functional)

(define (diff-contexts contexts-old contexts-new)
  (define old-map (contexts->uuid-map contexts-old))
  (define new-map (contexts->uuid-map contexts-new))
  
  (append
   (find-context-additions old-map new-map)
   (find-context-removals old-map new-map)
   (find-context-renames old-map new-map)))

(define (find-context-additions old-map new-map)
  (for/list ([(uuid ctx) (in-hash new-map)]
             #:unless (hash-has-key? old-map uuid))
    (add-context-op uuid (bounded-context-name ctx) 'safe)))

(define (find-context-removals old-map new-map)
  (for/list ([(uuid ctx) (in-hash old-map)]
             #:unless (hash-has-key? new-map uuid))
    (remove-context-op uuid (bounded-context-name ctx) 'destructive)))

(define (find-context-renames old-map new-map)
  (for/list ([(uuid new-ctx) (in-hash new-map)]
             #:when (hash-has-key? old-map uuid)
             #:unless (eq? (bounded-context-name (hash-ref old-map uuid))
                          (bounded-context-name new-ctx)))
    (define old-ctx (hash-ref old-map uuid))
    (rename-context-op uuid
                       (bounded-context-name old-ctx)
                       (bounded-context-name new-ctx)
                       'safe)))

Benefits of Functional Style

Aspect Imperative Functional
Testability Must test whole function Can test each finder separately
Composition Manual accumulation Natural with append
Parallelization Not possible (shared state) Trivial with pmap
Debugging Track mutation flow Each function is isolated

Consequences

Positive

  • Cleaner, more idiomatic Racket code
  • Each detection function is independently testable
  • Easier to add new operation types
  • Natural composition with append

Negative

  • One-time refactoring effort
  • Slightly different mental model

AI Agent Guidance

  • Use for/list with #:when/#:unless guards instead of for with set!
  • Break complex detection into small, focused functions
  • Use append to combine operation lists from different detectors
  • Each detector function should return a list (possibly empty)
