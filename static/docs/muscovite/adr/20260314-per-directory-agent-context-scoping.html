ADR 20260314: Per-Directory Agent Context Scoping

ADR 20260314: Per-Directory Agent Context Scoping

Status

Accepted

Context

The main .github/copilot-instructions.md (6KB) is auto-attached to every AI agent session regardless of which part of the monorepo the agent is working in. A generator agent editing dslgen/generators/cpp/ receives all 27 rules including Qt patterns, database conventions, release governance, and multi-agent protocol — most of which are irrelevant noise that consumes context window and dilutes focus.

The alternative of splitting the monorepo into sub-repositories was rejected because:

  • Rule 10 (test-first) requires tests and implementation in the same commit
  • Rule 27 (multi-layer gates) enforces cross-layer validation in a single release.sh run
  • The 17-file iteration reshuffle (PR #158) would require coordinated PRs across 4+ repos
  • Monorepo atomicity is Muscovite's architectural strength

Decision

Use .github/prompts/context-*.prompt.md files to provide directory-scoped agent context. Agents load the relevant context prompt based on their working directory, supplementing (not replacing) the main instructions.

Scoping Strategy

The main .github/copilot-instructions.md retains:

  • Universal rules (licensing, branching, project boundary, generated code integrity)
  • User profile and response format
  • Build & test quick reference (commands only)

Directory-specific rules move to context prompts:

Prompt File Covers Key Content
context-dslgen-generators.prompt.md dslgen/generators/ Template DSL (Rule 19), Racket style (Rule 18), naming conventions (GT §II), IR struct reference
context-dslgen-parser.prompt.md dslgen/parser/, dslgen/grammar/ Grammar-first (Rule 7), PEG format, lexer keywords, parser patterns
context-racket-tests.prompt.md dslgen/tests/ Test fixtures, golden files, tagged suites, heartbeat expectations
context-cpp-framework.prompt.md framework/, include/ C++23 patterns (Rule 22), Qt patterns (Rule 23), SPDX headers, doctest BDD
context-cpp-tests.prompt.md tests/ doctest vs QTest boundary, ctest presets, DB test labels, fixture patterns
context-database.prompt.md database/, Sqitch output PostgreSQL conventions (Rule 3), Sqitch deploy, migration safety, semantic diff

Agent Loading Protocol

  1. Worker claims FTR → determines primary directory from FTR spec
  2. Worker loads main instructions (auto-attached) + relevant context-*.prompt.md
  3. If FTR spans multiple directories, worker loads all applicable context prompts
  4. QA agent loads all context prompts for directories touched by the PR

Relationship to Main Instructions

Context prompts are additive — they provide detail, not overrides. If a context prompt conflicts with the main instructions, the main instructions win. Context prompts may cross-reference each other (e.g., generator context references parser context for IR struct definitions).

Consequences

  • Agents working on generators see ~2KB of focused Racket/template rules instead of 6KB of everything
  • New rules are added to the appropriate context prompt, not the monolithic main file
  • The main instructions file shrinks as directory-specific content migrates out
  • Agent orchestration (coordinator, AGENTS.md) documents which context prompts each role loads
  • Context prompts are versioned alongside the code they describe — no drift
