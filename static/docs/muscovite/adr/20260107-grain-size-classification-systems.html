Grain Size Classification Systems

Grain Size Classification Systems

  • Status: accepted
  • Deciders: Domain modeling team
  • Date: 2026-01-07
  • Tags: sedimentology, classification, lookup-tables

Context and Problem Statement

Sedimentary rock classification requires standardized grain size terminology. Multiple classification systems exist globally, and geologists need to work with all of them for different purposes (research, engineering, international collaboration).

Decision Drivers

  • Need for authoritative grain size classification (Udden-Wentworth is USGS standard)
  • Support for multiple classification schemes (Wentworth, ISO 14688, Krumbein phi)
  • Quantitative boundaries (phi values, metric sizes) for computational analysis
  • Folk-Ward sorting statistics require grain size class lookups

Considered Options

  • Single hardcoded Wentworth scale only
  • Multiple classification systems as lookup entities with FK relationships
  • Flexible string-based classification without constraints

Decision Outcome

Chosen option: "Multiple classification systems as lookup entities", because:

  1. Supports multiple authoritative standards (Wentworth, ISO, etc.)
  2. Provides quantitative phi and metric boundaries
  3. Enables FK relationships from samples to classification
  4. Seeds can be loaded from CSV with canonical values

Grain Size Scale Entity

ENTITY GrainSizeScale IDENTIFIED_BY id (
    id          UUID        NOT NULL,
    code        VARCHAR(30) NOT NULL @unique,
    name        VARCHAR(80) NOT NULL,
    authority   VARCHAR(100),
    year        INTEGER,
    description TEXT
)

Seed values: | code | name | authority | |------|------|-----------| | WENTWORTH | Udden-Wentworth Scale | USGS/Wentworth 1922 | | ISO_14688 | ISO 14688-1:2017 | International Organization for Standardization | | KRUMBEIN | Krumbein Phi Scale | Krumbein 1934 |

Grain Size Class Entity

ENTITY GrainSizeClass IDENTIFIED_BY id (
    id          UUID        NOT NULL,
    scale_id    UUID        NOT NULL,
    class_name  VARCHAR(50) NOT NULL,
    phi_min     DECIMAL     NOT NULL,
    phi_max     DECIMAL     NOT NULL,
    mm_min      DECIMAL     NOT NULL,
    mm_max      DECIMAL     NOT NULL,
    category    VARCHAR(30) NOT NULL,
    sort_order  INTEGER     NOT NULL
)
RELATIONSHIPS {
    scale -> GrainSizeScale REQUIRED ONE;
}

Positive Consequences

  • Samples can reference specific grain size classes via FK
  • Multiple scales supported for international collaboration
  • Quantitative phi/mm values enable computational analysis
  • CSV seeds provide canonical reference data

Negative Consequences

  • Additional complexity vs simple string field
  • Requires maintenance of seed data
