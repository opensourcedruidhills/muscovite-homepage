std::execution for Structured Concurrency

std::execution for Structured Concurrency

  • Status: proposed
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04
  • Tags: c++26, concurrency, async, grpc, std-execution

Technical Story: Muscovite roadmap v3.4+ — evaluating std::execution (sender/receiver) for gRPC async patterns and structured concurrency.

Context and Problem Statement

C++26 ships std::execution, providing a unified framework for expressing concurrency with structured lifetime guarantees. Muscovite's gRPC services currently use async patterns based on gRPC's completion queue and callback APIs with custom infrastructure.

However, Herb Sutter warns: "This feature is currently harder to adopt than most C++ features because it lacks great documentation and is missing some helper libraries."

Decision Outcome

Chosen option: "Prototype in v3.x, adopt in v4.x", because:

  • The feature is valuable (structured concurrency = safety)
  • But the ecosystem is too immature (no gRPC adapter, no Qt bridge)
  • Neither GCC nor Clang has implemented it yet
  • C++29 improvements reduce early-adoption API churn risk

Positive Consequences

  • Prototyping builds institutional knowledge
  • No production risk — current patterns remain the default

Negative Consequences

  • Prototype work may be throwaway if API changes in C++29

AI Agent Guidance

When This Decision Applies

  • When designing new async patterns in the framework
  • When evaluating gRPC or Qt async integration options

When This Decision Does NOT Apply

  • v2.x development (no std::execution in C++23)
  • Synchronous code paths

Enforcement

  • Prototype code lives in framework/execution/ (experimental)
  • Current actor_context / correlation remain production path until v4.x
