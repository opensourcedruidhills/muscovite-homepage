Leader-Worker Agent Coordination via NATS JetStream

Leader-Worker Agent Coordination via NATS JetStream

  • Status: superseded by 20260318-nats-agent-coordination-abandoned
  • Date: 2026-03-11
  • ADR: 20260311-leader-worker-agent-coordination
  • Supersedes: 20260309-nats-jetstream-agent-coordination (fallback philosophy)
  • Context: Multi-agent iteration execution requires centralized coordination

Context

ADR 20260306 established peer-based multi-agent coordination with file-based claims. ADR 20260309 added NATS JetStream as a transport layer but preserved "graceful degradation" (silent fallback to files when NATS is unavailable). In practice:

  • Human operator manually relays information between agent sessions
  • Agents self-claim FTRs without dependency awareness
  • No automated PR review routing or feedback loop
  • No observability into iteration progress without manual queries
  • 24 NATS calls in nats-agent.sh use || true (silent failures)

The target workflow: human plans an iteration with FTRs and dependency graph, starts 3 copilot CLI sessions (1 leader + 2 workers), and the coordinator handles the rest — assignment, review, merge, release.

Decision

1. NATS is Mandatory Infrastructure (NO Graceful Degradation)

The coordinator daemon REQUIRES a running NATS server on port 4223. If NATS is unavailable:

  • Coordinator refuses to start (exit 1)
  • Workers refuse to register (exit 1)
  • Leader refuses to attach (exit 1)
  • No fallback to file-based coordination

Rationale: Silent degradation masks coordination failures. When the coordination backbone is down, work MUST stop — not continue uncoordinated.

2. Python Coordinator Daemon

A long-running Python process (agents/coordinator.py) replaces shell-based agents/nats-agent.sh for coordination:

  • Single persistent NATS connection (no fork/exec per call)
  • asyncio for concurrent event handling
  • Typed state machine per FTR (claimed → progressing → pr-ready → reviewing → merged)
  • Structured logging to agents/logs/coordinator.log
  • CLI interface for leader and worker copilot sessions

3. Leader-Worker Model (Not Peer-Based)

Three roles with distinct permissions:

Role Count Capabilities
Leader 1 Assign FTRs, review PRs, approve/reject, merge, run release
Worker 1-N Register, receive assignment, implement, create PR, receive feedback
Observer 0-N Read-only event stream, no state mutations

Leader is the FIRST session to call coordinator.py leader attach. Workers register via coordinator.py worker register <name>. No election protocol — explicit role declaration.

4. Worker Discovery via NATS KV

Workers register in KV bucket worker-registry/{name} with TTL-based heartbeat:

  • Registration: coordinator.py worker register worker-1
  • Heartbeat: automatic (every 60s, TTL 180s)
  • Discovery: coordinator watches worker-registry bucket for changes
  • Expiry: missed heartbeats → worker marked unavailable → in-flight FTR reassignable

5. Dependency Graph as TOML

Each iteration includes DEPENDENCIES.toml:

[graph]
FTR-356 = []
FTR-357 = []
FTR-360 = ["FTR-356", "FTR-357", "FTR-358", "FTR-359"]
FTR-361 = ["FTR-360"]

Coordinator reads this on startup, validates (cycle detection, missing FTR references), and performs topological sort to determine assignment order. Generates .dot visualization.

6. Communication Flow

Worker → NATS → Coordinator → NATS → Leader
Leader → NATS → Coordinator → NATS → Worker

Message types:

  • worker.registered — worker announces presence
  • worker.progress — FTR implementation milestone
  • worker.pr-ready — PR created, ready for review
  • leader.assign — FTR assigned to worker
  • leader.review-result — approve or reject with feedback
  • leader.merge-complete — PR merged
  • coordinator.ftr-status — state machine transition
  • coordinator.iteration-complete — all FTRs merged, release ready

7. Observability

  • Coordinator log: agents/logs/coordinator.log (structured, daemon lifecycle)
  • Event stream: agents/logs/event-stream.log (human-friendly, tail -f)
  • Observer daemon: agents/observer.py — read-only NATS subscriber, formats events
  • Metrics: per-FTR wall time, review rounds, build cycles → metrics/coordination-report-{version}.json
  • Iteration rollup: total time, per-worker utilization, coordinator overhead

8. Proxy Cost Metrics

Since GitHub Copilot CLI exposes no billing API, track proxy metrics:

Metric Granularity Source
Wall-clock time per-FTR, per-iteration Coordinator timestamps
Agent actions per-FTR CLI invocation count
PR review rounds per-PR reject count before approve
Build/test cycles per-FTR cmake/ctest invocations
Coordinator message count per-iteration NATS stream

Consequences

Positive

  • Fully automated iteration execution (human plans, agents execute)
  • Observable: tail -f event stream, metrics rollup
  • Fail-loud: NATS down = work stops = no silent coordination failures
  • Dependency-aware scheduling: no out-of-order FTR execution

Negative

  • Python dependency added (nats-py, tomllib)
  • NATS server must be running before any agent work (systemd service)
  • More complex setup than file-based claims
  • Single coordinator is a SPOF (mitigated: restart resumes from NATS state)

Risks

  • Coordinator crash mid-iteration (mitigated: all state in NATS, restart replays)
  • Worker heartbeat false-positive expiry (mitigated: 3× heartbeat interval as TTL)
  • NATS server restart loses JetStream state if /tmp (mitigated: move to persistent path)
