ADR: Qt Widgets Only - No QML Client Generation

ADR: Qt Widgets Only - No QML Client Generation

Status: Superseded by 20260206-multi-frontend-architecture
Date: 2025-12-31
Context: Qt gRPC client generation for Muscovite

Decision

We will NOT generate QML-based clients. All frontends will use Qt Widgets exclusively.

Rationale

  1. Build System Complexity: QML module integration in CMake introduces build system complexity, moc file handling, and Qt Autotools conflicts.
  2. Maintenance: Qt Widgets is simpler to maintain and debug in a generated context.
  3. Performance: Widgets is faster for data-heavy (CRUD) applications like Geosciences.
  4. Deployment: Widgets distributes as a single executable without runtime QML dependencies.
  5. Focus: Avoids dividing effort between two incompatible frontend frameworks.

Implications

  • All Qt client generators produce Widgets-only code (no QML files, no QML modules).
  • Qt models and list models are linked into a Widgets executable.
  • CMake targets are straightforward: add_executable() + target_link_libraries().
  • Moc files are generated by set(CMAKE_AUTOMOC ON) and never included directly in source.

AI Agent Guidance

When to Apply

  • Generating Qt clients from Muscovite DDD models.
  • Building Muscovite framework Qt components.
  • Creating Qt-based tooling or UI applications.

When NOT to Apply

  • Qt library code (e.g., muscoviteqt library itself).
  • Pure C++ backends or gRPC servers.

Key Rules

  1. No QML generation: Never emit .qml files from generators.
  2. No moc includes: Moc files are ALWAYS auto-generated by CMake; never include them manually.
  3. CMake-only build: Use set(CMAKE_AUTOMOC ON) and standard Qt6 CMake helpers (qt_add_executable, qt_add_library).
  4. Widgets only: All UI code uses QWidget, QMainWindow, QTableWidget, etc.
