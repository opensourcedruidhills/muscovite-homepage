Build All Libraries as Shared Libraries

Build All Libraries as Shared Libraries

  • Status: accepted
  • Date: 2026-01-13
  • Tags: build, cmake, libraries, licensing

Context and Problem Statement

Muscovite generates C++ libraries (DBA, gRPC services, Qt models) and also provides framework libraries. The question is whether these should be built as static (.a) or shared (.so/.dll) libraries.

Decision Drivers

  • Build speed: Shared libraries compile once and link quickly; static libraries require full re-linking on any change
  • Link speed: Linking against shared libraries is significantly faster than static linking, especially for large projects
  • Licensing: LGPL dependencies (Qt, potentially others) require shared library linking to avoid copyleft obligations on application code
  • Disk space: Shared libraries are stored once; static linking duplicates code in every executable
  • Development iteration: Faster incremental builds improve developer productivity

Considered Options

  • Static libraries only (BUILD_SHARED_LIBS=OFF)
  • Shared libraries only (BUILD_SHARED_LIBS=ON)
  • Mixed approach (some static, some shared)

Decision Outcome

Chosen option: "Shared libraries only", because it provides the best build performance, respects LGPL licensing requirements, and reduces disk usage. All libraries in Muscovite and all generated sample projects MUST be built as shared libraries.

Implementation

CMake Settings:

set(BUILD_SHARED_LIBS ON)

Conan Settings (in conanfile.py):

options = {"shared": [True, False]}
default_options = {"shared": True}

Applies to:

  • Muscovite framework libraries (libmuscoviteserver.so, libmuscovitedba.so, libmuscoviteqt*.so)
  • Generated DBA libraries (lib*_dba.so)
  • Generated gRPC service libraries (lib*_grpc.so)
  • Generated Qt model/widget libraries (lib*_qt.so)

Positive Consequences

  • Faster incremental builds during development
  • Faster CI/CD pipeline execution
  • Proper LGPL compliance for Qt-based components
  • Smaller total disk footprint
  • Enables hot-reload scenarios in the future

Negative Consequences

  • Runtime requires all .so files to be available (via LD_LIBRARY_PATH or RPATH)
  • Slightly more complex deployment (must bundle shared libraries)
  • Minor runtime overhead for symbol resolution

AI Agent Guidance

When This Applies

  • Any CMakeLists.txt generation (muscovite or sample projects)
  • Any conanfile.py generation
  • Any library target creation

Enforcement

  • Generators MUST emit set(BUILD_SHARED_LIBS ON) in CMakeLists.txt
  • Generators MUST NOT create static library targets via add_library(... STATIC ...)
  • Use add_library(libname ...) without STATIC/SHARED (inherits BUILD_SHARED_LIBS)
  • Conan recipes should default shared=True

Key Files

  • src/dslgen/generators/cpp/dba/unified.rkt - DBA library CMake generation
  • src/dslgen/generators/grpc/main.rkt - gRPC library CMake generation
  • src/dslgen/generators/qt/cmake.rkt - Qt library CMake generation
  • CMakeLists.txt - Muscovite root build
