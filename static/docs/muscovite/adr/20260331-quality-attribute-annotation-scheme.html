Quality Attribute Annotation Scheme

Quality Attribute Annotation Scheme

  • Status: draft
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-31
  • Tags: dsl, ddd, quality-attributes, annotation, grammar, ir, architecture
  • Supersedes: none

Technical Story: RFC-001 Extension 4 — add quality attribute annotations to UUID-tracked DSL constructs for architectural decision support.

Context and Problem Statement

Architectural decisions require quality attribute analysis: which aggregates change frequently (volatility), which require high availability, which have strict consistency requirements, which handle large data volumes. This metadata drives service decomposition, infrastructure provisioning, and SLA definitions.

Currently this knowledge is tacit or lives in external documents. It should be expressible in the domain model so generators can produce quality-attribute-aware output and documentation includes architectural decision rationale.

Decision Drivers

  • Same annotation mechanism as compliance (reuse the scheme pattern)
  • Quality attributes differ per project — must be user-definable
  • Must apply to any UUID-tracked construct (aggregates, entities, contexts)
  • Generators need quality metadata for infrastructure hints (partitioning, isolation levels)

Considered Options

Option 1: QUALITY_SCHEME with @quality annotations (chosen)

Same mechanism as COMPLIANCE_SCHEME — user-defined schemes, @quality Scheme:Tag.

  • Good, because identical mechanism to compliance — grammar investment amortized
  • Good, because user-definable vocabulary per project
  • Good, because generators can derive infrastructure hints

Option 2: Combine with compliance into single ANNOTATION_SCHEME

A single ANNOTATION_SCHEME with a KIND discriminator.

  • Good, because fewer grammar rules
  • Bad, because @compliance and @quality have very different generator implications
  • Bad, because loses semantic clarity in the DSL — @annotate is too generic

Decision Outcome

Chosen option: "QUALITY_SCHEME with @quality annotations", because separate keywords preserve semantic clarity while sharing the same underlying mechanism. @compliance and @quality read differently, serve different audiences, and drive different generator behavior.

DSL Syntax

.mus project file — scheme definition

QUALITY_SCHEME Volatility {
    RARELY   DOC "Schema and data change less than once per quarter";
    NORMAL   DOC "Regular change cadence — monthly or per-sprint";
    OFTEN    DOC "High-frequency changes — multiple times per week";
}

QUALITY_SCHEME Criticality {
    AVAILABILITY_HIGH      DOC "99.99% uptime SLA — failover required";
    AVAILABILITY_NORMAL    DOC "99.9% uptime SLA — standard redundancy";
    CONSISTENCY_STRONG     DOC "ACID required — no eventual consistency";
    CONSISTENCY_EVENTUAL   DOC "Eventual consistency acceptable";
    SECURITY_HIGH          DOC "Encryption at rest and in transit, audit logging";
}

QUALITY_SCHEME DataVolume {
    TINY    DOC "< 10K rows expected";
    NORMAL  DOC "10K–10M rows expected";
    HUGE    DOC "10M+ rows expected — partitioning recommended";
}

.ddd model — annotations

ENTITY AuditLog @uuid "..." @quality Volatility:OFTEN DataVolume:HUGE
    IDENTIFIED_BY id ( ... )

AGGREGATE Order @uuid "..."
    @quality Criticality:CONSISTENCY_STRONG Criticality:AVAILABILITY_HIGH
    ROOT Order EVENT_SOURCED;

PEG Grammar

QualityAnnotation <- '@quality' QualityTag+
QualityTag        <- IDENTIFIER ':' IDENTIFIER

IR Implications

  • Every protobuf IR message that carries a UUID also carries repeated QualityTag quality_tags
  • QualityTag message: string scheme = 1; string tag = 2;
  • Scheme definitions stored at project level in the IR

Generator Implications

  • Documentation: architect-scoped docs include quality attribute matrix per context. SLA recommendations derived from criticality tags.
  • Heatmap: volatility heatmap, data volume heatmap, criticality heatmap.
  • PostgreSQL: HUGE data volume → table partitioning hints in generated comments. STRONG consistency → explicit SERIALIZABLE isolation level in transaction wrappers.
  • Infrastructure: feeds future Terraform/Kubernetes generators with sizing hints.

Positive Consequences

  • Quality attributes formalized in the model, not in external documents
  • Generators can produce infrastructure-aware output
  • Architect documentation automatically includes quality analysis

Negative Consequences

  • Adds annotation verbosity alongside compliance annotations
  • Quality scheme design requires architectural expertise

AI Agent Guidance

When This Decision Applies

  • When adding @quality annotations to UUID-tracked constructs
  • When generating documentation that includes quality attribute analysis
  • When generating infrastructure hints (partitioning, isolation, SLA)

When This Decision Does NOT Apply

  • Regulatory compliance concerns (use @compliance instead)
  • Runtime performance metrics (quality annotations are design-time, not runtime)

Enforcement

  • PEG grammar rules for QualityAnnotation
  • Parser validation: referenced scheme:tag must exist in project QUALITY_SCHEME
  • Generators emit quality-aware output only when annotations are present
