Sub-Agent Delegation Guardrails

Sub-Agent Delegation Guardrails

  • Status: superseded by 20260318-nats-agent-coordination-abandoned
  • Deciders: Johannes Lochmann
  • Date: 2026-03-15
  • Tags: multi-agent, delegation, work-orders, quality

Technical Story: v1.66.1 delegation experiment dispatched 3 of 5 FTRs to sonnet-4.5 sub-agents. FTR-418 (tr() wrapping) completed with zero rework. FTR-419 (Theme migration) introduced a library boundary violation (muscoviteqt → muscoviteqt-widgets). FTR-420 (header/source split) produced namespace scope errors, stray braces, and required 2 retries plus manual fixup. Root causes: agents lack architectural context (CMake target graph, library boundaries) and heuristic AC gets over-applied without skip-lists.

Context and Problem Statement

Work orders (ADR 20260313) define the dispatch unit. But the content of a work order determines whether delegation succeeds or fails. v1.66.1 proved that:

  1. Bounded mechanical tasks succeed — "wrap these 15 strings in tr()" → 0 rework.
  2. Tasks requiring architectural awareness fail — agents don't know which CMake target owns which source file, which library links to which, or which headers are template-only.
  3. Heuristic AC gets over-applied — "move all implementations to .cpp" sounds mechanical but requires judgment (templates must stay, namespace scope must be preserved).
  4. Large-scope tasks exceed context windows — 140 files can't fit in a single agent context. Agents either skip files or hallucinate changes.

Without guardrails, delegation is a coin flip: mechanical tasks save 40-50% cost, but cross-layer tasks can cost more than doing them manually due to rework.

Decision Drivers

  • v1.66.1 rework cost on FTR-419/420 partially offset delegation savings
  • v1.66.2 has 140-file FTR-425 and heuristic FTR-422 — same failure modes predicted
  • Coordinator work orders need content standards, not just complexity tiers
  • Agents are stateless — they can't discover architectural constraints at runtime

Considered Options

  1. Status quo — complexity tiers only, agent figures out constraints
  2. Guardrails in work order content — mandatory sections for context injection
  3. Pre-flight audit scripts — generate exact change lists before dispatching

Decision Outcome

Chosen option: "Guardrails in work order content + pre-flight audit scripts", because the combination addresses both the context gap (guardrails) and the scale problem (audit scripts chunk work into verifiable units).

Mandatory Work Order Sections (new)

Every work order MUST include these sections beyond the existing template:

1. Architectural Context Block

## Architectural Context
- **Target**: muscoviteqt-widgets (framework/qt/CMakeLists.txt:78-98)
- **Links to**: muscoviteqt, muscoviteqt-models, Qt6::Widgets
- **Linked by**: (nothing — leaf library)
- **Constraint**: Cannot reference symbols from muscoviteqt (circular dep)

For Racket work: module dependency chain. For C++: CMake target + link graph. For SQL: which Sqitch plan owns the change.

2. Explicit Skip-List

## Skip List (DO NOT modify)
- `entity_edit_dialog.hpp:186-199` — template method, must stay in header
- `qt_type_base.hpp` — CRTP template, entire file must stay in header
- `theme.hpp` inline getters — trivial, intentionally inline

Generated by a pre-flight audit script. Agent MUST NOT touch listed files/ranges.

3. Pre-Computed Change Sites

## Change Sites (exhaustive)
1. `framework/dba/src/transactor.cpp:45` — `fmt::format("...")` → `std::format("...")`
2. `framework/dba/src/transactor.cpp:89` — `fmt::format("...")` → `std::format("...")`
...

For mechanical tasks (≥10 sites), a pre-flight script enumerates every site. The agent applies exactly these changes — no discovery, no judgment.

4. Chunk Size Limit

  • Maximum 25 files per work order for mechanical tasks
  • Maximum 5 files per work order for cross-layer tasks
  • Large FTRs (e.g., 140 files) decompose into multiple WOs with explicit file lists

Pre-Flight Audit Scripts

Before creating work orders, the coordinator (or leader) runs an audit script that:

  1. Enumerates change sites — grep/AST scan produces exact file:line list
  2. Identifies skip-list entries — heuristics flag templates, CRTP, inline-intentional
  3. Chunks by CMake target — groups files by owning library to prevent cross-boundary errors
  4. Emits work order JSON — ready for coordinator.py leader work-order create

Scripts live in tools/audit/ and are FTR-specific. Examples:

# FTR-424: enumerate all fmt::format sites, group by CMake target
tools/audit/audit-fmt-format.sh > tmp/ftr-424-sites.json

# FTR-425: enumerate bare for-loops, classify iterator type, chunk by 20 files
tools/audit/audit-bare-for.sh > tmp/ftr-425-sites.json

Verifiable-Only AC

Work orders MUST NOT include unverifiable claims. Specifically:

  • ❌ "5-10% performance improvement" (no benchmark gate)
  • ❌ "binary size reduction" (no measurement command)
  • ✅ "0 bare for loops in dslgen/generators/postgres/" (grep-verifiable)
  • ✅ "fmt not in target_link_libraries" (grep-verifiable)

If a performance claim matters, add a benchmark command to the AC with a threshold.

Positive Consequences

  • Agents receive complete architectural context — no discovery needed
  • Skip-lists prevent over-application of heuristic rules
  • Pre-computed change sites eliminate judgment from mechanical tasks
  • Chunking keeps work within agent context windows
  • Every AC is verifiable by a shell command

Negative Consequences

  • More upfront work for coordinator/leader before dispatching
  • Audit scripts are FTR-specific — must be written per task type
  • Rigid chunking may split logically related changes across WOs

Pros and Cons of the Options

Option 1: Status quo (complexity tiers only)

  • Good, because zero overhead for coordinator
  • Bad, because v1.66.1 proved agents lack architectural awareness
  • Bad, because heuristic AC causes over-application
  • Bad, because 140-file tasks exceed context windows

Option 2: Guardrails in work order content

  • Good, because agents get complete context upfront
  • Good, because skip-lists prevent known failure modes
  • Bad, because requires manual work order authoring
  • Bad, because doesn't solve the enumeration problem

Option 3: Pre-flight audit scripts (chosen, combined with option 2)

  • Good, because automates change site discovery
  • Good, because chunks work by architectural boundary
  • Good, because produces machine-readable work order input
  • Bad, because scripts are FTR-type-specific (one-time cost per pattern)

Addendum: v1.66.3 Evidence — Constrained Delegation Pipeline (2026-03-15)

v1.66.3 delivered 4 FTRs using the guardrails from this ADR plus Python migration scripts. Results provide quantitative evidence for a 4-stage constrained delegation pipeline that combines input-space restriction (prevent wrong roads) with output verification (catch remaining errors).

Quantitative Evidence

FTR Method Coverage Rework
FTR-424 (fmt→std::format, 76 sites) Python script 100% 0
FTR-425 (in-list, 900 sites) Python script 100% 9 false positives in let bindings
FTR-422 (COMMENT ON, 182 indexes) sonnet-4.5 agent 8% (15/182) 92% manual completion
FTR-435 (context prompts, ≤4096 bytes) gpt-4.1 agent 6261 bytes (over budget) Full rewrite by coordinator

Key findings:

  1. Python scripts for >50-site mechanical transforms — zero agent compute, 100% accurate for straightforward patterns (FTR-424/425). Cost: ~0 vs ~3h agent compute.
  2. Scripts still have edge cases — FTR-425 script incorrectly wrapped 9 let/let* bindings with in-list. Caught by runtime contract violation, not grep.
  3. Agent coverage without quantitative targets is unreliable — FTR-422 agent self-reported done at 8% actual coverage. Without the Python counter script, this would have shipped.
  4. Size/budget constraints must be numeric — FTR-435 specified "≤4096 bytes"; agent produced 6261 bytes. Numeric constraints enable automated pre-flight rejection.

4-Stage Constrained Delegation Pipeline

The guardrails in this ADR address Stage 1 and parts of Stage 3. This addendum formalizes the complete pipeline:

Stage 1: Constraint Specification (prevent wrong roads)

Already in this ADR: Architectural Context Block, Skip-List, Pre-Computed Change Sites.

New constraint categories from v1.66.3:

Constraint Type Example Failure It Prevents
Quantitative target "181/181 indexes must have COMMENT ON" FTR-422: agent stops at 8%
Size budget "≤4096 bytes total" FTR-435: agent produces 153% of budget
Scope boundary "only for clauses, NOT let/let* bindings" FTR-425: false in-list insertion
Tool selection ">50 sites → Python script, not agent" FTR-424/425: eliminates agent compute
Negative examples "WRONG: (let ([x (in-list xs)]) ...)" FTR-425: pattern misapplication

Stage 2: Pre-Flight Validation (reject before execution)

Run automated checks BEFORE dispatching work to an agent:

# Verify chunk size
jq '.change_sites | length' tmp/ftr-NNN-sites.json  # must be ≤25

# Verify no skip-list files in change set
comm -12 <(jq -r '.skip_list[]' wo.json | sort) <(jq -r '.change_sites[].file' wo.json | sort)
# must be empty

# Verify tool selection rule
SITE_COUNT=$(jq '.change_sites | length' tmp/ftr-NNN-sites.json)
[ "$SITE_COUNT" -gt 50 ] && echo "REJECT: use Python script, not agent"

Stage 3: Embedded Verification (agent self-checks)

Work orders include verification commands the agent MUST run before reporting done:

## Verification (run before reporting done)
1. `grep -rn 'fmt::format' framework/ | wc -l` → must be 0
2. `python3 tools/audit/count-comment-on.py | tail -1` → must show "182/182"
3. `wc -c .github/copilot-instructions.md` → must be ≤4096
4. `racket tests/test-generators.rkt` → exit 0

Stage 4: Coordinator Spot-Check (catch remaining errors)

Coordinator runs independent quantitative checks after agent reports done:

# Coverage check (independent of agent's self-report)
python3 tools/audit/count-comment-on.py  # compare against target
wc -c .github/copilot-instructions.md    # compare against budget

# Runtime verification (catches edge cases scripts/agents miss)
racket tests/run-all-tests.rkt           # contract violations surface here

Tool Selection Decision Tree

Is the transform mechanical (pattern → replacement)?
├─ YES: How many sites?
│  ├─ >50 sites → Python script (NEVER agent)
│  ├─ 10-50 sites → Agent with pre-computed change list
│  └─ <10 sites → Agent with architectural context
└─ NO: Does it require cross-layer knowledge?
   ├─ YES: Opus-tier agent, max 5 files per WO
   └─ NO: Sonnet-tier agent, max 25 files per WO

Updated Work Order Template Additions

Work orders SHOULD include (in addition to existing mandatory sections):

## Quantitative Targets
- [ ] {metric}: {current} → {target} (command: `{verification command}`)

## Tool Selection
- Method: {python-script | agent-mechanical | agent-cross-layer}
- Justification: {site count | complexity reason}

## Negative Examples (DO NOT produce)
- WRONG: `{anti-pattern with explanation}`
- WRONG: `{anti-pattern with explanation}`

AI Agent Guidance

When This Decision Applies

  • When creating work orders for any FTR with ≥10 change sites
  • When delegating tasks that touch C++ framework code (CMake target boundaries)
  • When delegating Racket generator changes across multiple modules
  • When any AC contains heuristic language ("auto-derive", "move all", "migrate")

When This Decision Does NOT Apply

  • Single-file documentation tasks (e.g., updating one .md file)
  • Work orders with ≤3 change sites where the coordinator can verify manually
  • Architectural work orders (opus-tier) where the agent IS the architect

Enforcement

  • Coordinator validates work order sections before dispatch
  • tools/audit/ scripts generate change site lists and skip-lists
  • Work order template updated in workflow/templates/workorder.md
  • Release retrospective tracks delegation success rate per guardrail

Key Files

  • workflow/templates/workorder.md — work order template (to be updated)
  • tools/audit/ — pre-flight audit scripts (to be created per FTR pattern)
  • doc/AGENT_COORDINATION_LEARNINGS.md — delegation metrics per release
  • agents/coordinator.py — work order creation and validation
