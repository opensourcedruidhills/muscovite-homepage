FOREIGN CONTEXT — Generated Anti-Corruption Layer for External Systems

FOREIGN CONTEXT — Generated Anti-Corruption Layer for External Systems

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-23
  • Tags: ddd, acl, anti-corruption-layer, foreign-context, external-integration, translation

Technical Story: External systems (payment gateways, legacy ERPs, external EHRs) have their own models that Muscovite cannot annotate with UUIDs or own. The domain must be insulated from their concepts via a typed Anti-Corruption Layer. This ADR introduces FOREIGN CONTEXT as a first-class DSL construct that generates typed C++ adapters.

Context and Problem Statement

Domain aggregates may need to communicate with external systems. Without a generated ACL, developers must either:

  1. Expose external type models directly in domain code (coupling)
  2. Write translation adapters by hand (no contract enforcement, no versioning)

DDD prescribes the Anti-Corruption Layer pattern for exactly this case. FOREIGN CONTEXT makes the translation contract explicit, declarative, and generator-enforced.

Placement question: Should FOREIGN CONTEXT be in .ddd (domain) or .mus (project)?

The ACL translation mappings encode how the domain interprets external concepts — that PaymentGateway.status == "APPROVED" maps to success = true is a domain claim with domain consequences. This is core bounded-context knowledge. It belongs in .ddd. What credentials to use and which endpoint URL to call is infrastructure → .mus.

Decision Drivers

  • ACL translation is domain knowledge, not project configuration → .ddd
  • Translation expressions must be declarative-only (no Turing-complete logic in DSL)
  • The domain never sees the foreign type; the adapter is the boundary
  • Complex translations that cannot be expressed declaratively belong in Layer 2 Sidecar
  • Grammar-first: PEG grammar rule before parser implementation

Considered Options

  1. FOREIGN CONTEXT in .ddd — translation contract is domain knowledge
  2. FOREIGN CONTEXT in .mus — external system integration is project configuration
  3. Hand-written ACL with no DSL support — no generation, just convention

Decision Outcome

Chosen option: Option 1 — FOREIGN CONTEXT in .ddd.

DSL Syntax

FOREIGN CONTEXT PaymentGateway @uuid "..." {

    FOREIGN_TYPE PaymentRequest @uuid "..." (
        amount       DECIMAL NOT_NULL,
        currency     TEXT    NOT_NULL,
        reference    TEXT    NOT_NULL
    )

    FOREIGN_TYPE PaymentResult @uuid "..." (
        status       TEXT NOT_NULL,
        txn_id       TEXT,
        error_msg    TEXT
    )

    TRANSLATE PaymentRequest FROM Order.PaymentCommand @uuid "..." {
        amount    <- total_amount_cents / 100.0,
        currency  <- currency_code,
        reference <- order_id::TEXT
    }

    TRANSLATE Order.PaymentResult FROM PaymentResult @uuid "..." {
        success    <- status == "APPROVED",
        reference  <- txn_id
    }
}

IR Structs

foreign-context:

  • uuid — String: stable UUID for semantic diff
  • name — Symbol: foreign context name
  • foreign-types — List of foreign-type
  • translate-rules — List of translation-rule
  • doc — String or #f

foreign-type:

  • uuid — String
  • name — Symbol
  • fields — List of field (same as entity field struct)

translation-rule:

  • uuid — String
  • target-type — Symbol: Muscovite type being translated into or from
  • source-type — Symbol: FOREIGN_TYPE or Muscovite type
  • direction — Symbol: 'to-foreign | 'from-foreign
  • mappings — List of field-mapping

field-mapping:

  • target-field — Symbol
  • expression — String: declarative expression (validated against permitted sub-grammar)

Declarative TRANSLATE Sub-Grammar

TRANSLATE expressions are restricted to a declarative sub-grammar enforced by the parser. This is not convention — it is a parse error to use prohibited constructs.

Permitted:

  • Field references: field_name
  • Qualified field references: entity.field_name
  • Type casts: field_name::TEXT, field_name::DECIMAL
  • Arithmetic: +, -, *, /, with numeric literals
  • String concatenation: field1 || " " || field2
  • Comparison: field == "LITERAL", field != "LITERAL" (produces boolean)
  • Null coalescence: field ?? "default"

Prohibited (parse error):

  • Control flow (IF, CASE, WHEN)
  • Function calls (except type casts)
  • Variables or let bindings
  • Any multi-statement expression

Complex translations that cannot be expressed in this sub-grammar belong in a Layer 2 Sidecar stub. The generator emits a // TODO: complex translation stub method with the input/output types pre-filled.

Generated Output

For each FOREIGN CONTEXT, the generator produces:

Layer 2 (always regenerated):

  • {foreign_context}_adapter.hpp — typed C++ adapter class: 
    class PaymentGatewayAdapter {
public:
    PaymentRequest translate_to(const Order::PaymentCommand& cmd) const;
    Order::PaymentResult translate_from(const PaymentResult& result) const;
};
  • Simple declarative mappings are implemented inline in the header
  • Complex mappings (TODO stubs) are declared as virtual methods

Layer 2 Sidecar (generated once, user-owned):

  • {foreign_context}_complex_translations.cpp — only emitted if complex mappings exist
  • Contains the virtual method stubs marked with sentinel
  • Developer fills in the complex logic here

Transport stub (Layer 2 Sidecar, generated once):

  • gRPC client stub if external protocol is gRPC
  • REST client stub (using framework HTTP client) if protocol is HTTP
  • Protocol declared in .mus as infrastructure configuration: 
    EXTERNAL_SYSTEM PaymentGateway
    PROTOCOL: gRPC
    ENDPOINT_ENV: PAYMENT_GW_URL  # read from environment at runtime

Endpoint Configuration Placement

External system endpoints are not in the DSL. They are:

  • Endpoint URL: environment variable at runtime (12-factor)
  • Protocol selection: .mus EXTERNAL_SYSTEM block

This keeps the translation contract (domain knowledge, .ddd) separate from infrastructure configuration (.mus + environment).

Positive Consequences

  • Domain never sees foreign types — adapter is the strict boundary
  • Translation contract is machine-readable and versionable via UUID-based semantic diff
  • Declarative expressions are validated at generation time
  • Complex translations have a generated scaffold in the correct Layer 2 Sidecar location

Negative Consequences

  • Declarative sub-grammar is a subset of what developers might want to express (by design — forces complex logic to sidecar where it can be tested independently)
  • Foreign types must be defined in .ddd even though they are not domain types (they are boundary objects — this mirrors how DDD treats ACL value objects)

Pros and Cons of the Options

Option 1: FOREIGN CONTEXT in .ddd (chosen)

  • Good, because translation contract is domain knowledge with domain consequences
  • Good, because UUID-based semantic diff can detect breaking changes in the translation
  • Good, because consistent with DDD bounded context thinking
  • Bad, because .ddd contains types the domain does not own (mitigated: FOREIGN_TYPE is explicitly marked)

Option 2: FOREIGN CONTEXT in .mus

  • Good, because external system integration feels like project configuration
  • Bad, because translation mappings are domain knowledge, not project config
  • Bad, because breaking translation changes would be invisible to the semantic diff pipeline

Option 3: Hand-written ACL

  • Bad, because no contract enforcement
  • Bad, because no generator support for translation scaffolding
