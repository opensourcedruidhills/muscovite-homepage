PostgreSQL Idiomatic Output Quality

PostgreSQL Idiomatic Output Quality

  • Status: accepted
  • Date: 2026-02-27
  • Tags: postgresql, generators, sqitch, idiomacy, quality
  • Deciders: Johannes Lochmann

Technical Story: Systematic audit of PostgreSQL generator output revealed 10 categories of non-idiomatic SQL patterns. This ADR records the canonical PostgreSQL patterns and mandates their adoption across all Sqitch generators.

Context and Problem Statement

A comprehensive audit of 28 Racket generator files in dslgen/generators/postgres/ (6,527 lines) identified patterns that, while functionally correct, deviate from PostgreSQL best practices. These deviations affect maintainability, debuggability, and production DBA acceptance. The audit found 10 distinct categories affecting 13+ files.

See doc/POSTGRES-PATTERNS.md for the full pattern catalog with canonical forms, anti-patterns, and affected files.

Decision Drivers

  • Production DBAs expect idiomatic PostgreSQL — non-standard patterns erode trust
  • pg_stat_user_indexes and EXPLAIN output is harder to read with auto-generated names
  • Missing FK indexes cause sequential scans on JOIN/DELETE operations
  • Shallow verify scripts give false confidence in migration correctness
  • Inconsistent naming makes grep-based debugging unreliable

Decisions

1. UUIDv7 Everywhere (Replace gen_random_uuid)

All UUID primary key defaults MUST use uuidv7(), not gen_random_uuid().

sqitch-common.rkt already defines uuid-default-fn returning "uuidv7()" (L61-63), but 6 infrastructure generator files bypass it with hardcoded gen_random_uuid():

File Occurrences Lines
sqitch-event-changes.rkt 4 L38, L159, L180, L200
sqitch-state-machine.rkt 2 L121, L220
sqitch-saga.rkt 3 L132, L151, L166
sqitch-events.rkt 2 L46, L150
migration-sql-deploy.rkt 1 L174
sqitch-bitemporal.rkt 1 (system trigger)

Fix: Replace all 13 hardcoded gen_random_uuid() with (uuid-default-fn) call from sqitch-common.rkt.

2. Named FK Constraints via ALTER TABLE (No Inline REFERENCES)

All foreign keys MUST be created as named constraints using ALTER TABLE ... ADD CONSTRAINT, not inline REFERENCES in CREATE TABLE.

Inline REFERENCES produces auto-generated constraint names like encounter_patient_id_fkey that are unpredictable and hard to reference in migrations. Named constraints follow the pattern fk_{table}_{column}.

Affected files: sqitch-entity.rkt (L66, L111), sqitch-entity-columns.rkt (L93, L112, L146), sqitch-state-machine.rkt (L113, L122-126, L221-223), sqitch-event-changes.rkt (L181, L201), and others with inline REFERENCES.

Exception: migration-sql-deploy.rkt (L127) already uses named FK constraints — this is the correct pattern to follow.

3. Automatic B-tree Indexes on All FK Columns

Every FK column MUST have a corresponding CREATE INDEX idx_{table}_{column} ON {schema}.{table} ({column}).

PostgreSQL does NOT auto-create indexes on FK columns (unlike MySQL). Missing FK indexes cause sequential scans on:

  • JOIN operations through the FK
  • CASCADE DELETE from parent
  • Query planner lookups

indexes-triggers.rkt generates metadata-driven indexes (L58) but FK indexes are not systematically generated for all FK columns.

4. Column-Level COMMENT ON from field-doc

Every column with a non-null field-doc MUST generate COMMENT ON COLUMN {schema}.{table}.{column} IS '...';.

sqitch-entity-columns.rkt already generates COMMENT ON TABLE (L313-422) but column-level comments are not propagated from field-doc values. Per ADR 20260217, all database objects that support COMMENT ON must have generated comments.

5. Standard String Quoting (No E-prefix)

Seed data INSERT statements MUST use standard SQL string quoting ('...'), not PostgreSQL-specific escape strings (E'...').

sqitch-unified-seeds.rkt sql-escape-value (L68-85) uses E'...' prefix for all string values. Standard '...' quoting with doubled single quotes ('') is sufficient and portable.

6. Idempotent Seeds with ON CONFLICT

All seed INSERT statements MUST include ON CONFLICT (id) DO UPDATE SET to enable re-runnable deployments.

sqitch-unified-seeds.rkt generates bare INSERT INTO (L136-141) without conflict handling. Re-running a Sqitch deploy will fail on duplicate keys. Idempotent seeds are essential for development workflows.

7. Consistent SQL Naming Prefixes

All generated SQL objects MUST follow these naming conventions:

Object Prefix Pattern Example
Foreign key fk_ fk_{table}_{column} fk_encounter_patient_id
Index idx_ idx_{table}_{column} idx_encounter_patient_id
Trigger trg_ trg_{table}_{purpose} trg_patient_audit
Check constraint chk_ chk_{table}_{rule} chk_patient_birth_date_not_future
Unique constraint uq_ uq_{table}_{columns} uq_patient_mrn
Exclusion excl_ excl_{table}_{purpose} excl_patient_valid_time

Existing generators use inconsistent patterns — some use fk_, others generate unnamed constraints, triggers lack consistent trg_ prefix.

8. Deep Verify Scripts

Sqitch verify scripts MUST check actual schema structure, not just SELECT 1 FROM {table}.

Current verify scripts (generated across multiple files) use SELECT 1 FROM {schema}.{table} WHERE FALSE; which only verifies table existence. Deep verify scripts should check:

  • Column existence and types via information_schema.columns
  • Constraint existence via pg_constraint
  • Index existence via pg_indexes
  • Trigger existence via information_schema.triggers

Pattern: DO $$ BEGIN ... ASSERT (SELECT count(*) FROM information_schema.columns WHERE ...) = N; ... END $$;

9. JSONB for Event Outbox Payload

Decision: domain_events.payload columns use JSONB type. This is an explicit exception to the general "no JSONB columns" rule (enforced by parser for domain entity fields).

Rationale: The domain_events table is infrastructure, not a domain entity. Event outbox payloads are:

  • Serialized from typed domain event structs (type safety is at the C++ layer)
  • Consumed by NATS/messaging infrastructure that expects JSON
  • Never queried by field (no payload->>'field' in WHERE clauses)
  • Transient — consumed and archived, not a long-lived query target

The parser-level JSONB prohibition applies to domain model fields in .ddd files. Infrastructure tables generated by the framework (domain_events, saga_state, event_store) are exempt because their schemas are fixed by the generator, not user-defined.

No code change required — current generators already emit JSONB for event payloads. This decision documents the intentional exception.

10. PostgreSQL Pattern Enforcement Test

A dedicated Racket test (test-postgres-patterns.rkt) MUST scan generated SQL output and verify compliance with all patterns in this ADR.

The test should:

  • Parse generated .sql files from examples/hello-world/generated/
  • Verify no gen_random_uuid() outside of explicit exceptions
  • Verify all FK constraints are named (fk_ prefix)
  • Verify FK indexes exist for all FK columns
  • Verify no E'...' string literals in seed files
  • Verify ON CONFLICT clause in seed INSERT statements
  • Verify naming convention compliance

This is a regression safety net — once patterns are fixed, they must not regress.

Positive Consequences

  • Generated SQL passes DBA review without modification
  • FK indexes prevent production performance surprises
  • Named constraints enable reliable migration references
  • Deep verify scripts catch deployment drift
  • Enforceable via automated test

Negative Consequences

  • One-time migration of 13+ generator files
  • Existing generated output changes (requires regeneration of all examples)
  • Verify script size increases (more thorough checks)

Implementation Order

Features are ordered by dependency:

  1. FTR-159 (uuidv7) — standalone, no dependencies
  2. FTR-165 (naming conventions) — establishes naming patterns used by FTR-160, FTR-161
  3. FTR-160 (named FKs) — depends on FTR-165 for fk_ prefix convention
  4. FTR-161 (FK indexes) — depends on FTR-160 (indexes reference FK columns)
  5. FTR-162 (column COMMENT ON) — standalone, extends existing COMMENT ON
  6. FTR-163 (seed best practices) — standalone
  7. FTR-164 (deep verify) — depends on FTR-160, FTR-161 (verifies their output)
  8. FTR-166 (enforcement test) — depends on all above (verifies everything)

AI Agent Guidance

When This Decision Applies

  • When modifying ANY file in dslgen/generators/postgres/
  • When adding new PostgreSQL generator functions
  • When reviewing generated SQL output for correctness
  • When writing Sqitch deploy/verify/revert scripts

When This Decision Does NOT Apply

  • Domain entity JSONB prohibition still applies (parser-enforced)
  • C++, gRPC, Qt generators (separate concerns)
  • Hand-authored SQL in framework migrations (different lifecycle)
