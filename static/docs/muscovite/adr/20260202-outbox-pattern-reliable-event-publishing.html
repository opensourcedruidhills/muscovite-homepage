Outbox Pattern with NATS JetStream and Protobuf

Outbox Pattern with NATS JetStream and Protobuf

  • Status: accepted
  • Date: 2026-02-02
  • Tags: outbox, event-publishing, nats, jetstream, protobuf, reliability

Context and Problem Statement

The Muscovite architecture stores domain events transactionally within the domain_events table of each bounded context. Currently, these events are serialized as JSONB. However, JSONB lacks strict schema enforcement and type safety for external consumers. Furthermore, events currently remain trapped in the database without a mechanism to reach external systems reliably.

Current State (v0.7.0):

  • Events stored as JSONB in PostgreSQL.
  • No external publishing pipeline.
  • Clients poll via gRPC for updates (inefficient).

Goal for v0.8.0: Establish a strict, crash-only event publishing pipeline that bridges PostgreSQL to NATS JetStream using binary Protobuf payloads for maximum type safety and reliability.

Decision Drivers

  • Type Safety: External consumers need strong guarantees about event schemas (Protobuf).
  • Database Purity: The database should act only as a persistence buffer; it should not know about downstream consumers (Open/Closed Principle).
  • Reliability: Events must be persisted atomically with state changes (Outbox Pattern).
  • Traceability: The database tracks "published" state to prevent data loss.
  • Performance: NATS JetStream handles fan-out; the database is not queried by consumers.

Considered Options

  1. Option A: JSONB Payloads + Logical Replication (CDC)

    • Pros: Decoupled.
    • Cons: JSONB is loosely typed; CDC setup (Debezium) adds significant infrastructure complexity.

  2. Option B: JSONB Payloads + Background Polling

    • Pros: Simple.
    • Cons: JSONB payloads require runtime validation by consumers; write amplification if database tracks per-consumer state.

  3. Option C: Protobuf Payloads + NATS JetStream Outbox (Chosen)

    • Pros: Strong typing via Protobuf; database only tracks "handoff to NATS"; NATS handles fan-out; NATS JetStream provides durability.
    • Cons: Requires nats.c integration; binary payloads are harder to query ad-hoc (mitigated by event_type column).

Decision Outcome

Chosen option: "Option C - Protobuf Payloads + NATS JetStream Outbox" with JSON-in-DB / Protobuf-on-wire adaptation.

1. Event Serialization (Layered Architecture)

To maintain strict layer separation (DBA layer must not depend on gRPC/Protobuf), the implementation uses a hybrid approach:

  • Database Storage: Events are serialized as JSON TEXT in the payload column. This keeps the DBA layer independent of Protobuf (no proto includes).
  • Wire Format: The EventService (gRPC layer) parses the JSON payload and constructs typed Protobuf messages for streaming to clients.
  • Schema: Events are defined in .ddd and generate both JSON field mappings (DBA) and Protobuf message definitions (gRPC).
  • Wrapper: A ContextEvent oneof message wraps all possible events for a context in the gRPC layer.

Why JSON-in-DB?

  • DBA layer uses nlohmann::json (already a dependency) - no Protobuf dependency needed.
  • Maintains layer independence: DBA → Database only, gRPC → Protobuf/Wire.
  • JSON is human-readable in database (debugging, auditing).
  • EventService handles JSON→Protobuf conversion at streaming time.

2. Transactional Outbox

  • Persistence: Events are inserted into domain_events in the same transaction as the aggregate state change.
  • State Tracking: processed_at (timestamp) indicates if the event has been successfully handed off to NATS.
  • Crash Recovery: On startup/restart, the publisher polls for all events where processed_at IS NULL.

3. NATS JetStream Publishing

  • Infrastructure: A per-context background OutboxPublisher polls the database.
  • Broker: NatsMessageBroker (wrapping nats.c) publishes binary payloads to NATS.
  • Subject: muscovite.{context_name}.{event_type} (e.g., muscovite.patient.PatientCreated).
  • Persistence: JetStream enabled for durability.

4. Client Updates

  • Bridge: The EventService (gRPC) will subscribe to NATS subjects and forward events to connected clients (e.g., Qt).
  • Qt Models: Models will react to these gRPC streams to update UI state in real-time.

Consequences

  • Breaking Change: Existing domain_events tables must be migrated (drop/recreate or convert column) to support TEXT JSON.
  • New Dependency: nats.c C library required for the C++ backend.
  • Infrastructure: NATS server required in docker-compose.yml.
  • Observability: JSON payloads are human-readable in the database (easier debugging than binary).
  • Layer Independence: DBA layer has no dependency on gRPC/Protobuf generated code.

Implementation Plan (v0.8.0)

  1. Generators: Update Protobuf generators for event messages; DBA emits JSON; gRPC parses JSON and constructs typed Protobuf.
  2. Database: Migrate payload column to TEXT (JSON storage).
  3. Messaging: Implement NatsMessageBroker and OutboxPublisher (JSON→Protobuf conversion).
  4. Integration: Wire up NATS service in Docker and CMake.
