Bitemporal Data Support for Auditable Domain Models

Bitemporal Data Support for Auditable Domain Models

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-21
  • Tags: dsl, bitemporal, temporal, audit, iso-20022, valid-time, system-time, timestamptz

Technical Story: Financial, healthcare, and regulatory domains require tracking two independent timelines for data: when a fact was true in reality (Valid Time) and when it was recorded in the system (System Time). Muscovite currently has full TIMESTAMPTZ support across all layers but no formal bitemporal semantics — no auto-generated temporal columns, no "as-of" queries, no overlap prevention.

Context and Problem Statement

Bitemporality is the requirement to answer four classes of temporal queries:

  1. Current: What is the current known state? (standard query)
  2. As-of Valid Time: What was true at a specific real-world point in time?
  3. As-of System Time: What did the system believe at a specific recording time?
  4. As-of Both: What did the system believe about the real world at time X, as recorded at time Y?

Today, developers can manually add valid_from/valid_to fields using USES Basedata.Period, but this provides no auto-generated query methods, no system-time tracking, no overlap prevention, and no temporal integrity constraints. The bitemporal pattern is well-understood (SQL:2011 temporal tables, ISO 20022 financial messaging, HL7 FHIR audit requirements) but tedious to implement correctly.

Question: How should Muscovite support bitemporal data models in the DSL with full-stack code generation?

Decision Drivers

  • TIMESTAMPTZ already has complete 5-layer mappings (PostgreSQL → C++ muscovite::dba::Timestamp → Proto → Qt)
  • CONCURRENCY block on entities is the exact precedent for entity-level configuration clauses
  • No new type registration needed if using dual-column TIMESTAMPTZ approach
  • ISO 20022 / HL7 FHIR / GDPR audit requirements drive enterprise adoption
  • Non-destructive updates (no physical DELETE/UPDATE on temporal data) align with Event Sourcing (ADR 20260204)
  • PostgreSQL EXCLUDE USING GIST provides database-level temporal overlap prevention

Considered Options

Option 1: Dual-column TIMESTAMPTZ (valid_from, valid_to, system_from, system_to)

Use four TIMESTAMPTZ columns. Valid Time is user-managed (set explicitly in commands). System Time is auto-managed (set by trigger/repository on INSERT/UPDATE). Overlap prevention via EXCLUDE USING GIST on tstzrange(valid_from, valid_to).

Option 2: PostgreSQL range types (TSTZRANGE, DATERANGE)

Use native PostgreSQL range columns (TSTZRANGE valid_period, TSTZRANGE system_period). More idiomatic SQL:2011, native range operators (@>, &&, -|-).

Option 3: SQL:2011 temporal tables (PERIOD FOR)

Use PostgreSQL's emerging SQL:2011 PERIOD FOR syntax with SYSTEM VERSIONING. Most standards-compliant but PostgreSQL support is still evolving (PG 17+ for some features).

Decision Outcome

Chosen option: Option 1 — Dual-column TIMESTAMPTZ, because:

  1. Zero type-registry work: TIMESTAMPTZ is already mapped across all 5 layers. No new C++ strong type, no new Proto message, no new Qt widget needed.
  2. Proto/Qt compatibility: Protocol Buffers and Qt have no native range types. Range columns would decompose at the gRPC boundary anyway — dual columns avoid the impedance mismatch.
  3. GIST still works: EXCLUDE USING GIST (id WITH =, tstzrange(valid_from, valid_to) WITH &&) operates on an expression, not a stored range column. Full overlap prevention without range storage.
  4. Simpler C++ API: Timestamp valid_from, valid_to is clearer than a custom TemporalRange<Timestamp> wrapper.
  5. Migration simplicity: Adding 4 TIMESTAMPTZ columns to existing entities is a standard Sqitch migration. Range columns require btree_gist extension and are harder to backfill.

DSL Syntax

ENTITY Balance @uuid "..." IDENTIFIED_BY id (
    account_id UUID NOT_NULL
    amount Decimal(18,2) NOT_NULL
    currency_code Code NOT_NULL
)
BITEMPORAL {
    VALID_TIME valid_from, valid_to;     -- User-managed: when the fact is true in reality
    SYSTEM_TIME AUTO;                     -- Auto-managed: when the system recorded it
    GRANULARITY MICROSECOND;              -- Optional: MICROSECOND (default) | MILLISECOND | SECOND
}
RELATIONSHIPS {
    account -> Account REQUIRED ONE;
}

Shorthand for valid-time-only (no system versioning):

ENTITY InsurancePolicy @uuid "..." IDENTIFIED_BY id (
    holder_id UUID NOT_NULL
    premium Decimal(10,2) NOT_NULL
)
BITEMPORAL {
    VALID_TIME effective_from, effective_to;
}

Generated Artifacts Per Layer

Layer Artifact
PostgreSQL 4 TIMESTAMPTZ columns (2 if valid-time-only), EXCLUDE USING GIST constraint, system-time trigger (SET system_from = NOW() on INSERT, close + reopen on UPDATE), btree_gist extension
C++ DBA find_as_of(id, valid_at), find_as_of(id, valid_at, system_at), find_history(id), find_corrections(id) repository methods
gRPC valid_at and system_at optional filter fields on Get/List RPCs, GetHistory RPC
Qt Widgets QDateTimeEdit range filters for valid-time and system-time on list views
Qt QML ViewModel temporal filter properties
Sqitch btree_gist extension deploy, temporal columns in entity deploy scripts

Non-Destructive Update Pattern

Bitemporal entities use INSERT + close instead of UPDATE:

-- "Correct" a balance: close the old interval, insert a new one
UPDATE balance SET valid_to = NOW() WHERE id = $1 AND valid_to = 'infinity';
INSERT INTO balance (id, account_id, amount, valid_from, valid_to, system_from, system_to)
VALUES (gen_random_uuid(), $2, $3, $4, 'infinity', NOW(), 'infinity');

The C++ repository's update() method MUST implement this as an atomic operation within the UnitOfWork.

Temporal Query Precision

  • C++ side: Use muscovite::dba::Timestamp (wraps std::chrono::time_point<system_clock, microseconds>)
  • PostgreSQL side: TIMESTAMPTZ stores microsecond precision natively
  • Proto side: google.protobuf.Timestamp has nanosecond precision (superset)
  • No rounding errors in the existing mapping chain for microsecond-level financial reconciliation

Positive Consequences

  • Zero new type registrations — leverages existing TIMESTAMPTZ 5-layer mapping
  • Financial/healthcare/regulatory domains get auditable temporal data out of the box
  • Non-destructive updates align with Event Sourcing philosophy (ADR 20260204)
  • EXCLUDE USING GIST provides database-level overlap prevention ("temporal paradox" guard)
  • "As-of" queries are generated, not hand-written — eliminates a major source of temporal bugs

Negative Consequences

  • 4 extra columns per bitemporal entity — storage overhead and index cost
  • Non-destructive updates produce growing tables — need for archival/partitioning strategy (future ADR)
  • System-time trigger adds INSERT/UPDATE overhead (~5% based on PostgreSQL benchmarks)
  • btree_gist extension must be available (standard in PostgreSQL, but a deployment dependency)
  • Dual-column approach is less idiomatic than SQL:2011 PERIOD FOR — may need migration path when PostgreSQL fully supports temporal tables
