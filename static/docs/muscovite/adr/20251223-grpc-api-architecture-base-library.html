gRPC API Architecture with Base Library and Generated Services

gRPC API Architecture with Base Library and Generated Services

  • Status: accepted
  • Deciders: API architects, Backend developers
  • Date: 2025-12-23
  • Tags: grpc, api, architecture, codegen

Context and Problem Statement

Muscovite needs to expose domain operations via gRPC for client applications (Qt/QML, web, mobile). The API layer must:

  1. Be generated from DDD spec files for consistency
  2. Share common infrastructure (connection pools, error handling, interceptors)
  3. Support future authorization (OAuth2) without breaking changes
  4. Bridge PL/pgSQL stored procedures to gRPC services

How should we structure the gRPC layer to maximize code reuse and maintainability?

Decision Drivers

  • Consistent API generation across all DDD projects
  • Avoid duplicating infrastructure code in every generated project
  • Clean separation between reusable base and project-specific generated code
  • Prepare for OAuth2 integration via interceptors (Phase 2)
  • Leverage existing muscovite libraries (muscovitedba, muscoviteserver)

Considered Options

  • Single monolithic generator (all code in generated project)
  • Base library + generated services (layered architecture)
  • gRPC gateway to REST (indirect approach)

Decision Outcome

Chosen option: "Base library + generated services", because it provides maximum code reuse, clean separation of concerns, and a clear extension point for authorization interceptors.

Phase 1: No Authorization (Current)

  • All endpoints are publicly accessible
  • Focus on correct service implementation
  • Interceptors: logging, metrics, error handling only

Phase 2: OAuth2 Authorization (Future)

  • Add AuthInterceptor to base library
  • JWT validation via interceptor
  • Scope-based authorization per RPC method
  • Generated services annotated with required scopes
  • No changes to generated service code — interceptor handles auth

Positive Consequences

  • Single place for infrastructure (interceptors, error mapping)
  • Generated code is lean and focused on business logic
  • OAuth2 addition requires no regeneration of services
  • Consistent error handling across all projects
  • Easy to test base library independently

Negative Consequences

  • Additional dependency (muscovitegrpc) for all gRPC projects
  • Base library API changes affect all projects
  • Slightly more complex build setup

Base Library Components

muscovitegrpc Library

Racket Generator Design

Generator Responsibilities

  1. Proto Generation (generators/grpc/proto.rkt)

    • Generate message types from entities
    • Generate service definitions from repositories
    • Import common.proto from base library

  2. Service Implementation (generators/grpc/service.rkt)

    • Generate service class inheriting from ServiceBase
    • Map RPC methods to PL/pgSQL API calls
    • Use base library type converters

  3. CMake Integration (generators/grpc/cmake.rkt)

    • Find muscovitegrpc package
    • Proto compilation targets
    • Service library targets

Proto Mapping Rules

DDD Concept Proto Mapping
Entity Message with all fields
Value Object Message (embedded)
Repository.find_by_id rpc GetById(IdRequest) returns (Entity)
Repository.find_all rpc List(ListRequest) returns (stream Entity)
Repository.insert rpc Create(CreateRequest) returns (Entity)
Repository.update rpc Update(UpdateRequest) returns (Entity)
Repository.delete rpc Delete(IdRequest) returns (Empty)
Command rpc <CommandName>(CommandRequest) returns (CommandResponse)

AI Agent Guidance

When This Decision Applies

  • ALL gRPC service generation from DDD specs
  • Adding new interceptors or base functionality
  • Modifying proto mapping rules
  • Extending error handling

When This Decision Does NOT Apply

  • PostgreSQL schema generation (use Sqitch ADR)
  • Qt/QML client generation (separate architecture)
  • Non-gRPC API protocols (REST, GraphQL)

Enforcement

  • Racket generators import and use base library types
  • Generated CMakeLists.txt links muscovitegrpc
  • Code review verifies no infrastructure duplication
