Muscovite 2.0 — Protobuf IR Contract & Unix Pipeline Architecture

  • Status: accepted
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-30
  • Tags: architecture, muscomp, 2.0, IR, protobuf, pipeline
  • Supersedes: Refines Translate Layer 0 from Racket to C++

Technical Story: Muscovite 2.0 architectural foundation — the transition from monolithic compiler to a Unix-philosophy pipeline of standalone executables communicating through a protobuf IR file contract.

Context and Problem Statement

muscomp v1.78.0 successfully translated Layer 0 from Racket to C++. However, the translation preserved the monolithic architecture: a single muscomp_lib static library (195 files, 12K lines) containing the parser, validators, and all 10 generator targets. This creates several problems:

  1. Third-party generators are impossible — adding a C# or Java generator requires forking muscomp and linking into its build system
  2. All generators must be C++ — the monolith couples language choice to the compiler
  3. Compilation is all-or-nothing — changing one generator rebuilds the entire library
  4. The IR is an in-memory data structure — no stable, versioned contract between parser and generators

ADR 20260326 explicitly identified future targets (C#, Java, React, Flutter) as a driver for the C++ transition. The current monolith prevents this.

Decision Drivers

  • Third-party generators must be possible without coupling to muscomp internals
  • Generator authors should be free to use any programming language
  • The IR contract must be stable, versioned, and independently evolvable
  • Built-in generators should use the same integration path as third-party ones
  • muscomp itself should exercise the full Muscovite feature set (dogfooding)
  • Compilation time for the toolchain must improve, not regress

Considered Options

  1. dlopen plugin architecture — shared libraries loaded at runtime via factory functions
  2. Protobuf IR file contract — standalone executables communicating through a serialized IR file
  3. Status quo — keep the monolith, add generators by expanding muscomp_lib

Decision Outcome

Chosen option: "Protobuf IR file contract", because it achieves complete decoupling (any language, any build system, zero dependency on muscomp), provides a stable versioned contract via protobuf schema evolution rules, and reduces the compiler to a thin orchestrator following Unix philosophy.

The dlopen approach (Option 1) was considered and partially adopted as the InProcessDynamic sidecar transport (see ADR 20260330-inprocess-dynamic-transport) for performance-sensitive built-in generators. It complements rather than replaces the file-based contract.

Core Principles

  1. The IR file is the product — model.ir (protobuf binary) is the universal contract between parsing and generation. The .proto schema is versioned independently from muscomp and is the most important artifact in the project.

  2. Every stage is a standalone executable — muscomp parse produces the IR file. Each generator (muscomp-gen-postgres, muscomp-gen-cpp-dba, etc.) is a separate binary that reads the IR file and writes output files. No special distinction between built-in and third-party generators.

  3. A third-party generator is a program that reads a file — no SDK, no framework, no plugin loading mechanism. Deserialize protobuf in any language, write output, done.

  4. CLI split — muscomp parse (source → IR file) and muscomp generate (IR file → output) are separate subcommands. The legacy one-shot syntax (muscomp --project X.mus --postgres --cpp-dba) remains as sugar that runs both stages in-process without writing the IR file to disk.

  5. Parallel generation — standalone executables can run simultaneously. The orchestrator spawns all requested generators in parallel, collects exit codes.

  6. Open GENERATE block — the .mus project file's GENERATE {} block accepts arbitrary identifiers, not a closed keyword set. muscomp parse records the intent in the IR. The orchestrator resolves identifiers to executables at generation time.

CLI Design

muscomp parse

  • Input: .mus project file (references .ddd and .lay files)
  • Output: model.ir (protobuf binary)
  • Runs: PEG parse → AST → semantic validation → IR construction → serialization
  • Optional: --text-format for human-readable protobuf, --ir-version=N for backward compatibility

muscomp generate

  • Input: model.ir + --target=<name> (repeatable)
  • Output: generated files in -o <output-dir>
  • Resolves target name to executable (muscomp-gen-<name>), spawns in parallel
  • Validates IR version compatibility before dispatch

Convenience subcommands (non-blocking):

  • muscomp ir-diff old.ir new.ir — semantic diff between IR versions
  • muscomp ir-dump model.ir — human-readable IR summary

Backward-compatible sugar:

  • muscomp --project X.mus --postgres --cpp-dba internally runs parse + generate in-process, never writes IR to disk

IR Schema Design

  • Protobuf3 with explicit field presence where needed
  • Every IR node carries its UUID (for semantic diff tracking)
  • Cross-references use UUIDs, not pointers — flat serializable graph
  • oneof for type variants
  • ir_version field in root message for compatibility negotiation
  • Reserved field number ranges per major node type (100-199, 200-299, etc.)
  • Schema follows proto3 evolution rules: additive fields, never remove, deprecate only
  • Breaking changes (if ever): muscovite.ir.v1 → muscovite.ir.v2 with migration window
  • Published in repo under proto/muscovite/ir/

Self-Hosting: muscovite-ir.ddd

The IR schema is defined as a Muscovite domain model (muscovite-ir.ddd), making muscomp a Muscovite application whose domain is "DSL compilation."

  • Structural IR types modeled as entities, value objects, aggregates
  • Behavioral contracts (parser, validators, generators) modeled as EXTENSION_POINTs with sidecar implementations
  • Generated artifacts: .proto files, C++ types, documentation, validation stubs
  • Bootstrap protocol: generated .proto files are checked into proto/muscovite/ir/ as bootstrap artifacts. muscomp compiles against the checked-in protos. When muscovite-ir.ddd changes, regenerate protos, update checked-in copy, rebuild muscomp. Same pattern as protoc's descriptor.proto.
  • CI rule: any commit modifying muscovite-ir.ddd must include regenerated protos. CI verifies they match.

Self-hosting will likely require DSL additions (VARIANT/UNION types, MAP types) which benefit all users. This is a feature, not a cost — muscomp becomes the DSL's most demanding customer.

IR Caching and Incremental Builds

Since the IR is a file:

  • Cache: skip parsing if .ddd/.mus inputs unchanged (content hash or mtime)
  • Diff: muscomp ir-diff shows domain model changes between versions
  • Inspect: muscomp ir-dump or --text-format for human review
  • CI artifact: model.ir can be stored as a build artifact for reproducibility

CMake Integration

muscomp_generate(
    PROJECT ${CMAKE_SOURCE_DIR}/model/project.mus
    TARGETS postgres cpp-dba grpc qt-widgets cmake
    OUTPUT_DIR ${CMAKE_BINARY_DIR}/generated
    IR_FILE ${CMAKE_BINARY_DIR}/model.ir
)

Creates add_custom_command rules with proper dependency tracking: .ddd/.mus → model.ir → generated files.

Positive Consequences

  • Third-party generators in any language with zero coupling to muscomp
  • Built-in and third-party generators follow identical integration path
  • Parallel generation — spawn all targets simultaneously
  • IR schema versioned independently, stable contract for ecosystem
  • Self-hosting exercises and improves the DSL
  • Improved compile times via independent build targets
  • model.ir is inspectable, diffable, cacheable

Negative Consequences

  • Serialization overhead for the two-stage path (~5ms for protobuf, negligible)
  • Process spawn overhead for parallel generators (~10ms each, negligible)
  • Bootstrap complexity for self-hosting (well-understood, protoc precedent)
  • Iteration time for generator development includes a compile step (mitigated by per-target builds with mold+ccache)
  • Proto schema must be maintained alongside DSL evolution

Pros and Cons of the Options

Option 1: dlopen Plugin Architecture

  • Good, because zero serialization overhead (in-process)
  • Good, because familiar pattern (plugin frameworks)
  • Bad, because C++-only (or requires C FFI wrappers)
  • Bad, because ABI stability burden across versions
  • Bad, because requires plugin discovery mechanism, factory functions, API versioning
  • Bad, because two different integration patterns (built-in vs third-party)

Partially adopted as InProcessDynamic transport for performance-sensitive cases.

Option 2: Protobuf IR File Contract (Chosen)

  • Good, because any language with protobuf bindings can write a generator
  • Good, because zero coupling — a generator is a program that reads a file
  • Good, because protobuf handles forward/backward compatibility natively
  • Good, because no plugin framework, SDK, or discovery mechanism needed
  • Good, because the IR file is inspectable, diffable, cacheable, and archivable
  • Good, because parallel generation via independent processes
  • Bad, because serialization/deserialization on the two-stage path (~5ms, negligible)
  • Bad, because generator development requires a compile step (mitigated by targeted builds)

Option 3: Status Quo (Monolith)

  • Good, because no migration effort
  • Bad, because third-party generators require forking muscomp
  • Bad, because all generators must be C++
  • Bad, because no stable IR contract
  • Bad, because compilation is all-or-nothing (195 files)
  • Bad, because doesn't fulfill ADR 20260326's vision for new language targets

AI Agent Guidance

When This Decision Applies

  • When designing or implementing any muscomp architectural change
  • When adding a new generator target to muscomp
  • When modifying the IR data structures
  • When planning the transition from v1.x monolith to v2.0 pipeline
  • When a third party asks how to write a Muscovite generator

When This Decision Does NOT Apply

  • muscomp v1.78.x work (completing golden diff, parser coverage) — finish quality gates before architectural migration
  • Layer 1 (C++ framework) or Layer 2 (generated application) changes — this ADR concerns Layer 0 only
  • Racket dslgen maintenance during transition — Racket remains reference until muscomp produces byte-identical output

Enforcement

  • The IR .proto schema is the source of truth for the parser-generator contract
  • CI verifies that checked-in proto files match what muscovite-ir.ddd generates (once self-hosting is active)
  • muscomp generate validates IR version compatibility before dispatching to generators
  • The GENERATE {} block in .mus files accepts arbitrary identifiers (no closed keyword set in grammar)

Key Files

  • proto/muscovite/ir/ — IR protobuf schema (bootstrap artifacts)
  • toolchain/muscomp/src/ir/types.hpp — current in-memory IR (to be replaced)
  • toolchain/muscomp/src/main.cpp — CLI entry point (to be split into subcommands)
  • toolchain/muscomp/src/generators/generator_registry.cpp — current dispatch hub (to be replaced by executable spawning)
  • dslgen/model/ddd-types.rkt — Racket IR reference (98 struct types to map to proto)
