Concurrency Control: Optimistic Locking + Advisory Locks

Concurrency Control: Optimistic Locking + Advisory Locks

  • Status: accepted
  • Date: 2026-02-15
  • Tags: concurrency, locking, optimistic, advisory, database

Technical Story: FTR-062

Context and Problem Statement

Muscovite generates CRUD operations through CommandService/CommandRepository but has no concurrency control. Concurrent updates to the same aggregate can cause lost-update anomalies. Multi-step workflows (e.g., saga compensation) need mutual exclusion to prevent conflicting operations.

How should Muscovite provide concurrency control that is:

  1. Declarative (DSL-driven, not hand-coded)
  2. Generated across all layers (SQL, C++, gRPC, Qt)
  3. Composable (optimistic + pessimistic can coexist)
  4. Inheritable (context-level defaults, entity-level overrides)

Decision Drivers

  • Lost-update prevention for concurrent entity modifications
  • Long-running workflow exclusion (saga compensation, batch operations)
  • PostgreSQL-native mechanisms (no external lock managers)
  • Generated code must handle conflicts gracefully in all frontends
  • Minimal schema overhead (no lock tables, no extra infrastructure)

Considered Options

  1. Optimistic locking only — version column + conditional UPDATE
  2. Pessimistic locking only — SELECT FOR UPDATE or advisory locks
  3. Dual strategy: optimistic + advisory — version column for concurrent reads, advisory locks for workflow exclusion
  4. Application-level locking — in-memory lock manager in C++ service layer

Decision Outcome

Chosen option: 3. Dual strategy: optimistic + advisory, because:

  • Optimistic locking handles the common case (concurrent UI edits) with zero contention overhead
  • Advisory locks handle the rare case (saga compensation, batch imports) with explicit exclusion
  • Both are PostgreSQL-native, requiring no external infrastructure
  • The two strategies are orthogonal and composable
  • Version columns propagate naturally through gRPC and Qt layers

Design

DSL Syntax

Three-level declaration with inheritance:

-- Context-level default (applies to all entities unless overridden)
CONTEXT Patient @uuid "..." {
    CONCURRENCY {
        OPTIMISTIC version;
        LOCK_TIMEOUT "5s";
    }
    
    -- Entity-level override
    ENTITY Patient @uuid "..." IDENTIFIED_BY id (...)
    CONCURRENCY {
        OPTIMISTIC version;
        ADVISORY;
        LOCK_TIMEOUT "10s";
    }
}

Inheritance Resolution

  1. Entity-level CONCURRENCY block → used as-is
  2. Aggregate-level CONCURRENCY block → applied to root and children
  3. Context-level CONCURRENCY block → default for all entities in context
  4. No CONCURRENCY block at any level → no concurrency control (current behavior)

Entity-level overrides completely replace inherited config (no merging).

Optimistic Locking

PostgreSQL: Auto-generated version column (INTEGER NOT NULL DEFAULT 1).

ALTER TABLE patient.patient ADD COLUMN version INTEGER NOT NULL DEFAULT 1;

C++ CommandRepository::update():

auto result = txn.exec_params(
    "UPDATE patient.patient SET ..., version = version + 1 "
    "WHERE id = $1 AND version = $2 "
    "RETURNING version",
    id, expected_version);
if (result.empty()) {
    throw muscovite::dba::OptimisticLockException(id, expected_version);
}

gRPC: Version field added to entity messages and update requests.

Qt: Conflict dialog on OptimisticLockException — reload + re-edit.

Advisory Locks

PostgreSQL: Uses pg_advisory_xact_lock(hash) within transactions.

Hash function: hashtext(schema_name || '.' || table_name || '.' || id::text) cast to bigint.

C++ CommandService:

void acquire_advisory_lock(pqxx::work& txn, const boost::uuids::uuid& id) {
    txn.exec_params(
        "SELECT pg_advisory_xact_lock(hashtext($1))",
        schema_ + "." + table_ + "." + boost::uuids::to_string(id));
}

Advisory locks are transaction-scoped and automatically released on commit/rollback.

Lock Timeout

Configurable per entity/context. Generates SET LOCAL lock_timeout = '...' at transaction start.

Framework Exceptions

Exception Layer Trigger
OptimisticLockException framework/dba Version mismatch on UPDATE
AdvisoryLockTimeoutException framework/dba pg_advisory_xact_lock timeout

Both inherit from muscovite::dba::ConcurrencyException.

Consequences

Positive

  • Zero overhead for entities without CONCURRENCY (opt-in)
  • Version column costs 4 bytes per row (INTEGER)
  • Advisory locks use no disk I/O (in-memory hash table in PostgreSQL)
  • Conflict detection is atomic (single UPDATE ... WHERE version = $n)
  • Lock timeout prevents deadlock hangs

Negative

  • Version column adds one field to every gRPC message for optimistic entities
  • Advisory lock hash collisions possible (extremely unlikely with hashtext)
  • Optimistic lock failures require retry logic in frontends

Neutral

  • Generated code complexity increases moderately
  • Lock timeout values need tuning per deployment
