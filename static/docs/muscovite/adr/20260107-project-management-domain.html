Project Management Domain with Kanban Workflow

Project Management Domain with Kanban Workflow

  • Status: proposed
  • Date: 2026-01-07
  • Deciders: Development Team
  • Tags: domain-modeling, workflow, project-management

Context and Problem Statement

The geosciences domain model references "projects" in multiple contexts (field mapping, campaigns, drilling, etc.), but lacks a comprehensive project management domain to track:

  • Project hierarchy (projects, subprojects)
  • Goals, milestones, deliverables
  • Task management with workflow states
  • Team structure (persons, roles, workgroups)
  • Resource allocation (instruments, vehicles)
  • Time tracking and effort estimation
  • Work-in-progress (WIP) limits per person per status

Current Gap: No systematic way to:

  1. Plan and track multi-phase geoscience projects
  2. Assign work to team members with capacity constraints
  3. Track progress against goals and milestones
  4. Manage shared resources (field vehicles, instruments)
  5. Enforce WIP limits to prevent overload

Decision Drivers

  • Workflow Flexibility: Support various project methodologies (field campaigns, lab work, analysis phases)
  • WIP Limits: Prevent team overload via configurable work-in-progress constraints
  • Resource Management: Track expensive field equipment and vehicles
  • Time Tracking: Accurate effort estimation and actual hours for project accounting
  • Hierarchy: Support complex project structures (multi-year projects with phases/subprojects)
  • Status Configurability: Organizations define their own workflow states
  • Integration: Link tasks to domain entities (samples, analyses, maps, etc.)

Considered Options

Option 1: Simple Task List (Rejected)

  • ❌ No hierarchy or milestones
  • ❌ No resource management
  • ❌ No WIP enforcement
  • ✅ Easy to implement

Option 2: Fixed Workflow States (Rejected)

  • ❌ Not adaptable to different organizations
  • ❌ Hard-coded states don't fit all workflows
  • ✅ Predictable behavior

Option 3: Kanban-Based with Configurable States (Selected)

  • ✅ Flexible workflow states (lookup table)
  • ✅ WIP limits per person per status
  • ✅ Visual workflow management
  • ✅ Time tracking integration
  • ✅ Resource allocation
  • ✅ Project hierarchy support

Decision Outcome

Implement a Kanban-inspired project management domain with the following components:

Core Entities

CONTEXT ProjectManagement {
    
    -- =====================================================================
    -- Project Hierarchy
    -- =====================================================================
    
    ENTITY Project IDENTIFIED_BY id (
        id              UUID        NOT NULL,
        parent_id       UUID,           -- for subprojects
        code            TEXT        NOT NULL @unique,
        name            TEXT        NOT NULL,
        description     TEXT,
        status          TEXT        NOT NULL,  -- active, on_hold, completed, cancelled
        start_date      DATE,
        target_end_date DATE,
        actual_end_date DATE,
        budget_hours    DECIMAL,
        manager_id      UUID        NOT NULL  -- references Person
    )
    CHECK status_valid: status IN ('active', 'on_hold', 'completed', 'cancelled')
    DOC "Scientific project or phase (field campaign, lab analysis, report writing)";
    
    ENTITY Goal IDENTIFIED_BY id (
        id          UUID    NOT NULL,
        project_id  UUID    NOT NULL,
        title       TEXT    NOT NULL,
        description TEXT,
        priority    INTEGER NOT NULL DEFAULT 3,  -- 1=critical, 5=nice-to-have
        status      TEXT    NOT NULL             -- not_started, in_progress, achieved, deferred
    )
    DOC "High-level project objective (e.g., 'Complete stratigraphic correlation')";
    
    ENTITY Milestone IDENTIFIED_BY id (
        id           UUID    NOT NULL,
        project_id   UUID    NOT NULL,
        goal_id      UUID,               -- optional link to goal
        title        TEXT    NOT NULL,
        description  TEXT,
        target_date  DATE    NOT NULL,
        actual_date  DATE,
        status       TEXT    NOT NULL    -- pending, at_risk, achieved, missed
    )
    DOC "Measurable checkpoint (e.g., 'Field samples collected by June 30')";
    
    -- =====================================================================
    -- Task Management (Kanban Core)
    -- =====================================================================
    
    ENTITY TaskStatus IDENTIFIED_BY id (
        id               UUID    NOT NULL,
        code             TEXT    NOT NULL @unique,
        name             TEXT    NOT NULL,
        description      TEXT,
        category         TEXT    NOT NULL,  -- backlog, active, done, abandoned
        display_order    INTEGER NOT NULL,
        color_hex        TEXT,               -- UI hint: #FF5733
        wip_limit_global INTEGER             -- organization-wide WIP limit for this status
    )
    CHECK category_valid: category IN ('backlog', 'active', 'done', 'abandoned')
    DOC "Configurable workflow state (idea, planned, assigned, in_progress, qa, accepted, etc.)";
    
    -- Seed with defaults:
    IMPORT CSV "../../seeds/project-management/task_status.csv" INTO TaskStatus;
    
    ENTITY Task IDENTIFIED_BY id (
        id                  UUID    NOT NULL,
        project_id          UUID    NOT NULL,
        milestone_id        UUID,
        parent_task_id      UUID,           -- for task breakdown
        title               TEXT    NOT NULL,
        description         TEXT,
        status_id           UUID    NOT NULL,  -- references TaskStatus
        assigned_to_id      UUID,              -- references Person (nullable if unassigned)
        estimated_hours     DECIMAL,
        actual_hours        DECIMAL DEFAULT 0,
        priority            INTEGER NOT NULL DEFAULT 3,
        created_at          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
        started_at          TIMESTAMPTZ,
        completed_at        TIMESTAMPTZ,
        blocked             BOOLEAN NOT NULL DEFAULT FALSE,
        blocked_reason      TEXT
    )
    DOC "Work item with effort estimate and time tracking";
    
    ENTITY TaskStatusTransition IDENTIFIED_BY id (
        id             UUID        NOT NULL,
        task_id        UUID        NOT NULL,
        from_status_id UUID,              -- NULL if initial creation
        to_status_id   UUID        NOT NULL,
        transitioned_by UUID       NOT NULL,  -- Person
        transitioned_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
        comment        TEXT
    )
    DOC "Audit trail for task state changes";
    
    -- =====================================================================
    -- Team & Resources
    -- =====================================================================
    
    ENTITY Person IDENTIFIED_BY id (
        id              UUID    NOT NULL,
        employee_code   TEXT    NOT NULL @unique,
        first_name      TEXT    NOT NULL,
        last_name       TEXT    NOT NULL,
        email           TEXT    NOT NULL @unique,
        position        TEXT,               -- Geologist, Lab Technician, Project Manager
        capacity_hours  DECIMAL,            -- weekly capacity (e.g., 40.0)
        active          BOOLEAN NOT NULL DEFAULT TRUE
    )
    DOC "Team member (employee or contractor)";
    
    ENTITY Role IDENTIFIED_BY id (
        id          UUID    NOT NULL,
        code        TEXT    NOT NULL @unique,
        name        TEXT    NOT NULL,
        description TEXT
    )
    DOC "Functional role (Field Geologist, GIS Analyst, Driller, Lab Manager)";
    
    ENTITY PersonRole IDENTIFIED_BY id (
        id          UUID    NOT NULL,
        person_id   UUID    NOT NULL,
        role_id     UUID    NOT NULL,
        project_id  UUID                -- role assignment scoped to project (optional)
    )
    DOC "Person's role(s) in projects";
    
    ENTITY Workgroup IDENTIFIED_BY id (
        id          UUID    NOT NULL,
        code        TEXT    NOT NULL @unique,
        name        TEXT    NOT NULL,
        description TEXT,
        leader_id   UUID                -- references Person
    )
    DOC "Team unit (Field Team Alpha, Geochemistry Lab, GIS Group)";
    
    ENTITY WorkgroupMembership IDENTIFIED_BY id (
        id              UUID    NOT NULL,
        workgroup_id    UUID    NOT NULL,
        person_id       UUID    NOT NULL,
        joined_date     DATE    NOT NULL,
        left_date       DATE
    )
    DOC "Person's membership in workgroup";
    
    -- =====================================================================
    -- Resource Management
    -- =====================================================================
    
    ENTITY Instrument IDENTIFIED_BY id (
        id              UUID    NOT NULL,
        code            TEXT    NOT NULL @unique,
        name            TEXT    NOT NULL,
        description     TEXT,
        instrument_type TEXT    NOT NULL,  -- XRF, XRD, GPS, Core Drill, etc.
        serial_number   TEXT,
        owner_workgroup UUID,              -- references Workgroup
        status          TEXT    NOT NULL   -- available, in_use, maintenance, retired
    )
    CHECK status_valid: status IN ('available', 'in_use', 'maintenance', 'retired')
    DOC "Field/lab equipment";
    
    ENTITY Vehicle IDENTIFIED_BY id (
        id              UUID    NOT NULL,
        code            TEXT    NOT NULL @unique,
        name            TEXT    NOT NULL,
        vehicle_type    TEXT    NOT NULL,  -- truck, helicopter, ship, atv, drone
        license_plate   TEXT,
        capacity_persons INTEGER,
        status          TEXT    NOT NULL   -- available, in_use, maintenance, retired
    )
    CHECK status_valid: status IN ('available', 'in_use', 'maintenance', 'retired')
    DOC "Field vehicle (4x4, boat, helicopter, ATV)";
    
    ENTITY ResourceAllocation IDENTIFIED_BY id (
        id              UUID        NOT NULL,
        task_id         UUID        NOT NULL,
        resource_type   TEXT        NOT NULL,  -- instrument, vehicle
        resource_id     UUID        NOT NULL,  -- polymorphic reference
        allocated_from  TIMESTAMPTZ NOT NULL,
        allocated_to    TIMESTAMPTZ NOT NULL,
        allocated_by    UUID        NOT NULL   -- Person
    )
    CHECK resource_type_valid: resource_type IN ('instrument', 'vehicle')
    DOC "Time-based resource booking for tasks";
    
    -- =====================================================================
    -- Time Tracking
    -- =====================================================================
    
    ENTITY TimeEntry IDENTIFIED_BY id (
        id          UUID        NOT NULL,
        task_id     UUID        NOT NULL,
        person_id   UUID        NOT NULL,
        start_time  TIMESTAMPTZ NOT NULL,
        end_time    TIMESTAMPTZ,
        hours       DECIMAL,                -- computed or manual entry
        notes       TEXT
    )
    DOC "Actual time spent on tasks (for effort tracking and billing)";
    
    -- =====================================================================
    -- WIP Limit Enforcement
    -- =====================================================================
    
    ENTITY PersonWipLimit IDENTIFIED_BY id (
        id          UUID    NOT NULL,
        person_id   UUID    NOT NULL,
        status_id   UUID    NOT NULL,  -- TaskStatus
        max_hours   DECIMAL NOT NULL,  -- e.g., 20 hours max in 'in_progress' at once
        max_tasks   INTEGER             -- optional: max number of tasks (regardless of hours)
    )
    DOC "Per-person WIP limits by status (prevents overload)";
    
    -- =====================================================================
    -- Aggregates
    -- =====================================================================
    
    AGGREGATE Project ROOT Project (
        CONTAINS Goal, Milestone, Task
    );
    
    AGGREGATE Team ROOT Person (
        CONTAINS PersonRole, WorkgroupMembership
    );
    
    AGGREGATE Resources ROOT Instrument (
        CONTAINS ResourceAllocation
    );
    
    AGGREGATE VehicleFleet ROOT Vehicle (
        CONTAINS ResourceAllocation
    );
    
    -- =====================================================================
    -- Domain Events
    -- =====================================================================
    
    EVENT TaskCreated (
        id          UUID        NOT NULL,
        task_id     UUID        NOT NULL,
        project_id  UUID        NOT NULL,
        created_by  UUID        NOT NULL,
        created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
    );
    
    EVENT TaskAssigned (
        id              UUID        NOT NULL,
        task_id         UUID        NOT NULL,
        assigned_to_id  UUID        NOT NULL,
        assigned_by     UUID        NOT NULL,
        assigned_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
    );
    
    EVENT TaskStatusChanged (
        id             UUID        NOT NULL,
        task_id        UUID        NOT NULL,
        from_status_id UUID,
        to_status_id   UUID        NOT NULL,
        changed_by     UUID        NOT NULL,
        changed_at     TIMESTAMPTZ NOT NULL DEFAULT NOW()
    );
    
    EVENT TaskCompleted (
        id              UUID        NOT NULL,
        task_id         UUID        NOT NULL,
        completed_by    UUID        NOT NULL,
        completed_at    TIMESTAMPTZ NOT NULL DEFAULT NOW(),
        actual_hours    DECIMAL
    );
    
    EVENT WipLimitExceeded (
        id          UUID        NOT NULL,
        person_id   UUID        NOT NULL,
        status_id   UUID        NOT NULL,
        current_wip DECIMAL     NOT NULL,
        limit_wip   DECIMAL     NOT NULL,
        detected_at TIMESTAMPTZ NOT NULL DEFAULT NOW()
    );
    
    -- =====================================================================
    -- Business Rules (Specifications)
    -- =====================================================================
    
    SPECIFICATION TasksInProgress FOR Task
        WHERE status_id IN (
            SELECT id FROM TaskStatus WHERE category = 'active'
        )
        DOC "All tasks currently being worked on";
    
    SPECIFICATION OverdueTask FOR Task
        WHERE completed_at IS NULL 
          AND milestone_id IN (
              SELECT id FROM Milestone 
              WHERE target_date < CURRENT_DATE 
                AND status != 'achieved'
          )
        DOC "Tasks linked to overdue milestones";
    
    SPECIFICATION PersonCurrentWip FOR Task (person_id UUID, status_id UUID)
        WHERE assigned_to_id = person_id 
          AND status_id = status_id
        DOC "Current WIP for a person in a specific status";
    
    -- =====================================================================
    -- Policies (Workflow Enforcement)
    -- =====================================================================
    
    POLICY EnforceWipLimit
        WHEN TaskStatusChanged
        WHERE EXISTS (
            SELECT 1 FROM PersonWipLimit pwl
            JOIN Task t ON t.assigned_to_id = pwl.person_id
            WHERE t.status_id = pwl.status_id
              AND t.assigned_to_id = $task.assigned_to_id
            GROUP BY pwl.person_id, pwl.status_id, pwl.max_hours
            HAVING SUM(t.estimated_hours) > pwl.max_hours
        )
        THEN EMIT WipLimitExceeded(
            person_id = $task.assigned_to_id,
            status_id = $to_status_id
        )
        DOC "Emit event when person exceeds WIP limit for a status";
    
    POLICY NotifyMilestoneRisk
        WHEN TaskCompleted
        WHERE $task.milestone_id IS NOT NULL
          AND EXISTS (
              SELECT 1 FROM Milestone m
              WHERE m.id = $task.milestone_id
                AND m.target_date < CURRENT_DATE + INTERVAL '7 days'
                AND m.status = 'pending'
          )
        THEN NOTIFY ProjectManager(
            milestone_id = $task.milestone_id,
            days_remaining = (SELECT target_date - CURRENT_DATE FROM Milestone WHERE id = $task.milestone_id)
        )
        DOC "Alert manager when milestone is within 7 days and still pending";
    
    -- =====================================================================
    -- Repository
    -- =====================================================================
    
    REPOSITORY TaskRepository FOR Task (
        find_by_assignee(person_id UUID) -> Task[],
        find_by_status(status_id UUID) -> Task[],
        find_by_project(project_id UUID) -> Task[],
        find_overdue() -> Task[],
        find_blocked() -> Task[],
        calculate_wip_for_person(person_id UUID, status_id UUID) -> DECIMAL
    );
    
    REPOSITORY ProjectRepository FOR Project (
        find_active() -> Project[],
        find_by_manager(manager_id UUID) -> Project[],
        find_with_overdue_milestones() -> Project[]
    );
}

Kanban Workflow Model

Status Lifecycle:

Backlog:  idea → to_plan → planned
Active:   assigned → in_progress → in_qa → in_acceptance
Done:     accepted
Abandoned: rework_required → abandoned → aborted → obsolete

WIP Limit Enforcement:

  • Each person has per-status WIP limits (hours or task count)
  • Policy emits WipLimitExceeded event when limit breached
  • UI displays visual warnings (Kanban board shows red/amber capacity indicators)
  • Prevents task status change if WIP limit exceeded (optional strict mode)

Time Tracking:

  • TimeEntry records actual work sessions (start/stop timer or manual entry)
  • Task.actual_hours aggregated from time entries
  • Variance reports: estimated vs. actual (for future estimation improvement)

Resource Allocation:

  • Instruments and vehicles booked for task duration
  • Prevents double-booking via constraint checks
  • UI shows resource calendar (Gantt-style availability)

Consequences

Positive

✅ Flexible Workflow: Organizations configure states to match their processes
✅ Capacity Management: WIP limits prevent team overload
✅ Resource Visibility: Real-time instrument/vehicle availability
✅ Accurate Tracking: Time entry audit trail for accounting and estimation
✅ Hierarchy Support: Multi-level projects and task breakdown
✅ Event-Driven: Policies enable workflow automation and notifications
✅ Domain Integration: Tasks link to samples, analyses, maps, etc.

Negative

⚠️ Complexity: More entities than simple task list (15+ entities)
⚠️ Configuration Overhead: Organizations must define TaskStatus lookup data
⚠️ Enforcement Burden: WIP limits require discipline to maintain
⚠️ UI Complexity: Kanban board requires sophisticated drag-and-drop UI

Risks

🔥 Over-Engineering: Risk of unused features (start minimal, expand based on feedback)
🔥 Performance: Large projects (1000+ tasks) may need pagination/indexing
🔥 Polymorphic Resource References: resource_id in ResourceAllocation requires careful handling

Mitigation

  • Phased Implementation: Start with Project → Task → Person, add WIP limits later
  • Performance: Index on (project_id, status_id, assigned_to_id), use materialized views for dashboards
  • Resource Polymorphism: Consider separate tables (InstrumentAllocation, VehicleAllocation) instead of single polymorphic table

Implementation Notes

Polymorphic Resource Handling:

-- Option A: Discriminated union (type column)
CREATE TABLE resource_allocation (
    resource_type TEXT NOT NULL,  -- 'instrument' | 'vehicle'
    resource_id UUID NOT NULL,
    CONSTRAINT resource_type_check CHECK (
        (resource_type = 'instrument' AND EXISTS (SELECT 1 FROM instrument WHERE id = resource_id))
        OR (resource_type = 'vehicle' AND EXISTS (SELECT 1 FROM vehicle WHERE id = resource_id))
    )
);

-- Option B: Separate tables (cleaner, type-safe)
CREATE TABLE instrument_allocation (...);
CREATE TABLE vehicle_allocation (...);

Recommend Option B for generated code (better type safety, simpler queries).

WIP Limit Query (used by UI):

-- Current WIP for person in status
SELECT 
    p.id AS person_id,
    ts.id AS status_id,
    ts.name AS status_name,
    COUNT(t.id) AS task_count,
    COALESCE(SUM(t.estimated_hours), 0) AS total_hours,
    pwl.max_hours AS limit_hours,
    pwl.max_tasks AS limit_tasks,
    CASE 
        WHEN COALESCE(SUM(t.estimated_hours), 0) > pwl.max_hours THEN 'exceeded'
        WHEN COALESCE(SUM(t.estimated_hours), 0) > pwl.max_hours * 0.8 THEN 'warning'
        ELSE 'ok'
    END AS wip_status
FROM person p
JOIN task t ON t.assigned_to_id = p.id
JOIN task_status ts ON t.status_id = ts.id
LEFT JOIN person_wip_limit pwl ON pwl.person_id = p.id AND pwl.status_id = ts.id
WHERE ts.category = 'active'
GROUP BY p.id, ts.id, ts.name, pwl.max_hours, pwl.max_tasks;
