ADR: Agent Role Governance and Separation of Duties

ADR: Agent Role Governance and Separation of Duties

  • Date: 2026-03-12
  • Status: Accepted
  • Supersedes: Partially supersedes role definitions in ADR 20260306, 20260309, 20260311
  • Related: ADR 20260312-push-based-coordinator-with-durable-nats

Context

During the v1.63.0 iteration, multiple governance failures occurred because agent roles lacked clear privilege boundaries and code-level enforcement:

  1. State walk incident: A coordinator "state walk" marked 13 unimplemented FTRs as "merged" because no role check prevented the operation.
  2. Self-merge: The leader session merged its own governance fixes directly to develop — acting as both author and reviewer.
  3. Role confusion: Worker-2 fell back to file-based coordination when the coordinator returned wait, bypassing the assignment protocol entirely.
  4. Missing QA gate: No independent quality assessment exists — the same agent that assigns work also approves and merges it.

The root cause is a conflation of responsibilities: leader, reviewer, and quality gate were all the same agent session. Role definitions were scattered across 4 documents (AGENTS.md, COORDINATOR.md, copilot-instructions.md, log.sh) with no single source of truth and no code-level enforcement of role boundaries.

Decision

1. Five canonical roles with strict privilege separation

All agent roles are defined in a single authoritative document: doc/AGENTROLES.md. Other documents (AGENTS.md, copilot-instructions.md, COORDINATOR.md) reference it but do not redefine roles.

Role Scope NATS Integration Count
Planner Future iterations None 1 (human)
Leader Current iteration Yes — assignment commands 1
QA Current iteration Yes — merge/release approval 1
Worker Current iteration Yes — implementation lifecycle 1–N
Observer Any Yes — read-only subscription 0–N

2. Separation of duties (the core constraint)

No single role may both produce and approve work:

  • Leader assigns FTRs and manages iteration state. Leader cannot merge PRs or approve releases.
  • QA reviews PRs, validates quality gates, and signs off on releases. QA cannot assign FTRs or amend coordinator state.
  • Worker implements features. Worker cannot merge, review, assign, or release.
  • Planner produces iteration specs (ADRs, FTRs, ITERATION.md, DEPENDENCIES.toml). Planner operates outside the running iteration — no coordinator interaction.

3. Release requires dual sign-off

release.sh gains a mandatory QA sign-off gate:

  1. Leader triggers release: coordinator.py leader release
  2. Automated gates run (parser, generators, C++ build, ctest, clang-tidy, TLA+, DB deploy)
  3. QA reviews gate results and signs off: coordinator.py qa approve-release
  4. Only after QA approval does release.sh proceed to tag, merge to main, and push

If QA is absent, release is blocked. No override. No "emergency" bypass.

4. NATS subject-level role isolation

Coordinator commands are partitioned by role prefix:

Subject Prefix Accessible By Purpose
coord.cmd.leader.* Leader only assign, amend-state, trigger-release
coord.cmd.qa.* QA only review, approve-merge, approve-release, gate-failed
coord.cmd.worker.* Workers only progress, pr-ready, ack-sync
coord.event.> All (subscribe) State change notifications

Enforcement via NATS user credentials + permissions (not just subject naming conventions).

5. Role identity in coordinator protocol

Every coordinator command includes a role field verified against the registered identity:

{
  "role": "leader",
  "worker": "leader-1",
  "cmd": "assign",
  "ftr": "FTR-357",
  "target_worker": "worker-2"
}

The coordinator rejects commands where role does not match the registered identity of the NATS connection. This is enforced in code, not documentation.

6. Planner operates outside the coordinator

The planner role:

  • Works on feature/*-planning branches from develop
  • Creates ADRs, FTRs, ITERATION.md, DEPENDENCIES.toml
  • Submits via PR (reviewed by QA or human)
  • Has no NATS credentials and no coordinator commands
  • Produces the artifacts that the leader loads on iteration start

This is always the human operator, potentially in a dedicated Copilot session.

Consequences

Positive

  • No self-approval: The agent that assigns work cannot merge it
  • Release safety: Dual sign-off prevents accidental or premature releases
  • Audit trail: Role identity in every command enables forensic analysis
  • Single source of truth: doc/AGENTROLES.md is the canonical role reference
  • NATS-enforced: Role boundaries are in code and transport config, not just docs

Negative

  • Higher coordination overhead: QA must be staffed for every iteration
  • Latency: Merge approval requires QA availability — no instant self-merge
  • Complexity: NATS user management adds operational burden
  • Bootstrap problem: First iteration with new model requires careful role assignment

Migration

  • v1.63.0 (current): Document roles in doc/AGENTROLES.md, update cross-references
  • v1.64.0: Implement role identity in coordinator commands (code enforcement)
  • v1.64.0: Implement NATS credential-based subject isolation
  • v1.64.0: Add QA sign-off gate to release.sh

Risks

  • QA bottleneck: If QA agent is slow or unavailable, all merges block. Mitigation: Human operator can always act as QA via CLI.
  • Role escalation: An agent could attempt to register as multiple roles. Mitigation: One NATS credential per role, coordinator rejects duplicate registrations.
