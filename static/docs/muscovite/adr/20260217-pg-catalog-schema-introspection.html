pg_catalog Schema Introspection for Brownfield Analysis

pg_catalog Schema Introspection for Brownfield Analysis

  • Status: accepted
  • Date: 2026-02-17
  • Tags: brownfield, introspection, pg_catalog, schema-analysis, legacy
  • Deciders: Johannes Lochmann

Technical Story: The first step of brownfield migration is understanding the legacy schema. Muscovite needs a tool that connects to a legacy PostgreSQL database and emits a draft .ddd file. The choice between information_schema and pg_catalog determines how much schema detail (constraints, expressions, indexes) can be extracted.

Context and Problem Statement

To generate MAPPING blocks and target .ddd entities from a legacy database, Muscovite needs to introspect the legacy schema: tables, columns, types, constraints (CHECK, FK, UNIQUE, NOT NULL), indexes, and defaults.

Two PostgreSQL introspection APIs exist:

  1. information_schema — SQL-standard views, portable across databases, but limited detail (constraint names without expressions)
  2. pg_catalog — PostgreSQL-specific system catalogs, non-portable, but provides full constraint expressions, type OIDs, index definitions, and column statistics

Muscovite already uses information_schema in Sqitch verify scripts and dslgen/verification/verify-schema.rkt for deployment verification. However, brownfield introspection has different requirements: it needs to understand constraints, not just verify their existence.

Decision Drivers

  • CHECK constraint expressions (e.g., email ~* '^[^@]+@[^@]+$') are needed to infer STRONG_TYPE candidates
  • FK column mappings (which columns participate in the FK) are needed to infer RELATIONSHIPS blocks
  • UNIQUE constraint columns are needed to infer entity identity and specification candidates
  • Default expressions are needed for MAPPING @default annotations
  • Source is always PostgreSQL in v1.24.0 — portability is not a concern for the introspection tool
  • The existing verify-schema.rkt demonstrates Racket + psycopg-style introspection patterns

Considered Options

  1. pg_catalog system catalogs — pg_class, pg_attribute, pg_type, pg_constraint, pg_index
  2. information_schema standard views — tables, columns, table_constraints, key_column_usage
  3. pg_dump --schema-only parsing — parse DDL output text
  4. pgAdmin/DBeaver export — manual export + parse

Decision Outcome

Chosen option: Option 1 — pg_catalog system catalogs, because they provide the concrete detail needed for domain model inference:

Capability pg_catalog information_schema
Column types with precision pg_attribute.atttypmod columns.numeric_precision (partial)
CHECK expression text pg_constraint.conbin → pg_get_constraintdef() Constraint name only
FK target columns pg_constraint.confkey array Via key_column_usage (indirect)
UNIQUE constraint columns pg_constraint.conkey array Via constraint_column_usage (indirect)
Index definitions pg_index + pg_get_indexdef() Not available
Column defaults pg_attrdef.adbin → pg_get_expr() columns.column_default (text)
Type OID (for custom types) pg_type.oid Not available
Table/column comments pg_description.description Not available

Key Queries

-- Tables in a schema
SELECT c.relname, d.description
FROM pg_catalog.pg_class c
JOIN pg_catalog.pg_namespace n ON n.oid = c.relnamespace
LEFT JOIN pg_catalog.pg_description d ON d.objoid = c.oid AND d.objsubid = 0
WHERE n.nspname = $1 AND c.relkind = 'r';

-- Columns with types
SELECT a.attname, format_type(a.atttypid, a.atttypmod), a.attnotnull,
       pg_get_expr(ad.adbin, ad.adrelid) AS default_expr,
       d.description
FROM pg_catalog.pg_attribute a
LEFT JOIN pg_catalog.pg_attrdef ad ON ad.adrelid = a.attrelid AND ad.adnum = a.attnum
LEFT JOIN pg_catalog.pg_description d ON d.objoid = a.attrelid AND d.objsubid = a.attnum
WHERE a.attrelid = $1::regclass AND a.attnum > 0 AND NOT a.attisdropped;

-- CHECK constraints with expressions
SELECT conname, pg_get_constraintdef(oid) AS definition
FROM pg_catalog.pg_constraint
WHERE conrelid = $1::regclass AND contype = 'c';

-- Foreign key constraints with column mappings
SELECT conname, pg_get_constraintdef(oid) AS definition,
       confrelid::regclass AS referenced_table
FROM pg_catalog.pg_constraint
WHERE conrelid = $1::regclass AND contype = 'f';

Positive Consequences

  • CHECK expressions enable automatic STRONG_TYPE inference (regex patterns → strong types)
  • FK definitions enable automatic RELATIONSHIPS block generation
  • UNIQUE constraints enable entity identity inference
  • Table/column comments provide documentation seeds for .ddd doc clauses
  • Full constraint detail reduces human review burden in the "Grasping" phase

Negative Consequences

  • PostgreSQL-specific — other source databases will need their own introspection queries
  • pg_catalog API stability: rarely changes but not SQL-standard guaranteed
  • Complex queries for multi-column constraints and indexes

AI Agent Guidance

When implementing the introspection tool:

  • Create dslgen/tools/introspect-legacy-schema.rkt — Racket script using db library for PostgreSQL connection
  • Emit draft .ddd with -- REVIEW: annotations on ambiguous mappings
  • Emit -- UNKNOWN TYPE: {pg_type} for unmappable types
  • CHECK expressions matching common patterns (email regex, URL regex, positive number) should suggest STRONG_TYPE candidates
  • FK constraints should generate RELATIONSHIPS blocks with -- INFERRED FROM FK: {constraint_name}
  • UNIQUE constraints should suggest IDENTIFIED_BY candidates
  • Output should be valid .ddd syntax that parses (with review comments)
  • Existing dslgen/verification/verify-schema.rkt provides Racket patterns for PostgreSQL introspection
