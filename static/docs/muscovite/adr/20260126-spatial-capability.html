Spatial Capability as Framework Feature

Spatial Capability as Framework Feature

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-01-26

Context and Problem Statement

Muscovite's spatial/GIS features were initially implemented as hardcoded support for the geosciences example project. This couples framework capabilities to a specific domain example, violating separation of concerns and preventing other projects from using spatial features.

Decision Drivers

  • Separation of framework from examples
  • Reusability across different domains (healthcare facility mapping, logistics, etc.)
  • Consistency with existing capability pattern (security)
  • Clean generator architecture

Decision Outcome

Spatial support is implemented as a framework capability, enabled per-project via CAPABILITIES { spatial }, with field-level @spatial-queryable annotations for query generation.

Project-Level Capability

PROJECT "FHIR Health" @uuid "..." {
    DESCRIPTION "Healthcare domain model";
    
    CAPABILITIES {
        security,    -- Enables Security context
        spatial      -- Enables PostGIS, geometry types, spatial queries
    }
    
    CONTEXTS "contexts/";
}

Field-Level Annotation

The @spatial-queryable annotation marks geometry fields for spatial query RPC generation:

ENTITY Facility IDENTIFIED_BY id @uuid "..." (
    id       UUID      NOT NULL @uuid "...",
    name     TEXT      NOT NULL @uuid "...",
    location POINT     NOT NULL @uuid "..." @spatial-queryable,
    boundary POLYGON            @uuid "..." @spatial-queryable
)
DOC "Healthcare facility with location";

What @spatial-queryable Generates

For a field marked @spatial-queryable, the generators produce:

gRPC Service RPCs:

service FacilityService {
    // Standard CRUD...
    
    // Spatial queries for @spatial-queryable fields
    rpc FindFacilitiesWithinExtent(WithinExtentRequest) returns (stream Facility);
    rpc FindFacilitiesNearPoint(NearPointRequest) returns (stream Facility);
    rpc FindFacilitiesIntersecting(IntersectingRequest) returns (stream Facility);
}

C++ Repository Methods:

class FacilityQueryRepository {
    std::vector<Facility> find_within_extent(const Polygon<4326>& extent);
    std::vector<Facility> find_near_point(const Point<4326>& point, double radius_meters);
    std::vector<Facility> find_intersecting(const Geometry<4326>& geometry);
};

Capability Detection in Generators

Generators check for spatial capability before emitting spatial-specific code:

;; In generator code
(define (has-spatial-capability? project)
  (member 'spatial (project-capabilities project)))

(define (field-is-spatial-queryable? field)
  (hash-ref (field-annotations field) 'spatial-queryable #f))

(define (is-geometry-type? type-name)
  (member type-name '(POINT LINESTRING POLYGON MULTIPOINT MULTILINESTRING 
                      MULTIPOLYGON GEOMETRYCOLLECTION POINTZ LINESTRINGZ POLYGONZ)))

Builtin Spatial Context

When spatial capability is enabled, the framework automatically loads:

CONTEXT Spatial @builtin @uuid "..." {
    ENTITY SpatialReferenceSystem IDENTIFIED_BY id @uuid "..." (
        id          UUID    NOT NULL @uuid "...",
        srid        INTEGER NOT NULL @uuid "..." @unique,
        name        TEXT    NOT NULL @uuid "...",
        auth_name   TEXT    NOT NULL @uuid "...",  -- EPSG, ESRI, etc.
        auth_srid   INTEGER NOT NULL @uuid "...",
        proj4text   TEXT             @uuid "...",
        srtext      TEXT             @uuid "...",  -- WKT definition
        is_active   BOOLEAN NOT NULL @uuid "..." DEFAULT true
    )
    DOC "Spatial reference system metadata (parallels PostGIS spatial_ref_sys)";
    
    AGGREGATE SpatialReferenceSystem ROOT SpatialReferenceSystem;
    
    IMPORT CSV "seeds/spatial/common_srs.csv" INTO SpatialReferenceSystem;
}

Consequences

Good:

  • Clean separation: framework vs examples
  • Any project can enable spatial features
  • Explicit annotation controls query generation
  • Consistent with security capability pattern

Bad:

  • Slight complexity increase in parser
  • Two-level configuration (project + field)

Migration from Hardcoded Implementation

Remove from generators:

  • (eq? (entity-name e) 'GeologicMapFeature) checks in proto.rkt
  • (string=? ctx-name "GeologicMapping") checks in cpp-service-v2.rkt

Replace with:

  • (field-is-spatial-queryable? field) for field-level decisions
  • (has-spatial-capability? project) for project-level decisions
  • (is-geometry-type? (field-type field)) for type detection

AI Agent Guidance

When this applies:

  • Adding spatial features to any project
  • Modifying generators to support spatial types
  • Creating new spatial query operations

Critical rules:

  1. Never hardcode entity/context names for spatial features
  2. Always check @spatial-queryable annotation for query generation
  3. Always check project capability before emitting PostGIS-specific code
  4. Geometry type detection is by DSL type name, not field name

Generator pattern:

;; CORRECT: Check capability and annotation
(when (and (has-spatial-capability? project)
           (field-is-spatial-queryable? field))
  (emit-spatial-query-rpc entity field))

;; WRONG: Hardcoded entity name
(when (eq? (entity-name e) 'GeologicMapFeature)
  (emit-spatial-query-rpc e))
