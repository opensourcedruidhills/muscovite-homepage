Underscore-Joined Keywords — No Whitespace in DSL Keywords

Underscore-Joined Keywords — No Whitespace in DSL Keywords

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-13
  • Tags: grammar, dsl, lexer, keywords, breaking-change

Context and Problem Statement

Muscovite's DSL has 17 multi-word keyword sequences where two or more space-separated tokens form a single syntactic construct:

Current Syntax Grammar Rule Category
VALUE_OBJECT ValueObject Construct keyword
STRONG_TYPE StrongType Construct keyword
LOOKUP_TABLE LookupTable Construct keyword
SOURCED_EVENT SourcedEvent Construct keyword
EVENT_SOURCED EventSourcedClause Modifier keyword
NOT NULL NullClause Constraint clause
IDENTIFIED_BY Entity Entity clause
BASED_ON StrongType Strong type clause
APPLIES_TO SourcedEvent Event clause
TRIGGERED_BY Saga Saga clause
CORRELATE_BY CorrelateClause Saga clause
COMPENSATES_WITH CompensatesClause Saga clause
SEED_JSON LookupTable Data clause
IMPORT_JSON Import Data clause
NATURAL_KEY NaturalKeyClause Value object clause
POSITIVE_INTEGER BaseType Type keyword
POSITIVE_INT BaseType Type keyword

This creates several concrete problems:

  1. Lexer complexity: The tokenizer must handle ambiguous lookahead — VALUE could be a standalone identifier or the start of VALUE_OBJECT. Similarly, EVENT is both a construct keyword (EVENT PatientCreated) and part of SOURCED_EVENT and EVENT_SOURCED.

  2. Construct-map derivation: The DSL reference generator (dslgen/tools/generate-dsl-reference.rkt) maintains a hardcoded 24-entry construct-map because construct keywords cannot be derived from grammar rule names — ValueObject maps to the two-token sequence VALUE_OBJECT, not a single keyword. This mapping must be manually synced.

  3. Example extraction: The generator extracts examples from .ddd files by matching construct keywords. Two-token keywords require special-case handling with fallback logic (e.g., LOOKUP_TABLE needs a regex that spans whitespace).

  4. Parser fragility: The recursive descent parser needs explicit two-token lookahead to distinguish EVENT (domain event) from SOURCED_EVENT (sourced event) and EVENT_SOURCED (aggregate modifier). This is a known source of parse ambiguity.

  5. AI generation: LLMs generating .ddd files sometimes split multi-word keywords across lines or insert extra whitespace, producing parse failures.

Meanwhile, the grammar already uses underscore-joined tokens for relationship types (CUSTOMER_SUPPLIER, SHARED_KERNEL, ANTICORRUPTION_LAYER, OPEN_HOST_SERVICE, PUBLISHED_LANGUAGE, SEPARATE_WAYS), compute timing (INSERT_OR_UPDATE), and data formats (JSON_SCHEMA, XML_SCHEMA). There is no consistent convention.

Decision Drivers

  • C/C++ macro convention: Underscore-joined identifiers are the universal convention for multi-word constants in C, C++, and preprocessor macros — the primary target languages of Muscovite's generated output
  • Lexer simplification: Single-token keywords eliminate lookahead ambiguity — every keyword is one token, unambiguously
  • Self-describing documentation: If every construct keyword is a single token, the construct-map in the DSL reference generator becomes derivable from grammar rule names rather than hardcoded
  • Grammar-first evolution (ADR 20260211): Simpler grammar rules when keywords are atomic tokens
  • Consistency: Half the multi-word concepts already use underscores; the other half use spaces — pick one convention

Considered Options

Option 1: Underscore-Joined Keywords (strict policy)

All DSL keywords become single underscore-joined tokens:

-- Before
VALUE_OBJECT Address @inline @uuid "..." ( ... )
STRONG_TYPE Email BASED_ON TEXT @uuid "..."
LOOKUP_TABLE Status @uuid "..." SEED_JSON "..."
SOURCED_EVENT PatientRegistered @uuid "..." ( ... ) APPLIES_TO Patient
ENTITY Patient @uuid "..." IDENTIFIED_BY id ( id UUID NOT NULL )
SAGA Fulfillment @uuid "..." TRIGGERED_BY OrderPlaced CORRELATE_BY order_id { ... }

-- After (17 replacements)
VALUE_OBJECT Address @inline @uuid "..." ( ... )
STRONG_TYPE Email BASED_ON TEXT @uuid "..."
LOOKUP_TABLE Status @uuid "..." SEED_JSON "..."
SOURCED_EVENT PatientRegistered @uuid "..." ( ... ) APPLIES_TO Patient
ENTITY Patient @uuid "..." IDENTIFIED_BY id ( id UUID NOT_NULL )
SAGA Fulfillment @uuid "..." TRIGGERED_BY OrderPlaced CORRELATE_BY order_id { ... }

Option 2: Keep Current Mixed Convention

Leave multi-word keywords as-is. Accept the lexer complexity and hardcoded construct-map as acceptable trade-offs.

Option 3: Merge to Single Words (no separator)

VALUEOBJECT, STRONGTYPE, LOOKUPTABLE — readable for short combinations but degrades for longer ones (COMPENSATESWITH, IDENTIFIEDBY).

Decision Outcome

Chosen option: Option 1 — Underscore-Joined Keywords, because it applies a single consistent convention that already exists in the grammar (relationship types, compute timing, data formats), follows C/C++ naming tradition, and eliminates an entire class of lexer/parser complexity.

Strict Policy

All DSL keywords MUST be single tokens. The underscore _ is the word separator for multi-word keywords. No exceptions.

Implementation Order (Grammar-First per ADR 20260211)

1. GRAMMAR    →  Replace all multi-word terminals with underscore-joined tokens
2. VERSION    →  Bump grammar to v1.2.0 (breaking change)
3. LEXER      →  Update keyword token list in lexer.rkt
4. PARSER     →  Simplify lookahead — remove two-token matching
5. MIGRATION  →  Batch-rename all .ddd files (sed/Racket script)
6. GENERATOR  →  Simplify construct-map derivation, example extraction
7. TESTS      →  Update grammar conformance, parser tests, generator tests
8. DOCS       →  Regenerate DSL.md (self-updating — picks up automatically)

Migration Tooling

A batch migration script MUST be provided to transform existing .ddd files:

# Pseudocode — exact tool TBD (sed, Racket, or Python)
for file in $(find examples/ -name "*.ddd"); do
    sed -i \
        -e 's/VALUE_OBJECT/VALUE_OBJECT/g' \
        -e 's/STRONG_TYPE/STRONG_TYPE/g' \
        -e 's/LOOKUP_TABLE/LOOKUP_TABLE/g' \
        -e 's/SOURCED_EVENT/SOURCED_EVENT/g' \
        -e 's/EVENT_SOURCED/EVENT_SOURCED/g' \
        -e 's/NOT NULL/NOT_NULL/g' \
        -e 's/IDENTIFIED_BY/IDENTIFIED_BY/g' \
        -e 's/BASED_ON/BASED_ON/g' \
        -e 's/APPLIES_TO/APPLIES_TO/g' \
        -e 's/TRIGGERED_BY/TRIGGERED_BY/g' \
        -e 's/CORRELATE_BY/CORRELATE_BY/g' \
        -e 's/COMPENSATES_WITH/COMPENSATES_WITH/g' \
        -e 's/SEED_JSON/SEED_JSON/g' \
        -e 's/IMPORT_JSON/IMPORT_JSON/g' \
        -e 's/NATURAL_KEY/NATURAL_KEY/g' \
        -e 's/POSITIVE_INTEGER/POSITIVE_INTEGER/g' \
        -e 's/POSITIVE_INT/POSITIVE_INT/g' \
        "$file"
done

Note: Order matters — POSITIVE_INTEGER must be replaced before POSITIVE_INT to avoid partial matches. Similarly, BASED_ON must not be confused with standalone ON in other contexts. The migration script must handle these correctly.

Positive Consequences

  • Lexer simplification: Every keyword is a single token — no lookahead, no ambiguity
  • Derivable construct-map: DSL reference generator can derive keyword↔︎rule mapping from grammar rule names (e.g., ValueObject → VALUE_OBJECT) instead of hardcoding 24 entries
  • Simpler example extraction: Single-token keyword matching replaces multi-token regex with fallback logic
  • AI-friendlier: LLMs can't accidentally split keywords across lines
  • Consistent convention: All keywords follow one rule — the same one used by C/C++ macros and the existing underscore-joined tokens in the grammar
  • Parser simplification: No more EVENT vs SOURCED_EVENT vs EVENT_SOURCED disambiguation

Negative Consequences

  • Breaking change: All existing .ddd files (45 files, ~18K lines) must be migrated
  • Visual density: IDENTIFIED_BY is arguably less readable than IDENTIFIED_BY — underscore-joined keywords look more "code-like" than "prose-like"
  • Migration risk: sed-based replacement could have edge cases in string literals or comments containing these keyword sequences
  • Readability trade-off: The DSL moves slightly further from natural language toward programming language syntax

Future Evolution

Once all keywords are single tokens:

  1. Construct-map derivation: The construct-map in the DSL reference generator becomes a simple PascalCase-to-UPPER_SNAKE_CASE conversion — no hardcoded list needed
  2. Grammar-to-parser generation (ADR 20260211, Option 3): Single-token keywords are a prerequisite for machine-generated parsers from PEG grammars
  3. Language Server Protocol: Single-token keywords simplify completion, highlighting, and hover documentation in editor extensions
