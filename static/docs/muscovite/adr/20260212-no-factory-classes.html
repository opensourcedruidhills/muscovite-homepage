No Factory Classes — C++ Constructors Are Factories

No Factory Classes — C++ Constructors Are Factories

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-12
  • Tags: ddd, patterns, c++, code-generation

Context and Problem Statement

Evans' Factory pattern (Domain-Driven Design, 2003, Chapter 6) addressed a real limitation in Java and C#: constructors in those languages cannot express complex creation logic cleanly, leading to separate Factory classes that encapsulate aggregate assembly, invariant enforcement, and reconstitution from persistence.

Muscovite generates C++ code. Should the generator emit Factory classes for aggregates and entities?

Decision Drivers

  • C++ constructors are fundamentally more capable than Java/C# constructors (RAII, exceptions, move semantics)
  • Muscovite already generates Repository classes with create() methods that serve factory semantics
  • Additional Factory classes would increase generated code volume without adding capability
  • Evans himself described Factories as addressing constructor inadequacy, not as a universal requirement

Considered Options

  1. Generate Factory classes per aggregate — PatientFactory::create(...) returning fully constructed aggregate
  2. Generate Factory methods on entities — Patient::create(...) static methods
  3. No Factory classes — use constructors + Repository::create() (chosen)

Decision Outcome

Chosen option: "No Factory classes — use constructors + Repository::create()", because C++ constructors already provide factory semantics through RAII, and the generated Repository layer handles the persistence-creation boundary.

Rationale

C++ constructors are factories. Unlike Java/C#, C++ constructors can:

  • Validate all invariants and throw exceptions on invalid input (fail-fast)
  • Use member initializer lists for efficient, single-phase construction
  • Guarantee RAII — objects are valid from construction or don't exist
  • Be overloaded for multiple creation paths
  • Accept move-only types for zero-copy aggregate assembly

Two-phase initialization is an anti-pattern in C++. A Factory that returns an uninitialized object requiring subsequent init() calls violates RAII. C++ constructors eliminate this class of bugs by design.

Repository provides the persistence-creation boundary. The generated CommandRepository::insert() method:

  1. Accepts a fully-constructed entity (validated by its constructor)
  2. Persists to PostgreSQL within a transaction
  3. Returns the entity with database-assigned fields (e.g., created_at)
  4. Emits domain events via UnitOfWork

This is exactly what Evans' Factory was designed to do — but without a separate class.

Named constructors remain available. When domain-specific creation logic is needed (e.g., reconstitution from an external format), developers can add static factory methods manually:

// Hand-written in extension code, NOT generated
static Patient Patient::fromFhirBundle(const fhir::Bundle& bundle);
static Patient Patient::reconstitute(const EventStream& events);

These are application-layer concerns that the generator cannot anticipate.

Positive Consequences

  • Less generated code — no *Factory classes to maintain across all layers
  • Simpler mental model — constructor = creation, repository = persistence
  • RAII enforced — invalid objects cannot exist
  • Aligned with modern C++ idioms (C++23, move semantics, std::expected)
  • Developers retain full control via manually authored named constructors

Negative Consequences

  • Complex multi-entity aggregate assembly (e.g., Order + OrderLine[] + Address) has no single generated entry point — developers compose via CommandService
  • Reconstitution from non-standard formats (FHIR, HL7v2, legacy CSV) requires manual factory methods

AI Agent Guidance

When This Decision Applies

  • Generating C++ entity structs, repository classes, or domain services
  • Reviewing DDD pattern checklists that list "Factory" as a required pattern
  • Evaluating whether a missing *Factory class is a generation bug (it is not)

When This Decision Does Not Apply

  • Non-C++ targets where constructors lack RAII semantics (evaluate separately if a non-C++ frontend target is added)
  • Builder pattern for complex query objects (this ADR is about domain entity creation, not query construction)

Key Files Affected

  • dslgen/generators/cpp/dba/entity.rkt — generates entity structs with constructors
  • dslgen/generators/cpp/dba/unified.rkt — generates CommandRepository with insert()
  • framework/dba/include/muscovite/dba/unit_of_work.hpp — transaction + event boundary

Enforcement

  • No enforcement needed — the generator simply does not emit Factory classes
  • If a future DSL construct implies factory semantics (e.g., FACTORY keyword), reject it per this ADR
