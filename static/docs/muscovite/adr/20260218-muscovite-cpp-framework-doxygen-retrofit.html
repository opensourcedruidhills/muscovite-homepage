Muscovite-cpp Framework Doxygen Retrofit

Muscovite-cpp Framework Doxygen Retrofit

  • Status: accepted
  • Date: 2026-02-17
  • Tags: documentation, doxygen, framework, retrofit, c++
  • Deciders: Johannes Lochmann

Technical Story: The muscovite-cpp framework (Layer 1) public API lacks systematic Doxygen documentation. As AI-prompted code, it needs structured documentation with ADR cross-references for maintainability and onboarding.

Context and Problem Statement

The muscovite-cpp framework under framework/ provides the runtime infrastructure for generated applications: strong types, DBA base classes (CommandRepository, QueryRepository), NATS integration (NatsMessageBroker, NatsEventConsumer, OutboxPublisher), gRPC server utilities, Qt base classes, and spatial types. This code is AI-prompted and hand-reviewed.

Currently, public headers have minimal or no Doxygen documentation. Key design decisions (why OutboxPublisher uses advisory locks, why CommandRepository separates from QueryRepository, why NatsMessageBroker reconnects with exponential backoff) are not documented at the point of implementation.

This is distinct from ADR 20260217 (documentation as generated artifact) because framework code is hand-authored, not generated. It requires a systematic retrofit effort, not generator changes.

Decision Drivers

  • Framework code is AI-prompted — documentation must survive context switches between sessions
  • Design rationale tied to ADRs (CQRS, event sourcing, outbox pattern, optimistic locking) must be traceable at the code level
  • cmake --build --preset debug --target doxygen-docs (from ADR 20260217) will include framework headers — they should be documented
  • Onboarding new developers to the framework requires understanding the "why" behind each component

Considered Options

  1. Systematic Doxygen retrofit with ADR cross-references — file-by-file, AI-prompted with human review
  2. README-per-module approach — one README.md per framework subdirectory
  3. No action — rely on ADRs and CONTEXT.md for understanding

Decision Outcome

Chosen option: Option 1 — Systematic Doxygen retrofit, because Doxygen comments are collocated with the code they describe, visible in IDE tooltips, and automatically included in the generated Doxygen HTML output.

Scope

All public headers in:

  • framework/include/muscovite/ — top-level types
  • framework/dba/include/muscovite/dba/ — database access layer (CommandRepository, QueryRepository, NatsMessageBroker, OutboxPublisher, EventStore)
  • framework/types/include/muscovite/types/ — strong types
  • framework/grpc/include/muscovite/grpc/ — gRPC server utilities
  • framework/qt/include/muscovite/qt/ — Qt base classes
  • framework/server/include/muscovite/server/ — server configuration and lifecycle
  • framework/spatial/include/muscovite/spatial/ — spatial/GIS types
  • framework/geo/include/muscovite/geo/ — geographic types

Documentation Standard

Each public header gets:

  • @file — file purpose
  • @class / @struct — class brief + detailed description
  • @brief on all public methods
  • @param / @return on non-trivial methods
  • @note for important behavioral characteristics
  • @see ADR-NNNN cross-references to relevant architecture decisions
  • @invariant for class invariants

Positive Consequences

  • IDE tooltips show domain-meaningful documentation for framework classes
  • Doxygen HTML output covers both framework and generated code
  • ADR traceability at the code level — grep for @see ADR finds all implementation sites
  • Onboarding time reduction for contributors

Negative Consequences

  • One-time retrofit effort (~3-5 days of prompted work)
  • Maintenance burden — documentation must be updated when framework changes
  • Risk of documentation drift if not enforced in review

Mandatory Documentation Maintenance (NO EXCEPTIONS)

All changes to muscovite-cpp framework code MUST update existing Doxygen documentation in the same commit. This is not optional and not a follow-up task.

  • New public class/struct → full @class, @brief, @file documentation required
  • Changed method signature → update @brief, @param, @return
  • Changed behavior → update @note, @invariant, @see as applicable
  • New ADR implementation → add @see ADR-NNNN reference

FORBIDDEN: Submitting framework code changes that leave documentation stale. "Will document later" is not acceptable.

Same rule applies to Racket code — see ADR 20260217 and copilot-instructions.md Rule 14.

AI Agent Guidance

When This Decision Applies

  • When modifying any public header in framework/ — update or add Doxygen comments
  • When adding new framework classes — include full Doxygen documentation from the start
  • When implementing ADR decisions in framework code — add @see ADR-NNNN references

When This Decision Does NOT Apply

  • Generated code documentation (covered by ADR 20260217)
  • Internal/private implementation files (.cpp files) — document only when non-obvious
  • Racket source code comments (separate concern)

Enforcement

  • Code review: public headers must have Doxygen comments
  • cmake --build --preset debug --target doxygen-docs must produce warnings-free output
  • Periodic audit: grep for undocumented public classes

Key Files

  • framework/dba/include/muscovite/dba/ — primary retrofit target (most complex classes)
  • framework/types/include/muscovite/types/ — strong type documentation
  • framework/server/include/muscovite/server/config.hpp — configuration documentation
  • framework/grpc/include/muscovite/grpc/ — gRPC server documentation
