Generated Server Configuration via INI Files

Generated Server Configuration via INI Files

  • Status: accepted
  • Deciders: jolo, AI agent
  • Date: 2026-01-16
  • Tags: configuration, server, ini, deployment

Context and Problem Statement

Generated gRPC servers currently hardcode default database connection parameters (host=localhost, port=5432, dbname=muscovite) and rely on environment variables for configuration. This causes problems:

  1. Wrong defaults: Servers try to connect to non-existent "muscovite" database instead of the project's actual database
  2. No config file support: Despite muscovite framework having INI file support, generated servers don't use it
  3. Environment variable fragility: Must set multiple env vars correctly to run the server
  4. No project-specific defaults: Each project has different database names/ports defined in project.ddd

Decision Drivers

  • Zero-config startup: Server should work out-of-the-box with project-specific defaults
  • Deployment flexibility: Support config files, environment variables, and CLI arguments
  • Framework consistency: Use existing muscovite::server::Config infrastructure
  • Self-documenting: Generated config file shows all available options

Considered Options

  1. Continue with environment variables only - Status quo, requires correct env setup
  2. Generate INI config file - Emit config with project-specific defaults from project.ddd
  3. Generate YAML/TOML config - More modern formats but require additional dependencies

Decision Outcome

Chosen option: Generate INI config file using existing muscovite framework support.

Implementation

  1. Racket generator emits config/server.ini containing:

    • Database connection settings from project.ddd DATABASE blocks
    • Server name derived from project name
    • Default port 50051
    • Logging defaults

  2. Generated server_main.cpp uses muscovite::server::Config::load() instead of manual env parsing

  3. Server accepts --config / -c parameter to specify config file path

  4. Precedence order (highest to lowest):

    1. Command-line arguments (--port, --database-url)
    2. Environment variables (FHIRHEALTH_DATABASE_URL)
    3. Config file (config/server.ini)
    4. Default values

Generated Config File Format

; Generated by muscovite for FhirHealth project
; Edit this file or override with environment variables / CLI args

[server]
name = fhirhealth-server
port = 50051
bind_address = 0.0.0.0

[database]
; PostgreSQL connection string
connection_string = postgresql://postgres:postgres@localhost:5555/fhir_health
pool_min = 4
pool_max = 20

[logging]
level = info
console = true
; file = /var/log/fhirhealth.log

Environment Variable Prefix

Each project uses its own prefix derived from the project name:

  • FhirHealth → FHIRHEALTH_
  • Geosciences → GEOSCIENCES_

This prevents conflicts when running multiple servers.

Consequences

Positive

  • Works out-of-the-box: Server starts with correct database from config file
  • Self-documenting: Config file shows all options with comments
  • Flexible deployment: Override specific settings via env vars or CLI
  • Framework consistency: Uses existing tested Config infrastructure
  • Project isolation: Each project has its own prefix and config

Negative

  • Extra file to manage: Config file must be deployed with server
  • Path resolution: Server must find config file relative to executable

Neutral

  • Config file uses INI format (simple, well-supported, already implemented)

AI Agent Guidance

When Generating New Projects

The gRPC generator must:

  1. Emit config/server.ini with project-specific database settings
  2. Generate server_main.cpp that uses Config::load() with the project's env prefix
  3. Look for config file at ../config/server.ini relative to executable by default

Key Files Affected

File Change
src/dslgen/generators/grpc/server-main.rkt Use Config::load(), add --config handling
src/dslgen/generators/grpc/config.rkt NEW: Generate server.ini from project settings
generated-*/grpc/config/server.ini Generated config file

Testing

# Run server with default config (looks for ../config/server.ini)
./fhirhealth_server

# Run server with explicit config path
./fhirhealth_server --config /path/to/server.ini

# Override database via environment
FHIRHEALTH_DATABASE_URL="postgresql://user:pass@prod:5432/fhir" ./fhirhealth_server

# Override via CLI (highest priority)
./fhirhealth_server --database-url "postgresql://user:pass@staging:5432/fhir"
