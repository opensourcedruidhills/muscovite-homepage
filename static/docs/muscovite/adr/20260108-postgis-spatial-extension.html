Use PostGIS for Spatial Data Storage and Queries

Use PostGIS for Spatial Data Storage and Queries

Date: 2026-01-08
Status: Accepted
Deciders: System Architect, Database Team, GIS Team
Tags: postgresql, postgis, spatial, geometry, gis, fgdc

Context and Problem Statement

Geologic mapping requires storing and querying spatial data for features like fault traces, volcanic deposits, map extents, and sample locations. How should we store spatial geometries and perform spatial operations (containment, intersection, distance) in a way that supports professional GIS workflows and FGDC standard compliance?

Decision Drivers

  • FGDC Standard Compliance: USGS Federal Geographic Data Committee standard requires spatial data for all map features
  • Spatial Operations: Need to query "features within map extent", "faults intersecting polygon", "samples within 100m of outcrop"
  • Coordinate Systems: Support multiple SRIDs (spatial reference systems) with on-the-fly transformation
  • Industry Standard: Integrate with QGIS, ArcGIS, and other professional GIS tools
  • Performance: Spatial indexes for efficient large-dataset queries
  • Data Interchange: Export GeoJSON, WKT, KML for interoperability

Considered Options

Option 1: Store WKT as TEXT (REJECTED)

CREATE TABLE fault (
    trace_geometry TEXT  -- 'LINESTRING(139.7 35.7, 139.8 35.8)'
);

Pros:

  • Simple implementation
  • Human-readable
  • No extension required

Cons:

  • No spatial operations: Cannot query "faults within extent" without parsing
  • No validation: Can store invalid WKT
  • No SRID tracking: Coordinate system unknown
  • No indexes: Linear scan for all spatial queries
  • Manual parsing: Application must parse WKT for rendering

Option 2: Store Binary Geometry (BYTEA) (REJECTED)

CREATE TABLE fault (
    trace_geometry BYTEA  -- Well-Known Binary format
);

Pros:

  • Compact storage
  • Industry standard format (WKB)

Cons:

  • No spatial operations: Cannot query geometries without extension
  • No indexes: Cannot create spatial indexes
  • Manual implementation: Must implement spatial functions in application
  • No SRID: Coordinate system not stored

Option 3: PostGIS Extension (ACCEPTED)

CREATE EXTENSION IF NOT EXISTS postgis;

CREATE TABLE structural_geology.fault (
    trace_geometry GEOMETRY(LineString, 4326)  -- WGS84
);

CREATE INDEX idx_fault_trace ON structural_geology.fault 
    USING GIST(trace_geometry);

-- Spatial queries
SELECT * FROM structural_geology.fault
WHERE ST_Intersects(trace_geometry, ST_MakeEnvelope(-180, -90, 180, 90, 4326));

Pros:

  • Full OGC compliance: Implements Open Geospatial Consortium standards
  • 400+ spatial functions: ST_Intersects, ST_Contains, ST_Distance, ST_Transform, etc.
  • Spatial indexes: GIST/BRIN indexes for fast queries on millions of features
  • SRID support: Built-in coordinate system tracking and transformation
  • Industry standard: Used by QGIS, ArcGIS, GeoServer, etc.
  • Data interchange: ST_AsGeoJSON, ST_AsKML, ST_AsText for export
  • Validation: Enforces valid geometries at INSERT time
  • Topology: Advanced topological analysis (overlaps, gaps, slivers)

Cons:

  • Extension dependency (must be installed in container)
  • Slightly larger binary size (~15MB)

Decision Outcome

Chosen Option: PostGIS extension (Option 3)

PostGIS is the industry-standard solution for spatial data in PostgreSQL and provides comprehensive GIS functionality required for professional geologic mapping.

PostGIS Installation

Docker Image (docker/postgres/Dockerfile):

FROM postgres:18

RUN apt-get update && \
    apt-get install -y postgresql-18-postgis-3 && \
    rm -rf /var/lib/apt/lists/*

Database Initialization (deploy/000-init.sql):

-- Enable PostGIS extension (must be first migration)
CREATE EXTENSION IF NOT EXISTS postgis;
CREATE EXTENSION IF NOT EXISTS postgis_topology;  -- Optional

-- Verify installation
SELECT PostGIS_Version(), PostGIS_Full_Version();

DSL Type Mapping

DSL Type PostgreSQL Type SRID Description
POINT GEOMETRY(Point, 4326) 4326 Single coordinate (longitude, latitude)
LINESTRING GEOMETRY(LineString, 4326) 4326 Fault trace, stream path, cross-section line
POLYGON GEOMETRY(Polygon, 4326) 4326 Map extent, deposit outline, exclusion zone
GEOMETRY GEOMETRY(Geometry, 4326) 4326 Mixed geometry types (point, line, or polygon)
GEOGRAPHY GEOGRAPHY(Geometry, 4326) 4326 Geodetic calculations (true distance on sphere)

Default SRID: 4326 (WGS84) - Global latitude/longitude coordinate system

SRID Strategy

Per-Map SRID with Default:

ENTITY GeologicMap IDENTIFIED_BY id (
    id UUID NOT NULL,
    map_extent GEOMETRY(Polygon, 4326) NOT NULL,
    crs_epsg INTEGER NOT NULL DEFAULT 4326
)
DOC "Map extent defines viewable area in map's native CRS";

On-the-Fly Transformation (API layer):

CREATE OR REPLACE FUNCTION geologic_mapping_api.map_feature_insert(
    p_map_id UUID,
    p_feature_geometry GEOMETRY,
    ...
) RETURNS geologic_mapping.geologic_map_feature AS $$
DECLARE
    v_target_srid INTEGER;
    v_result geologic_mapping.geologic_map_feature%ROWTYPE;
BEGIN
    -- Get target map's SRID
    SELECT crs_epsg INTO v_target_srid 
    FROM geologic_mapping.geologic_map 
    WHERE id = p_map_id;
    
    -- Transform geometry to map's SRID if needed
    INSERT INTO geologic_mapping.geologic_map_feature (
        map_id,
        geometry,
        ...
    ) VALUES (
        p_map_id,
        ST_Transform(p_feature_geometry, v_target_srid),
        ...
    )
    RETURNING * INTO v_result;
    
    RETURN v_result;
END;
$$ LANGUAGE plpgsql;

Spatial Index Generation

Automatic GIST Indexes (generated by sqitch-unified.rkt):

-- For every GEOMETRY/GEOGRAPHY column
CREATE INDEX idx_<table>_<column> ON <schema>.<table> 
    USING GIST(<column>);

Example:

CREATE INDEX idx_fault_trace_geometry 
    ON structural_geology.fault 
    USING GIST(trace_geometry);

CREATE INDEX idx_map_feature_geometry 
    ON geologic_mapping.geologic_map_feature 
    USING GIST(geometry);

Geometry Validation

PostGIS enforces valid geometries:

  • Self-intersecting polygons: Rejected with error
  • Unclosed polygons: Rejected with error
  • Invalid WKT: Rejected with error
  • SRID mismatch: Function calls check SRID compatibility

Common Spatial Operations

1. Features Within Map Extent:

SELECT f.* 
FROM geologic_mapping.geologic_map_feature f
JOIN geologic_mapping.geologic_map m ON f.map_id = m.id
WHERE ST_Intersects(f.geometry, m.map_extent);

2. Distance Queries:

SELECT * FROM field_observation.field_sample
WHERE ST_DWithin(
    location::geography,
    ST_SetSRID(ST_MakePoint(139.7, 35.7), 4326)::geography,
    100  -- 100 meters
);

3. Coordinate Transformation:

-- Transform from WGS84 (4326) to UTM Zone 54N (32654)
SELECT ST_Transform(
    ST_SetSRID(ST_MakePoint(139.7, 35.7), 4326),
    32654
) AS utm_point;

4. GeoJSON Export:

SELECT ST_AsGeoJSON(geometry) AS geojson
FROM geologic_mapping.geologic_map_feature
WHERE map_id = '...';

Implementation Rules

1. DSL Syntax

Entity with Geometry:

ENTITY Fault IDENTIFIED_BY id (
    id UUID NOT NULL,
    trace_geometry LINESTRING NOT NULL
)
DOC "Fault with spatial trace geometry";

Generated SQL:

CREATE TABLE structural_geology.fault (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    trace_geometry GEOMETRY(LineString, 4326) NOT NULL
);

CREATE INDEX idx_fault_trace_geometry 
    ON structural_geology.fault 
    USING GIST(trace_geometry);

2. Geometry Type Constraints

Feature type → Geometry type mapping:

Feature Type Geometry Type Example Entity
Fault, Fold, Lineation LINESTRING structural_geology.fault.trace_geometry
Volcanic deposit, aquifer POLYGON volcanology.pyroclastic_deposit.outline_geometry
Sample location, station POINT field_observation.field_sample.location
Map extent, AOI POLYGON geologic_mapping.geologic_map.map_extent
Mixed/unknown GEOMETRY geologic_mapping.geologic_map_feature.geometry

3. API Function Patterns

Insert with Geometry (WKT input):

CREATE OR REPLACE FUNCTION structural_geology_api.fault_insert(
    p_trace_geometry TEXT,  -- WKT: 'LINESTRING(139.7 35.7, 139.8 35.8)'
    ...
) RETURNS structural_geology.fault AS $$
DECLARE
    v_result structural_geology.fault%ROWTYPE;
BEGIN
    INSERT INTO structural_geology.fault (trace_geometry, ...)
    VALUES (
        ST_GeomFromText(p_trace_geometry, 4326),
        ...
    )
    RETURNING * INTO v_result;
    
    RETURN v_result;
END;
$$ LANGUAGE plpgsql;

Select with GeoJSON Output:

CREATE OR REPLACE FUNCTION structural_geology_api.fault_select_geojson(
    p_id UUID
) RETURNS JSON AS $$
BEGIN
    RETURN (
        SELECT json_build_object(
            'type', 'Feature',
            'id', id,
            'geometry', ST_AsGeoJSON(trace_geometry)::json,
            'properties', json_build_object(
                'name', name,
                'fault_type', fault_type
            )
        )
        FROM structural_geology.fault
        WHERE id = p_id
    );
END;
$$ LANGUAGE plpgsql;

Consequences

Positive

  • Professional GIS Capability: Full spatial analysis without external tools
  • Performance: GIST indexes enable sub-second queries on millions of features
  • FGDC Compliance: Meets USGS spatial data requirements
  • Industry Integration: Direct export to QGIS, ArcGIS, GeoServer
  • Coordinate Flexibility: 6000+ SRID definitions built-in with ST_Transform
  • Data Validation: Invalid geometries rejected at database level
  • SQL Spatial Queries: Complex spatial analysis in pure SQL

Negative

  • Extension Dependency: Requires PostGIS installation in Docker container
  • Learning Curve: Team must learn PostGIS function syntax (ST_*)
  • Binary Size: PostgreSQL image grows by ~15MB
  • Version Coupling: PostGIS version must match PostgreSQL major version

Migration Impact

Existing Projects:

  • Run CREATE EXTENSION postgis; in existing databases
  • ALTER TABLE to add geometry columns
  • Migrate WKT TEXT columns: UPDATE table SET geom_col = ST_GeomFromText(wkt_col, 4326)
  • CREATE GIST indexes

New Projects:

  • 000-init.sql automatically enables PostGIS
  • All spatial types generate GEOMETRY columns
  • All GEOMETRY columns get GIST indexes

AI Agent Guidance

When This Decision Applies

  • Always: All spatial data (coordinates, traces, outlines, extents)
  • Always: Any feature that will be rendered on a map
  • Always: Any query involving "within", "near", "intersecting"

When This Decision Does NOT Apply

  • Simple X/Y coordinates for non-spatial calculations (use DECIMAL columns)
  • Address strings without coordinates (use TEXT)

How It's Enforced

  1. Docker Image: Dockerfile installs postgresql-18-postgis-3
  2. Database Init: 000-init.sql enables postgis extension
  3. DSL Generator: sqitch-unified.rkt maps LINESTRING → GEOMETRY(LineString, 4326)
  4. Index Generation: Automatic GIST index for all GEOMETRY columns
  5. API Templates: WKT input via ST_GeomFromText(), GeoJSON output via ST_AsGeoJSON()

Notes

PostGIS version 3.x is used with PostgreSQL 18. The extension is backward-compatible with existing spatial data standards (WKT, WKB, GeoJSON, KML).

For geodetic calculations (true distance on earth's surface), use GEOGRAPHY type instead of GEOMETRY:

-- Geography uses spherical calculations
SELECT ST_Distance(
    point1::geography,
    point2::geography
) AS distance_meters;

For planar calculations (projected coordinates, faster), use GEOMETRY:

-- Geometry uses Pythagorean distance
SELECT ST_Distance(point1, point2) AS distance_units;

Default to GEOMETRY(type, 4326) unless doing global-scale distance calculations, then use GEOGRAPHY.
