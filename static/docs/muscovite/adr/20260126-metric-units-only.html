Metric Units Only

Metric Units Only

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-01-26

Context and Problem Statement

Muscovite needs a consistent unit system for all measurements including distances, areas, elevations, and other physical quantities. The choice affects spatial queries, data storage, API contracts, and user interfaces.

Decision Drivers

  • Scientific rigor and international standards
  • Consistency across the entire system
  • Interoperability with PostGIS (which uses meters for GEOGRAPHY)
  • Elimination of unit conversion errors

Decision Outcome

The entire Muscovite system works exclusively with metric (SI) units. Imperial units are forbidden and not in scope.

Unit Standards

Measurement Unit Symbol
Distance meters m
Area square meters m²
Elevation/Depth meters m
Mass kilograms kg
Temperature Celsius °C
Volume liters / cubic meters L / m³
Pressure pascals Pa
Coordinates decimal degrees (WGS84) °

Consequences

Good:

  • No unit conversion bugs
  • Direct compatibility with PostGIS GEOGRAPHY (always returns meters)
  • Scientific data is already metric
  • Single code path, no unit handling complexity
  • Clear API contracts

Bad:

  • Users in US/UK must convert from imperial (their problem, not ours)
  • Display formatting may need locale-aware conversion in UI layer only

Implementation Rules

  1. Database: All numeric fields representing physical measurements store metric values
  2. gRPC API: All distance/area parameters and responses are in meters
  3. C++ Types: No imperial constants or conversion functions in framework code
  4. Qt UI: Display can format to locale preferences, but internal values remain metric
  5. Spatial Queries: near_point(radius_meters) - parameter is always meters

Examples

// Correct: meters
message NearPointRequest {
    Point location = 1;
    double radius_meters = 2;  // Always meters
}

// FORBIDDEN: imperial
message NearPointRequest {
    Point location = 1;
    double radius_feet = 2;  // NO!
}
// Correct
auto features = repo.features_near_point(location, 1000.0);  // 1km radius

// FORBIDDEN - no imperial constants
constexpr double FEET_PER_METER = 3.28084;  // NO!

AI Agent Guidance

When this applies:

  • Any code involving distances, areas, elevations, depths
  • Spatial query parameters
  • Physical measurements of any kind
  • API design for numeric quantities

Enforcement:

  • Code review should reject imperial unit references
  • Generators must not emit imperial conversion code
  • Documentation must specify metric units

Key files that must follow this:

  • src/muscframework/dba/include/muscovite/dba/geometry/ - All spatial types
  • src/dslgen/generators/grpc/proto.rkt - API messages
  • src/dslgen/generators/cpp/dba/unified.rkt - Repository queries
