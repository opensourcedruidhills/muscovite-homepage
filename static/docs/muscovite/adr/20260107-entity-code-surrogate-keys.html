Use Entity Code as User-Facing Surrogate Key

Use Entity Code as User-Facing Surrogate Key

Date: 2026-01-07
Status: Accepted
Deciders: Architecture Team
Tags: database, domain-modeling, usability

Context and Problem Statement

UUID primary keys provide excellent technical properties (global uniqueness, no collisions, decentralized generation) but are user-hostile. Users cannot remember or communicate UUIDs verbally. For professional geologists working with map symbols, mineral samples, or field localities, they need human-readable codes like CONT-001, MIN-QZ, or SMP-2024-001.

Current challenges:

  • Users must reference entities by UUID in UI/API
  • No way to quickly identify entities in logs or conversations
  • Import/export requires UUID management
  • Natural keys (like FGDC symbol codes) aren't enforced consistently

Decision Drivers

  • Usability: Geologists need memorable codes for field work
  • Auditability: Logs and reports should show readable identifiers
  • Import/Export: CSV imports should use codes, not UUIDs
  • Flexibility: Users should be able to define their own codes
  • Automation: System should auto-generate codes when users don't provide them
  • Uniqueness: Codes must be unique per entity type

Considered Options

Option 1: Composite Natural Keys

Use domain-specific natural keys (e.g., fgdc_code + section for MapSymbol).

Pros:

  • No surrogate needed
  • Aligns with domain concepts

Cons:

  • Different per entity (no consistency)
  • Some entities lack natural keys
  • Composite keys complicate FKs
  • Hard to predict format

Option 2: User-Managed Codes Only

Require users to always provide a code.

Pros:

  • Simple implementation
  • Users control naming

Cons:

  • Breaks bulk imports (must generate 1000+ codes manually)
  • Creates friction in UI forms
  • Inconsistent formatting across users

Option 3: Auto-Generated Codes with Prefix System (CHOSEN)

Central prefix table + sequences + auto-generation on insert.

Pros:

  • Best of both worlds: manual OR automatic
  • Consistent formatting across entities
  • Easy bulk imports (codes auto-assigned)
  • Prefix identifies entity type (MIN-001 = Mineral, SMP-001 = Sample)
  • Sequence ensures uniqueness
  • Users can override for special cases

Cons:

  • Adds database complexity (triggers/functions)
  • Prefix table needs maintenance
  • Migration required for existing data

Decision Outcome

Chosen Option: Auto-Generated Codes with Prefix System

Every entity gets:

  1. code TEXT NOT NULL UNIQUE field
  2. CONSTRAINT entity_code_check CHECK (code ~ '^[A-Z0-9_-]{3,50}$') (uppercase alphanumeric + separators)
  3. Auto-generation via trigger if code is NULL/empty on INSERT

Architecture:

-- Central prefix registry
CREATE TABLE _system.entity_code_prefix (
    schema_name TEXT NOT NULL,
    entity_name TEXT NOT NULL,
    prefix      TEXT NOT NULL,
    next_seq    INTEGER NOT NULL DEFAULT 1,
    seq_width   INTEGER NOT NULL DEFAULT 6,
    PRIMARY KEY (schema_name, entity_name),
    CONSTRAINT prefix_format CHECK (prefix ~ '^[A-Z]{2,10}$')
);

-- Sequences per entity (created dynamically)
CREATE SEQUENCE geologicmapping.mapsymbol_code_seq;

-- Auto-generation trigger
CREATE TRIGGER entity_code_autogen
    BEFORE INSERT ON geologicmapping.mapsymbol
    FOR EACH ROW
    WHEN (NEW.code IS NULL OR NEW.code = '')
    EXECUTE FUNCTION _system.generate_entity_code();

-- Generation function
CREATE FUNCTION _system.generate_entity_code() RETURNS TRIGGER AS $$
DECLARE
    v_prefix TEXT;
    v_seq_width INTEGER;
    v_next_seq INTEGER;
    v_schema TEXT := TG_TABLE_SCHEMA;
    v_table TEXT := TG_TABLE_NAME;
BEGIN
    SELECT prefix, seq_width, next_seq
    INTO v_prefix, v_seq_width, v_next_seq
    FROM _system.entity_code_prefix
    WHERE schema_name = v_schema AND entity_name = v_table;
    
    IF NOT FOUND THEN
        RAISE EXCEPTION 'No code prefix registered for %.%', v_schema, v_table;
    END IF;
    
    -- Generate code: PREFIX-000001
    NEW.code := v_prefix || '-' || LPAD(v_next_seq::TEXT, v_seq_width, '0');
    
    -- Increment sequence
    UPDATE _system.entity_code_prefix
    SET next_seq = next_seq + 1
    WHERE schema_name = v_schema AND entity_name = v_table;
    
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

Positive Consequences

  • ✅ UUIDs remain PKs (globally unique, no sequence gaps)
  • ✅ Codes provide user-facing identifiers (searchable, memorable)
  • ✅ Bulk imports work: leave code empty → auto-generated
  • ✅ Custom codes work: provide code → used as-is
  • ✅ Consistent format across system
  • ✅ Prefix identifies entity type at a glance
  • ✅ Sequences prevent collisions
  • ✅ Audit logs show readable codes

Negative Consequences

  • ❌ Adds database complexity (triggers on all entity tables)
  • ❌ Prefix table must be maintained (new entities need registration)
  • ❌ Sequence gaps if transactions rollback (acceptable trade-off)
  • ❌ Code uniqueness constraint may fail for user-provided codes (handled via constraint error)

Implementation Details

Generator Changes

DSL Syntax (add to all entities):

ENTITY MapSymbol IDENTIFIED_BY id (
    id          UUID NOT NULL,
    code        TEXT NOT NULL,
    category_id UUID NOT NULL,
    ...
)
@code_prefix("MS")
@code_seq_width(6)

SQL Generation:

  1. Add code TEXT NOT NULL UNIQUE to all CREATE TABLE statements
  2. Generate CREATE SEQUENCE {schema}.{table}_code_seq
  3. Generate trigger for each table
  4. Populate _system.entity_code_prefix from @code_prefix annotations

API Changes:

  • {entity}_insert() accepts p_code TEXT DEFAULT NULL
  • If NULL, trigger generates code
  • If provided, used as-is (uniqueness enforced by constraint)

CSV Import:

  • Code column optional in seed CSVs
  • If empty, auto-generated on INSERT
  • If present, used verbatim

Migration Strategy

  1. Add code column to all existing tables (nullable initially)
  2. Backfill codes for existing rows using sequence
  3. Make NOT NULL after backfill
  4. Add UNIQUE constraint
  5. Create triggers

For USGS seed data:

  • MapSymbol: use fgdc_code as code (already unique)
  • LithologyPattern: use fgdc_code as code
  • Other entities: auto-generate

Key Files

  • src/dslgen/model/ddd-types.rkt: Add code-prefix to entity struct

Common Mistakes to Avoid

  • ❌ Forgetting UNIQUE constraint on code
  • ❌ Not registering prefix in _system.entity_code_prefix
  • ❌ Using lowercase in prefix (must be uppercase)
  • ❌ Making code a PK (UUID remains PK)
  • ❌ Not handling NULL code in INSERT functions
