Remove PL/pgSQL Stored Procedure API Layer

Remove PL/pgSQL Stored Procedure API Layer

  • Status: accepted
  • Date: 2026-01-20
  • Tags: architecture, postgresql, dba, stored-procedures
  • Supersedes: 20251223-api-only-testing-with-ddd-seeding.md (partially)

Context and Problem Statement

Muscovite currently generates PL/pgSQL stored procedures ({schema}_api.{entity}_insert/select/update/delete) as a database API layer. The C++ DBA repositories call these stored procedures via libpqxx. However, these stored procedures contain only pure CRUD operations with no business logic—they are essentially SQL wrappers that add complexity without benefit.

The question is: should we retain this stored procedure layer, or move to direct SQL execution in C++ repositories?

Decision Drivers

  • Simplicity: Two layers (C++ → SP → SQL) vs one layer (C++ → SQL) for simple CRUD
  • Debugging: Stored procedures obscure the actual SQL being executed
  • Portability: Business logic in C++ is portable; PL/pgSQL is PostgreSQL-specific
  • Performance: Minor overhead from SP calls (negligible, not primary driver)
  • Testing: Current pgTAP tests test SPs, not business logic
  • DDD Alignment: Stored procedures bypass aggregate boundaries (see ADR: Domain Services as Aggregate Boundary)

Considered Options

  1. Keep stored procedures as public API: Current architecture
  2. Remove stored procedures, direct SQL in repositories: Proposed change
  3. Keep SPs as internal implementation detail: Hybrid approach

Decision Outcome

Chosen option: "Remove stored procedures, direct SQL in repositories", because:

  • Stored procedures contain zero business logic—they are pure CRUD wrappers
  • Direct SQL in C++ is easier to debug (see actual queries in logs)
  • Business logic in C++ enables proper aggregate boundary enforcement
  • Removes an unnecessary abstraction layer
  • Enables CQRS-ready architecture (see ADR: CQRS-Ready Architecture)

Implementation

Before (current):

// DBA Repository calls stored procedure
auto result = txn.exec_params(
    "SELECT * FROM patient_api.patient_insert($1, $2, $3)",
    id, name, birth_date
);

After (new):

// DBA Repository executes SQL directly
auto result = txn.exec_params(
    "INSERT INTO fhir.patient (id, name, birth_date) "
    "VALUES ($1, $2, $3) RETURNING *",
    id, name, birth_date
);

Generator Changes:

  • Remove plpgsql-api.rkt from generation pipeline
  • Modify repository.rkt to emit direct SQL instead of SP calls
  • Keep schema/table generation in sqitch-unified.rkt
  • Keep domain_events table for event storage

Positive Consequences

  • Simpler architecture: one less layer to understand and maintain
  • Better debugging: actual SQL visible in logs and profilers
  • Portable business logic: C++ code works with any SQL database (future)
  • Enables proper DDD: aggregate boundaries enforced in domain services
  • Cleaner Sqitch migrations: only DDL, no function definitions

Negative Consequences

  • Loss of \df {schema}_api.* for quick API discovery in psql
  • Existing pgTAP tests become obsolete (replaced by Catch2 tests)
  • FK validation errors surface at SQL execution time, not SP pre-check
    • Mitigation: PostgreSQL FK constraints provide clear error messages
    • Mitigation: Domain services can add application-level validation if needed

Amendment: Defense-in-Depth Invariant Triggers (2026-02-23)

Context: BUG-003 (TransitionState lost-update) and the introduction of STATE_MACHINE blocks (ADR 20260221) raised the question of whether PostgreSQL triggers should guard state machine transitions at the storage level.

The Distinction That Matters

Category Banned? Rationale
CRUD API stored procedures ({schema}_api.entity_insert, etc.) YES Pure wrappers with zero business logic; added a useless abstraction layer
Triggers with business logic (pricing, fee calculation, complex derivations) YES Business logic belongs in C++ domain services (ADR 20260120-domain-services)
Invariant enforcement triggers (state machine guards, cross-row constraints that CHECK cannot express) NO — explicitly allowed Defense-in-depth; guards the DB against bypasses of the app layer
Standard CHECK constraints, FK constraints, UNIQUE constraints Always allowed Core PostgreSQL invariants

Invariant Enforcement Triggers: When to Use

A trigger is appropriate for defense-in-depth when all of these are true:

  1. The constraint cannot be expressed as a standard CHECK constraint (which is row-local)
  2. The constraint enforces an invariant (a rule that must always hold), not business logic
  3. Direct database modifications (e.g., psql, migration scripts, admin tools) would silently violate the invariant without the trigger
  4. The application layer (C++ domain service or StateMachine class) is the primary enforcement point — the trigger is a secondary guard, not a substitute

Prime example: state machine transition guards.

  • The C++ OrderStateMachine::transition() is the primary guardian.
  • A trigger BEFORE UPDATE ON order WHEN (NEW.current_state_id <> OLD.current_state_id) that validates the transition UUID pair against order_state_transition_log is the secondary guard.
  • Without the trigger, a direct UPDATE order SET current_state_id = '...' bypasses all guards.

Decision: State Machine Transition Triggers — Generate as Opt-In

State machine transition enforcement triggers are architecturally legitimate and recommended for production domains. They will be generated as opt-in via a STATE_MACHINE annotation (exact DSL syntax TBD, pending grammar extension):

STATE_MACHINE OrderLifecycle @uuid "..." FOR Order
    ENFORCE_AT DB_LEVEL  -- generates trigger; default is app-level only
{ ... }

When ENFORCE_AT DB_LEVEL is present, the generator emits an additional Sqitch change with a BEFORE UPDATE trigger that:

  1. Raises SQLSTATE '23514' (CHECK violation) on an invalid transition
  2. Delegates to a single PL/pgSQL function that reads from the lookup-table constants (no business logic — pure UUID pair validation against a allowed-transitions table)

Without ENFORCE_AT DB_LEVEL: no trigger is generated; the default is app-layer-only enforcement via the C++ StateMachine class (existing behavior).

Prohibition Boundary (NO EXCEPTIONS)

This amendment does not open the door to stored procedures containing business logic. The following remain banned:

  • PL/pgSQL functions with conditional branching on domain values (e.g., pricing, discounts)
  • Stored procedures replacing domain service methods
  • Triggers that perform DML on other tables (cascading side-effect triggers)
  • Functions called by application code as an API substitute

AI Agent Guidance

When This Decision Applies

  • When generating C++ DBA repository code
  • When generating database migrations (Sqitch)
  • When implementing CRUD operations for any entity
  • When debugging database operations

When This Decision Does NOT Apply

  • Complex queries that benefit from database-side optimization (use views instead)
  • Standard CHECK, UNIQUE, FOREIGN KEY constraints — always use these freely
  • domain_events table and related infrastructure
  • Invariant enforcement triggers — explicitly permitted (see Amendment above)
  • State machine transition guard triggers with ENFORCE_AT DB_LEVEL — permitted and generated

Enforcement

  • plpgsql-api.rkt generation is disabled/removed
  • repository.rkt generates direct SQL statements
  • Code review rejects stored procedure creation for CRUD operations
  • Catch2 tests validate repository behavior directly

Key Files

  • src/dslgen/generators/postgres/plpgsql-api.rkt - TO BE REMOVED
  • src/dslgen/generators/cpp/repository.rkt - Modified for direct SQL
  • src/dslgen/generators/postgres/sqitch-unified.rkt - Schema/table only
