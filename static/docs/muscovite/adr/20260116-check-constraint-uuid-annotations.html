CHECK Constraint UUID Annotations

CHECK Constraint UUID Annotations

  • Status: accepted
  • Deciders: Architecture Team
  • Date: 2026-01-16
  • Tags: dsl, uuid, identity, check-constraints

Context and Problem Statement

CHECK constraints currently generate synthetic pseudo-UUIDs by concatenating entity UUID and constraint name:

(format "~a-~a" ent-uuid (invariant-name inv))
;; Result: "26374859-6071-8293-0415-3758578517fb-clinical_status_valid"

This approach breaks semantic identity tracking:

  1. If the constraint is renamed, it appears as remove + add (not rename)
  2. Pseudo-UUIDs are not valid UUIDv4/v7 format
  3. Inconsistent with all other DSL elements which have explicit @uuid

Decision Drivers

  • All DSL elements should have stable UUID identity (ADR 2)
  • Renames should be detected as renames, not delete + create
  • UUID format should be consistent across the model
  • Semantic diff relies on UUID-based identity

Decision Outcome

Chosen option: Add @uuid annotation support for CHECK constraints

DSL Syntax Update

-- Before (no UUID)
CHECK clinical_status_valid: clinical_status IN ('active', 'inactive', 'resolved')

-- After (with UUID)
CHECK clinical_status_valid @uuid "12345678-1234-5678-1234-567812345678": 
    clinical_status IN ('active', 'inactive', 'resolved')

Parser Changes

;; invariant struct update
(struct invariant
  (name              ; Symbol: constraint name
   uuid              ; String: stable UUID (NEW)
   description       ; String: human description
   sql-check)        ; String or #f: SQL expression
  #:transparent)

;; Parser update
(define (parse-check-constraint ps)
  (expect-keyword! ps "CHECK")
  (define name (string->symbol (expect-identifier! ps)))
  (define uuid (parse-optional-uuid-annotation ps))  ; NEW
  (expect-token! ps ":")
  (define expr (parse-sql-expression ps))
  (invariant name uuid #f expr))

Diff Algorithm Update

(define (diff-invariants contexts-old contexts-new)
  (define old-map (invariants->uuid-map contexts-old))  ; keyed by UUID
  (define new-map (invariants->uuid-map contexts-new))
  
  (append
   (find-check-additions old-map new-map)
   (find-check-removals old-map new-map)
   (find-check-renames old-map new-map)      ; NEW: can detect renames
   (find-check-modifications old-map new-map))) ; NEW: expression changes

Migration Path

Existing CHECK constraints without UUIDs:

  1. Generator assigns UUIDs during first generation (like entity fields)
  2. UUIDs are written back to the .ddd file
  3. Subsequent diffs use the assigned UUIDs

Backward Compatibility

For constraints without @uuid annotation:

  • Parser assigns a deterministic UUID based on hash of (entity-uuid, constraint-name)
  • Warning emitted recommending explicit UUID
  • This ensures existing models continue to work

Consequences

Positive

  • CHECK constraints have proper identity tracking
  • Renames detected correctly (same UUID, different name)
  • Expression changes detected (same UUID, different expression)
  • Consistent with other DSL element identity

Negative

  • Existing .ddd files need UUID annotations added
  • Slightly more verbose syntax

AI Agent Guidance

  • Always include @uuid when creating CHECK constraints
  • When converting existing constraints, generate and add UUIDs
  • UUID should be placed between name and colon: CHECK name @uuid "..." : expr
