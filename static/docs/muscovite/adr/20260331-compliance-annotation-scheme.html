Compliance Annotation Scheme

Compliance Annotation Scheme

  • Status: draft
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-31
  • Tags: dsl, ddd, compliance, gdpr, hipaa, annotation, grammar, ir, generators
  • Supersedes: none

Technical Story: RFC-001 Extension 3 — add regulatory compliance annotations to any UUID-tracked DSL construct.

Context and Problem Statement

Regulated industries require formal classification of data elements for privacy (DSGVO/GDPR), health data (HIPAA), financial controls (SOX, PCI-DSS), and information security (ISO 27001). Today this classification lives in external spreadsheets and Confluence pages that drift from the code within weeks.

Muscovite's domain model is the single source of truth for schema, code, and documentation — but not for regulatory classification. The classification should live in the model itself, propagate through the IR, and drive generator behavior: auditor-facing documentation, compliance heatmaps, column-level access policies, and field masking rules.

Decision Drivers

  • Consulting engagements in healthcare, finance, public sector require formal data classification
  • Classification must be per-field granular (not just per-entity)
  • Scheme vocabularies differ per regulation (GDPR vs. HIPAA vs. PCI-DSS)
  • Must apply to any UUID-tracked construct — fields, events, transitions, saga steps
  • Annotation mechanism must be reusable (same pattern as quality annotations)
  • Grammar-first mandate (ADR 20260211)

Considered Options

Option 1: COMPLIANCE_SCHEME with @compliance annotations (chosen)

User-defined schemes in .mus; @compliance Scheme:Tag annotations on any UUID-tracked construct in .ddd files.

  • Good, because scheme is user-definable per engagement/regulation
  • Good, because applies to any UUID-tracked construct uniformly
  • Good, because multiple tags per construct (a field can be both DSGVO:PII and ISO27001:CONFIDENTIAL)
  • Good, because annotation mechanism is reused for quality attributes (Extension 4)
  • Bad, because verbose on heavily-annotated fields

Option 2: Built-in PII/SENSITIVE/PUBLIC enum

Hardcoded classification levels on fields.

  • Good, because simple
  • Bad, because one-size-fits-all — cannot represent GDPR Art. 9 distinctions
  • Bad, because only applies to fields, not events/transitions/saga steps

Option 3: External classification file (.compliance.yaml)

Side-by-side YAML file mapping UUIDs to classifications.

  • Good, because no grammar changes needed
  • Bad, because drifts from the model (the exact problem being solved)
  • Bad, because not visible in the DSL — auditors must cross-reference two files

Decision Outcome

Chosen option: "COMPLIANCE_SCHEME with @compliance annotations", because it places regulatory classification inside the model as a first-class concern, uses the same scheme mechanism as other annotation types, and supports per-engagement vocabularies.

DSL Syntax

.mus project file — scheme definition

COMPLIANCE_SCHEME DSGVO {
    PII
        DOC "Personally identifiable information — Art. 4(1) GDPR";
    SENSITIVE_HEALTH
        DOC "Special category health data — Art. 9(1) GDPR";
    FINANCIAL
        DOC "Payment and banking data";
    MINOR
        DOC "Data relating to persons under 16 — Art. 8 GDPR";
    RETENTION_LIMITED
        DOC "Subject to storage limitation — Art. 5(1)(e) GDPR";
}

COMPLIANCE_SCHEME ISO27001 {
    CONFIDENTIAL
        DOC "Annex A.8.2 — Information classification: confidential";
    INTERNAL
        DOC "Internal use only";
    PUBLIC
        DOC "Publicly accessible";
}

.ddd model — annotations

ENTITY Patient @uuid "..." IDENTIFIED_BY id (
    id           UUID   NOT_NULL @uuid "...",
    mrn          TEXT   NOT_NULL @uuid "..." @compliance DSGVO:PII ISO27001:CONFIDENTIAL,
    given_name   TEXT   NOT_NULL @uuid "..." @compliance DSGVO:PII,
    diagnosis    TEXT            @uuid "..." @compliance DSGVO:SENSITIVE_HEALTH,
    ssn          TEXT            @uuid "..." @compliance DSGVO:PII PCI_DSS:CARDHOLDER_DATA
)

Annotations are valid on any grammar rule that accepts UuidAnnotation — entities, fields, value objects, events, commands, sagas, saga steps, state machine transitions, extension points, aggregates, relationships, projections, policies, specifications, services, repositories, strong types, lookup tables, and any future UUID-tracked construct.

PEG Grammar

ComplianceAnnotation <- '@compliance' ComplianceTag+
ComplianceTag        <- IDENTIFIER ':' IDENTIFIER

IR Implications

  • Every protobuf IR message that carries a UUID also carries repeated ComplianceTag compliance_tags
  • ComplianceTag message: string scheme = 1; string tag = 2;
  • Scheme definitions stored at project level in the IR

Generator Implications

  • PostgreSQL: COMMENT ON COLUMN ... IS 'DSGVO:PII'. Optional: row-level security policy stubs, column-level encryption annotations, retention trigger stubs.
  • C++ DBA: PII field accessors routed through auditable wrapper. Deletion service method generated for entities with PII fields (GDPR Art. 17 right to erasure).
  • gRPC: sensitive = true in proto field comments. Field masking for non-privileged service accounts.
  • Documentation: auditor-scoped docs list all tagged fields, storage locations, access paths, retention policies. Generates GDPR Article 30 record of processing activities. ISO 27001 control mapping per entity.
  • Heatmap: compliance density per context — risk visualization for stakeholder review.

Positive Consequences

  • Regulatory classification becomes part of the single source of truth
  • Auditor documentation generated automatically from the model
  • Compliance drift eliminated — classification travels with the field definition
  • GDPR Article 30 reports generated, not hand-authored

Negative Consequences

  • Annotation verbosity on heavily-regulated entities
  • Scheme definitions require domain expertise to author correctly
  • Generator complexity increases for compliance-aware output

AI Agent Guidance

When This Decision Applies

  • When adding @compliance annotations to any UUID-tracked construct
  • When generating documentation for regulated industries
  • When building generators that produce compliance-aware output (masking, RLS, audit)

When This Decision Does NOT Apply

  • Non-regulated domains where compliance classification is not required
  • Quality attributes (use @quality annotation instead — separate ADR)

Enforcement

  • PEG grammar rules for ComplianceAnnotation
  • Parser validation: referenced scheme:tag must exist in project COMPLIANCE_SCHEME
  • Generator-specific validation per target (e.g., PII fields must have retention policy)
