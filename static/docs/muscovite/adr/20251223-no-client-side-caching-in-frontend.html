No Client-Side Data Caching in Frontend

No Client-Side Data Caching in Frontend

  • Status: accepted
  • Deciders: jolo
  • Date: 2025-12-23
  • Tags: qt, frontend, caching, architecture

Technical Story: Defining data management strategy for Qt frontend applications.

Context and Problem Statement

When building a Qt frontend that communicates with a gRPC backend, we must decide whether to cache data on the client side. Caching can improve perceived performance but introduces complexity around cache invalidation, consistency, and memory management.

Decision Drivers

  • Data consistency between multiple clients
  • Simplicity of frontend architecture
  • Real-time data requirements (multiple users editing)
  • Memory footprint on client devices
  • Offline capability requirements (none currently)
  • Server as single source of truth principle

Considered Options

  1. No Client-Side Caching - Always fetch from server
  2. In-Memory Cache with TTL - Cache with time-based expiration
  3. Optimistic UI with Cache - Update UI immediately, sync with server
  4. Full Offline-First Cache - SQLite/local storage with sync

Decision Outcome

Chosen option: "No Client-Side Caching", because it ensures data consistency, simplifies the frontend architecture, and aligns with the principle that the server (via PostgreSQL) is the single source of truth.

Positive Consequences

  • Data is always current from server
  • No cache invalidation complexity
  • Simpler frontend code (no cache management)
  • No stale data bugs
  • Multiple clients always see consistent data
  • Reduced memory footprint

Negative Consequences

  • Every operation requires network round-trip
  • Perceived latency for repeated queries
  • More server load for redundant queries
  • No offline capability

Pros and Cons of the Options

No Client-Side Caching

  • Good, because always shows current server state
  • Good, because simple architecture
  • Good, because no cache invalidation bugs
  • Good, because low memory usage
  • Bad, because network latency on every operation
  • Bad, because no offline support

In-Memory Cache with TTL

  • Good, because faster repeated queries
  • Good, because reduces server load
  • Bad, because stale data within TTL window
  • Bad, because complex invalidation logic
  • Bad, because memory management overhead

Optimistic UI with Cache

  • Good, because instant UI feedback
  • Good, because smooth user experience
  • Bad, because complex conflict resolution
  • Bad, because can show incorrect state temporarily
  • Bad, because significant implementation complexity

Full Offline-First Cache

  • Good, because works without network
  • Good, because best perceived performance
  • Bad, because very complex sync logic
  • Bad, because conflict resolution nightmare
  • Bad, because significant storage requirements
  • Bad, because overkill for our use case

Implementation Notes

Service Layer Pattern

All data access goes through Application Services that:

  1. Make gRPC call to server
  2. Convert response to Qt types
  3. Emit signal with result
  4. UI updates from signal
// Example: No caching, direct server fetch
QFuture<QList<Mineral*>> MineralService::list() {
    return QtConcurrent::run([this]() {
        // Always fetch from server
        auto response = stub_->List(...);
        return convertToQt(response);
    });
}

UI Refresh Patterns

  • List views: Refresh on navigation/focus
  • Detail views: Fetch on open, refresh on save
  • Forms: Submit to server, refresh parent list on success

AI Agent Guidance

When This Decision Applies

  • Implementing data fetching in Qt frontend
  • Designing service layer for entity access
  • Adding new entity views or editors

When This Decision Does NOT Apply

  • Reference data that truly never changes (could cache indefinitely)
  • UI state (selections, scroll positions - not server data)
  • Authentication tokens (managed separately by AuthManager)

Enforcement

  • Code review: No local data stores or caches for entity data
  • Service classes: Must always call server, no local lookup

Key Files Affected

  • src/muscframework/qt/service/ - Base service classes
  • generated-*/qt/service/ - Generated entity services
