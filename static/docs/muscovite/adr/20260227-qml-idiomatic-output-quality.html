QML Idiomatic Output Quality

QML Idiomatic Output Quality

  • Status: accepted
  • Date: 2026-02-27
  • Tags: qt, qml, qt-quick, generators, idiom, quality, best-practices
  • Deciders: Johannes Lochmann

Technical Story: Comprehensive audit of QML generators and generated Qt Quick output against official Qt 6 best practices, coding conventions, and performance guidelines. Establishes canonical patterns and feature roadmap for v1.42.0.

Context and Problem Statement

The QML generators (dslgen/generators/qt/) produce Qt Quick UIs from .ddd domain models. An audit against three authoritative Qt documents revealed 15 categories of deviation from official Qt 6 guidelines:

  • Qt Quick Best Practices
  • QML Coding Conventions
  • Qt Quick Performance

The muscovite-cpp framework (framework/qt/) is Qt Widgets-based (QWidget, QFormLayout), NOT QML. Generated QML output is produced entirely by Racket generators. This audit covers only the generators and their output.

Decision Drivers

  • Qt 6 best practices mandate specific patterns (qsTr, versionless imports, typed properties)
  • Generated QML should be indistinguishable from hand-written Qt 6 code
  • Performance patterns (Loader over Qt.createComponent, QAbstractListModel over QVariantList) affect end-user experience
  • Consistency: identical audit methodology as PostgreSQL audit (ADR 20260227-postgres-idiomatic-output-quality)

Audit Methodology

  • Read ALL 27 Racket generator files in dslgen/generators/qt/ (QML-related)
  • Read ALL 56 C++ framework files in framework/qt/
  • Examined generated QML output in examples/{hello-world,fhir-health,geosciences}/generated/qt-quick/
  • Cross-referenced every finding with specific Qt documentation section
  • Confirmed findings with grep across the codebase
  • Every finding includes exact file:line references

Decisions

Decision 1: qsTr() for ALL User-Facing Strings (HIGH)

Finding: Form field labels, dialog titles, placeholderText, and ToolTip.text are emitted as raw string literals without qsTr() wrapping in qml-dialog.rkt. The qsTr call IS used correctly in confirmation dialogs (L305, L318) and button text (L295-302), proving awareness of the pattern — but it's inconsistently applied.

Evidence:

  • qml-dialog.rkt L81: Label { text: "~a:" } — missing qsTr
  • qml-dialog.rkt L85: Label { text: "~a *:" } — missing qsTr
  • qml-dialog.rkt L90: Label { text: "~a:" } — missing qsTr
  • qml-dialog.rkt L95: Label { text: "~a:" } — missing qsTr
  • qml-dialog.rkt L112: Label { text: "~a:" } — missing qsTr
  • qml-dialog.rkt L134: Label { text: "~a:" } + placeholderText: "~a" — missing qsTr
  • qml-dialog.rkt L186: dialog title — missing qsTr
  • qml-dialog.rkt L305/L318: qsTr("Are you sure?") — correctly wrapped (confirms partial awareness)

Qt Reference: "Make User-Facing Strings Translatable" — Best Practices

Remediation: Wrap all 6 label format strings, title, placeholderText, and ToolTip.text in qsTr(). Feature: FTR-167.

Decision 2: Typed required property Declarations (MEDIUM)

Finding: Dialog and list components declare required property var viewModel instead of using the actual ViewModel type. This defeats QML's type system, prevents static analysis by qmllint, and blocks AOT compilation optimization.

Evidence:

  • qml-dialog.rkt L184: required property var viewModel
  • qml-list.rkt L80: required property var viewModel
  • qml-dialog-children.rkt: child section components

Counter-example: qml-saga.rkt L212 correctly emits required property {Type}ViewModel viewModel — proving the pattern is known and achievable.

Qt Reference: "Type Safety — ALWAYS use actual types instead of var" — Best Practices

Remediation: Emit required property {EntityName}ViewModel viewModel. Feature: FTR-169.

Decision 3: Versionless Qt 6 Imports (MEDIUM)

Finding: All QML-emitting generators use versioned Qt 5-style imports (import QtQuick 2.15). Qt 6 requires versionless imports for qt_add_qml_module, AOT compilation, and access to Qt 6 types.

Evidence (grep confirmed — 8 files):

  • qml-dialog.rkt L189: import QtQuick 2.15
  • qml-list.rkt L75: import QtQuick 2.15
  • qml-saga.rkt L204
  • qml-state-machine.rkt L210
  • qml-search.rkt L285
  • qml-app.rkt L35, L106
  • e2e-qml-test.rkt L84

Qt Reference: Qt 6 migration guide, qt_add_qml_module documentation

Remediation: Create centralized (qml-imports) function in common.rkt, replace all inline import strings. Feature: FTR-168.

Decision 4: QAbstractListModel for ListView Data (MEDIUM)

Finding: ViewModel header generator emits Q_PROPERTY(QVariantList items ...) for list data. This forces full model replacement on every change, destroys ListView animations, breaks delegate recycling, and prevents role-based data access.

Evidence:

  • qml-viewmodel-header.rkt L85: Q_PROPERTY(QVariantList items READ items NOTIFY itemsChanged)
  • qml-viewmodel-header.rkt L89-93: FK item list properties also QVariantList

Qt Reference: "Models/Views — use proper model for incremental updates, animations, and delegate recycling" — Performance

Remediation: Generate QAbstractListModel subclass per entity, add framework base class EntityListModel. Feature: FTR-170.

Decision 5: Loader Pattern for Dynamic Dialogs (MEDIUM)

Finding: qml-list.rkt uses Qt.createComponent() for dialog instantiation. This is synchronous and blocks the UI thread during QML parsing and compilation.

Evidence:

  • qml-list.rkt L83-92: Qt.createComponent("../dialogs/entity_dialog.qml")

Qt Reference: "Lazy Initialization — use Loader with asynchronous: true" — Performance

Remediation: Replace with Loader { asynchronous: true; source: "..." }. Feature: FTR-171.

Decision 6: onValueModified for SpinBox (LOW)

Finding: SpinBox generates onValueChanged handler, which fires on programmatic value changes (e.g., ViewModel loading), creating feedback loops and false dirty-flag triggers.

Evidence:

  • qml-dialog.rkt L95: onValueChanged: { viewModel.set~a(value); dialog.dirty = true }

Qt Reference: "Prefer explicit interaction signals over value-changed signals" — Best Practices

Remediation: Replace onValueChanged with onValueModified. Feature: FTR-172.

Decision 7: Remove clip:true from ListView (LOW)

Finding: qml-list.rkt sets clip: true on ListView. This is unnecessary (ListView manages delegate visibility internally) and forces extra GPU work.

Evidence:

  • qml-list.rkt L136: clip: true

Qt Reference: "Clipping — disabled by default, increases renderer complexity" — Performance

Remediation: Remove clip: true. Feature: FTR-173 (cosmetic bundle).

Decision 8: Theme Constants for Layout Dimensions (LOW)

Finding: qml-dialog.rkt hardcodes Layout.preferredWidth: 140 in 6 form field generators. Magic pixel values prevent DPI-aware scaling and violate Qt's scalable UI guidelines.

Evidence:

  • qml-dialog.rkt L81, L85, L90, L95, L112, L134: Layout.preferredWidth: 140

Qt Reference: "Scalable UIs — avoid explicit pixel sizes; use anchors, layouts, and scalable measurements" — Best Practices

Remediation: Extract to Theme.formLabelWidth property. Feature: FTR-173 (cosmetic bundle).

Decision 9: QML Property Ordering Convention (LOW)

Finding: Generated Dialog components don't follow the Qt coding conventions property ordering: id → required properties → property declarations → signal declarations → JavaScript functions → object properties → child objects.

Evidence:

  • qml-dialog.rkt — generate-qml-edit-dialog function mixes property ordering

Qt Reference: "Property ordering" — Coding Conventions

Remediation: Reorder emitted properties. Feature: FTR-173 (cosmetic bundle).

Decision 10: Grouped Anchor Notation (LOW)

Finding: When 3+ anchors properties are set, dot notation is used instead of group notation.

Evidence:

  • qml-list.rkt — delegate RowLayout anchors

Qt Reference: "Use group notation when setting several properties from one group" — Coding Conventions

Remediation: Use anchors { } block when 3+ properties. Feature: FTR-173 (cosmetic bundle).

Decision 11: Consolidate Type Dispatch Blocks (MEDIUM)

Finding: Four separate QML generator files maintain independent (case type-sym ...) dispatch blocks for converting DSL types to C++ expressions. These blocks drift independently, and adding a new type requires updating all four.

Evidence:

  • qml-viewmodel-mapping.rkt — type dispatch block
  • qml-viewmodel-grpc.rkt — type dispatch block
  • qml-child-viewmodel.rkt — type dispatch block
  • display-helper-codegen.rkt — type dispatch block

Remediation: Consolidate into common.rkt functions. Feature: FTR-174.

Decision 12: Resolve except-in Import Fragility (MEDIUM)

Finding: 21 QML generator files use (require (except-in "../../model/ddd-types.rkt" ...)) to exclude conflicting identifiers. This is fragile — any new conflicting export from ddd-types.rkt breaks all 21 files simultaneously.

Evidence (grep confirmed — 21 files):

dslgen/generators/qt/*.rkt: (except-in ../../model/ddd-types.rkt dsl-type->cpp-type dsl-type->proto-type)

Remediation: Export curated identifiers through common.rkt. Feature: FTR-174.

Decision 13: Template DSL Compliance (HIGH — Deferred to v1.36.0)

Finding: Zero QML generator files use define-template per ADR 20260226. All 17 files use raw string-append/format chains. This is tracked in the idiomatic Racket migration roadmap (Wave 2b, v1.36.0).

Evidence (grep confirmed):

dslgen/generators/qt/*.rkt: 0 matches for "define-template"

Note: This is NOT a new decision — ADR 20260226 already mandates template DSL adoption. Listed here for completeness. Wave 2b (v1.36.0) will address this. Patterns 1–12 in this ADR can be fixed within the current string-append/format generators before the template DSL migration.

Cross-reference: FTR-174 for immediate consolidation; v1.36.0 for full template migration.

Decision 14: Legacy Project-DSL Path Assessment (LOW)

Finding: 6 files in dslgen/generators/qt/ reference /project-dsl/ paths or appear to be orphaned from a pre-.mus era. These may be dead code.

Evidence:

  • qt-project-dsl.rkt — project DSL consumer
  • qml-app.rkt — references project metadata via legacy path
  • 4 additional files with project-dsl imports

Remediation: Assess which code paths are dead and remove. Feature: FTR-175.

Decision 15: QML Pattern Enforcement Test (HIGH)

Finding: No automated test verifies generated QML output for compliance with the patterns in this ADR. Without a test, regressions will occur as generators are modified.

Remediation: Create test-qml-patterns.rkt that parses generated QML output and checks:

  • No raw string literals (non-qsTr) in Label.text, Dialog.title, placeholderText
  • No versioned imports
  • No property var viewModel
  • No Qt.createComponent
  • No onValueChanged on SpinBox
  • No clip: true on ListView

Feature: FTR-176.

Positive Consequences

  • Generated QML will pass qmllint without warnings
  • Qt Linguist / lupdate can discover all translatable strings
  • AOT compilation enabled via versionless imports + typed properties
  • ListView performance improved via QAbstractListModel + no clip
  • Dialog loading non-blocking via Loader pattern
  • Generator maintenance reduced via type dispatch consolidation
  • Pattern compliance enforced by automated test (FTR-176)

Negative Consequences

  • 10 features to implement (~3-5 iterations depending on scope)
  • QAbstractListModel (FTR-170) requires new framework base class
  • except-in refactoring (FTR-174) touches 21 files

AI Agent Guidance

When This Decision Applies

  • When modifying any QML generator in dslgen/generators/qt/
  • When adding new QML output to any generator
  • When reviewing generated QML output for quality

When This Decision Does NOT Apply

  • Qt Widgets generators (separate pattern set)
  • Framework C++ code in framework/qt/ (Widgets-based, not QML)
  • Template DSL migration (covered by ADR 20260226, Wave 2b)

Enforcement

  • test-qml-patterns.rkt (FTR-176) enforces Decisions 1-7 via automated regex checks
  • Code review: verify qsTr() on all new string literals in QML generators
  • doc/QML-PATTERNS.md is the canonical reference for pattern compliance
