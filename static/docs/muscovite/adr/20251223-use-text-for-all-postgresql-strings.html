Use TEXT for All String Columns in PostgreSQL

Use TEXT for All String Columns in PostgreSQL

  • Status: accepted
  • Deciders: Database architects
  • Date: 2024-06-01
  • Tags: postgresql, schema, types

Context and Problem Statement

When generating PostgreSQL schemas from DSL entity definitions, we need to decide how to map string fields to PostgreSQL types. Should we use VARCHAR(n) with length constraints, VARCHAR without length, or TEXT?

Decision Drivers

  • PostgreSQL internal storage is identical for VARCHAR and TEXT
  • Length validation belongs at the application layer, not database
  • Simplifies generator logic (no need to track/propagate lengths)
  • Avoids artificial truncation of data
  • Migrations are easier (no ALTER COLUMN for length changes)

Considered Options

  • TEXT for all string fields
  • VARCHAR(n) with explicit lengths from DSL
  • VARCHAR without length (equivalent to TEXT)

Decision Outcome

Chosen option: "TEXT for all string fields", because PostgreSQL stores VARCHAR and TEXT identically, and length constraints are better enforced at the application layer where error messages can be more meaningful.

Positive Consequences

  • Simpler generator code (no length tracking)
  • No migration required when business requirements change string lengths
  • Consistent behavior across all string fields
  • No risk of silent truncation

Negative Consequences

  • No database-level length enforcement (must be done in application/API)
  • Schema doesn't document expected maximum lengths
  • CHECK constraints needed if hard limits are truly required

Pros and Cons of the Options

TEXT for all strings

  • Good, because identical storage performance to VARCHAR
  • Good, because no artificial limits
  • Good, because simpler migrations
  • Good, because generator logic is simpler
  • Bad, because no schema-level documentation of expected lengths
  • Bad, because relies on application-layer validation

VARCHAR(n) with explicit lengths

  • Good, because documents expected lengths in schema
  • Good, because database enforces constraints
  • Bad, because requires migrations when lengths change
  • Bad, because silent truncation risk if not careful
  • Bad, because generator must track and propagate lengths

VARCHAR without length

  • Good, because functionally equivalent to TEXT
  • Bad, because inconsistent with common PostgreSQL practice
  • Bad, because no advantage over TEXT

AI Agent Guidance

When This Decision Applies

  • ALL PostgreSQL DDL generation for string/text fields
  • DSL types: String, VARCHAR, VARCHAR(n), TEXT all map to TEXT
  • Generated schemas, lookup tables, domain events, sagas

When This Decision Does NOT Apply

  • Non-PostgreSQL targets (C++ uses std::string, Proto uses string)
  • CHAR(n) for fixed-width codes (rare, must be explicit in DSL)

Enforcement

  • Racket generators map all string types to TEXT
  • No VARCHAR appears in generated DDL
  • Length validation in PL/pgSQL API functions if needed
