MAPPING DSL Block for Brownfield Migration

MAPPING DSL Block for Brownfield Migration

  • Status: accepted
  • Date: 2026-02-17
  • Tags: dsl, grammar, mapping, migration, brownfield, context-item
  • Deciders: Johannes Lochmann

Technical Story: The brownfield migration factory (ADR 20260217) requires a DSL construct to declare field-level mappings between legacy schema columns and target entity fields, including data transformations and DLQ routing.

Context and Problem Statement

The migration factory needs declarative configuration for how legacy data maps to target entities. This configuration must be:

  1. Expressible in the .ddd DSL (not external config files)
  2. Parseable by the existing recursive-descent parser
  3. Rich enough to capture field transforms (@trim, @default, @transform)
  4. Tied to a specific bounded context (access to entities, strong types, validation rules)

The existing ACL/Translator grammar construct (AclDecl → Translator → TranslatorMapping → MappingExpr) provides ~70% of the needed structure.

Decision Drivers

  • Grammar-first rule (ADR 20260211): PEG grammar change must precede parser changes
  • MAPPING needs access to context-local entities and strong types for validation
  • Consistency with existing DSL constructs (ACL, Saga, Projection)
  • The ACL TranslatorMapping rule (field = MappingExpr) is structurally identical to migration field mapping

Considered Options

  1. MAPPING as a ContextItem — lives inside CONTEXT block, alongside entities and sagas
  2. MAPPING as a top-level construct — lives at file root level, like context maps
  3. MAPPING in a separate .map file — new file extension with dedicated parser
  4. External YAML/JSON config — migration config outside the DSL

Decision Outcome

Chosen option: Option 1 — MAPPING as a ContextItem, because a mapping is target-context-specific and needs access to the context's entities, strong types, and validation rules. This follows the same pattern as Saga, Projection, and ACL which are all context-scoped.

Grammar Extension

# New ContextItem variant (Section 2 of muscovite.peg)
ContextItem <- ... / MappingBlock

# New rules (new section)
MappingBlock     <- 'MAPPING' IDENTIFIER UuidAnnotation '{' MappingClause+ '}'
MappingClause    <- MappingSource / MappingTarget / MappingFields / MappingDlq
MappingSource    <- 'SOURCE' STRING_LITERAL
MappingTarget    <- 'TARGET' DottedIdentifier
MappingFields    <- 'FIELDS' '{' MappingField+ '}'
MappingField     <- IDENTIFIER '->' IDENTIFIER MappingAnnotation*
MappingAnnotation <- '@transform' '(' IDENTIFIER ')' / '@trim' / '@default' '(' Literal ')' / '@format' '(' STRING_LITERAL ')'
MappingDlq       <- 'DLQ' STRING_LITERAL

DottedIdentifier <- IDENTIFIER ('.' IDENTIFIER)*

DSL Syntax Example

CONTEXT PatientCare @uuid "..." {
    ENTITY Patient IDENTIFIED_BY id (
        id UUID NOT_NULL
        mrn TEXT NOT_NULL
        given_name TEXT NOT_NULL
        email TEXT
    )

    MAPPING LegacyPatientImport @uuid "a1b2c3d4-..." {
        SOURCE "legacy_ehr.patients"
        TARGET Patient

        FIELDS {
            patient_id   -> id          @transform(to_uuidv5)
            medical_num  -> mrn         @trim
            first_name   -> given_name  @trim @default("Unknown")
            email_addr   -> email       @trim
        }

        DLQ "patient_import_errors"
    }
}

IR Struct

(struct mapping (name uuid source-table target-entity fields dlq-name doc) #:transparent)
(struct mapping-field (source-name target-name transforms doc) #:transparent)
(struct field-transform (type args) #:transparent)
;; type is one of: 'transform, 'trim, 'default, 'format
;; args is a list: '() for trim, '("value") for default, '("to_uuidv5") for transform

Positive Consequences

  • MAPPING follows established ContextItem patterns — parser, IR, generators all have clear precedent
  • Field mapping syntax (source -> target @transform(...)) is readable and self-documenting
  • ACL MappingExpr infrastructure is reusable
  • Grammar conformance tests validate the extension automatically

Negative Consequences

  • Transform annotation language is a fixed set — extending with custom transforms requires grammar changes
  • DottedIdentifier for TARGET introduces a new grammar concept (but is simple: Aggregate.Entity)

AI Agent Guidance

When implementing the MAPPING block:

  • PEG grammar first: add MappingBlock to Section 2's ContextItem alternatives in dslgen/grammar/muscovite.peg
  • IR struct: add mapping, mapping-field, field-transform to dslgen/model/ddd-types.rkt near the ACL structs
  • Parser: add parse-mapping-block to dslgen/parser/parser.rkt, following parse-acl-declaration pattern
  • Wire into context parsing loop: add 'MAPPING case alongside 'SAGA, 'PROJECTION, 'ACL
  • Add MAPPING keyword to dslgen/parser/lexer.rkt
  • Verify: racket tests/test-grammar-conformance.rkt must pass
  • Built-in transforms: trim, default, to_uuidv5, format_date, lowercase, uppercase, coalesce, truncate, map_value
