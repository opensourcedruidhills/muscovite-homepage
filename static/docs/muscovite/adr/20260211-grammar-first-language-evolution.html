Grammar-First Language Evolution

Grammar-First Language Evolution

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: grammar, dsl, evolution, process, testing, onboarding

Context and Problem Statement

Muscovite's DSL has a formal PEG grammar (dslgen/grammar/muscovite.peg, 767 lines, 43 rules) and a recursive descent parser (dslgen/parser/parser.rkt, ~2,900 lines). Both were developed in parallel: the parser was written first, the grammar was extracted and formalized in v1.1.0.

As the DSL matures, the question is: which artifact leads when the language evolves?

Historically, new DSL features were added by:

  1. Editing the parser directly (adding keywords, parse functions)
  2. Updating the model structs
  3. Writing generators
  4. Documenting the syntax retroactively

This works for a solo developer who holds the full parser in their head. It does not scale to:

  • AI agents who need an unambiguous specification to generate syntactically valid .ddd files
  • Future contributors who need a reference that isn't 2,900 lines of Racket
  • Cross-version compatibility where syntax changes must be tracked as deliberate, versioned decisions

Decision Drivers

  • The formal PEG grammar already exists — this inverts the workflow, not creates it
  • AI agents generate better .ddd files when constrained by a formal grammar (per VISION.md)
  • Parser bugs are harder to detect without a reference specification to test against
  • Grammar changes are small, reviewable diffs; parser changes are large, procedural diffs
  • Onboarding (human or AI) is faster with a declarative spec than with imperative parser code

Considered Options

Option 1: Grammar-First (spec leads, parser follows)

  1. Propose syntax change in PEG grammar
  2. Review grammar diff (small, declarative, reviewable)
  3. Implement in parser to match grammar
  4. Validate with grammar conformance test
  5. Version-stamp the grammar

Option 2: Parser-First (current approach, grammar follows)

  1. Edit parser directly
  2. Update grammar afterward (if remembered)
  3. Risk: grammar and parser drift apart

Option 3: Machine-Executable Grammar (parser generated from PEG)

  1. Write grammar in a Racket PEG library (e.g., parsack)
  2. Generate parser from grammar automatically
  3. No drift possible — single source of truth

Decision Outcome

Chosen option: Option 1 — Grammar-First, because it provides maximum rigor with minimal tooling change. Option 3 is a future possibility but requires replacing the hand-tuned recursive descent parser, which is well-tested and handles error recovery gracefully. Option 1 gives us 90% of the benefit at 10% of the cost.

The Process

1. GRAMMAR    →  Edit muscovite.peg (add/modify rule)
2. VERSION    →  Bump grammar version header
3. IMPLEMENT  →  Update parser.rkt + lexer.rkt to match
4. MODEL      →  Update ddd-types.rkt if new structs needed
5. GENERATORS →  Add/update generators for new construct
6. TEST       →  Run grammar conformance + generator test suite
7. EXAMPLES   →  Add usage to example .ddd files

Step 1 is the gate. No parser change without a grammar change first.

What Counts as a Grammar Change

Change Requires Grammar-First? Example
New keyword ✅ Yes Adding PROJECTION construct
New clause on existing construct ✅ Yes Adding TIMEOUT to COMMAND
New field type ✅ Yes Adding Duration to type system
Parser bug fix (syntax unchanged) ❌ No Fixing precedence in expression parsing
Generator change (no syntax impact) ❌ No Changing C++ output format
Error message improvement ❌ No Better error on missing semicolon
Refactoring parser internals ❌ No Splitting a parse function

Grammar Versioning

The grammar file carries a version header. Grammar versions track language versions:

Version: 1.2.0

When the grammar changes, the version increments. The grammar conformance test (tests/test-grammar-conformance.rkt) validates that example .ddd files exercise the grammar rules.

Is the Current Test Harness Sufficient?

Assessment of current safety nets:

Safety Net Coverage Strength Gap
Grammar conformance test 19/24 rules exercised by examples ✅ Validates parser↔︎grammar alignment 5 rules uncovered
Generator test suite (test-output-generator-suite.rkt) 10 generators, 751 lines ⚠️ Currently failing (exit code 1) Must pass before this ADR is actionable
Parser feature tests (test-parser-features.rkt) Core parse paths ✅ rackunit based Doesn't cover new constructs until added
Cross-layer verification 6 modules, 1,926 lines ✅ Infrastructure exists ADR still proposed; not wired to CTest
Example .ddd files 37 files across 2 projects ✅ Real-world coverage Not all grammar rules exercised

Verdict: The test harness is necessary but not yet sufficient. Specifically:

  1. Generator test suite must pass — it's the primary safety net for "does grammar change X break any generator output?" Currently failing.
  2. Grammar conformance test should reach 24/24 rules — add example .ddd coverage for the 5 uncovered rules.
  3. CTest integration (ADR 20260210) should be completed — so ctest --preset racket runs all Racket tests in one command.

Once these three are green, the grammar-first workflow has a closed-loop safety net:

Grammar change → Parser change → Conformance test (grammar ↔ parser)
                                → Generator tests (parser ↔ generators)
                                → Example parsing (real-world validation)

Minimum Viable Safety Net (before enforcing grammar-first)

This is achievable within the v1.2.0 iteration (FTR-004).

Positive Consequences

  • Onboarding: New contributors (human or AI) read 768 lines of PEG, not 2,900 lines of Racket
  • AI generation: LLMs can follow formal grammar to produce valid .ddd files
  • Reviewability: Grammar diffs are 5–20 lines; parser diffs are 50–200 lines for the same feature
  • Drift prevention: Grammar and parser can't diverge if grammar leads
  • Versioning: Language evolution is explicitly tracked via grammar version stamps
  • Robustness: Specification-first catches design issues before implementation

Negative Consequences

  • Process overhead: Every syntax feature requires a grammar edit first (even if trivial)
  • Two artifacts to maintain: Grammar and parser must stay synchronized
  • Not machine-enforced: Discipline required — no tooling prevents parser-first edits (yet)
  • Grammar doesn't cover error recovery: Parser's error handling is richer than what PEG can express

Future Evolution

If grammar-first proves valuable, Option 3 (machine-executable grammar) becomes attractive:

  1. Translate PEG grammar to a Racket parser combinator library
  2. Generate parser skeleton from grammar
  3. Hand-tune error recovery in generated code
  4. Single source of truth, zero drift

This is a v2.x consideration. Grammar-first process is sufficient for v1.x.

AI Agent Guidance

When This Decision Applies

  • Adding any new DSL syntax (keywords, clauses, constructs)
  • Proposing new field types for the type registry
  • Modifying existing syntax (adding optional clauses, changing delimiters)
  • Planning new DSL features in iteration planning

When This Decision Does NOT Apply

  • Parser bug fixes that don't change accepted syntax
  • Generator changes (output format, template strings)
  • Error message improvements
  • Internal parser refactoring
  • Model struct changes that don't affect surface syntax

Enforcement

  • Before implementing any parser change: Check if it changes accepted syntax
  • If syntax changes: Edit dslgen/grammar/muscovite.peg first, bump version
  • After parser change: Run racket tests/test-grammar-conformance.rkt to validate alignment
  • After generator change: Run racket tests/test-output-generator-suite.rkt to validate output

Key Files

  • dslgen/grammar/muscovite.peg — formal PEG grammar (spec leads)
  • dslgen/parser/parser.rkt — recursive descent parser (implementation follows)
  • dslgen/parser/lexer.rkt — tokenizer (keywords must match grammar terminals)
  • dslgen/tests/test-grammar-conformance.rkt — grammar ↔︎ parser alignment test
  • dslgen/tests/test-output-generator-suite.rkt — generator output validation
