Responsive Multi-Form-Factor Layouts as Future Extension

Responsive Multi-Form-Factor Layouts as Future Extension

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: layout, form-factor, responsive, future, deferred

Context and Problem Statement

The Abstract Layout Vocabulary (ADR abstract-layout-vocabulary) introduces a single LAYOUT block per aggregate. A natural extension is per-form-factor layouts within the same aggregate:

AGGREGATE Patient ROOT Patient
    LAYOUT desktop {
        STYLE split_view
        MASTER { ORIENTATION horizontal }
    }
    LAYOUT tablet {
        STYLE multi_section
        MASTER { ORIENTATION grid }
    }
    LAYOUT mobile {
        STYLE single_page
        MASTER { ORIENTATION vertical GROUPING flat }
    }
    DOC "Patient aggregate";

This would allow one .ddd file to generate optimized UIs for desktop, tablet, mobile, watch, and embedded devices — each with appropriate layout decisions.

Question: Should multi-form-factor LAYOUT blocks be part of the initial grammar extension?

Decision Drivers

  • Grammar complexity — adding form-factor qualifiers multiplies grammar rules
  • Implementation risk — only 2 UI generators exist (Qt Widgets, QML); no mobile/tablet generators
  • YAGNI — no mobile/tablet targets are scheduled for implementation
  • Prove-before-scale — the single LAYOUT block must work before extending to N form factors
  • Reversibility — easier to add form-factor qualifiers later than to remove them

Considered Options

Option 1: Include Multi-Form-Factor in Initial Grammar

Add LAYOUT <form-factor>? { ... } now. Generators select the matching layout for their form factor.

Option 2: Defer to Future Extension (Single LAYOUT Only)

Ship single LAYOUT block first. Form-factor selection happens in target configuration, not DSL. Extend grammar later when mobile/tablet generators exist.

Decision Outcome

Chosen option: Option 2 — Defer, because:

  • Only 2 generators exist today (Qt Widgets Desktop, Qt QML Desktop) — both are desktop
  • A mobile generator does not exist yet
  • Adding form-factor to grammar now would be speculative — untestable until mobile generators exist
  • Target configuration already carries form-factor field — generators can select defaults per form factor without DSL involvement
  • Grammar extension is additive — LAYOUT <form-factor>? { ... } can be added later without breaking LAYOUT { ... }

Current Mechanism (Form Factor in Target Config, Not DSL)

;; Target config carries form-factor
(target-config
  #:name 'qt-qml-mobile
  #:form-factor 'mobile    ; <-- form factor lives HERE, not in DSL
  ...)

When generating for a mobile target, the generator consults its target config for mobile-appropriate defaults. The DSL's LAYOUT block provides intent; the target config adapts to form factor.

Future Grammar Extension Path

When mobile/tablet generators exist and the single LAYOUT block is proven:

;; Current (v1.2.0)
LayoutBlock <- 'LAYOUT' '{' LayoutDirective+ '}'

;; Future (v2.x)
LayoutBlock <- 'LAYOUT' FormFactorQualifier? '{' LayoutDirective+ '}'
FormFactorQualifier <- 'desktop' / 'tablet' / 'mobile' / 'watch' / 'embedded'

This is a backward-compatible grammar extension (optional qualifier).

Positive Consequences

  • Simpler initial grammar — fewer rules, less parser complexity
  • Lower implementation risk — focus on proving 2 targets
  • Future-safe — grammar can be extended without breaking existing files

Negative Consequences

  • Cannot express form-factor-specific layouts in DSL until this is implemented
  • Projects targeting mobile must rely on target-config defaults, not explicit DSL intent
  • May need to be revisited sooner if a mobile frontend target lands

AI Agent Guidance

When This Decision Applies

  • When proposing form-factor-specific syntax in the DSL
  • When discussing mobile/tablet layout optimization
  • When planning mobile-specific UI generation

When This Decision Does NOT Apply

  • When implementing the single LAYOUT block (that's ADR abstract-layout-vocabulary)
  • When adding target configuration for new form factors (that's purely Racket config)

Enforcement

  • Do not add form-factor qualifiers to the LAYOUT grammar rule until this ADR is superseded
  • Form-factor selection happens in target configuration (dslgen/targets/), not in the DSL
  • If mobile/tablet generators are implemented, revisit this ADR
