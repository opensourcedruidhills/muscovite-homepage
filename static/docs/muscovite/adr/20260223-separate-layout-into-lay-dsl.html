Separate Layout and Presentation Directives into `.lay` DSL Files

Separate Layout and Presentation Directives into .lay DSL Files

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-23
  • Tags: layout, presentation, dsl, lay, domain-modeling, separation-of-concerns
  • Supersedes: Abstract Layout Vocabulary, Form-Factor Layout Selection Cascade

Technical Story: Layout and presentation directives (LAYOUT { ... }, form-factor overrides, column selection) are mixed into .ddd domain model files. This violates separation of concerns and prevents multiple presentation profiles from targeting the same domain model. The fix is a third DSL file type: .lay.

Context and Problem Statement

.ddd files currently carry three categories of content:

  1. Domain modeling — entities, aggregates, relationships, events, sagas: domain expert's concern, belongs in .ddd
  2. Identity display strings (DISPLAY_FORMAT { SHORT, FULL }) — how the domain identifies an aggregate in human-readable text: a domain claim, belongs in .ddd
  3. Layout and rendering directives (LAYOUT { ... }, TABLE: [...] column lists) — rendering decisions. A patient aggregate should not need to say "use TABS on desktop." That is a UI design concern. It does not belong in .ddd.

ADR 20260211 (Abstract Layout Vocabulary) and ADR 20260219 (Form-Factor Layout Selection) placed the right concepts in the wrong file tier. This ADR remedies that.

Additionally: multiple presentation profiles (desktop, tablet, accessibility) on one domain model cannot be expressed without duplicating aggregates or adding indirection inside .ddd.

Decision Drivers

  • Domain experts authoring .ddd should not be exposed to UI rendering concepts
  • Multiple presentation profiles on one domain model without duplicating aggregates
  • .ddd files should be parseable and meaningful with no UI target present
  • Aligns with the existing .ddd / .mus separation (domain model vs. project config)
  • Grammar-first evolution (ADR 20260211-grammar-first-language-evolution)

Considered Options

  1. New .lay DSL file type — separate file, separate grammar, separate parser, explicitly declared in .mus
  2. LAYOUT section inside .mus — reuse project config file; .ddd stays clean
  3. Keep in .ddd — reject separation (status quo)

Decision Outcome

Chosen option: Option 1 — New .lay DSL file type, because:

  • Multiple presentation profiles require separate files with separate names
  • .mus already has a defined concern (project config, capabilities, plugin wiring); layout is not project config
  • Separation is auditable: ls *.lay shows all presentation profiles
  • Follows prior art: .mus has its own grammar file muscovite-mus.peg

Grammar and Parser

  • Formal grammar: dslgen/grammar/muscovite-lay.peg (separate file — follows .mus/muscovite-mus.peg precedent; one concern, one grammar file)
  • Parser module: dslgen/parser/lay-parser.rkt
  • IR produced: lay-profile struct containing lay-aggregate-layout entries

File Discovery (Explicit)

.mus project files declare presentation profiles explicitly:

PRESENTATION_PROFILE "path/to/desktop.lay"
PRESENTATION_PROFILE "path/to/mobile.lay"

Auto-discovery was rejected: explicit declaration is version-controlled intent and enables two profiles for the same domain without ambiguity.

.lay File Syntax

LAYOUT_PROFILE "hello-world-desktop" FORM_FACTOR desktop {

    LAYOUT Todo.TodoItem {
        MASTER { TABLE: [title, due_date, is_done]; }
        DETAIL {
            STYLE: FORM;
            GROUPING: TABS;
            ACTIONS: [create, edit, delete];
            CHILDREN: [TodoComment as "Comments"];
        }
    }

    LAYOUT desktop Todo.TodoItem {
        MASTER { TABLE: [title, due_date, is_done]; }
    }

    LAYOUT mobile Todo.TodoItem {
        MASTER { TABLE: [title, is_done]; }
        DETAIL { STYLE: CARD; GROUPING: FLAT; }
    }
}

Aggregate references: CONTEXT.AggregateName qualified form required. Form-factor qualification follows ADR 20260219 semantics (selection cascade preserved).

DISPLAY_FORMAT in .ddd (Rename from PRESENTATION)

The existing PRESENTATION block is renamed DISPLAY_FORMAT. It retains only:

  • SHORT: "format string" — short text representation (combo box, breadcrumb)
  • FULL: "format string" — full text representation (detail header, notification)

TABLE: [...] is removed from .ddd and moves to .lay.

ENTITY Patient @uuid "..." IDENTIFIED_BY id (...)
DISPLAY_FORMAT {
    SHORT: "{family_name}, {given_name}";
    FULL:  "{family_name}, {given_name} (MRN: {mrn})";
}
DOC "A person receiving healthcare services";

This amends ADR 20260123 (PRESENTATION Hints for Entity Display): the concept is retained, the keyword is renamed, and TABLE is removed from its scope.

Migration Steps

  1. FTR-132 — Implement .lay grammar, parser, IR, PRESENTATION_PROFILE in .mus
  2. FTR-133 — Mechanical migration of all example .ddd files (fhir-health × 9 contexts, geosciences × 26 contexts, hello-world): extract LAYOUT blocks and TABLE column lists to .lay files; rename PRESENTATION → DISPLAY_FORMAT
  3. FTR-134 — Enable parser enforcement: LAYOUT or TABLE: inside .ddd is a hard parse error (depends on FTR-133 completing first)

Positive Consequences

  • .ddd files contain only domain modeling concerns
  • Multiple presentation profiles per domain model via multiple .lay files
  • Domain experts and UI designers can work on separate files without merge conflicts
  • Generator discovers which layouts exist per profile without re-parsing domain models

Negative Consequences

  • New parser and grammar to maintain
  • Breaking change for all .ddd files containing LAYOUT blocks
  • Three DSL file types: .ddd, .mus, .lay

Pros and Cons of the Options

Option 1: New .lay DSL file type (chosen)

  • Good, because enables multiple presentation profiles on same domain model
  • Good, because .ddd is a clean domain modeling language
  • Good, because follows the .mus precedent (one concern, one file type)
  • Bad, because adds a third DSL with its own parser and grammar

Option 2: LAYOUT section inside .mus

  • Good, because no new file type
  • Bad, because .mus has a defined concern (project config); layout is not project config
  • Bad, because one .mus cannot cleanly express two profiles for the same aggregate

Option 3: Keep in .ddd

  • Bad, because domain experts are exposed to UI rendering concepts
  • Bad, because multiple profiles require aggregate duplication
