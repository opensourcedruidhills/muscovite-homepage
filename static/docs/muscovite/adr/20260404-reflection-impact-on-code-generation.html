Reflection Impact on Code Generation Architecture

Reflection Impact on Code Generation Architecture

  • Status: proposed
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04
  • Tags: c++26, reflection, generators, architecture

Technical Story: Muscovite roadmap v3.2+ — evaluating whether C++26 reflection can replace portions of generated boilerplate with generic framework code.

Context and Problem Statement

Muscovite generates substantial per-entity C++ boilerplate: type converters (struct ↔︎ pqxx row), protobuf serializers (struct ↔︎ proto message), Qt model column bindings, and repository CRUD methods. This code is repetitive — each entity gets the same structural pattern with different field names and types.

C++26 reflection enables compile-time introspection of struct members. In principle, a single generic function like reflect_to_row<T>() could replace per-entity generated type converters.

Decision Drivers

  • Less generated code = fewer generator bugs, less output to review
  • Reflection-based code lives in the framework (Layer 1), reducing Layer 2 surface area
  • But: reflection operates on C++ types, not DSL semantics — some mappings require DSL knowledge that C++ types alone cannot express
  • Compile-time reflection can increase build times significantly
  • The feature is brand new — ecosystem tooling is immature

Considered Options

  1. Full reflection adoption — Replace all per-entity boilerplate with generic code
  2. Selective reflection — Replace type converters and serializers; keep repository generation
  3. Annotation-driven reflection — Generate annotated structs; framework reflects on annotations
  4. Defer — Wait for C++29 when reflection ecosystem matures

Decision Outcome

Chosen option: TBD — requires prototyping. Leaning toward option 3 (annotation-driven).

What Stays Generated (even with reflection)

Artifact Why
C++ entity structs Struct layout IS the domain model projection
.proto files Protobuf has its own schema
SQL/Sqitch migrations Database schema is a separate target
CMakeLists.txt Build system is not reflectable
QML components QML is a separate language

What Could Become Generic (reflection candidates)

Current generated code Reflection replacement
type_converters.cpp (struct ↔︎ pqxx row) reflect_to_row<T>()
Protobuf ↔︎ struct mappers reflect_to_proto<T, ProtoT>()
Entity comparison operators reflect_equal<T>()
JSON serialization (if added) reflect_to_json<T>()

Positive Consequences

  • Dramatic reduction in generated C++ code volume
  • Bug fixes in mapping logic apply to all entities immediately

Negative Consequences

  • Compile time may increase significantly
  • Debugging reflection-based code is harder than generated code
  • Risk: reflection API may evolve in C++29

AI Agent Guidance

When This Decision Applies

  • When designing new per-entity generated code patterns
  • When prototyping C++26 reflection features

When This Decision Does NOT Apply

  • v2.x development (no reflection in C++23)
  • Non-C++ targets (SQL, proto, QML, CMake)

Enforcement

  • Prototyping results documented before ADR acceptance
  • Framework reflection code lives in framework/reflection/ (new module)
