Forbid PostgreSQL ENUM Types - Use Lookup Tables

Forbid PostgreSQL ENUM Types - Use Lookup Tables

Date: 2026-01-08
Status: Accepted
Deciders: System Architect, Database Team
Tags: postgresql, data-modeling, extensibility, schema-evolution

Context and Problem Statement

PostgreSQL ENUM types provide compile-time type safety for categorical data, but they create significant challenges for schema evolution in production systems. How should we represent categorical data (sample types, confidence levels, statuses) in a way that supports business evolution without database downtime or migration complexity?

Decision Drivers

  • Schema Evolution: Business requirements change frequently; new categories emerge, old ones are retired
  • Soft Delete: Need to mark values as inactive without breaking referential integrity
  • Migration Complexity: ALTER TYPE operations require exclusive locks and can break function signatures
  • Audit Trail: Need to track when values were added, retired, and who made changes
  • Documentation: Values need descriptions, display order, and business context
  • Cross-Context Consistency: Shared lookup values must be accessible across bounded contexts

Considered Options

Option 1: PostgreSQL ENUM Types (REJECTED)

CREATE TYPE sample_status AS ENUM ('pending', 'analyzed', 'archived');

Pros:

  • Type safety at database level
  • Compact storage (4 bytes)
  • Fast comparisons

Cons:

  • Cannot drop values (requires DROP TYPE + recreate)
  • Breaking changes: Dropping ENUM type invalidates function signatures
  • No soft delete: Cannot mark values as retired without migration
  • No metadata: No description, display order, or business context
  • No audit trail: Cannot track who added/retired values
  • Global lock: ALTER TYPE requires exclusive lock on type

Option 2: CHECK Constraints on TEXT (REJECTED)

CREATE TABLE sample (
    status TEXT NOT NULL CHECK (status IN ('pending', 'analyzed', 'archived'))
);

Pros:

  • Simple implementation
  • No migration overhead

Cons:

  • Constraint modification requires table lock
  • No centralized value management
  • No metadata or audit trail
  • Duplicate constraint definitions across tables
  • Cannot share values across contexts

Option 3: Lookup Tables in Basedata Context (ACCEPTED)

CREATE TABLE basedata.confidence_level (
    id           UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    code         TEXT NOT NULL UNIQUE,
    name         TEXT NOT NULL,
    description  TEXT,
    display_order INTEGER,
    is_active    BOOLEAN NOT NULL DEFAULT true,
    created_at   TIMESTAMPTZ NOT NULL DEFAULT now(),
    created_by   TEXT
);

CREATE TABLE geologic_mapping.geologic_map_feature (
    confidence_id UUID NOT NULL REFERENCES basedata.confidence_level(id)
);

Pros:

  • Soft delete: Set is_active = false instead of dropping values
  • Extensible: Add new values without schema changes (INSERT only)
  • Rich metadata: Description, display order, business context
  • Audit trail: Track creation, retirement, and ownership
  • No locks: Adding values is a simple INSERT (no DDL)
  • Cross-context sharing: Basedata context provides canonical values
  • Backward compatible: Retired values remain for historical data
  • UI-friendly: Descriptions and display order support ComboBox rendering

Cons:

  • Slightly larger storage (16 bytes UUID vs 4 bytes ENUM)
  • No compile-time type checking (runtime FK validation instead)
  • More complex queries (JOIN required for display name)

Decision Outcome

Chosen Option: Lookup tables in Basedata context (Option 3)

All categorical data MUST be modeled as lookup tables with this structure:

CONTEXT Basedata {
    ENTITY ConfidenceLevel IDENTIFIED_BY id (
        id            UUID        NOT NULL,
        code          TEXT        NOT NULL @unique,
        name          TEXT        NOT NULL,
        description   TEXT,
        display_order INTEGER,
        is_active     BOOLEAN     NOT NULL DEFAULT TRUE,
        created_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
        created_by    TEXT
    )
    DOC "Observation confidence levels for geologic mapping";
    
    AGGREGATE ConfidenceLevel ROOT ConfidenceLevel;
    
    IMPORT_JSON "../../seeds/basedata/confidence_level.json" INTO ConfidenceLevel;
}

Standard Lookup Table Pattern

All lookup tables follow this pattern:

Column Type Purpose
id UUID Primary key (referenced by domain entities)
code TEXT UNIQUE Short identifier for API/export (e.g., "CERTAIN", "APPROX")
name TEXT Display name for UI (e.g., "Certain", "Approximate")
description TEXT Business definition and usage guidance
display_order INTEGER Sort order for ComboBox/dropdown (optional)
is_active BOOLEAN Soft delete flag (retired values set to false)
created_at TIMESTAMPTZ Audit trail - when added
created_by TEXT Audit trail - who added

Implementation Rules

  1. Generator Enforcement: DSL generator MUST reject ENUM declarations with compilation error
  2. Basedata Context: All shared lookup tables MUST be in Basedata context
  3. Seed Data Required: Every lookup table MUST have JSON seed file in seeds/basedata/
  4. Foreign Keys: Domain entities reference lookup tables via <lookup>_id UUID NOT NULL REFERENCES basedata.<lookup>(id)
  5. API Layer: Select functions MUST include lookup table JOINs to return code and name fields
  6. Qt Client: ComboBox widgets MUST query SELECT id, name FROM basedata.<lookup> WHERE is_active = true ORDER BY display_order

Lookup Table Location Used By
ConfidenceLevel Basedata GeologicMapping (map feature confidence)

Validation Rules

When domain-specific validation is critical, use CHECK constraints in addition to FK:

CREATE TABLE geologic_mapping.geologic_map_feature (
    confidence_id UUID NOT NULL 
        REFERENCES basedata.confidence_level(id),
    -- Additional CHECK if business rules require specific values
    CHECK (EXISTS (
        SELECT 1 FROM basedata.confidence_level 
        WHERE id = confidence_id 
        AND code IN ('CERTAIN', 'APPROX', 'INFERRED')
    ))
);

Use sparingly - most validation should be at application layer.

Consequences

Positive

  • Zero-downtime value additions: INSERT INTO basedata.confidence_level requires no DDL
  • Graceful retirement: Set is_active = false preserves historical data integrity
  • Rich business context: Descriptions document usage and semantics
  • Audit compliance: Track when/who added or retired values
  • UI consistency: Standardized ComboBox queries across all Qt forms
  • Cross-context sharing: Basedata provides single source of truth
  • Migration safety: No function signature invalidation

Negative

  • Storage overhead: 16 bytes (UUID) vs 4 bytes (ENUM) per reference
  • Query complexity: JOINs required for display names in reports
  • Runtime validation: FK checks happen at INSERT time, not compile time
  • Developer discipline: Must remember to check is_active flag in queries

Migration Path for Existing ENUMs

If ENUMs already exist in the database:

  1. Create lookup table with values from ENUM
  2. Add <lookup>_id column to referencing tables
  3. Populate via: UPDATE table SET <lookup>_id = (SELECT id FROM lookup WHERE code = old_enum_column::text)
  4. Add FK constraint on <lookup>_id
  5. Drop old ENUM column
  6. Drop ENUM type

AI Agent Guidance

When This Decision Applies

  • Always: All categorical data (statuses, types, categories, classifications)
  • Always: Any data that might need new values added in production
  • Always: Data that needs soft delete (retirement without data loss)

When This Decision Does NOT Apply

  • Never (PostgreSQL ENUMs are forbidden project-wide)

How It's Enforced

  1. DSL Parser (src/dslgen/parser/parser.rkt): Rejects ENUM keyword with error
  2. Code Review: All .ddd files checked for lookup table pattern compliance
  3. Generator Validation: Sqitch generator verifies all categorical types are lookup tables

Key Files Affected

  • src/dslgen/parser/parser.rkt - Remove/reject ENUM parsing
  • src/dslgen/generators/postgres/sqitch-unified.rkt - Validate lookup table references
  • src/examples/geosciences/basedata.ddd - Canonical lookup table definitions
  • seeds/basedata/*.json - Seed data for all lookup tables

Common Patterns

Creating a Lookup Table:

CONTEXT Basedata {
    ENTITY MyLookup IDENTIFIED_BY id (
        id UUID NOT NULL,
        code TEXT NOT NULL @unique,
        name TEXT NOT NULL,
        description TEXT,
        is_active BOOLEAN NOT NULL DEFAULT TRUE
    );
    AGGREGATE MyLookup ROOT MyLookup;
    IMPORT_JSON "../../seeds/basedata/my_lookup.json" INTO MyLookup;
}

Referencing from Domain Entity:

CONTEXT MyDomain {
    ENTITY MyEntity IDENTIFIED_BY id (
        id UUID NOT NULL,
        lookup_id UUID NOT NULL
    )
    RELATIONSHIPS {
        lookup -> MyLookup REQUIRED ONE;
    }
}

Seed Data Format (seeds/basedata/my_lookup.json):

[
  {
    "code": "VALUE1",
    "name": "Value One",
    "description": "First value in the lookup",
    "display_order": 1,
    "is_active": true
  },
  {
    "code": "VALUE2",
    "name": "Value Two",
    "description": "Second value in the lookup",
    "display_order": 2,
    "is_active": true
  }
]

Notes

This decision aligns with DDD principles: lookup tables are lightweight aggregate roots in the Basedata bounded context, providing a published language for categorical data across all domain contexts.

The is_active pattern supports Temporal Modeling: retired values remain in the database for historical queries but are filtered out of current operations. This is critical for compliance and audit requirements in geoscience data management.
