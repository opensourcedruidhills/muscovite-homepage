QUDT-Driven Unit Standard Library

QUDT-Driven Unit Standard Library

  • Status: proposed
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04

Technical Story: IDEA-134 — DSL-Native Units of Measure, Decision 1

Context and Problem Statement

Muscovite needs a standard library of unit and dimension definitions that projects can import (IMPORT "muscovite://units/si"). This library must cover the 7 SI base dimensions, all SI prefixes, 22+ named derived dimensions, and domain-specific packs (medical, geoscience, engineering). The question is how to author and maintain these definitions: hand-code them from ISO 80000 tables, or generate them from a machine-readable ontology?

Decision Drivers

  • Completeness: 2500+ units across SI, medical, and geoscience domains
  • Correctness: conversion factors must be exact — errors propagate to all generated code
  • Maintainability: new units and corrections from upstream must be incorporable
  • Auditability: every unit must trace to an authoritative source (ISO 80000, UCUM, QUDT)
  • Parse performance: the DSL compiler must load unit libraries quickly
  • Licensing: data source must be compatible with AGPL-3.0 + Commercial dual license

Considered Options

  1. QUDT-generated with manual curation — Parse QUDT RDF/Turtle at build time, generate .ddd standard library files, review and commit
  2. Hand-coded standard library — Manually transcribe units from ISO 80000 tables and UCUM specification into .ddd files

Decision Outcome

Chosen option: "QUDT-generated with manual curation" (option 1), because it provides 2500+ units with machine-verified conversion factors while the curation step ensures only relevant units ship in each domain pack.

The workflow is: generate → review → commit. The generated .ddd files are checked into the repository (not generated at user build time). This means:

  • Users never need QUDT tooling or RDF parsers
  • The curation step filters irrelevant units (e.g., obscure CGS units excluded from SI core)
  • Conversion factors are auditable in plain-text .ddd files
  • Updates are explicit: re-run generator, diff, review, commit

Positive Consequences

  • 2500+ units available from day one, with correct conversion factors
  • Automatic updates when QUDT releases new versions (re-run generator, diff, commit)
  • Every unit traces to a QUDT URI → ISO 80000 / UCUM source
  • Domain packs (medical, geoscience) generated from curated QUDT subsets
  • Licensing is clean: QUDT is CC BY 4.0, requires only attribution in docs

Negative Consequences

  • Requires a QUDT-to-DDD generator tool (one-time build cost, written in Racket or Python)
  • Generated .ddd files may be large (SI core + prefixes ≈ 500 lines)
  • Manual curation step adds friction to updates (by design — prevents accidental unit additions)
  • RDF/Turtle parsing dependency in the generator tool (not in the DSL compiler itself)

Pros and Cons of the Options

Option 1: QUDT-Generated with Manual Curation

Parse QUDT RDF/Turtle data, emit .ddd unit definitions, human reviews and commits.

  • Good, because 2500+ units with machine-verified conversion factors
  • Good, because automatic updates when QUDT publishes new versions
  • Good, because every unit has a traceable URI to ISO 80000 / UCUM
  • Good, because CC BY 4.0 license — compatible with AGPL + Commercial
  • Good, because curation step prevents irrelevant units from bloating the library
  • Bad, because requires building a QUDT-to-DDD generator tool
  • Bad, because RDF/Turtle parsing adds complexity to the build toolchain
  • Bad, because generated files may include units nobody uses (mitigated by curation)

Option 2: Hand-Coded Standard Library

Manually author .ddd files from ISO 80000 tables and UCUM specification.

  • Good, because simpler toolchain — no RDF parser needed
  • Good, because fully curated from day one — only relevant units included
  • Good, because faster to parse (smaller files)
  • Good, because no external data dependency
  • Bad, because error-prone — manual transcription of conversion factors
  • Bad, because incomplete — unlikely to cover 2500+ units
  • Bad, because maintenance burden — every new unit is manual work
  • Bad, because no traceable link to authoritative source unless manually documented
  • Bad, because domain packs (medical, geoscience) require domain expertise to curate

AI Agent Guidance

When This Decision Applies

  • When creating or modifying the units standard library files (muscovite://units/*.ddd)
  • When adding new domain packs (medical, geoscience, engineering)
  • When updating conversion factors or adding new units to existing packs
  • When building the QUDT-to-DDD generator tool

When This Decision Does NOT Apply

  • When using units in project .ddd files (those just IMPORT the standard library)
  • When modifying generators that consume QUANTITY fields (see other ADRs in this series)
  • When working on the DSL compiler's unit resolution logic (that reads the .ddd files)

Enforcement

  • The QUDT-to-DDD generator tool produces .ddd files; manual edits to generated files are overwritten on next generation run
  • Code review ensures curation step is performed (no blind commit of generated output)
  • Attribution to QUDT (CC BY 4.0) must appear in generated file headers and in LICENSE
