ADR: Release Cadence Deceleration — Stabilization Phase

ADR: Release Cadence Deceleration — Stabilization Phase

  • Status: accepted
  • Date: 2026-03-09
  • Deciders: Johannes Lochmann

Context

Current State

Between v1.35.0 (early February 2026) and v1.60.0 (March 9, 2026), Muscovite shipped 26 releases in approximately 5 weeks — averaging roughly one release per working day. This velocity was appropriate during rapid feature buildout when the priority was establishing the core generator pipeline, database conventions, and multi-target code generation.

Release Ceremony Cost

The release ceremony has grown substantially as quality gates were added (ADR 20260307):

Gate Approximate Duration Added In
validate-release.sh 30s v1.55.0
Racket full test suite 5–10 min v1.55.0
C++ build (3 scopes) 3–5 min v1.55.0
C++ tests (ctest) 1–2 min v1.55.0
Database deploy 1–2 min v1.55.0
clang-tidy 2–4 min v1.58.0
Code coverage collection 3–5 min v1.60.0
Build profiling + metrics 1–2 min v1.60.0
Full code generation 2–3 min v1.60.0
Documentation publishing 1–2 min v1.60.0
Subtree push (muscovite-cpp) 5–15 min v1.55.0
Submodule updates 2–3 min v1.55.0
Metrics collection + trending 1–2 min v1.60.0
Total ceremony ~30–55 min

At daily release cadence, this means 30–55 minutes per day spent on release ceremony alone — a significant fraction of productive time. The ratio of feature development time to QA/release overhead is deteriorating.

Feature Base Maturity

The product has reached a strong feature base:

  • 8 generation targets (PostgreSQL, Sqitch, gRPC, C++ DBA, C++ DBA tests, Qt Widgets, QML, CMake/Conan)
  • Expression AST with collection predicates (v1.60.0)
  • Behavioral Sovereignty roadmap defined (v1.61.0–v1.63.0)
  • Documentation pipeline automated
  • Metrics and trending operational
  • 191 ADRs documenting all architectural decisions

The priority is shifting from "ship features fast" to "ship features correctly." Stability, correctness, and documentation quality matter more than velocity.

The Algebra

For N features requiring F hours of development and a release ceremony costing C hours:

  • Daily releases, 1 feature each: N × (F + C) total hours
  • Bi-weekly releases, 5 features each: ⌈N/5⌉ × (5F + C) = NF + ⌈N/5⌉C total hours
  • Savings: (N - ⌈N/5⌉) × C = ~80% reduction in ceremony overhead

With C ≈ 0.75 hours and N = 20 features per quarter:

  • Daily: 20 × 0.75 = 15 hours of ceremony
  • Bi-weekly: 4 × 0.75 = 3 hours of ceremony
  • 12 hours saved per quarter — roughly 1.5 full working days reclaimed for features

Decision

Shift from near-daily releases to bi-weekly (or larger) releases, effective after v1.60.0.

New Cadence Rules

Rule Description
Minimum iteration size 4+ FTRs per release (exceptions: hotfixes, security patches)
Target cadence Bi-weekly releases (~every 10 working days)
Maximum cadence No more than one release per week unless critical fix
Hotfix path Patch releases (vX.Y.Z, Z>0) permitted anytime for regressions or security
RC practice For iterations with ≥8 FTRs, tag an RC (vX.Y.0-rc1) and soak for 48h

Iteration Sizing Guidelines

Iteration Size FTRs Appropriate For
Small 4–6 Focused infrastructure or tooling iterations
Medium 7–10 Standard feature iterations (Behavioral Sovereignty tiers)
Large 11–15 Cross-cutting iterations with parallel agent tracks
Extra-large 16+ Discouraged — split into two iterations

Quality Gate Efficiency

The ceremony cost is justified — removing gates is not an option. Instead:

  1. Per-PR gates via agent-pr.sh catch most regressions at merge time, meaning the release ceremony should be a confirmation, not a discovery.
  2. Parallel gate execution (future): run Racket tests, C++ build, and clang-tidy concurrently. Potential savings: ~10 minutes.
  3. Incremental subtree push (future): git subtree push rewrites full history (~10 min at 1,100 commits). CI-driven sync would eliminate this.

What This Changes for Agents

Before After
Agent completes 1–2 FTRs → release Agent completes FTRs → merge to develop → wait for iteration
Pressure to release each FTR individually FTRs accumulate on develop, tested continuously
Release ceremony per 1–2 features Release ceremony per 5–10 features
develop and main nearly identical develop is the working branch, main is release-only (as designed)

Branch Strategy Alignment

This decision reinforces ADR 20260225 (branching model):

  • develop becomes the true integration branch where features soak together
  • main is release-only — updated bi-weekly, always production-quality
  • Feature branches merge to develop via agent-pr.sh as before
  • The release branch (release/vX.Y.0) is cut from develop when the iteration is complete and all gates pass

Version Number Semantics

With fewer, larger releases, version numbers gain more meaning:

Component Meaning
Major (X) Breaking DSL changes (reserved, currently 1.x)
Minor (Y) Feature iteration — now represents 2 weeks of work, not 1 day
Patch (Z) Hotfix for regressions in the corresponding minor release

The gap from v1.55.x to v1.58.0 (intentional renumbering for stabilization sprints) becomes unnecessary — larger iterations naturally provide stabilization time.

Consequences

Positive

  • Feature/QA ratio improves: Ceremony cost amortized over 5–10 features instead of 1–2
  • Higher release quality: Features soak together on develop before release
  • Meaningful versions: Each minor version represents a substantial capability increment
  • Reduced context switching: Agents focus on implementation, not ceremony
  • Better changelogs: Each release tells a coherent story (e.g., "Behavioral Sovereignty Tier 2")
  • RC practice: Large iterations get a soak period, catching integration issues

Negative

  • Longer feedback cycle: Bugs in merged features wait for the next release (mitigated: develop is the working branch, issues found there are fixed immediately)
  • Larger diffs per release: More changes to review in release notes (mitigated: each FTR is reviewed at PR time; release is confirmation)
  • Temptation to slip scope: "While we're waiting for the release, add one more feature" (mitigated: iteration scope locked at planning time per existing governance)

Neutral

  • All quality gates remain mandatory — no gates removed or weakened
  • agent-pr.sh continues to enforce per-PR quality gates
  • Hotfix path remains available for critical regressions
  • The v1.61.0–v1.63.0 Behavioral Sovereignty iterations already follow this pattern naturally (9, 4, and 4 FTRs respectively)

Review Trigger

Re-evaluate this decision if:

  • Release ceremony drops below 15 minutes (parallel gates, CI subtree sync)
  • Team size grows beyond 3 concurrent agents (may need faster integration feedback)
  • A critical regression in a released version is traced to features soaking too long on develop
