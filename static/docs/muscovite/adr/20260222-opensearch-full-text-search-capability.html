OpenSearch Full-Text Search Capability

OpenSearch Full-Text Search Capability

  • Status: accepted
  • Date: 2026-02-22
  • Tags: search, opensearch, capability, nats, domain-events, full-text, fhir
  • Deciders: Johannes Lochmann

Technical Story: Muscovite-generated applications need cross-entity, cross-context full-text and typed-filter search. PostgreSQL tsvector/tsquery is insufficient for production-grade FHIR search parameters, faceting, and typo tolerance. An external search engine, populated via the existing NATS/Outbox domain event pipeline, provides a scalable, eventually-consistent search index.

Context and Problem Statement

Domain applications need to search across all entity types with:

  • Full-text search with relevance ranking and highlighting
  • Typed filters: date ranges, numeric comparators, boolean, enum/facet
  • Cross-context search (e.g., "find everything related to patient John Doe")
  • FHIR-grade search (1,400+ search parameters, chained searches, composite parameters, :contains, :exact, :above/:below for hierarchical codes)
  • Sub-second response times independent of OLTP database load
  • Faceted results grouped by entity type / bounded context

PostgreSQL full-text search (tsvector, GIN indexes) is:

  • Coupled to the transactional database (search load competes with writes)
  • Limited in relevance ranking and typo tolerance
  • No native faceting or aggregation on search results
  • Cannot handle FHIR's complex search parameter semantics

The existing NATS JetStream + Outbox Publisher infrastructure (ADR 20260202) already delivers domain events reliably to consumers. A search indexer is a natural new consumer.

Decision Drivers

  • FHIR Health example has 9 bounded contexts with complex search requirements
  • Geosciences example has 26 bounded contexts — cross-context search is essential for field geologists
  • Existing NATS + Outbox infrastructure provides reliable event delivery to new consumers at zero additional transport cost
  • Must be Apache 2.0 compatible (no SSPL/proprietary licensing conflicts with AGPL-3.0 + Commercial dual license)
  • Must run in Docker alongside existing PostgreSQL + NATS services
  • Must integrate with existing libcurl + nlohmann_json dependencies (no new Conan packages)
  • Must fit the capability-gating pattern (ADR 20260212): opt-in via CAPABILITIES { search } in .mus

Considered Options

  1. OpenSearch — Apache 2.0 fork of Elasticsearch, Linux Foundation governed
  2. Elasticsearch — SSPL + Elastic License 2.0 (proprietary since 7.11)
  3. Meilisearch — Rust-based, MIT licensed, UI-focused search
  4. Typesense — C++-based, GPL-3.0, fast typed search
  5. PostgreSQL FTS — Built-in tsvector/tsquery with GIN indexes
  6. Zinc/ZincSearch — Go-based, ES-compatible lightweight search

Decision Outcome

Chosen option: Option 1 — OpenSearch, because:

  1. License: Apache 2.0 — no interaction with AGPL-3.0, no risk for commercial licensees deploying generated code. SSPL (Elasticsearch) creates unpredictable licensing for code-generated infrastructure.
  2. Feature completeness: Same Lucene core as Elasticsearch — identical JSON query DSL, full aggregations, nested queries, date/numeric range filters, custom analyzers. Handles all FHIR search parameter types.
  3. Docker simplicity: No enrollment ceremony (unlike ES 8.x). Single-node dev mode with one environment variable.
  4. Free security + alerting: Built-in, no paywall (ES paywalls cross-cluster replication, some ML features).
  5. Community: Linux Foundation governance, multi-vendor (AWS, SAP, Red Hat).
  6. Query DSL portability: Generated query JSON works unchanged if someone swaps to Elasticsearch — the APIs are compatible at the REST level.

Negative Consequences

  • Additional infrastructure service (Docker container, ~512MB memory for dev)
  • Eventual consistency — search results lag domain events by outbox poll interval (default 100ms)
  • Full reindex required on mapping changes (automated via CMake target)
  • C++ client is a thin libcurl wrapper, not an official SDK (but same situation for all search engines)

Detailed Decisions

1. Capability: SEARCH in Project DSL

New capability in .mus project configuration:

CAPABILITIES {
    search
}

When search capability is enabled:

  • Docker Compose includes OpenSearch service
  • Build system links muscovite_search library
  • Server startup initializes SearchIndexer NATS consumer
  • gRPC GlobalSearchService is generated
  • Qt clients include search bar widget

When search is not enabled: zero search-related code generated. No OpenSearch dependency.

2. DSL Annotation: @searchable on Fields

Field-level annotation marking which fields are indexed for search:

ENTITY Patient @uuid "..." IDENTIFIED_BY id (
    id          UUID        NOT_NULL @uuid "...",
    mrn         TEXT        NOT_NULL @unique @searchable @uuid "...",
    given_name  TEXT        NOT_NULL @searchable @uuid "...",
    family_name TEXT        NOT_NULL @searchable @uuid "...",
    birth_date  DATE        @searchable @uuid "...",
    active      BOOLEAN     NOT_NULL @uuid "..."
)

Entity-level @searchable shorthand (all TEXT fields): evaluated during implementation.

3. Search Field Type Derivation

The generator derives OpenSearch mapping types from the existing type registry — no new type mappings:

DSL Type OpenSearch Mapping Search Operators
TEXT, String, Email, Phone, URL text + keyword (multi-field) full-text, exact, prefix, wildcard
INTEGER, BIGINT long / integer =, !=, <, <=, >, >=, range
Decimal, NUMERIC double =, !=, <, <=, >, >=, range
DATE date (format: yyyy-MM-dd) =, !=, <, <=, >, >=, range
TIMESTAMPTZ date (format: strict_date_optional_time) =, !=, <, <=, >, >=, range
BOOLEAN boolean =, !=
UUID (FK to lookup) keyword + denormalized text display exact match, facet

4. Framework Module: muscovite::search

New framework library at framework/search/:

framework/search/
├── CMakeLists.txt
├── include/muscovite/search/
│   ├── opensearch_client.hpp    # libcurl + nlohmann_json REST client
│   ├── search_indexer.hpp       # NATS consumer → OpenSearch upsert/delete
│   ├── search_query.hpp         # Type-safe query builder
│   ├── search_result.hpp        # Typed result (context, entity_type, entity_id, score, highlights)
│   ├── search_config.hpp        # Connection config
│   └── search_reindexer.hpp     # Bulk reindex from PostgreSQL
└── src/
    ├── opensearch_client.cpp
    ├── search_indexer.cpp
    ├── search_query.cpp
    └── search_reindexer.cpp

Dependencies: CURL::libcurl (already linked in oauth2), nlohmann_json::nlohmann_json (already in Conan). Zero new Conan packages.

5. C++ Client: libcurl + nlohmann_json

No official C++ OpenSearch SDK exists. The client wraps 6 REST operations:

Operation HTTP Endpoint
Create index PUT /{index} JSON mapping
Index document PUT /{index}/_doc/{id} JSON document
Delete document DELETE /{index}/_doc/{id} —
Search POST /{index}/_search JSON query
Multi-search POST /{index1},{index2}/_search JSON query
Bulk index POST /_bulk NDJSON

This follows the same pattern as authorization_code_flow.cpp in framework/grpc/src/oauth2/.

6. Event-Driven Index Population

Search indexes are populated and maintained via the existing Outbox → NATS pipeline:

CommandService → domain_events table → OutboxPublisher → NATS JetStream
                                                              │
                                                    SearchIndexer (new consumer)
                                                              │
                                                    OpenSearch REST API

Event handling:

  • *Created / *Updated → Fetch full entity from QueryRepository, map to JSON document, upsert
  • *Deleted → Delete document by entity ID

The SearchIndexer extends the NatsEventConsumer pattern already proven in the framework.

7. Index Naming Convention

One index per entity within a context: {context}_{entity} (snake_case).

Context Entity Index Name
patient Patient patient_patient
observation Observation observation_observation
todo TodoItem todo_todo_item

Cross-context search targets multiple indexes: POST /patient_patient,observation_observation/_search.

8. Lookup Table Denormalization

When a @searchable field is a FK to a lookup table, the search document includes the resolved display value:

{
    "id": "...",
    "title": "Buy groceries",
    "priority_id": "uuid-of-high",
    "priority_display": "High",
    "status_id": "uuid-of-active",
    "status_display": "Active"
}

This enables full-text search on display values and faceting on lookup codes without runtime joins.

9. gRPC SearchService (Generated)

service GlobalSearchService {
    rpc Search(SearchRequest) returns (SearchResponse);
    rpc SearchInContext(ContextSearchRequest) returns (SearchResponse);
    rpc Suggest(SuggestRequest) returns (SuggestResponse);
}

Generated per project when search capability is active. The service delegates to the framework OpenSearchClient.

10. Docker Infrastructure

OpenSearch added to docker/docker-compose.yml, gated by capability:

opensearch:
  image: opensearchproject/opensearch:2.19.0
  container_name: muscovite-opensearch
  environment:
    - discovery.type=single-node
    - OPENSEARCH_INITIAL_ADMIN_PASSWORD=muscovite-dev-Admin1!
    - DISABLE_SECURITY_PLUGIN=true
  ports:
    - "9200:9200"
  volumes:
    - muscovite-searchdata:/usr/share/opensearch/data

CMake targets:

  • search-reindex-{project} — full reindex from PostgreSQL
  • search-status — check index health and document counts

11. Qt Client Integration

Qt Widgets: GlobalSearchWidget — QLineEdit + filterable QTableView overlay. Debounced search-as-you-type (150ms). Click result → navigate to entity edit dialog. Uses PRESENTATION SHORT format for display.

QML: SearchBar component + SearchResultsModel. Q_PROPERTY(QString query ...) bound to TextField. Click → StackView navigation to entity detail.

12. Display Text from PRESENTATION

Search results use the entity's PRESENTATION SHORT format string for human-readable display text. The document mapper resolves format placeholders at index time.

13. Grammar-First and Test-First Governance

Per ADR 20260211 (Grammar-First) and ADR 20260212 (doctest + TDD):

  1. @searchable annotation → PEG grammar edit first (dslgen/grammar/muscovite.peg)
  2. SEARCH capability → PEG grammar edit in project grammar section
  3. Parser changes only after grammar is updated
  4. Framework muscovite::search — doctest BDD tests written before implementation
  5. Generator tests — rackunit tests with expected output before generator code
  6. Grammar conformance test must pass after changes

Pros and Cons of the Options

Option 1: OpenSearch (chosen)

  • Good: Apache 2.0 — safe for AGPL + Commercial dual licensing
  • Good: Same Lucene/query DSL as Elasticsearch — full feature parity for our use case
  • Good: Free security, alerting, cross-cluster replication
  • Good: Docker-friendly, no enrollment ceremony
  • Good: Linux Foundation governance, multi-vendor community
  • Bad: Slightly behind Elasticsearch on cutting-edge features (ESQL, ELSER)
  • Bad: Additional infrastructure service

Option 2: Elasticsearch

  • Good: Industry standard, largest ecosystem
  • Good: Slightly ahead on ML/vector search features
  • Bad: SSPL license — incompatible with code generation model. Commercial licensees deploying generated infrastructure could need Elastic commercial license
  • Bad: ES 8.x enrollment ceremony complicates Docker dev setup
  • Bad: Some features paywalled (CCR, ML)

Option 3: Meilisearch

  • Good: Simple, fast, typo-tolerant, great UX for search-as-you-type
  • Good: MIT licensed
  • Bad: No complex query DSL — cannot handle FHIR chained/composite search parameters
  • Bad: Limited aggregation/faceting compared to OpenSearch
  • Bad: No nested object queries
  • Bad: Not production-grade for enterprise search workloads

Option 4: Typesense

  • Good: C++ based, fast, good typed filters
  • Good: Simpler API than OpenSearch
  • Bad: GPL-3.0 (not Apache 2.0 — interaction with AGPL needs legal review)
  • Bad: Smaller ecosystem, fewer analyzers
  • Bad: Cannot handle FHIR's complex search requirements

Option 5: PostgreSQL FTS

  • Good: Zero additional infrastructure
  • Good: Transactionally consistent
  • Bad: Search load competes with OLTP writes
  • Bad: No typo tolerance, limited relevance ranking
  • Bad: No faceting/aggregation
  • Bad: Cannot scale search independently of database
  • Bad: Insufficient for FHIR search parameter complexity

Option 6: Zinc/ZincSearch

  • Good: Lightweight, ES-compatible API
  • Good: Apache 2.0
  • Bad: Less mature, limited feature set
  • Bad: Single-developer project risk
  • Bad: Missing advanced analyzers and aggregations

AI Agent Guidance

When This Decision Applies

  • When adding @searchable annotations to DSL fields
  • When implementing SEARCH capability in .mus project configuration
  • When modifying framework search components in framework/search/
  • When generating gRPC GlobalSearchService or Qt search UI components
  • When extending Docker infrastructure with OpenSearch service

When This Decision Does NOT Apply

  • PostgreSQL-internal queries (specifications, projections) — these use SQL directly
  • Spatial queries (@spatial-queryable) — these use PostGIS, not OpenSearch
  • Domain event persistence — governed by Outbox Pattern ADR
