Use grpc++ with Qt Wrapper Layer for Frontend Communication

Use grpc++ with Qt Wrapper Layer for Frontend Communication

  • Status: accepted
  • Deciders: jolo
  • Date: 2025-12-23
  • Tags: qt, grpc, frontend, architecture

Technical Story: Implementing Qt-based frontend that communicates with generated gRPC backend services.

Context and Problem Statement

Muscovite generates gRPC services for domain entities. We need a Qt-based frontend that can communicate with these services. The question is whether to use Qt's native gRPC implementation (QtGrpc, Qt 6.5+) or the standard grpc++ library with a Qt wrapper layer.

Decision Drivers

  • OAuth2 authentication support with full control over token injection
  • Compatibility with both Qt 5.15 LTS and Qt 6.x
  • Type conversion between Proto types and Qt types (QString, QDateTime, QUuid)
  • Signal/slot integration for async operations
  • Support for both QtWidgets and QML UI technologies
  • Maturity and community support of the solution

Considered Options

  1. QtGrpc (Qt 6.5+) - Native Qt gRPC module
  2. grpc++ with Qt Wrapper - Standard gRPC with custom Qt integration layer
  3. REST API Layer - Generate REST endpoints instead of using gRPC directly

Decision Outcome

Chosen option: "grpc++ with Qt Wrapper", because it provides full control over authentication interceptors, works with Qt 5.15+, has mature documentation, and allows us to build a clean separation between transport and Qt types.

Positive Consequences

  • Full control over OAuth2 interceptor implementation
  • Works with Qt 5.15 LTS (broader enterprise adoption)
  • Mature, well-documented gRPC library
  • Clean type conversion layer we control
  • Can optimize conversion for our specific types

Negative Consequences

  • More boilerplate code for Qt integration
  • Manual async pattern integration with Qt event loop
  • Must maintain type converter implementations

Pros and Cons of the Options

QtGrpc (Qt 6.5+)

  • Good, because native Qt integration with signals/slots
  • Good, because generates Qt-native types automatically
  • Good, because seamless QML integration via QML_ELEMENT
  • Bad, because requires Qt 6.5+ (newer dependency)
  • Bad, because less community adoption and documentation
  • Bad, because no direct OAuth2 interceptor support

grpc++ with Qt Wrapper

  • Good, because mature, well-documented library
  • Good, because full interceptor support for OAuth2
  • Good, because works with any Qt version (5.15+)
  • Good, because large community and examples
  • Bad, because requires manual type conversion
  • Bad, because threading model mismatch with Qt event loop
  • Bad, because more boilerplate for QML exposure

REST API Layer

  • Good, because simpler HTTP-based communication
  • Good, because Qt has excellent HTTP support (QNetworkAccessManager)
  • Bad, because loses strong typing from Proto definitions
  • Bad, because requires additional REST layer generation
  • Bad, because less efficient than binary gRPC protocol

AI Agent Guidance

When This Decision Applies

  • Implementing Qt frontend communication with gRPC backend
  • Adding new service clients to the Qt layer
  • Designing async operation patterns in Qt

When This Decision Does NOT Apply

  • Server-side gRPC implementation (use standard grpc++)
  • Non-Qt C++ clients (use standard grpc++)
  • Web frontends (use grpc-web or REST)

Enforcement

  • Code review: Qt frontend code must use muscoviteqt wrapper, not raw grpc++
  • Generated code: DSL generator creates Qt wrappers, not raw stubs

Key Files Affected

  • src/muscframework/qt/ - muscoviteqt base library
  • generated-*/qt/ - Generated Qt client code
