ADR: Qt gRPC Client Auth Architecture

ADR: Qt gRPC Client Auth Architecture

Status

Accepted

Context

  • We need a Qt-friendly client library that talks to the Muscovite gRPC backend using OAuth2 (and optionally built-in auth) without committing to Widgets vs QML.
  • We already have a production-ready gRPC C++ + OAuth2 stack (muscovite_grpc_oauth2). Reusing it avoids divergence from the server and avoids an extra abstraction layer.
  • The frontend must expose asynchronous APIs that feel natural in Qt while remaining UI-neutral.
  • TLS artifacts (CA/cert/key) must be generated by existing tooling (Racket/CMake) rather than manually provisioned.

Decision

  • Use standard gRPC C++ with the existing OAuth2 interceptor; do not switch to QtGRPC in this phase.
  • Provide a dual async surface: QFuture/QPromise as the primary API, plus signals/slots for convenience.
  • Keep auth state headless (no UI), with pluggable adapters for Widgets or QML later.
  • Generate TLS artifacts via tooling (Racket generator or CMake target) and load from configured paths; no hardcoded defaults.
  • Error model: distinct domains for (a) gRPC transport/status, (b) OAuth2/auth failures, (c) token storage errors. On UNAUTHENTICATED, attempt one refresh before surfacing the error.
  • Defaults: 5-minute refresh margin before access token expiry; 60s clock skew tolerance; per-call deadline ~10s; retries off by default except when explicitly enabled for idempotent calls.
  • Threading: gRPC work on a worker thread; signals marshalled to the Qt main thread via queued connections.
  • Streaming is deferred; focus on unary calls first.
  • Token storage prefers OS keychain; in-memory refresh token fallback is opt-in and explicit.

Consequences

  • Minimal risk: we reuse the battle-tested gRPC C++ stack and OAuth2 interceptor already shipped.
  • UI stacks stay decoupled; both Widgets and QML can bind to the same headless auth/viewmodel objects.
  • Operations are simpler: TLS material is produced by existing pipelines; no manual cert steps baked into code.
  • We avoid QtGRPC lock-in but keep the door open to add an adapter later if needed.
  • Testing focus: concurrency/thread-handoff, token refresh paths, TLS load errors, and clear error surfacing for UI layers.
