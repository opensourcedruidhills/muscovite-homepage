Circuit Breaker as Generated Infrastructure State Machine

Circuit Breaker as Generated Infrastructure State Machine

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-23
  • Tags: plugins, circuit-breaker, state-machine, reliability, infrastructure, dsl

Technical Story: Plugin calls (especially LocalGRPC SideEffect plugins) may fail. The domain service must not cascade failures from optional plugins. A circuit breaker is the correct reliability pattern. Muscovite already generates domain state machines; the circuit breaker should be generated by the same machinery under a distinct DSL construct and C++ base class hierarchy.

Context and Problem Statement

LocalGRPC and QtRemoteObjects plugin transports involve out-of-process calls that can fail. Without protection, a failing audit-export plugin crashes the domain service call chain. A circuit breaker isolates this failure.

A circuit breaker is a state machine (Closed → Open → Half-Open → Closed). Muscovite already generates domain state machines (ADR 20260221-entity-level-state-machines). The insight is that the generator machinery is reusable, but the two constructs must remain architecturally distinct:

Class Purpose DSL in Generated base
Domain state machine Business lifecycle (Patient: Draft → Active) .ddd DomainStateMachineBase
Infrastructure state machine Reliability contract (Closed → Open → Half-Open) .mus InfrastructureStateMachineBase

The types are siblings in the C++ type hierarchy, not derived from each other.

Decision Drivers

  • Plugin failures must not crash domain services (isolation requirement)
  • Circuit breaker is operational/deployment configuration, not domain knowledge → .mus
  • Generator machinery reuse without semantic conflation
  • Backoff algorithm is a generation-time decision (emit exactly the declared algorithm)
  • Frontend circuit breaker must integrate with Qt's existing state machine (QStateMachine)
  • Jitter is production-critical for preventing thundering herd on recovery → include now

Considered Options

  1. CIRCUIT_BREAKER in .mus generates InfrastructureStateMachine (separate hierarchy)
  2. Reuse STATE_MACHINE DSL keyword — domain and infrastructure states in same hierarchy
  3. Runtime-only circuit breaker — no DSL, configure entirely at runtime via .mus values

Decision Outcome

Chosen option: Option 1 — CIRCUIT_BREAKER in .mus, separate InfrastructureStateMachine hierarchy.

DSL Construct (in .mus)

The CIRCUIT_BREAKER block is nested inside a PLUGIN declaration:

PLUGIN EnterpriseAudit
    IMPLEMENTS: AuditExporter
    TRANSPORT: LocalGRPC
    CIRCUIT_BREAKER:
        FAILURE_THRESHOLD: 3
        TIMEOUT: 30s
        BACKOFF: Exponential BASE 500ms MAX 30s JITTER 10%
        ON_OPEN:      Warn
        ON_HALF_OPEN: Probe
        ON_TRIP:      Ignore

Fields:

  • FAILURE_THRESHOLD — integer: consecutive failures before tripping to Open
  • TIMEOUT — duration: time in Open state before transitioning to Half-Open
  • BACKOFF — backoff algorithm (see below)
  • ON_OPEN — action on Closed → Open transition
  • ON_HALF_OPEN — action on Open → Half-Open transition
  • ON_TRIP — action fired when the triggering failure trips the circuit

Failure mode actions:

  • Warn — log warning, return cached value or error response to caller
  • Block — reject caller immediately with error (no cached value)
  • Ignore — swallow the triggering failure silently
  • Probe — allow one call through to test recovery (Half-Open behavior)
  • Emit — publish a NATS/domain event for observability

Three-State Model

Standard circuit breaker:

  • Closed — normal operation; failures counted; threshold not yet reached
  • Open — threshold exceeded; all calls rejected immediately; ON_OPEN fires; timeout timer starts
  • Half-Open — timeout expired; one probe call allowed; success → Closed; failure → Open (reset timeout)

BACKOFF Declaration

The backoff algorithm is declared in the DSL and generated as a compile-time constant (not a runtime configuration). The generated code contains exactly the declared algorithm.

BACKOFF: Exponential BASE 500ms MAX 30s JITTER 10%
BACKOFF: Linear      BASE 1s    MAX 10s JITTER 5%
BACKOFF: Fixed       INTERVAL   5s

JITTER is mandatory for Exponential and Linear (default 10% if omitted). JITTER prevents thundering herd on mass recovery. Fixed backoff has no jitter.

Generation rules:

  • Exponential: delay = min(BASE * 2^attempt, MAX) ± JITTER%
  • Linear: delay = min(BASE * attempt, MAX) ± JITTER%
  • Fixed: delay = INTERVAL

C++ Framework Base Classes (Layer 1)

Two new base classes added to framework/:

framework/
├── dba/include/muscovite/dba/
│   └── infrastructure_state_machine.hpp  # BE circuit breaker base
└── qt/include/muscovite/qt/
    └── q_infrastructure_state_machine.hpp  # FE QStateMachine-based base

InfrastructureStateMachineBase (BE):

  • Template parameter: backoff algorithm type (generated as specialisation)
  • Methods: call(F&&), on_success(), on_failure(), state(), reset()
  • Emits NATS observability event if ON_OPEN: Emit configured

QInfrastructureStateMachineBase (FE):

  • Derives from QStateMachine
  • States map to QState instances: m_closed, m_open, m_half_open
  • Transitions fire Qt signals: circuitOpened(), circuitClosed(), circuitHalfOpened()
  • UI responds to state changes (disable plugin-dependent UI sections when circuit is Open)

Generated Per-Plugin

For each plugin with CIRCUIT_BREAKER config:

  1. Specialised InfrastructureStateMachine<BackoffPolicy> instantiation (BE)
  2. Specialised QInfrastructureStateMachine subclass (FE)
  3. Wrapper method in generated domain service that routes through the circuit breaker
  4. CMake: circuit breaker code is Layer 2 (regenerated), not sidecar

Positive Consequences

  • Plugin failures are isolated; domain service continues functioning
  • Circuit breaker state is observable (NATS events, Qt signals)
  • Backoff algorithm is generated code — no runtime configuration drift
  • Domain and infrastructure state machines are architecturally distinct despite sharing generator machinery
  • FE circuit breaker integrates naturally with existing QStateMachine patterns

Negative Consequences

  • Two new framework base classes to maintain in framework/
  • Circuit breaker is per-plugin — many plugins means many generated circuit breakers (this is correct; each plugin has an independent failure domain)

Pros and Cons of the Options

Option 1: Separate InfrastructureStateMachine hierarchy (chosen)

  • Good, because domain and infrastructure states are never conflated
  • Good, because FE integrates naturally with QStateMachine
  • Good, because backoff is generated (no runtime config drift)
  • Bad, because two new base classes to maintain

Option 2: Reuse STATE_MACHINE keyword

  • Good, because reuses existing grammar and parser
  • Bad, because domain and infrastructure states are semantically different
  • Bad, because QStateMachine integration loses the explicit separation

Option 3: Runtime-only circuit breaker

  • Good, because no generator changes needed
  • Bad, because configuration can drift from code at runtime
  • Bad, because no compile-time backoff algorithm guarantee
