Form-Factor Layout Selection Cascade

Form-Factor Layout Selection Cascade

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-19
  • Tags: layout, ui, dsl, parser, qt, qml, responsive

Context and Problem Statement

ADR 20260211 introduced an abstract LAYOUT vocabulary, but each aggregate could still define only one layout. Multi-frontend generation now needs different layout intent per form factor (desktop/tablet/mobile) without duplicating aggregates.

Question: How should Muscovite select the effective aggregate layout per target form factor while preserving backward compatibility?

Decision Drivers

  • Grammar-first evolution (ADR 20260211-grammar-first-language-evolution)
  • Backward compatibility for existing unqualified LAYOUT { ... }
  • Deterministic generator behavior across Qt Widgets and QML
  • Single source of truth in .ddd domain models
  • Future extensibility for additional form factors

Considered Options

Option 1: Form-factor-qualified LAYOUT blocks with fallback cascade

Allow multiple LAYOUT blocks per aggregate, each optionally qualified (desktop, tablet, mobile, watch, embedded).

Option 2: Per-target external configuration

Keep one DSL layout and move form-factor selection to external target configuration files.

Option 3: Generator-only heuristics

Infer form-factor behavior in each generator without DSL support.

Decision Outcome

Chosen option: Option 1 — Form-factor-qualified LAYOUT blocks with fallback cascade.

Grammar and IR

  • Grammar supports repeated LAYOUT blocks with optional qualifier:
    • LAYOUT { ... } (unqualified fallback)
    • LAYOUT desktop { ... }, LAYOUT mobile { ... }, etc.
  • layout-spec stores form-factor (#f for unqualified).
  • aggregate stores layouts as a list.

Selection Rules

Generators must resolve layout in this order:

  1. Exact form-factor match
  2. Optional cascade list (target-specific)
  3. Unqualified layout (form-factor = #f)
  4. No layout (#f), then existing generator defaults apply

Initial Target Mapping

  • Qt Widgets: desktop
  • Qt QML desktop: desktop
  • Future QML mobile: mobile with cascade tablet

Positive Consequences

  • One aggregate can express responsive intent cleanly
  • Existing .ddd files remain valid and keep behavior
  • Generator logic is explicit and testable
  • New form factors require minimal extension

Negative Consequences

  • Additional parser and IR complexity
  • More keyword surface in the lexer/grammar
  • Need to keep serialization backward-compatible (layout + layouts)

AI Agent Guidance

When This Decision Applies

  • When parsing or generating aggregate LAYOUT blocks
  • When selecting UI layout for a target frontend form factor
  • When serializing/deserializing aggregate layout IR

When This Decision Does NOT Apply

  • Non-UI generators that do not consume aggregate layout hints
  • Domains with no LAYOUT blocks (default generator behavior remains valid)

Enforcement

  • Enforced by PEG grammar and parser tests
  • Enforced by IR resolution functions in ddd-types.rkt
  • Enforced by Qt generator layout-map resolution at context generation time
