Arc Constraints for Polymorphic Types

Arc Constraints for Polymorphic Types

  • Status: accepted
  • Date: 2026-01-23
  • Tags: ddd, constraints, polymorphism, validation, fhir

Context and Problem Statement

Many domain models require exactly N of M fields to be populated—a pattern called "arc constraints" or "exclusive arc." This is essential for polymorphic types like FHIR's Observation.value[x] where exactly one of several value types must be set.

ENTITY Observation @uuid "..." IDENTIFIED_BY id (
    id              UUID NOT NULL,
    value_quantity  DECIMAL,
    value_string    TEXT,
    value_boolean   BOOLEAN,
    value_date      DATE
)
CHECK value_arc @uuid "...":
    ((value_quantity IS NOT NULL)::int + 
     (value_string IS NOT NULL)::int + 
     (value_boolean IS NOT NULL)::int + 
     (value_date IS NOT NULL)::int) = 1
DOC "Clinical observation with polymorphic value";

Questions:

  1. How should arc constraints be validated at each layer?
  2. Should the DSL have dedicated syntax or use CHECK expressions?
  3. How do Qt edit dialogs enforce mutual exclusivity?

Decision Drivers

  • FHIR Compliance: FHIR R4 extensively uses [x] polymorphic elements
  • Database Integrity: PostgreSQL CHECK constraints are the final enforcement
  • User Experience: Qt dialogs should guide users to valid states
  • Type Safety: Arc constraint violations should be caught early
  • Expressiveness: Support "exactly N", "at least N", "at most N" patterns

Considered Options

  1. PostgreSQL CHECK only: Database-level enforcement, UI discovers on save
  2. Dedicated ARC syntax: ARC EXACTLY 1 OF { field1, field2, field3 }
  3. CHECK expression with Qt generation: Current approach enhanced (chosen)

Decision Outcome

Chosen option: "CHECK expression with Qt generation", because:

  • Already have CHECK constraint parsing with @uuid annotations
  • PostgreSQL CHECK syntax is flexible and well-understood
  • Qt generator can detect arc patterns and generate appropriate UI

DSL Syntax

The existing CHECK syntax supports arc constraints:

-- Exactly 1 of 4 (polymorphic value)
CHECK value_arc @uuid "...":
    ((value_quantity IS NOT NULL)::int + 
     (value_string IS NOT NULL)::int + 
     (value_boolean IS NOT NULL)::int + 
     (value_date IS NOT NULL)::int) = 1

-- At least 1 of 3 (required contact method)
CHECK contact_arc @uuid "...":
    ((email IS NOT NULL)::int + (phone IS NOT NULL)::int + (fax IS NOT NULL)::int) >= 1

-- At most 2 of 4 (optional ceiling)
CHECK options_arc @uuid "...":
    ((opt_a IS NOT NULL)::int + (opt_b IS NOT NULL)::int + 
     (opt_c IS NOT NULL)::int + (opt_d IS NOT NULL)::int) <= 2

Generated PostgreSQL

ALTER TABLE observation.observation
    ADD CONSTRAINT chk_observation_value_arc CHECK (
        ((value_quantity IS NOT NULL)::INT + 
         (value_string IS NOT NULL)::INT + 
         (value_boolean IS NOT NULL)::INT + 
         (value_date IS NOT NULL)::INT) = 1
    );

Generated Qt Validation

The Qt generator detects arc constraint patterns and generates validation logic:

class ObservationValidator : public muscovite::qt::EntityValidator {
public:
    ValidationResult validateArcConstraints() const override {
        ValidationResult result;
        
        // value_arc: exactly 1 of 4
        int valueFieldsSet = 0;
        if (!valueQuantity_->text().isEmpty()) valueFieldsSet++;
        if (!valueString_->text().isEmpty()) valueFieldsSet++;
        if (valueBoolean_->isChecked()) valueFieldsSet++;
        if (valueDate_->date().isValid()) valueFieldsSet++;
        
        if (valueFieldsSet != 1) {
            result.addError(ValidationCode::ARC_CONSTRAINT_VIOLATION,
                tr("Exactly one value field must be set (quantity, string, boolean, or date)"));
        }
        
        return result;
    }
};

Qt UI Behavior

For "exactly 1 of N" arcs, the Qt generator can emit:

  1. Radio-button group: Fields grouped with radio selection
  2. Tab widget: Each option on a separate tab
  3. Combo + dynamic field: Select type, show corresponding input
  4. Mutual clearing: Setting one field clears others (optional behavior)
// Option 3: Combo + dynamic field
void ObservationEditDialog::setupValueField() {
    valueTypeCombo_ = new QComboBox();
    valueTypeCombo_->addItem(tr("Quantity"), "quantity");
    valueTypeCombo_->addItem(tr("Text"), "string");
    valueTypeCombo_->addItem(tr("Yes/No"), "boolean");
    valueTypeCombo_->addItem(tr("Date"), "date");
    
    valueStack_ = new QStackedWidget();
    valueStack_->addWidget(valueQuantityWidget_);
    valueStack_->addWidget(valueStringWidget_);
    valueStack_->addWidget(valueBooleanWidget_);
    valueStack_->addWidget(valueDateWidget_);
    
    connect(valueTypeCombo_, &QComboBox::currentIndexChanged,
            valueStack_, &QStackedWidget::setCurrentIndex);
}

Positive Consequences

  • FHIR polymorphic types fully supported
  • Database guarantees data integrity
  • Qt dialogs guide users to valid states
  • Flexible: exactly/at-least/at-most patterns all work

Negative Consequences

  • Arc pattern detection requires parsing CHECK expressions
  • Qt UI generation adds complexity
  • Schema evolution: adding/removing arc members needs migration awareness

Implementation Notes

Parser Enhancement

Arc constraint detection function:

;; Detect arc constraint pattern in CHECK expression
(define (check-is-arc-constraint? check-constraint)
  (define expr (check-constraint-expression check-constraint))
  ;; Look for pattern: ((field IS NOT NULL)::int + ...) = N
  (and (string-contains? expr "IS NOT NULL")
       (string-contains? expr "::int")
       (or (string-contains? expr "= 1")
           (string-contains? expr ">= 1")
           (string-contains? expr "<= "))))
