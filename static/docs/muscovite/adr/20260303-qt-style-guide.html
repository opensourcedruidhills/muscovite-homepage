Qt/QML Modern Style Guide for Muscovite

Qt/QML Modern Style Guide for Muscovite

  • Status: accepted
  • Date: 2026-03-03
  • Tags: qt, qml, widgets, style-guide, patterns, generator
  • Deciders: Johannes Lochmann

Technical Story: FTR-188 — Codify doc/QT-WIDGETS-PATTERNS.md (16 sections) and doc/QML-PATTERNS.md (13 sections) as binding Qt/QML style guides for all Muscovite Qt code (framework Layer 1 and generated Layer 2).

Context and Problem Statement

Muscovite generates Qt Widgets and Qt Quick/QML desktop clients from the same .ddd domain model. The framework (framework/qt/) provides base classes (EntityListModel, LoginDialog, GrpcChannelManager, validation). Generated code must follow Qt idioms precisely — Qt has strong conventions around:

  • Signal/slot naming, Q_PROPERTY patterns, memory management (parent-child ownership)
  • QML import syntax (Qt 6 versionless), property ordering, type-safe required property
  • Testing: QTest for anything touching QObject/QWidget; doctest only for pure value types

Two compliance audits exist:

  • doc/QT-FRAMEWORK-AUDIT.md — framework code patterns
  • doc/QT-GENERATOR-AUDIT.md — generated code patterns

This ADR elevates the pattern catalogs to binding guides and explicitly clarifies the testing boundary.

Decision Drivers

  • Generated QML must pass Qt Quick Compiler static analysis (qmllint)
  • Qt 6 versionless imports avoid build breakage on minor version bumps
  • AI agents need a single reference for Qt-specific patterns (separate from C++23 patterns)
  • Testing boundary confusion (QTest vs doctest) caused wrong test framework choices in past PRs

Considered Options

  1. Adopt QT-WIDGETS-PATTERNS.md + QML-PATTERNS.md as binding guides
  2. Use Qt documentation conventions only — no project-specific guide
  3. Merge into CPP-PATTERNS.md — single document for all C++

Decision Outcome

Chosen option: Option 1 — Separate Qt/QML binding guides, because Qt patterns are domain-specific to the GUI layer and do not apply to DBA/gRPC code. Merging would bloat the C++ guide and confuse scope.

Qt Widgets Binding Sections (16 Sections)

# Section Severity Key Rule
1 Asynchronous gRPC Calls P0 Never block the event loop with synchronous stubs
2 gRPC Stub Ownership P0 Per-instance unique_ptr<StubInterface>, not static
3 Theming and Colors P1 QPalette system colors, no hardcoded RGB
4 QML Property Naming P1 camelCase for Q_PROPERTY, snake_case for C++ members
5 Validator Widget Types P1 Use QDoubleValidator for Decimal fields
6 Memory Management P1 Qt parent-child ownership; raw new only with parent
7 Signal/Slot Connections P1 connect(sender, &Class::signal, ...) — no string-based
8 Internationalization P1 tr() for Widgets, qsTr() for QML
9 Threading and Blocking P0 No QThread::msleep; use QTimer::singleShot
10 Layout Management P2 QFormLayout for forms, QGridLayout for tables
11 Q_PROPERTY for ViewModels P1 All fields as Q_PROPERTY with NOTIFY signal
12 Forward Declarations P2 Forward-declare Qt classes in headers
13 Header/Source Split P1 Never inline QWidget method bodies
14 QRegularExpression P2 static const QRegularExpression for reuse
15 Member Initialization P1 NSDMI (in-class init) for Qt member pointers
16 Qt6-Specific APIs P1 qOverload<> over QOverload<>::of

QML Binding Sections (13 Sections)

# Section Severity Key Rule
1 Internationalization (qsTr) P0 All user-facing strings wrapped in qsTr()
2 Type-Safe Required Properties P1 required property type name for ViewModel injection
3 Versionless Qt 6 Imports P0 import QtQuick — no version numbers
4 QAbstractListModel for ListView P0 EntityListModel base class, no ListModel/modelData
5 Loader for Dynamic Dialogs P1 Loader { active: false } for dialog creation
6 SpinBox Value Handling P1 onValueModified (user action) over onValueChanged (binding loop)
7 ListView Clipping P2 clip: true on all ListView instances
8 Theme Constants P2 MuscoviteTheme.qml pragma singleton
9 QML Property Ordering P2 id → dimensions → anchors → properties → children → handlers
10 Grouped Anchor Notation P2 anchors { left: parent.left; ... } grouped block
11 Type Dispatch Consolidation P1 Central common.rkt dispatch, no per-file type→widget maps
12 Module Import (except-in) P1 Racket except-in to resolve binding conflicts
13 Template DSL Compliance P1 define-template for all new generator functions

Testing Boundary (EXPLICIT — NO EXCEPTIONS)

Code Under Test Test Framework Rationale
QObject subclass QTest Requires QCoreApplication, signal/slot testing
QWidget subclass QTest Requires QApplication, event loop
QAbstractItemModel QTest QSignalSpy, QModelIndex
ViewModel (Q_PROPERTY) QTest Signal verification, property binding
Pure value types (no Qt) doctest No Qt event loop needed
pqxx repository code doctest Database, no Qt dependency

FORBIDDEN: Using doctest for any class inheriting from QObject. Using QTest for pure C++ value types with no Qt dependency.

Top 5 Forbidden Qt/QML Patterns

# Forbidden Required Rationale
1 Synchronous gRPC stub calls in Qt (stub_->Method(...)) TaskTree GrpcTaskAdapter async pattern Blocks event loop → frozen UI
2 QOverload<int>::of(&QSpinBox::valueChanged) qOverload<int>(&QSpinBox::valueChanged) Qt6 free function, shorter and cleaner
3 import QtQuick 2.15 import QtQuick Qt6 versionless imports
4 ListModel { ListElement { ... } } in QML C++ QAbstractListModel subclass Typed, testable, sortable, filterable
5 QThread::msleep(500) QTimer::singleShot(500, ...) Non-blocking event-loop-friendly

Positive Consequences

  • Clear testing boundary prevents framework choice errors
  • Generated QML passes qmllint static analysis
  • AI agents have an explicit Qt-specific forbidden-pattern list

Negative Consequences

  • Two separate documents to maintain (Widgets + QML) — mitigated by Generator Cross-Reference sections in each
  • Retrofit of existing code is phased per Priority sections
