C++23 Style Guide for Muscovite

C++23 Style Guide for Muscovite

  • Status: accepted
  • Date: 2026-03-03
  • Tags: c++, style-guide, cpp23, patterns, framework, generator
  • Deciders: Johannes Lochmann

Technical Story: FTR-187 — Codify doc/CPP-PATTERNS.md as the binding C++23 style guide for all Muscovite C++ code (framework Layer 1 and generated Layer 2).

Context and Problem Statement

Muscovite generates ~56K lines of C++ code across DBA repositories, gRPC services, and Qt clients. The framework (framework/) adds ~35K lines. Both layers must maintain consistent, modern C++23 style to:

  1. Ensure generated output is production-quality — not "code-gen smell"
  2. Provide a stable target for AI agents writing/modifying C++ code
  3. Prevent regression of patterns already modernized

The doc/CPP-PATTERNS.md catalog was created as a compliance audit (v1.43.0) documenting 20 pattern sections with severity ratings. This ADR elevates that catalog to a binding style guide.

Decision Drivers

  • Generated code must be indistinguishable from hand-written modern C++23
  • AI agents (Copilot, Claude) need explicit forbidden/required patterns to avoid regressions
  • Framework and generated code share the same C++ compiler — style must be unified
  • The compliance audit (doc/CPP23-COMPLIANCE-AUDIT.md) identified specific gaps to close

Considered Options

  1. Adopt CPP-PATTERNS.md as the binding style guide — reference from copilot-instructions.md
  2. Use an external style guide (e.g., Google C++ Style Guide) — adapt to project needs
  3. No formal guide — rely on code review and convention

Decision Outcome

Chosen option: Option 1 — CPP-PATTERNS.md as binding style guide, because it is already tailored to Muscovite's specific needs (pqxx, gRPC, Qt, code generation), covers the exact patterns seen in the codebase, and has severity ratings that prioritize fixes.

Binding Sections (20 Sections)

# Section Severity Key Rule
1 Error Handling P0 Use std::expected<T,E> for recoverable errors; [[nodiscard]] on all
2 Attribute Annotations P1 [[nodiscard]] on factories/queries; [[maybe_unused]] on interface params
3 Ranges and Views P1 std::ranges::transform over raw for+push_back
4 Structured Bindings P2 Use for multi-return, tuple/pair decomposition
5 Concepts and Constraints P2 requires clauses over static_assert SFINAE
6 Designated Initializers P2 Named initialization for config structs
7 String Handling P1 std::format over fmt::format; constexpr pqxx::zview for SQL
8 Const Correctness P1 const everywhere possible; constexpr for compile-time values
9 Forward Declarations P2 Forward-declare in headers; include in sources
10 Header/Source Separation P1 No inline method bodies in headers (except templates)
11 Namespace Structure P2 muscovite::{module}:: hierarchy
12 Memory and Ownership P1 unique_ptr default; shared_ptr only when shared ownership proven
13 C Legacy Elimination P1 No reinterpret_cast, C-style casts, #define constants
14 SQL Query Management P0 constexpr pqxx::zview for all SQL; parameterized queries ($1)
15 Duplicate Code Elimination P2 Template-based deduplication
16 Include Guards P2 #pragma once (compiler-standard, faster than #ifndef)
17 NATS/Messaging Patterns P1 Framework-level publish helpers
18 constexpr Where Possible P1 Compile-time constants, UUIDv5 lookup values
19 Testing Patterns P1 doctest BDD for pure C++; QTest for QObject classes (see ADR Qt Style Guide)
20 pqxx API Usage P0 pqxx::work for writes, pqxx::nontransaction for reads, pqxx::params

Top 5 Forbidden Patterns (Mandatory CI/Review Enforcement)

# Forbidden Required Rationale
1 Raw for + push_back accumulation std::ranges::transform / ranges::to<vector>() C++23 ranges eliminate off-by-one, improve readability
2 fmt::format(...) std::format(...) C++23 <format> is standard; eliminates {fmt} dependency long-term
3 std::string sql = "SELECT ..." constexpr pqxx::zview sql{"SELECT ..."} Compile-time string, zero-copy to libpq
4 reinterpret_cast / C-style casts static_cast / std::bit_cast Type safety, grep-ability
5 QThread::msleep() in non-test code QTimer::singleShot() Non-blocking event loop (see ADR Qt Style Guide)

Positive Consequences

  • AI agents have an explicit forbidden-pattern list → fewer regressions
  • Generated C++ code quality is auditable against a single document
  • New contributors have a single style reference

Negative Consequences

  • Existing code that predates this ADR may not fully comply — retrofit is phased per CPP-PATTERNS.md Priority and Phasing section
  • Some patterns (e.g., std::expected) require C++23 compiler support — already enforced via CMAKE_CXX_STANDARD 23
