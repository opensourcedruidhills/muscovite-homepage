Relationship Target Resolution in Migration Operations

Relationship Target Resolution in Migration Operations

  • Status: accepted
  • Deciders: Architecture Team
  • Date: 2026-01-16
  • Tags: migration, relationships, foreign-keys

Context and Problem Statement

When the semantic diff detects new relationships, the add-relationship-op stores target entity information incompletely:

{"target-entity": false, "type": "add-relationship"}

The target entity UUID is false because cross-context relationship resolution happens at generation time, not during diff computation. This breaks incremental migration SQL generation because:

  1. FK constraint SQL requires the target table name
  2. Target table may be in a different schema
  3. Without resolution, generator cannot produce valid SQL

Decision Drivers

  • Relationships frequently cross bounded context boundaries
  • FK constraints require fully qualified table names
  • Migration SQL must be complete and executable
  • Diff operations should be self-contained

Considered Options

  1. Resolve at generation time (current broken approach)
  2. Resolve during diff computation
  3. Store relationship reference and resolve lazily

Decision Outcome

Chosen option: Resolve during diff computation

The add-relationship-op struct will store fully resolved target information:

(struct add-relationship-op
  (uuid                    ; Relationship UUID
   source-entity-uuid      ; Source entity UUID
   source-schema-name      ; Source entity's schema
   target-entity-uuid      ; Target entity UUID (RESOLVED)
   target-entity-name      ; Target entity name (RESOLVED)
   target-schema-name      ; Target entity's schema (RESOLVED)
   field-name              ; FK field name in source
   cardinality             ; 'one or 'many
   required?               ; NOT NULL constraint
   rel                     ; Original relationship struct
   safety)
  #:transparent)

Resolution Algorithm

(define (resolve-relationship-target rel all-contexts)
  ;; 1. Get target entity name from relationship
  (define target-name (relationship-target-entity rel))
  
  ;; 2. Search all contexts for entity with that name
  (for*/first ([ctx all-contexts]
               [ent (all-entities ctx)]
               #:when (eq? (entity-name ent) target-name))
    (values (entity-uuid ent)
            (entity-name ent)
            (context->schema-name ctx))))

Cross-Context Resolution

When resolving targets:

  1. First search the same context as the source entity
  2. Then search imported contexts (via IMPORT statements)
  3. Finally search all contexts (with warning about implicit dependency)

Consequences

Positive

  • Migration SQL is complete and executable
  • FK constraints have correct target table references
  • Cross-schema relationships handled correctly

Negative

  • Diff computation needs access to full model (both old and new)
  • Slightly more complex diff algorithm

SQL Generation Example

With resolved relationship:

ALTER TABLE condition.condition
ADD CONSTRAINT condition_clinical_status_id_fkey
FOREIGN KEY (clinical_status_id)
REFERENCES basedata.conditionclinicalstatus(id);

AI Agent Guidance

  • When detecting relationship additions, always resolve target entity
  • If target cannot be resolved, emit a validation error (not a placeholder)
  • Include both source and target schema names in the operation
