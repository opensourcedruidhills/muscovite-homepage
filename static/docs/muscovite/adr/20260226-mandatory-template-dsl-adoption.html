Mandatory Template DSL Adoption for All Generators

Mandatory Template DSL Adoption for All Generators

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-26
  • Tags: racket, generators, template-dsl, code-quality, enforcement
  • Supersedes: 20260218-template-dsl-for-code-generation.md (supersedes optional adoption; now mandatory)
  • Relates to: 20260226-racket-idiomatic-style-and-byte-compilation.md
  • Relates to: 20260223-racket-generator-unit-testing-strategy.md

Technical Story: ADR 20260218 introduced the template DSL (FTR-069) as an optional alternative to format/string-append code generation. Eight months later, adoption is 8 define-template invocations in 1 file out of ~200 generator files (53,529 lines). The remaining 99.9% of generators still use raw format (3,461 calls) and string-append (1,488 calls). Optional adoption has failed — the template DSL must become mandatory for all new and modified generator code.

Context and Problem Statement

The template DSL was introduced in v1.21.0 (ADR 20260218) with an incremental migration plan:

  1. ✅ Value objects (v1.21.0) — done, 8 templates
  2. ❌ Entities (v1.22.0 candidate) — never started
  3. ❌ Repositories, services (v1.23.0+) — never started
  4. ❌ Cross-target templates (v1.24.0+) — never started

The migration stalled because:

  • No enforcement mechanism — ADR 20260218 says "migrate incrementally" but doesn't define when or block non-template code
  • Path of least resistance — format strings are faster to write for a single generator; the template DSL's value is in composition and reuse, which emerges only when multiple generators use it
  • AI agents default to format — agents copy existing patterns; 99.9% of existing patterns are format/string-append

The result: 3,461 format calls and 1,488 string-append calls across generators, each one a potential positional placeholder error, indentation bug, or duplication that the template DSL was designed to prevent.

The Duplication Problem (Quantified)

Pattern Approximate Duplications Template DSL Fix
C++ struct generation 5 generators (entity, VO, event, projection, migration) cpp-struct template
C++ include guard + header every .hpp generator cpp-header-file template
C++ Doxygen comment block every C++ generator cpp-doc-comment template
SQL CREATE TABLE + columns entity, lookup, event store, saga, projection sql-create-table template
SQL COMMENT ON generation 6+ generators sql-comment-on template
Proto message definition entity, event, saga, projection proto-message template
Proto field mapping entity, relationship, lookup proto-field template
Qt property declaration widgets and QML generators qt-property template
CMake target_sources every build generator cmake-target-sources template

Each duplication is a site where a fix in one generator must be manually replicated to N others. The template DSL eliminates this by composition.

Decision Drivers

  • Optional adoption failed after 8 months — 0.1% adoption rate
  • 3,461 format + 1,488 string-append = 4,949 raw string generation sites
  • Duplication across generators is a correctness multiplier (per ADR 20260223's amplification argument)
  • AI agents will follow the dominant pattern — which must become template DSL
  • The template DSL is proven (byte-identical output, 4× acceptable overhead, passing tests)
  • Cross-target template sharing is impossible with raw format strings
  • Source location in template errors (via define-template macro) is strictly better than format arity errors

Considered Options

  1. Status quo — keep template DSL optional, hope for organic adoption
  2. Mandatory for new code only — all new generator functions must use templates; existing code migrates when touched
  3. Big-bang migration — rewrite all generators to templates in one release
  4. Abandon template DSL — accept format/string-append as the permanent pattern

Decision Outcome

Chosen option: Option 2 — Mandatory for new code, migrate existing when touched, because it avoids a feature freeze while establishing an irreversible trend toward template adoption. This is the same strategy used for for/fold vs set! in ADR 20260226.

Rules (NO EXCEPTIONS for new code)

Rule 1: All new generator functions MUST use the template DSL

;; FORBIDDEN in new generator code
(define (generate-foo entity ctx)
  (string-append
   (format "struct ~a {\n" (symbol->cpp-class (entity-name entity)))
   (format "    ~a ~a;\n" cpp-type field-name)
   "};\n"))

;; REQUIRED in new generator code
(define-template generate-foo
  (entity ctx)
  (block 0
    (line (format "struct ~a {" (symbol->cpp-class (entity-name entity))))
    (map-items
     (λ (fld)
       (indent 1 (line (format "~a ~a;" (field->cpp-type fld) (field->cpp-name fld)))))
     (entity-fields entity))
    (line "};")))

Rule 2: Modified generator functions MUST be converted to template DSL

When a generator function is modified for any reason (feature, bugfix, refactor), the entire function must be converted to template DSL. Do not mix format/string-append with template nodes in the same function.

Rule 3: Cross-target patterns MUST be shared templates

When two or more targets generate structurally similar output (e.g., C++ struct for entity vs C++ struct for value object), extract a shared template. Shared templates live in dslgen/generators/shared/templates/.

Rule 4: Template DSL node types are the ONLY permitted output mechanism

Permitted Forbidden
(line ...) (string-append ... "\n")
(block ...) Nested string-append
(join ...) (string-join ... sep) for template output
(when-true ...) (if cond (format ...) "")
(map-items ...) (apply string-append (map (λ (x) (format ...)) xs))
(indent ...) Manual " " whitespace strings
(text ...) Raw string returns from generator functions

Exception: Utility functions that compute names, types, or other non-template values may use format/string-append for string computation (not output construction).

Migration Plan

Tier 1 — Foundation Templates (v1.36.0)

Create shared templates for the most duplicated patterns:

dslgen/generators/shared/templates/
├── cpp-templates.rkt       # cpp-struct, cpp-header-file, cpp-doc-comment, cpp-namespace
├── sql-templates.rkt       # sql-create-table, sql-comment-on, sql-alter-table
├── proto-templates.rkt     # proto-message, proto-field, proto-service
├── qt-templates.rkt        # qt-property, qt-signal, qt-model
└── cmake-templates.rkt     # cmake-target, cmake-find-package

Tier 2 — High-Value Generator Migration (v1.37.0–v1.38.0)

Convert generators with most duplication:

  1. cpp/dba/entity.rkt — entity struct generation (used by ~5 downstream generators)
  2. postgres/sqitch-entity.rkt — CREATE TABLE + COMMENT ON
  3. grpc/proto.rkt — proto message definitions
  4. qt/widgets-views.rkt + qt/quick-views.rkt — shared widget/QML patterns

Tier 3 — Remaining Generators (Ongoing)

All remaining generators convert when touched. Track progress:

# Migration audit
grep -rl "define-template" dslgen/generators/ | wc -l       # files using templates
grep -rL "define-template" dslgen/generators/ --include="*.rkt" | wc -l  # files NOT using templates

Template DSL Extensions Required

The current template DSL (233 lines, 7 node types) may need extensions for full adoption:

Extension Purpose Priority
(indent-block n ...) Shorthand for (block n (indent n ...)) High — most common pattern
(blank-line) Empty line separator High — all generators need it
(comment style text) Target-aware comments (//, --, #) High — replaces manual comment formatting
(guard cond template) Like when-true but raises on #f Medium — for required sections
(choose pred items) Filter + map in one node Medium — replaces filter-map chains
(include-template name args) Cross-file template reference High — enables shared templates

Positive Consequences

  • Duplication eliminated: N×M (N patterns × M targets) reduced to N shared templates
  • Indentation bugs eliminated: indent node handles all whitespace
  • Positional placeholder errors eliminated: named template parameters
  • Source location in errors: define-template macro provides file:line
  • AI agents naturally adopt templates when dominant pattern
  • Cross-target consistency: shared templates guarantee identical structure
  • Testable output: template nodes can be tested structurally without string matching

Negative Consequences

  • Learning curve: contributors must learn template DSL before writing generators
  • Transition period: two output mechanisms coexist (shrinking, never growing)
  • Template overhead: 4× slower than format (acceptable for offline generators; optimize if needed)
  • Initial investment: shared template library (~500–1000 lines) must be written before Tier 2 migration

Add to copilot-instructions.md, CLAUDE.md, AGENTS.md:

### N. Template DSL Mandatory (NO EXCEPTIONS for new generator code)

Per ADR 20260226-mandatory-template-dsl-adoption:
- ALL new generator functions MUST use `define-template` + template nodes
- ALL modified generator functions MUST be converted to template DSL
- Shared patterns (C++ struct, SQL CREATE TABLE, proto message) MUST use shared templates from `generators/shared/templates/`

**FORBIDDEN**: New `format`/`string-append` for output construction in generators |
  Mixed template + format in same function | Duplicating output patterns across targets

Code Review Gate

Generator PRs must not introduce new format-based output construction. The audit command:

# Should trend toward zero over time
grep -rc "string-append\|format " dslgen/generators/ --include="*.rkt" | awk -F: '{s+=$2} END {print s}'

Test Requirement

Per ADR 20260223, every new generator function gets a unit test. Template-based functions are tested by:

  1. Constructing a minimal IR fixture
  2. Calling the template function
  3. Asserting structural properties of the emitted string (not exact match — structural)

AI Agent Guidance

When This Decision Applies

  • When modifying an existing generator function for any reason
  • When adding a new generator target
  • When reviewing AI-generated generator code

When This Decision Does NOT Apply

  • Utility functions computing names, types, or paths (these return strings, not templates)
  • Parser code (dslgen/parser/)
  • Model/IR code (dslgen/model/)
  • Migration pipeline (dslgen/migration/) — uses format for SQL, not code generation patterns
  • Test code (dslgen/tests/)

Template Authoring Quick Reference

(require "../template-dsl.rkt")

(define-template my-generator
  (entity ctx)
  (block 0
    ;; Section comment
    (line (format "// ~a" (entity-doc entity)))
    (blank-line)
    ;; Conditional section
    (when-true (entity-has-events? entity)
      (block 0
        (line "// Events")
        (map-items
         (λ (evt) (indent 1 (line (format "~a," (event-name evt)))))
         (entity-events entity))))
    ;; Join with separator
    (join ",\n" (map field->declaration (entity-fields entity)))))

;; Use it
(define output (emit (my-generator entity ctx)))
