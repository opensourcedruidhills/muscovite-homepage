Plugin System — Extension Points and the Layer 2 Sidecar

Plugin System — Extension Points and the Layer 2 Sidecar

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-23
  • Tags: plugins, extension-points, layer2-sidecar, dsl, generator, architecture

Technical Story: Muscovite's generated applications need well-defined, typed extension points that third-party code can implement without modifying generated code. This ADR establishes the plugin system architecture: the EXTENSION_POINT DSL construct, the PLUGIN wiring in .mus, and the Layer 2 Sidecar concept for generated-once user-owned code.

Context and Problem Statement

The three-layer invariant states that Layer 2 (generated applications) is never manually edited. This is correct — but it creates a problem: how does user-written domain behavior hook into the generated application? Without extension points, users must either:

  1. Fork the generator (wrong — violates DSL-first)
  2. Edit generated code (wrong — breaks regeneration guarantee)
  3. Insert hooks by hand (wrong — no contract, no tooling)

The plugin system solves this by making extension points first-class DSL citizens. The .ddd file declares the slot; the generator produces the interface; the plugin provides the implementation; the framework wires them at startup.

Decision Drivers

  • Extension mechanism must not introduce Turing-complete complexity in the DSL
  • Three-layer invariant must be preserved: generated code is always regeneratable
  • Plugin implementations must have a stable, versioned contract
  • Generated-once user files need a clear protocol to coexist with continuous regeneration
  • Grammar-first evolution (ADR 20260211-grammar-first-language-evolution)

Considered Options

  1. EXTENSION_POINT in .ddd + PLUGIN in .mus + Layer 2 Sidecar stubs (extension points as domain contracts, wiring as project config)
  2. External plugin registry file (separate plugin manifest, not in DSL)
  3. Convention-based hooks (no DSL construct, just naming conventions)

Decision Outcome

Chosen option: Option 1 — EXTENSION_POINT in .ddd + PLUGIN in .mus + Layer 2 Sidecar.

Three Plugin Classes

Class Contract Interface Transport
Validator Synchronous, pure, stateless Pure virtual C++ function None — in-process
SideEffect Asynchronous, potentially slow gRPC over localhost (BE) / Qt Remote Objects (FE) Local socket only
UIWidget Frontend UI injection Pure virtual C++ / Qt Remote Objects Qt-native or local socket

EXTENSION_POINT in .ddd (Domain Contract)

Extension points are declared in .ddd because they are domain contracts: they define what the aggregate exposes, what payload it carries, and when it fires. They are not project configuration.

EXTENSION_POINT PasswordValidator
    CLASS: Validator
    TARGET: User.password
    LAYER: Backend
    INTERFACE: SyncVirtual
    DOC "Validates password strength against configured policy"

EXTENSION_POINT AuditExporter
    CLASS: SideEffect
    TARGET: PatientAggregate
    TRIGGER: OnCommit
    LAYER: Backend
    INTERFACE: LocalGRPC
    DOC "Exports audit records to external audit log"

EXTENSION_POINT CustomPatientFormSection
    CLASS: UIWidget
    TARGET: PatientEditDialog
    LAYER: Frontend
    INTERFACE: QtWidget
    DOC "Custom form section injected into patient edit dialog"

IR struct (extension-point):

  • uuid — stable UUID for semantic diff
  • name — Symbol: extension point name
  • class — Symbol: 'Validator | 'SideEffect | 'UIWidget
  • target — Symbol or qualified name: entity/aggregate/field/dialog being extended
  • trigger — Symbol or #f: 'OnCommit | 'OnValidate | 'OnSave | #f (Validators: #f)
  • layer — Symbol: 'Backend | 'Frontend
  • interface — Symbol: 'SyncVirtual | 'LocalGRPC | 'QtWidget | 'QtRemoteObjects
  • doc — String or #f

Validation rules (enforced at parse time):

  • TARGET must reference an existing entity, aggregate, field, or dialog in the same context
  • INTERFACE must be consistent with CLASS and LAYER (e.g., Validator ↔︎ SyncVirtual)
  • TRIGGER required for SideEffect and UIWidget, forbidden for Validator

PLUGIN in .mus (Project Wiring)

Plugins wire against extension points in the .mus project file because plugin wiring is a project/deployment decision, not a domain decision.

PLUGIN MyPasswordPolicy
    IMPLEMENTS: PasswordValidator
    TRANSPORT: SyncVirtual

PLUGIN EnterpriseAudit
    IMPLEMENTS: AuditExporter
    TRANSPORT: LocalGRPC
    CIRCUIT_BREAKER:
        FAILURE_THRESHOLD: 3
        TIMEOUT: 30s
        BACKOFF: Exponential BASE 500ms MAX 30s JITTER 10%
        ON_OPEN: Warn
        ON_HALF_OPEN: Probe
        ON_TRIP: Ignore

IR struct (plugin-config):

  • name — Symbol: plugin name
  • implements — Symbol: must reference a declared EXTENSION_POINT
  • transport — Symbol: transport (validated against EXTENSION_POINT.INTERFACE)
  • circuit-breaker — circuit-breaker-config or #f

Cross-reference validation: IMPLEMENTS target must name a declared EXTENSION_POINT in the same project. Validated by the project parser after all .ddd files are loaded.

Layer 2 Sidecar

The three-layer invariant states Layer 2 is always regeneratable. The plugin system introduces a new territory: generated-once, user-owned stub files.

Territory Ownership Regenerated? Named
Layer 2 Generator Yes — every run Interface files
Layer 2 Sidecar Developer No — once + sentinel-protected Stub implementation files

Naming convention:

  • {ExtensionPoint}_interface.hpp — Layer 2, always regenerated, contains abstract base
  • {ExtensionPoint}_impl.cpp — Layer 2 Sidecar, generated once, user-owned thereafter

Sentinel mechanism:

The second line of every freshly generated sidecar file contains:

// MUSCOVITE:SIDECAR:UNMODIFIED — remove this line when editing this file

On every generation run, the generator checks each sidecar file:

  • Sentinel present → file is pristine. If the interface has changed, regenerate.
  • Sentinel absent → user has edited. Never touch. Emit a notice: → sidecar {file} user-owned, skipped.

Why sentinel over checksum: checksums are fragile (whitespace normalization, line endings, BOM). A sentinel comment survives any encoding normalization and is human-readable. A developer who opens the file and sees the sentinel understands immediately what it means.

--force-regen-stubs Flag

Regenerates sidecar files even when user-modified. Output:

  • Produces {ExtensionPoint}_impl.cpp.new alongside the existing user-owned file
  • Does NOT overwrite in place
  • Developer reconciles via git merge {ExtensionPoint}_impl.cpp.new or three-way merge

This is the correct place for reconciliation — git history is the right tool.

CMake Integration

Both files are added to target_sources() with distinct labels:

target_sources(patient_context
    PRIVATE
        # Layer 2 — regenerated every run
        generated/plugins/password_validator_interface.hpp

    PRIVATE
        # Layer 2 Sidecar — user-owned, generated once
        sidecars/plugins/password_validator_impl.cpp
)

The sidecar directory (sidecars/) is separate from generated/ to make the distinction visible in the file tree.

Generated Artifacts per Extension Point

  1. Abstract C++ interface class (Layer 2)
  2. Proto definition (for LocalGRPC transport, Layer 2)
  3. Sidecar stub implementation (Layer 2 Sidecar, generated once)
  4. CMake integration: both files in target_sources()
  5. Circuit breaker wrapping (if CIRCUIT_BREAKER configured in .mus)

Positive Consequences

  • Extension points are first-class DSL citizens with typed contracts
  • Domain model is not polluted with implementation concerns
  • Layer 2 regeneration guarantee preserved for interface files
  • Sidecar pattern gives users a stable place to write plugin implementations
  • Interface changes are detectable (sentinel → regenerate → .new file for merge)

Negative Consequences

  • Two-territory filesystem (generated/ + sidecars/) adds navigation complexity
  • Sentinel protocol requires discipline: removing the sentinel by accident marks the file as user-modified permanently (until manually re-adding sentinel or deleting)
  • IMPLEMENTS cross-reference validation requires loading all .ddd files before validating .mus plugin declarations

Pros and Cons of the Options

Option 1: EXTENSION_POINT + PLUGIN + Layer 2 Sidecar (chosen)

  • Good, because typed contracts are enforceable by the generator
  • Good, because three-layer invariant preserved for interface files
  • Good, because sidecar protocol is explicit and tooling-enforced
  • Bad, because adds complexity (new DSL construct, new file territory)

Option 2: External plugin registry file

  • Good, because no new DSL construct
  • Bad, because no cross-reference validation against domain model
  • Bad, because interface contracts are implicit (no generator enforcement)

Option 3: Convention-based hooks

  • Good, because no tooling changes
  • Bad, because no contract, no versioning, no generator enforcement — purely convention
