PostgreSQL Quantity Storage Strategy

PostgreSQL Quantity Storage Strategy

  • Status: proposed
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04
  • Tags: units, postgresql, storage, domain-types

Technical Story: IDEA-134 — DSL-Native Units of Measure, Decision 3

Context and Problem Statement

When a DSL entity declares depth QUANTITY<Length, km>, the PostgreSQL generator must decide how to store the quantity value. Two viable strategies exist:

  • Strategy A: Use the postgresql-unit extension, which provides a native unit column type with built-in dimensional analysis (SELECT depth @ 'm' FROM drill_hole)
  • Strategy B: Use CREATE DOMAIN types (e.g., CREATE DOMAIN length_km AS NUMERIC), which are portable across all PostgreSQL installations without extensions

The choice affects portability, query capabilities, deployment complexity, and dimensional safety at the database level.

Decision Drivers

  • Portability: many deployments (managed cloud PostgreSQL) cannot install extensions
  • Dimensional safety: preventing unit confusion at the SQL level
  • Query capabilities: unit conversion in SQL (SELECT depth @ 'km' FROM ...)
  • Deployment complexity: extension installation, version management, CI setup
  • Licensing: extension must be compatible with AGPL-3.0 + Commercial dual license
  • DSL philosophy: project-level configuration, not one-size-fits-all

Considered Options

  1. Domain types (portable, default) — CREATE DOMAIN pressure_mmhg AS NUMERIC
  2. postgresql-unit extension — Native unit column type with dimensional analysis
  3. Both, project-configurable — Default to domain types, opt-in to extension via .mus

Decision Outcome

Chosen option: "Both, project-configurable" (option 3), with postgresql-unit as the primary/default and domain types as a fallback for environments that cannot install extensions. The storage strategy is declared in the .mus project file:

PROJECT "fhir-health" {
    -- Default: postgresql-unit (full dimensional analysis in SQL)
    QUANTITY_STORAGE postgresql_unit;

    -- Fallback for managed cloud PG without extension support:
    -- QUANTITY_STORAGE domain_types;
}

Rationale for making postgresql-unit the default: If a project declares units and dimensions in the DSL, it deserves dimensional safety at every layer — not just in C++ (mp-units) and Proto (typed messages), but also in SQL. Domain types are NUMERIC aliases that silently allow pressure_mmhg + length_km. The postgresql-unit extension catches this at the database level, completing the safety chain. The extension uses the same unit definition database (GNU Units, rooted in NIST/SI) that UCUM and mp-units derive from, ensuring consistency across all layers.

Primary strategy: postgresql-unit extension

-- Extension provides the 'unit' column type with dimensional analysis
CREATE EXTENSION IF NOT EXISTS unit;

CREATE TABLE vital_sign (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    systolic    unit NOT NULL,       -- stores '120 mm[Hg]'
    diastolic   unit NOT NULL,
    temperature unit NOT NULL        -- stores '37.2 Cel'
);

-- Dimensional analysis in SQL:
SELECT systolic @ 'Pa' FROM vital_sign;  -- convert mmHg → Pascal
SELECT temperature @ 'K' FROM vital_sign; -- Celsius → Kelvin

-- Dimensional safety:
SELECT systolic + temperature;  -- ERROR: dimension mismatch

-- Periodic table with real units:
SELECT symbol, atomic_mass FROM element
WHERE atomic_mass > '100 u'::unit;

Fallback strategy: Domain types (for managed cloud PG)

CREATE DOMAIN pressure_mmhg AS NUMERIC;
CREATE DOMAIN temperature_cel AS NUMERIC;
CREATE DOMAIN length_km AS NUMERIC;

CREATE TABLE drill_hole (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    depth       length_km NOT NULL,
    temperature temperature_cel NOT NULL
);

-- No in-SQL conversion — C++ layer handles unit conversion
-- No dimensional safety — domain types are NUMERIC aliases

Licensing Note

The postgresql-unit extension is licensed under the PostgreSQL License (BSD-like), which is compatible with both AGPL-3.0 and Commercial licensing. Muscovite does not bundle the extension — it generates SQL that uses it when the project opts in (default).

Positive Consequences

  • Dimensional safety in SQL by default — pressure + length is a runtime error
  • In-SQL unit conversion: SELECT depth @ 'km' FROM drill_hole
  • Consistency: same unit definitions (NIST/SI) as mp-units (C++) and UCUM (Proto)
  • Values self-describe their unit: '120 mm[Hg]' not just 120
  • Fallback (domain types) works on managed cloud PG without extension support
  • Configuration is explicit in .mus — deployment requirements are auditable

Negative Consequences

  • Two code paths in the PostgreSQL generator (extension vs domain types)
  • Default requires extension installation on every database instance
  • Managed cloud PostgreSQL (RDS, Cloud SQL) may require domain_types fallback
  • Extension version must be tracked and tested
  • Testing must cover both strategies

Pros and Cons of the Options

Option 1: Domain Types Only (Portable)

CREATE DOMAIN pressure_mmhg AS NUMERIC — semantic naming, no extension required.

  • Good, because works on any PostgreSQL installation (including managed cloud)
  • Good, because no extension version management or CI setup
  • Good, because domain types provide semantic naming in DDL and error messages
  • Good, because Sqitch migrations are standard SQL
  • Bad, because no dimensional safety — PostgreSQL treats domains as aliases
  • Bad, because no in-SQL unit conversion (must be done in C++ layer)
  • Bad, because pressure_mmhg + length_km compiles without error in SQL

Option 2: postgresql-unit Extension Only

Native unit column type with built-in dimensional analysis.

  • Good, because full dimensional safety in SQL (mixing dimensions is a runtime error)
  • Good, because in-SQL unit conversion: SELECT depth @ 'km' FROM ...
  • Good, because values self-describe their unit ('120 mm[Hg]')
  • Good, because PostgreSQL License (BSD-like) — compatible with Muscovite licensing
  • Bad, because requires extension installation on every database instance
  • Bad, because managed cloud PostgreSQL (RDS, Cloud SQL) may not support custom extensions
  • Bad, because extension version must be tracked and tested
  • Bad, because single-vendor dependency for a core storage mechanism

Option 3: Both, Project-Configurable

Default to domain types; opt-in to postgresql-unit via .mus project configuration.

  • Good, because portable default — works everywhere out of the box
  • Good, because projects that need dimensional safety can opt in
  • Good, because configuration is explicit and auditable in .mus file
  • Good, because each strategy generates idiomatic SQL for its approach
  • Bad, because two code paths in the PostgreSQL generator
  • Bad, because both strategies must be tested in CI
  • Bad, because project owners must understand the trade-off to choose

AI Agent Guidance

When This Decision Applies

  • When generating PostgreSQL DDL (Sqitch plans) for entities with QUANTITY fields
  • When generating CREATE DOMAIN statements for quantity types
  • When generating SQL that references postgresql-unit extension features
  • When modifying the .mus project file schema to add QUANTITY_STORAGE setting

When This Decision Does NOT Apply

  • When generating protobuf messages (see typed proto messages)
  • When generating C++ code (C++ always uses mp-units regardless of storage strategy)
  • When working with non-quantity fields (TEXT, DECIMAL, DATE, etc.)
  • When working on the QUDT-to-DDD generator tool

Enforcement

  • PostgreSQL generator reads QUANTITY_STORAGE from .mus project configuration
  • Default is domain_types if not specified
  • Generator emits CREATE EXTENSION IF NOT EXISTS unit; only when postgresql_unit is selected
  • Racket tests verify both strategies produce valid SQL
