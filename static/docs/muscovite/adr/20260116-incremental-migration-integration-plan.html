ADR: Incremental Migration Integration Plan

ADR: Incremental Migration Integration Plan

Status: Accepted
Date: 2026-01-16
Context: Schema Evolution Pipeline

Summary

This document describes the plan to achieve incremental database migrations from DDD model changes, ensuring:

  • The database is never dropped and recreated
  • All schema changes go through Sqitch migrations
  • C++ code can be regenerated from scratch at any time (stateless)
  • The DDD model is the single source of truth

Implementation Plan

Phase 1: LOOKUP_TABLE Migration Support (Day 1)

Goal: LOOKUP_TABLE operations integrated into diff and SQL generation

1.1 Add LOOKUP_TABLE operations to diff-model.rkt

;; In diff-entities, detect lookup tables by presence of standard fields
(define (entity-is-lookup-table? ent)
  (define fields (entity-fields ent))
  (define field-names (map field-name fields))
  (and (member 'code field-names)
       (member 'display field-names)
       (member 'is_active field-names)))

;; Generate add-lookup-table-op instead of add-entity-op for lookup tables
(define (diff-entities old-contexts new-contexts)
  (for/list ([ent (find-added-entities old-contexts new-contexts)])
    (if (entity-is-lookup-table? ent)
        (add-lookup-table-op ...)
        (add-entity-op ...))))

1.2 Add SQL generation for lookup tables

;; In sqitch-generator.rkt
[(add-lookup-table-op uuid ctx-uuid name ent schema-name safety)
 (string-append
  (format "-- Create lookup table: ~a.~a\n" schema-name name)
  (entity->create-table-sql ent schema-name)
  "\n\n"
  (format "-- Create index on code\n")
  (format "CREATE UNIQUE INDEX ~a_code_idx ON ~a.~a(code);\n"
          name schema-name name))]

1.3 Files to modify

  • migration/operations.rkt - Re-export from operation-versioning.rkt
  • migration/diff-model.rkt - Detect lookup tables, emit correct ops
  • migration/sqitch-generator.rkt - Handle LOOKUP_TABLE ops

Phase 2: Seed Data Migration (Day 2)

Goal: IMPORT CSV changes → INSERT/UPDATE/DELETE statements

2.1 Track seed data in IR

The parser already captures csv-import structs. Need to:

  1. Include seed file paths in semantic IR
  2. Diff seed data between versions
;; New operation types
(struct add-seed-row-op
  (entity-uuid table-name schema-name row-data safety)
  #:transparent)

(struct update-seed-row-op  
  (entity-uuid table-name schema-name pk-value changes safety)
  #:transparent)

(struct remove-seed-row-op
  (entity-uuid table-name schema-name pk-value safety)
  #:transparent)

2.2 Seed diff algorithm

  1. Extract CSV content from both git refs (via muscovite-version)
  2. Parse CSV → list of row hashes
  3. Compare by PK (code column for lookup tables)
  4. Generate add/update/remove seed ops

2.3 SQL generation

-- add-seed-row-op
INSERT INTO basedata.clinical_status (id, code, display, definition, is_active, sort_order)
VALUES ('uuid-here', 'active', 'Active', 'The condition is active', true, 1);

-- update-seed-row-op  
UPDATE basedata.clinical_status SET display = 'Active Status', sort_order = 2 WHERE code = 'active';

-- remove-seed-row-op
DELETE FROM basedata.clinical_status WHERE code = 'deprecated';

Phase 3: Complete Relationship FK Migrations (Day 2-3)

Goal: add-relationship-op → actual ALTER TABLE ADD CONSTRAINT

3.1 Current state

;; Currently generates:
"-- Add relationship: patient.condition.encounter_id -> encounter.encounter (cardinality: one, required: true)"

3.2 Target state

-- Add FK column if not exists
ALTER TABLE patient.condition 
  ADD COLUMN IF NOT EXISTS encounter_id UUID;

-- Add foreign key constraint  
ALTER TABLE patient.condition 
  ADD CONSTRAINT fk_condition_encounter 
  FOREIGN KEY (encounter_id) 
  REFERENCES encounter.encounter(id);

-- Add NOT NULL if required
ALTER TABLE patient.condition
  ALTER COLUMN encounter_id SET NOT NULL;

3.3 Implementation

Enhance operation->deploy-sql in sqitch-generator.rkt to emit full FK DDL.

Phase 4: sqitch.plan Dependency Management (Day 3)

Goal: Correct topological order in sqitch.plan

4.1 Problem

Current update-sqitch-plan just appends. But migrations may have dependencies:

  • FK migrations depend on both tables existing
  • Seed data depends on table existing
  • API functions depend on table existing

4.2 Solution

;; Extract dependencies from operations
(define (operation-dependencies op)
  (match op
    [(add-relationship-op ... src-ent-name src-schema-name tgt-ent-name tgt-schema-name ...)
     (list (format "~a-table-~a" src-schema-name src-ent-name)
           (format "~a-table-~a" tgt-schema-name tgt-ent-name))]
    [(seed-lookup-table-op ... entity-name schema-name ...)
     (list (format "~a-table-~a" schema-name entity-name))]
    [_ '()]))

;; Write sqitch.plan entry with dependencies
(define (write-plan-entry change-name dependencies)
  (format "~a [~a] ~a Migration <...>\n"
          change-name
          (string-join dependencies " ")
          (current-date-time)))

Phase 5: End-to-End Integration Test (Day 4)

Goal: Prove the pipeline works with real FHIR project git history

5.1 Test scenario

  1. Create git tag v1.0.0 on current FHIR model
  2. Add a new lookup table AppointmentStatus in Basedata context
  3. Commit as v1.1.0
  4. Run: racket migrate.rkt --repo examples/fhir-health --from v1.0.0 --to v1.1.0
  5. Verify:
    • deploy/migrate-v1_0_0-to-v1_1_0.sql contains CREATE TABLE
    • revert/migrate-v1_0_0-to-v1_1_0.sql contains DROP TABLE
    • sqitch deploy succeeds
    • sqitch verify passes

Success Criteria

  1. ✅ Never DROP DATABASE - All changes via ALTER TABLE
  2. ✅ UUID identity - Renamed entities detected, not dropped/recreated
  3. ✅ Lookup tables - CREATE TABLE + seeds in single migration
  4. ✅ Seed data delta - INSERT/UPDATE/DELETE for changed rows only
  5. ✅ FK constraints - Full ALTER TABLE ADD CONSTRAINT
  6. ✅ Revertible - Every deploy has matching revert
  7. ✅ Verifiable - Every migration has verify script
  8. ✅ Dependency order - sqitch.plan respects FK dependencies

Non-Goals (Phase 1)

  • API function migrations (regenerate full API is acceptable)
  • pgTAP test migrations (regenerate is acceptable)
  • C++ code migrations (regenerate is acceptable)
  • gRPC proto migrations (regenerate is acceptable)

These stateless artifacts can be regenerated from the model at any point without data loss.

Risk Mitigation

Risk Mitigation
Destructive ops slip through Safety classification + --force required
Wrong FK order Topological sort in sqitch-generator
Missing revert Always generate deploy/revert/verify together
Seed data conflicts Use UPSERT (ON CONFLICT DO UPDATE)
