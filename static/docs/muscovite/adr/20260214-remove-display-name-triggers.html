Remove display_name PL/pgSQL Triggers — Assemble in C++ Instead

Remove display_name PL/pgSQL Triggers — Assemble in C++ Instead

  • Status: accepted
  • Deciders: jolo, AI agent
  • Date: 2026-02-14
  • Tags: database, triggers, plpgsql, display-name, presentation
  • Supersedes: 20260115-denormalized-display-name-with-triggers

Context and Problem Statement

ADR 20260115 introduced PL/pgSQL triggers to maintain denormalized display_name fields on parent entities (Patient, Practitioner). These triggers fire AFTER INSERT/UPDATE/DELETE on child name entities and update the parent's display_name field.

This directly contradicts ADR 20260120 (remove PL/pgSQL API layer) and ADR 20260120 (domain services as aggregate boundary), which establish that all business logic lives in C++ domain services, not in database procedures or triggers.

The display_name triggers are the only remaining PL/pgSQL code in the generated stack.

Decision Drivers

  • Consistency: ADR 20260120 prohibits stored procedures and PL/pgSQL — no exceptions
  • Simplicity: Removing triggers eliminates cross-table mutation in the database
  • Testability: C++ display logic is unit-testable; PL/pgSQL triggers are not (in our stack)
  • Maintainability: One less generator module to maintain (display-name-trigger.rkt)

Decision Outcome

Remove all PL/pgSQL display_name triggers. The display_name field remains as a regular TEXT column on parent entities. Population and maintenance of this field becomes the responsibility of C++ CommandService code.

What Changes

  1. Delete dslgen/generators/postgres/display-name-trigger.rkt (258 lines)
  2. Remove Phase 8b from dslgen/generators/postgres/sqitch-unified.rkt
  3. Remove generated trigger SQL files from example projects
  4. Supersede ADR 20260115

What Does NOT Change

  • The display_name TEXT field on entities — it remains a valid column
  • PRESENTATION blocks referencing {display_name} — they still work
  • Qt/QML generators reading display_name — unchanged
  • gRPC proto for display_name — unchanged

Future Work (Not in This ADR)

A future feature could generate a display_name() const method on C++ entity structs using the PRESENTATION SHORT format pattern. This is not required for correctness — the field can be populated by application code at insert/update time.

Consequences

Positive

  • Zero PL/pgSQL in generated output — full compliance with ADR 20260120
  • Simpler Sqitch plans (2 fewer changes for fhir-health)
  • One less generator module to maintain
  • Eliminates cross-table trigger mutation

Negative

  • display_name no longer auto-populated by database triggers — application code must set it
  • Existing databases with deployed triggers need a revert migration
