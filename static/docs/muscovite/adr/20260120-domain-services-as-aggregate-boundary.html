Domain Services as Aggregate Boundary

Domain Services as Aggregate Boundary

  • Status: accepted
  • Date: 2026-01-20
  • Tags: ddd, architecture, aggregate, domain-services, cqrs

Context and Problem Statement

The current Muscovite architecture generates independent CRUD APIs for every entity, including child entities within aggregates. This creates an anemic domain model where:

  • Child entities can be modified directly, bypassing the aggregate root
  • No transactional boundary around aggregate operations
  • Business invariants spanning multiple entities cannot be enforced
  • The aggregate root is not the sole access point as DDD prescribes

Example of the problem:

# Current (anemic): Child entity directly accessible
gRPC: CreateOrderLine(order_id, product_id, quantity)
      → OrderLineRepository.insert() → PostgreSQL
      # Order aggregate invariants NOT checked!

# DDD-correct: Access through aggregate root
gRPC: AddLineToOrder(order_id, product_id, quantity)
      → OrderCommandService.add_line() 
      → Validates order state, checks invariants
      → OrderRepository.save(order_with_new_line)

Decision Drivers

  • DDD Compliance: Aggregate root must be sole entry point for modifications
  • Invariant Enforcement: Business rules spanning entities must be validated
  • Transaction Boundaries: All changes to an aggregate in single transaction
  • Testability: Business logic in C++ is testable with Catch2
  • Consistency: All aggregates (including lookups) follow same pattern

Considered Options

  1. Keep current per-entity APIs: Status quo, anemic model
  2. Domain services for multi-entity aggregates only: Partial fix
  3. Domain services for ALL aggregates: Full DDD compliance (chosen)

Decision Outcome

Chosen option: "Domain services for ALL aggregates", because:

  • Consistent architecture: all access patterns follow same model
  • Lookup tables (self-referential aggregates) also get proper encapsulation
  • Simplifies gRPC service generation (always route through services)
  • Enables future features: caching, validation, event emission

Implementation

CommandService (generated for each aggregate):

class PatientCommandService {
public:
    PatientCommandService(
        PatientCommandRepository& patient_repo,
        EncounterCommandRepository& encounter_repo,  // child
        ConnectionPool& pool
    );

    // Aggregate root operations
    Patient create(const CreatePatientRequest& req);
    Patient update(const UpdatePatientRequest& req);
    void remove(const boost::uuids::uuid& id);

    // Child entity operations (through root)
    Encounter add_encounter(const boost::uuids::uuid& patient_id, 
                           const AddEncounterRequest& req);
    void remove_encounter(const boost::uuids::uuid& patient_id,
                         const boost::uuids::uuid& encounter_id);

private:
    // All operations wrapped in single transaction
    template<typename F>
    auto execute(F&& func) {
        return muscovite::dba::execute(pool_, std::forward<F>(func));
    }
};

QueryService (generated for each aggregate):

class PatientQueryService {
public:
    PatientQueryService(PatientQueryRepository& repo);

    std::optional<Patient> find_by_id(const boost::uuids::uuid& id);
    std::vector<Patient> find_all(const PageRequest& page);
    std::vector<Patient> find_active();  // From SPECIFICATION
};

Lookup Tables follow same pattern:

// Even simple lookups get services for consistency
class ClinicalStatusCommandService { /* create, update, delete */ };
class ClinicalStatusQueryService { /* find_by_id, find_all, find_by_code */ };

Positive Consequences

  • Proper DDD: Aggregate root enforces invariants
  • Single transaction: All aggregate changes atomic
  • Testable: Business logic in C++, tested with Catch2
  • Consistent: All aggregates (including lookups) same pattern
  • CQRS-ready: Command/query paths already separated

Negative Consequences

  • More generated code (service classes for each aggregate)
  • Slight indirection for simple lookups
    • Mitigation: Services are thin wrappers, minimal overhead
  • Child entities not directly accessible
    • Mitigation: This is intentional—DDD compliance

AI Agent Guidance

When This Decision Applies

  • When generating gRPC service implementations
  • When generating C++ domain services
  • When designing aggregate operations
  • When adding child entity operations

When This Decision Does NOT Apply

  • Database schema generation (unchanged)
  • Direct database queries for reporting/analytics (use views)
  • Internal repository operations (services call repositories)

Enforcement

  • gRPC services MUST route through domain services
  • Child entity operations MUST go through aggregate root service
  • Repositories accept pqxx::work& for transaction injection
  • Code review rejects direct child entity access
