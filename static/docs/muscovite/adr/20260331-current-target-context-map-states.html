Current/Target Context Map States

Current/Target Context Map States

  • Status: draft
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-31
  • Tags: dsl, ddd, strategic, context-map, brownfield, grammar, ir
  • Supersedes: none

Technical Story: RFC-001 Extension 2 — add CURRENT/TARGET state to MAP blocks for brownfield modernization.

Context and Problem Statement

Brownfield modernization requires modeling two realities: the legacy system as it exists today, and the target architecture being built. Currently the MAP block has no concept of state — it models one reality. During extraction, the AI discovers the current landscape. The architect defines the target. But there is no formal way to express both states and track convergence from CURRENT to TARGET topology.

Decision Drivers

  • Brownfield consulting engagements need before/after visualization
  • Must be backward compatible — MAP without STATE retains current behavior
  • Both maps must coexist in the same project for convergence tracking
  • Grammar-first mandate (ADR 20260211)

Considered Options

Option 1: STATE clause on MAP block (chosen)

MAP STATE CURRENT { ... } and MAP STATE TARGET { ... } as two separate blocks.

  • Good, because explicit and readable
  • Good, because backward compatible (MAP without STATE is UNSPECIFIED)
  • Good, because generators can select which state to render
  • Bad, because contexts may appear in both maps with different relationships

Option 2: Versioned MAP blocks

MAP VERSION "2024-Q1" { ... } with arbitrary version strings.

  • Good, because supports more than two states
  • Bad, because no semantic meaning — generators can't distinguish current from target
  • Bad, because opens the door to unbounded proliferation

Option 3: Annotations on individual relationships

PatientAdmin -> Billing : SHARED_KERNEL @state CURRENT

  • Good, because fine-grained per-relationship
  • Bad, because clutters the relationship syntax
  • Bad, because hard to visualize as a coherent map

Decision Outcome

Chosen option: "STATE clause on MAP block", because it provides exactly the two states needed for brownfield modernization with clear semantics and full backward compatibility.

DSL Syntax

MAP STATE CURRENT {
    PatientAdmin -> Billing : SHARED_KERNEL VIA SHARED_DB
        DOC "Legacy: both contexts read the same patient table";
    Reporting <- PatientAdmin : CONFORMIST VIA SHARED_DB
        DOC "Legacy: reporting queries patient tables directly";
}

MAP STATE TARGET {
    PatientAdmin -> Billing : CUSTOMER_SUPPLIER VIA GRPC
        DOC "Target: billing consumes patient events via gRPC";
    Reporting <- PatientAdmin : ANTICORRUPTION_LAYER VIA EVENTS
        DOC "Target: reporting subscribes to domain events";
}

Rules

  • A project may have at most one CURRENT map and one TARGET map
  • Both are optional — greenfield projects have only TARGET (or no state)
  • MAP without STATE retains current behavior (state = UNSPECIFIED)
  • Contexts in CURRENT not in TARGET are legacy-only (expected to be retired)
  • Contexts in TARGET not in CURRENT are new (being built)

PEG Grammar Changes

MapBlock <- 'MAP' MapState? '{' MapEntry* '}'
MapState <- 'STATE' ('CURRENT' / 'TARGET')

IR Implications

  • MapBlock gains a state field: CURRENT, TARGET, or UNSPECIFIED
  • Both maps coexist in the IR; generators select which to render

Generator Implications

  • Documentation: side-by-side or overlay diagrams showing current vs. target. Diff highlighting for changed relationships. Migration progress tracking.
  • Heatmap: convergence heatmap — contexts that have completed transition vs. those still in CURRENT topology.
  • Validation: warn if a CURRENT relationship type is SHARED_DB and the TARGET relationship for the same context pair is still SHARED_DB (no improvement planned).

Positive Consequences

  • Brownfield modernization has formal before/after representation
  • Convergence tracking is derivable from the model
  • AI extraction agents can populate CURRENT maps automatically

Negative Consequences

  • Two maps to maintain for brownfield projects
  • Validation rules become more complex (cross-map consistency)

AI Agent Guidance

When This Decision Applies

  • When modeling brownfield systems with legacy and target architectures
  • When generating context map documentation with migration progress

When This Decision Does NOT Apply

  • Greenfield projects (single MAP block without STATE is sufficient)
  • The MAP block without STATE clause — existing behavior unchanged

Enforcement

  • PEG grammar enforces STATE keyword placement
  • Parser validation: at most one CURRENT and one TARGET map per project
  • Generators select map state via configuration flag
