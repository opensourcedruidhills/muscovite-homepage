Migration Extractor Generator Architecture

Migration Extractor Generator Architecture

  • Status: accepted
  • Date: 2026-02-23
  • Tags: brownfield, migration, extractor, generator, pqxx, nats, cursor, batching
  • Deciders: Johannes Lochmann

Technical Story: The first execution stage of the brownfield migration factory (ADR 20260217) is the Extractor: a generated C++ binary that reads rows from a legacy PostgreSQL database and emits typed JSON envelopes to a NATS JetStream subject. This ADR specifies how the extractor generator works and what code it produces.

Context and Problem Statement

The migration factory pipeline starts at the legacy database. The extractor must:

  1. Connect to the legacy PostgreSQL instance (separate connection string from the target DB)
  2. Read all rows from MAPPING.SOURCE table in manageable batches (cursor-based, not LIMIT/OFFSET which is unstable on live tables)
  3. For each row, construct a typed JSON envelope (ADR 20260217-typed-json) with field metadata
  4. Publish the envelope to a NATS JetStream subject
  5. Be idempotent: re-running the extractor on a row already migrated must produce the same target UUID (no duplicate inserts)
  6. Support resumability: if extraction is interrupted, it must be possible to resume from the last committed batch

Without a generator, every migration project implements this from scratch — inconsistently, untested, and often without idempotency.

Decision Drivers

  • All migration code is derived from MAPPING blocks in .ddd files (ADR 20260217)
  • CommandRepository is the mandated persistence path (ADR 20260120) — but the extractor is a read-only source-side component; it writes only to NATS, not to the target DB
  • Cursor-based batching is mandatory over LIMIT/OFFSET to handle live tables without row drift
  • Idempotency via UUIDv5 with namespace UUID from MAPPING @uuid + legacy PK string (ADR 20260217)
  • The generated extractor binary must be standalone — it runs against the legacy DB without the full Muscovite stack
  • NatsMessageBroker from framework/dba/ is the mandated NATS client — no new clients created

Considered Options

  1. Generated standalone C++ binary per context — one {context}-extractor binary that handles all MAPPING blocks in that context; configured via INI/env vars
  2. Generated library + shared bootstrap main — generator emits only the per-entity extraction logic as functions; a common main() from the framework wires them together
  3. Racket CLI tool (no C++ generation) — an interpreted Racket script using the db library reads legacy DB and publishes to NATS

Decision Outcome

Chosen option: Option 1 — Generated standalone C++ binary per context, because:

  • Deployed independently of the main application (extraction runs before the target stack is live)
  • Compiles against the same Conan toolchain — no runtime interpreter dependency on migration host
  • Follows generator patterns established by domain service generators
  • One binary per context maps cleanly to one NATS stream per context

The generator emits a CMake executable target {context}-extractor alongside the existing DBA targets.

Generated File Structure

generated/
└── migration/
    ├── include/{project}/migration/{context}/
    │   ├── {entity}_extractor.hpp      # Per-entity extraction class
    │   └── {entity}_envelope.hpp       # Typed JSON envelope struct + serializers
    ├── src/{context}/
    │   ├── {entity}_extractor.cpp
    │   └── {context}_extractor_main.cpp  # main() — connects, iterates MAPPING blocks
    └── CMakeLists.txt

Cursor-Based Batching Strategy

// Generated pattern for LegacyPatientImport
void PatientExtractor::extract(pqxx::connection& conn,
                                NatsMessageBroker& nats,
                                const ExtractionConfig& cfg) {
    pqxx::nontransaction txn(conn);
    // Declare server-side cursor — stable across batches
    txn.exec(fmt::format(
        "DECLARE migration_cursor_{} CURSOR FOR "
        "SELECT patient_id, medical_num, first_name, email_addr, last_visit, balance "
        "FROM legacy_ehr.patients", cfg.run_uuid));

    while (true) {
        auto rows = txn.exec(fmt::format(
            "FETCH {} FROM migration_cursor_{}", cfg.batch_size, cfg.run_uuid));
        if (rows.empty()) break;

        for (const auto& row : rows) {
            auto envelope = build_envelope(row, cfg);
            nats.publish("migration.extract.patient_care.patient",
                         envelope.to_json().dump());
        }
    }
    txn.exec(fmt::format("CLOSE migration_cursor_{}", cfg.run_uuid));
}

Idempotency via UUIDv5

// Target UUID is deterministic: UUIDv5(mapping_namespace_uuid, legacy_pk_string)
// mapping_namespace_uuid comes from MAPPING @uuid annotation
boost::uuids::uuid target_id = muscovite::types::uuid_v5(
    mapping_namespace,                   // from MAPPING @uuid "a1b2c3d4-..."
    row["patient_id"].as<std::string>()  // legacy PK as string seed
);

Re-running extraction never produces a different target UUID for the same source row.

NATS Subject and Stream Configuration

  • Publish subject: migration.extract.{sql_id(context)}.{sql_id(entity)}
  • JetStream stream: MIGRATION_{SQL_ID(context)}
  • Message retention: LimitsPolicy with MaxAge = 7 days (configurable via INI)

Positive Consequences

  • Migration code is 100% generated from MAPPING blocks — no hand-written extraction code
  • Cursor-based batching is safe on live tables (no row drift between batches)
  • Idempotency via UUIDv5 makes re-extraction safe — exactly-once semantics at the target
  • Standalone binary: extraction can run before the target application is deployed
  • Follows established DBA generator patterns — low cognitive overhead for contributors

Negative Consequences

  • C++ binary requires Conan/CMake build on the migration host
  • Cursor lifetime is scoped to one connection — network interruption requires full cursor restart (resumability is batch-level, not row-level)
  • Source is PostgreSQL-only in v1.30.0; MySQL/SQL Server sources need new generators

Constraints

  • Extractor MUST NOT write to the target database (ADR 20260120 CommandRepository is transformer's job)
  • Extractor MUST NOT use LIMIT/OFFSET pagination on production source tables
  • Idempotency namespace UUID MUST come from MAPPING @uuid annotation — not generated at runtime
  • Generated code MUST carry dual-license SPDX header

AI Agent Guidance

When implementing the extractor generator:

  • New file: dslgen/generators/cpp/migration-extractor.rkt
  • Follow patterns from dslgen/generators/cpp/dba/command-repository.rkt for pqxx query generation
  • MAPPING block IR: (bounded-context-mappings context) returns list of mapping structs
  • For each mapping: generate {entity}_extractor.hpp/cpp using (mapping-source-table m), (mapping-fields m), (mapping-uuid m) for namespace
  • Cursor name: "migration_cursor_" + run_uuid_env_var — use getenv("MIGRATION_RUN_UUID") or generate at startup
  • NatsMessageBroker: include framework/dba/include/muscovite/dba/nats_message_broker.hpp
  • CMake target: emit add_executable({context}-extractor ...) in migration/CMakeLists.txt
  • Test: generator test verifying SQL cursor DECLARE statement is emitted with correct table name from MAPPING SOURCE
