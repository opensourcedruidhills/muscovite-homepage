Quantity Storage Unit Convention

Quantity Storage Unit Convention

  • Status: proposed
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04
  • Tags: units, storage, conversion, convention

Technical Story: IDEA-134 — DSL-Native Units of Measure, Decision 4

Context and Problem Statement

When a DSL entity declares depth QUANTITY<Length, km>, the value must be stored somewhere — in the database, in protobuf messages, in C++ structs. The question is: in which unit is the numeric value stored?

Two conventions are possible:

  1. Declared unit: store in the unit the field declares (km). The database column contains kilometres, the proto message carries kilometres, and conversion happens at layer boundaries when a different unit is needed.
  2. SI base unit: always store in the SI base unit (m for length, Pa for pressure, kg for mass). The database always contains metres, and km is only a display concern.

This convention affects every generator (PostgreSQL, proto, C++, QML) and every converter.

Decision Drivers

  • User intention: the DSL author chose km for a reason — it matches their domain
  • Human readability: database values should be understandable without conversion tables
  • Conversion simplicity: fewer conversions means fewer opportunities for rounding errors
  • Canonical form: a single storage unit eliminates ambiguity across systems
  • Domain expectations: geoscientists expect depth in km, clinicians expect blood pressure in mmHg
  • Precision: some units have exact representations that SI base units do not (e.g., mmHg = 133.322 Pa)

Considered Options

  1. Store in declared unit — depth QUANTITY<Length, km> stores kilometres
  2. Always store in SI base unit — depth QUANTITY<Length, km> stores metres

Decision Outcome

Chosen option: "Store in declared unit" (option 1), because it preserves user intention, produces human-readable database values, and avoids unnecessary precision loss from round-trip conversions.

The convention is:

  • Database: depth column contains the numeric value in kilometres
  • Protobuf: Length message carries the value with LENGTH_UNIT_KILOMETRE enum
  • C++: mp_units::quantity<mp_units::si::kilo<mp_units::si::metre>, double> — mp-units tracks the unit at compile time; conversion is explicit
  • QML: displays the value with the declared unit symbol

Conversion happens at boundaries, not at storage:

DSL: depth QUANTITY<Length, km>
                    │
            ┌───────┴────────┐
            ▼                ▼
  PostgreSQL: 3.5          Proto: { value: 3.5,
  (length_km column)               unit: KILOMETRE,
                                    ucum_code: "km" }
            │                │
            └───────┬────────┘
                    ▼
  C++: quantity<kilo<metre>>{3.5}
                    │
            ┌───────┴────────┐
            ▼                ▼
  Display: "3.5 km"    Calculation: auto-converted
                       by mp-units when mixing units

When a C++ function needs metres (e.g., for a calculation), mp-units handles the conversion implicitly and type-safely:

auto depth_km = drill_hole.depth;           // quantity<kilo<metre>>{3.5}
auto depth_m  = depth_km.in(si::metre);     // quantity<metre>{3500.0}

Positive Consequences

  • Database values are human-readable: a geoscientist sees 3.5 in the depth column and knows it means 3.5 km (the domain type name confirms: length_km)
  • No surprise conversions: what you declare is what you get
  • No precision loss from unnecessary round-trip conversions
  • Domain experts can query the database directly without mental conversion
  • Proto messages carry the declared unit — receivers know what they are getting
  • mp-units handles cross-unit calculations safely at compile time

Negative Consequences

  • Different fields of the same dimension may store in different units (depth in km, diameter in mm) — comparison requires explicit conversion
  • Aggregation queries across fields with different units require explicit conversion in SQL
  • Proto consumers must check the unit enum to interpret the value correctly
  • Changing a field's declared unit requires a data migration

Pros and Cons of the Options

Option 1: Store in Declared Unit

The numeric value in the database, proto, and C++ is in the unit the DSL field declares.

  • Good, because preserves user intention — QUANTITY<Length, km> means kilometres everywhere
  • Good, because human-readable database values (domain experts can SELECT depth FROM ...)
  • Good, because no unnecessary conversions — value enters and exits in the same unit
  • Good, because avoids precision loss from round-trip conversion (e.g., mmHg → Pa → mmHg)
  • Good, because proto enum documents what unit the value is in
  • Bad, because mixed units in same dimension require explicit conversion for comparison
  • Bad, because changing declared unit requires data migration
  • Bad, because aggregation queries across different units are harder

Option 2: Always Store in SI Base Unit

Every quantity is converted to SI base before storage: km → m, mmHg → Pa, mg → kg.

  • Good, because single canonical form — all lengths are in metres, no ambiguity
  • Good, because cross-field comparison is trivial (same unit, always)
  • Good, because simpler converter logic (always convert to/from SI base)
  • Good, because aggregation queries are straightforward
  • Bad, because database values are unintuitive (blood pressure as 15999.86 Pa instead of 120 mmHg)
  • Bad, because user intention lost — the DSL said mmHg but the DB stores Pa
  • Bad, because precision loss: 120 mmHg → 15999.8594... Pa → 119.9999... mmHg
  • Bad, because domain experts cannot query the database without conversion tables
  • Bad, because every read/write crosses a conversion boundary (always converting)

AI Agent Guidance

When This Decision Applies

  • When generating PostgreSQL columns for QUANTITY fields (store in declared unit)
  • When generating proto converter functions (proto value is in declared unit)
  • When generating C++ struct members for QUANTITY fields (mp-units type uses declared unit)
  • When generating database-to-C++ type converters (type_converters)
  • When generating QML display code for quantity fields

When This Decision Does NOT Apply

  • When performing calculations inside C++ business logic (mp-units auto-converts as needed)
  • When the postgresql-unit extension is used (it stores value + unit together, self-describing)
  • When working on the QUDT-to-DDD standard library generator (that defines units, not storage)

Enforcement

  • PostgreSQL generator creates domain types named with the declared unit (length_km)
  • Proto generator sets the enum value matching the declared unit
  • C++ generator uses mp-units type alias with the declared unit
  • Converter generators produce conversion code only at layer boundaries
  • Changing a field's declared unit in the DSL triggers a semantic migration (ADR 20260109)
