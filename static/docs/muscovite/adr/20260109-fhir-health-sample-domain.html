Technical Story: Establish a well-defined, machine-readable sample domain in a separate git repository to test and validate the schema evolution pipeline (ADRs 1-7) with clean version history.

Context and Problem Statement

The geosciences project is comprehensive (26 bounded contexts, 331 entities) but too large for iterative development and testing of the schema evolution system. We need a smaller, well-defined domain that:

  1. Is machine-readable - Has formal specifications we can validate against
  2. Is well-documented - Official documentation for every element
  3. Has realistic complexity - Multiple aggregates, relationships, value objects
  4. Exercises all ADR features - UUIDs, diffs, migrations, policies, verification
  5. Has isolated git history - Version history independent from muscovite development

HL7 FHIR (Fast Healthcare Interoperability Resources) is ideal because:

  • Industry-standard healthcare data model
  • Machine-readable JSON/XML definitions
  • Extensive official documentation
  • Small enough for rapid iteration, rich enough to be meaningful

Git Submodule Approach: The schema evolution system (ADR 2.5) uses git to extract model versions. Testing requires:

  • Creating deliberate "v1.0", "v1.1", "v2.0" commits
  • Running muscovite-version against these commits
  • Diffing versions to generate migrations

If FHIR sample lives in muscovite repo, its commits are interleaved with muscovite development. A separate repository as git submodule provides clean, testable version history.

Decision Drivers

  • Need a test domain for schema evolution pipeline
  • Must have machine-readable definitions for validation
  • Must be small enough to iterate quickly
  • Must exercise: aggregates, entities, value objects, relationships, reference data
  • Must support semantic diffing and migration scenarios
  • Must have isolated git history for version extraction testing

Considered Options

  • Subset of geosciences: Rejected - still too coupled to main project, pollutes git history
  • Generic e-commerce: Rejected - no machine-readable standard
  • FHIR subset in muscovite repo: Rejected - git history interleaved with framework development
  • FHIR subset as git submodule: Accepted - isolated history, formal spec, right complexity
  • OpenAPI Petstore: Rejected - too simple, no aggregate complexity

Decision Outcome

Chosen option: "FHIR subset as git submodule", implementing a focused healthcare domain in a separate repository (fhir-health-sample) linked to muscovite via git submodule.

Git Submodule Setup

# In muscovite repository
cd ${PROJECT_ROOT}

# Add as submodule (after creating fhir-health-sample repo)
git submodule add https://github.com/your-org/fhir-health-sample.git examples/fhir-health

# Clone muscovite with submodule
git clone --recurse-submodules https://github.com/your-org/muscovite.git

# Update submodule to specific version
cd examples/fhir-health
git checkout v1.0.0

Version History Strategy

The fhir-health-sample repo will have tagged releases for testing:

Tag Description Migration Test
v1.0.0 Initial: Security + Basedata + Patient Baseline
v1.1.0 Add middle_name to Patient Safe: add nullable field
v1.2.0 Add Practitioner context Safe: add entity
v1.3.0 Add Organization context Safe: add entity

This enables testing muscovite-version against real git history:

# Extract v1.0.0 model
muscovite-version --repo examples/fhir-health --ref v1.0.0 --format sexpr > v1.sexpr

# Extract v2.0.0 model  
muscovite-version --repo examples/fhir-health --ref v2.0.0 --format sexpr > v2.sexpr

# Diff in Racket
racket -e '(diff-model (read-ir "v1.sexpr") (read-ir "v2.sexpr"))'

Positive Consequences

  • Rapid iteration on schema evolution pipeline
  • Machine-readable validation against FHIR spec
  • Clear documentation for every model element
  • Realistic complexity without overwhelming size
  • Isolated from geosciences project changes
  • Clean git history for version extraction testing
  • Controlled version tags for migration scenario testing

Negative Consequences

  • Healthcare domain may be unfamiliar to some contributors
  • FHIR spec is extensive; must carefully limit scope
  • Not directly related to geosciences use case
  • Additional repository to maintain
  • Submodule complexity for new contributors

CONTEXT Basedata @uuid "fhir-basedata-0000-0000-000000000001" {

    -- Administrative Gender (http://hl7.org/fhir/administrative-gender)
    -- Only reference data needed for Patient in v1.0.0
    
    ENTITY AdministrativeGender @uuid "fhir-basedata-0001-0001-000000000001"
    IDENTIFIED_BY id (
        id          UUID        NOT NULL,
        code        VARCHAR(20) NOT NULL @unique,
        display     TEXT        NOT NULL,
        definition  TEXT
    )
    DOC "Administrative gender - not clinical gender";
    
    AGGREGATE AdministrativeGender @uuid "fhir-basedata-0001-0002-000000000001"
    ROOT AdministrativeGender
    DOC "Gender value set";
    
    IMPORT_JSON "../../seeds/gender.json" INTO AdministrativeGender;
}

Phase 7: Encounter Aggregate

contexts/encounter.ddd:

CONTEXT Encounter @uuid "fhir-encounter-0000-0000-000000000001" {

    ENTITY Encounter @uuid "fhir-encounter-0001-0001-000000000001"
    IDENTIFIED_BY id (
        id              UUID        NOT NULL,
        status_id       UUID        NOT NULL,           -- FK to Basedata.EncounterStatus
        type_id         UUID,                           -- FK to Basedata.EncounterType
        patient_id      UUID        NOT NULL,           -- FK to Patient
        
        -- Time period (embedded)
        period_start    TIMESTAMPTZ NOT NULL,
        period_end      TIMESTAMPTZ,
        
        -- Location (simple for prototype)
        location_name   TEXT,
        
        -- Reason (JSONB array of CodeableConcept)
        reason_codes    JSONB
    )
    RELATIONSHIPS {
        status -> EncounterStatus REQUIRED ONE;
        type -> EncounterType OPTIONAL ONE;
        patient -> Patient REQUIRED ONE;
    }
    DOC "An interaction between a patient and healthcare provider(s)";
    
    -- Participant join table (Encounter <-> Practitioner)
    ENTITY EncounterParticipant @uuid "fhir-encounter-0002-0001-000000000001"
    IDENTIFIED_BY id (
        id              UUID        NOT NULL,
        encounter_id    UUID        NOT NULL,
        practitioner_id UUID        NOT NULL,
        role_id         UUID                            -- Role in this encounter
    )
    RELATIONSHIPS {
        encounter -> Encounter REQUIRED ONE;
        practitioner -> Practitioner REQUIRED ONE;
        role -> PractitionerRole OPTIONAL ONE;
    }
    DOC "Practitioner participation in an encounter";
    
    AGGREGATE Encounter @uuid "fhir-encounter-0001-0002-000000000001"
    ROOT Encounter
    CONTAINS EncounterParticipant
    DOC "Encounter aggregate - clinical visit/admission";
}

Phase 8: Clinical Context (Condition + Observation)

contexts/clinical.ddd:

CONTEXT Clinical @uuid "fhir-clinical-0000-0000-000000000001" {

    -- Condition (diagnosis, problem)
    ENTITY Condition @uuid "fhir-clinical-0001-0001-000000000001"
    IDENTIFIED_BY id (
        id              UUID        NOT NULL,
        patient_id      UUID        NOT NULL,           -- FK to Patient
        encounter_id    UUID,                           -- FK to Encounter (when diagnosed)
        code_id         UUID        NOT NULL,           -- FK to Basedata.ConditionCode
        
        clinical_status VARCHAR(20) NOT NULL,           -- active | recurrence | relapse | inactive | remission | resolved
        verification    VARCHAR(20) NOT NULL,           -- unconfirmed | provisional | differential | confirmed
        onset_date      DATE,
        abatement_date  DATE,
        recorded_date   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
        
        note            TEXT
    )
    RELATIONSHIPS {
        patient -> Patient REQUIRED ONE;
        encounter -> Encounter OPTIONAL ONE;
        code -> ConditionCode REQUIRED ONE;
    }
    DOC "A clinical condition, problem, or diagnosis";
    
    AGGREGATE Condition @uuid "fhir-clinical-0001-0002-000000000001"
    ROOT Condition
    DOC "Condition aggregate";

    -- Observation (vital signs, lab results)
    ENTITY Observation @uuid "fhir-clinical-0002-0001-000000000001"
    IDENTIFIED_BY id (
        id              UUID        NOT NULL,
        patient_id      UUID        NOT NULL,           -- FK to Patient
        encounter_id    UUID,                           -- FK to Encounter
        code_id         UUID        NOT NULL,           -- FK to Basedata.ObservationCode
        
        status          VARCHAR(20) NOT NULL,           -- registered | preliminary | final | amended | cancelled
        effective_time  TIMESTAMPTZ NOT NULL,
        issued          TIMESTAMPTZ NOT NULL DEFAULT NOW(),
        
        -- Value (one of these will be populated)
        value_quantity  JSONB,                          -- Quantity value object
        value_string    TEXT,
        value_boolean   BOOLEAN,
        
        interpretation  VARCHAR(20),                    -- normal | abnormal | high | low | critical
        note            TEXT
    )
    RELATIONSHIPS {
        patient -> Patient REQUIRED ONE;
        encounter -> Encounter OPTIONAL ONE;
        code -> ObservationCode REQUIRED ONE;
    }
    DOC "Measurements and simple assertions about a patient";
    
    AGGREGATE Observation @uuid "fhir-clinical-0002-0002-000000000001"
    ROOT Observation
    DOC "Observation aggregate";
}
