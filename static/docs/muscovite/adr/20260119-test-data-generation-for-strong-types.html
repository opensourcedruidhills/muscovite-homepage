ADR: Test Data Generation for Strong Types

ADR: Test Data Generation for Strong Types

  • Status: Accepted
  • Date: 2026-01-19
  • Deciders: Architecture Team
  • Technical Story: Generate valid test data for strong-typed fields across all layers

Context and Problem Statement

With the introduction of strongly-typed fields across PostgreSQL, gRPC, C++, and Qt layers (see Strong Types Across All Layers), test data generation becomes more complex:

  1. pgTAP tests need valid domain values that pass CHECK constraints
  2. C++ unit tests need values that pass wrapper class validation
  3. gRPC integration tests need properly serialized strong type messages
  4. Qt widget tests need values that pass QValidator checks
  5. Seed data (CSV/JSON) needs valid values for all strong types

Current test generation uses simple placeholders like 'test_email_' || v_idx which will fail CHECK constraints like VALUE ~ '^[^@]+@[^@]+\.[^@]+$'.

Decision Drivers

  • Deterministic - Same test run produces same values (for debugging)
  • Valid - Generated values pass all validation constraints
  • Unique - Support for unique constraints (each call generates distinct value)
  • Realistic - Values look like real data for better debugging
  • Configurable - Allow test-specific overrides
  • Cross-layer consistency - Same patterns in pgTAP, C++, Qt tests

Decision Outcome

Implement a type-aware test data generation system that:

  1. Parses CHECK constraint patterns from STRONG_TYPE definitions
  2. Generates valid values matching those patterns
  3. Provides deterministic generators with optional index/seed parameters
  4. Works identically across PostgreSQL (PL/pgSQL), C++, and Qt

1. DSL Strong Type with Test Data Hints

Extend STRONG_TYPE syntax with optional TEST DATA clause:

STRONG_TYPE Email BASED_ON TEXT
    CHECK email_format: VALUE ~ '^[^@]+@[^@]+\.[^@]+$'
    TEST DATA pattern: '{name}@{domain}.com'
             names: ['alice', 'bob', 'charlie', 'diana', 'eve']
             domains: ['example', 'test', 'sample']
    DOC "Valid email address";

STRONG_TYPE Phone BASED_ON TEXT
    CHECK phone_format: VALUE ~ '^\+?[0-9]{1,3}[-\s]?[0-9]{3}[-\s]?[0-9]{4}$'
    TEST DATA pattern: '+1-{area}-{number}'
             area: sequence(100, 999)
             number: sequence(1000, 9999)
    DOC "US phone number";

STRONG_TYPE PostalCode BASED_ON TEXT
    CHECK postal_format: VALUE ~ '^[0-9]{5}(-[0-9]{4})?$'
    TEST DATA pattern: '{zip5}'
             zip5: sequence(10000, 99999)
    DOC "US postal code (ZIP)";

STRONG_TYPE PositiveDecimal BASED_ON DECIMAL(18,4)
    CHECK positive: VALUE > 0
    TEST DATA range: (0.01, 9999.99)
    DOC "Positive decimal";

STRONG_TYPE Latitude BASED_ON DECIMAL(9,6)
    CHECK valid_lat: VALUE >= -90 AND VALUE <= 90
    TEST DATA range: (-90.0, 90.0)
    DOC "Geographic latitude";

2. PostgreSQL Test Data Functions

Generate PL/pgSQL functions for each strong type:

-- In basedata schema (test utilities)
CREATE OR REPLACE FUNCTION basedata.test_email(p_index INTEGER DEFAULT 0)
RETURNS basedata.email AS $$
DECLARE
    v_names TEXT[] := ARRAY['alice', 'bob', 'charlie', 'diana', 'eve', 'frank', 'grace', 'henry'];
    v_domains TEXT[] := ARRAY['example', 'test', 'sample', 'demo'];
    v_name TEXT;
    v_domain TEXT;
BEGIN
    v_name := v_names[(p_index % array_length(v_names, 1)) + 1];
    v_domain := v_domains[(p_index / array_length(v_names, 1) % array_length(v_domains, 1)) + 1];
    RETURN (v_name || p_index || '@' || v_domain || '.com')::basedata.email;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

CREATE OR REPLACE FUNCTION basedata.test_phone(p_index INTEGER DEFAULT 0)
RETURNS basedata.phone AS $$
BEGIN
    RETURN ('+1-' || lpad((100 + (p_index % 900))::TEXT, 3, '0') || '-' || 
            lpad((1000 + (p_index % 9000))::TEXT, 4, '0'))::basedata.phone;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

CREATE OR REPLACE FUNCTION basedata.test_postal_code(p_index INTEGER DEFAULT 0)
RETURNS basedata.postal_code AS $$
BEGIN
    RETURN lpad((10000 + p_index)::TEXT, 5, '0')::basedata.postal_code;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

CREATE OR REPLACE FUNCTION basedata.test_positive_decimal(p_index INTEGER DEFAULT 0)
RETURNS basedata.positive_decimal AS $$
BEGIN
    RETURN ((p_index % 9999) + 0.01 + (p_index % 100) / 100.0)::basedata.positive_decimal;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

CREATE OR REPLACE FUNCTION basedata.test_latitude(p_index INTEGER DEFAULT 0)
RETURNS basedata.latitude AS $$
BEGIN
    -- Distribute across valid range -90 to +90
    RETURN (-90.0 + (p_index % 18000) / 100.0)::basedata.latitude;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

CREATE OR REPLACE FUNCTION basedata.test_longitude(p_index INTEGER DEFAULT 0)
RETURNS basedata.longitude AS $$
BEGIN
    -- Distribute across valid range -180 to +180
    RETURN (-180.0 + (p_index % 36000) / 100.0)::basedata.longitude;
END;
$$ LANGUAGE plpgsql IMMUTABLE;

3. pgTAP Test Usage

Update pgTAP generator to use test data functions:

-- Before (fails CHECK constraint):
INSERT INTO patient (id, email) VALUES (gen_random_uuid(), 'test_email_1');

-- After (uses generated test function):
INSERT INTO patient (id, email) 
VALUES (gen_random_uuid(), basedata.test_email(1));

-- Full example with multiple strong types:
DO $$
DECLARE
    v_patient patient%ROWTYPE;
    v_idx INTEGER := 1;
BEGIN
    v_patient := patient_api.patient_insert(
        NULL,  -- id (auto-generated)
        'John Doe',  -- name
        basedata.test_email(v_idx),      -- email
        basedata.test_phone(v_idx),      -- phone
        basedata.test_postal_code(v_idx) -- postal_code
    );
    
    PERFORM ok(v_patient.email ~ '^[^@]+@[^@]+\.[^@]+$', 'Email matches pattern');
    PERFORM ok(v_patient.phone ~ '^\+1-[0-9]{3}-[0-9]{4}$', 'Phone matches pattern');
END;
$$;

4. C++ Test Data Generators

Generate C++ test utilities in include/muscovite/types/test_data.hpp:

#pragma once
#include <muscovite/types/email.hpp>
#include <muscovite/types/phone.hpp>
#include <muscovite/types/postal_code.hpp>
#include <muscovite/types/decimal.hpp>

namespace muscovite::types::test {

class TestDataGenerator {
public:
    explicit TestDataGenerator(uint64_t seed = 0) : index_(seed) {}
    
    // Deterministic generators (same index → same value)
    [[nodiscard]] Email email(uint64_t index) const {
        static constexpr std::array names = {"alice", "bob", "charlie", "diana", "eve"};
        static constexpr std::array domains = {"example", "test", "sample"};
        auto name = names[index % names.size()];
        auto domain = domains[(index / names.size()) % domains.size()];
        return Email::from_trusted(std::format("{}{}@{}.com", name, index, domain));
    }
    
    [[nodiscard]] Phone phone(uint64_t index) const {
        auto area = 100 + (index % 900);
        auto number = 1000 + (index % 9000);
        return Phone::from_trusted(std::format("+1-{:03d}-{:04d}", area, number));
    }
    
    [[nodiscard]] PostalCode postal_code(uint64_t index) const {
        return PostalCode::from_trusted(std::format("{:05d}", 10000 + index));
    }
    
    [[nodiscard]] PositiveDecimal positive_decimal(uint64_t index) const {
        return PositiveDecimal::from_trusted(
            boost::multiprecision::cpp_dec_float<50>((index % 9999) + 0.01));
    }
    
    [[nodiscard]] Latitude latitude(uint64_t index) const {
        return Latitude::from_trusted(-90.0 + (index % 18000) / 100.0);
    }
    
    [[nodiscard]] Longitude longitude(uint64_t index) const {
        return Longitude::from_trusted(-180.0 + (index % 36000) / 100.0);
    }
    
    // Auto-incrementing generators
    [[nodiscard]] Email next_email() { return email(index_++); }
    [[nodiscard]] Phone next_phone() { return phone(index_++); }
    [[nodiscard]] PostalCode next_postal_code() { return postal_code(index_++); }
    // ... etc
    
private:
    uint64_t index_ = 0;
};

} // namespace muscovite::types::test

Catch2 Test Usage:

#include <muscovite/types/test_data.hpp>
#include <catch2/catch_test_macros.hpp>

TEST_CASE("Patient CRUD with strong types", "[patient]") {
    muscovite::types::test::TestDataGenerator gen;
    
    auto patient = PatientService::Create(
        "John Doe",
        gen.email(1),
        gen.phone(1),
        gen.postal_code(1)
    );
    
    REQUIRE(patient.email().value().contains("@"));
    REQUIRE(patient.phone().value().starts_with("+1-"));
}

5. Qt Test Data Generators

Generate Qt test utilities in include/muscovite/qt/types/test_data.hpp:

#pragma once
#include <muscovite/qt/types/email.hpp>
#include <muscovite/qt/types/phone.hpp>
#include <QString>

namespace muscovite::qt::test {

class TestDataGenerator {
public:
    explicit TestDataGenerator(quint64 seed = 0) : index_(seed) {}
    
    [[nodiscard]] Email email(quint64 index) const {
        static const QStringList names = {"alice", "bob", "charlie", "diana", "eve"};
        static const QStringList domains = {"example", "test", "sample"};
        auto name = names[index % names.size()];
        auto domain = domains[(index / names.size()) % domains.size()];
        return Email::fromTrusted(QStringLiteral("%1%2@%3.com").arg(name).arg(index).arg(domain));
    }
    
    [[nodiscard]] Phone phone(quint64 index) const {
        auto area = 100 + (index % 900);
        auto number = 1000 + (index % 9000);
        return Phone::fromTrusted(QStringLiteral("+1-%1-%2")
            .arg(area, 3, 10, QChar('0'))
            .arg(number, 4, 10, QChar('0')));
    }
    
    // ... etc
    
private:
    quint64 index_ = 0;
};

} // namespace muscovite::qt::test

8. Validation of Generated Test Data

Each test data function includes a sanity check:

-- After creating test_email function:
DO $$
BEGIN
    -- Verify generated values pass the domain constraint
    FOR i IN 0..99 LOOP
        PERFORM basedata.test_email(i);  -- Will raise if invalid
    END LOOP;
    RAISE NOTICE 'test_email: 100 values validated OK';
END;
$$;

Consequences

Positive

  • All tests pass - Generated values satisfy CHECK constraints
  • Deterministic - Reproducible test failures
  • Self-documenting - Test data patterns defined in DSL
  • Cross-layer - Same patterns in pgTAP, C++, Qt
  • Seed-friendly - Valid CSV/JSON seed data

Negative

  • More generated code - Test data functions for each strong type
  • Maintenance - Test data patterns must match CHECK constraints

Neutral

  • Index-based generation may produce "unrealistic" patterns (alice99@example.com)

AI Agent Guidance

When This Applies

  • Generating pgTAP tests with STRONG_TYPE fields → Use test_* functions
  • Generating C++ tests with strong types → Use TestDataGenerator
  • Generating seed CSV/JSON with strong types → Generate valid patterns
  • Writing manual tests → Import test data utilities

When This Does NOT Apply

  • Production code → Use real user input, not test data
  • Built-in types (UUID, Date) → Use standard generators (gen_random_uuid())
