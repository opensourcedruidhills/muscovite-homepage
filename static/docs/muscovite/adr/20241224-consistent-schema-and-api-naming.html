Consistent Schema and API Naming for All Contexts

Consistent Schema and API Naming for All Contexts

  • Status: accepted
  • Deciders: jolo
  • Date: 2024-12-24
  • Tags: database, naming, consistency

Context and Problem Statement

Muscovite generates PostgreSQL schemas from bounded contexts. We need a consistent naming pattern that applies equally to:

  • User-defined contexts (e.g., mineralogy, stratigraphy)
  • Built-in infrastructure contexts (e.g., security, audit)

Decision Drivers

  • Consistency across all generated schemas
  • Clear separation between data tables and API functions
  • Predictable naming for all contexts regardless of origin

Considered Options

  1. Special naming for built-ins: Use prefixed schema names like musc_security
  2. Consistent naming for all: Same <context> + <context>_api pattern for everything
  3. Single schema approach: Put everything in one schema per context

Decision Outcome

Chosen option: Consistent naming for all, because it maintains uniformity and makes built-in capabilities indistinguishable from user-defined contexts at the schema level.

Schema Naming Pattern

For ANY bounded context (user-defined or built-in):

Type Schema Name Example
Data Tables <context> security, mineralogy
API Functions <context>_api security_api, mineralogy_api

Examples

Security (Built-in):

-- Data schema
CREATE SCHEMA security;
CREATE TABLE security.principal (...);
CREATE TABLE security.principal_group (...);

-- API schema  
CREATE SCHEMA security_api;
CREATE FUNCTION security_api.principal_insert(...) RETURNS ...;
CREATE FUNCTION security_api.principal_select_by_id(...) RETURNS ...;

Mineralogy (User-defined):

-- Data schema
CREATE SCHEMA mineralogy;
CREATE TABLE mineralogy.mineral (...);

-- API schema
CREATE SCHEMA mineralogy_api;
CREATE FUNCTION mineralogy_api.mineral_insert(...) RETURNS ...;

Positive Consequences

  • Predictable naming across all contexts
  • Built-in capabilities look and behave like user contexts
  • No special-casing in generators
  • Easy to reason about security grants

Negative Consequences

  • Built-in schema names could conflict with user context names (mitigation: reserved names list)

AI Agent Guidance

When This Decision Applies

  • When generating PostgreSQL schemas for ANY bounded context
  • When generating PL/pgSQL API functions
  • When naming schemas for built-in capabilities
  • When creating Sqitch migrations

When This Decision Does NOT Apply

  • Non-PostgreSQL targets (C++, gRPC, Qt naming follows different conventions)
  • Internal Racket module naming

Enforcement

  • Enforced by sqitch-unified.rkt and plpgsql-api.rkt generators
  • Same code path for built-in and user-defined contexts
  • No special prefix logic for built-in schemas

Key Files

  • src/dslgen/generators/postgres/sqitch-unified.rkt - Schema generation
  • src/dslgen/generators/postgres/plpgsql-api.rkt - API function generation
