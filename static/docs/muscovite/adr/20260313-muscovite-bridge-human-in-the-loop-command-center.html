ADR: Muscovite Bridge — Human-in-the-Loop Command Center

ADR: Muscovite Bridge — Human-in-the-Loop Command Center

  • Date: 2026-03-13
  • Status: Proposed
  • Related: ADR 20260312-agent-role-governance-and-separation-of-duties, ADR 20260312-push-based-coordinator-with-durable-nats, ADR 20260309-nats-jetstream-agent-coordination

Context

The closed-circuit problem

The Muscovite multi-agent coordinator (v1.63.0–v1.64.0) manages an iteration's lifecycle through structured NATS commands: workers register, receive assignments, report progress, signal PR readiness, and get released. The coordinator is a deterministic state machine — crash-recoverable via JetStream event replay, formally verified via TLA+ model checking.

However, the circuit is not closed. Four seams require human intervention with no protocol support:

Seam Current state Required
Worker spawn Manual copilot-cli launch Auto-spawn or trigger
QA review No automated QA agent Gates-pass = auto-approve
PR merge execution qa.approve updates state but nobody runs gh pr merge Merge agent or inline merge
Release trigger Human runs release.sh Coordinator-initiated

The steering problem

When the circuit eventually closes (all four seams automated), the human principal loses visibility and control. Today, the only intervention mechanisms are:

  • CLI commands (coordinator.py leader amend-state) — out-of-band, no context
  • Raw NATS requests (nats req coord.cmd...) — requires protocol knowledge
  • Event subscription (nats sub coord.event.>) — firehose, no filtering, no response path

None of these allow an agent to ask the human a question and wait for an answer.

The missing role

The coordinator protocol defines five roles (ADR 20260312): Planner, Leader, QA, Worker, Observer. The human principal is not a role — they operate through the Leader or QA CLI, which conflates "human who makes decisions" with "agent that executes protocol commands."

This matters because:

  1. Workers get stuck. A worker implementing FTR-418 might face an ambiguous acceptance criterion: "should qDebug strings be wrapped in tr()?" Today, the worker either guesses or stops. There is no way to ask the human and block until answered.

  2. The human can't address specific agents. If worker-3 takes a wrong approach, the human has no way to say "use approach X instead" without manually amending FTR state and hoping the worker re-reads its brief.

  3. Natural language has no channel. The structured protocol correctly uses typed JSON for state machine transitions. But human↔︎agent communication is inherently natural language. There is no NATS subject for it.

The reasoning chain

This ADR emerged from a systematic assessment of the Qt Widgets pattern catalog (QT-WIDGETS-PATTERNS.md, 2026-03-13). While deriving FTRs 417–420 for v1.66.0, the question arose: "Can agents execute these FTRs without human intervention?" The analysis revealed:

  1. The NATS protocol handles agent↔︎coordinator state transitions well.
  2. The PR lifecycle has a gap: qa.approve changes coordinator state but doesn't execute gh pr merge. No merge agent exists.
  3. The QA review step is unnecessary for mechanical FTRs — if acceptance criteria are measurable and gates pass, merge should be automatic.
  4. If the circuit closes (auto-approve + merge agent), the human needs a way back in — but only on demand, when an agent escalates.
  5. The human must be able to address any agent (worker, QA, leader), not just the coordinator. And any agent must be able to address the human.
  6. This communication is natural language — the one place where NL belongs in the system. Agent↔︎agent communication stays structured JSON. Human↔︎agent communication is NL routed through the coordinator.

Conclusion: The protocol needs a sixth role (human) and a bidirectional NL channel. The QML dashboard (deferred FTR-381/382) becomes the UI for this channel — transforming from a read-only monitor into an interactive command center.

Decision

1. Introduce muscovite-bridge as the human command center

Name: muscovite-bridge — the bridge between human principal and autonomous agents, analogous to a ship's bridge where the captain observes and commands.

Technology: QML application communicating via gRPC bidirectional streams (see ADR 20260313-grpc-gateway-for-agent-coordination-protocol). The bridge never connects to NATS directly — it talks to the gRPC gateway, which handles NATS internally. Built on the existing framework/qt/ stack with grpc_task_adapter for async I/O.

Not a dashboard. A dashboard is passive (Observer role). The bridge is interactive: it observes, but also sends commands, responds to escalations, and communicates decisions in natural language.

2. Add human as the sixth protocol role

Role NATS Prefix Direction Content
Human coord.cmd.human.* Human → Agent Decisions, steering, overrides
Human coord.escalate.* Agent → Human Questions, blockers, confirmations

The human role is singular (like QA) — exactly one bridge session per iteration.

Registration: coord.cmd.human.attach (returns session token, like leader/QA).

Privileges:

  • Respond to any escalation
  • Send directed message to any agent (worker, QA, leader)
  • All Leader privileges (status, amend-state, reload, release)
  • All QA privileges (approve, reject, gate-failed)
  • Override any agent decision

The human role is a superset of all other roles. This is intentional — the human principal is the ultimate authority.

3. Escalation protocol

Agent → Human (escalation)

Any role can escalate:

Subject: coord.escalate.<role>.<agent-name>
Payload: {
  "ftr": "FTR-418",
  "question": "The audit command catches qDebug strings too. Should I exclude
               qDebug from tr() wrapping or wrap them?",
  "options": ["Exclude qDebug (recommended)", "Wrap all strings"],
  "context": { "file": "dslgen/generators/qt/qml-viewmodel.rkt", "line": 142 },
  "blocking": true
}

When blocking: true:

  • FTR transitions to BLOCKED_HUMAN (new state in state machine)
  • Worker suspends work on this FTR
  • Bridge UI highlights the escalation prominently
  • Coordinator will not reassign the FTR

When blocking: false:

  • Advisory notification only
  • Agent continues working
  • Human may respond or ignore

Human → Agent (response)

Subject: coord.cmd.human.respond
Payload: {
  "escalation_id": "<uuid>",
  "decision": "Exclude qDebug — they are developer-facing, not user-facing",
  "ftr": "FTR-418"
}

On response:

  • FTR transitions from BLOCKED_HUMAN back to PROGRESSING
  • Decision is delivered to the agent via its push subscription
  • Decision is persisted in JetStream for audit trail

Human → Agent (directed message, unsolicited)

Subject: coord.cmd.human.direct.<role>.<agent-name>
Payload: {
  "ftr": "FTR-420",
  "message": "Start with spatial_widgets.hpp — it's the worst offender at 443 lines.
              Leave validation_codes.hpp alone, it's pure enums."
}

Delivered to the agent's push subscription. Not blocking — advisory steering.

4. FTR state machine extension

Add BLOCKED_HUMAN state:

stateDiagram-v2
    PENDING --> ASSIGNED
    ASSIGNED --> PROGRESSING
    PROGRESSING --> PR_READY
    PR_READY --> REVIEWING
    REVIEWING --> MERGED

    PROGRESSING --> BLOCKED_HUMAN
    BLOCKED_HUMAN --> PROGRESSING

    REVIEWING --> ASSIGNED: gate fail / rejection
    BLOCKED_HUMAN --> ASSIGNED: gate fail / rejection

Transitions:

  • PROGRESSING → BLOCKED_HUMAN: Agent escalates with blocking: true
  • BLOCKED_HUMAN → PROGRESSING: Human responds via coord.cmd.human.respond
  • BLOCKED_HUMAN → PENDING: Human cancels and reassigns (amend-state)

6. Closing the remaining circuit seams

With the bridge in place, the remaining open seams close:

Seam Solution Mechanism
PR merge Merge agent (new) Polls merge.next, runs gh pr merge, calls merge.complete
QA review Auto-approve policy If all gates pass and FTR has auto_approve: true in DEPENDENCIES.toml, QA auto-approves
Release trigger Bridge button Human clicks "Release" in bridge → coord.cmd.leader.release
Worker spawn Out of scope Workers run on customer infra with customer AI credentials (ADR 20260313-grpc-gateway). Spawn is environment-dependent (local CLI, CI pipeline, cloud container)

The auto_approve policy per FTR enables lights-out operation for mechanical FTRs (refactoring, code generation fixes) while preserving human review for architectural changes (grammar modifications, new DSL constructs).

7. NL belongs here, nowhere else

Human ↔︎ Agent: Natural language via coord.escalate.* and coord.cmd.human.*. Messages are free-form text with optional structured metadata (FTR, file, line, options).

Agent ↔︎ Agent: Structured JSON via coord.cmd.* and coord.event.*. No change. The coordinator remains a deterministic state machine with typed, validated inputs.

Agent ↔︎ Coordinator: Structured JSON. No change. State transitions, resource locks, merge queue — all remain typed and TLA+-verifiable.

Natural language enters the system at exactly one boundary: the human↔︎agent interface. This preserves the formal properties of the coordinator protocol while giving the human principal the communication medium they need.

Consequences

Positive

  • Closed circuit: The iteration lifecycle can run without human intervention when all gates pass and auto_approve is set.
  • On-demand steering: Human can intervene at any point without disrupting the protocol.
  • Audit trail: All escalations and decisions are persisted in JetStream — every human decision is traceable.
  • Single pane of glass: One UI for observation, escalation response, directed messaging, and release control.
  • Graceful degradation: If the bridge is disconnected, agents continue autonomously (non-blocking escalations are advisory; blocking escalations pause the FTR until the human reconnects).

Negative

  • New state: BLOCKED_HUMAN adds complexity to the state machine. TLA+ model must be updated.
  • Single point: One human bridge per iteration. No delegation to multiple humans (acceptable for current team size; revisit if team grows).
  • QML dependency: Bridge requires Qt 6.11+ and the full QML stack. No web fallback (acceptable — Muscovite already requires Qt for generated frontends).

Implementation scope

SaaS deployment model

In a SaaS deployment (see ADR 20260313-grpc-gateway-for-agent-coordination-protocol), the bridge runs on the customer's desktop and connects to the hosted gRPC gateway. Key implications for the bridge:

  • Agent execution is customer-side. The bridge observes agent state via the gateway but never spawns or hosts agents. Workers run on customer infrastructure with customer AI credentials.
  • The bridge sees event metadata only. FTR states, worker names, timings, escalation text — but never source code, AI prompts, or credentials. Escalation messages are natural language authored by agents and humans; they may reference code by file:line but do not transmit file contents through the gateway.
  • Offline-capable observation. The bridge can cache the last-known iteration state locally. If the gateway connection drops, the UI shows stale data with a "disconnected" indicator. Agents continue autonomously (non-blocking escalations proceed; blocking ones pause until the human reconnects).
  • Multi-tenant readiness. A single bridge instance connects to one gateway endpoint (one iteration). Multiple bridge windows can run simultaneously against different gateway endpoints (different projects or tenants).

Alternatives Considered

A. Web-based dashboard (rejected)

A browser-based UI (React/Vue) would avoid the Qt dependency but would require:

  • A WebSocket bridge to NATS (additional infrastructure)
  • A separate tech stack outside Muscovite's Qt ecosystem
  • No reuse of existing framework/qt/ components

Muscovite already generates Qt frontends. Building the bridge in QML is consistent and reuses existing async gRPC patterns, theme system, and validation framework.

B. CLI-only interaction (rejected)

Extending the existing coordinator.py CLI with escalate and respond subcommands. This works mechanically but fails on the UX:

  • No real-time event stream (must poll)
  • No visual overview of iteration state
  • No notification of pending escalations
  • Context-switching between terminal windows

The CLI remains available as a fallback and for scripted automation, but the primary human interface should be the bridge.

C. Slack/Discord integration (rejected)

Routing escalations to a chat platform provides notifications but:

  • Adds external service dependency
  • Messages are not part of the NATS audit trail
  • Response routing back to the coordinator requires a bot
  • Violates the project boundary rule (Rule 13 — no external service dependencies for core workflow)

D. Keep human out of protocol, use GitHub only (rejected)

Using GitHub Issues or PR comments for all human↔︎agent communication. This is where review feedback naturally lives, but:

  • No real-time notification (GitHub webhooks require infrastructure)
  • Cannot block an agent pending human response
  • No iteration-level overview
  • Mixes project management with code review
