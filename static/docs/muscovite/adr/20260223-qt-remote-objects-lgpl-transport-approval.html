Qt Remote Objects LGPL Transport Approval for Frontend Plugins

Qt Remote Objects LGPL Transport Approval for Frontend Plugins

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-23
  • Tags: qt, lgpl, licensing, plugins, transport, qt-remote-objects

Technical Story: The plugin system design (ADR 20260223-plugin-transport-selection) requires a transport mechanism for frontend (Qt-side) plugins that inject UI behavior or respond to property changes. Qt Remote Objects (LGPL) is the ergonomically correct choice over gRPC for this use case. This ADR closes the licensing question.

Context and Problem Statement

Frontend plugins (UIWidget class extension points) need a transport mechanism that:

  • Speaks Qt's native object model (signals, slots, Q_PROPERTY) across process boundaries
  • Operates over a local socket (no network exposure)
  • Integrates naturally with the existing EntityDialogStateMachine pattern

Qt Remote Objects (QRemoteObjectNode, QRemoteObjectHost) provides transparent proxy objects over local sockets and is ergonomically superior to gRPC for Qt-native plugins. However, it carries LGPL-2.1 licensing (with Qt LGPL exception), which must be assessed against Muscovite's AGPL-3.0 + Commercial dual license.

Decision Drivers

  • Legal compatibility with AGPL-3.0 + Commercial dual license
  • Commercial customer responsibility for Qt module licensing
  • Ergonomics: Qt native object model vs. gRPC serialization for FE plugins
  • Fallback availability if QtRemoteObjects is rejected

Considered Options

  1. Approve Qt Remote Objects (LGPL-2.1 + Qt exception) — primary FE transport
  2. Reject Qt Remote Objects — use gRPC for all plugin transports — uniform transport
  3. Defer decision — use gRPC now, migrate later

Decision Outcome

Chosen option: Option 1 — Qt Remote Objects is approved.

Licensing Analysis

Qt Remote Objects is licensed under LGPL-2.1 with the Qt LGPL Exception 1.1. The exception permits linking Qt LGPL modules into proprietary applications without triggering LGPL copyleft as long as the Qt library itself remains dynamically linked and replaceable by the user.

  • AGPL-3.0 compatibility: LGPL-2.1 (with exceptions) is OSI-approved and compatible with AGPL-3.0 under standard FSF guidance. No conflict.
  • Commercial track: Muscovite's commercial customers are already Qt licensees (Qt Widgets and QML are already used). Qt Remote Objects is a standard Qt module. Adding it does not introduce any new licensing constraint on commercial customers beyond what they have already accepted via the Qt license.
  • Muscovite generated code: Generated code that uses QRemoteObjectNode inherits the LGPL terms for that Qt module. This is not a new constraint — generated Qt code already inherits LGPL from other Qt modules.

Conclusion

Qt Remote Objects is an approved transport for frontend (UIWidget class) plugins. Transport is a per-plugin .mus configuration (TRANSPORT: QtRemoteObjects). Switching transport is a one-line .mus change + regeneration.

gRPC-over-localhost remains a valid alternative if the plugin author prefers a language-agnostic contract or does not want Qt-native coupling.

This question is closed. No further licensing review is required for Qt Remote Objects in the Muscovite project.

Positive Consequences

  • FE plugins speak Qt-native object model across process boundaries
  • Signals and Q_PROPERTY work transparently via proxy objects
  • Generated circuit breaker integrates with QStateMachine naturally
  • Commercial customers: no new licensing burden

Negative Consequences

  • Qt Remote Objects is an additional Qt module dependency in generated projects
  • Plugin authors on the FE side must speak Qt (cannot use arbitrary language) — gRPC fallback is available for language-agnostic FE plugins

Pros and Cons of the Options

Option 1: Approve Qt Remote Objects (chosen)

  • Good, because native Qt signals/slots work transparently via proxy
  • Good, because no new licensing burden on AGPL or Commercial tracks
  • Good, because familiar to Qt developers
  • Bad, because FE plugin authors must use Qt (gRPC fallback available for others)

Option 2: Reject — use gRPC for all plugin transports

  • Good, because uniform transport across BE and FE
  • Good, because any language can implement FE plugins
  • Bad, because serialization overhead for Qt-native property changes is unergonomic
  • Bad, because loses Qt signal/slot integration
