Unified Racket Test Runner

Unified Racket Test Runner

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: testing, racket, ci, developer-experience

Context and Problem Statement

Muscovite has 5+ Racket test files in dslgen/tests/ covering parser features, grammar conformance, generator output, integration, and context mapping. Running them requires knowing each file name and executing them individually. There is no single command to validate the entire Racket layer.

This creates friction for:

  • CI/CD: No single entry point for "all Racket tests pass"
  • AI agents: Must be told which tests to run and in what order
  • Contributors: Must discover test files by reading directory listings
  • Release ceremonies: Quality gateway lists 5 separate test commands

Decision Drivers

  • Developer experience: one command to validate everything
  • CI readiness: single exit code for pass/fail
  • Test ordering: some tests have implicit dependencies (parser before generators)
  • Reporting: aggregated pass/fail counts across all suites

Decision Outcome

Create dslgen/tests/run-all-tests.rkt — a unified test runner that:

  1. Discovers and runs all test files in deterministic order
  2. Reports per-suite and aggregate pass/fail counts
  3. Exits with code 0 (all pass) or 1 (any failure)
  4. Captures and displays output from each suite

Test Execution Order

1. test-parser.rkt              — Core parser functionality
2. test-parser-features.rkt     — Extended parser features
3. test-grammar-conformance.rkt — PEG grammar ↔ parser alignment
4. test-generator-suite.rkt     — All generator output validation
5. test-dsl-integration.rkt     — End-to-end parse + generate
6. test-context-mapping.rkt     — Context map parsing (if exists)

Order matters: parser tests first (foundation), then conformance, then generators (depend on parsing), then integration (depends on everything).

What This Is NOT

  • Not a test framework (we use rackunit and ad-hoc checks)
  • Not a replacement for ctest (C++ tests stay in CTest)
  • Not a parallel executor (sequential for determinism)

Positive Consequences

  • Single command for full Racket validation: racket dslgen/tests/run-all-tests.rkt
  • Quality gateway simplifies to one line instead of five
  • AI agents need only one test command
  • Release checklist shortens

Negative Consequences

  • Slower than running targeted tests during development
  • Aggregated output may obscure individual failures (mitigated by per-suite reporting)

AI Agent Guidance

When This Applies

  • Running full validation before commits
  • Release ceremony quality gates
  • CI pipeline Racket test stage

Key Files

  • dslgen/tests/run-all-tests.rkt — the unified runner
  • All test files in dslgen/tests/test-*.rkt — individual suites
