Migration Safety and Validation Policies

Migration Safety and Validation Policies

  • Status: accepted
  • Deciders: Project maintainers
  • Date: 2026-01-09
  • Tags: policy, safety, migration, validation, schema-evolution
  • Related: Declarative Migration Language, Migration SQL Generation

Technical Story: Implement policy evaluation to classify and control migration operations based on safety level.

Context and Problem Statement

Not all inferred migrations are safe. A migration that drops a column destroys data. A migration that changes a type may truncate values. Without guardrails:

  1. Destructive changes execute silently: DROP COLUMN runs without warning
  2. No approval workflow: All changes treated equally
  3. No audit trail: Cannot prove destructive change was intentional
  4. CI/CD risk: Automated pipelines may execute dangerous migrations

Policy evaluation provides:

  • Safety classification for all operations
  • Approval requirements for dangerous operations
  • Audit trail of decisions
  • CI/CD-friendly gate mechanism

Decision Drivers

  • Distinguish safe vs unsafe vs destructive migration operations
  • Fail or warn on potentially destructive changes
  • Enforce policies consistently across the pipeline
  • Support explicit override for intentional destructive changes
  • Enable CI/CD integration with approval gates
  • Audit trail for compliance

Considered Options

  • Auto-execute all migrations: Rejected - too dangerous for production
  • Manual review only: Rejected - too slow, error-prone
  • Binary safe/unsafe classification: Rejected - too coarse
  • Tiered policy evaluation with overrides: Accepted

Decision Outcome

Chosen option: "Tiered policy evaluation with overrides", implementing a policy evaluator that:

  1. Classifies each operation by safety tier
  2. Applies configurable policies per tier
  3. Requires explicit approval for dangerous operations
  4. Produces audit trail of decisions

Safety Tiers

Tier Description Default Policy
safe No data loss possible Auto-approve
unsafe Potential data issues Require strategy
destructive Certain data loss Require explicit approval
review Needs human judgment Flag for review

Positive Consequences

  • Prevents unintended data loss
  • Enables safe automated migration for additive changes
  • Supports compliance audit requirements
  • Clear separation of concerns (diff → policy → execute)

Negative Consequences

  • Adds complexity to migration workflow
  • Requires explicit intent for some legitimate changes
  • May slow down rapid iteration (by design)

Implementation

Phase 1: Policy Definition

Create src/dslgen/migration/policy.rkt:

#lang racket/base

(provide (all-defined-out))

;; Policy configuration
(struct migration-policy
  (safe-action          ;; 'approve | 'flag | 'reject
   unsafe-action        ;; 'approve | 'require-strategy | 'reject
   destructive-action   ;; 'approve | 'require-approval | 'reject
   review-action        ;; 'approve | 'flag | 'reject
   allowed-overrides    ;; List of operation types that can be overridden
   require-audit)       ;; Boolean - require audit log entry
  #:transparent)

;; Default policy (safe for CI/CD)
(define default-policy
  (migration-policy
   'approve              ;; safe → auto-approve
   'require-strategy     ;; unsafe → must provide strategy
   'reject               ;; destructive → blocked by default
   'flag                 ;; review → flag for human review
   '()                   ;; no overrides allowed
   #t))                  ;; require audit

;; Development policy (more permissive)
(define development-policy
  (migration-policy
   'approve              ;; safe → auto-approve
   'approve              ;; unsafe → auto-approve in dev
   'require-approval     ;; destructive → need explicit --allow-destructive
   'approve              ;; review → auto-approve in dev
   '(remove-field-op remove-entity-op change-type-op)
   #f))                  ;; no audit in dev

;; Strict policy (for production)
(define production-policy
  (migration-policy
   'approve              ;; safe → auto-approve
   'require-strategy     ;; unsafe → must provide strategy
   'reject               ;; destructive → blocked
   'reject               ;; review → blocked
   '()                   ;; no overrides
   #t))                  ;; require audit

Phase 2: Policy Evaluation

;; Evaluation result
(struct policy-result
  (approved?          ;; Boolean
   operation          ;; The original migration-op
   decision           ;; 'approved | 'rejected | 'needs-strategy | 'needs-approval | 'flagged
   reason             ;; Human-readable explanation
   required-action)   ;; What user must do, or #f
  #:transparent)

;; Evaluate single operation against policy
(define (evaluate-operation policy op)
  (define safety (migration-op-safety op))
  (define action (policy-action-for-safety policy safety))
  
  (case action
    [(approve)
     (policy-result #t op 'approved "Auto-approved by policy" #f)]
    
    [(reject)
     (policy-result #f op 'rejected 
                    (format "~a operations are blocked by policy" safety)
                    #f)]
    
    [(require-strategy)
     (define strategy (migration-op-strategy op))
     (if strategy
         (policy-result #t op 'approved 
                        (format "Approved with strategy: ~a" strategy)
                        #f)
         (policy-result #f op 'needs-strategy
                        "Unsafe operation requires a migration strategy"
                        "Provide strategy via --strategy flag or migration annotation"))]
    
    [(require-approval)
     (define approval (migration-op-approval op))
     (if approval
         (policy-result #t op 'approved
                        (format "Approved by: ~a" approval)
                        #f)
         (policy-result #f op 'needs-approval
                        "Destructive operation requires explicit approval"
                        "Provide approval via --allow-destructive flag"))]
    
    [(flag)
     (policy-result #t op 'flagged
                    "Flagged for human review"
                    "Review this operation before deployment")]))

;; Evaluate entire migration plan
(define (evaluate-migration-plan policy plan)
  (define results 
    (map (λ (op) (evaluate-operation policy op))
         (migration-plan-operations plan)))
  
  (define all-approved? 
    (andmap policy-result-approved? results))
  
  (define blocked 
    (filter (λ (r) (not (policy-result-approved? r))) results))
  
  (define flagged
    (filter (λ (r) (eq? (policy-result-decision r) 'flagged)) results))
  
  (values all-approved? results blocked flagged))

Phase 3: Strategy Definitions

;; Strategies for unsafe operations
(struct migration-strategy
  (name              ;; Symbol
   description       ;; Human-readable
   applies-to        ;; List of operation types
   sql-template)     ;; Function that generates safe SQL
  #:transparent)

;; Built-in strategies
(define add-not-null-with-default
  (migration-strategy
   'add-not-null-with-default
   "Add NOT NULL column with default value for existing rows"
   '(add-field-op)
   (λ (op)
     (format "ALTER TABLE ~a ADD COLUMN ~a ~a NOT NULL DEFAULT ~a"
             (add-field-op-entity-name op)
             (add-field-op-field-name op)
             (add-field-op-field-type op)
             (add-field-op-default-value op)))))

(define backfill-then-constrain
  (migration-strategy
   'backfill-then-constrain
   "Add nullable column, backfill data, then add NOT NULL"
   '(add-field-op)
   (λ (op)
     (string-append
      (format "ALTER TABLE ~a ADD COLUMN ~a ~a;\n"
              (add-field-op-entity-name op)
              (add-field-op-field-name op)
              (add-field-op-field-type op))
      (format "UPDATE ~a SET ~a = ~a WHERE ~a IS NULL;\n"
              (add-field-op-entity-name op)
              (add-field-op-field-name op)
              (add-field-op-default-value op)
              (add-field-op-field-name op))
      (format "ALTER TABLE ~a ALTER COLUMN ~a SET NOT NULL;"
              (add-field-op-entity-name op)
              (add-field-op-field-name op))))))

(define type-change-with-using
  (migration-strategy
   'type-change-with-using
   "Change column type with explicit USING clause"
   '(change-type-op)
   (λ (op)
     (format "ALTER TABLE ~a ALTER COLUMN ~a TYPE ~a USING ~a"
             (change-type-op-entity-name op)
             (change-type-op-field-name op)
             (change-type-op-new-type op)
             (change-type-op-conversion op)))))

Phase 4: CLI Integration

;; Command-line flags
;; --policy development|production|default
;; --allow-destructive  (for CI/CD with explicit approval)
;; --strategy <name>    (for unsafe operations)
;; --approval <ticket>  (audit reference)

(define (apply-cli-overrides policy args)
  (define allow-destructive? (member "--allow-destructive" args))
  (define strategy (find-arg-value args "--strategy"))
  (define approval (find-arg-value args "--approval"))
  
  (if allow-destructive?
      (struct-copy migration-policy policy
                   [destructive-action 'approve])
      policy))

Phase 5: Audit Trail

;; Audit log entry
(struct audit-entry
  (timestamp
   migration-id
   operation-uuid
   decision
   policy-used
   override-used?
   approval-reference
   user)
  #:transparent)

;; Write audit log
(define (write-audit-log entries path)
  (call-with-output-file path
    (λ (out)
      (for ([entry entries])
        (displayln (audit-entry->json entry) out)))
    #:exists 'append))

AI Agent Guidance

When This Decision Applies

  • When implementing migration execution pipeline
  • When adding new migration operation types
  • When configuring CI/CD for schema changes

When This Decision Does NOT Apply

  • Model validation (that's parsing, not migration)
  • SQL syntax validation (that's generation)
  • Runtime authorization

Enforcement

  • All migrations MUST pass policy evaluation before SQL generation
  • Destructive operations MUST be rejected by default policy
  • Overrides MUST be audited when require-audit is true
  • CI/CD pipelines MUST use explicit --allow-destructive flag
