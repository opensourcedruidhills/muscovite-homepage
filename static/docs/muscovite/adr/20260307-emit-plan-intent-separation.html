EMIT/PLAN Intent Separation — Internal State vs External Effects

EMIT/PLAN Intent Separation — Internal State vs External Effects

  • Status: accepted
  • Date: 2026-03-07
  • Deciders: Johannes Lochmann
  • Tags: emit, plan, event, outbox, behavioral-sovereignty

Context

The current EMITS clause (muscovite.peg:347) declares which domain events a command produces, but does not distinguish between:

  1. Internal state changes — events that modify the aggregate's own state within the same transaction
  2. External side effects — actions that affect systems outside the aggregate boundary (email, notifications, external APIs)

The POLICY construct already has this distinction implicitly: EMIT (internal), INVOKE (sync external), NOTIFY (async external). But COMMAND lacks it entirely — all outputs are undifferentiated event names.

The existing infrastructure already supports both paths:

  • UnitOfWork::add_event() → domain_events table → same transaction (internal)
  • OutboxPublisher polls domain_events → NATS relay (external)

But the DSL cannot express which events are internal state changes vs which are planned external effects. This conflation means generated code treats all events identically, losing the architectural distinction.

Decision

Introduce EMIT and PLAN as distinct intent keywords within DECIDE blocks:

EMIT — Internal state change, applied atomically within the transaction:

EMIT PatientDischarged(discharge_date, reason);
  • Generates: uow.add_event(DomainEvent{...}) within the active transaction
  • Persistence: Written to domain_events table in the same COMMIT
  • Guarantee: All-or-nothing with the aggregate state change

PLAN — External side effect, queued for asynchronous execution:

PLAN SendDischargeNotification(patient.email, reason);
  • Generates: Outbox entry with typed payload
  • Execution: OutboxPublisher relays via NATS after transaction commits
  • Guarantee: At-least-once delivery (outbox pattern)
  • Failure: Does not roll back the aggregate state change

Validation rules:

  • Every EMIT in a DECIDE block must reference a declared DOMAIN_EVENT
  • Every PLAN must reference a declared EFFECT (see ADR 20260307-effect-catalog)
  • EMITS clause on the COMMAND must list all EMIT targets (parser validates)
  • PLAN targets are validated against the EFFECT catalog at parse time

Consequences

  • Clean separation of "things that happened" (EMIT) vs "things to do" (PLAN)
  • Generators produce different code paths for each intent type
  • PostgreSQL: EMIT → INSERT INTO domain_events (same txn); PLAN → INSERT INTO outbox (same txn, different relay)
  • C++ DomainService: EMIT → uow.add_event(); PLAN → uow.plan_effect()
  • Docs generator can visualize internal vs external data flow distinctly
  • POLICY's EMIT/INVOKE/NOTIFY remain unchanged (reactive context, different semantics)
