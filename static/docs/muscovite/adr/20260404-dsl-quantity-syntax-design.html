DSL Quantity Syntax Design

DSL Quantity Syntax Design

  • Status: proposed
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04
  • Tags: units, dsl, syntax, grammar

Technical Story: IDEA-134 — DSL-Native Units of Measure, Decision 5

Context and Problem Statement

Muscovite's DSL needs syntax for declaring quantity fields on entities. The core question is how unit symbols with SI prefixes are written: as concatenated symbols (km, mg, µm) that match conventional notation, or as compositional expressions (kilo metre, milli gram) that make the prefix explicit.

Beyond field declarations, the DSL also needs syntax for:

  • Importing unit systems (IMPORT "muscovite://units/si")
  • Declaring custom dimensions (DIMENSION Density = Mass / Length^3)
  • Declaring custom units (UNIT millidarcy = 9.869233e-16 m^2)
  • Configuring prefixes available in a project

Decision Drivers

  • Readability: DSL should look natural to domain experts (geoscientists, clinicians)
  • Familiarity: km, mg, mmHg are universally recognized; kilo metre is not
  • Consistency with UCUM: UCUM uses concatenated symbols (km, mg, mm[Hg])
  • Compiler complexity: concatenated symbols require prefix resolution in the compiler
  • Ambiguity: some concatenated symbols could be ambiguous (mitigated by standard library)
  • Grammar simplicity: compositional syntax has simpler tokenization

Considered Options

  1. Concatenated unit symbols — QUANTITY<Length, km>, QUANTITY<Pressure, mmHg>
  2. Compositional syntax — QUANTITY<Length, kilo metre>, QUANTITY<Pressure, milli metre Hg>

Decision Outcome

Chosen option: "Concatenated unit symbols" (option 1), because it matches how scientists, engineers, and clinicians actually write and think about units.

Field Declaration Syntax

field_name QUANTITY<Dimension, unit_symbol>

Examples:

depth         QUANTITY<Length, km>
diameter      QUANTITY<Length, mm>
weight        QUANTITY<Mass, kg>
temperature   QUANTITY<Temperature, Cel>
systolic      QUANTITY<Pressure, mmHg>
core_density  QUANTITY<Density, kg/m3>
wavelength    QUANTITY<Length, µm>

The compiler resolves prefixed symbols internally:

  • km → prefix kilo + base unit metre → factor 1000, dimension Length
  • mg → prefix milli + base unit gram → factor 0.001, dimension Mass
  • mmHg → named unit (not a prefix composition), dimension Pressure
  • kg/m3 → derived unit expression, dimension Density

Unit System Import Syntax

IMPORT "muscovite://units/si";          -- 7 base + prefixes + 22 derived
IMPORT "muscovite://units/medical";     -- mmHg, mg/dL, IU/L, Cel, ...
IMPORT "muscovite://units/geoscience";  -- mGal, ppm, Ma, ...

Dimension Declaration Syntax

For custom or derived dimensions not in the standard library:

DIMENSION Density = Mass / Length^3;
DIMENSION Velocity = Length / Time;
DIMENSION Concentration = Amount / Volume;

Unit Declaration Syntax

For custom units not in the standard library:

UNIT millidarcy = 9.869233e-16 m^2;
UNIT Ma = 3.1536e13 s;                 -- megayear in seconds

Prefix Configuration

Projects can restrict which prefixes are available (to prevent nonsensical combinations):

PROJECT "fhir-health" {
    PREFIXES micro, milli, centi, deci, deca, hecto, kilo, mega;
}

Positive Consequences

  • Natural syntax: QUANTITY<Length, km> reads like documentation
  • Consistent with UCUM encoding (which healthcare systems already use)
  • Domain experts can read and write DSL files without learning a new unit notation
  • Prefix resolution is a compiler concern, invisible to DSL authors
  • Derived units (kg/m3) use familiar mathematical notation

Negative Consequences

  • Compiler must implement prefix resolution (decompose km → kilo + metre)
  • Some symbols could theoretically be ambiguous (e.g., is mGal milli-Gal or mega-litre?) — resolved by standard library precedence rules
  • Unicode symbols (µm) require UTF-8 handling in the parser (UCUM uses u instead)
  • Grammar is more complex: unit symbols are looked up, not tokenized from parts

Pros and Cons of the Options

Option 1: Concatenated Unit Symbols

QUANTITY<Length, km> — prefix is part of the symbol, compiler resolves it.

  • Good, because matches universal scientific notation (km, mg, mmHg)
  • Good, because consistent with UCUM standard encoding
  • Good, because shorter, more readable DSL declarations
  • Good, because domain experts immediately understand the syntax
  • Good, because derived unit expressions (kg/m3) look natural
  • Bad, because compiler must implement prefix resolution algorithm
  • Bad, because potential ambiguity in rare cases (resolved by standard library)
  • Bad, because Unicode symbols (µm) require UTF-8 parser support

Option 2: Compositional Syntax

QUANTITY<Length, kilo metre> — prefix and base unit are separate tokens.

  • Good, because simpler tokenization (prefix and unit are separate tokens)
  • Good, because no ambiguity — prefix is always explicit
  • Good, because no prefix resolution needed in the compiler
  • Bad, because unnatural — nobody writes "kilo metre" in practice
  • Bad, because verbose: QUANTITY<Pressure, milli metre mercury> vs QUANTITY<Pressure, mmHg>
  • Bad, because breaks consistency with UCUM, ISO 80000, and scientific convention
  • Bad, because domain experts must learn an unfamiliar notation
  • Bad, because derived units become awkward: kilo gram / metre^3 vs kg/m3

AI Agent Guidance

When This Decision Applies

  • When extending the PEG grammar (dslgen/grammar/muscovite.peg) with QUANTITY syntax
  • When implementing the parser for QUANTITY field declarations
  • When implementing prefix resolution in the DSL compiler (decomposing km → kilo + metre)
  • When writing documentation or examples that show QUANTITY field syntax
  • When adding new keywords: QUANTITY, DIMENSION, UNIT, PREFIXES, UNIT_SYSTEM

When This Decision Does NOT Apply

  • When working on generator output (generators consume the parsed IR, not raw syntax)
  • When working on proto or PostgreSQL representation (see the respective ADRs)
  • When editing existing non-quantity DSL constructs (ENTITY, CONTEXT, RELATIONSHIP, etc.)

Enforcement

  • PEG grammar defines the QUANTITY syntax rule (grammar-first, per ADR 20260211)
  • Grammar conformance tests verify parsing of all documented examples
  • New keywords (QUANTITY, DIMENSION, UNIT, PREFIXES) must follow contextual keyword rules (ADR 20260316) — they are contextual within entity field declarations
  • lint-ddd.sh validates QUANTITY declarations against imported unit systems
