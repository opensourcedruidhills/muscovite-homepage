Disable C++20 Module Scanning

Disable C++20 Module Scanning

  • Status: accepted
  • Deciders: Build system architects
  • Date: 2025-12-23
  • Tags: cpp, cmake, build, clang-tidy

Context and Problem Statement

CMake 3.28+ automatically enables C++20 module scanning for C++23 projects. This causes GCC to add -fmodules-ts, -fmodule-mapper, and -fdeps-format=p1689r5 flags. However, clang-tidy uses clang's frontend which doesn't understand these GCC-specific flags, causing hard errors.

Decision Drivers

  • clang-tidy is essential for code quality enforcement
  • We are not currently using C++20 modules
  • GCC module flags are incompatible with clang's frontend
  • Cannot suppress these errors via .clang-tidy (they're frontend errors)

Considered Options

  • Disable C++20 module scanning globally
  • Disable module scanning only when clang-tidy enabled
  • Switch to clang compiler
  • Drop clang-tidy

Decision Outcome

Chosen option: "Disable C++20 module scanning globally", because we are not using C++20 modules and this is the simplest solution that allows both GCC compilation and clang-tidy analysis.

# In root CMakeLists.txt
set(CMAKE_CXX_SCAN_FOR_MODULES OFF)  # Not using C++20 modules

Positive Consequences

  • clang-tidy works correctly with GCC-compiled code
  • Build is simpler without module dependency scanning
  • No conditional logic needed
  • Can enable modules in future if needed

Negative Consequences

  • Cannot use C++20 modules without changing this setting
  • Must revisit if modules become desirable

Pros and Cons of the Options

Disable globally

  • Good, because simplest solution
  • Good, because no conditional logic
  • Good, because clearly documents we don't use modules
  • Bad, because prevents future module use without change

Disable conditionally

  • Good, because could use modules when not running clang-tidy
  • Bad, because split behavior between build modes
  • Bad, because more complex CMake logic

Switch to clang

  • Good, because clang-tidy works natively
  • Bad, because GCC has better C++23 support currently
  • Bad, because disrupts existing build setup

Drop clang-tidy

  • Good, because no compatibility issues
  • Bad, because lose valuable static analysis
  • Bad, because code quality suffers

AI Agent Guidance

When This Decision Applies

  • CMake configuration for C++23 targets
  • Adding new C++ libraries to the build
  • Troubleshooting clang-tidy errors about unknown flags

When This Decision Does NOT Apply

  • If project decides to adopt C++20 modules (supersede this ADR)
  • Non-CMake build systems

Enforcement

  • CMAKE_CXX_SCAN_FOR_MODULES OFF in root CMakeLists.txt
  • No .cppm or module interface files in codebase

  • CMakeLists.txt - Global setting at line 16
  • cmake/StaticAnalysis.cmake - clang-tidy configuration
