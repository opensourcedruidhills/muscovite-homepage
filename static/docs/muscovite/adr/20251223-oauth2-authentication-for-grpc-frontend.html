OAuth2 Authentication for gRPC Frontend (External Providers)

OAuth2 Authentication for gRPC Frontend (External Providers)

  • Status: accepted
  • Deciders: jolo
  • Date: 2025-12-23
  • Tags: qt, frontend, authentication, oauth2, grpc

Technical Story: Implementing secure authentication for Qt frontend communicating with gRPC backend via external identity providers.

Related Decision

Built-in Authentication: For self-hosted deployments without external identity providers, see Security Domain Architecture. The built-in auth uses username/password against the muscoviteserver security module with JWT tokens.

This ADR covers OAuth2 integration with external providers (Keycloak, Auth0, Azure AD, etc.) for enterprise deployments.

Context and Problem Statement

The Qt frontend needs to authenticate users against the gRPC backend. For enterprise deployments, integration with existing identity providers (IdP) is required. We need a secure, standard authentication mechanism that works with gRPC's interceptor model and supports token refresh without user intervention.

Decision Drivers

  • Security best practices (no password storage in app)
  • Standard protocol support (OAuth2/OIDC)
  • Token refresh without user re-authentication
  • Integration with gRPC call credentials
  • Support for multiple identity providers
  • Qt signal/slot integration for auth state changes

Considered Options

  1. OAuth2 Authorization Code Flow with PKCE - Browser-based auth
  2. OAuth2 Resource Owner Password Credentials (ROPC) - Direct username/password
  3. API Keys - Static keys per user
  4. mTLS Client Certificates - Certificate-based auth

Decision Outcome

Chosen option: "OAuth2 Authorization Code Flow with PKCE" for external provider integration, because it follows security best practices, never exposes user passwords to the application, and is the recommended flow for native applications per RFC 8252.

Note: This is an optional authentication mode. The default/primary mode is built-in username/password authentication against the muscoviteserver security module.

Positive Consequences

  • User passwords never touch our application
  • Standard flow supported by all identity providers
  • Supports SSO and enterprise identity providers
  • Refresh tokens enable long sessions
  • Secure against MITM with PKCE

Negative Consequences

  • Requires browser redirect (or embedded browser)
  • More complex than simple username/password
  • Requires identity provider setup
  • Token storage must be secure

Implementation Architecture

AuthManager Dual-Mode Support

The AuthManager supports both built-in and OAuth2 authentication:

class AuthManager : public QObject {
    Q_OBJECT
    Q_PROPERTY(bool authenticated READ isAuthenticated NOTIFY authStateChanged)
    Q_PROPERTY(QString userName READ userName NOTIFY authStateChanged)
    Q_PROPERTY(AuthMode authMode READ authMode NOTIFY authModeChanged)

public:
    enum AuthMode { BuiltIn, OAuth2 };
    Q_ENUM(AuthMode)
    
    static AuthManager& instance();
    
    // Built-in auth (default)
    void login(const QString& username, const QString& password);
    
    // OAuth2 auth (enterprise)
    void loginWithOAuth2();           // Starts OAuth2 flow
    
    void logout();
    bool isAuthenticated() const;
    QString accessToken() const;
    QString userName() const;
    
signals:
    void authStateChanged();
    void authModeChanged();
    void loginFailed(const QString& error);
    void tokenRefreshed();
    
    QString accessToken_;
    QString refreshToken_;
    QDateTime tokenExpiry_;
    QOAuth2AuthorizationCodeFlow* oauth_;
};

AuthInterceptor Class (muscoviteqt)

gRPC interceptor adding bearer token to all calls:

class AuthInterceptor : public grpc::MetadataCredentialsPlugin {
public:
    explicit AuthInterceptor(AuthManager& authManager);
    
    grpc::Status GetMetadata(
        grpc::string_ref service_url,
        grpc::string_ref method_name,
        const grpc::AuthContext& channel_auth_context,
        std::multimap<grpc::string, grpc::string>* metadata
    ) override {
        auto token = authManager_.accessToken();
        if (!token.isEmpty()) {
            metadata->insert({"authorization", 
                "Bearer " + token.toStdString()});
        }
        return grpc::Status::OK;
    }
    
private:
    AuthManager& authManager_;
};

Token Storage

  • Access tokens: Memory only (short-lived)
  • Refresh tokens: Qt Keychain (secure OS storage)
  • Token expiry: Track and auto-refresh before expiry

Login Flow

sequenceDiagram
    participant QtApp as Qt App

    participant IdP as Identity Provider

    QtApp->>Auth: login()
    Auth->>IdP: Open browser with auth URL + PKCE
    Note over IdP: User authenticates
    IdP-->>Auth: Redirect with code
    Auth->>IdP: Exchange code for tokens
    IdP-->>Auth: tokens
    Auth-->>QtApp: authStateChanged()

AI Agent Guidance

When This Decision Applies

  • Implementing authentication in Qt frontend
  • Setting up gRPC channel credentials
  • Adding login/logout UI components

When This Decision Does NOT Apply

  • Server-side authentication (gRPC server validates tokens)
  • Service-to-service authentication
  • Development/testing scenarios (may use simplified auth)

Enforcement

  • All gRPC channels must use AuthInterceptor
  • AuthManager is singleton, never create duplicate instances
  • Token refresh happens automatically, don't manually manage tokens
