Strong Type Template Families for Field Constraints

Strong Type Template Families for Field Constraints

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-07
  • Tags: cpp, strong-types, templates, validation, constraints

Technical Story: ADR 20260119 introduced strong_type_base<Derived> CRTP with from_trusted(). Only PositiveInteger exists as a concrete type. The DSL needs a family of parameterized strong types to represent arbitrary field constraints from DSL annotations.

Context and Problem Statement

The existing strong type infrastructure (ADR 20260119) provides a CRTP base class strong_type_base<Derived> with:

  • static auto from(value_type) -> std::expected<Derived, std::string> for validated construction
  • static auto from_trusted(value_type) -> Derived for database loads (bypasses validation)
  • Value semantics, comparison operators, hashing

However, only one concrete type exists: PositiveInteger. The DSL constraint annotations (ADR 20260307-field-constraint-annotations) need to map to a family of parameterized strong types that cover the full range of per-field constraints: bounded integers, bounded decimals, constrained dates/datetimes, bounded strings, and pattern strings.

How should the strong type family be structured to support arbitrary DSL constraint annotations while maintaining compile-time safety where possible?

Decision Drivers

  • Compile-time validation where possible (integer/string bounds are known at compile time)
  • Runtime validation where necessary (temporal constraints depend on now())
  • Must extend strong_type_base<Derived> to inherit from_trusted() for DB loads
  • Existing PositiveInteger must remain backward-compatible (becomes a type alias)
  • C++23 features (NTTP, StringLiteral, constexpr) should be used where they improve safety
  • Generator must map DSL annotations to specific template instantiations

Considered Options

  1. Four template families — BoundedInteger, ConstrainedDate/DateTime, BoundedDecimal, BoundedString/PatternString
  2. Single generic BoundedType<T, Min, Max> — one template for all types
  3. Policy-based design — validated_type<T, Policy1, Policy2, ...> with composable policies
  4. Runtime-only validation — no templates, all validation in from() with runtime parameters

Decision Outcome

Chosen option: Option 1 — Four template families, because each family has distinct validation semantics that cannot be unified into a single template without sacrificing type safety or compile-time guarantees.

Family 1: BoundedInteger<Min, Max>

template<std::int64_t Min = std::numeric_limits<std::int64_t>::min(),
         std::int64_t Max = std::numeric_limits<std::int64_t>::max()>
class BoundedInteger : public strong_type_base<BoundedInteger<Min, Max>> {
public:
    static auto from(std::int64_t v) -> std::expected<BoundedInteger, std::string>;
    static auto from_trusted(std::int64_t v) -> BoundedInteger;
};

// Backward compatibility
using PositiveInteger = BoundedInteger<1>;
// DSL: age Integer @min(0) @max(150) → BoundedInteger<0, 150>
// DSL: quantity Integer @min(1) → BoundedInteger<1>

Sentinel values numeric_limits::min()/max() represent "unbounded" on that side, enabling half-open ranges.

Family 2: ConstrainedDate / ConstrainedDateTime

enum class TemporalPolicy { PastOrPresent, Future, PastOnly };

template<TemporalPolicy P>
class ConstrainedDate : public strong_type_base<ConstrainedDate<P>> {
public:
    static auto from(std::chrono::year_month_day v) -> std::expected<ConstrainedDate, std::string>;
    static auto from_trusted(std::chrono::year_month_day v) -> ConstrainedDate;
};

// DSL: birth_date Date @not_in_future → ConstrainedDate<TemporalPolicy::PastOrPresent>
// DSL: appointment_date DateTime @not_in_past → ConstrainedDateTime<TemporalPolicy::Future>

Runtime validation because now() is not constexpr. The policy tag is a compile-time NTTP that selects the runtime check.

Family 3: BoundedDecimal<MinStr, MaxStr>

template<StringLiteral MinStr = "-inf", StringLiteral MaxStr = "+inf">
class BoundedDecimal : public strong_type_base<BoundedDecimal<MinStr, MaxStr>> {
public:
    static auto from(decimal64 v) -> std::expected<BoundedDecimal, std::string>;
    static auto from_trusted(decimal64 v) -> BoundedDecimal;
};

// DSL: temperature Decimal @min(-273.15) @max(1000.0) → BoundedDecimal<"-273.15", "1000.0">

C++23 StringLiteral NTTP allows decimal precision to be carried as a compile-time parameter. The string is parsed to decimal64 at construction time.

Family 4: BoundedString<MinLen, MaxLen> and PatternString

template<std::size_t MinLen = 0,
         std::size_t MaxLen = std::numeric_limits<std::size_t>::max()>
class BoundedString : public strong_type_base<BoundedString<MinLen, MaxLen>> {
public:
    static auto from(std::string_view v) -> std::expected<BoundedString, std::string>;
    static auto from_trusted(std::string v) -> BoundedString;
};

template<StringLiteral Pattern>
class PatternString : public strong_type_base<PatternString<Pattern>> {
public:
    static auto from(std::string_view v) -> std::expected<PatternString, std::string>;
    static auto from_trusted(std::string v) -> PatternString;
};

// DSL: name String @not_empty → BoundedString<1>
// DSL: email String @pattern("[^@]+@[^@]+\.[^@]+") → PatternString<"[^@]+@[^@]+\\.[^@]+">

@not_empty is syntactic sugar for BoundedString<1>. PatternString compiles the regex once (static local) and validates on each from() call.

Positive Consequences

  • Compile-time bounds for integers and strings — invalid template parameters caught at compile time
  • PositiveInteger becomes a backward-compatible alias — no migration needed
  • from_trusted() inherited from strong_type_base — DB loads bypass validation consistently
  • Generator maps DSL annotations directly to template instantiations — no runtime configuration
  • Each family has focused, understandable validation semantics

Negative Consequences

  • Four families means more framework code to maintain in framework/include/muscovite/
  • StringLiteral NTTP (C++23) may not be supported by all compilers yet — requires GCC 13+ or Clang 17+
  • Temporal validation is runtime-only — cannot be checked at compile time
  • Complex template instantiations may produce verbose compiler error messages

AI Agent Guidance

When This Decision Applies

  • When generating C++ entity field types from DSL constraint annotations
  • When mapping @min/@max → BoundedInteger<Min, Max> or BoundedDecimal<MinStr, MaxStr>
  • When mapping @not_empty → BoundedString<1>
  • When mapping @not_in_future/@not_in_past → ConstrainedDate<Policy> or ConstrainedDateTime<Policy>
  • When mapping @pattern(...) → PatternString<Pattern>
  • When generating Entity::try_from(Command) — use strong type from() for Tier 1 validation

When This Decision Does NOT Apply

  • Cross-field constraints (@after, @before) — those are Tier 2, not strong types
  • Unconstrained fields — fields without annotations use raw types (std::string, std::int64_t)
  • Lookup table foreign keys — these are UUIDs, not constrained value types
  • Database loads — always use from_trusted(), never from()

Enforcement

  • Generator maps DSL annotations to template instantiations in dslgen/generators/dba/unified.rkt
  • Framework provides template families in framework/include/muscovite/types/
  • Unit tests verify from() rejects invalid values and from_trusted() accepts any value
  • Compile-time tests verify BoundedInteger<5, 3> (inverted range) produces a static_assert failure
