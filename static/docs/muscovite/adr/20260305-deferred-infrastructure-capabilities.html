ADR: Deferred Infrastructure Capabilities — APIM and Cache Layer

ADR: Deferred Infrastructure Capabilities — APIM and Cache Layer

  • Date: 2026-03-05
  • Status: accepted
  • Tags: infrastructure, architecture, deferred

Context

During v1.49.0 planning, we evaluated whether Muscovite should add an API Management (APIM) layer and a cache layer to the generated stack. Both are common in production microservice deployments but add operational and code-generation complexity.

Muscovite currently generates: PostgreSQL/Sqitch, C++23 DBA, gRPC/Protobuf, Qt Widgets, Qt QML, CMake/Conan, Documentation, and OpenSearch targets — all from .ddd models.

Decision Drivers

  • No external API consumers today — Qt desktop clients speak gRPC natively
  • No read-latency SLA — desktop clients are low-concurrency, direct-connection
  • Generator architecture already provides the seams — adding a target is ~1K–5K lines of Racket
  • Premature infrastructure increases operational burden without delivering user value
  • YAGNI — defer until a concrete trigger demands the capability

Considered Options

  1. Implement APIM + Cache now — full REST gateway (Kong/Envoy/APISIX) + Redis/Valkey cache
  2. Add OpenAPI spec generation only — generate specs but no runtime gateway
  3. Defer both, document the insertion points ← chosen

Decision Outcome

Chosen option: Defer both capabilities. The current architecture is pre-wired for both without requiring future refactoring.

APIM — Deferred

Planned path: gRPC-Gateway transcoding via google.api.http annotations on already-generated .proto files.

Why it's safe to defer:

  • gRPC services are generated with well-defined request/response types
  • Proto files already contain complete service definitions
  • Adding google.api.http annotations is a template-level change in generators/grpc/proto.rkt
  • A future generators/apim/ target would read the same IR structs every other generator reads
  • Estimated effort: ~2K lines of Racket for OpenAPI spec generation + proto annotation

Triggers to revisit:

  • External API consumers (mobile apps, partner integrations) requiring REST
  • Multi-tenant SaaS deployment requiring rate limiting, API keys, OAuth2
  • FHIR R4 REST conformance requirement (mandates HTTP REST endpoints)
  • Public developer portal needing OpenAPI specs

Cache Layer — Deferred

Planned path: CachedQueryService wrapper generated around QueryService, with domain_events-driven invalidation.

Why it's safe to defer:

  • QueryService is the single read entry point (ADR 20260120) — natural cache insertion point
  • CommandService already emits domain events unconditionally (ADR 20260124) — invalidation signal exists
  • CQRS separation (separate CommandRepository / QueryRepository types) prevents cache-transaction contamination
  • A future @cacheable annotation + generator wrapper is additive, not structural

Implementation sketch (for future reference):

gRPC → QueryService → Cache → QueryRepository → PostgreSQL
                        ↑
         domain_events ─┘  (invalidation via event subscription)
  • Generator: ~1K–2K lines of Racket to emit CachedQueryService wrapper
  • Framework: muscovite::cache::Cache<K,V> interface with Redis + in-memory backends (~500 lines)
  • DSL: @cacheable TTL 60s annotation (requires PEG grammar change)

Triggers to revisit:

  • High read:write ratio (>100:1) on specific aggregates
  • gRPC response latency exceeding SLA budget
  • Multi-instance deployment with shared read workload
  • Expensive projection materialization (complex JOINs)

Positive Consequences

  • No additional operational dependencies (Redis, API gateway) in the deployment
  • Generated code stays lean — no cache/APIM boilerplate for aggregates that don't need it
  • Future implementation is additive (new generator target or wrapper), not a refactor
  • Architectural seams are documented for future developers/agents

Negative Consequences

  • If an external REST API is needed urgently, there will be a lead time to implement the generator
  • Cache-worthy read patterns must be addressed ad-hoc until the generator exists
