Use JSON for Seed Data Files

Use JSON for Seed Data Files

Status: Accepted
Date: 2026-01-08
Deciders: System Architect, Database Team
Related: 20260107-entity-code-surrogate-keys.md

Context

Seed data files populate lookup tables and reference data during database deployment. Initially, CSV format was chosen for its simplicity and human-readability. However, CSV parsing has proven fragile due to:

  1. Ambiguous field delimiting: Commas inside quoted fields require complex parsing
  2. Embedded newlines: Multi-line field values break naive line-splitting
  3. Quote escaping: CSV's "" → " convention is inconsistent across implementations
  4. Empty field handling: Different parsers treat consecutive commas differently
  5. No type information: All values are strings; type coercion errors are common
  6. Mixed quoting: Some CSVs quote all fields, others only fields with special chars

Real-world failures:

  • map_symbol.csv: Field "Internal contact—Identity questionable, location accurate" split incorrectly
  • lithology_pattern.csv: Embedded newlines in descriptions caused SQL syntax errors
  • dunham_class.csv: Empty fields after trimming caused column misalignment

Decision

Adopt JSON as the standard format for all seed data files.

Seed files will use .json extension and contain an array of objects:

[
  {
    "id": "536568a9-71d8-4467-ac34-7d7f5b92d433",
    "code": "LITHPAT-601",
    "fgdc_code": "601",
    "name": "Gravel or conglomerate (1st option)",
    "rock_category": "sedimentary",
    "description": "Coarse-grained clastic sedimentary rock\nwith particles >2mm"
  }
]

Key properties:

  • Each object represents one table row
  • Keys correspond to table column names
  • Omitted keys result in NULL or DEFAULT values
  • JSON native types (string, number, boolean, null) map naturally to SQL types
  • Arrays and objects serialize to JSONB columns

Generator behavior:

  • DSL: IMPORT_JSON "path/to/file.json" INTO EntityName
  • Generated SQL escapes values appropriately for PostgreSQL
  • Code field omitted from JSON → trigger auto-generates
  • Explicit code field → preserves user value

Rationale

Why JSON over alternatives?

Format Pros Cons Verdict
CSV Simple, compact, Excel-friendly Fragile parsing, no structure ❌ Current pain point
JSON Unambiguous, structured, Racket built-in More verbose ✅ Selected
SQL INSERT No parsing, direct execution Not data, harder to maintain ❌ Too imperative
S-expressions Racket-native, simple Non-standard, learning curve ❌ Too niche
YAML Human-friendly, less verbose Complex spec, security issues ❌ Overkill
TOML Simple, readable Poor Racket support ❌ Limited tooling

JSON advantages:

  1. Robust parsing: Racket's built-in json module handles all edge cases correctly
  2. Unambiguous syntax: No confusion about quoting, escaping, or delimiters
  3. Type preservation: Numbers, booleans, nulls are distinct from strings
  4. Validation: JSON Schema can enforce structure before generation
  5. Tooling: Every editor has JSON support (syntax highlighting, formatting, validation)
  6. PostgreSQL affinity: Native JSONB type, json_populate_recordset(), jsonb_to_record()
  7. Version control friendly: Structured diffs are meaningful

Migration path:

Existing CSV files remain valid during transition:

  • Generator detects file extension (.csv vs .json)
  • CSV parsing uses fallback legacy mode (best-effort)
  • New seed files MUST use JSON
  • Existing CSVs can be converted incrementally

Implementation

Generator changes:

sqitch-unified.rkt:

(require json)

(define (parse-seed-file path)
  (cond
    [(string-suffix? path ".json") (parse-json-seed path)]
    [(string-suffix? path ".csv")  (parse-csv-seed path)]  ; legacy
    [else (error "Unsupported seed format")]))

(define (parse-json-seed path)
  (call-with-input-file path
    (lambda (in)
      (define data (read-json in))
      (unless (list? data)
        (error "JSON seed must be array of objects"))
      data)))

INSERT generation:

;; JSON objects directly map to column-value pairs
(for ([row json-data])
  (define cols (hash-keys row))
  (define vals (map (lambda (k) (hash-ref row k)) cols))
  (format "INSERT INTO ~a (~a) VALUES (~a);"
          table
          (string-join cols ", ")
          (string-join (map sql-escape vals) ", ")))

SQL value escaping:

(define (sql-escape val)
  (cond
    [(null? val) "NULL"]
    [(boolean? val) (if val "true" "false")]
    [(number? val) (number->string val)]
    [(string? val) (format "E'~a'" 
                          (string-replace 
                           (string-replace val "\\" "\\\\")
                           "'" "''"))]
    [(hash? val) (format "E'~a'" (jsexpr->string val))]  ; JSONB columns
    [(list? val) (format "E'~a'" (jsexpr->string val))]
    [else (error "Unsupported JSON type")]))

DSL syntax (unchanged):

ENTITY LithologyPattern IDENTIFIED_BY id (
    id UUID NOT NULL,
    code TEXT NOT NULL @unique,
    ...
)
IMPORT_JSON "../../seeds/geologic-mapping/lithology_pattern.json" INTO LithologyPattern;

Consequences

Positive:

  • ✅ Eliminates CSV parsing fragility - no more split errors, quote issues, or newline problems
  • ✅ Correct by construction - JSON syntax errors caught by parser, not at SQL execution
  • ✅ Better type safety - numbers as numbers, booleans as booleans
  • ✅ Easier debugging - JSON structure is self-documenting
  • ✅ No external dependencies - Racket's json module is built-in
  • ✅ Migration-friendly - CSV files still work during transition

Negative:

  • ❌ More verbose - JSON overhead (braces, quotes) increases file size ~20%
  • ❌ Less Excel-friendly - can't directly edit in spreadsheet (but tools exist)
  • ❌ Migration effort - need to convert ~38 existing CSV files

Neutral:

  • 🔄 Learning curve - team must learn JSON syntax (minimal for most developers)
  • 🔄 Tooling required - need CSV-to-JSON conversion script (one-time)

Migration Strategy

Phase 1: Generator support (immediate)

  1. Add JSON parsing to sqitch-unified.rkt
  2. Keep CSV parsing as fallback
  3. Test with one converted seed file

Phase 2: Conversion (incremental)

  1. Create convert-seeds-to-json.py script
  2. Convert basedata seeds first (simple, no FK lookups)
  3. Validate each conversion with jq and jsonlint
  4. Deploy and test incrementally

Phase 3: Deprecation (after all converted)

  1. Remove CSV parsing code
  2. Update documentation
  3. Add .csv extension check that emits error with migration instructions

Conversion script:

#!/usr/bin/env python3
import csv, json, sys

with open(sys.argv[1]) as f:
    reader = csv.DictReader(f)
    data = [row for row in reader]
    print(json.dumps(data, indent=2))

Usage:

python convert-seeds-to-json.py seeds/geologic-mapping/lithology_pattern.csv > \
       seeds/geologic-mapping/lithology_pattern.json

Validation

Before deployment:

# Validate JSON syntax
jq empty lithology_pattern.json || echo "Invalid JSON"

# Validate structure (all rows have same keys)
jq -e 'map(keys) | unique | length == 1' lithology_pattern.json

After deployment:

# Verify row count matches
psql -c "SELECT COUNT(*) FROM geologicmapping.lithologypattern;" | grep 100

AI Agent Guidance

When this applies:

  • Creating new seed data files
  • Converting existing CSV seed files to JSON
  • Debugging seed import errors
  • Adding new lookup tables with reference data

When this does NOT apply:

  • User data import (use COPY or bulk insert APIs)
  • Large datasets (>1000 rows) - consider direct SQL
  • Legacy CSV files during transition period

Key enforcement:

  1. New seed files MUST use JSON format - generator emits warning for CSV
  2. JSON arrays MUST contain objects - generator validates structure
  3. Object keys MUST match table columns - generator checks schema
  4. Code field SHOULD be omitted - let trigger auto-generate (unless explicit user codes)
