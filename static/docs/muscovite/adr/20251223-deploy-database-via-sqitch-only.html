Deploy Database Changes Exclusively via Sqitch

Deploy Database Changes Exclusively via Sqitch

  • Status: accepted
  • Deciders: Database architects, DevOps
  • Date: 2024-07-01
  • Tags: postgresql, sqitch, deployment, migrations

Context and Problem Statement

Muscovite generates PostgreSQL schemas, tables, functions, and stored procedures. How should these database artifacts be deployed to development, staging, and production environments?

Decision Drivers

  • Track all database changes in version control
  • Support rollback of failed deployments
  • Verify successful deployment
  • Consistent deployment across all environments
  • Support dependencies between migrations

Considered Options

  • Sqitch migration management
  • Plain SQL scripts with manual ordering
  • ORM-based migrations (like Alembic, Flyway)
  • Direct psql execution from generators

Decision Outcome

Chosen option: "Sqitch migration management", because it provides dependency tracking, verification scripts, and rollback capabilities with pure SQL that matches our PostgreSQL-native approach.

ABSOLUTE REQUIREMENT: ALL database modifications—schemas, tables, functions, procedures, triggers, indexes, API layers—MUST be deployed EXCLUSIVELY via Sqitch migrations. NO OTHER METHOD IS PERMITTED.

Positive Consequences

  • Complete audit trail in version control
  • Dependency ordering handled automatically
  • Verify scripts confirm successful deployment
  • Revert scripts enable rollback
  • Works with any PostgreSQL version
  • No ORM abstraction layer

Negative Consequences

  • Learning curve for Sqitch commands
  • Must maintain deploy/revert/verify triplets
  • Manual dependency declaration required

Forbidden Practices

  • ❌ Direct psql commands to create/modify database objects
  • ❌ Shell scripts that load SQL files directly
  • ❌ Python/Node scripts that execute DDL statements
  • ❌ Manual SQL execution in any database client
  • ❌ Generators outputting SQL outside of deploy/revert/verify directories
  • ❌ Any database change not tracked in sqitch.plan

Required Structure

generated/sqitch/
├── sqitch.conf           # Database connection targets
├── sqitch.plan           # ALL migrations in dependency order
├── deploy/               # CREATE/INSERT statements
├── revert/               # DROP statements (inverse of deploy)
└── verify/               # SELECT queries to verify deployment

Migration Dependency Order

  1. Data schemas first: schema-name-schema
  2. Tables depend on schemas: schema-table-entity [schema-schema]
  3. API schemas depend on data schemas
  4. API functions depend on API schema AND tables
  5. Foreign keys depend on both tables

AI Agent Guidance

When This Decision Applies

  • ALL generated SQL for PostgreSQL database
  • ALL schema creation, alteration, or deletion
  • ALL function/procedure creation
  • ALL index and constraint changes
  • Manual database fixes (must create migration)

When This Decision Does NOT Apply

  • Non-PostgreSQL targets (C++, gRPC, Qt)
  • Local development queries (SELECT only)
  • pgTAP test execution (reads deployed schema)

Enforcement

  • Generators output to deploy/, revert/, verify/ directories
  • sqitch.plan lists all migrations with dependencies
  • CI runs sqitch verify to validate deployments
