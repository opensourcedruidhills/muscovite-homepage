Lookup Tables Over CHECK Constraints for Enumerated Values

Lookup Tables Over CHECK Constraints for Enumerated Values

  • Status: accepted
  • Deciders: Architecture team
  • Date: 2026-01-16
  • Tags: postgresql, schema, lookup-tables, enums, extensibility

Context and Problem Statement

When modeling domain concepts with enumerated values (e.g., clinical_status, verification_status, severity, category), we need to decide how to enforce valid values. Should we use PostgreSQL CHECK constraints with hard-coded value lists, or dedicated lookup tables that reference these values?

FHIR and similar healthcare standards define value sets that may evolve over time. Organizations need the flexibility to:

  • Extend standard value sets with local codes
  • Modify display names or descriptions
  • Retire obsolete values without breaking existing data
  • Track value metadata (effective dates, definitions, sources)

Decision Drivers

  • Extensibility: Users MUST be able to extend enumerated value sets
  • Modifiability: Users MUST be able to modify existing entries (display name, description)
  • Retirability: Users MUST be able to retire values without deleting them
  • Auditability: Changes to value sets must be trackable
  • Referential integrity: Values must be validated at the database level
  • Migration safety: Schema evolution must not require ALTER TABLE for value changes

Considered Options

  1. CHECK constraints with hard-coded value lists
  2. PostgreSQL ENUM types
  3. Lookup tables with foreign key references

Decision Outcome

Chosen option: "Lookup tables with foreign key references", because they provide full extensibility, modifiability, and retirability while maintaining referential integrity.

Implementation Pattern

Every enumerated value set MUST be implemented as a lookup table in the Basedata context:

CONTEXT Basedata {
    ENTITY ClinicalStatus @uuid "..." IDENTIFIED_BY id (
        id           UUID        NOT NULL,
        code         TEXT        NOT NULL @unique,
        display      TEXT        NOT NULL,
        definition   TEXT,
        is_active    BOOLEAN     NOT NULL DEFAULT true,
        effective_from TIMESTAMPTZ,
        effective_to   TIMESTAMPTZ,
        source_system  TEXT,
        sort_order     INTEGER
    )
    DOC "Clinical status codes for conditions (active, inactive, resolved, etc.)";
    
    AGGREGATE ClinicalStatus ROOT ClinicalStatus;
    
    IMPORT CSV "../../seeds/basedata/clinical_status.csv" INTO ClinicalStatus;
}

Domain entities reference lookups via foreign keys:

ENTITY Condition IDENTIFIED_BY id (
    id                   UUID NOT NULL,
    clinical_status_id   UUID NOT NULL,  -- FK to Basedata.ClinicalStatus
    ...
)
RELATIONSHIPS {
    clinical_status -> ClinicalStatus REQUIRED ONE;
}

Standard Lookup Table Columns

Column Type Required Purpose
id UUID Yes Primary key
code TEXT Yes Unique machine-readable code
display TEXT Yes Human-readable display name
definition TEXT No Full description/definition
is_active BOOLEAN Yes Soft-delete / retirement flag
effective_from TIMESTAMPTZ No When value became valid
effective_to TIMESTAMPTZ No When value was retired
source_system TEXT No Origin (e.g., "FHIR R4", "LOCAL")
sort_order INTEGER No UI display ordering

Positive Consequences

  • Users can add local codes without schema changes
  • Display names can be updated without migrations
  • Retired codes remain in database with is_active = false
  • Full audit trail via standard audit patterns
  • Referential integrity maintained via FK constraints
  • FHIR value sets can be seeded via CSV import
  • UI dropdowns populated from API, not hard-coded

Negative Consequences

  • More tables in schema (one per value set)
  • Joins required for display names
  • Initial seed data must be provided
  • Slightly more complex queries

FORBIDDEN Patterns

The following patterns are STRICTLY PROHIBITED:

❌ CHECK Constraints for Enumerated Values

-- FORBIDDEN: Hard-coded CHECK constraint
CREATE TABLE condition (
    clinical_status TEXT NOT NULL 
        CHECK (clinical_status IN ('active', 'inactive', 'resolved'))
);

Why forbidden:

  • Cannot extend without ALTER TABLE
  • Cannot modify display names
  • Cannot retire values
  • No metadata (definitions, sources)
  • Schema migration required for any change

❌ PostgreSQL ENUM Types

-- FORBIDDEN: PostgreSQL ENUM
CREATE TYPE clinical_status_enum AS ENUM ('active', 'inactive', 'resolved');

Why forbidden:

  • Adding values requires ALTER TYPE
  • Cannot remove values without dropping and recreating
  • No metadata storage
  • Ordering changes require recreation

❌ Application-Level Constants

// FORBIDDEN: Hard-coded constants in application
enum class ClinicalStatus { Active, Inactive, Resolved };

Why forbidden:

  • Changes require code deployment
  • No database-level validation
  • Multiple applications may have inconsistent values

Migration from CHECK Constraints

Existing CHECK constraints must be migrated to lookup tables:

  1. Create lookup table in Basedata context
  2. Seed with existing valid values
  3. Add FK column to referencing table
  4. Populate FK column from existing text values
  5. Drop text column with CHECK constraint
  6. Rename FK column if needed

Pros and Cons of All Options

Lookup Tables (Chosen)

  • ✅ Fully extensible by users
  • ✅ Modifiable without schema changes
  • ✅ Soft-delete via is_active flag
  • ✅ Rich metadata (definitions, sources, dates)
  • ✅ Referential integrity via FKs
  • ✅ Auditable via standard patterns
  • ⚠️ More tables in schema
  • ⚠️ Joins required for display names

CHECK Constraints

  • ✅ Simple to implement
  • ✅ No additional tables
  • ❌ Cannot extend without migration
  • ❌ Cannot modify without migration
  • ❌ No soft-delete capability
  • ❌ No metadata storage
  • ❌ Violates extensibility requirement

PostgreSQL ENUM Types

  • ✅ Type-safe at database level
  • ✅ Compact storage
  • ❌ Adding values requires ALTER TYPE
  • ❌ Removing values impossible without recreation
  • ❌ No metadata storage
  • ❌ Ordering changes require recreation
  • ❌ Violates extensibility requirement

AI Agent Guidance

When This Decision Applies

  • ANY field that represents an enumerated value set
  • Examples: status codes, categories, types, severities, priorities
  • FHIR CodeableConcept, Coding, or code elements
  • Any "dropdown" or "picklist" field in the UI

When This Decision Does NOT Apply

  • Boolean fields (true/false)
  • Numeric ranges (use CHECK for min/max)
  • Pattern validation (use CHECK for regex)
  • Unique constraints (use @unique annotation)

How It's Enforced

  1. Parser validation: CHECK constraints with IN clauses should warn
  2. Code review: Reject PRs with hard-coded status/type/category values
  3. Generator behavior: Never emit CHECK constraints for enumerated values
  4. Seed files: All lookup tables must have corresponding CSV seeds
