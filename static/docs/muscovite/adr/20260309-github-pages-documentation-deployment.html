GitHub Pages Documentation Deployment via muscovite-homepage

GitHub Pages Documentation Deployment via muscovite-homepage

  • Status: accepted
  • Date: 2026-03-09
  • Deciders: Johannes Lochmann
  • Tags: documentation, github-pages, deployment, local-build

Context and Problem Statement

Muscovite generates multiple documentation artifacts (Doxygen C++ API, AsciiDoc system manuals, DSL references, Fact Base JSON, pattern guides, ADRs) but these are only available locally after a build. Contributors and evaluators cannot browse framework or sample project documentation without cloning the repo, installing all toolchains, and running CMake targets. This creates a high barrier to understanding the project.

Decision Drivers

  • Documentation must be browsable without cloning or building
  • No additional subscription cost (GitHub Free account)
  • Must serve under the existing muscovite.johanneslochmann.net domain
  • Private GitHub Pages requires Enterprise Cloud — not available
  • Generated documentation must be publishable on each release
  • No GitHub Actions minutes consumed for documentation builds — all heavy computation runs locally
  • Must cover: DSL framework docs, C++ framework API, and per-example-project docs

Considered Options

  1. Deploy into muscovite-homepage under /docs/ path — local script builds docs, pushes to muscovite-homepage repo
  2. Separate public muscovite-docs repo — new repo with GitHub Pages, CI pushes built docs there
  3. GitHub Pages from muscovite repo (gh-pages branch) — requires GitHub Pro ($4/month minimum)
  4. Commit rendered docs into the source repo — bloats git history, merge noise

Decision Outcome

Chosen option: "Deploy into muscovite-homepage under /docs/ path", because it reuses the existing public repo with its CNAME (muscovite.johanneslochmann.net), requires no DNS changes, no additional repos, and no paid plan upgrade. Hugo already serves the homepage; the /docs/ subtree bypasses Hugo and serves static HTML directly.

Build and Deploy Pipeline

A local script (tools/publish-docs.sh) performs the entire build-and-deploy sequence. No GitHub Actions CI is used for documentation — all heavy computation (Racket, Doxygen, AsciiDoctor) runs on the developer machine where these tools are already installed.

The script:

  1. Builds Doxygen HTML (cmake --build --preset debug --target doxygen-docs)
  2. Generates DSL output for all example projects (if not already generated)
  3. Builds AsciiDoc system manuals, HTML only (cmake --build --preset debug --target system-manual-html-*)
  4. Builds Fact Base JSON (cmake --build --preset debug --target docs-fact-base)
  5. Converts Markdown docs to HTML (lightweight — pandoc or raw copy with index)
  6. Assembles the complete docs/ directory tree in a staging area
  7. Copies the staging area into the muscovite-homepage checkout at static/docs/
  8. Commits and pushes to muscovite-homepage
  9. Hugo's existing hugo.yml GitHub Actions workflow triggers on that push, copies static/docs/ verbatim into the site output, and deploys

Typical invocation (after a release):

tools/publish-docs.sh --homepage-repo ~/src/muscovite-homepage

Hugo Coordination

The muscovite-homepage repo uses Hugo with GitHub Pages. Hugo's static/ directory is a passthrough — files placed there are copied verbatim to the site output without Hugo processing. The publish script writes exclusively to static/docs/; Hugo's existing workflow handles the final deployment.

The only GitHub Actions minutes consumed are Hugo's build (~30 seconds), which is unavoidable since that's how muscovite-homepage already deploys.

Positive Consequences

  • All documentation browsable at muscovite.johanneslochmann.net/docs/
  • Zero cost — reuses existing GitHub Pages on public repo
  • No GitHub Actions minutes for heavy builds — only Hugo's ~30s rebuild
  • No DNS changes required
  • No deploy keys or secrets needed — developer pushes with their existing git credentials
  • Uniform per-example structure scales to new projects automatically
  • All toolchains already present on the developer machine

Negative Consequences

  • Manual step required after each release (run tools/publish-docs.sh)
  • Requires local checkout of muscovite-homepage repo
  • Can be integrated into release.sh as an optional post-release step in the future

Pros and Cons of the Options

Option 2: Separate muscovite-docs repo

  • Good, because clean separation of concerns
  • Good, because independent Pages configuration
  • Bad, because requires new CNAME DNS record and propagation wait
  • Bad, because one more repo to maintain

Option 3: gh-pages branch in muscovite repo

  • Good, because simplest CI (same repo)
  • Bad, because requires GitHub Pro ($4/month) for Pages from private repos
  • Bad, because Pages would be public anyway (no private Pages without Enterprise Cloud)

Option 4: Commit rendered docs into source repo

  • Good, because no CI needed
  • Bad, because bloats git history with binary/HTML artifacts
  • Bad, because merge conflicts on every regeneration
  • Bad, because not browsable as a website

AI Agent Guidance

When This Decision Applies

  • When adding new documentation targets or doc types to the build
  • When adding new example projects (must add to the publish script)
  • When modifying the muscovite-homepage Hugo configuration

When This Decision Does NOT Apply

  • Local documentation builds (CMake targets remain unchanged)
  • Audience-specific LLM reports (deferred — require API key secret, not in initial scope)
  • PDF generation (only HTML is deployed to Pages)

Enforcement

  • tools/publish-docs.sh is the single entry point for documentation deployment
  • Hugo static passthrough via static/docs/ directory (no config change needed — Hugo convention)
  • Can be added as optional post-release step in workflow/scripts/release.sh

Key Files

  • cmake/Documentation.cmake — existing CMake doc targets
  • cmake/Doxyfile.in — Doxygen configuration template
