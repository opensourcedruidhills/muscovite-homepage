Migration Transformer Generator Architecture

Migration Transformer Generator Architecture

  • Status: accepted
  • Date: 2026-02-23
  • Tags: brownfield, migration, transformer, generator, nats, consumer, strong-types, dlq, command-repository
  • Deciders: Johannes Lochmann

Technical Story: The second execution stage of the brownfield migration factory (ADR 20260217) is the Transformer: a generated NatsEventConsumer subclass that subscribes to the extractor's NATS subject, applies field transforms, validates via strong-type constructors, and persists via CommandRepository. Failed records are routed to a DLQ subject for human triage.

Context and Problem Statement

The transformer is the most complex component of the migration pipeline. It must:

  1. Consume typed JSON envelopes from NATS (migration.extract.{context}.{entity})
  2. Apply ordered field transforms from MAPPING @trim, @default, @transform annotations
  3. Validate the transformed values against strong-type constructors (e.g., Email("value") throws if invalid)
  4. Validate entity invariants (CHECK constraints)
  5. Persist valid records via CommandRepository::create() — which automatically emits domain events
  6. Route invalid records to a DLQ NATS subject with the envelope + validation error message attached
  7. Compensate transient NATS/database errors by requeueing — not by silently dropping

Without a generator, every migration project hand-codes this logic differently. Worse, most implementations bypass CommandRepository and write directly to SQL — violating ADR 20260120 and silencing domain event emission.

Decision Drivers

  • CommandRepository is the mandated persistence path (ADR 20260120) — transformer MUST use it, never raw SQL
  • NatsEventConsumer from framework/dba/ is the mandated consumer pattern (ADR 20260203)
  • Domain events must be emitted for migrated records (audit trail of what arrived via migration)
  • Strong-type validation (ADR 20260119) is the primary validation layer — the transformer leverages existing constructors
  • DLQ pattern (domain_events_dlq) is already established; migration uses a parallel NATS subject, not the database table
  • Generated transformers must be idempotent: re-processing the same envelope (same target UUID via UUIDv5) MUST result in upsert, not duplicate

Considered Options

  1. Generated NatsEventConsumer subclass per MAPPING block — subscribes, deserializes, transforms, validates, persists via CommandRepository
  2. Generated SQL UPSERT procedure — transformer emits SQL INSERT ... ON CONFLICT directly to target DB
  3. Shared transformer framework class + generated config — one C++ class parameterized by a generated transform table
  4. Racket CLI transformer — interpreted, reads from NATS and writes to DB

Decision Outcome

Chosen option: Option 1 — Generated NatsEventConsumer subclass per MAPPING block, because it inherits the established consumer infrastructure (connection management, acknowledgement policy, backpressure), enforces the CommandRepository path, and follows the code-generation pattern already established for domain service generators.

Generated Class Structure

// Generated: patient_care_patient_transformer.hpp
namespace patient_care::migration {

class PatientTransformer : public muscovite::dba::NatsEventConsumer {
public:
    explicit PatientTransformer(
        PatientCommandRepository& repo,
        muscovite::dba::ConnectionPool& pool,
        NatsMessageBroker& broker
    );

protected:
    // Called per message by NatsEventConsumer::run()
    ProcessResult process(const nats::Message& msg) override;

private:
    // Apply @trim, @default, @transform in declaration order
    std::string apply_transforms(const std::string& value,
                                  const std::vector<FieldTransform>& transforms);
    void route_to_dlq(const MigrationEnvelope& env,
                      const std::string& error_msg);

    PatientCommandRepository& repo_;
    muscovite::dba::ConnectionPool& pool_;
    NatsMessageBroker& broker_;
};

} // namespace patient_care::migration

Transform Execution Order

For each MAPPING field:

  1. Deserialize {"value": ..., "target_type": "..."} from envelope
  2. Apply transforms in declaration order (left-to-right as written in MAPPING block):
    • @trim — strip leading/trailing whitespace
    • @default("x") — substitute if value is null or empty
    • @transform(to_uuidv5) — apply named transform function
    • @format("YYYY-MM-DD") — reformat string values
  3. Validate via strong-type constructor or entity invariant CHECK

DLQ Routing

On validation failure, the transformer publishes to migration.dlq.{context}.{entity} with an augmented envelope:

{
  "$mapping": "LegacyPatientImport",
  "$source": "legacy_ehr.patients",
  "$batch_id": "...",
  "$sequence": 42,
  "$error": "Email validation failed: 'not-an-email' is not a valid email address",
  "$error_field": "email_addr",
  "$error_stage": "strong_type_validation",
  "fields": { ... }
}

The DLQ subject is consumed by the Triage UI (ADR 20260223-migration-triage-ui) for human review.

Idempotency

The target entity id is the UUIDv5 computed by the extractor (embedded in the envelope as fields.{pk_field}.value). The CommandRepository create() call uses ON CONFLICT (id) DO NOTHING for migration mode — a generated variant of the standard create().

Positive Consequences

  • CommandRepository path is enforced — domain events are emitted for all migrated records (audit trail)
  • Transform annotations are applied consistently in declaration order
  • Strong-type validation reuses existing C++ constructors — no separate validation logic
  • DLQ routing with full envelope + error context enables precise human triage
  • Idempotency via UUIDv5 + ON CONFLICT DO NOTHING makes re-runs safe

Negative Consequences

  • One NatsEventConsumer subclass per MAPPING block — potentially many generated classes for large migrations
  • ON CONFLICT DO NOTHING migration mode is a parallel code path in CommandRepository (two create() variants)
  • Transient infrastructure errors (NATS disconnect, PostgreSQL restart) during transform are retried by requeueing — delivery order may shift

Constraints

  • Transformer MUST use CommandRepository::create() — never raw pqxx::work.exec("INSERT ...")
  • DLQ MUST carry the full original envelope, not just the error message
  • Transformer MUST NOT silently discard records — route to DLQ or requeue, never drop
  • Generated code MUST carry dual-license SPDX header

AI Agent Guidance

When implementing the transformer generator:

  • New file: dslgen/generators/cpp/migration-transformer.rkt
  • Follow patterns from dslgen/generators/cpp/dba/saga-repository.rkt for multi-step orchestration
  • Base class for generated consumer: muscovite::dba::NatsEventConsumer (framework/dba/include/muscovite/dba/nats_event_consumer.hpp)
  • Subscribe subject: (format "migration.extract.~a.~a" (symbol->sql-id ctx-name) (symbol->sql-id entity-name))
  • DLQ subject: (format "migration.dlq.~a.~a" (symbol->sql-id ctx-name) (symbol->sql-id entity-name))
  • Transform application: emit a switch/if-chain on FieldTransform::type in declaration order
  • For ON CONFLICT DO NOTHING: add a second create_migration() method to the generated CommandRepository that appends ON CONFLICT (id) DO NOTHING
  • Test: integration test — transformer consumes a pre-seeded NATS subject, valid records appear in target DB, invalid records appear on DLQ subject
