Racket Generator Test Suite with CTest Integration and Coverage

Racket Generator Test Suite with CTest Integration and Coverage

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-10
  • Tags: testing, racket, generators, ctest, coverage, rackunit

Technical Story: Generator correctness is only verified by "file exists" checks. Need structured content-validation tests, unified runner, CTest integration, and code coverage tracking as a v1.0.0 release gate.

Context and Problem Statement

Muscovite has ~30 Racket test files across dslgen/tests/ and dslgen/test-*.rkt, but the test infrastructure has several gaps that undermine confidence in generator correctness:

  1. Mixed frameworks: ~12 files use rackunit (test-case, check-*), ~10 use manual displayln + unless/error. Manual-style tests don't propagate exit codes reliably, making automation impossible.
  2. No CTest integration: Zero CMake targets exist for any Racket test. All CTest infrastructure covers only C++ Catch2 tests. Racket tests must be run individually via racket dslgen/tests/test-xxx.rkt.
  3. No unified runner: No script discovers and executes all test files. No CI configuration for Racket tests exists.
  4. Shallow validation: Generator tests check file existence but never validate file content. A generator could emit syntactically broken SQL or C++ and tests would pass.
  5. Orphaned fixtures: Three fixture files in dslgen/tests/fixtures/ (specifications.ddd, value-objects.ddd, advanced-entity-features.ddd) have no consuming test.
  6. No coverage tracking: No raco cover integration exists to identify untested generator code paths.

Decision Drivers

  • Single source of truth: Generators are the only path from DSL to code — they must be provably correct
  • Fail-fast philosophy: Catch generation bugs before the C++ build (saves minutes per iteration)
  • Existing rackunit investment: 12 files already use it, standardizing reduces cognitive overhead
  • CTest as unified entry point: Developers already use ctest --preset debug — Racket tests should appear alongside C++ tests
  • No CI budget: No GitHub Actions pipeline — CTest is the only automation vehicle
  • Coverage visibility: Know which generator code paths are untested before declaring v1.0.0

Considered Options

Option A: Keep ad-hoc manual tests as-is

Continue running tests individually, accept mixed styles, no CTest integration.

  • Good, because zero effort
  • Bad, because manual-style tests don't fail reliably (exit code 0 on displayln errors)
  • Bad, because developers forget to run Racket tests
  • Bad, because no content validation — broken generators go undetected

Option B: Structured rackunit suite + CTest + raco cover

Migrate all tests to rackunit, add per-generator content-validation tests, integrate with CTest via add_test(), add raco cover for coverage tracking.

  • Good, because generators are verified at content level
  • Good, because CTest makes Racket tests first-class citizens
  • Good, because coverage identifies untested code paths
  • Good, because rackunit exit codes work with CTest
  • Neutral, because requires migrating ~10 manual tests
  • Bad, because upfront effort (~2 days)

Option C: Full CI pipeline with GitHub Actions

Everything in Option B, plus GitHub Actions workflow for ctest --preset racket.

  • Good, because automated on every push
  • Bad, because no budget for CI infrastructure
  • Bad, because requires Racket installation in CI image

Decision Outcome

Chosen option: Option B — Structured rackunit suite + CTest + raco cover.

This is a v1.0.0 release gate. Generator correctness must be verified before declaring the release.

Prerequisite: Migrate Manual Tests to rackunit

Convert ~10 manual-style tests from displayln + unless/error to proper rackunit test-suite + check-* assertions with exit-code propagation via rackunit/text-ui run-tests.

Migration pattern:

;; BEFORE (manual)
(define ctx (parse-file "examples/fhir-health/patient.ddd"))
(displayln (bounded-context-name ctx))  ;; no assertion

;; AFTER (rackunit)
(require rackunit rackunit/text-ui)
(define patient-suite
  (test-suite "Patient context parsing"
    (test-case "parses patient context"
      (define ctx (parse-file "examples/fhir-health/patient.ddd"))
      (check-equal? (bounded-context-name ctx) 'Patient))))
(module+ main (run-tests patient-suite))

Phase 1: Generator Smoke Tests

One new test-generator-{target}.rkt per generator family. Each test:

  1. Parses a minimal fixture DDD (not fhir-health/geosciences — too slow)
  2. Calls the generator
  3. Asserts no errors (no exceptions thrown)
  4. Asserts output files exist with expected names
  5. Uses make-temporary-directory for isolation, cleans up after

New files:

File Generator Target Key Assertions
test-generator-postgres.rkt postgres, postgres-unified sqitch.conf exists, deploy/*.sql files exist, sqitch.plan non-empty
test-generator-cpp-dba.rkt cpp-dba, unified-dba include/*/dba/entities/*.hpp exist, src/*/repositories/*.cpp exist
test-generator-grpc.rkt grpc proto/*.proto exist, src/*_service.cpp exist
test-generator-qt.rkt qt-widgets, qt-quick Widget *_edit_dialog.hpp exist, QML *ViewModel.hpp + *.qml exist
test-generator-cmake.rkt cmake CMakeLists.txt exists, CMakePresets.json exists

Test isolation pattern:

(define (with-temp-dir proc)
  (define dir (make-temporary-directory))
  (dynamic-wind
    void
    (λ () (proc dir))
    (λ () (delete-directory/files dir))))

Performance: Smoke tests use the small fixture DDD only (~50 lines). Full fhir-health (9 contexts) and geosciences (26 contexts) parsing/generation reserved for racket-integration CTest label (nightly/on-demand).

Phase 2: Scenario Content Validation

Create dslgen/tests/fixtures/generator-test.ddd — a compact but comprehensive DDD file covering all major DSL constructs:

CONTEXT TestDomain @uuid "..." {
    STRONG_TYPE Email BASED_ON TEXT
        CHECK email_format: VALUE ~ '^.+@.+$'
        DOC "Test email type";

    VALUE_OBJECT Address @inline @uuid "..." (
        street TEXT NOT NULL @uuid "...",
        city   TEXT NOT NULL @uuid "..."
    ) DOC "Test address";

    ENTITY Customer @uuid "..." IDENTIFIED_BY id (
        id         UUID        NOT NULL @uuid "...",
        name       TEXT        NOT NULL @uuid "...",
        email      Email       NOT NULL @uuid "...",
        created_at TIMESTAMPTZ NOT NULL @uuid "...",
        score      DECIMAL     NOT NULL @uuid "..."
    )
    RELATIONSHIPS {
        status -> CustomerStatus REQUIRED ONE;
    }
    PRESENTATION {
        SHORT: "{name}";
        FULL: "{name} ({email})";
        TABLE: [name, email, created_at];
    }
    DOC "Test customer entity";

    LOOKUP_TABLE CustomerStatus
        SEED "seeds/customer_status.json"
        DOC "Customer status codes";

    AGGREGATE Customer @uuid "..." ROOT Customer
        DOC "Customer aggregate";

    EVENT CustomerCreated @uuid "..." (
        customer_id UUID    NOT NULL,
        name        TEXT    NOT NULL,
        email       Email   NOT NULL
    ) DOC "Customer was created";

        WHERE status_id IS NOT NULL
        DOC "Customers with active status";

Phase 3: CTest Integration + Coverage

New file: dslgen/CMakeLists.txt

# Racket generator test suite
# Included from top-level CMakeLists.txt

find_program(RACKET_EXECUTABLE racket REQUIRED)

# --- Smoke tests (Phase 1) ---
set(RACKET_SMOKE_TESTS
    test-generator-postgres
    test-generator-cpp-dba
    test-generator-grpc
    test-generator-qt
    test-generator-cmake
)

foreach(test_name IN LISTS RACKET_SMOKE_TESTS)
    add_test(
        NAME racket::smoke::${test_name}
        COMMAND ${RACKET_EXECUTABLE}
            ${CMAKE_SOURCE_DIR}/dslgen/tests/${test_name}.rkt
        WORKING_DIRECTORY ${CMAKE_SOURCE_DIR}
    )
    set_tests_properties(racket::smoke::${test_name} PROPERTIES
        LABELS "racket;racket-smoke"
        TIMEOUT 60
    )
endforeach()

# --- Parser tests (existing, migrated to rackunit) ---
set(RACKET_PARSER_TESTS
    test-parser
    test-parser-features
    test-dsl-integration
    test-lookup-table-parsing
    test-lang-components
    test-tpt-inheritance
    test-instruct-block
    test-context-mapping
)

foreach(test_name IN LISTS RACKET_PARSER_TESTS)
    add_test(
        NAME racket::parser::${test_name}
        COMMAND ${RACKET_EXECUTABLE}
            ${CMAKE_SOURCE_DIR}/dslgen/tests/${test_name}.rkt
        WORKING_DIRECTORY ${CMAKE_SOURCE_DIR}
    )
    set_tests_properties(racket::parser::${test_name} PROPERTIES
        LABELS "racket;racket-parser"

endforeach()

# --- Scenario tests (Phase 2) ---

    COMMAND ${RACKET_EXECUTABLE}
        ${CMAKE_SOURCE_DIR}/dslgen/tests/test-dsl-integration.rkt
    WORKING_DIRECTORY ${CMAKE_SOURCE_DIR}

# --- Coverage target ---
find_program(RACO_EXECUTABLE raco)
if(RACO_EXECUTABLE)
    add_custom_target(racket-coverage
        COMMAND ${RACO_EXECUTABLE} cover
            -f html
            -d ${CMAKE_BINARY_DIR}/racket-coverage
            ${CMAKE_SOURCE_DIR}/dslgen/tests/
        WORKING_DIRECTORY ${CMAKE_SOURCE_DIR}
        COMMENT "Running raco cover on Racket test suite"
    )
endif()

# --- Convenience target: run all Racket tests ---
add_custom_target(racket-tests
    COMMAND ${RACKET_EXECUTABLE}
        ${CMAKE_SOURCE_DIR}/dslgen/tests/run-all-tests.rkt
    WORKING_DIRECTORY ${CMAKE_SOURCE_DIR}
    COMMENT "Running all Racket generator tests"
)

Unified runner: dslgen/tests/run-all-tests.rkt

Discovers all test-*.rkt files in the tests directory, requires each, runs all exported test suites via rackunit/text-ui, and exits with non-zero on any failure.

CTest preset (add to CMakePresets.json):

{
    "name": "racket",
    "displayName": "Racket Generator Tests",
    "configurePreset": "debug",
    "filter": {
        "include": {
            "label": "racket"
        }
    }
}

Usage:

ctest --preset racket                    # All Racket tests
ctest --preset debug -L racket-smoke     # Smoke tests only
ctest --preset debug -L racket-parser    # Parser tests only
ctest --preset debug -L racket-migration # Migration tests only
cmake --build --preset debug --target racket-coverage  # Coverage report

AI Agent Guidance

When This Decision Applies

  • Adding a new generator target (e.g., Rust backend) → must add test-generator-{target}.rkt
  • Adding a new DSL construct (e.g., SAGA) → must add assertions in the relevant generator test files
  • Modifying type-registry.rkt mappings → must verify content assertions still pass
  • Modifying any generator template → run ctest --preset racket to validate

When This Decision Does NOT Apply

  • C++ framework tests → use Catch2 (ADR 20260120)
  • Qt Quick UI tests → use QML TestCase (ADR 20260210)
  • Database integration tests → use ctest --preset debug -L db
  • Manual exploratory testing of generated code

Enforcement

  • CTest preset racket runs all Racket tests
  • rackunit exit codes propagate failures to CTest
  • raco cover generates coverage report to build/debug/racket-coverage/
  • All tests use make-temporary-directory — never write to generated-*/
