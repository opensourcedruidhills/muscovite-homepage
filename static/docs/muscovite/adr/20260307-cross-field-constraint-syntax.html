Cross-Field Constraint Annotation Syntax

Cross-Field Constraint Annotation Syntax

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-07
  • Tags: dsl, grammar, constraints, validation, cross-field

Technical Story: Some invariants depend on relationships between fields in the same entity (e.g., end_date must be after start_date). These cannot be expressed as per-field strong types and require a distinct annotation mechanism.

Context and Problem Statement

Per-field constraints (ADR 20260307-field-constraint-annotations) handle validation of individual field values in isolation. However, many domain invariants are relational — they express a constraint between two or more fields within the same entity:

  • end_date must be after start_date
  • discharge_date must not precede admission_date
  • max_value must not equal min_value
  • follow_up_date must be on the same day as appointment_date

These constraints cannot be encoded as strong types (Tier 1) because they require access to multiple field values simultaneously. They need a different generation target: the EntityValidator (Tier 2).

How should the DSL express cross-field constraints that generate EntityValidator rules?

Decision Drivers

  • Must reuse the annotation syntax pattern established by per-field constraints
  • Parser must validate that referenced fields exist in the same entity at parse time
  • Nullable fields must have well-defined semantics (what happens when one side is NULL?)
  • Cross-field constraints must be clearly distinguishable from per-field constraints in the IR
  • Generated code must target EntityValidator (Tier 2), not strong types (Tier 1)

Considered Options

  1. Cross-field annotations on the constrained field — @after(start_date) on the end_date field
  2. Separate INVARIANTS block per entity — INVARIANTS { end_date > start_date; }
  3. Expression-based constraints — @constraint("end_date > start_date") as free-form expression

Decision Outcome

Chosen option: Option 1 — Cross-field annotations on the constrained field, because it maintains the annotation-per-field pattern, is parseable with the existing grammar extension mechanism, and keeps constraints co-located with the field they primarily affect.

Supported Cross-Field Annotations

Annotation Applies To Meaning
@after(field_name) Date, DateTime This field's value > referenced field's value
@before(field_name) Date, DateTime This field's value < referenced field's value
@same_day(field_name) Date, DateTime Same calendar day as referenced field
@not_equal(field_name) Any comparable type This field's value ≠ referenced field's value

IR Representation

Cross-field constraints reuse the field-constraint struct with relative? = #t:

(field-constraint 'after 'start_date #t)
;; kind: 'after
;; value: symbol — the referenced field name
;; relative?: #t — marks this as a cross-field constraint

The parser validates at parse time that the referenced field symbol exists within the same entity. A parse error is raised if the referenced field is not found.

Nullable Semantics

If either the annotated field or the referenced field is nullable and its value is NULL, the constraint is vacuously true. This follows SQL NULL semantics: NULL > '2026-01-01' evaluates to UNKNOWN, which is not a constraint violation.

Rationale: requiring both fields to be non-null would make nullable fields unusable with cross-field constraints, which contradicts real-world domain models (e.g., discharge_date is nullable because the patient may still be admitted).

Generation Target

Cross-field constraints generate EntityValidator rules (Tier 2), not strong types (Tier 1). The EntityValidator receives the fully constructed entity and checks relational invariants:

// Generated EntityValidator rule for: end_date @after(start_date)
if (entity.end_date().has_value() && entity.start_date().has_value()) {
    if (*entity.end_date() <= *entity.start_date()) {
        errors.add("end_date", "must be after start_date");
    }
}

Positive Consequences

  • Consistent annotation syntax — cross-field constraints look like per-field constraints
  • Parse-time validation catches typos in referenced field names immediately
  • Clear IR distinction (relative? = #t) enables generators to route to the correct tier
  • Nullable semantics are well-defined and follow SQL conventions

Negative Consequences

  • SQL cannot express these as CHECK constraints (PostgreSQL CHECK constraints cannot reference other columns in a meaningful cross-row way, and cross-column CHECKs are limited) — enforcement is application-side only
  • Only binary constraints (between two fields) — ternary or higher-arity constraints need a future extension
  • The annotation is on one field but references another, which could be confusing when reading .ddd files

AI Agent Guidance

When This Decision Applies

  • When parsing field annotations that reference other field names (the value is a symbol, not a number)
  • When generating EntityValidator code — cross-field constraints produce Tier 2 validation rules
  • When extending the PEG grammar for new cross-field annotation types
  • When handling nullable fields in cross-field validation — apply vacuous truth semantics

When This Decision Does NOT Apply

  • Per-field constraints (@min, @max, @not_empty) — those are relative? = #f, see ADR 20260307-field-constraint-annotations
  • Cross-entity constraints (relationships between different entities) — those use RELATIONSHIPS {} blocks
  • Business rules requiring external state (database lookups, API calls) — those are Tier 3 (PluginValidator)

Enforcement

  • Parser validates referenced field exists in same entity at parse time
  • Grammar conformance test verifies cross-field annotation syntax
  • Generator tests verify EntityValidator output for cross-field constraints
  • Nullable semantics enforced by generator template (vacuous truth pattern)
