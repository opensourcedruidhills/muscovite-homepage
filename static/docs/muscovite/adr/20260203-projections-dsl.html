Projections DSL for Read-Optimized Views

Projections DSL for Read-Optimized Views

  • Status: accepted
  • Date: 2026-02-03
  • Tags: projections, cqrs, read-models, materialized-views, dsl

Context and Problem Statement

With the Event Consumers infrastructure complete (v0.9.0 Phase 1), we can now build projections — read-optimized views that aggregate data from domain events. Projections enable:

  • Denormalized read models for dashboard/reporting queries
  • Cross-aggregate views joining data from multiple bounded contexts
  • Pre-computed aggregations (counts, sums, latest values)
  • Eventual consistency between write and read models

The question is: how do we declare projections in the DSL and generate the appropriate infrastructure?

Decision Drivers

  • DDD principles: Projections are derived from domain events, not source tables
  • Declarative: Users specify WHAT the projection contains, not HOW to refresh
  • Multiple refresh strategies: Event-driven (real-time), scheduled, on-demand
  • SQL generation: Target PostgreSQL VIEWs or MATERIALIZED VIEWs
  • Type safety: Projections are strongly typed across layers

Considered Options

  1. SQL-only projections: Hand-write VIEWs, no DSL support
  2. Event handler DSL: Define projections as event handlers with state
  3. Declarative projection DSL: PROJECTION keyword with source/select syntax (chosen)
  4. External tool: Use dbt or similar for projections

Decision Outcome

Chosen option: Declarative projection DSL, because it fits naturally with the existing DSL syntax, enables code generation for all layers, and keeps the single-source-of-truth principle.

DSL Syntax

Note: The FROM clause uses a string literal to contain the SQL query. This allows arbitrary SQL including commas, subqueries, and complex expressions without conflicting with the DSL grammar.

CONTEXT Reporting {

    -- Simple denormalized view joining entities
    PROJECTION ActivePatientSummary @uuid "..." (
        patient_id          UUID        NOT NULL,
        mrn                 TEXT        NOT NULL,
        full_name           TEXT        NOT NULL,
        gender_display      TEXT,
        encounter_count     INTEGER     NOT NULL DEFAULT 0,
        last_encounter_date DATE,
        primary_condition   TEXT
    )
    FROM "SELECT p.id, p.mrn, pn.family || ', ' || pn.given, g.display,
          COALESCE(e.cnt, 0), e.last_date, c.display
          FROM Patient.Patient p
          LEFT JOIN Patient.PatientName pn ON p.id = pn.patient_id AND pn.use = 'official'
          LEFT JOIN Basedata.AdministrativeGender g ON p.gender_id = g.id
          LEFT JOIN (
              SELECT patient_id, COUNT(*) as cnt, MAX(period_start) as last_date
              FROM Encounter.Encounter
              GROUP BY patient_id
          ) e ON p.id = e.patient_id
          WHERE p.active = true"
    REFRESH ON_DEMAND
    DOC "Active patients with encounter statistics for dashboard";

    -- Event-driven projection updated by domain events
    PROJECTION PatientEncounterTimeline @uuid "..." (
        id                  UUID        NOT NULL,
        patient_id          UUID        NOT NULL,
        event_timestamp     TIMESTAMPTZ NOT NULL,
        event_type          TEXT        NOT NULL,
        event_summary       TEXT
    )
    REFRESH ON_EVENT Patient.PatientCreated, Encounter.EncounterCreated
    HANDLER patient_timeline_handler
    DOC "Timeline of patient events for clinical view";

    -- Scheduled aggregation
    PROJECTION DailyEncounterStats @uuid "..." (
        stat_date           DATE        NOT NULL,
        encounter_count     INTEGER     NOT NULL,
        unique_patients     INTEGER     NOT NULL,
        avg_duration_hours  DECIMAL(5,2)
    )
    REFRESH SCHEDULED "0 1 * * *"  -- cron expression: 1 AM daily
    DOC "Daily encounter statistics for reporting";

}

Refresh Strategies

Strategy Keyword Generated Use Case
On-demand REFRESH ON_DEMAND PostgreSQL MATERIALIZED VIEW Reporting dashboards, batch analytics
Event-driven REFRESH ON_EVENT <events> EventConsumer handler + table Real-time read models, notifications
Scheduled REFRESH SCHEDULED "<cron>" Cron job + REFRESH MATERIALIZED VIEW Periodic aggregations
Live REFRESH LIVE (default) PostgreSQL VIEW Always-current data, small datasets

Generated Artifacts

1. PostgreSQL (Sqitch migrations)

ON_DEMAND / SCHEDULED → MATERIALIZED VIEW:

-- deploy/reporting-projection-active_patient_summary.sql
CREATE MATERIALIZED VIEW reporting.active_patient_summary AS
SELECT 
    p.id AS patient_id,
    p.mrn,
    COALESCE(pn.given, '') || ' ' || COALESCE(pn.family, '') AS full_name,
    g.display AS gender_display,
    COALESCE(e.cnt, 0) AS encounter_count,
    e.last_date AS last_encounter_date,
    NULL AS primary_condition  -- computed separately
FROM patient.patient p
LEFT JOIN patient.patient_name pn ON p.id = pn.patient_id AND pn.use = 'official'
LEFT JOIN basedata.administrative_gender g ON p.gender_id = g.id
LEFT JOIN (
    SELECT patient_id, COUNT(*) as cnt, MAX(period_start) as last_date
    FROM encounter.encounter
    GROUP BY patient_id
) e ON p.id = e.patient_id
WHERE p.active = true;

CREATE UNIQUE INDEX idx_active_patient_summary_patient_id 
    ON reporting.active_patient_summary(patient_id);

-- Refresh function
CREATE OR REPLACE FUNCTION reporting.refresh_active_patient_summary()
RETURNS void AS $$
BEGIN
    REFRESH MATERIALIZED VIEW CONCURRENTLY reporting.active_patient_summary;
END;
$$ LANGUAGE plpgsql;

Projection Lifecycle

┌─────────────────────────────────────────────────────────────┐
│                    Projection Lifecycle                     │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ON_DEMAND:                                                 │
│    App startup → Check if exists → Create if not           │
│    User request → Call refresh_<projection>() function     │
│                                                             │
│  SCHEDULED:                                                 │
│    App startup → Register cron job                         │
│    Cron fires → Call REFRESH MATERIALIZED VIEW CONCURRENTLY│
│                                                             │
│  ON_EVENT:                                                  │
│    App startup → Register EventConsumer handlers           │
│    Event received → Handler inserts/updates projection row │
│    Query → Read from projection table                      │
│                                                             │
│  LIVE:                                                      │
│    Query → View executes underlying SELECT                 │
│                                                             │
└─────────────────────────────────────────────────────────────┘

Parser Changes

Add to parser/parser.rkt:

;; Projection struct
(struct projection
  (name          ; Symbol
   uuid          ; String (UUID)
   fields        ; List of field
   source        ; String (FROM ... clause) or #f
   refresh-mode  ; Symbol: 'on-demand, 'scheduled, 'on-event, 'live
   refresh-spec  ; String (cron) or list of event types
   handler       ; Symbol (handler function name) or #f
   doc           ; String or #f
   synonyms)     ; List of string
  #:transparent)

;; Parse PROJECTION keyword
(define (parse-projection ps)
  (expect-keyword! ps "PROJECTION")
  ...)

Generator Updates

Generator Output
postgres/sqitch-unified.rkt CREATE VIEW / MATERIALIZED VIEW, refresh functions
cpp/dba/projections.rkt Projection service with event handlers (NEW)
grpc/proto.rkt Query messages for projection data
grpc/cpp-service-v2.rkt QueryService methods for projections

Positive Consequences

  • Declarative: Projections defined alongside domain model
  • Type-safe: Generated structs/messages for projection data
  • Flexible refresh: Multiple strategies for different use cases
  • Testable: Event-driven projections can be unit tested
  • Migrations: Projections versioned with Sqitch like tables

Negative Consequences

  • Learning curve: New DSL syntax to learn
  • Complexity: Event-driven projections require careful handler design
  • Consistency: ON_EVENT projections are eventually consistent
  • Performance: Complex joins in LIVE views may be slow

Example: Patient Dashboard

CONTEXT Dashboard {

    -- Real-time patient count (event-driven)
    PROJECTION PatientCountByGender @uuid "..." (
        gender_code         TEXT        NOT NULL PRIMARY KEY,
        patient_count       INTEGER     NOT NULL DEFAULT 0,
        last_updated        TIMESTAMPTZ NOT NULL DEFAULT NOW()
    )
    REFRESH ON_EVENT Patient.PatientCreated, Patient.PatientUpdated, Patient.PatientDeleted
    HANDLER gender_count_handler
    DOC "Patient count by gender, updated in real-time";

    -- Scheduled daily stats (batch)
    PROJECTION DailyPatientStats @uuid "..." (
        stat_date           DATE        NOT NULL PRIMARY KEY,
        new_patients        INTEGER     NOT NULL,
        active_patients     INTEGER     NOT NULL,
        encounters_started  INTEGER     NOT NULL
    )
    REFRESH SCHEDULED "0 2 * * *"
    DOC "Daily statistics computed at 2 AM";

}

Migration Path

  1. Phase 1 (this ADR): Define syntax, implement parser
  2. Phase 2: SQL generator for VIEW/MATERIALIZED VIEW
  3. Phase 3: C++ ProjectionService with EventConsumer integration
  4. Phase 4: gRPC QueryService bindings
  5. Phase 5: Scheduled refresh infrastructure (cron)

AI Agent Guidance

When This Decision Applies

  • When adding projection support to the DSL parser
  • When generating SQL for read-optimized views
  • When implementing event-driven projection handlers
  • When adding query methods for projection data

When This Decision Does NOT Apply

  • Simple queries on single entities (use QueryRepository)
  • Temporary analysis queries (use SQL directly)
  • Ad-hoc reporting (use BI tools)
