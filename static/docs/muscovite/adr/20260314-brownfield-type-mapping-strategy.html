ADR: Brownfield Type Mapping Strategy — Grammar Types vs Extractor Aliases

ADR: Brownfield Type Mapping Strategy — Grammar Types vs Extractor Aliases

Date: 2026-03-14 Status: accepted Deciders: Johannes Lochmann

Context

Real-world brownfield extraction encounters 23+ data types from legacy database systems (SQL Server, Oracle, MySQL) that must be represented in Muscovite .ddd files. The DSL type system (PEG grammar BaseType rule) already covers 15 core types plus geospatial. The question is: which legacy types should become first-class grammar types, and which should be normalized by the extractor to existing types?

This decision has compounding effects — every grammar type must be mapped across all 8 generator targets (PostgreSQL, C++ entity, C++ DBA, gRPC Proto, Qt Widgets, QML, fact base, documentation). Adding a type that's semantically identical to an existing one creates maintenance burden without semantic value.

Types Encountered

From extraction of 757 entities across 36 legacy SQL Server models:

  • 8 types that already exist in the grammar (false alarms from agents using wrong DSL names)
  • 5 types that are semantically distinct and have direct PostgreSQL equivalents
  • 6 types that are vendor-specific aliases for existing DSL types
  • 4 types that need structural constructs beyond simple type additions

Decision

Principle: Grammar Types Must Be Semantically Distinct

A type enters the grammar only if it represents a semantic distinction that PostgreSQL (our primary target) treats differently. Vendor-specific spellings of the same semantic type are normalized by the extractor.

Decision framework for any future type:

  1. Does PostgreSQL have a distinct native type for it? → Grammar candidate
  2. Does it generate different DDL than any existing grammar type? → Grammar candidate
  3. Is it just a spelling variant of an existing type? → Extractor alias
  4. Is it vendor-specific with no PG equivalent? → Extractor maps to closest PG type

Grammar Additions (5 types)

DSL Type PG Type C++ Type Why grammar-level
SMALLINT SMALLINT int16_t Distinct storage size; PG treats differently from INTEGER
FLOAT DOUBLE PRECISION double IEEE 754 double; semantically different from DECIMAL
BYTEA BYTEA std::vector<uint8_t> Binary data; no existing grammar type covers this
TIME TIME std::chrono::hh_mm_ss Time-of-day without date; distinct from DATETIME/TIMESTAMPTZ
XML XML std::string PG has native XML type with XPath support

Traceability Requirement

Every extractor-normalized type must preserve the original via @source:

amount DECIMAL(19,4) @source "money" @uuid "...",

This ensures modernization agents can trace back to the legacy schema.

VIEW_ENTITY Keyword

Entities extracted from database views use the VIEW_ENTITY keyword instead of ENTITY. This signals to generators:

  • Always generate surrogate id UUID PK (drop synthetic composite PKs)
  • Generate read-only operations only (no INSERT/UPDATE/DELETE)
  • Tag as view-sourced in fact base and documentation

FK Inference

Per ADR 20260106 (explicit relationships only), the grammar never infers FKs. The extractor runs a post-extraction heuristic pass that generates candidate RELATIONSHIPS blocks with confidence scores. HIGH/MEDIUM are emitted active; LOW are emitted as comments for human review.

Consequences

Positive

  • Grammar stays clean: 20 core types (15 existing + 5 new), not 35+ vendor spellings
  • Extractors for different source databases (SQL Server, Oracle, MySQL, PostgreSQL) share the same target type system
  • @source annotation provides full traceability without grammar pollution
  • Decision framework scales to future vendor types without case-by-case debate

Negative

  • Extraction agents must consult the mapping table — extracting raw DDL types produces invalid .ddd files
  • Lossy mappings exist: MONEY→DECIMAL loses the "this is currency" semantic; REAL→FLOAT widens precision. Both are documented via @source.

Neutral

  • Grammar version bump required (coordinate with FTR-412 grammar merge)
  • All 8 generator targets must handle 5 new types — mechanical but wide impact

Implementation

  • v1.65.3: FTR-431 (grammar types), FTR-432 (VIEW_ENTITY)
  • v1.67.0: FTR-433 (extractor type mapping table), FTR-434 (FK inference)
