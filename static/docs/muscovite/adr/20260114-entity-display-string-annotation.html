Entity Display String Annotation

Entity Display String Annotation

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-01-14
  • Tags: dsl, qt, localization, ui

Context and Problem Statement

When entities are displayed in FK comboboxes, table cells, navigation breadcrumbs, or detail labels, we need human-readable display strings. Currently the generator uses heuristics to pick a display field (name, code, or fallback to id), but this approach:

  1. Cannot compose multiple fields (e.g., "Dr. John Smith (Cardiology)")
  2. Has no localization support - format patterns need translation for different locales
  3. Provides no control over different display contexts (short vs. full vs. table row)
  4. Cannot handle fallback chains when preferred fields are empty

Decision Drivers

  • Flexibility: Different UI contexts need different display formats
  • Localization: Format patterns must work with Qt's tr() and qsTr() systems
  • Type Safety: Field references must be validated at generation time
  • Simplicity: DSL syntax should be intuitive and readable
  • Performance: Display composition should be efficient (client-side)

Considered Options

  1. Simple template string: DISPLAY AS "{family_name}, {given_name}"
  2. With fallback chain: DISPLAY AS "{name}" FALLBACK "{code}" FALLBACK "{id}"
  3. Explicit localization key: DISPLAY AS "{family_name}, {given_name}" KEY "patient_display"
  4. Multiple formats for contexts: DISPLAY (SHORT: ..., FULL: ..., TABLE: ...)

Decision Outcome

Chosen option: Option 4 - Multiple formats for different contexts, because:

  • Different UI elements need different verbosity levels
  • ComboBox needs SHORT, detail view needs FULL, table cells need TABLE format
  • Each format can have its own fallback chain
  • Translation files can provide locale-specific patterns per format

DSL Syntax

ENTITY Patient IDENTIFIED_BY id (
    id              UUID        NOT NULL,
    given_name      TEXT        NOT NULL,
    family_name     TEXT        NOT NULL,
    mrn             TEXT        NOT NULL @unique,
    prefix          TEXT,
    birth_date      DATE
)
PRESENTATION (
    SHORT: "{mrn}",
    FULL: "{prefix} {given_name} {family_name} ({mrn})",
    TABLE: "{family_name}, {given_name}"
)
DOC "A patient in the healthcare system";

Note: The keyword is PRESENTATION rather than DISPLAY to avoid conflicts with the common field name display used in lookup tables (e.g., AdministrativeGender.display).

Format Types: | Format | Usage | Example Output | |--------|-------|----------------| | SHORT | ComboBox items, breadcrumbs | "MRN12345" | | FULL | Detail headers, tooltips | "Dr. John Smith (MRN12345)" | | TABLE | Table cells, list items | "Smith, John" |

Default Behavior:

  • If only one format specified, it becomes the default for all contexts
  • If no DISPLAY clause, fall back to heuristic: name > code > first TEXT field > id

Client-Side Composition

Display strings are composed in the Qt client, NOT in proto messages or server-side code. Reasons:

  1. Localization: Qt's tr() system handles runtime language switching
  2. Format patterns: Translators can reorder placeholders (%1 %2 → %2 %1)
  3. No proto bloat: Proto messages stay lean with actual field data
  4. Flexibility: Client can choose format based on UI context

Generated Code

Per-entity DisplayHelper class:

// Generated in patient_display.hpp
namespace patient::qt {

enum class DisplayFormat { Short, Full, Table, Default = Full };

class PatientDisplayHelper {
public:
    static QString displayString(const patient::Patient& entity,
                                 DisplayFormat format = DisplayFormat::Default) {
        switch (format) {
        case DisplayFormat::Short:
            // Pattern: "{mrn}"
            return QString::fromStdString(entity.mrn());
            
        case DisplayFormat::Table:
            // Pattern: "{family_name}, {given_name}"
            return QObject::tr("%1, %2", "Patient.TABLE")
                .arg(QString::fromStdString(entity.family_name()))
                .arg(QString::fromStdString(entity.given_name()));
            
        case DisplayFormat::Full:
        default:
            // Pattern: "{prefix} {given_name} {family_name} ({mrn})"
            return QObject::tr("%1 %2 %3 (%4)", "Patient.FULL")
                .arg(QString::fromStdString(entity.prefix()))
                .arg(QString::fromStdString(entity.given_name()))
                .arg(QString::fromStdString(entity.family_name()))
                .arg(QString::fromStdString(entity.mrn()));
        }
    }
};

} // namespace patient::qt

Integration Points

  1. Qt Model data(): Use DisplayHelper::displayString(item, DisplayFormat::Table) for DisplayRole
  2. ComboBox population: Use DisplayFormat::Short for dropdown items
  3. Detail dialogs: Use DisplayFormat::Full for window titles
  4. QML: Expose via Q_INVOKABLE QString displayString(int row, int format)

Validation Rules

Code Severity Description
E030 Error Field referenced in DISPLAY pattern doesn't exist
E031 Error DISPLAY format string has unclosed brace
E032 Error Unknown format type (not SHORT/FULL/TABLE)
W030 Warning Entity has no DISPLAY and no suitable display field

Fallback Chain Support

For optional fields that might be empty:

DISPLAY (
    SHORT: "{code}" OR "{id}",
    FULL: "{name}" OR "{code}" OR "{id}"
)

Generated code handles fallbacks:

case DisplayFormat::Short:
    if (!entity.code().empty())
        return QString::fromStdString(entity.code());
    return QString::fromStdString(entity.id());

Consequences

Positive

  • Full control over entity display in all UI contexts
  • Proper Qt localization with translator-controllable patterns
  • No proto message bloat (display computed client-side)
  • Compile-time validation of field references
  • Fallback chains handle optional fields gracefully

Negative

  • Additional DSL syntax complexity
  • Generated DisplayHelper classes add to code size
  • Translators need to understand placeholder semantics

Neutral

  • Requires Qt linguist workflow for translations
  • Display patterns defined at model level, not UI level

AI Agent Guidance

When This ADR Applies

  • Generating Qt models or widgets for entities
  • Populating FK comboboxes with entity references
  • Displaying entity identifiers in UI elements

When This ADR Does NOT Apply

  • Server-side code (proto messages, gRPC services)
  • Database layer (SQL, stored procedures)
  • Non-Qt clients (CLI tools, web frontends)

Key Implementation Details

  • Display patterns use {field_name} syntax in DSL
  • Generated code uses Qt tr() with %1, %2, ... placeholders
  • Format types: SHORT, FULL, TABLE (case-insensitive in DSL)
  • Default format is FULL when unspecified
  • Heuristic fallback when no DISPLAY clause: name > code > first TEXT > id
