Complete Example Submodule Decoupling — Internal Test Harnesses

Complete Example Submodule Decoupling — Internal Test Harnesses

  • Status: accepted (supersedes initial version merged in PR #167)
  • Deciders: Johannes Lochmann
  • Date: 2026-03-16
  • Tags: testing, architecture, submodules, cmake, documentation
  • Related: ADR 20260312 — Example Repos as Independent Consumers

Technical Story: Complete the decoupling started in v1.65.2 (FTR-407). Example submodules are independent consumer repos, but 101 references across the monorepo still depend on their content — Racket tests, CMake build targets, documentation publishing, workflow scripts, and shell tools.

Context and Problem Statement

ADR 20260312 (v1.65.2) freed example projects from the release ceremony — they no longer get generated, committed, or tagged by release.sh. However, the decoupling was incomplete. A full audit reveals 101 references to examples/ across ~40 files in the monorepo:

Coupling by Category

Category Files References Impact
Racket tests 12 17 Tests fail without submodule init
CMake build system 5 17 DB deploy, NATS setup, doc build, L2 verification all reference examples/
Shell tools 8 22 Doc publishing, metrics, linting iterate over examples/
Racket tools 5 9 DSL reference generators read hello-world; diagram/validator/viewer default to examples/
Workflow scripts 4 7 agent-pr.sh, effort estimation, system brief reference examples/
Coordinator 1 1 Hardcoded fhir-health generation command
Generator comments ~8 ~8 Non-functional but misleading
Config/tmp 2 3 IDE config, throwaway scripts

Specific High-Cost Coupling

  1. Database CMake targets (database/CMakeLists.txt): Registers fhir and geosciences as database projects. Every cmake --preset debug configures Sqitch targets, NATS setup, and CTest fixtures for these example databases.
  2. Release gate db-fresh-fhir (release.sh): Drops, creates, and deploys the entire fhir-health schema on every release. Requires Docker + PostgreSQL.
  3. Documentation publishing (tools/publish-docs.sh, cmake/Documentation.cmake): Generates system manuals, fact bases, and audience reports from example project .mus files. Deploys to homepage.
  4. Layer 2 verification (cmake/Layer2Verification.cmake): Checks sidecar file integrity against hello-world, fhir-health, and geosciences generated output.
  5. DSL reference generators (dslgen/tools/generate-dsl-reference.rkt, generate-project-dsl-reference.rkt): Read hello-world to produce DSL.md and PROJECT-DSL.md.

Cost per Iteration

  • git submodule update --init --force: ~85 MB download, ~30s
  • Database deploy gate (fhir-health): Docker + Sqitch, ~2–5 min
  • Documentation generation from examples: ~3–5 min
  • Example-dependent Racket tests: adds submodule fragility to every test run
  • Total per release ceremony: ~10–15 min of example-dependent overhead

Decision Drivers

  • Complete decoupling: Every reference to examples/ in the framework must be eliminated or replaced with internal harness paths.
  • Test isolation: Tests must depend only on files within the muscovite repo.
  • Coverage preservation: Every DSL construct currently exercised by example projects must remain covered by internal harnesses.
  • Wall-clock savings: Eliminating submodule init, example DB deploy, and example doc generation saves ~10–15 min per release.
  • Token/compute savings: No example code generation, compilation, or deployment during framework iterations.
  • Documentation sovereignty: Example projects publish their own docs independently; the framework publishes framework docs only.

Considered Options

Option A: Kitchen-Sink Test Project

A single large .mus + multiple .ddd files exercising every construct.

  • Good, because single target to maintain.
  • Bad, because grows unwieldy — one failure blocks all tests.

Option B: Multiple Focused Test Harnesses

Several small .mus + .ddd sets under dslgen/tests/harness/, each targeting one concern. All framework references point to harnesses instead of submodules.

  • Good, because each harness is small and purposeful.
  • Good, because failure in one harness isolates the broken construct.
  • Good, because harnesses are versioned with the generator code.

Option C: Grammar-Derived Fuzz Testing

A Racket script synthesizes valid .ddd from the PEG grammar.

  • Good, because maximum coverage.
  • Bad, because complex to build and doesn't replace functional references.

Option D: Hybrid (B now, add C later)

Deploy focused harnesses immediately. Add grammar-derived fuzzing as a future FTR.

Decision Outcome

Chosen option: Option D (Hybrid), because it delivers immediate, complete decoupling (Option B) while preserving a clear path to grammar-derived fuzz testing (Option C) as a future enhancement.

What Harnesses Are NOT

  • Not example projects — they have no independent lifecycle, no repo, no release.
  • Not consumer-facing — they are internal test fixtures, not documentation.
  • Not domain models — they use synthetic domains designed to exercise DSL edges.
  • The test runner owns them. They exist to break when the generator regresses.

Positive Consequences

  • Zero submodule coupling: Tests, builds, docs, and releases depend only on files within the muscovite repo.
  • ~10–15 min saved per release: No example DB deploy, no example doc generation, no submodule fetch.
  • Reduced CI fragility: No submodule auth failures, no stale submodule refs.
  • Clean documentation boundary: Framework publishes framework docs; consumer repos publish their own docs.
  • Accurate metrics: Test and build metrics reflect the framework only.

Negative Consequences

  • Harness .ddd files must be authored — one-time migration effort (~40 files modified, ~16 files created).
  • Harnesses lack real-world domain richness (acceptable — that's what consumer repos demonstrate).
  • Example submodule removal is a one-way door — re-adding is possible but noisy.
  • DSL reference documentation (DSL.md, PROJECT-DSL.md) will use synthetic examples from harness/minimal instead of hello-world's real domain.

Migration Sequence

  1. Create harnesses under dslgen/tests/harness/ (6 directories).
  2. Repoint Racket tests — change define-runtime-path from ../../examples/ to harness/ paths. Include migration test files in dslgen/migration/.
  3. Decouple CMake — update database/CMakeLists.txt (replace example DB projects with harness), remove example targets from Documentation.cmake, DockerNats.cmake, Layer2Verification.cmake.
  4. Decouple tools — update publish-docs.sh (remove example sections), lint-ddd.sh (scan harness/), generate-dsl-reference.rkt and generate-project-dsl-reference.rkt (use harness/minimal), update default paths in diagram.rkt, validator.rkt, viewer.rkt.
  5. Decouple workflow scripts — update agent-pr.sh, generate-effort- estimation.sh, generate-system-brief.sh, release.sh DB gate.
  6. Move tpt-inheritance-test.ddd from examples/ root into dslgen/tests/fixtures/.
  7. Remove example submodules from .gitmodules and clean up with git rm --cached examples/*.
  8. Update documentation — AGENTS.md, README.md metrics section.
  9. Future FTR: Grammar-derived fuzz testing (Option C / path to D).

AI Agent Guidance

When This Decision Applies

  • When adding a new DSL construct: add coverage to the appropriate harness.
  • When writing or modifying Racket tests: reference harness/ paths, never examples/.
  • When modifying CMake targets: do not add example project dependencies.
  • When modifying documentation tools: do not reference example projects.
  • When modifying CI/workflow scripts: do not iterate over examples/.

When This Decision Does NOT Apply

  • Consumer repos (fhir-health, geosciences, etc.) manage their own build, test, documentation, and release strategies independently.
  • tools/build-example.sh may be kept as a consumer utility shipped with the framework, but should not be invoked from monorepo build/CI.

Enforcement

  • test-grammar-conformance.rkt scans harness/ instead of examples/. Any define-runtime-path pointing to ../../examples/ is a conformance violation.
  • CI does not run git submodule update --init for example submodules.
  • release.sh DB gate targets db-fresh-harness, not db-fresh-fhir.
  • PR review should reject new references to examples/ in framework code.

Key Files

  • dslgen/tests/harness/ — all test harness .mus and .ddd files
  • dslgen/tests/fixtures/ — existing single-file test fixtures (unchanged)
  • database/CMakeLists.txt — DB project registrations (harness replaces examples)
  • cmake/Documentation.cmake — example doc targets removed
  • tools/publish-docs.sh — example doc publishing removed
  • workflow/scripts/release.sh — DB gate uses harness
  • .gitmodules — example submodules removed after migration
