Unconditional Domain Events Table Generation

Unconditional Domain Events Table Generation

  • Status: accepted
  • Date: 2026-01-24
  • Tags: domain-events, sqitch, cqrs, event-sourcing, sql-generation

Context and Problem Statement

The C++ DBA generator unconditionally generates DomainEventRepository for every bounded context, which emits domain events via:

INSERT INTO {schema}.domain_events (event_type, aggregate_id, ...)

However, the SQL Sqitch generator conditionally generates domain_events tables only when the context has DSL EVENT definitions. This mismatch causes runtime errors:

ERROR: relation "practitioner.domain_events" does not exist

When a user creates or updates an entity via gRPC, the CommandService emits a domain event, but the table doesn't exist because no EVENT was declared in the DDD model.

Decision Drivers

  • Runtime errors: C++ code crashes when no domain_events table exists
  • CQRS-ready architecture: ADR 20260120 requires events for all commands
  • Audit trail: All CRUD operations should leave an audit trail
  • Consistency: Same behavior across all bounded contexts
  • Minimal code change: Prefer smallest fix with highest impact

Considered Options

  1. Option A: Make C++ event emission conditional - Skip emit() when context has no EVENT definitions
  2. Option B: Make SQL table generation unconditional - Always generate domain_events table
  3. Option C: Add EVENT declarations to DDD files - Explicit event modeling in DSL

Decision Outcome

Chosen option: "Option B - Make SQL table generation unconditional", because it aligns with the CQRS-ready architecture (ADR 20260120) which expects events to be emitted for all commands, and requires minimal code change (1 file, ~5 lines).

Implementation

Modify src/dslgen/generators/postgres/sqitch-unified.rkt lines 765-769:

;; BEFORE (conditional)
(define event-changes
  (filter identity
   (for/list ([ctx contexts])
     (if (not (null? (bounded-context-events ctx)))
         (add-change-if-new (make-events-table-change ctx))
         #f))))

;; AFTER (unconditional)
(define event-changes
  (for/list ([ctx contexts])
    (add-change-if-new (make-events-table-change ctx))))

Regeneration Required

After the fix:

  1. Regenerate: racket generate-fhir-health.rkt --sqitch
  2. Redeploy: sqitch deploy db:pg://postgres:postgres@localhost:5555/fhir

Positive Consequences

  • All CommandService operations emit events (audit trail)
  • CQRS-ready architecture fully supported
  • Consistent behavior: every context has domain_events table
  • No runtime errors from missing tables
  • Minimal code change (1 file)

Negative Consequences

  • Creates domain_events tables in contexts that may never emit events (negligible storage)
  • Slight increase in migration complexity (one extra table per schema)

Pros and Cons of the Options

Option A: Make C++ Event Emission Conditional

Modify generators to skip event emission when context has no EVENT definitions.

  • Good, because less database overhead
  • Good, because consistent with DSL "no events = no table" semantics
  • Bad, because loses audit trail for operations
  • Bad, because when events are later added, historical data isn't available
  • Bad, because requires changes to multiple C++ generators

Option B: Make SQL Table Generation Unconditional (Chosen)

Always generate domain_events table for every bounded context.

  • Good, because all CRUD operations leave audit trail
  • Good, because CQRS-ready architecture already expects this
  • Good, because minimal code change (1 file, ~5 lines)
  • Good, because consistent behavior across all contexts
  • Bad, because creates empty tables in contexts that never emit events (negligible)

Option C: Add EVENT Declarations to DDD Files

Add explicit EVENT PatientCreated (...) declarations to each context.

  • Good, because explicit domain modeling
  • Good, because full DDD event sourcing with proper event types
  • Bad, because requires extensive DDD file modifications
  • Bad, because overkill if events are only used for audit logging
  • Bad, because adds DSL complexity for generic CRUD events
