Geography Over Geometry for Spatial Data

Geography Over Geometry for Spatial Data

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-01-26
  • Supersedes: PostGIS Spatial Extension

Context and Problem Statement

PostGIS provides two spatial type systems: GEOMETRY (planar/Cartesian math) and GEOGRAPHY (spherical/geodetic math). The original ADR chose GEOMETRY for performance. After further analysis, we need to reconsider for scientific and commercial accuracy requirements.

Decision Drivers

  • Correctness and precision over speed
  • Global applicability (not limited to small geographic areas)
  • Scientific rigor for geology, environmental, and mapping applications
  • Consistent behavior regardless of location on Earth

Considered Options

  1. GEOMETRY - Planar Cartesian calculations
  2. GEOGRAPHY - Spherical geodetic calculations

Decision Outcome

Chosen option: GEOGRAPHY, because correctness and precision are more important than speed for scientific and commercial work.

Consequences

Good:

  • Accurate distance calculations anywhere on Earth (meters, not degrees)
  • Correct area calculations for large regions
  • No projection distortion artifacts
  • Always uses WGS84 (SRID 4326) - universal standard

Bad:

  • Slower than GEOMETRY (typically 2-5x for complex operations)
  • Some spatial functions not available (can cast to GEOMETRY when needed)
  • Always meters for distance (no configurable units - which we consider good, see ADR: Metric Units Only)

Technical Details

Type Mapping:

DSL Type PostgreSQL Type
POINT GEOGRAPHY(POINT, 4326)
POINTZ GEOGRAPHY(POINTZ, 4326)
LINESTRING GEOGRAPHY(LINESTRING, 4326)
LINESTRINGZ GEOGRAPHY(LINESTRINGZ, 4326)
POLYGON GEOGRAPHY(POLYGON, 4326)
POLYGONZ GEOGRAPHY(POLYGONZ, 4326)
MULTIPOINT GEOGRAPHY(MULTIPOINT, 4326)
MULTILINESTRING GEOGRAPHY(MULTILINESTRING, 4326)
MULTIPOLYGON GEOGRAPHY(MULTIPOLYGON, 4326)
GEOMETRYCOLLECTION GEOGRAPHY(GEOMETRYCOLLECTION, 4326)

SRID Strategy:

  • GEOGRAPHY always uses WGS84 (SRID 4326) internally
  • DSL allows specifying SRID for documentation: location POINT(4326)
  • If non-4326 SRID specified, data is transformed on insert/select
  • Default SRID is 4326; omitting generates a warning

C++ Template SRID:

// Compile-time SRID safety
muscovite::dba::Point<4326> location;  // WGS84
muscovite::dba::Point<3857> webMercator;  // Web Mercator

// Compilation error: cannot assign different SRIDs
location = webMercator;  // ERROR

// Explicit transform required
location = webMercator.transform<4326>();  // OK

Index Generation:

  • GIST spatial indexes auto-generated for @spatial-queryable fields
  • Uses USING GIST (column geography_ops) for GEOGRAPHY

AI Agent Guidance

When this applies:

  • Any entity with spatial/geographic data
  • Location fields, boundaries, routes, areas
  • Whenever POINT, LINESTRING, POLYGON, etc. types are used

When this does NOT apply:

  • Pure coordinate storage without spatial queries (use two DECIMAL fields)
  • Image/raster coordinates (pixel space, not geographic)

Key implementation files:

  • src/dslgen/generators/postgres/sqitch-unified.rkt - GEOGRAPHY type emission
  • src/dslgen/generators/cpp/common.rkt - C++ type mapping
