Three-Tier Validation Architecture

Three-Tier Validation Architecture

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-07
  • Tags: validation, architecture, strong-types, sidecar, domain-services

Technical Story: Validation currently happens ad-hoc. Field constraints in the DSL are not enforced at the C++ boundary. There is no structured way to compose field-level, entity-level, and business-rule validation into a coherent pipeline.

Context and Problem Statement

Muscovite generates domain entities with typed fields, but validation is inconsistent across the generated codebase:

  • Per-field validation exists only for PositiveInteger (ADR 20260119) — no other field type has construction-time validation
  • Cross-field invariants (e.g., end_date > start_date) are not enforced anywhere in generated code
  • Business rules (e.g., "a patient cannot be discharged before all lab results are reviewed") have no extension point

This creates three problems: (1) invalid data can be persisted because validation is incomplete, (2) validation logic is duplicated between frontend and backend, and (3) users have no structured way to add custom business rules without editing generated code (which is forbidden).

How should validation be structured so that field constraints, entity invariants, and business rules compose into a single pipeline with clear ownership boundaries?

Decision Drivers

  • Validation must be composable — all three tiers run in sequence, merging errors
  • Error accumulation (not fail-fast) — clients should see all validation failures in one response
  • Tier 1 and Tier 2 are fully generated — no manual code required
  • Tier 3 must be user-extensible without editing generated code (sidecar pattern, ADR 20260223)
  • Frontend (Qt Widgets + QML) must mirror the same three tiers for UI symmetry
  • The architecture must be expressible as generated C++23 code

Considered Options

  1. Three-tier pipeline — Strong Types (Tier 1) → EntityValidator (Tier 2) → PluginValidator (Tier 3)
  2. Single validator class — one monolithic validator per entity handling all concerns
  3. Aspect-oriented validation — validation aspects woven into entity setters
  4. Schema-based validation — JSON Schema-like validation descriptors interpreted at runtime

Decision Outcome

Chosen option: Option 1 — Three-tier pipeline, because it provides clear ownership boundaries (generated vs. user-owned), supports error accumulation, and maps naturally to both backend and frontend generation targets.

Tier 1: Strong Types (Per-Field Construction Validation)

  • Mechanism: Entity::try_from(Command) factory method
  • What it validates: Per-field constraints from @min, @max, @not_empty, @not_in_future, @not_in_past, @pattern annotations
  • Implementation: Compile-time template parameters where possible (BoundedInteger, BoundedString), runtime checks where necessary (temporal constraints)
  • Lifecycle: Fully generated, regenerated every run
  • Error type: std::expected<Entity, ValidationErrors>
  • C++ pattern: Strong type template families (ADR 20260307-strong-type-template-families)

Tier 2: EntityValidator (Cross-Field Invariants)

  • Mechanism: EntityValidator::validate(const Entity&) -> ValidationErrors
  • What it validates: Cross-field constraints from @after, @before, @same_day, @not_equal annotations
  • Implementation: Generated method that checks relational invariants between entity fields
  • Lifecycle: Fully generated, regenerated every run
  • Nullable semantics: Vacuously true if either field is NULL (ADR 20260307-cross-field-constraint-syntax)

Tier 3: PluginValidator (User-Owned Business Rules)

  • Mechanism: PluginValidator::validate(const Entity&, const Context&) -> ValidationErrors
  • What it validates: Business rules that require external state, domain knowledge, or user-specific logic
  • Implementation: Virtual method in sidecar file (ADR 20260223). Default implementation is no-op (returns empty errors)
  • Lifecycle: Generated once as sidecar stub, then user-owned. Protected by sentinel comment
  • Extension point: Users override validate() in the sidecar implementation file

Composition in DomainService

auto DomainService::create(const CreateCommand& cmd) -> std::expected<Entity, ValidationErrors> {
    // Tier 1: Field-level construction validation
    auto entity_result = Entity::try_from(cmd);
    if (!entity_result) return std::unexpected(entity_result.error());

    // Tier 2: Cross-field invariant validation
    auto tier2_errors = entity_validator_.validate(*entity_result);
    if (!tier2_errors.empty()) return std::unexpected(tier2_errors);

    // Tier 3: Plugin business rules
    auto tier3_errors = plugin_validator_.validate(*entity_result, context_);
    if (!tier3_errors.empty()) return std::unexpected(tier3_errors);

    // All tiers passed — persist
    return command_service_.create(*entity_result);
}

ValidationErrors

struct ValidationError {
    std::string field;     // field name or "" for entity-level errors
    std::string message;   // human-readable error message
    std::string code;      // machine-readable error code (e.g., "min_violation")
};

class ValidationErrors {
    std::vector<ValidationError> errors_;
public:
    void add(std::string field, std::string message, std::string code = "");
    void merge(const ValidationErrors& other);
    [[nodiscard]] bool empty() const;
    [[nodiscard]] auto begin() const;
    [[nodiscard]] auto end() const;
};

Positive Consequences

  • Clear ownership: Tier 1+2 are generated (hands-off), Tier 3 is user-owned (sidecar)
  • Error accumulation: clients get all validation failures in one response
  • Testable: each tier can be tested independently with doctest
  • Frontend symmetry: Qt Widgets and QML mirror the same three tiers (ADR 20260307-frontend-validation-symmetry)
  • Composable: DomainService orchestrates all three tiers in a predictable sequence

Negative Consequences

  • Three validation classes per entity increases generated code volume
  • Error accumulation across tiers requires careful merge semantics
  • Tier 3 no-op default means users must discover the extension point (documentation burden)
  • Temporal constraints (Tier 1) cannot be compile-time — runtime validation adds overhead

AI Agent Guidance

When This Decision Applies

  • When generating any validation-related C++ code — follow the three-tier structure
  • When generating DomainService — compose all three tiers in order
  • When generating entity factory methods (try_from) — this is Tier 1
  • When generating EntityValidator — this is Tier 2
  • When generating sidecar stubs for PluginValidator — this is Tier 3
  • When generating frontend validation — mirror the same three tiers

When This Decision Does NOT Apply

  • Query operations — read paths do not require validation
  • Lookup table values — these are compile-time constants, not validated at runtime
  • Authentication/authorization — these are infrastructure concerns, not domain validation

Enforcement

  • DomainService generator must compose all three tiers
  • EntityValidator generator must produce rules from relative? = #t constraints
  • PluginValidator generator must produce sidecar stub with sentinel comment
  • Integration tests verify all three tiers are invoked in sequence
  • ValidationErrors is a framework class in framework/ (Layer 1)
