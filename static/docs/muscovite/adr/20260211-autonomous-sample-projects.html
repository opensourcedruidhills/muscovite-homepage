Sample Projects Must Be Fully Autonomous

Sample Projects Must Be Fully Autonomous

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: examples, infrastructure, docker, build

Technical Story: Sample projects (hello-world, fhir-health, geosciences) must be self-contained repositories that treat Muscovite as an external dependency and manage their own infrastructure.

Context and Problem Statement

Muscovite sample projects currently live in separate git repositories added as submodules under examples/. However, there is no formal policy on how autonomous these projects must be. Questions arise:

  • Can they rely on the root Muscovite docker-compose.yml for PostgreSQL?
  • Can they assume the Muscovite framework is already built locally?
  • Can they depend on Muscovite CI to validate them?

Muscovite has no CI infrastructure and will not have one due to budget constraints. This means sample projects cannot rely on a parent CI pipeline to catch breakage. Each sample must be independently cloneable, buildable, and runnable by a user who has never seen the Muscovite monorepo.

Decision Drivers

  • No Muscovite CI: No budget for CI — samples cannot rely on upstream validation
  • Onboarding: A new user should git clone a sample and have it working without cloning Muscovite itself
  • Reproducibility: Every sample must start from zero and produce a deterministic result
  • Template quality: Samples serve as copy-paste templates for new projects — they must model best practices for standalone Muscovite consumers

Considered Options

  • Option 1: Fully autonomous — own Docker, own build, Muscovite as FetchContent/installed dependency
  • Option 2: Semi-autonomous — own build, but share Muscovite's Docker infrastructure
  • Option 3: Integrated — build as part of Muscovite's CMake tree, share everything

Decision Outcome

Chosen option: Option 1 — Fully autonomous, because without CI there is no safety net for integration assumptions, and because samples must demonstrate the real-world consumer experience.

Requirements for Every Sample Project

Requirement Detail
Own git repository Hosted at opensourcedruidhills/{name}-sample, added as submodule under examples/
Own docker-compose.yml Starts its own PostgreSQL container. Port configurable via DB_PORT env var with per-sample defaults (fhir=5556, geosciences=5557, hello-world=5558)
Own conanfile.py or conanfile.txt Declares all C++ dependencies independently
Own Conan profiles conan_profiles/debug and release with compiler.cppstd=gnu23 + compiler.cstd=17 (libpq is C, requires C17)
Own Python venv python3 -m venv .venv + requirements.txt pinning Conan version. No system-level Python dependencies
Own CMakeLists.txt Standalone build that uses find_package(muscovite) — never add_subdirectory() of Muscovite sources
Own CMakePresets.json Standard presets (debug, release) with no references to parent paths
Muscovite version pinned MUSCOVITE_VERSION file at repo root contains exact tag (e.g., v1.18.0). Used by FetchContent_Declare(GIT_TAG ...). Updated during Muscovite release ceremony
Own setup.sh Full bootstrap: venv → Conan → Docker → Racket generate → CMake build → Sqitch deploy. One command from clone to running
Own Sqitch project Self-contained migrations with own sqitch.conf and sqitch.plan
Own README.md Complete setup instructions: clone → ./setup.sh → ctest — no "see Muscovite docs" for critical steps
Own seed data All lookup table seeds included in the repository

Release Ceremony Integration

When Muscovite releases a new version (vX.Y.Z), the release script must update MUSCOVITE_VERSION in all sample submodules, commit, tag, and push. This is a mandatory release checklist item.

Positive Consequences

  • Any user can git clone + docker compose up + cmake --build without Muscovite source
  • Samples prove that the framework packaging and install targets actually work
  • No hidden coupling to monorepo structure
  • Samples serve as realistic integration tests for the Muscovite install/export chain
  • Each sample can evolve and tag releases independently

Negative Consequences

  • Duplication of Docker config across samples (mitigated: it's ~20 lines of YAML)
  • Must keep FetchContent / Conan references in sync when Muscovite releases new versions
  • Slightly more setup for contributors who work on multiple samples

Pros and Cons of the Options

Option 1: Fully Autonomous

  • Good, because zero assumptions about host environment
  • Good, because proves real-world consumer experience works
  • Good, because survives without CI
  • Bad, because Docker config is duplicated across samples

Option 2: Semi-Autonomous (shared Docker)

  • Good, because less duplication
  • Bad, because user must clone Muscovite monorepo to get Docker config
  • Bad, because port conflicts when running multiple samples

Option 3: Integrated

  • Good, because simplest for monorepo developers
  • Bad, because useless as a template — users can't replicate the build
  • Bad, because hides packaging bugs (everything "just works" inside add_subdirectory)

AI Agent Guidance

When This Decision Applies

  • When creating any new sample/example project under examples/
  • When modifying an existing sample project's build or infrastructure
  • When advising users on how to start a new Muscovite-based project

When This Decision Does NOT Apply

  • Muscovite framework itself (framework/, dslgen/) — these are the monorepo
  • Test fixtures in tests/fixtures/ — these are internal test data, not standalone projects
  • .ddd files used only for parser/generator testing

Enforcement

  • Code review on sample project PRs
  • README.md in each sample must include full standalone setup instructions
  • docker-compose.yml must be present in each sample repository root

Key Files

  • examples/*/docker-compose.yml — per-sample Docker config
  • examples/*/CMakeLists.txt — standalone build with find_package(muscovite)
  • examples/*/CMakePresets.json — self-contained presets
  • examples/*/README.md — complete setup walkthrough
  • .gitmodules — submodule registration
