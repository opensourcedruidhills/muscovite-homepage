PlantUML Salt Wireframes in Generated Documentation

PlantUML Salt Wireframes in Generated Documentation

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-22
  • Tags: documentation, wireframe, plantuml, salt, ui, layout, asciidoc, code-generation

Technical Story: Generated AsciiDoc system manuals document the data model (ER diagrams, field tables) but not the generated UI structure. Stakeholders reviewing a domain model cannot see what the application will look like until C++ compiles and runs. Wireframes derived from the same IR that drives Qt Widgets and QML generation close this gap.

Context and Problem Statement

Muscovite generates full-stack applications from .ddd models: PostgreSQL schemas, C++ DBA, gRPC services, Qt Widgets, and Qt Quick/QML frontends. The AsciiDoc system manual generator (glossary-adoc.rkt) already produces ER diagrams, event flow diagrams, and saga workflow diagrams — all via PlantUML.

However, the generated UI structure is invisible in documentation. The IR contains rich layout metadata:

  • layout-spec with form-factor qualifiers (desktop, tablet, mobile)
  • master-spec with table column selection
  • detail-spec with style (form/sidebar/card/wizard), grouping (tabs/sections/accordion/flat), actions, and child tabs
  • Complete type → widget mapping (Boolean → QCheckBox, FK → QComboBox, Date → QDateEdit, etc.)
  • FK detection with display field resolution for dropdowns

This metadata drives Qt Widgets and QML generation today but is not surfaced in documentation. A domain expert reviewing the system manual sees entity fields but not the master-detail layout, tab grouping, or form arrangement that will be generated.

Three use cases identified:

  1. Default wireframes: Auto-generated from IR structural properties — show what the generated UI looks like without any DSL customization
  2. Customizable wireframes: Reflect LAYOUT {} block customizations from .ddd files — the DSL is the wireframe editor, regeneration shows changes
  3. Structural documentation: Wireframes embedded in AsciiDoc system manuals alongside ER diagrams and event flows

Decision Drivers

  • PlantUML is the standardized diagram format (ADR 20260217, Decision 5)
  • AsciiDoc + asciidoctor-diagram pipeline already renders PlantUML in system manuals
  • Layout IR is mature: layout-spec, master-spec, detail-spec, child-tab-spec fully parsed and resolved
  • Type → widget mapping exists in widgets-field-helpers.rkt (15+ type mappings)
  • FK → ComboBox detection exists in widgets-fk-helpers.rkt
  • Target config abstraction maps abstract vocabulary to concrete widgets (target-config.rkt)
  • Zero new external dependencies preferred

Considered Options

Option 1: PlantUML Salt Wireframes (Recommended)

PlantUML's salt dialect is a built-in wireframe/mockup syntax supporting forms, tabs, trees, tables, dropdowns, checkboxes, spinners, and buttons.

@startsalt
{+
  {* Entity: TodoItem                       }
  {#
    . | Title        | Due Date   | Done
    1 | Buy groceries | 2026-03-01 | [X]
    2 | Write report  | 2026-03-15 | [ ]
  }
  ---
  {/ <b>General | Comments }
  {
    Title:       | "Buy groceries              "
    Description: | {SI
                    Multi-line description
                   }
    Priority:    | ^High^
    Status:      | ^In Progress^
    Due Date:    | "2026-03-01"
    Done:        | [X]
    ---
    [Create] | [Edit] | [Delete]
  }
}
@endsalt

Option 2: ASCII Art Wireframes

Plain-text box-drawing wireframes in AsciiDoc source blocks.

┌─────────────────────────────────────────────┐
│ Master: TodoItem                            │
├─────────┬──────────┬─────────┬──────────────┤
│ Title   │ Due Date │ Done    │              │
├─────────┼──────────┼─────────┤  Title: ___  │
│ Buy gro │ 2026-03  │ [X]     │  Desc:  ___ │
└─────────┴──────────┴─────────┤  Prio:  [v] │
                                └─────────────┘

Option 3: HTML/CSS Mockups

Generate standalone .html files with pure CSS wireframe styling.

Option 4: SVG via Racket racket/draw

Generate .svg wireframes directly using Racket's drawing library (as diagram.rkt does for architecture diagrams).

Option 5: Figma/External Tool Export

Generate Figma-compatible JSON or Sketch artboards from the IR.

Decision Outcome

Chosen option: Option 1 — PlantUML Salt Wireframes, because it requires zero new dependencies, integrates natively with the existing AsciiDoc + PlantUML pipeline, and the Salt widget vocabulary maps directly to the IR's layout metadata.

Positive Consequences

  • Wireframes appear in system manuals alongside ER diagrams, event flows, and saga workflows — unified documentation
  • Same PlantUML rendering pipeline, same [plantuml] block syntax, same build toolchain
  • Layout IR metadata → Salt syntax is a straightforward, deterministic transformation
  • LAYOUT {} block customizations in .ddd immediately reflect in wireframes on regeneration
  • Form-factor-qualified layouts produce separate wireframes per form factor (desktop, mobile, tablet)
  • Stakeholders can review UI structure without building C++

Negative Consequences

  • Salt syntax is schematic, not pixel-perfect — adequate for structural wireframes, not visual design
  • Salt has no interactivity (no hover, no click simulation) — wireframes are static
  • Requires understanding Salt's somewhat unusual syntax for the generator author
  • Additional PlantUML rendering time per aggregate (one wireframe diagram each)

Detailed Decisions

1. Generator Module: dslgen/generators/docs/wireframe-salt.rkt

New Racket module consuming the same IR as widgets-views.rkt and quick-views.rkt, but emitting PlantUML Salt syntax instead of C++/QML code.

IR Input Salt Output
master-spec.columns {# . | Col1 | Col2 } — table with header row and sample data rows
detail-spec.style = 'form { Label: | "field" } — labeled form fields
detail-spec.style = 'sidebar {+ {/ Master | Detail } } — split layout
detail-spec.style = 'card { ... } — framed card layout
detail-spec.style = 'wizard {/ Step 1 | Step 2 | Step 3 } — wizard pages
detail-spec.grouping = 'tabs {/ Tab1 | Tab2 } — tabbed content
detail-spec.grouping = 'sections Labeled sections with separators
detail-spec.grouping = 'accordion Collapsible sections
detail-spec.grouping = 'flat Sequential fields
detail-spec.children Additional tabs with child entity tables
detail-spec.actions [Create] | [Edit] | [Delete] — button row

2. Field Type → Salt Widget Mapping

Reuse the existing type → widget mapping from widgets-field-helpers.rkt:

Widget Type Salt Syntax Example
QLineEdit (String, UUID, Email, URL) "placeholder text " "john@example.com "
QTextEdit (description, notes) {SI multi-line text } Multi-line input area
QSpinBox (Integer) "42 " Numeric input
QDoubleSpinBox (Decimal) "3.14 " Decimal input
QCheckBox (Boolean) [X] or [ ] Checkbox
QDateEdit (Date) "2026-03-01" Date field
QDateTimeEdit (DateTime) "2026-03-01 12:00" DateTime field
QTimeEdit (Time) "14:30" Time field
QComboBox (FK relationship) ^Selected Value^ Dropdown

3. Integration into AsciiDoc System Manual

Wireframes are embedded in glossary-adoc-entities.rkt as a new subsection per aggregate, immediately after the ER diagram:

==== UI Wireframe: TodoItem

[plantuml, todoitem-wireframe, svg]
----
@startsalt
...
@endsalt
----

4. Form-Factor-Qualified Wireframes

If the aggregate has multiple LAYOUT blocks with form-factor qualifiers, generate one wireframe per form factor:

===== Desktop Layout

[plantuml, todoitem-wireframe-desktop, svg]
----
@startsalt
...sidebar layout with full table columns...
@endsalt
----

===== Mobile Layout

[plantuml, todoitem-wireframe-mobile, svg]
----
@startsalt
...compact form with minimal columns...
@endsalt
----

5. Default Wireframe Generation (No LAYOUT Block)

When an aggregate has no explicit LAYOUT {} block, generate a default wireframe based on:

  • Master: All non-id, non-audit fields as table columns
  • Detail: STYLE: form, GROUPING: flat, ACTIONS: [create, edit, delete]
  • FK fields: Rendered as dropdowns with target entity name
  • Boolean fields: Rendered as checkboxes
  • Audit fields (created_at, updated_at): Omitted from master, shown read-only in detail

This mirrors the default layout resolution in target-config.rkt (default-layout-for-target).

6. Scope: DSL-to-Docs Only (Phase 1)

Phase 1 generates wireframes from .ddd → AsciiDoc only. The mapping is:

.ddd model → IR (layout-spec + fields + relationships)
           → wireframe-salt.rkt
           → PlantUML Salt blocks
           → AsciiDoc system manual
           → HTML/PDF via asciidoctor-diagram

Future phases may add:

  • Interactive wireframe editing that propagates back to LAYOUT {} blocks
  • Wireframes as a standalone CLI output (--wireframes)
  • Wireframe diff (before/after layout changes)

Pros and Cons of the Options

Option 1: PlantUML Salt (chosen)

  • Good, because zero new dependencies — PlantUML already required
  • Good, because native AsciiDoc embedding via [plantuml] blocks
  • Good, because Salt widget vocabulary maps 1:1 to the IR's layout metadata
  • Good, because deterministic generation — same IR always produces same wireframe
  • Bad, because schematic aesthetics, not production-quality mockups
  • Bad, because no interactivity

Option 2: ASCII Art

  • Good, because zero dependencies, works in plain text
  • Bad, because column alignment and box-drawing is fragile to generate
  • Bad, because not renderable as images for PDF/HTML
  • Bad, because limited widget vocabulary (no tabs, no dropdowns)

Option 3: HTML/CSS Mockups

  • Good, because closest to actual UI, potentially interactive
  • Bad, because new toolchain separate from AsciiDoc pipeline
  • Bad, because not embeddable in AsciiDoc documents (only linkable)
  • Bad, because HTML/CSS generation is complex and fragile

Option 4: SVG via racket/draw

  • Good, because full rendering control
  • Bad, because must build a complete form renderer from scratch (~1K+ lines)
  • Bad, because no standard wireframe vocabulary — custom rendering engine
  • Bad, because diagram.rkt renders architecture diagrams, not forms

Option 5: Figma/External

  • Good, because professional-quality mockups
  • Bad, because proprietary dependency, violates single-source-of-truth
  • Bad, because no round-trip back to DSL
  • Bad, because requires API key / cloud service

AI Agent Guidance

When This Decision Applies

  • When modifying glossary-adoc-entities.rkt — add wireframe section per aggregate
  • When adding new DSL types — update the Salt widget mapping in wireframe-salt.rkt
  • When changing LAYOUT {} grammar or IR — update wireframe generation accordingly
  • When adding new detail-spec.style or grouping values — add Salt rendering

When This Decision Does NOT Apply

  • Interactive UI prototyping tools — out of scope
  • Pixel-perfect visual design mockups — Salt is structural, not visual
  • Generated Qt/QML code — wireframes are documentation, not runtime artifacts
