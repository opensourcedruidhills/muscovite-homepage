Anti-Meridian Handling Limitation

Anti-Meridian Handling Limitation

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-01-26

Context and Problem Statement

The anti-meridian (international date line at 180°/-180° longitude) poses challenges for geographic calculations. Polygons and linestrings that cross this boundary require special handling because a simple bounding box query might miss features or produce incorrect results.

Decision Drivers

  • Correctness of spatial queries
  • Complexity of anti-meridian handling
  • Geographic distribution of typical use cases
  • PostGIS GEOGRAPHY behavior

Decision Outcome

Anti-meridian crossing geometries are documented as a known limitation. The framework does not provide special handling for this edge case in the initial release.

Technical Background

Consider a polygon covering the Bering Strait:

  • Western point: 170°E (positive longitude)
  • Eastern point: 170°W (-170°, or equivalently 190°)

A naive bounding box would span from 170° to -170°, which most GIS systems interpret as covering most of the globe (going the "long way around") rather than the small region intended.

PostGIS GEOGRAPHY Behavior

PostGIS GEOGRAPHY type handles anti-meridian correctly for:

  • Distance calculations (ST_Distance)
  • Area calculations (ST_Area)
  • Point-in-polygon (ST_Contains, ST_Within)

But spatial indexes and bounding box queries may behave unexpectedly for:

  • Large polygons crossing 180°
  • Bounding box operations (&&, ST_Intersects with boxes)

Guidance for Users

  1. Split geometries at anti-meridian: Create two polygons instead of one crossing
  2. Use smaller query extents: Avoid extent queries that cross 180°
  3. Test edge cases: Verify behavior if your domain involves Pacific regions

Affected Regions

This limitation primarily affects:

  • Pacific Ocean (Russia/Alaska, Japan/Hawaii routes)
  • Fiji, Tonga, Samoa (island nations near date line)
  • Antarctic regions crossing 180°

Most use cases (Europe, Americas, Africa, most of Asia) are unaffected.

Future Consideration

If anti-meridian handling becomes critical:

  1. Implement geometry splitting at 180° during insert
  2. Add extent normalization in query layer
  3. Provide utility functions for Pacific-centric queries

AI Agent Guidance

When this applies:

  • User's domain involves Pacific Ocean regions
  • Geometries span more than 180° of longitude
  • Spatial queries return unexpected results near 180°/-180°

What to do:

  • Warn users about this limitation when they define large extents
  • Suggest splitting geometries at anti-meridian
  • Point to this ADR for detailed explanation

Not an error, but a warning:

Warning: Geometry appears to cross the anti-meridian (180°/-180° longitude).
Spatial queries may behave unexpectedly for features crossing this boundary.
Consider splitting into separate geometries at the anti-meridian.
See ADR: 20260126-anti-meridian-limitation.md

Detection heuristic:

bool crosses_antimeridian(const Polygon<4326>& poly) {
    double min_lon = std::numeric_limits<double>::max();
    double max_lon = std::numeric_limits<double>::lowest();
    for (const auto& point : poly.exterior_ring()) {
        min_lon = std::min(min_lon, point.longitude());
        max_lon = std::max(max_lon, point.longitude());
    }
    // If span > 180°, likely crosses anti-meridian
    return (max_lon - min_lon) > 180.0;
}
