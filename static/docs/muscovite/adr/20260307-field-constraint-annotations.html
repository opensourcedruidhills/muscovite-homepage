Field Constraint Annotations in DSL Grammar

Field Constraint Annotations in DSL Grammar

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-07
  • Tags: dsl, grammar, constraints, validation, peg

Technical Story: The DSL type system declares field types (Integer, String, Date) but has no syntax for declaring constraints like minimum/maximum values, non-emptiness, or temporal restrictions. The PEG grammar needs extension per ADR 20260211 (grammar-first mandate).

Context and Problem Statement

Muscovite's DSL currently allows declaring entity fields with types (Integer, String, Date, DateTime, Decimal, Boolean, UUID) but provides no mechanism for expressing value constraints. A field declared as Integer accepts any 64-bit integer; a String field accepts any text including empty strings; a Date field accepts any valid date including far-future or far-past values.

Domain models routinely need constraints: ages must be positive, names must be non-empty, birth dates must not be in the future, quantities must fall within ranges. Without DSL-level constraint syntax, these rules exist only in documentation or ad-hoc C++ code, defeating the DSL's purpose of being the single source of truth.

How should the DSL grammar be extended to support per-field constraint annotations that generate validation code across all 8 targets?

Decision Drivers

  • Grammar-first mandate (ADR 20260211) — PEG grammar must change before parser
  • Constraints must be expressible inline with field declarations (not in separate blocks)
  • The annotation syntax must be extensible for future constraint types
  • Half-open ranges (min-only or max-only) are a common pattern and must be supported
  • Constraints must map cleanly to all 8 generator targets (SQL, C++, Proto, Qt Widgets, QML, CMake, Sqitch, Doc)

Considered Options

  1. Annotation syntax after field type — @min(N), @max(N), @not_empty, etc.
  2. Constraint block per entity — CONSTRAINTS { field_name: min(1), max(100); }
  3. Type aliases — PositiveInteger, NonEmptyString as built-in DSL types
  4. External constraint files — .constraints files alongside .ddd files

Decision Outcome

Chosen option: Option 1 — Annotation syntax after field type, because it keeps constraints co-located with the field they apply to, is extensible, and follows the existing @uuid annotation pattern.

Grammar Extension

field_constraint  ← '@min(' number ')' / '@max(' number ')' / '@not_empty'
                   / '@not_in_future' / '@not_in_past'
                   / '@pattern(' string_literal ')'

field_declaration ← identifier type_reference field_constraint* ('@uuid' uuid_value)? doc_string? ';'

Supported Annotations

Annotation Applies To Meaning
@min(N) Integer, Decimal Minimum value (inclusive)
@max(N) Integer, Decimal Maximum value (inclusive)
@not_empty String Minimum length 1
@not_in_future Date, DateTime Value ≤ now
@not_in_past Date, DateTime Value ≥ now
@pattern("regex") String Must match POSIX ERE pattern

Half-open ranges are supported: @min(0) without @max means "≥ 0, no upper bound". @max(100) without @min means "no lower bound, ≤ 100".

IR Representation

(struct field-constraint (kind value relative?) #:transparent)
;; kind: symbol — 'min, 'max, 'not-empty, 'not-in-future, 'not-in-past, 'pattern
;; value: number, string, or #f (for boolean constraints like not-empty)
;; relative?: #f for per-field constraints (cross-field uses #t, see ADR 20260307-cross-field)

Target Mappings

Target Generation
C++ Strong type template instantiation (Tier 1 validation)
SQL CHECK constraint on column
Proto Field comments (protobuf has no native constraints)
Qt Widgets Widget configuration (e.g., QSpinBox::setMinimum)
QML Validator components (IntValidator, RegularExpressionValidator)
Documentation Constraint table in entity documentation

Positive Consequences

  • Single source of truth — constraints declared once in DSL, generated everywhere
  • DSL amplification increases — one annotation generates validation across all 8 targets
  • Co-located with field — easy to read, easy to maintain
  • Extensible — new annotation types can be added following the same pattern
  • Grammar-first compliance — PEG grammar change drives all downstream changes

Negative Consequences

  • Grammar complexity increases — parser must handle optional annotation chains
  • Not all constraints map to all targets (e.g., Proto has no native validation)
  • Regex patterns in DSL may need escaping rules for special characters
  • Existing .ddd files are unaffected (annotations are optional) but cannot benefit until updated

AI Agent Guidance

When This Decision Applies

  • When extending the PEG grammar in dslgen/grammar/muscovite.peg — add constraint annotation rules
  • When modifying the parser in dslgen/parser/parser.rkt — parse annotations after field type
  • When adding field-constraint structs to the IR in dslgen/model/ddd-types.rkt
  • When generating validation code in any of the 8 generator targets

When This Decision Does NOT Apply

  • Cross-field constraints (e.g., end_date > start_date) — see ADR 20260307-cross-field-constraint-syntax
  • Business rules that depend on external state — those are Tier 3 (PluginValidator)
  • Lookup table constraints — those use the existing LOOKUP_TABLE mechanism

Enforcement

  • PEG grammar conformance test: dslgen/tests/test-grammar-conformance.rkt
  • Parser tests validate annotation parsing: dslgen/tests/test-parser.rkt
  • Generator tests verify constraint output per target
