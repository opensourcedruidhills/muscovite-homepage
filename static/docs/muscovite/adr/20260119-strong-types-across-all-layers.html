ADR: Strong Types Across All Layers

ADR: Strong Types Across All Layers

  • Status: Accepted
  • Date: 2026-01-19
  • Deciders: Architecture Team
  • Technical Story: Implement end-to-end strong typing from PostgreSQL through gRPC to C++ and Qt

Context and Problem Statement

Muscovite's STRONG_TYPE DSL construct currently generates PostgreSQL DOMAINs with CHECK constraints for database-level validation. However, these strong types are flattened to primitives when crossing layer boundaries:

Layer Current Mapping Problem
PostgreSQL CREATE DOMAIN email AS TEXT ✅ Validated
PL/pgSQL API p_email email ✅ Domain-typed
gRPC Proto string email = 1; ❌ Just a string
C++ Service std::string email ❌ Just a string
Qt Client QString email ❌ Just a string

This creates several problems:

  1. Type safety lost - Any string can be passed where an email is expected
  2. No compile-time checks - Mismatched types only caught at runtime (DB constraint)
  3. Self-documenting code - std::string says nothing about the domain meaning
  4. IDE support - No type-specific autocomplete or validation
  5. Interoperability - Existing boost/Qt types (boost::uuids::uuid, QUuid) not leveraged

Decision Drivers

  • Type safety: Compile-time errors when mixing incompatible types
  • Reuse existing types: boost::uuids::uuid, QUuid, std::chrono types
  • Validation locality: Validate in C++/Qt, not just at database
  • Serialization: Efficient proto serialization without string conversion overhead
  • Performance: Avoid unnecessary string ↔︎ native type conversions
  • Ergonomics: Idiomatic C++ and Qt code

Considered Options

  1. Keep primitives - ❌ No type safety
  2. C++20 strong typedefs only - Partial solution, no Qt/proto integration
  3. Custom wrapper classes per strong type - Maximum safety, but verbose
  4. Hybrid: Reuse existing + generate wrappers - ✅ Selected

Decision Outcome

Implement strong types across all layers using a hybrid approach:

2. STRONG_TYPE → Generated Wrapper Classes

For each user-defined STRONG_TYPE:

STRONG_TYPE Email BASED_ON TEXT
    CHECK email_format: VALUE ~ '^[^@]+@[^@]+\.[^@]+$'
    DOC "Valid email address";

Generated C++ (muscovite::types::Email):

namespace muscovite::types {

class Email {
public:
    // Explicit construction from string - validates format
    explicit Email(std::string_view value);
    
    // Factory method for unchecked construction (from trusted source like DB)
    static Email from_trusted(std::string value) noexcept;
    
    // Accessors
    [[nodiscard]] const std::string& value() const noexcept { return value_; }
    [[nodiscard]] std::string_view view() const noexcept { return value_; }
    
    // Implicit conversion to string_view for ergonomics
    operator std::string_view() const noexcept { return value_; }
    
    // Comparison (defaulted)
    auto operator<=>(const Email&) const = default;
    
    // Validation (static, usable before construction)
    [[nodiscard]] static bool is_valid(std::string_view value) noexcept;
    
    // Proto serialization
    void to_proto(std::string* out) const { *out = value_; }
    static Email from_proto(const std::string& proto) { return from_trusted(proto); }

private:
    std::string value_;
};

} // namespace muscovite::types

Generated Qt Wrapper (muscovite::qt::Email):

namespace muscovite::qt {

class Email {
public:
    explicit Email(const QString& value);
    static Email fromTrusted(QString value) noexcept;
    
    [[nodiscard]] const QString& value() const noexcept { return value_; }
    [[nodiscard]] static bool isValid(const QString& value) noexcept;
    
    // Conversion to/from C++ type
    explicit Email(const muscovite::types::Email& cpp);
    [[nodiscard]] muscovite::types::Email toCpp() const;
    
    // QVariant support for Qt models
    operator QVariant() const { return QVariant::fromValue(value_); }
    
private:
    QString value_;
};

} // namespace muscovite::qt

5. Validation Strategy

Level When Validated Action on Invalid
Qt Input User input in QLineEdit Show validation error, don't submit
Qt Model Before gRPC call Throw validation exception
gRPC Service Request received Return INVALID_ARGUMENT status
PostgreSQL INSERT/UPDATE Raise constraint violation

Validation regex is single source of truth in the DDD model:

STRONG_TYPE Email BASED_ON TEXT
    CHECK email_format: VALUE ~ '^[^@]+@[^@]+\.[^@]+$'

This regex is:

  1. Used in PostgreSQL DOMAIN CHECK
  2. Compiled into C++ static validator (std::regex or RE2)
  3. Compiled into Qt QRegularExpressionValidator
  4. Documented in proto message comment

Phase 3: C++ DBA Layer (Primitives for pqxx Compatibility)

Phase 6: Qt Widget Generation (In Progress)

Layer Boundary Architecture (Implementation Decision)

The strong types library (muscovite::types) provides application-layer validation that sits between the transport layer (gRPC) and the database layer (DBA). Here's how types flow:

flowchart TD
    A["Qt Client Layer\nQLineEdit → EmailValidator::validate → gRPC call with string"]
    B["gRPC Transport Layer\nProto uses simple string type for cross-language compatibility\nNo proto wrapper messages — validation happens elsewhere"]
    C["C++ gRPC Service Layer\nOption A: Validate here using muscovite::types::Email::is_valid\nOption B: Pass through to DBA, let PostgreSQL DOMAIN reject\nCurrent: Option B · Future: Add Option A for early rejection"]
    D["DBA Layer (Repository)\nEntity structs use std::string for strong type fields\npqxx::params only accepts string-like types"]
    E["PostgreSQL Layer\nColumn type: basedata.email (DOMAIN with CHECK constraint)\nINSERT/UPDATE → CHECK → constraint violation → DBA layer throws\nThis is the AUTHORITATIVE validation layer"]

    A -->|"proto: string"| B
    B -->|"std::string (from proto)"| C
    C -->|"std::string"| D
    D -->|"std::string via pqxx"| E

Why Primitives in DBA Layer?

  1. pqxx Compatibility: pqxx::params constructor requires types convertible to string/numeric
  2. PostgreSQL DOMAINs: The database enforces constraints authoritatively
  3. Performance: No redundant validation; DB checks on every write
  4. Simplicity: DBA layer is a thin pass-through, not a validation layer

Where Strong Types Are Used

Component Strong Types? Reason
muscovite::types::* library ✅ Yes Provide validation utilities
muscovite::qt::* library ✅ Yes QValidator integration for UI
Generated DBA entities ❌ No pqxx requires primitives
Generated proto messages ❌ No Cross-language compatibility
Generated gRPC services ❌ No Transport layer, no validation
Application service layer ✅ Optional Add validation before DBA calls

Using Strong Types in Application Code

Application code that calls the generated DBA layer can use strong types for validation:

// Application service (hand-written, not generated)
grpc::Status PatientService::UpdateEmail(
    grpc::ServerContext* ctx,
    const UpdateEmailRequest* req,
    UpdateEmailResponse* resp) {
    
    // Validate using strong type
    if (!muscovite::types::Email::is_valid(req->email())) {
        return grpc::Status(grpc::INVALID_ARGUMENT, "Invalid email format");
    }
    
    // Call generated DBA layer with primitive string
    auto patient = repo_.find_by_id(uuid_from_proto(req->patient_id()));
    patient.email = req->email();  // std::string, not Email
    repo_.update(patient);
    
    return grpc::Status::OK;
}

Consequences

Positive

  • Compile-time safety - Email and Phone are distinct types
  • Self-documenting - Code clearly shows domain intent
  • Validation at all layers - Invalid data caught early
  • IDE support - Type-specific autocomplete
  • Reuse - Leverage mature boost/Qt types

Negative

  • More code - Wrapper classes for each strong type
  • Conversion overhead - Proto ↔︎ C++ ↔︎ Qt conversions
  • Build time - More templates to compile

Neutral

  • Regex validation compiled once at startup (negligible overhead)

AI Agent Guidance

When This Applies

  • Generating C++ code for entities with STRONG_TYPE fields → Use wrapper types
  • Generating proto messages → Use muscovite.common.{TypeName} messages
  • Generating Qt models/dialogs → Use Qt wrapper types with validators
  • Converting between layers → Use generated conversion functions

When This Does NOT Apply

  • Built-in types (UUID, Date, DateTime) → Use existing library types directly
  • PostgreSQL generation → Already uses DOMAINs correctly
  • PL/pgSQL API → Already uses domain-typed parameters
