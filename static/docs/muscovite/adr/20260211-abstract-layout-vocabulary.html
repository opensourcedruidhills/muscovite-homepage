Abstract Layout Vocabulary for Presentation Layer

Abstract Layout Vocabulary for Presentation Layer

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: presentation, layout, grammar, dsl, qt, qml, ui

Context and Problem Statement

Muscovite generates functional full-stack UIs from .ddd files, but all layout decisions are hardcoded in generators:

  • Qt Widgets always uses QVBoxLayout → QFormLayout → QTabWidget for aggregates (generators/qt/widgets-views.rkt)
  • Qt QML always uses ColumnLayout → collapsible GroupBox for aggregates (generators/qt/qml-main.rkt)
  • Field ordering matches declaration order with no DSL-level override
  • No control over detail view display (table vs. cards vs. tree)
  • No control over action button placement or style

When adding new UI targets, each generator re-invents layout logic. There is no shared vocabulary for expressing presentation intent.

Question: How should the DSL express layout and presentation structure without prescribing target-specific widgets?

Decision Drivers

  • Grammar-first evolution (ADR 20260211-grammar-first-language-evolution)
  • Target independence — one DSL model, multiple UI outputs
  • Backward compatibility — existing .ddd files must generate unchanged output
  • Aggregates are the UI boundary (ADR 20260120-domain-services-as-aggregate-boundary)
  • Code generation philosophy — generate specialized classes, not runtime abstractions

Considered Options

Option 1: Abstract Layout Vocabulary in DSL Grammar

Add a LAYOUT block to the Aggregate grammar rule with target-agnostic primitives: STYLE, ORIENTATION, GROUPING, DISPLAY, ACTIONS, etc. Parser produces IR structs (layout-spec, master-spec, detail-spec, action-spec). Generators map abstract concepts to concrete widgets via target configuration modules.

Option 2: External Configuration Files (YAML/JSON)

Keep layout out of the DSL entirely. Use per-project YAML/JSON files to configure UI layout per target.

Option 3: Annotation-Based Hints

Use @layout annotations on entities and fields instead of a dedicated block.

Decision Outcome

Chosen option: Option 1 — Abstract Layout Vocabulary in DSL Grammar, because:

  • Grammar-first: layout is part of the formal PEG spec, not external config
  • Co-located with domain model: layout intent lives next to the aggregate it describes
  • Testable in Racket: grammar conformance + IR validation in the Racket test suite
  • Target-independent: generators map abstract concepts to concrete widgets
  • Backward compatible: when no LAYOUT block is present, generators use sensible defaults matching current hardcoded behavior

Grammar Extension

Add to muscovite.peg Section 9 (Aggregate):

Aggregate           <- 'AGGREGATE' IDENTIFIER UuidAnnotation? 'ROOT' IDENTIFIER
                       ContainsClause?
                       EmbedsClause?
                       EventSourcedClause?
                       LayoutBlock?
                       DocClause?
                       ';'?

LayoutBlock         <- 'LAYOUT' '{' LayoutDirective+ '}'
LayoutDirective     <- StyleHint / MasterLayout / DetailLayout / ActionLayout

StyleHint           <- 'STYLE' StyleValue
StyleValue          <- 'single_page' / 'multi_section' / 'wizard' / 'split_view'

MasterLayout        <- 'MASTER' IDENTIFIER '{' MasterProperty+ '}'
MasterProperty      <- 'ORIENTATION' OrientationValue
                     / 'GROUPING' GroupingValue
                     / 'FIELDS' FieldSelector
                     / 'READONLY' IdentifierList
                     / 'VALIDATION' ValidationValue

DetailLayout        <- 'DETAIL' IDENTIFIER '{' DetailProperty+ '}'
DetailProperty      <- 'DISPLAY' DisplayValue
                     / 'COLUMNS' IdentifierList
                     / 'SORTABLE' BoolValue
                     / 'FILTERABLE' BoolValue
                     / 'SEARCHABLE' BoolValue
                     / 'ACTIONS' IdentifierList
                     / 'ACTION_STYLE' ActionStyleValue
                     / 'INLINE_EDIT' BoolValue
                     / 'INITIAL_STATE' InitialStateValue
                     / 'PAGE_SIZE' NUMBER

ActionLayout        <- 'ACTIONS' '{' ActionProperty+ '}'
ActionProperty      <- 'PRIMARY' IdentifierList
                     / 'SECONDARY' IdentifierList
                     / 'PLACEMENT' PlacementValue

OrientationValue    <- 'vertical' / 'horizontal' / 'grid'
GroupingValue       <- 'flat' / 'sections' / 'tabs'
DisplayValue        <- 'table' / 'cards' / 'list' / 'tree'
ActionStyleValue    <- 'toolbar' / 'context_menu' / 'inline' / 'fab'
InitialStateValue   <- 'expanded' / 'collapsed'
PlacementValue      <- 'top' / 'bottom' / 'sidebar' / 'floating'
ValidationValue     <- 'inline' / 'on_submit'
FieldSelector       <- 'ALL' / IdentifierList
BoolValue           <- 'true' / 'false'
IdentifierList      <- IDENTIFIER ( ',' IDENTIFIER )*

IR Structs

Add to dslgen/model/ddd-types.rkt:

(struct layout-spec
  (style master-specs detail-specs action-spec)
  #:transparent)

(struct master-spec
  (name orientation grouping fields readonly-fields validation)
  #:transparent)

(struct detail-spec
  (collection display columns sortable filterable searchable
   actions action-style inline-edit initial-state page-size)
  #:transparent)

(struct action-spec
  (primary secondary placement)
  #:transparent)

Add layout field to aggregate struct (appended at end, after doc):

(struct aggregate
  (uuid name root entities value-objects event-sourced? doc layout)
  #:transparent)

(define (make-aggregate name root
                        #:uuid [uuid (generate-uuid)]
                        #:entities [entities '()]
                        #:value-objects [value-objects '()]
                        #:event-sourced? [event-sourced? #f]
                        #:doc [doc #f]
                        #:layout [layout #f])
  (aggregate uuid name root entities value-objects event-sourced? doc layout))

LAYOUT Attaches to AGGREGATE Only

The LAYOUT block is only valid inside Aggregate, not on standalone entities. Rationale:

  • Aggregates define the consistency boundary and the UI interaction boundary
  • All UI access goes through aggregate services (ADR 20260120)
  • Child entities appear as DETAIL sections within the aggregate's LAYOUT
  • DETAIL sections reference child entity names from the CONTAINS clause

DETAIL References Child Entities by Name

DETAIL sections use the child entity name from the CONTAINS clause:

AGGREGATE Patient @uuid "..." ROOT Patient
    (CONTAINS Appointment, MedicalRecord)
    LAYOUT {
        DETAIL Appointment { DISPLAY table }
        DETAIL MedicalRecord { DISPLAY cards }
    }
    DOC "Patient aggregate";

The parser validates that DETAIL names match CONTAINS names.

Relationship to PRESENTATION Block

  • PRESENTATION (SHORT/FULL/TABLE) — controls data formatting (how an entity displays as text)
  • LAYOUT { DETAIL { COLUMNS ... } } — controls which columns are visible in a detail view

These are complementary, not overlapping. PRESENTATION SHORT provides display text for FK comboboxes; LAYOUT DETAIL COLUMNS configures which columns appear in a table/list.

Default Behavior (No LAYOUT Block)

When no LAYOUT block is present, generators produce the same output as today:

(define (default-layout-for-aggregate aggregate)
  (layout-spec
   'multi-section
   (list (master-spec (aggregate-name aggregate) 'vertical 'flat 'all '() 'inline))
   (for/list ([child (aggregate-entities aggregate)])
     (detail-spec (entity-name child) 'table #f #t #f #f
                  '(add edit delete) 'toolbar #f 'expanded #f))
   (action-spec '(save cancel) '() 'bottom)))

Context-Sensitive Keywords

DISPLAY, ACTIONS, STYLE, PRIMARY, SECONDARY, etc. are context-sensitive — only treated as keywords inside LAYOUT { } blocks. This prevents breaking existing .ddd files that may use these as field names or identifiers.

Positive Consequences

  • One vocabulary for all UI targets — adding new frontends requires only a target config, not DSL changes
  • Gradual adoption — existing .ddd files work unchanged
  • Testable — grammar conformance + IR struct tests in Racket
  • Self-documenting — LAYOUT block makes UI intent explicit in the domain model

Negative Consequences

  • More grammar complexity — 30+ new terminals in the PEG spec
  • Parser implementation effort — new parse functions, context-sensitive keyword handling
  • Default behavior must exactly match current hardcoded output to avoid regressions

AI Agent Guidance

When This Decision Applies

  • Adding LAYOUT blocks to .ddd files
  • Modifying UI layout behavior in Qt Widgets or QML generators
  • Adding new UI targets (future mobile frontends, etc.)
  • Creating target configuration modules under dslgen/targets/

When This Decision Does NOT Apply

  • Changing PRESENTATION (SHORT/FULL/TABLE) display formatting
  • Modifying non-UI generators (PostgreSQL, gRPC, C++ DBA)
  • Parser bug fixes that don't change accepted syntax

Enforcement

  • LAYOUT block only valid inside Aggregate (parser rejects elsewhere)
  • DETAIL names must match CONTAINS clause names (parser validates)
  • Grammar conformance test must cover LayoutBlock rule
  • Default layout-spec must produce output identical to current hardcoded behavior (golden-file test)
