Command Behavioral Evolution — DECIDE Block

Command Behavioral Evolution — DECIDE Block

  • Status: accepted
  • Date: 2026-03-07
  • Deciders: Johannes Lochmann
  • Tags: command, decide, behavioral-sovereignty, rich-domain-model

Context

The existing COMMAND construct (muscovite.peg:340) declares intent structure but not behavior:

Command <- 'COMMAND' IDENTIFIER 'ON' IDENTIFIER FieldList RequiresClause? EmitsClause? ';'?

It has REQUIRES (precondition) and EMITS (event name list), but no logic body. The actual command handling is generated as an opaque DomainService method stub that developers must implement manually. This creates an anemic domain model where the DSL captures structure but not behavior.

An alternative proposal suggested a new ACTION top-level construct. This was rejected because COMMAND already occupies this semantic space — adding ACTION would create a parallel construct with overlapping responsibility.

Decision

Evolve COMMAND with an optional DECIDE block containing pure functional logic:

Command <- 'COMMAND' IDENTIFIER 'ON' IDENTIFIER FieldList
           RequiresClause*
           DecideBlock?
           EnsuresClause?
           EmitsClause
           DocClause?
           ';'?

DecideBlock <- 'DECIDE' '{' DecideStatement+ '}'

DecideStatement <- LetStatement / EmitStatement / PlanStatement

LetStatement  <- 'LET' IDENTIFIER '=' Expression ';'
EmitStatement <- 'EMIT' IDENTIFIER ActionParams ';'
PlanStatement <- 'PLAN' IDENTIFIER ActionParams ';'

EnsuresClause <- 'ENSURES' IDENTIFIER ':' Expression

Key properties:

  • DECIDE is optional — commands without it remain stub-generated (backward compatible)
  • DECIDE body is a pure function: no I/O, no service calls, no mutation
  • Output is an intent list: EMIT (internal state change) + PLAN (external side effect)
  • ENSURES is a postcondition checked after DECIDE execution; transaction rolls back on failure
  • Multiple REQUIRES clauses allowed (named preconditions)

IR representation (ddd-types.rkt):

(struct decide-block (statements) #:transparent)
(struct let-stmt (name expr) #:transparent)
(struct emit-stmt (event-name args) #:transparent)
(struct plan-stmt (effect-name args) #:transparent)
(struct ensures-clause (name expr) #:transparent)

Generation target: The DECIDE block generates into the body of DomainService::{command_name}() (ADR 20260307-domain-service-layer). The DomainService method:

  1. Evaluates REQUIRES preconditions
  2. Executes DECIDE logic (pure computation)
  3. Checks ENSURES postcondition
  4. Applies EMITs via uow.add_event() (atomic, same transaction)
  5. Applies PLANs via outbox entries (OutboxPublisher relay)

Commands without DECIDE blocks continue generating as sidecar stubs (ADR 20260223, FTR-138). This preserves the plugin escape hatch for logic too complex for the expression language.

Consequences

  • COMMAND grammar grows significantly but remains backward compatible
  • command IR struct gains optional decide-block and ensures fields
  • DomainService generator must handle both stub and DECIDE-based commands
  • EMITS list must be validated against EMIT statements in DECIDE block at parse time
  • Variable scoping in DECIDE is sequential (each LET can reference prior LETs)
  • Depends on functional expression language (ADR 20260307-functional-expression-language)
