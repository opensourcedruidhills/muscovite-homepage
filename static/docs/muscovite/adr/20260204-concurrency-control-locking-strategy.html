Concurrency Control and Locking Strategy

Concurrency Control and Locking Strategy

  • Status: proposed
  • Date: 2026-02-04
  • Deciders: (pending)
  • Tags: concurrency, locking, optimistic, pessimistic, advisory-locks

Context and Problem Statement

Multi-user applications require concurrency control to prevent lost updates and ensure data consistency. Muscovite-generated applications need a strategy for:

  1. Concurrent writes: Two users updating the same entity simultaneously
  2. UI edit sessions: User opens edit dialog, another user wants to edit same entity
  3. Crash recovery: Application crash or network disconnect leaves entity in "locked" state

The challenge is balancing user experience (minimal blocking) with data integrity (no lost updates).

Decision Drivers

  • Type safety across all layers (consistent with Muscovite philosophy)
  • Automatic cleanup on session end or crash
  • Integration with event sourcing (version as event sequence)
  • Visibility of lock status across distributed clients
  • No stale locks from crashed processes

Considered Options

Option 1: Optimistic Locking via Version Field

Add explicit version INTEGER NOT NULL DEFAULT 1 column to entities. Increment on each update with conflict detection.

UPDATE patient SET 
    active = false, 
    version = version + 1, 
    updated_at = NOW()
WHERE id = $1 AND version = $2
RETURNING version;  -- Returns 0 rows if version mismatch

PostgreSQL Intrinsic Columns Evaluation:

Column Purpose Suitable?
xmin Transaction ID of inserting txn ❌ 32-bit, wraps at ~4B txns
xmax Transaction ID of deleting txn ❌ Same wraparound issue
ctid Physical row location ❌ Changes on VACUUM

Verdict: Intrinsic columns are not suitable. Explicit version field required.

Pros:

  • Simple, well-understood pattern
  • No special database features required
  • Works with replicas (read-your-writes)
  • No deadlock risk
  • Natural fit for event sourcing (version = event sequence number)

Cons:

  • Requires retry logic on conflict
  • Client must track version they loaded
  • Concurrent writes cause one to fail (must retry)

Option 2: Pessimistic Locking via SELECT FOR UPDATE

BEGIN;
SELECT * FROM patient WHERE id = $1 FOR UPDATE;
-- Other transactions block here until COMMIT/ROLLBACK
UPDATE patient SET ...;
COMMIT;

Pros:

  • Guarantees exclusive access during transaction
  • Simple transaction semantics

Cons:

  • Lock held only during transaction—not useful for UI edit sessions
  • Deadlock risk when locking multiple rows
  • Doesn't solve "user opens dialog and walks away" problem

Option 3: Advisory Locks + Event Bus

PostgreSQL advisory locks with application-defined semantics, combined with event bus notifications.

-- When user opens edit dialog
SELECT pg_try_advisory_lock(hashtext('patient'), uuid_hash($1));
-- Returns true if acquired, false if already locked

-- When user saves or closes dialog  
SELECT pg_advisory_unlock(hashtext('patient'), uuid_hash($1));

Critical Property (from PostgreSQL docs):

"Advisory locks are automatically cleaned up by the server at the end of the session."

This solves the crash problem—if application crashes or network disconnects, PostgreSQL automatically releases all session-level advisory locks.

Lock Types:

Type Behavior Use Case
pg_advisory_lock Session-level, blocking Long edit sessions
pg_try_advisory_lock Session-level, non-blocking Check availability
pg_advisory_xact_lock Transaction-level Short operations

Pros:

  • Survives crashes (automatic cleanup on session end)
  • Can check lock status before waiting
  • Visible in pg_locks system view
  • Can broadcast lock acquisition via NATS for UI updates

Cons:

  • Requires session-per-entity or connection pooling awareness
  • Advisory = application must enforce (not automatic)
  • User closing browser without proper cleanup needs heartbeat mechanism

Option 4: Hybrid (Optimistic + Advisory + Event Bus)

Combine approaches for different scenarios:

flowchart TD
    A["1. User opens edit dialog\nTRY pg_advisory_lock(entity_type_hash, entity_uuid_hash)\nIf locked: Show 'Being edited by [user]'\nIf acquired: Publish lock.acquired event to NATS"]
    B["2. User edits and saves\nUPDATE ... WHERE id = $1 AND version = $2\nIf version mismatch: Optimistic lock conflict\nIf success: Release advisory lock, publish lock.released"]
    C["3. User cancels or closes dialog\nRelease advisory lock\nPublish lock.released event"]
    D["4. Session ends (crash, disconnect, timeout)\nPostgreSQL auto-releases advisory lock\nHeartbeat timeout triggers lock.released (via service)"]

    A --> B
    A --> C
    A --> D

Pros:

  • Best of both worlds: OCC for data integrity, advisory for UI coordination
  • Graceful degradation (advisory lock failure doesn't block saves)
  • Full visibility via event bus

Cons:

  • More complex implementation
  • Two mechanisms to understand and maintain

Stale Lock Comparison

Scenario Flag in Table Advisory Lock Version Field
App crashes ❌ Stale lock ✅ Auto-released ✅ N/A
Network drops ❌ Stale lock ✅ Auto-released ✅ N/A
User closes tab ❌ Stale lock* ⚠️ Session-dependent ✅ N/A
Server restart ❌ Stale lock ✅ All locks cleared ✅ N/A

*With proper beforeunload handler, can be mitigated but not guaranteed.

Proposed DSL Syntax

ENTITY Patient IDENTIFIED_BY id (
    id          UUID        NOT NULL,
    version     INTEGER     NOT NULL DEFAULT 1,  -- Optimistic lock
    -- ...
)
CONCURRENCY {
    OPTIMISTIC version;           -- Column for OCC
    ADVISORY;                     -- Enable advisory lock support
    LOCK_TIMEOUT "5m";            -- Auto-release after inactivity
}

Open Questions

  2. Default behavior: All entities get version, or opt-in via CONCURRENCY block?

  4. Lock timeout: Configurable per-entity or global setting?
  5. Event bus integration: Required, or optional enhancement?

AI Agent Guidance

When This ADR Applies

  • Implementing concurrent edit functionality
  • Adding version fields to entities
  • Generating lock management code
  • Qt dialog edit session handling

When This ADR Does Not Apply

  • Read-only queries (no locking needed)
  • Batch imports (use transaction isolation instead)
  • Event sourcing aggregates (use event version, not row version)

Key Implementation Notes

  • Never use PostgreSQL xmin/xmax for optimistic locking
  • Advisory locks require stable database sessions (beware connection pooling)
  • Always implement heartbeat for long-running UI sessions
  • Event sourcing version ≠ entity version (different semantics)
