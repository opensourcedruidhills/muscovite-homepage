Denormalized display_name Field with Database Triggers

Denormalized display_name Field with Database Triggers

  • Status: superseded by 20260214-remove-display-name-triggers
  • Deciders: jolo, AI agent
  • Date: 2026-01-15
  • Tags: database, triggers, denormalization, presentation, ui

Context and Problem Statement

In domain models following FHIR patterns (Patient, Practitioner), the entity's name information is stored in separate child entities (PatientName, PractitionerName) via @embedded relationships. This is correct relational design but creates a UI problem: when displaying these entities in ComboBox dropdowns or list views, the parent entity has no direct name field to display—only a UUID primary key.

The PRESENTATION syntax allows defining display formats like SHORT: "{display_name}", but if the entity has no display_name field, it falls back to showing the UUID, which is useless for users.

Decision Drivers

  • User Experience: Users need to see human-readable names, not UUIDs
  • Data Consistency: Display name must stay in sync with child name records
  • Performance: Avoid N+1 queries when loading dropdowns/lists
  • DRY Principle: Centralize name composition logic; don't duplicate in every frontend
  • Audit Trail: Changes to name records must automatically reflect in display_name

Considered Options

  1. Frontend composition - Load child name records and compose in UI
  2. View-based approach - Create PostgreSQL views joining parent and child
  3. Denormalized field with triggers - Store display_name on parent, maintain via triggers
  4. Computed column - Use PostgreSQL generated column (not possible with cross-table data)

Decision Outcome

Chosen option: Denormalized field with database triggers

Implementation

  1. Add display_name TEXT field to parent entities (Patient, Practitioner)

  2. Generate PostgreSQL triggers via Racket that:

    • Fire AFTER INSERT/UPDATE/DELETE on child name entities
    • Select the "best" name using priority ordering: official > usual > first found
    • Compose name string: "{prefix} {given} {family} {suffix}" (trimmed)
    • Update parent's display_name field
    • Handle FK changes (when name is reassigned to different parent)
    • Set fallback placeholder when no names exist: "Patient #UUID"

  3. Update PRESENTATION syntax to reference display_name:

    PRESENTATION (
    SHORT: "{display_name}",
    FULL: "{display_name} (DOB: {birth_date})",
    TABLE: "{display_name}"
)

Trigger Generation

The trigger generator (display-name-trigger.rkt) automatically detects:

  • Parent entities with a display_name field
  • Child entities with @embedded relationship to parent
  • Child entities with name-related fields (family, given, prefix, suffix)

Generated SQL files follow Sqitch conventions:

  • deploy/{context}-trigger-{child}-display-name.sql
  • revert/{context}-trigger-{child}-display-name.sql
  • verify/{context}-trigger-{child}-display-name.sql

Priority Ordering

The trigger uses FHIR-standard use values for priority:

ORDER BY 
    CASE WHEN use = 'official' THEN 1
         WHEN use = 'usual' THEN 2
         ELSE 3
    END,
    id  -- Deterministic tiebreaker
LIMIT 1

Consequences

Positive

  • Instant lookups: ComboBox/list loads require only parent table query
  • Consistent formatting: Name composition logic centralized in database
  • Automatic sync: Any INSERT/UPDATE/DELETE on names immediately updates display_name
  • Works with existing architecture: No changes to gRPC services or Qt client logic
  • PRESENTATION syntax just works: "{display_name}" is a regular field reference

Negative

  • Storage overhead: Redundant data (display_name duplicates child data)
  • Trigger complexity: Must handle INSERT/UPDATE/DELETE and FK changes
  • Migration required: Existing data needs backfill (run trigger logic manually)

Neutral

  • Fallback placeholder ("Patient #UUID") is functional but not ideal for empty states

AI Agent Guidance

When This Pattern Applies

  • Entity has child records containing display-relevant data (names, titles)
  • Entity needs human-readable representation in dropdowns/lists
  • Child records have @embedded relationship to parent
  • Multiple child records possible with priority selection needed

When This Pattern Does NOT Apply

  • Entity has all display fields directly (no child decomposition)
  • Simple lookup tables where code or name field suffices
  • Entities never shown in ComboBox/list contexts

Checklist for Adding display_name to New Entity

  1. Add display_name TEXT field to parent entity in .ddd file
  2. Ensure child name entity has @embedded relationship to parent
  3. Ensure child has family, given, prefix, suffix fields (or subset)
  4. Update PRESENTATION to use "{display_name}"
  5. Regenerate project
  6. Deploy new migrations (trigger will auto-populate on first INSERT)
