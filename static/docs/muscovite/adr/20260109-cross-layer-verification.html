Cross-Layer Consistency Verification

Cross-Layer Consistency Verification

Technical Story: Verify that generated migrations and code maintain consistency across all architectural layers.

Context and Problem Statement

Generated migrations and code must not break consistency between layers:

Layer Artifact Risk if Inconsistent
PostgreSQL Tables, columns, types Runtime SQL errors
PL/pgSQL API Functions, parameters API call failures
C++ DBA Entities, repositories Compilation errors, runtime crashes
gRPC Proto definitions, services Client/server mismatch
Qt Client Models, widgets UI crashes, data corruption

Currently, Muscovite generates all layers from the same IR, which provides generation-time consistency. However:

  1. No deployed-state verification: Cannot verify migration was applied correctly
  2. No cross-layer type checking: Type mismatches may not be caught until runtime
  3. No API signature verification: API changes may break clients silently
  4. No regression detection: Model changes may break existing integrations

Decision Drivers

  • Ensure semantic fidelity across all generated layers
  • Detect mismatched types, relations, and invariants before deployment
  • Maintain correctness in CRUD operations and generated UI
  • Enable CI/CD verification gates
  • Support incremental model evolution safely

Considered Options

  • Manual verification: Rejected - error-prone, doesn't scale
  • Automated tests per layer: Partial - catches some issues
  • Integration testing only: Partial - catches runtime issues, slow
  • Semantic verification against IR: Accepted - catches issues at generation time

Decision Outcome

Chosen option: "Semantic verification against IR", implementing an automated verification stage that:

  1. Compares deployed schema against IR
  2. Validates API signatures against schema
  3. Verifies C++ types against API
  4. Validates gRPC contracts against backend
  5. Checks Qt models against gRPC

Positive Consequences

  • Early detection of inconsistencies (before runtime)
  • CI/CD-friendly verification gates
  • Supports safe incremental model evolution
  • Documents expected behavior per layer
  • Enables automated regression testing

Negative Consequences

  • Adds runtime cost for verification
  • Requires database connection for schema verification
  • Some verifications require compilation (C++)
  • May produce false positives for intentional variations

Implementation

Phase 1: Schema Verification

Create src/dslgen/verification/verify-schema.rkt:

#lang racket/base

(require db)
(provide verify-schema-against-ir)

;; Connect and verify PostgreSQL schema matches IR
(define (verify-schema-against-ir ir connection-string)
  (define conn (postgresql-connect #:database "geosciences"
                                   #:user "postgres"
                                   #:password "postgres"
                                   #:server "localhost"
                                   #:port 5555))
  
  (define issues '())
  
  ;; Verify each entity exists as table
  (for ([ctx (ddd-model-contexts ir)])
    (define schema-name (context->schema-name ctx))
    (for ([agg (bounded-context-aggregates ctx)])
      (define entity (aggregate-root agg))
      (define table-name (entity->table-name entity))
      
      ;; Check table exists
      (unless (table-exists? conn schema-name table-name)
        (set! issues (cons (verification-issue
                            'error 'missing-table
                            (format "~a.~a" schema-name table-name)
                            "Table does not exist in database")
                           issues)))
      
      ;; Check columns match
      (when (table-exists? conn schema-name table-name)
        (define db-columns (get-table-columns conn schema-name table-name))
        (for ([field (entity-fields entity)])
          (define col-name (field->column-name field))
          (define expected-type (field->pg-type field))
          
          (cond
            [(not (hash-has-key? db-columns col-name))
             (set! issues (cons (verification-issue
                                 'error 'missing-column
                                 (format "~a.~a.~a" schema-name table-name col-name)
                                 "Column does not exist")
                                issues))]
            
            [(not (type-compatible? (hash-ref db-columns col-name) expected-type))
             (set! issues (cons (verification-issue
                                 'error 'type-mismatch
                                 (format "~a.~a.~a" schema-name table-name col-name)
                                 (format "Expected ~a, found ~a" 
                                         expected-type 
                                         (hash-ref db-columns col-name)))
                                issues))])))))
  
  (disconnect conn)
  issues)

;; Query information_schema for table columns
(define (get-table-columns conn schema table)
  (define rows 
    (query-rows conn
      "SELECT column_name, data_type, is_nullable 
       FROM information_schema.columns 
       WHERE table_schema = $1 AND table_name = $2"
      schema table))
  (for/hash ([row rows])
    (values (vector-ref row 0)
            (hash 'type (vector-ref row 1)
                  'nullable (equal? (vector-ref row 2) "YES")))))

Phase 2: API Verification

Create src/dslgen/verification/verify-api.rkt:

;; Verify PL/pgSQL API functions match schema
(define (verify-api-against-schema ir connection-string)
  (define conn (postgresql-connect ...))
  (define issues '())
  
  (for ([ctx (ddd-model-contexts ir)])
    (define api-schema (format "~a_api" (context->schema-name ctx)))
    
    (for ([agg (bounded-context-aggregates ctx)])
      (define entity (aggregate-root agg))
      (define entity-name (symbol->string (entity-name entity)))
      
      ;; Check insert function
      (define insert-fn (format "~a_insert" entity-name))
      (define insert-params (entity->insert-params entity))
      
      (unless (function-exists? conn api-schema insert-fn)
        (set! issues (cons (verification-issue
                            'error 'missing-function
                            (format "~a.~a" api-schema insert-fn)
                            "API function does not exist")
                           issues)))
      
      (when (function-exists? conn api-schema insert-fn)
        (define db-params (get-function-params conn api-schema insert-fn))
        (define param-issues (verify-params-match insert-params db-params))
        (set! issues (append param-issues issues)))
      
      ;; Check select_by_id function
      (define select-fn (format "~a_select_by_id" entity-name))
      (unless (function-exists? conn api-schema select-fn)
        (set! issues (cons (verification-issue
                            'warning 'missing-function
                            (format "~a.~a" api-schema select-fn)
                            "Select function does not exist")
                           issues)))))
  
  (disconnect conn)
  issues)

;; Query pg_proc for function parameters
(define (get-function-params conn schema fn-name)
  (query-rows conn
    "SELECT p.parameter_name, p.data_type
     FROM information_schema.parameters p
     JOIN information_schema.routines r ON p.specific_name = r.specific_name
     WHERE r.routine_schema = $1 AND r.routine_name = $2
     ORDER BY p.ordinal_position"
    schema fn-name))

Phase 3: gRPC Contract Verification

Create src/dslgen/verification/verify-grpc.rkt:

;; Verify gRPC proto matches IR
(define (verify-grpc-against-ir ir proto-dir)
  (define issues '())
  
  (for ([ctx (ddd-model-contexts ir)])
    (define ctx-name (symbol->string (bounded-context-name ctx)))
    (define proto-path (build-path proto-dir ctx-name (format "~a.proto" ctx-name)))
    
    (if (file-exists? proto-path)
        (begin
          (define proto-content (file->string proto-path))
          (define proto-messages (parse-proto-messages proto-content))
          
          (for ([agg (bounded-context-aggregates ctx)])
            (define entity (aggregate-root agg))
            (define msg-name (entity->proto-message-name entity))
            
            (unless (hash-has-key? proto-messages msg-name)
              (set! issues (cons (verification-issue
                                  'error 'missing-message
                                  msg-name
                                  (format "Proto message not found in ~a" proto-path))
                                 issues)))
            
            ;; Verify fields match
            (when (hash-has-key? proto-messages msg-name)
              (define proto-fields (hash-ref proto-messages msg-name))
              (for ([field (entity-fields entity)])
                (define proto-field-name (field->proto-field-name field))
                (unless (member proto-field-name proto-fields)
                  (set! issues (cons (verification-issue
                                      'error 'missing-proto-field
                                      (format "~a.~a" msg-name proto-field-name)
                                      "Field not in proto message")
                                     issues)))))))
        
        (set! issues (cons (verification-issue
                            'error 'missing-proto
                            proto-path
                            "Proto file does not exist")
                           issues))))
  
  issues)

Phase 4: Cross-Layer Type Verification

Create src/dslgen/verification/verify-types.rkt:

;; Verify type mappings are consistent across layers
(define (verify-type-consistency ir)
  (define issues '())
  
  (for ([ctx (ddd-model-contexts ir)])
    (for ([agg (bounded-context-aggregates ctx)])
      (define entity (aggregate-root agg))
      
      (for ([field (entity-fields entity)])
        (define ir-type (field-type field))
        (define pg-type (dsl-type->pg-type ir-type))
        (define cpp-type (dsl-type->cpp-type ir-type))
        (define proto-type (dsl-type->proto-type ir-type))
        (define qt-type (dsl-type->qt-type ir-type))
        
        ;; Verify round-trip compatibility
        (unless (types-roundtrip-compatible? pg-type cpp-type)
          (set! issues (cons (verification-issue
                              'warning 'type-roundtrip
                              (format "~a.~a" (entity-name entity) (field-name field))
                              (format "PG type ~a may not round-trip with C++ type ~a"
                                      pg-type cpp-type))
                             issues)))
        
        (unless (types-roundtrip-compatible? proto-type cpp-type)
          (set! issues (cons (verification-issue
                              'warning 'type-roundtrip
                              (format "~a.~a" (entity-name entity) (field-name field))
                              (format "Proto type ~a may not round-trip with C++ type ~a"
                                      proto-type cpp-type))
                             issues))))))
  
  issues)

;; Check if types can round-trip without data loss
(define (types-roundtrip-compatible? type1 type2)
  (case (list type1 type2)
    [((uuid string)) #t]           ;; UUID as string is safe
    [((text string)) #t]           ;; TEXT as string is safe
    [((integer int64_t)) #t]       ;; INTEGER fits in int64
    [((bigint int64_t)) #t]        ;; BIGINT fits in int64
    [((numeric double)) #f]        ;; NUMERIC to double loses precision!
    [((numeric string)) #t]        ;; NUMERIC as string is safe
    [((timestamptz Timestamp)) #t] ;; TIMESTAMPTZ to proto Timestamp is safe
    [else #t]))                    ;; Default assume compatible

Phase 5: Unified Verification Report

Create src/dslgen/verification/verify-all.rkt:

(provide verify-all-layers
         verification-report)

(struct verification-report
  (timestamp
   ir-hash
   schema-issues
   api-issues
   grpc-issues
   type-issues
   summary)
  #:transparent)

;; Run all verifications
(define (verify-all-layers ir connection-string proto-dir)
  (define schema-issues (verify-schema-against-ir ir connection-string))
  (define api-issues (verify-api-against-schema ir connection-string))
  (define grpc-issues (verify-grpc-against-ir ir proto-dir))
  (define type-issues (verify-type-consistency ir))
  
  (define all-issues (append schema-issues api-issues grpc-issues type-issues))
  (define error-count (count (λ (i) (eq? (verification-issue-severity i) 'error)) all-issues))
  (define warning-count (count (λ (i) (eq? (verification-issue-severity i) 'warning)) all-issues))
  
  (verification-report
   (current-iso-timestamp)
   (ir-content-hash ir)
   schema-issues
   api-issues
   grpc-issues
   type-issues
   (hash 'total (length all-issues)
         'errors error-count
         'warnings warning-count
         'passed? (zero? error-count))))

;; Print human-readable report
(define (print-verification-report report)
  (printf "=== Muscovite Cross-Layer Verification Report ===\n")
  (printf "Timestamp: ~a\n" (verification-report-timestamp report))
  (printf "IR Hash: ~a\n\n" (verification-report-ir-hash report))
  
  (printf "Schema Verification:\n")
  (for ([issue (verification-report-schema-issues report)])
    (printf "  [~a] ~a: ~a\n" 
            (verification-issue-severity issue)
            (verification-issue-location issue)
            (verification-issue-message issue)))
  
  ;; ... similar for other issue types
  
  (printf "\nSummary: ~a errors, ~a warnings\n"
          (hash-ref (verification-report-summary report) 'errors)
          (hash-ref (verification-report-summary report) 'warnings))
  
  (if (hash-ref (verification-report-summary report) 'passed?)
      (printf "✓ Verification PASSED\n")
      (printf "✗ Verification FAILED\n")))

AI Agent Guidance

When This Decision Applies

  • After generating or applying migrations
  • In CI/CD verification pipelines
  • When debugging layer inconsistencies
  • During model evolution validation

When This Decision Does NOT Apply

  • During initial project setup (no existing schema)
  • For non-PostgreSQL databases
  • For code review of generated files (that's manual)

Enforcement

  • CI/CD pipelines MUST run verification after migration
  • Verification errors MUST block deployment
  • Warnings SHOULD be reviewed but may not block
  • Reports MUST be archived for audit
