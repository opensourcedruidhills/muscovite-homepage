Compile-Time Lookup Constants via Deterministic Seed UUIDs

Compile-Time Lookup Constants via Deterministic Seed UUIDs

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-12
  • Tags: lookup-tables, c++, code-generation, type-safety, seeds

Context and Problem Statement

Muscovite generates lookup tables (ADR 20260116) for all enumerated domain values. Current seed JSON files omit the id field, relying on DEFAULT uuidv7() at the database layer. This means UUIDs are non-deterministic — they differ across deployments.

When C++ code needs to reference a lookup value (e.g., "set saga status to completed"), it must perform a runtime query to resolve the UUID:

// Runtime lookup — wasteful, fragile, untestable without DB
auto status_id = query_repo.find_by_code("completed");
txn.exec_params("UPDATE ... SET status_id = $1", status_id);

This is especially problematic for framework infrastructure tables (saga status, step results) where the set of values is known at generation time and will never change at runtime.

Decision Drivers

  • Lookup table UUIDs must be stable across regenerations (deterministic)
  • C++ code must reference lookup values without runtime queries
  • Mismatches between C++ constants and database rows must be caught at compile time, not runtime
  • Pattern must work for both generator-created seeds (saga infrastructure) and user-authored seeds (domain basedata)
  • Must not break existing seed JSON files that omit id

Decision Outcome

Generate deterministic UUIDs (UUIDv5) in seed JSON files + matching C++ headers with compile-time constants.

Two-Part Generation

For every lookup table whose values are referenced in generated C++ code:

Part 1 — Seed JSON with deterministic UUIDs:

The Racket generator produces UUIDv5 from a fixed namespace UUID + "{context}.{table}.{code}" as the name. The resulting UUID is stable across regenerations because the inputs are deterministic.

[
  {
    "id": "a1b2c3d4-...",
    "code": "completed",
    "display": "Completed",
    "definition": "Saga completed successfully",
    "is_active": true,
    "sort_order": 4
  }
]

Part 2 — C++ header with compile-time constants:

// Generated — DO NOT EDIT
// Lookup constants for workflows.saga_status
#pragma once
#include <boost/uuid/uuid.hpp>

namespace workflows::lookup::saga_status {

inline constexpr boost::uuids::uuid not_started{0xa1, 0xb2, ...};
inline constexpr boost::uuids::uuid in_progress{0xc3, 0xd4, ...};
inline constexpr boost::uuids::uuid compensating{0xe5, 0xf6, ...};
inline constexpr boost::uuids::uuid completed{0x17, 0x28, ...};
inline constexpr boost::uuids::uuid failed{0x39, 0x4a, ...};
inline constexpr boost::uuids::uuid cancelled{0x5b, 0x6c, ...};

} // namespace workflows::lookup::saga_status

Usage in generated repositories:

#include "workflows/lookup/saga_status.hpp"

void MetamorphicPathwaySagaCommandRepository::complete(
    pqxx::work& txn, const boost::uuids::uuid& saga_id)
{
    txn.exec_params(
        "UPDATE workflows.metamorphic_pathway_instances "
        "SET status_id = $1, completed_at = NOW(), updated_at = NOW() "
        "WHERE id = $2",
        lookup::saga_status::completed,  // compile-time constant
        saga_id);
}

UUIDv5 Namespace

A single fixed namespace UUID for all Muscovite lookup seeds:

;; Muscovite lookup seed namespace (UUIDv5)
(define MUSCOVITE_LOOKUP_NS "6ba7b810-9dad-11d1-80b4-00c04fd430c8")

Name string format: "{context_name}.{table_name}.{code}" — e.g., "workflows.saga_status.completed".

Applicability

Seed Source Deterministic UUIDs C++ Header When
Generator-created (saga_status, saga_step, etc.) Mandatory Mandatory Always — framework infrastructure
User-authored with @constants Mandatory Mandatory When DSL declares LOOKUP_TABLE ... @constants
User-authored without @constants Optional Not generated Default — no compile-time references needed

For user-authored seeds: if the JSON already contains id fields, those are used verbatim. If id is absent, the generator injects UUIDv5. This preserves backward compatibility.

Seed INSERT Generation Change

The make-seed-change function in sqitch.rkt must be updated to include the id column in INSERT statements when the seed JSON contains id fields. Currently it passes through all JSON keys verbatim, so this works automatically — no INSERT logic change needed, only the JSON content changes.

Consequences

Good

  • Zero runtime lookups for framework constants
  • Compile-time detection of code/UUID mismatches (typo in constant name = build error)
  • Deterministic UUIDs enable reproducible database state across environments
  • Same UUIDs in seed JSON and C++ header — single source of truth (Racket generator)
  • Pattern extends naturally to any lookup table, not just saga infrastructure

Bad

  • UUIDv5 dependency in Racket generator (minor — uuid package)
  • Slightly more generated code (one .hpp per lookup table with constants)
  • Existing deployments with non-deterministic UUIDs need migration if adopting this pattern retroactively

Neutral

  • Does not affect lookup tables where C++ never references specific rows
  • gRPC/Qt layers can use code strings (resolved server-side) — constants are a C++ DBA optimization

AI Agent Guidance

When This Applies

  • Generating lookup tables for saga infrastructure (status, step, step_result)
  • Any lookup table where generated C++ code references specific rows by identity
  • When a LOOKUP_TABLE has @constants annotation in the DSL

When This Does NOT Apply

  • User-facing lookup tables where C++ never references specific values
  • Lookup tables populated at runtime (not seeded from DSL)
  • Non-C++ targets (use code strings via gRPC)

Enforcement

  • Generator-created lookup tables (saga infrastructure) MUST produce both seed JSON with UUIDs and C++ constants header
  • Generated C++ repositories MUST use constants, never hardcoded UUID strings or runtime lookups
  • Verify: grep for find_by_code in generated saga repositories — should return zero matches

Common Mistakes

  • ❌ Hardcoding UUID strings in generated C++ ("a1b2c3d4-...")
  • ❌ Runtime SELECT id FROM ... WHERE code = 'completed' in hot paths
  • ❌ Using gen_random_uuid() for lookup tables that need compile-time constants
  • ✅ lookup::saga_status::completed — named constant, compile-time checked
