OpenTelemetry Observability Architecture

OpenTelemetry Observability Architecture

  • Status: proposed
  • Date: 2026-03-02
  • Tags: observability, opentelemetry, tracing, metrics, correlation-id, versioning
  • Deciders: Johannes Lochmann

Technical Story: Generated Muscovite gRPC services need production-grade observability: distributed tracing, metrics export, correlation ID propagation, and DDD model version exposure. IDEA-046, IDEA-047, IDEA-048.

Context and Problem Statement

Generated gRPC services currently have basic observability infrastructure:

  • MetricsInterceptorFactory: In-memory counters (request count, duration histogram, error count) — not exported anywhere
  • LoggingInterceptorFactory: spdlog per-request logging (method, duration, status) — local only
  • ErrorInterceptorFactory: Exception → gRPC status mapping

This infrastructure collects useful data but has three critical gaps:

  1. No distributed tracing: In a multi-context deployment (FHIR Health: 9 contexts, Geosciences: 26 contexts), operators cannot trace a request across service boundaries or correlate gRPC calls with database operations.

  2. No request correlation: A single user action that triggers operations across multiple bounded contexts produces isolated log lines with no shared identifier. Debugging requires manual timestamp correlation.

  3. No version introspection: Running services cannot report which .ddd model version they were generated from, making deployment verification and version-mismatch debugging impossible.

How should Muscovite generate production-grade observability into every service with zero manual effort from domain modelers?

Decision Drivers

  • License compatibility: All new dependencies must be compatible with AGPL-3.0 + Commercial dual licensing
  • Zero-effort for domain modelers: Observability is infrastructure, not domain concern — it should be generated, not hand-coded
  • Opt-in with zero overhead when disabled: Projects that don't need OTEL should have zero runtime cost
  • Industry standard: OpenTelemetry is the CNCF-graduated standard for observability
  • Existing infrastructure: Current interceptor pattern in framework/grpc/interceptors.hpp is a natural extension point
  • Cross-cutting: Correlation ID must flow through all layers (gRPC → service → repository → PostgreSQL → events → NATS)

Considered Options

  1. OpenTelemetry C++ SDK with capability-gated generation
  2. Prometheus metrics + Jaeger tracing (separate libraries)
  3. Enhance existing in-memory metrics + spdlog only

Decision Outcome

Chosen option: Option 1 — OpenTelemetry C++ SDK, because:

  • Apache-2.0 licensed (compatible with AGPL-3.0 + Commercial)
  • Single SDK covers traces, metrics, and logs (three signals, one dependency)
  • OTLP exporter works with any OTEL Collector backend (Jaeger, Prometheus, Grafana, Datadog, etc.)
  • Industry standard — operators expect OTEL in cloud-native services
  • gRPC transport for OTLP is natural since we already depend on gRPC

Architecture: Three Pillars + Two Cross-Cutting Concerns

Pillar 1: Distributed Tracing (OpenTelemetry Traces)

Generated services emit spans at three granularity levels:

Span Scope Attributes
RPC span Full gRPC method execution rpc.method, rpc.service, rpc.grpc.status_code, correlation_id
Service span CommandService/QueryService method muscovite.context, muscovite.aggregate, muscovite.operation
Database span Individual pqxx operation db.system=postgresql, db.statement (sanitized), db.operation

Span context is propagated via W3C traceparent/tracestate gRPC metadata headers (standard gRPC OTEL propagation).

Pillar 2: Metrics (OpenTelemetry Metrics)

Replace MetricsInterceptorFactory with OTEL-native metrics:

Metric Type Labels
muscovite.grpc.request.duration Histogram method, status, context
muscovite.grpc.request.count Counter method, status, context
muscovite.db.query.duration Histogram operation, context, aggregate
muscovite.db.pool.active Gauge context
muscovite.outbox.publish.count Counter context, event_type
muscovite.outbox.publish.duration Histogram context

Pillar 3: Logs (OpenTelemetry Log Bridge)

Keep spdlog for local development. When OTEL is enabled, add an OTEL LogBridge that:

  • Enriches spdlog output with trace_id and span_id from current span context
  • Exports structured logs via OTLP for cloud-native log aggregation
  • Adds correlation_id to every log record

Cross-Cutting Concern 1: Correlation ID

Always generated (not gated by TELEMETRY capability) because correlation is valuable even without full OTEL:

  1. gRPC interceptor extracts x-correlation-id from incoming metadata
  2. If absent, generates UUIDv7 (time-ordered for natural log ordering)
  3. Stores in CorrelationContext (scoped, thread-local)
  4. Propagated to:
    • All spdlog output: [correlation_id={id}]
    • Domain events: correlation_id column in domain_events table
    • NATS outbox messages: Nats-Correlation-Id header
    • Outgoing gRPC calls: x-correlation-id metadata
    • PostgreSQL: SET LOCAL muscovite.correlation_id = '{id}' for log_line_prefix correlation
  5. When OTEL is enabled, correlation ID is added as span attribute and the OTEL trace ID can serve as the correlation ID

Cross-Cutting Concern 2: DDD Model Version

Always generated — every service reports its lineage:

  1. Generator emits model_version.hpp with compile-time constants:

    • kModelVersion: from project.mus VERSION
    • kGeneratedAt: ISO-8601 timestamp of generation
    • kGeneratorVersion: from MUSCOVITE_CPP_VERSION
    • kModelHash: SHA-256 of the serialized IR (stable across whitespace changes)
    • kGitCommit: from libgit2 at generation time

  2. Generated VersionService gRPC service:

    • GetVersion() → returns all version attributes
    • Auto-registered in server_main.cpp

  3. Every gRPC response includes x-model-version trailer metadata

  4. When OTEL is enabled, version attributes are OTEL resource attributes on all telemetry:

    • service.version = kModelVersion (OTEL semantic convention)
    • muscovite.generator.version = kGeneratorVersion
    • muscovite.model.hash = kModelHash

Implementation Phases

Phase Features Iteration
Phase 1: Foundation Correlation ID interceptor, CorrelationContext, domain_events column, log enrichment, model_version.hpp, VersionService v1.51.0
Phase 2: OTEL Core TELEMETRY capability, OTEL SDK init, RPC spans, gRPC OTEL interceptor, OTLP exporter config in .mus v1.52.0
Phase 3: Deep Instrumentation Database spans, outbox spans, metrics, OTEL log bridge, resource attributes v1.53.0

DSL Integration

Project DSL (.mus) — TELEMETRY Block

CAPABILITIES {
    TELEMETRY;
}

TELEMETRY {
    OTLP_ENDPOINT "http://otel-collector:4317";
    SERVICE_NAME "fhir-health";
    SAMPLING_RATE 1.0;
    EXPORT_INTERVAL "30s";
}

Parsed by project-parser.rkt, stored in project-config IR struct, consumed by config.rkt and server-main.rkt generators.

Generated Config (server.ini)

[telemetry]
enabled = true
otlp_endpoint = http://otel-collector:4317
service_name = fhir-health
sampling_rate = 1.0
export_interval_ms = 30000

Overridable at runtime via MUSCOVITE_TELEMETRY_OTLP_ENDPOINT env var (following existing Config precedence: CLI > env > INI > defaults).

Dependency

# conanfile.py addition (Phase 2 only)
"opentelemetry-cpp/1.14.2",  # Apache-2.0 — OTEL traces, metrics, logs, OTLP exporter

When TELEMETRY capability is not enabled, opentelemetry-cpp is not linked (CMake conditional). Zero overhead for projects without telemetry.

Positive Consequences

  • DSL amplification extends to operations: .ddd + TELEMETRY → fully instrumented, production-observable services
  • Zero-effort observability: Domain modelers declare capability, get traces + metrics + correlation
  • Backend-agnostic: OTLP works with any OTEL Collector backend
  • Incremental adoption: Correlation ID works standalone (Phase 1), OTEL adds depth (Phase 2–3)
  • Version provenance: Every service self-reports its model version — deployment verification is trivial

Negative Consequences

  • opentelemetry-cpp SDK is large (~50 headers) — may increase compile time for OTEL-enabled targets
  • OTLP gRPC exporter adds another gRPC connection per service instance
  • correlation_id column in domain_events requires a migration (additive, nullable initially)
  • Phase 2–3 scope is significant (est. 3–5 days per phase)

Pros and Cons of the Options

Option 1: OpenTelemetry C++ SDK (chosen)

  • Good, because Apache-2.0 license — dual-license compatible
  • Good, because single SDK for traces + metrics + logs
  • Good, because CNCF-graduated industry standard
  • Good, because OTLP exporter = backend-agnostic
  • Good, because gRPC interceptor pattern matches existing infrastructure
  • Bad, because large dependency (~50 headers)
  • Bad, because OTEL C++ SDK maturity is younger than Go/Java equivalents

Option 2: Prometheus + Jaeger separately

  • Good, because mature, well-understood libraries
  • Bad, because two separate dependencies and integration patterns
  • Bad, because Prometheus C++ client is GPL-2.0 — incompatible with commercial license
  • Bad, because no unified correlation between metrics and traces

Option 3: Enhance existing spdlog + in-memory metrics

  • Good, because zero new dependencies
  • Good, because minimal change surface
  • Bad, because no distributed tracing (fundamental gap remains)
  • Bad, because no metrics export (data collected but invisible)
  • Bad, because no industry-standard integration

AI Agent Guidance

When This Decision Applies

  • When generating server_main.cpp — include correlation interceptor and (optionally) OTEL setup
  • When generating ServiceImpl classes — propagate correlation context, emit spans (if OTEL)
  • When generating domain_events tables — include correlation_id UUID column
  • When generating NATS outbox publishing — include correlation header
  • When generating project CMake — conditionally link opentelemetry-cpp
  • When generating server config — include [telemetry] section
  • When modifying project-parser.rkt — parse TELEMETRY block

When This Decision Does NOT Apply

  • Framework-internal logging (keep spdlog as-is for Layer 1)
  • Racket test infrastructure (no OTEL in test runners)
  • Build-time tooling (CMake, Conan configuration scripts)

Enforcement

  • Correlation interceptor generated for ALL gRPC-enabled projects (not capability-gated)
  • OTEL instrumentation generated only when TELEMETRY capability is declared
  • model_version.hpp generated for ALL projects (always useful)
  • VersionService generated for ALL gRPC-enabled projects
