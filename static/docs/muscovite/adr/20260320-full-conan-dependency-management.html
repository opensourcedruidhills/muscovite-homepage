ADR: Full Conan Dependency Management — All C++ Dependencies via Conan

ADR: Full Conan Dependency Management — All C++ Dependencies via Conan

  • Status: accepted
  • Date: 2026-03-20
  • Deciders: Johannes Lochmann

Context

Muscovite's C++ dependency stack is split across two sources:

  1. System packages (zypper/apt): libpqxx, spdlog, fmt, boost, gRPC, protobuf, libcurl
  2. Conan: jwt-cpp, nlohmann_json, libgit2, cli11, cnats, opentelemetry-cpp

This split creates compounding problems:

  1. Version freeze: System package repos (SUSE, Ubuntu) lag 6–18 months behind upstream. libpqxx is stuck at 7.10.3 (ConanCenter has 8.0.0). spdlog at 1.15.3 (latest: 1.17.0). There is no mechanism to advance without either upgrading the OS or building from source.

  2. Non-reproducible builds: Different developers and CI environments have different system package versions. A successful build on one machine can fail on another. There is no lockfile for system dependencies.

  3. ABI conflict risk: The OTel SDK (FTR-480) is conan-managed and transitively pulls in a conan-managed gRPC. The framework simultaneously links against the system gRPC. These are built with different flags, ABIs, and protobuf versions — an ABI mismatch waiting to happen.

  4. Platform dependency: Every new developer or CI runner must install the correct system packages via zypper/apt before any build works. The list is documented but not enforced.

  5. Lost worktree benefit: Git worktrees (Rule 26) give per-FTR isolated build environments, but system packages are global and unversioned. A dep upgrade on the host affects all worktrees simultaneously.

The root cause: mixing two dependency management systems with incompatible guarantees.

Decision

Move all C++ dependencies to Conan. Eliminate system package dependencies entirely.

Concretely:

  • Add grpc/1.78.1, libpqxx/8.0.0, spdlog/1.17.0, fmt/12.1.0, boost/1.90.0, openssl/3.6.1 to conanfile.py
  • Rewrite cmake/Dependencies.cmake to use conan find_package() targets for all deps, removing all find_package(spdlog), find_package(libpqxx), pkg_check_modules(GRPC) calls that rely on system packages
  • Migrate framework/dba/ from libpqxx 7 API to libpqxx 8 API
  • Keep the conan profiles (conan_profiles/debug, release, relwithdebinfo) — they are the correct mechanism for reproducible builds and are version-controlled with the repo
  • Establish a policy: conan is the single source of truth for C++ deps; zypper/apt installs only build tooling (gcc, cmake, ninja, mold) and Qt

Qt is explicitly excluded from this decision. Qt is managed via Qt's own installer (aqtinstall in CI, Qt Online Installer for local development). Qt Company's installer always has the latest release; ConanCenter lags by months. Qt builds from source take 4–8 hours and require matching tool versions (moc, uic, rcc) that the installer provides automatically.

Rationale

  1. Version sovereignty: Conan gives precise, version-controlled, reproducible dep selection. Upgrading libpqxx is a one-line change in conanfile.py, not a system package upgrade.

  2. Cold cache build once, all worktrees benefit: Conan's ~/.conan2/p/ cache is global. The first conan install --build=missing on a machine pays the build cost. Every subsequent worktree hits the cache. New FTR worktrees come up in seconds, not minutes.

  3. Eliminates the system/conan gRPC split: Once gRPC is conan-managed, OTel and the framework share a single gRPC binary. No ABI conflict is possible.

  4. "Living on the edge": Conan makes it trivial to track latest stable versions. The target policy is: bump deps to latest at each minor release unless a known breakage exists.

  5. CI simplification: Docker/container CI only needs to install gcc cmake ninja mold (and aqtinstall for Qt builds). All C++ deps are fetched and built by conan.

  6. libpqxx 8.0 enables modern API: libpqxx 8 drops deprecated pqxx::work in favour of pqxx::transaction<>, adds prepared-statement caching, and improves error reporting. The framework is already on C++23 — there is no reason to stay on the 7.x API.

Consequences

Benefits

  • Fully reproducible builds on any machine with gcc + conan
  • libpqxx, spdlog, gRPC always at latest stable
  • OTel ABI conflict permanently resolved
  • Git worktree model remains fully viable — cold cache built once per machine
  • Simplified CI setup (no system dep list to maintain)

Costs

  • One-time migration work across three FTRs (FTR-488, FTR-489, FTR-490)
  • First conan install --build=missing on a clean machine is slow (~30–60 min for grpc + abseil + protobuf + libpqxx). Subsequent installs use the cache.
  • libpqxx 8 API migration requires touching framework/dba/ and reviewing generated code

Not Changed

  • Qt management (aqtinstall / Qt Online Installer)
  • Conan profiles in conan_profiles/ — these remain version-controlled
  • CMakePresets.json structure — only the CMAKE_PREFIX_PATH for Qt changes
  • Build presets (debug, release, asan, tsan, coverage) — unaffected

System Packages Policy (post-migration)

Required on any build host:

gcc g++ cmake ninja-build mold   # build tooling
conan                             # dep management
aqtinstall + Qt 6.x              # Qt only

No C++ library headers from system packages.

Implementation

See FTR-488 (gRPC + protobuf via conan), FTR-489 (libpqxx 8.0 migration), FTR-490 (spdlog/fmt/boost + dep bumps) for the implementation breakdown.
