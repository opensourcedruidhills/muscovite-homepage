ADR: clang-tidy as C++ Pattern Enforcement Mechanism

ADR: clang-tidy as C++ Pattern Enforcement Mechanism

  • Date: 2026-03-05
  • Status: accepted
  • FTR: FTR-218

Context

doc/CPP-PATTERNS.md defines 20 sections of mandatory C++ style rules (severity-rated). Previously these were enforced only by documentation review and agent instructions. Agents and developers could introduce violations (e.g., m_ prefixed members, missing noexcept on move constructors, NULL macro usage) without any automated feedback.

Decision

Adopt clang-tidy as the mechanical enforcement layer for CPP-PATTERNS.md rules:

  1. .clang-tidy at repository root with check-to-pattern mapping in comments
  2. cmake/ClangTidy.cmake provides add_clang_tidy_target(target) for per-target control
  3. CMakePresets.json adds a clang-tidy configure+build preset
  4. StaticAnalysis.cmake MUSCOVITE_ENABLE_CLANG_TIDY=ON activates during build

Check → Pattern Mapping

clang-tidy Check CPP-PATTERNS Section FTR
readability-identifier-naming.PrivateMemberSuffix: '_' §7 Naming FTR-213
modernize-use-nullptr §2 nullptr FTR-209
performance-noexcept-move-constructor §4 Move Semantics FTR-203
modernize-use-nodiscard §6 [[nodiscard]] —
modernize-use-using §3 Type Aliases —
misc-const-correctness §10 const —

Integration Points

  • Local gate: agent-pr.sh runs cmake --build --preset clang-tidy before PR (FTR-219)
  • Release gate: release.sh verifies zero clang-tidy warnings (FTR-219)
  • On-demand: cmake --build --preset clang-tidy --target clang-tidy-check

Consequences

  • Positive: Mechanical enforcement catches pattern violations before review
  • Positive: .clang-tidy serves as living documentation of enforced rules
  • Negative: Build time increases ~30% when clang-tidy preset is active
  • Mitigation: clang-tidy preset is opt-in (not enabled in debug/release presets)
