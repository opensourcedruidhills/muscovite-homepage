ADR: Muscovite Embedded Prompts (MEP) for Platform-Agnostic AI Orchestration

ADR: Muscovite Embedded Prompts (MEP) for Platform-Agnostic AI Orchestration

  • Status: Accepted
  • Deciders: jolo, Muscovite AI
  • Date: 2026-02-05
  • Refers to: (Domain-Specific AI Interactions)

1. Context and Problem Statement

Muscovite aims to be a "Single Source of Truth" across a full application stack (PostgreSQL, C++, gRPC, Qt). While structural modeling (entities, relationships) is well-defined, complex behavioral logic like validations and long-running sagas often requires manual C++ or SQL coding and drifts between platforms.

As the project shifts toward a workflow where 95% of the code is generated via AI agents, we need a way to embed business intent into the DSL that is optimal for an AI agent to interpret and implement across multiple, divergent platforms without causing "model drift".

2. Decision Drivers

  • Platform Agnosticism: Logic must be defined once and be applicable to C++ domain services, PostgreSQL constraints, and Qt frontend validators.
  • Offline Determinism: The build system (CI/CD) must work without LLM API keys. Generation must be deterministic.
  • Verification: AI-generated code must be verifiable against the original intent to prevent "ghost in the machine" bugs.
  • Maintainability: The embedded prompt must be a first-class citizen in the DSL, versioned alongside the schema.

3. Proposed Architecture: The INSTRUCT Block

The DSL will be extended with a structured INSTRUCT block. This block does not contain code, but a formalized contract of intent.

A. Syntax Specification

ENTITY Patient IDENTIFIED_BY id {
    -- ... structural fields ...

    INSTRUCT VALIDATION Eligibility {
        INTENT: "A patient is eligible for trial enrollment only if they are over 18 
                and have a 'Stable' clinical status."
        INPUTS: { self.birth_date, self.status_id }
        OUTPUT: VAL700_INELIGIBLE
        STRATEGY: DETERMINISTIC
    }
}

B. Two-Phase Generation (Offline Support)

To ensure builds are fast and deterministic, generation is split:

  1. Enrichment Phase (racket enrich):

    • Detects changed INSTRUCT blocks (via SHA-256 hash).
    • Calls LLM API with type context + INTENT.
    • Generates Artifacts: Implementation (C++/SQL/Qt) AND Test Vectors (JSON).
    • Writes to dslgen/instruct-cache/<hash>.json.
    • This file is committed to Git.

  2. Generation Phase (racket generate):

    • Reads dslgen/instruct-cache/.
    • Emits code into target files (.hpp, .sql, .cpp).
    • Requirements: Runs offline, zero latency.

C. Platform-Agnostic Generation Logic

The Muscovite generators will transform the cached artifacts into target-specific code:

Target Generator Action
C++ Emits a bool checkEligibility() function in the Domain Service.
PostgreSQL Emits a CHECK constraint or trigger function to enforce logic at the data layer.
Qt (UI) Generates an AsyncValidator that provides real-time feedback in the PatientEditDialog.
Verification Generates a Catch2 test suite that loads the Test Vectors and asserts that C++ and SQL implementations match the vector's expected output.

4. Mitigating Non-Determinism & Drift

To ensure AI-generated code remains stable and platform-agnostic:

  1. Instruction Anchoring: The generator prompts the LLM with the full ENTITY and VALUE_OBJECT definitions to ensure type correctness.
  2. Logic Hashing: A SHA-256 hash of the INSTRUCT block tracks changes. Regeneration only occurs if the block's text or inputs change.
  3. Universal Test Vectors: The Enrichment Phase must produce a set of input/output pairs (e.g., {age: 17, status: 'Stable'} -> False, {age: 18, status: 'Stable'} -> True).
  4. Cross-Platform Verification: The build fails if the Generated C++ or SQL implementation disagrees with the Test Vectors.

5. Consequences

Positive

  • Intent as Truth: The "What" is stored in the DSL; the "How" is handled by the platform-specific generators.
  • Zero-Drift Validations: Ensures that a validation rule in the UI is identical to the rule in the database.
  • Offline CI/CD: Committing the instruct-cache ensures reproducible builds without API dependencies.

Negative

  • Context Overhead: Prompting for every small logic change increases token usage during the enrichment phase.
  • Cache Management: The instruct-cache must be managed (pruning old hashes) to prevent repository bloat.

6. Implementation Plan

  • Phase 1: Update dslgen/parser/lexer.rkt and parser.rkt to support INSTRUCT keyword and struct.
  • Phase 2: Create dslgen/enrich.rkt (Enrichment Tool) to handle LLM orchestration and cache management.
  • Phase 3: Update dslgen/generators/ to read from Cache and emit code.
