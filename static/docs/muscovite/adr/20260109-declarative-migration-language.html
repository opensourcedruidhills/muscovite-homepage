Declarative Migration Language

Declarative Migration Language

  • Status: accepted
  • Deciders: Project maintainers
  • Date: 2026-01-09
  • Tags: migration, ir, declarative, schema-evolution
  • Related: Semantic Model Delta, Migration Safety Policies

Technical Story: Define a human-reviewable, machine-executable migration IR that represents schema changes before SQL generation.

Context and Problem Statement

Migration operations need a reviewable, declarative representation before being rendered to SQL. Currently:

  1. No intermediate representation: Changes go directly from DSL diff to SQL
  2. No review step: Cannot inspect or modify migrations before execution
  3. No serialization: Migrations cannot be saved, versioned, or shared
  4. No backend abstraction: Tied directly to PostgreSQL SQL syntax

A declarative migration language enables:

  • Human review of proposed changes
  • Policy-based approval workflow
  • Backend-agnostic representation
  • Audit trail of schema evolution

Decision Drivers

  • UUID-based operation references (from ADR 0002/0003)
  • Serializable and editable (YAML, JSON, or S-expressions)
  • Complete representation of adds/removes/renames/changes
  • Supports policy-based execution from ADR 0005
  • Human-readable with clear operation semantics
  • Machine-executable for automation

Considered Options

  • Inline SQL generation: Rejected - not reviewable, not backend-agnostic
  • Text diff scripts: Rejected - ambiguous, hard to parse
  • JSON migration format: Considered - good for tooling, verbose for humans
  • Racket structs with S-expression serialization: Accepted - consistent with existing codebase
  • YAML migration format: Considered - human-friendly but adds dependency

Decision Outcome

Chosen option: "Racket structs with S-expression serialization", because it maintains consistency with the existing codebase and enables seamless integration with generators. Additionally provide optional JSON export for external tooling.

Positive Consequences

  • Human-reviewable migration plans
  • Supports automated policy checks (ADR 0005)
  • Compatible with multiple backends (PostgreSQL, MySQL, SQLite)
  • Enables approval workflow for destructive changes
  • Audit trail via git-versioned migration files

Negative Consequences

  • Requires mapping layer to each target system
  • S-expressions less familiar than JSON/YAML for some users
  • Additional serialization/deserialization code

Implementation

Phase 1: Migration Operation Structs

Create src/dslgen/migration/migration-types.rkt:

#lang racket/base

(provide (all-defined-out))

;; Base operation with common metadata
(struct migration-op 
  (uuid           ;; UUID of affected element
   safety         ;; 'safe | 'unsafe | 'destructive | 'review
   description    ;; Human-readable description
   requires)      ;; List of prerequisite operation UUIDs
  #:transparent)

;; ============================================================================
;; Context Operations
;; ============================================================================

(struct add-context-op migration-op
  (context-name
   context-uuid)
  #:transparent)

(struct remove-context-op migration-op
  (context-name
   cascade?)      ;; Whether to drop all contained objects
  #:transparent)

;; ============================================================================
;; Entity Operations
;; ============================================================================

(struct add-entity-op migration-op
  (context-name
   entity-name
   entity-uuid
   fields         ;; List of field definitions
   pk-field
   parent-uuid)   ;; For TPT inheritance, or #f
  #:transparent)

(struct remove-entity-op migration-op
  (context-name
   entity-name
   cascade?)
  #:transparent)

(struct rename-entity-op migration-op
  (context-name
   old-name
   new-name)
  #:transparent)

(struct move-entity-op migration-op
  (old-context
   new-context
   entity-name)
  #:transparent)

;; ============================================================================
;; Field Operations
;; ============================================================================

(struct add-field-op migration-op
  (entity-uuid
   field-name
   field-uuid
   field-type
   nullable?
   default-value) ;; SQL expression or #f
  #:transparent)

(struct remove-field-op migration-op
  (entity-uuid
   field-name)
  #:transparent)

(struct rename-field-op migration-op
  (entity-uuid
   old-name
   new-name)
  #:transparent)

(struct change-type-op migration-op
  (entity-uuid
   field-name
   old-type
   new-type
   conversion)    ;; SQL expression for USING clause, or #f
  #:transparent)

(struct change-nullable-op migration-op
  (entity-uuid
   field-name
   old-nullable?
   new-nullable?
   default-for-nulls) ;; Value to use for existing NULLs when adding NOT NULL
  #:transparent)

(struct change-default-op migration-op
  (entity-uuid
   field-name
   old-default
   new-default)
  #:transparent)

;; ============================================================================
;; Relationship Operations
;; ============================================================================

(struct add-relationship-op migration-op
  (source-entity-uuid
   target-entity-uuid
   rel-name
   rel-uuid
   cardinality    ;; 'one-to-one | 'one-to-many | 'many-to-many
   on-delete)     ;; 'cascade | 'set-null | 'restrict
  #:transparent)

(struct remove-relationship-op migration-op
  (source-entity-uuid
   rel-name)
  #:transparent)

(struct change-cardinality-op migration-op
  (rel-uuid
   old-cardinality
   new-cardinality)
  #:transparent)

;; ============================================================================
;; Index Operations
;; ============================================================================

(struct add-index-op migration-op
  (entity-uuid
   index-name
   columns
   unique?)
  #:transparent)

(struct remove-index-op migration-op
  (entity-uuid
   index-name)
  #:transparent)

Phase 2: Migration Plan Container

;; A migration plan is a sequence of operations with metadata
(struct migration-plan
  (id             ;; UUID for this migration
   name           ;; Human-readable name (e.g., "add-sample-type-field")
   description    ;; Longer description
   created-at     ;; ISO timestamp
   source-version ;; Hash of source IR
   target-version ;; Hash of target IR
   operations)    ;; List of migration-op
  #:transparent)

;; Create migration plan from diff result
(define (diff->migration-plan diff-ops source-ir target-ir)
  (migration-plan
   (make-uuid)
   (generate-migration-name diff-ops)
   (generate-migration-description diff-ops)
   (current-iso-timestamp)
   (ir-content-hash source-ir)
   (ir-content-hash target-ir)
   (sort-by-dependencies diff-ops)))

Phase 3: Serialization

;; Write migration plan to file
(define (write-migration-plan plan path)
  (call-with-output-file path
    (λ (out)
      (pretty-write (migration-plan->sexp plan) out))
    #:exists 'replace))

;; Read migration plan from file
(define (read-migration-plan path)
  (define sexp (call-with-input-file path read))
  (sexp->migration-plan sexp))

;; Convert to S-expression
(define (migration-plan->sexp plan)
  `(migration-plan
    (id ,(migration-plan-id plan))
    (name ,(migration-plan-name plan))
    (description ,(migration-plan-description plan))
    (created-at ,(migration-plan-created-at plan))
    (source-version ,(migration-plan-source-version plan))
    (target-version ,(migration-plan-target-version plan))
    (operations ,@(map migration-op->sexp (migration-plan-operations plan)))))

;; JSON export for external tools
(define (migration-plan->json plan)
  (hasheq
   'id (migration-plan-id plan)
   'name (migration-plan-name plan)
   'operations (map migration-op->json (migration-plan-operations plan))))

AI Agent Guidance

When This Decision Applies

  • When implementing migration generation pipeline
  • When creating migration review/approval tooling
  • When implementing migration execution

When This Decision Does NOT Apply

  • Direct SQL execution (that's ADR 0006)
  • Model parsing (that's ADR 0001-0002)
  • Runtime schema changes

Enforcement

  • All diff operations MUST be converted to migration-op structs
  • All migration-ops MUST include safety classification
  • Migration plans MUST be serializable and deserializable
  • Operations MUST include dependency ordering via requires field
