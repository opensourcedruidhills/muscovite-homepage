Adopt-On-Touch Contract Policy

Adopt-On-Touch Contract Policy

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-27
  • Tags: racket, contracts, quality, refactoring

Technical Story: FTR-513 — Contract-enforce parser, migration, and verification modules.

Context and Problem Statement

Racket's contract system provides runtime boundary checking between modules, catching type errors and invalid arguments at module boundaries. The contract system is opt-in: modules can use provide (untyped) or (provide (contract-out ...)) (typed).

The Muscovite DSL generator codebase (~50 modules) has inconsistent contract adoption. High-churn modules like generators/cpp/main.rkt and generators/postgres/main.rkt lack contracts, while isolated utility modules like shared/naming.rkt have them. Parser modules (parser/lexer.rkt, parser/parser.rkt), migration modules (migration/*.rkt), and verification modules (verification/*.rkt) lack contracts despite being imported by 10+ modules each.

This creates two problems:

  1. Type mismatches at module boundaries go undetected until deep in the call stack
  2. Refactoring is risky — changing a function signature in a widely-imported module requires manual inspection of all call sites

Decision Drivers

  • Parser, migration, and verification modules are stable APIs (low churn)
  • These modules are imported by 10+ call sites each
  • Contract violations caught at boundaries improve debugging clarity
  • Adding contracts to high-churn modules (generators) is low ROI (frequent rewrites)
  • Full-codebase contract retrofit is >200 function signatures (not viable in one PR)

Considered Options

  1. Adopt-on-touch — Add contracts when modifying modules with 10+ importers
  2. Full retrofit — Contract all modules in one pass
  3. High-churn exemption — Contract all modules except generators
  4. Status quo — Leave contracts opt-in with no policy

Decision Outcome

Chosen option: "Adopt-on-touch", because it balances incremental improvement with pragmatic scope control. FTR-513 demonstrates viability: 13 modules contracted in one PR, covering parser, migration, and verification boundaries.

Policy

When modifying any module imported by 10 or more other modules, add contract-out to its public exports in the same PR or commit. This includes:

  • Parser modules (parser/lexer.rkt, parser/parser.rkt)
  • Migration modules (migration/migration-types.rkt, migration/diff-model.rkt, migration/policy.rkt)
  • Verification modules (verification/verify-*.rkt)
  • Shared utilities (shared/naming.rkt, shared/expression-emitter.rkt)

Exemptions:

  • Generator main modules (generators/*/main.rkt) — high churn, low stability
  • Test modules — no external importers
  • Modules with <10 importers — opt-in contracts remain acceptable

Contract Design Patterns

  1. Structs: Use [struct foo ([field1 type1] [field2 type2])] for field contracts
  2. Multi-value returns: Use any as return type (values contracts are complex)
  3. Parameters (make-parameter): Keep as bare provide (cannot be contracted)
  4. Macros: Keep as bare provide (macros expand at compile time, not runtime)
  5. Re-exports (all-from-out): Keep as separate bare provide block
  6. Complex nested types: Use any/c to avoid circular module dependencies

Positive Consequences

  • Type mismatches caught at module boundaries, not deep in call stack
  • Self-documenting API — contract signatures serve as inline type hints
  • Incremental: no "big bang" retrofit required
  • Low risk: only touches modules during actual changes

Negative Consequences

  • Slight runtime overhead (negligible in code generation context)
  • Parser code becomes more verbose (contract declarations are 2-3× function count)
  • Requires discipline to remember the policy during refactoring

Pros and Cons of the Options

Option 2: Full retrofit

  • Good, because uniform contract coverage
  • Good, because frontloads all contract work
  • Bad, because >200 function signatures in one PR (high review burden)
  • Bad, because touches stable modules unnecessarily (churn without value)

Option 3: High-churn exemption

  • Good, because avoids contract churn in generators
  • Good, because focuses on stable APIs
  • Bad, because requires upfront decision of "high-churn" (subjective)
  • Bad, because no clear threshold for when to contract (arbitrary)

Option 4: Status quo

  • Good, because no process overhead
  • Bad, because type errors remain undetected until runtime
  • Bad, because FTR-513 demonstrates contracts catch real issues (e.g., bounded-context? vs any/c mismatches)

AI Agent Guidance

When This Decision Applies

  • When modifying any module with ≥10 importers (check with grep -rl "require.*module-name")
  • When adding a new widely-used utility module
  • When refactoring parser, migration, or verification modules

When This Decision Does NOT Apply

  • Generator main modules (generators/*/main.rkt) — exempt due to high churn
  • Modules with <10 importers — contracts remain opt-in
  • Test modules — no external contract boundary needed

Enforcement

  • Code review: changes to parser/migration/verification modules without contracts trigger review comment
  • Grep check: grep -rl "require.*<module>" dslgen | wc -l to count importers
  • FTR template: future FTRs touching contracted modules must update contracts
