Opportunity Solution Tree as Built-in Discovery Capability

Opportunity Solution Tree as Built-in Discovery Capability

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-16
  • Tags: capability, discovery, ost, product-discovery, opportunity-solution-tree

Technical Story: Teresa Torres's Continuous Discovery Habits (2021) provides a structured reasoning framework connecting measurable business outcomes to user needs to candidate solutions to validation experiments. Muscovite should offer this as a reusable capability so any consumer product can opt into discovery tracking without reinventing the domain model.

Context and Problem Statement

Muscovite's capability system (CAPABILITIES {} in project.mus) ships reusable infrastructure domains: security provides auth/authz entities, spatial provides PostGIS reference data. These capabilities auto-load framework-provided .ddd contexts so consumers get generated code across all 8 targets with zero modeling effort.

Product teams using Muscovite currently model what to build (entities, aggregates, services) and how it behaves (state machines, events, policies), but have no structured way to model why — the discovery chain from business outcomes through user needs to candidate solutions. Without this:

  1. Features lack traceability to validated user needs.
  2. Multiple solution candidates for the same problem cannot be compared.
  3. Production telemetry anomalies have no structured path back to discovery.
  4. Teams make solution-first decisions instead of need-first decisions.

The Opportunity Solution Tree (OST) model provides this missing layer. The question is how to deliver it in the Muscovite framework.

Decision Drivers

  • Reusability: Multiple consumer products (muscovite-agent-coordination, fhir-health, geosciences) would benefit from a shared discovery model.
  • Opt-in: Discovery tracking is not universally needed. Products that don't need it should pay zero cost.
  • Existing mechanism: The capability registry (dslgen/builtin/registry.rkt) already handles framework-provided contexts. No new infrastructure needed.
  • Generator coverage: Standard ENTITY + STATE_MACHINE constructs get full 8-target generation without generator changes.
  • Precedent: The security capability proves the pattern works.

Considered Options

Option A: No framework support — consumers model it themselves

Consumers write their own Outcome/Opportunity entities in their .ddd files.

  • Good, because zero framework work.
  • Bad, because every consumer reinvents the same model.
  • Bad, because no consistency across products.
  • Bad, because no framework-level documentation or best practices.

Option B: New DSL keywords — OUTCOME, OPPORTUNITY as grammar-level constructs

Add first-class grammar rules, IR structs, parser support, and specialized generator output (tree views, flow metric dashboards).

  • Good, because full semantic awareness at the DSL level.
  • Good, because generators can produce specialized output (OST tree queries).
  • Bad, because grammar changes are a strong commitment (Rule 7: grammar-first).
  • Bad, because multi-iteration effort (grammar + IR + parser + 8 generators).
  • Bad, because the OST model is opinionated — baking it into grammar couples the framework to one discovery methodology.

Option C: Built-in capability with framework-provided .ddd context

Register discovery as a capability. Ship dslgen/builtin/discovery/discovery.ddd containing Outcome and Opportunity entities with state machines, events, and policies. Consumers opt in via CAPABILITIES { discovery }.

  • Good, because uses existing capability infrastructure — proven pattern.
  • Good, because full 8-target generation with zero generator changes.
  • Good, because opt-in: one word in project.mus.
  • Good, because the .ddd file can evolve independently of the grammar.
  • Good, because promotion path to Option B exists if adoption proves the model.
  • Bad, because generators have no semantic awareness of OST (no specialized tree views or flow metric dashboards — those require consumer-side work).

Decision Outcome

Chosen option: Option C — Built-in discovery capability, because it delivers the OST domain model to all consumers via the proven capability mechanism with minimal framework changes and a clear promotion path.

The Four-Layer OST Model

Outcome (OTC) — measurable, time-bound business goal
  └── Opportunity (OPP) — user need, pain point, or desire
      └── Solution (FTR) — candidate feature (consumer's work items)
      └── Assumption Test (SPK) — cheapest validation experiment

What the Capability Provides

The discovery capability auto-loads a Discovery bounded context containing:

Entity Purpose State Machine
Outcome Measurable business goal (metric + target + date) Draft → Active → Achieved / Abandoned
Opportunity User need discovered from research/telemetry/retrospectives Identified → Validating → Validated → Addressed / Parked

Plus:

  • AnomalyDetected domain event (telemetry → discovery feedback loop)
  • AutoCreateOpportunity policy (anomaly auto-creates opportunity for human review)
  • CHECK constraints for controlled vocabularies (source, impact, confidence)

What Consumers Must Provide

  • Cross-context relationships from Opportunity to their work items (FTR, EPC, SPK) via FOREIGN CONTEXT declarations in their own .ddd files.
  • The telemetry pipeline that fires AnomalyDetected events (product-specific).
  • UI for OST tree visualization (the data model supports it; the layout is product-specific, potentially a .lay concern).

Consumer Opt-In

PROJECT MyProduct {
    CAPABILITIES {
        security
        discovery    -- Outcome + Opportunity entities, state machines, events
    }
    CONTEXTS {
        include "contexts/my-domain.ddd"
    }
}

Positive Consequences

  • Any Muscovite consumer gets a validated OST domain model with one keyword.
  • All 8 generator targets produce output: PostgreSQL tables, gRPC service, C++ repositories, Qt views, Sqitch migrations, CMake integration, documentation.
  • The model is battle-tested across consumers before any grammar promotion.
  • Closes the telemetry → discovery feedback loop when combined with the telemetry capability (ADR 20260302).

Negative Consequences

  • Generators produce generic CRUD for Outcome/Opportunity — no OST-specific tree queries or flow metric dashboards without additional generator work.
  • The discovery.ddd model embeds Teresa Torres's methodology. Products using different discovery frameworks would need to ignore or extend it.
  • Cross-context relationship wiring remains consumer responsibility.

Future: Promotion to Grammar-Level (Option B)

If multiple consumers adopt the discovery capability and demand specialized generator output (e.g., OST tree queries in PostgreSQL, tree visualization in QML, flow metric dashboards), the model can be promoted to grammar-level constructs in a future iteration:

  • OUTCOME and OPPORTUNITY as first-class keywords
  • Specialized IR structs with semantic meaning
  • Generator-produced tree queries, dashboards, and documentation sections

This promotion would follow the grammar-first mandate (ADR 20260211) and require its own ADR. The capability .ddd would then become the reference implementation.

AI Agent Guidance

When This Decision Applies

  • When a consumer product needs structured product discovery tracking (Outcomes, Opportunities, assumption testing).
  • When adding the discovery capability to a project.mus file.
  • When modeling cross-context relationships between Opportunity and consumer work items (Feature, Epic, Spike).

When This Decision Does NOT Apply

  • When modeling domain-specific entities unrelated to product discovery.
  • When the consumer uses a different discovery methodology and prefers to model their own entities from scratch (Option A remains valid).
  • When discussing grammar-level OUTCOME/OPPORTUNITY keywords (that's a future decision, not this one).

Enforcement

  • Capability registration enforced by dslgen/builtin/registry.rkt.
  • Grammar rule enforced by PEG grammar CapabilityName production.
  • Model correctness enforced by parser and integration tests.
