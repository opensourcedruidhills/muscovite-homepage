Evaluate qmltc (QML Type Compiler) for AOT Compilation

Evaluate qmltc (QML Type Compiler) for AOT Compilation

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04
  • Tags: qt, qml, performance, aot

Technical Story: FTR-529

Context and Problem Statement

Qt 6.11 ships qmltc (QML Type Compiler) which compiles QML to C++ at build time, enabling ahead-of-time (AOT) compilation. This eliminates JIT overhead for QML type instantiation and property binding evaluation. Should Muscovite adopt qmltc for generated QML modules?

Decision Drivers

  • Generated QML is fully under our control (no user-authored QML in generated modules)
  • All QML types are registered via QML_ELEMENT macro (prerequisite for qmltc)
  • all_qmltyperegistrations aggregate target now exists (FTR-524)
  • .qmllint.ini configuration ensures type-safe QML (FTR-523)
  • No dynamic setContextProperty usage (FTR-527 confirmed already clean)

Considered Options

  1. Adopt qmltc now — add qt_add_qml_module(... ENABLE_TYPE_COMPILER) to generated CMake
  2. Defer to Qt 7 — qmltc is still marked as tech preview in 6.11; wait for stable API
  3. Do not adopt — JIT performance is sufficient for CRUD/admin UIs

Decision Outcome

Chosen option: "Defer to Qt 7", because:

  1. Tech preview status: qmltc in Qt 6.11 does not support all QML constructs. Known limitations include dynamic property lookups, certain JavaScript expressions, and Loader/Repeater with delegate factories. Our generated QML uses Repeater in list views and dynamic viewModel property access.

  2. Build time tradeoff: qmltc adds a compilation step per QML file. For projects with 50+ generated QML files (typical for a bounded context with 10 entities), this adds ~30s to incremental builds. The runtime benefit (sub-millisecond type instantiation) is not meaningful for admin/CRUD UIs where network latency dominates.

  3. Prerequisite infrastructure is in place: FTR-523 (qmllint), FTR-524 (all_qmltyperegistrations), and FTR-527 (no setContextProperty) satisfy all qmltc prerequisites. When Qt 7 promotes qmltc to stable, adoption requires only a single-line CMake change: ENABLE_TYPE_COMPILER in qt_add_qml_module().

Consequences

  • Good: No build time regression. No risk from tech-preview API changes.
  • Good: Infrastructure is ready — future adoption is a one-line change.
  • Bad: No AOT performance benefit for now (acceptable for admin UIs).

When to Revisit

  • Qt 7.0 release removes tech-preview label from qmltc
  • A consumer project reports QML instantiation as a measured performance bottleneck
  • qmltc gains full support for Repeater delegates and dynamic property access
