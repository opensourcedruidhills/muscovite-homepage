CQRS-Ready Architecture

CQRS-Ready Architecture

  • Status: accepted
  • Date: 2026-01-20
  • Tags: cqrs, architecture, event-sourcing, scalability

Context and Problem Statement

Domain-Driven Design often benefits from Command Query Responsibility Segregation (CQRS), where write and read paths are separated. This enables:

  • Independent scaling of read vs write workloads
  • Optimized read models (denormalized views, caches)
  • Event-driven projections for complex queries
  • Eventually consistent read replicas

However, full CQRS adds significant complexity. The question is: how do we prepare for CQRS without implementing full event sourcing now?

Decision Drivers

  • Future scalability: Read-heavy workloads may need separate optimization
  • Simplicity now: Full CQRS/event-sourcing is overkill for current needs
  • Architecture clarity: Separate command/query paths improve code organization
  • Incremental adoption: Enable CQRS features without rewriting

Considered Options

  1. Traditional CRUD: Single repository for all operations
  2. Full CQRS with event sourcing: Event store as source of truth
  3. CQRS-ready architecture: Separate paths, same datasource (chosen)

Decision Outcome

Chosen option: "CQRS-ready architecture", because it provides architectural separation without the complexity of event sourcing. The same PostgreSQL database serves both command and query paths now, but the architecture allows future separation.

Architecture Layers

flowchart TD
    GS["gRPC Service"] --> WP["Write Path\n(Commands)"]
    GS --> RP["Read Path\n(Queries)"]

    WP --> CS["CommandService\ncreate, update, delete\n+ emit event"]
    RP --> QS["QueryService\nfind_by_id, find_all\nfind_by_spec, find_by_criteria"]

    CS --> CR["CommandRepo\npqxx::transaction<>"]
    QS --> QR["QueryRepo\npqxx::nontransaction"]

    CR --> PG["PostgreSQL\n(Same Database)"]
    QR --> PG
    PG --- DE["domain_events table\nEvents stored inline"]

Future Evolution:

flowchart LR
    WDB["Write DB\n(Primary)"] -->|"Events (Outbox)"| RDB["Read DB\n(Replica)"]

Event Emission Strategy

Current (Phase 1): Inline event storage within command transaction

Patient PatientCommandService::create(const CreatePatientRequest& req) {
    return execute([&](pqxx::work& txn) {
        // 1. Insert entity
        auto patient = patient_repo_.insert(txn, req);
        
        // 2. Emit event (same transaction)
        event_repo_.emit(txn, DomainEvent{
            .event_type = "PatientCreated",
            .aggregate_type = "Patient",
            .aggregate_id = patient.id,
            .payload = serialize(patient)
        });
        
        return patient;
    });
}

Future (Phase 2): Outbox pattern for reliable event publishing

// Same transaction writes to outbox table
// Background publisher reads outbox and publishes to message broker
// Consumers update read models / projections

Future (Phase 3): Full CQRS with read replicas

// CommandService → Write DB (primary)
// QueryService → Read DB (replica) or materialized views

Implementation Details

Transaction Types: | Path | Transaction | Reason | |------|-------------|--------| | Command | pqxx::work | Full ACID for writes | | Query | pqxx::nontransaction | No locks, read-only |

Repository Separation:

// CommandRepository: write operations only
class PatientCommandRepository {
public:
    Patient insert(pqxx::work& txn, const Patient& entity);
    Patient update(pqxx::work& txn, const Patient& entity);
    void remove(pqxx::work& txn, const boost::uuids::uuid& id);
};

// QueryRepository: read operations only
class PatientQueryRepository {
public:
    std::optional<Patient> find_by_id(pqxx::connection& conn, 
                                      const boost::uuids::uuid& id);
    std::vector<Patient> find_all(pqxx::connection& conn, 
                                  const PageRequest& page);
    std::vector<Patient> find_active(pqxx::connection& conn);
};

gRPC Routing:

// Write RPCs → CommandService
grpc::Status Create(ctx, req, resp) {
    auto result = command_service_.create(req);
    // ...
}

// Read RPCs → QueryService (or direct QueryRepository)
grpc::Status Get(ctx, req, resp) {
    auto result = query_service_.find_by_id(req.id());
    // ...
}

Positive Consequences

  • Clear separation of concerns (commands vs queries)
  • Easy to understand: writes go one path, reads go another
  • Future-proof: can split databases without architectural rewrite
  • Event infrastructure ready: domain_events table populated
  • Optimized reads: can use nontransaction for better performance

Negative Consequences

  • More generated code (two repositories per aggregate)
  • Events stored but not yet consumed (no projections)
    • Roadmap: Add projection handlers
  • Same database for now (no actual read scaling)
    • Roadmap: Add read replica support

Roadmap Items Enabled

  1. Outbox Pattern (v0.8.0): Reliable event publishing to message brokers — see 20260202-outbox-pattern-reliable-event-publishing.md
  2. Read Replicas (v0.9.0): QueryService points to replica
  3. Materialized Views (v0.9.0): Pre-computed read models
  4. Event Projections (v0.9.0): Build specialized read models from events
  5. Lookup Caching (v0.9.0+): In-memory cache for reference data

AI Agent Guidance

When This Decision Applies

  • When generating repository classes (split command/query)
  • When generating domain services (command vs query)
  • When implementing gRPC service methods
  • When adding event emission to commands

When This Decision Does NOT Apply

  • Schema generation (no CQRS impact)
  • Simple utility queries (can use QueryRepository directly)
  • Batch operations / data migrations

Enforcement

  • Repositories MUST be split into Command and Query variants
  • Commands MUST emit events to domain_events table
  • Queries MUST NOT modify data
  • gRPC write RPCs MUST route through CommandService
