Operation Semantic Versioning

Operation Semantic Versioning

  • Status: accepted
  • Deciders: Architecture Team
  • Date: 2026-01-16
  • Tags: migration, versioning, compatibility
  • Implementation: src/dslgen/migration/operation-versioning.rkt

Context and Problem Statement

Migration operations don't carry version metadata. If the operation schema changes (new fields, renamed fields, changed semantics), previously serialized diffs become incompatible.

Example problem:

// Old format (v1.0)
{"type": "add-entity", "uuid": "...", "name": "Patient"}

// New format (v1.1) - added schema-name
{"type": "add-entity", "uuid": "...", "name": "Patient", "schema-name": "patient"}

// New format (v1.2) - new operation type
{"type": "add-lookup-table", "uuid": "...", "name": "Gender", "schema-name": "basedata", ...}

A v1.1 consumer cannot process v1.2 LOOKUP_TABLE operations without understanding the new type.

Decision Drivers

  • Long-lived projects may have stored migration history
  • Tool upgrades should not break existing migrations
  • Clear upgrade path when operation format changes
  • Backward compatibility for diff consumers
  • LOOKUP_TABLE feature requires new operation types

Decision Outcome

Implemented version envelope pattern with upgrade functions.

Version Envelope

All serialized operations wrapped in version envelope:

{
  "version": "1.2.0",
  "generator": "muscovite-diff",
  "generated-at": "2026-01-16T10:00:00Z",
  "from-ref": "v1.0.0",
  "to-ref": "HEAD",
  "operations": [...]
}

LOOKUP_TABLE Operations (v1.2.0)

New operation types introduced for the LOOKUP_TABLE DSL feature:

;; Creating a lookup table with standard columns
(struct add-lookup-table-op
  (uuid           ; String: lookup table UUID
   context-uuid   ; String: containing context UUID
   name           ; Symbol: table name
   schema-name    ; Symbol: PostgreSQL schema name
   extra-fields   ; List of field: additional columns
   seed-path      ; String: path to seed file
   seed-format    ; Symbol: 'json or 'csv
   safety)        ; Symbol: 'safe
  #:transparent)

;; Seeding lookup table data
(struct seed-lookup-table-op
  (uuid           ; String: synthetic UUID
   lookup-uuid    ; String: lookup table UUID
   name           ; Symbol: table name
   schema-name    ; Symbol: PostgreSQL schema name
   seed-path      ; String: path to seed file
   seed-format    ; Symbol: 'json or 'csv
   safety)        ; Symbol: 'safe
  #:transparent)

Upgrade Functions

(define (upgrade-operations ops from-version to-version)
  (cond
    [(version=? from-version to-version) ops]
    [else
     ;; Apply upgrade chain
     (define upgraded-1.0->1.1
       (if (and (version<? from-version "1.1.0")
                (version>=? to-version "1.1.0"))
           (map upgrade-1.0-to-1.1 ops)
           ops))
     ;; v1.2 adds new op types - no upgrade needed for existing ops
     upgraded-1.0->1.1]))

Usage

(require "migration/operation-versioning.rkt")

;; Serialize operations with version metadata
(define json-str 
  (operations->versioned-json ops #:from-ref "v1.0.0" #:to-ref "HEAD"))

;; Deserialize with automatic version upgrade
(define upgraded-ops (versioned-json->operations json-str))

;; Check compatibility
(version-compatible? "1.2.0" "1.1.0")  ; #t - consumer can upgrade
(version-compatible? "1.0.0" "1.2.0")  ; #f - major mismatch

Consequences

Positive

  • Migration history can be persisted safely
  • Tool upgrades don't break existing migrations
  • Clear upgrade path documented in version history
  • LOOKUP_TABLE operations properly versioned

Negative

  • Adds serialization/deserialization overhead
  • Need to maintain upgrade functions for each version
  • Version number must be bumped for any operation schema change

AI Agent Guidance

  • Current version: 1.2.0
  • When adding new operation types, bump minor version
  • When changing operation struct fields, bump minor version with upgrade function
  • When changing compatibility rules, bump major version
  • Always include version envelope when persisting migrations
