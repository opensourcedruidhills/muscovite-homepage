Domain/Subdomain Hierarchy with User-Definable Classification

Domain/Subdomain Hierarchy with User-Definable Classification

  • Status: draft
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-31
  • Tags: dsl, ddd, strategic, domain, classification, grammar, ir
  • Supersedes: none

Technical Story: RFC-001 Extension 1 — add strategic DDD domain hierarchy to the Muscovite DSL.

Context and Problem Statement

Muscovite has CONTEXT as a flat construct. There is no way to express that contexts belong to a domain, that domains have subdomains, or that some domains are strategically more important than others (core vs. supporting vs. generic). During brownfield extraction, AI agents discover bounded contexts but cannot classify them — a classification that drives prioritization, resource allocation, and documentation scoping.

Evans' strategic DDD vocabulary (domains, subdomains, core/supporting/generic classification) is missing from the DSL entirely. This forces strategic reasoning into external documents that drift from the model.

Decision Drivers

  • Brownfield modernization use case requires domain classification before entity modeling
  • Classification vocabularies differ per engagement (healthcare vs. finance vs. public sector)
  • Must be purely additive — existing models without DOMAIN/SUBDOMAIN continue unchanged
  • Grammar-first mandate (ADR 20260211) — PEG grammar changes before parser/IR

Considered Options

Option 1: Hardcoded Core/Supporting/Generic enum

Built-in CLASSIFICATION enum with Evans' three values.

  • Good, because simple and immediately recognizable
  • Bad, because clients need custom vocabularies (e.g., "revenue-generating" vs. "cost-center")
  • Bad, because adding new classifications requires grammar changes

Option 2: User-definable CLASSIFICATION_SCHEME (chosen)

A CLASSIFICATION_SCHEME in the .mus project file defines the vocabulary. DOMAIN and SUBDOMAIN reference scheme entries.

  • Good, because each engagement defines its own strategic taxonomy
  • Good, because adding classifications requires no grammar or tool changes
  • Good, because the scheme mechanism is reusable (same pattern as COMPLIANCE_SCHEME)
  • Bad, because slightly more complex than a fixed enum

Option 3: Free-text tags on contexts

CONTEXT Foo @tag "core" @tag "clinical" — unstructured metadata.

  • Good, because maximally flexible
  • Bad, because no validation, no scheme, no consistency across the model
  • Bad, because generators cannot reliably drive behavior from unstructured tags

Decision Outcome

Chosen option: "User-definable CLASSIFICATION_SCHEME", because it balances expressiveness with structure. The scheme pattern is reused across four extensions (classification, compliance, quality, governance), amortizing the grammar investment.

DSL Syntax

.mus project file — scheme definition

CLASSIFICATION_SCHEME DomainClassification {
    CORE_DOMAIN
        DOC "Differentiating business capability — maximum investment";
    SUPPORTING_DOMAIN
        DOC "Necessary but not differentiating — build or buy";
    GENERIC_SUBDOMAIN
        DOC "Commodity capability — buy or open-source";
}

.ddd domain model — usage

DOMAIN Healthcare @uuid "..." CLASSIFICATION DomainClassification:CORE_DOMAIN {

    SUBDOMAIN ClinicalCare @uuid "..."
        CLASSIFICATION DomainClassification:CORE_DOMAIN {

        SUBDOMAIN EmergencyMedicine @uuid "..."
            CLASSIFICATION DomainClassification:CORE_DOMAIN
            DOC "Time-critical patient care workflows";
    }

    SUBDOMAIN Billing @uuid "..."
        CLASSIFICATION DomainClassification:SUPPORTING_DOMAIN
        DOC "Insurance claims and payment processing";
}

Context implements clause

CONTEXT EmergencyDepartment @uuid "..."
    IMPLEMENTS EmergencyMedicine
{ ... }

DAG Rules

  • Domains can contain zero or more subdomains; nesting is arbitrarily deep
  • A context IMPLEMENTS exactly one subdomain (or domain, if no subdomains exist)
  • Multiple contexts can implement the same subdomain
  • The graph must be acyclic — no circular containment
  • Every domain/subdomain carries a @uuid for semantic diff tracking
  • Classification is optional on each node — inherits from parent if omitted

PEG Grammar Additions

Domain             <- 'DOMAIN' IDENTIFIER UuidAnnotation
                      ClassificationClause? '{' DomainBody '}' DocClause?
DomainBody         <- (Subdomain / DocClause)*
Subdomain          <- 'SUBDOMAIN' IDENTIFIER UuidAnnotation
                      ClassificationClause? ('{' DomainBody '}')? DocClause?
ClassificationClause <- 'CLASSIFICATION' IDENTIFIER ':' IDENTIFIER
ImplementsClause   <- 'IMPLEMENTS' IDENTIFIER

IR Implications

  • New IR node types: Domain, Subdomain with parent references (UUIDs)
  • Context gains an implements field (UUID reference to domain/subdomain)
  • Classification stored as scheme_name:tag_name string pair
  • Protobuf IR schema adds DomainNode message with repeated DomainNode children

Generator Implications

  • Documentation: domain hierarchy diagram, classification legend, per-domain scoped reports. Auditor docs show only CORE_DOMAIN contexts. C-level docs show the full taxonomy.
  • Heatmap: classification-colored context maps. Core = red (high attention), generic = grey (commodity).

Positive Consequences

  • Strategic DDD vocabulary formalized in the model
  • AI extraction agents can classify discovered contexts using the scheme
  • Documentation generators produce domain-scoped reports automatically

Negative Consequences

  • Grammar complexity increases (new top-level constructs)
  • DAG cycle detection required in validation

AI Agent Guidance

When This Decision Applies

  • When adding domain hierarchy or classification constructs to the DSL
  • When generating documentation that needs strategic domain context
  • When building brownfield extraction pipelines that classify bounded contexts

When This Decision Does NOT Apply

  • Greenfield projects with a single bounded context (DOMAIN is optional)
  • Tactical DDD constructs (entities, aggregates, value objects) — unchanged

Enforcement

  • PEG grammar rules for DOMAIN/SUBDOMAIN
  • Parser validation: DAG cycle detection, IMPLEMENTS target resolution
  • Classification scheme validation: referenced tags must exist in the scheme
