Use Racket for DSL Code Generation

Use Racket for DSL Code Generation

  • Status: accepted
  • Deciders: Project founders
  • Date: 2024-01-15
  • Tags: dsl, codegen, language

Context and Problem Statement

Muscovite needs a language to implement its Domain-Specific Language (DSL) parser and multi-target code generators. The language must handle complex text transformations, support recursive descent parsing, and generate code for PostgreSQL, C++, gRPC, and Qt/QML targets.

Decision Drivers

  • Excellent support for language-oriented programming and DSL creation
  • Powerful macro system for code generation templates
  • Strong pattern matching for parsing
  • Interactive REPL for rapid development
  • Functional programming paradigm fits transformation pipelines
  • No compilation step needed during development

Considered Options

  • Racket (Lisp dialect designed for language creation)
  • Python with PLY/ANTLR
  • TypeScript with tree-sitter
  • Haskell with Parsec

Decision Outcome

Chosen option: "Racket", because it was designed specifically for creating languages and DSLs, provides the most expressive code generation capabilities through its macro system, and offers an excellent REPL-driven development workflow.

Positive Consequences

  • Very concise parser implementation (~500 lines for full DDD grammar)
  • Pattern matching makes AST transformations readable
  • REPL allows instant testing of parser changes
  • Single language for parsing, validation, and all generators
  • Educational: demonstrates language-oriented programming

Negative Consequences

  • Smaller talent pool compared to Python/TypeScript
  • IDE support less mature than mainstream languages
  • Learning curve for Lisp syntax
  • Deployment requires Racket runtime

Pros and Cons of the Options

Racket

  • Good, because designed for language creation (Language-Oriented Programming)
  • Good, because powerful macro system for code templates
  • Good, because excellent pattern matching with match
  • Good, because interactive REPL development
  • Bad, because smaller community and talent pool
  • Bad, because unfamiliar syntax for most developers

Python with PLY/ANTLR

  • Good, because large community and easy hiring
  • Good, because extensive library ecosystem
  • Good, because familiar syntax
  • Bad, because grammar definitions separate from processing code
  • Bad, because less elegant pattern matching
  • Bad, because string-heavy code generation is verbose

TypeScript with tree-sitter

  • Good, because type safety and IDE support
  • Good, because familiar syntax for web developers
  • Good, because tree-sitter is fast and incremental
  • Bad, because tree-sitter grammar is complex to define
  • Bad, because code generation requires template libraries
  • Bad, because less suited for multi-stage transformations

Haskell with Parsec

  • Good, because excellent parser combinator library
  • Good, because strong type system catches errors
  • Good, because pure functional fits transformations
  • Bad, because steep learning curve
  • Bad, because lazy evaluation can be surprising
  • Bad, because deployment is more complex

AI Agent Guidance

When This Decision Applies

  • All DSL parser development in src/dslgen/parser/
  • All code generators in src/dslgen/generators/
  • Any new target language generators
  • DSL validation logic

When This Decision Does NOT Apply

  • Build system (CMake)
  • Generated C++ code (output, not input)
  • Runtime server code (C++23)
  • Shell scripts for deployment

Enforcement

  • All generator logic must be in Racket
  • Entry point is generate-geosciences-unified.rkt
  • Tests use Racket's test framework

Key Files

  • src/dslgen/parser/parser.rkt - DSL parser
  • src/dslgen/parser/lexer.rkt - Tokenizer
  • src/dslgen/model/ddd-types.rkt - AST structures
  • src/dslgen/generators/ - All code generators
