Use Shared Basedata Context for Cross-Cutting Lookup Entities

Use Shared Basedata Context for Cross-Cutting Lookup Entities

  • Status: accepted
  • Deciders: Project maintainers
  • Date: 2026-01-06
  • Tags: dsl, context, lookups, basedata, cross-context

Context and Problem Statement

Multiple bounded contexts in the Geosciences domain need the same lookup types:

  • DatingMethod used in Stratigraphy, Paleontology, Geochronology
  • SampleType used in FieldObservation, LaboratoryAnalysis, Drilling
  • MeasurementUnit used across nearly all contexts
  • OpticalType, HabitType, LusterType for Mineralogy properties

Without a shared context, each domain context would either:

  1. Duplicate these entities (violating DRY, causing sync issues)
  2. Use plain TEXT fields with CHECK constraints (losing referential integrity)
  3. Hard-code values in application logic (no database validation)

Decision Drivers

  • Single source of truth for cross-cutting lookup values
  • Referential integrity via foreign keys to lookup tables
  • Seed data loaded once, referenced everywhere
  • Clean context boundaries (domain contexts don't own infrastructure)
  • Project mode enables cross-context relationships

Considered Options

  • Duplicate lookup entities in each context that needs them
  • Create shared Basedata context with all cross-cutting lookups
  • Use PostgreSQL ENUM types instead of lookup tables
  • Keep CHECK constraints with hardcoded value lists

Decision Outcome

Chosen option: "Create shared Basedata context", because it provides a single source of truth, enables FK relationships across contexts, and keeps domain contexts focused on their core concerns.

Positive Consequences

  • One authoritative definition per lookup type
  • Cross-context FKs work via project mode context loading
  • Seed data managed in one location (seeds/basedata/)
  • Domain contexts remain focused on business logic
  • Easy to add new lookup values without touching domain files

Negative Consequences

  • Additional context to maintain
  • Must be loaded before any context that references it
  • Cross-context dependency (but explicit and validated)

Implementation

Basedata Context Structure

-- basedata.ddd
CONTEXT Basedata {
    description "Shared lookup entities for cross-cutting concerns"
    
    -- Each lookup is a standalone aggregate root
    AGGREGATE OpticalType ROOT OpticalType {
        ENTITY OpticalType IDENTIFIED_BY id (
            id UUID NOT NULL,
            code VARCHAR(50) NOT NULL UNIQUE,
            name VARCHAR(100) NOT NULL,
            description TEXT
        )
    }
    IMPORT CSV "seeds/basedata/optical_type.csv" INTO OpticalType;
    
    -- Repeat for each lookup type...
}

Lookup Entities to Include

Mineralogy lookups (properties that were CHECK IN constraints):

  • OpticalType - isotropic, uniaxial_positive, uniaxial_negative, biaxial_positive, biaxial_negative
  • HabitType - prismatic, tabular, massive, acicular, etc.
  • LusterType - vitreous, metallic, pearly, silky, etc.
  • FractureType - conchoidal, uneven, hackly, splintery, etc.
  • TenacityType - brittle, malleable, ductile, sectile, etc.
  • TransparencyType - transparent, translucent, opaque
  • CleavageType - perfect, good, poor, none

Cross-cutting lookups (used by multiple contexts):

  • DatingMethod - radiometric, biostratigraphy, magnetostratigraphy, etc.
  • SampleType - rock, soil, water, sediment, core, etc.
  • AnalysisType - XRF, XRD, ICP-MS, thin_section, etc.
  • MeasurementUnit - meters, kilometers, ppm, percent, etc.
  • QualityFlag - verified, provisional, suspect, rejected
  • ConfidenceLevel - high, medium, low, uncertain

Project.ddd Loading Order

CONTEXTS {
    include "context-map.ddd"    -- Strategic patterns
    include "basedata.ddd"       -- MUST come before domain contexts
    include "chemistry.ddd"      -- Foundation (no basedata deps)
    include "crystallography.ddd"
    include "mineralogy.ddd"     -- References basedata lookups
    ...
}

Cross-Context Relationship Example

-- In mineralogy.ddd
ENTITY Mineral IDENTIFIED_BY id (
    id UUID NOT NULL,
    optical_type_id UUID,  -- FK to basedata.optical_type
)
RELATIONSHIPS {
    optical_type -> OpticalType OPTIONAL ONE;  -- Cross-context to Basedata
}

AI Agent Guidance

When This Decision Applies

  • Adding new lookup types used by multiple contexts
  • Replacing CHECK IN constraints with FK relationships
  • Modifying cross-context dependency order
  • Adding seed data for lookup values

When This Decision Does NOT Apply

  • Context-specific lookups (e.g., MineralClass stays in Mineralogy)
  • Entities with business logic beyond simple code/name/description
  • Relationships within a single context

Enforcement

  • Basedata must be included in project.ddd before any dependent context
  • Validation checks that relationship targets exist (catches missing include)
  • All lookup entities follow code/name/description pattern

Key Files

  • src/examples/geosciences/basedata.ddd - Shared lookup entities
  • src/examples/geosciences/project.ddd - Context loading order
  • seeds/basedata/*.csv - Seed data for lookups
