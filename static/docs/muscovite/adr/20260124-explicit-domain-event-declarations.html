Explicit Domain Event Declarations in DDD Model

Explicit Domain Event Declarations in DDD Model

  • Status: accepted
  • Date: 2026-01-24
  • Tags: domain-events, dsl, event-sourcing, ddd, cqrs

Context and Problem Statement

Currently, domain events are emitted generically by CommandServices for CRUD operations (e.g., "PatientCreated", "PatientUpdated", "PatientDeleted"). These events are useful for audit logging but lack the expressiveness of proper domain events.

True Domain-Driven Design advocates for explicit, business-meaningful events that capture domain semantics beyond simple CRUD. For example:

  • PatientAdmitted instead of just EncounterCreated
  • PrescriptionFilled instead of MedicationUpdated
  • LabResultsReceived instead of ObservationCreated

This ADR evaluates whether to require explicit EVENT declarations in the DDD model for richer event-driven architectures.

Decision Drivers

  • Domain expressiveness: Business events communicate intent better than CRUD events
  • Event-driven architecture: Explicit events enable proper event sourcing
  • Backward compatibility: Existing projects work with generic CRUD events
  • Developer effort: Adding EVENT declarations requires model changes
  • Complexity tradeoff: More DSL complexity vs. richer domain modeling

Considered Options

  1. Keep generic CRUD events: Current approach, implicit events for all commands
  2. Require explicit EVENT declarations: Only emit events that are defined in DSL
  3. Hybrid approach: Generic CRUD events + optional explicit events

Evaluation Criteria

Option 1: Generic CRUD Events (Current)

Pros:

  • Zero DSL changes required
  • Works out-of-the-box for all entities
  • Good enough for audit logging
  • Simple mental model

Cons:

  • Events lack business semantics
  • Can't distinguish business operations (admit vs. discharge)
  • Payload is entire entity, not operation-specific
  • Event consumers must infer intent

Option 2: Require Explicit EVENT Declarations

Pros:

  • Full domain expressiveness
  • Events carry business meaning
  • Proper event sourcing capability
  • Clear contract for event consumers

Cons:

  • Requires extensive DSL model changes
  • Higher barrier to entry
  • Overkill for simple CRUD applications
  • Migration effort for existing projects

Option 3: Hybrid Approach

Pros:

  • Backward compatible (generic events still work)
  • Opt-in for explicit events where needed
  • Gradual adoption path
  • Best of both worlds

Cons:

  • Two event systems to understand
  • Potential confusion about when to use which
  • More complex generator logic

DSL Syntax (If Adopted)

CONTEXT Patient {
    ENTITY Patient IDENTIFIED_BY id (
        id          UUID NOT NULL,
        active      BOOLEAN NOT NULL DEFAULT true,
        mrn         TEXT NOT NULL @unique,
        ...
    )
    
    -- Explicit domain events
    EVENT PatientAdmitted (
        patient_id  UUID NOT NULL,
        encounter_id UUID NOT NULL,
        admitted_at TIMESTAMPTZ NOT NULL,
        department  TEXT
    )
    EMITS ON Patient.create
    DOC "Raised when a patient is admitted to the facility";
    
    EVENT PatientDischarged (
        patient_id   UUID NOT NULL,
        encounter_id UUID NOT NULL,
        discharged_at TIMESTAMPTZ NOT NULL,
        disposition  TEXT
    )
    DOC "Raised when a patient is discharged";
    
    AGGREGATE Patient ROOT Patient;
}

// Explicit event struct
struct PatientAdmitted {
    boost::uuids::uuid patient_id;
    boost::uuids::uuid encounter_id;
    std::chrono::system_clock::time_point admitted_at;
    std::optional<std::string> department;
    
    nlohmann::json to_json() const;
    static PatientAdmitted from_json(const nlohmann::json&);

// CommandService emits typed event
void PatientCommandService::admit(const AdmitPatientRequest& req) {
    execute([&](pqxx::work& txn) {
        auto patient = patient_repo_.find_by_id(txn, req.patient_id());
        // ... update logic ...
        
        event_repo_.emit(txn, PatientAdmitted{
            .patient_id = patient.id,
            .encounter_id = encounter.id,
            .admitted_at = std::chrono::system_clock::now(),
            .department = req.department()
        });
    });
}

Evaluation Questions

  1. How many FHIR Health operations need explicit events?

    • Admission, discharge, transfer, lab results, prescription fill, etc.

  2. What's the migration path for existing projects?

    • Can generic CRUD events coexist with explicit events?

  3. How do consumers distinguish event types?

    • Event type field already exists; explicit events add typed payloads

  4. Is this needed for audit logging alone?

    • Probably not; generic CRUD events suffice for audit

  5. Is this needed for event-driven microservices?

    • Yes; explicit events enable proper choreography

Preliminary Assessment

Recommendation: Adopt Option 3 (Hybrid) in a future version (v0.7.0+).

  • Keep generic CRUD events as default behavior
  • Allow optional explicit EVENT declarations for rich domain modeling
  • Generic events use "EntityCreated/Updated/Deleted" naming
  • Explicit events use custom names and typed payloads

This preserves backward compatibility while enabling advanced event-driven architectures for projects that need them.
