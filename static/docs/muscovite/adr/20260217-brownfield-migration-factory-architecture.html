Brownfield Migration Factory Architecture

Brownfield Migration Factory Architecture

  • Status: accepted
  • Date: 2026-02-17
  • Tags: brownfield, migration, nats, event-driven, strangler-fig, data-migration
  • Deciders: Johannes Lochmann

Technical Story: Legacy system modernization requires repeatable, industrial-grade data migration. The current approach — artisanal scripts, manual ETL — is high-risk, non-reproducible, and unscalable. Muscovite's existing NATS/outbox/DLQ infrastructure enables a generated, event-driven migration pipeline.

Context and Problem Statement

Brownfield migration is the highest-risk phase of legacy modernization. Traditional approaches suffer from:

  1. Brittle throwaway scripts — hand-written ETL that cannot be re-run safely
  2. Data quality blind spots — dirty/incomplete legacy data discovered only at cutover
  3. All-or-nothing cutover — no incremental migration, no backpressure, no triage
  4. No type safety — legacy TEXT columns mapped to target strong types without validation

Muscovite already generates the target system (schema, repositories, services, UIs) from .ddd files. The missing piece is a generated pipeline that moves data from legacy schemas into the generated target, with type-safe validation and human-in-the-loop triage for data quality issues.

Decision Drivers

  • Muscovite's NATS JetStream infrastructure is production-grade (NatsMessageBroker, OutboxPublisher, NatsEventConsumer)
  • The DLQ pattern (domain_events_dlq table + move logic + CLI tool) is already established
  • Saga orchestration (multi-step process with compensation/timeout/failure) maps directly to migration pipeline orchestration
  • ACL/Translator grammar constructs provide a template for field-level MAPPING blocks
  • CommandRepository is the mandated persistence path (ADR 20260120) — migration MUST NOT bypass it
  • The brownfield market is significant and growing: legacy system replacement + AI-generated slop remediation

Considered Options

  1. Generated event-driven pipeline via NATS — Extraction → NATS → Transformation/Validation → CommandRepository → DLQ triage
  2. Batch SQL migration scripts — Generated INSERT INTO ... SELECT FROM with data transformations inline
  3. External ETL tool integration — Generate config files for Apache NiFi, Airbyte, or similar
  4. No migration support — Users handle data migration outside Muscovite

Decision Outcome

Chosen option: Option 1 — Generated event-driven pipeline via NATS, because it leverages existing infrastructure (NATS, DLQ, CommandRepository, strong-type validation), provides backpressure and replay capability, enables incremental migration with human triage, and keeps all data flow through the CommandService layer per ADR 20260120.

The pipeline implements an "Automated Strangler Fig" pattern through four stages:

  1. Extraction: Generated C++ utility reads legacy schema via pqxx cursor-based batching, emits typed JSON events to NATS JetStream
  2. NATS Bridge: Buffer and backpressure controller — migration runs at its own pace without crashing legacy DB or target API
  3. Transformation & Load: Generated consumer maps legacy fields to target entities, validates against strong-type constructors and entity invariants, persists via CommandRepository (which automatically emits domain events)
  4. Dead Letter Triage: Records failing validation are routed to DLQ NATS subject for human review via generated Qt triage UI

Positive Consequences

  • 90%+ of migration code is generated from MAPPING DSL blocks
  • Idempotent via deterministic UUIDv5 — re-processing same legacy row produces same target UUID
  • Backpressure via NATS JetStream prevents migration from overwhelming target system
  • Human triage is focused on data quality decisions, not infrastructure
  • Migration pipeline uses same CommandRepository/event-sourcing path as production — no special migration code paths
  • Extends .ddd amplification to brownfield scenarios

Negative Consequences

  • NATS becomes a runtime dependency for migration (already a dependency for production event publishing)
  • Initial scope limited to PostgreSQL-to-PostgreSQL; other source databases require additional adapters
  • Transform expression language is a fixed set — custom business logic requires manual extension

Constraints

  • Migrations MUST NOT bypass the CommandService/CommandRepository layer (ADR 20260120)
  • All generated migration code MUST carry the dual-license SPDX header
  • Source database access is PostgreSQL-only in v1.24.0; other adapters are future iterations
  • The Loader MUST use generated STRONG_TYPE domains in PostgreSQL to prevent data corruption (ADR 20260119)

AI Agent Guidance

When implementing migration generators:

  • Extractor generator follows dslgen/generators/cpp/dba/command-repository.rkt patterns for pqxx query generation
  • Transformer generator follows dslgen/generators/cpp/dba/saga-repository.rkt patterns for multi-step orchestration
  • All generated C++ code uses NatsMessageBroker from framework/dba/ — do not create a new NATS client
  • DLQ routing reuses existing domain_events_dlq table schema from dslgen/generators/postgres/sqitch-events.rkt
  • Idempotency: use muscovite::types::uuid_v5() with namespace UUID from MAPPING @uuid annotation + legacy PK
