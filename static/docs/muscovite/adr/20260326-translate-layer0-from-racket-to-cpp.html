Translate Layer 0 from Racket to C++

Translate Layer 0 from Racket to C++

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-26
  • Tags: architecture, layer0, toolchain, ai-accessibility

Technical Story: IDEA-098, IDEA-099, deep-dive analysis session 2026-03-26

Context and Problem Statement

Layer 0 (the DSL parser, IR model, and all code generators) is implemented in Racket (~140K lines, 349 files, 254 generator modules). Muscovite is planned to grow from 10 generator targets to 14+ (adding C#, Java, React, Flutter backends and frontends). Each new target adds ~50-80 generator modules (~15-20K lines).

Frontier AI models (Claude Opus/Sonnet, GPT-4) are measurably less capable in Racket than in C++. This manifests as:

  1. Lower first-draft quality — more correction cycles per generator module
  2. Weaker debugging — models lose track of parenthesis nesting, confuse for/list with for/fold, break apply string-append by returning wrong types
  3. Slower refactoring — Racket-specific patterns (match expressions, s-expression templates, contract-out) have fewer training examples than C++ equivalents
  4. Model dispatch workarounds — the decision heuristics recommend Mistral for Racket because Opus/Sonnet are weak in Lisp, fragmenting the model strategy

This productivity tax is a variable cost on a growing codebase. As Layer 0 doubles with new targets, the cumulative friction compounds.

Additionally, translating to C++ provides:

  • IP protection: distribute a compiled binary, not generator source
  • Zero-dependency distribution: no Racket runtime on target machines
  • Sub-second startup: no 71s cold start / 10s warm start
  • Dogfooding: Layer 0 uses the same language as Layer 1 and Layer 2

Decision Drivers

  • AI agent productivity on Layer 0 (primary driver — 20-30% efficiency gain per session, compounding across all future generator targets)
  • Distribution simplicity (single static binary vs Racket runtime dependency)
  • IP protection for commercial licensing (compiled binary)
  • Startup performance (<1s vs 10-71s)
  • Consistency with Layer 1 (C++ framework) and Layer 2 (generated C++)
  • The Racket codebase is undergoing a final idiomatic refactoring wave (v1.77.0) that will bring all generators to consistent template-DSL style — the optimal translation source

Considered Options

  1. Translate Layer 0 to C++ (chosen)
  2. Translate Layer 0 to Python
  3. Translate Layer 0 to TypeScript
  4. Keep Racket, mitigate via documentation and style enforcement

Decision Outcome

Chosen option: "Translate Layer 0 to C++", because it maximizes AI agent productivity on the language where frontier models are strongest, provides IP protection and zero-dependency distribution, dogfoods the same stack as Layer 1/2, and eliminates the Racket runtime dependency.

The translation is a parallel build — Racket remains the reference implementation until the C++ tool produces byte-identical golden output for all targets. Retirement is gradual, per-target, with a kill switch in generate.rkt.

Positive Consequences

  • Every new generator target (C#, Java, React, Flutter) written in C++ from day one — no Racket tax on greenfield development
  • Unified language across all three layers (C++ everywhere)
  • Distributable as a single binary for commercial customers
  • Sub-second cold start enables tighter edit-generate-test loops
  • AI agents operate at peak capability on Layer 0 work
  • The v1.77.0 style ratchet produces maximally idiomatic Racket — the cleanest possible translation source

Negative Consequences

  • 6-7 week translation effort (heavily AI-assisted, parallelizable)
  • Loss of Racket's REPL-driven development for generator prototyping
  • C++ compile step for generator changes (mitigated by ccache + mold)
  • Temporary maintenance of two implementations during transition
  • The template DSL library requires design work (not mechanical translation)

Pros and Cons of the Options

Option 1: Translate to C++

  • Good, because C++ is the highest-resource language for frontier AI models
  • Good, because it dogfoods the same stack (Layer 0 = Layer 1 = Layer 2)
  • Good, because compiled binary protects IP and eliminates runtime dependency
  • Good, because Boost (already a Conan dependency) provides Spirit.X3 for PEG parsing
  • Good, because CLI11 (already a Conan dependency) handles arg parsing
  • Good, because nlohmann_json (already a Conan dependency) handles JSON seed data
  • Bad, because C++ has a compile step (Racket runs directly)
  • Bad, because 6-7 weeks of translation effort

Option 2: Translate to Python

  • Good, because Python is well-modeled by AI and has a large developer pool
  • Good, because no compile step (interpreted like Racket)
  • Good, because rich ecosystem (Jinja2, lark, pydantic)
  • Bad, because introduces a third language to the project
  • Bad, because Python performance is significantly worse than C++ for large models
  • Bad, because no IP protection (source distribution)
  • Bad, because runtime dependency (Python 3.x)

Option 3: Translate to TypeScript

  • Good, because TypeScript is well-modeled by AI
  • Good, because Node.js has fast startup
  • Bad, because introduces a third language and runtime (Node.js)
  • Bad, because no IP protection (source distribution)
  • Bad, because npm ecosystem is a maintenance burden
  • Bad, because no Conan integration — separate build system

Option 4: Keep Racket with mitigations

  • Good, because zero translation cost
  • Good, because Racket is technically well-suited for DSL work
  • Bad, because the AI productivity tax is permanent and growing
  • Bad, because every new target (C#, Java, React, Flutter) pays the tax
  • Bad, because model dispatch fragmentation (Mistral for Racket) persists
  • Bad, because Racket runtime remains a distribution dependency

Implementation Strategy

Naming

The C++ tool is called muscomp (muscovite compiler). Located in toolchain/muscomp/.

PEG Library

Use cpp-peglib (MIT license, header-only, C++17, Packrat parsing, available on Conan). The existing 365-rule PEG grammar (dslgen/grammar/muscovite.peg) is the spec — cpp-peglib reads PEG grammars directly, minimizing translation risk.

Alternative: Boost.Spirit.X3 (BSL-1.0, already a dependency). More verbose but zero additional dependency. Decision deferred to implementation phase.

Translation Phases

Phase Scope Approach
1. Skeleton CLI, file I/O, IR structs, naming Mechanical translation
2. Parser PEG grammar → cpp-peglib Grammar-driven, minimal translation
3. Template DSL Block/Line/Indent builder Design required (~500 lines)
4. Generators 254 modules, target-by-target AI-assisted, golden test validated
5. Validation Cross-reference, relationships Mechanical translation
6. Migration Semantic diff engine Mechanical translation
7. Integration Kill switch in generate.rkt Gradual per-target cutover

Acceptance Criteria

  • Byte-identical golden output for all 160 golden files
  • All 96 Racket test suite scenarios reproducible as C++ integration tests
  • muscomp --project examples/fhir-health/project.mus --full produces identical output to racket generate.rkt --project examples/fhir-health/project.mus --full
  • Sub-second cold start (measured)

Retirement Protocol

  1. Each generator target has an independent --use-muscomp flag
  2. When a target passes all golden tests via muscomp, it graduates
  3. When all targets graduate, Racket generators move to archive/dslgen-racket/
  4. generate.rkt becomes a thin wrapper that calls muscomp
  5. After one full release cycle with muscomp-only, remove Racket dependency

AI Agent Guidance

When This Decision Applies

  • When writing NEW generator targets (C#, Java, React, Flutter) — write in C++ inside toolchain/muscomp/, not in Racket
  • When translating existing Racket generators to C++
  • When choosing where to implement new DSL features (parser, validation)

When This Decision Does NOT Apply

  • During the v1.77.0 Racket refactoring wave — complete the style ratchet in Racket first, then translate the clean result
  • For bug fixes to existing Racket generators — fix in Racket until that target graduates to muscomp
  • For the PEG grammar itself (dslgen/grammar/muscovite.peg) — this remains the single source of truth regardless of parser implementation

Enforcement

  • New generator targets MUST be implemented in toolchain/muscomp/
  • Golden test parity is the acceptance gate — byte-identical output required
  • release.sh runs both Racket and muscomp generators during transition, diff-checking output

Key Files

  • toolchain/muscomp/ — C++ tool source (new)
  • dslgen/grammar/muscovite.peg — PEG grammar (shared, single source of truth)
  • dslgen/tests/test-output-golden/ — golden files (shared acceptance criteria)
  • dslgen/generate.rkt — kill switch / dispatch to muscomp
