Generate QML TestCase UI Tests from DSL

Technical Story: Automate UI testing for Qt Quick/QML frontends generated by Muscovite.

Context and Problem Statement

Muscovite generates complete Qt Quick/QML frontends (ViewModels, dialogs, list views) from .ddd domain models. These UIs are currently tested only manually. How should we automate UI testing in a way that is DSL-driven, maintainable, and integrates with the existing CTest infrastructure?

Decision Drivers

  • Tests must be generated from DSL (no manual test authoring for generated UI)
  • Must integrate with CTest (ctest --preset debug -L qt)
  • Must run headless in CI (Xvfb / QT_QPA_PLATFORM=offscreen)
  • Must not require commercial tooling (Squish)
  • Must test the actual QML binding layer, not just C++ logic
  • Mock ViewModels should allow testing without a running gRPC server

Considered Options

  • Option 1: QML TestCase (QtQuick.Test) — generated per entity
  • Option 2: C++ QTest with QQuickView — load QML from C++
  • Option 3: Squish for Qt — commercial record/replay
  • Option 4: Playwright/Selenium-style — browser-based (not applicable to Qt)

Decision Outcome

Chosen option: Option 1 — QML TestCase, with Option 2 for integration tests that require a gRPC server.

Tier 1: Generated QML TestCase (Unit-Level)

Add a --qt-test generator flag that emits per-entity test files using Qt's built-in QtQuick.Test module:

import QtQuick 2.15
import QtTest 1.15
import FhirHealthQml

TestCase {
    name: "PatientDialogTest"
    when: windowShown

    // ViewModel with mock channel (no gRPC)
    PatientViewModel {
        id: viewModel
    }

    Patient_dialog {
        id: dialog
        viewModel: viewModel
    }

    function test_initialState() {
        compare(dialog.errorLabel.visible, false)
        compare(viewModel.id, "")
    }

    function test_requiredFieldValidation() {
        // active is required bool, defaults to false
        compare(viewModel.active, false)
    }

    function test_setFieldUpdatesViewModel() {
        viewModel.setMrn("MRN-001")
        compare(viewModel.mrn, "MRN-001")
    }

    function test_clearResetsFields() {
        viewModel.setMrn("MRN-001")
        viewModel.clear()
        compare(viewModel.mrn, "")
    }

    function test_toJsonContainsFields() {
        viewModel.setMrn("MRN-TEST")
        viewModel.setActive(true)
        var json = JSON.parse(viewModel.toJson())
        compare(json.mrn, "MRN-TEST")
        compare(json.active, true)
    }
}

Generated test categories per entity:

Category What It Tests Requires Server?
Initial state Default property values, error label hidden No
Field binding Set property → read back No
Clear/reset createNew() / clear() resets all fields No
toJson round-trip JSON serialization includes all fields No
Required field guards Optional Date/UUID not sent when empty No
FK dropdown loading Mock data populates ComboBox No (mock)
Dialog lifecycle Open/close, visibility toggles No

Tier 2: C++ Integration Tests (with gRPC Server)

For full round-trip testing (create → load → verify → delete), use C++ QTest:

TEST_CASE("Patient CRUD via ViewModel", "[qt][patient][integration]") {
    auto channel = grpc::CreateChannel("localhost:50051", grpc::InsecureChannelCredentials());
    PatientViewModel vm(channel);

    vm.setActive(true);
    vm.setDisplayName("Test Patient");
    
    QSignalSpy saveSpy(&vm, &PatientViewModel::saveSucceeded);
    vm.save();
    REQUIRE(saveSpy.wait(5000));
    REQUIRE(!vm.id().isEmpty());

    // Verify round-trip
    PatientViewModel vm2(channel);
    QSignalSpy loadSpy(&vm2, &PatientViewModel::loadingChanged);
    vm2.load(vm.id());
    REQUIRE(loadSpy.wait(5000));
    REQUIRE(vm2.displayName() == "Test Patient");

    // Cleanup
    QSignalSpy delSpy(&vm, &PatientViewModel::deleteSucceeded);
    vm.remove(vm.id());
    REQUIRE(delSpy.wait(5000));
}

CTest Integration

# Tier 1: QML unit tests (no server required)
add_test(
    NAME qml_patient_test
    COMMAND qmltestrunner -input ${CMAKE_CURRENT_SOURCE_DIR}/tests/qml/tst_Patient.qml
)
set_tests_properties(qml_patient_test PROPERTIES
    LABELS "qt;qml;unit"
    ENVIRONMENT "QT_QPA_PLATFORM=offscreen"
)

# Tier 2: Integration tests (requires running server + DB)
add_test(
    NAME qt_patient_integration
    COMMAND qt_integration_tests
)
set_tests_properties(qt_patient_integration PROPERTIES
    LABELS "qt;integration;db"
    FIXTURES_REQUIRED "grpc_server"
)

Positive Consequences

  • Every generated entity automatically gets UI tests
  • Tests run headless in CI without manual intervention
  • Mock ViewModels test QML binding layer without server dependency
  • CTest labels (qt, qml, unit, integration) enable selective execution
  • Same DSL model drives both UI generation and test generation

Negative Consequences

  • qmltestrunner requires Qt Quick Test module (already a dependency)
  • Offscreen rendering may miss platform-specific visual bugs
  • Integration tests need database fixtures (existing CTest infrastructure handles this)

Pros and Cons of the Options

Option 1: QML TestCase

  • Good, because it's built into Qt — no external dependencies
  • Good, because tests are written in QML, testing the actual binding layer
  • Good, because qmltestrunner integrates with CTest
  • Good, because it can be generated from DSL (same field/type metadata)
  • Bad, because limited to functional testing (no visual regression)

Option 2: C++ QTest with QQuickView

  • Good, because it tests the full C++ ↔︎ QML stack
  • Good, because it integrates with Catch2 test infrastructure
  • Good, because QSignalSpy verifies async operations
  • Bad, because more boilerplate than QML TestCase
  • Bad, because requires loading QML engine in C++ (heavier setup)

Option 3: Squish for Qt

  • Good, because record/replay is fast to author
  • Good, because it supports visual verification
  • Bad, because commercial license required
  • Bad, because not generatable from DSL
  • Bad, because test scripts are fragile to UI layout changes

When This Decision Applies

  • When adding --qt-test generator flag to produce QML test files
  • When creating CTest targets for QML/Qt Quick UI tests
  • When deciding how to test generated QML dialogs and list views
  • When setting up CI pipelines for Qt Quick frontends

When This Decision Does NOT Apply

  • Qt Widgets testing (use C++ QTest directly)
  • Non-UI C++ testing (use Catch2 per ADR 20260120)
  • Server-side / database testing (use existing CTest + Catch2)
