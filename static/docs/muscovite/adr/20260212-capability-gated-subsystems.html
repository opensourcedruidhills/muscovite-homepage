Capability-Gated Subsystems

Capability-Gated Subsystems

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-12
  • Tags: architecture, capabilities, modularity, kernel

Context and Problem Statement

Muscovite's DSL kernel (entity, aggregate, field, relationship, value-object, strong-type, lookup-table) is stable and well-isolated. However, five subsystems have grown alongside the kernel with inconsistent gating mechanisms:

Subsystem Lines (generators) Current Gating Isolation Quality
Security ~1,200 + ~2,500 (framework) CAPABILITIES { security } + unconditional server includes ⚠️ Partial — headers always emitted
Saga Engine 1,469 Data-driven (bounded-context-sagas) ✅ Clean
Event Sourcing 1,382 Data-driven (aggregate-event-sourced?) ✅ Clean
Spatial ~465 + ~200 (framework) PostGIS always on; RPCs per @spatial-queryable ⚠️ PostGIS unconditional
Frontend (Qt) 11,769 CLI flag --qt-frontend ✅ Clean

At 72K lines of Racket and growing, the risk is kernel creep: every new construct (PROJECTION, POLICY, more saga features) adds parser complexity unconditionally. A hello-world project pays the cost of saga/event-sourcing parser overhead for no benefit.

The existing CAPABILITIES mechanism only controls context injection (adding a .ddd file to the parse list). It does NOT gate generator behavior. Two subsystems already claim to be capabilities but leak:

  1. Security: Server-main generator unconditionally emits #include <muscovite/server/security/*.hpp> regardless of whether security is in CAPABILITIES. CMake uses if(TARGET) guards as a fallback.
  2. Spatial: 000-init Sqitch change unconditionally creates CREATE EXTENSION IF NOT EXISTS postgis. Spatial RPCs/indexes are data-driven by @spatial-queryable annotations, bypassing the capability flag entirely.

Decision Drivers

  • Kernel stability: The 7 core constructs must remain the strict kernel. Everything else is a feature module.
  • Explicit over clever: A project.ddd should declare what infrastructure it uses. "Does this project use sagas?" should be answerable from the project file, not from scanning all context files.
  • 72K monolith risk: Without module boundaries, the Racket codebase becomes harder to reason about.
  • New user experience: hello-world and simple projects should not pay parser/generator overhead for unused subsystems.
  • Existing pattern: Security and spatial already use CAPABILITIES — the mechanism exists, it just needs to be completed and made consistent.
  • Grammar purity: PEG grammar rules should not become context-dependent (ADR 20260211). Parser gating must happen at the validation layer, not the grammar layer.

Considered Options

Option 1: Factor all five into capabilities

Move saga, event sourcing, security (fix), spatial (fix), and frontends into CAPABILITIES { ... }.

Rejected: Frontends are build-time output selections ("same model → different output format"), not domain-level infrastructure declarations. CLI flags (--qt-frontend) are the correct mechanism for that concern.

Option 2: Factor four, keep frontends CLI-gated

Factor saga + event sourcing into capabilities. Fix security + spatial leaks. Keep frontends as CLI flags.

Selected: Clean separation of concerns — capabilities declare "what infrastructure does this project use?" while CLI flags declare "what outputs should this build produce?"

Option 3: Status quo — leave everything data-driven

Keep saga/event-sourcing as implicit data-driven features, accept security/spatial leaks.

Rejected: Violates explicit-over-clever principle. The monolith risk grows with each new construct.

Decision Outcome

Chosen option: Option 2 — Factor four subsystems into capabilities, keep frontends CLI-gated.

Target State

PROJECT "FHIR Health" {
    CAPABILITIES {
        security          -- auth/authz/audit (existing, fix leaks)
        spatial           -- PostGIS + spatial queries (existing, fix leaks)
        event-sourcing    -- typed event store + snapshots (new)
        saga              -- orchestrated multi-step workflows (new)
    }
}

Implementation Phases

Phase Work Effort Risk
1. Fix security leaks Gate server-main #include directives on security capability presence. Remove unconditional security wiring. 2 days Medium — must verify both example projects still compile
2. Fix spatial leaks Gate CREATE EXTENSION postgis on spatial capability. PostGIS only created when declared. 1 day Medium — geosciences uses spatial, fhir-health does not
3. Saga → capability Register saga in capability registry. Validate: SAGA blocks rejected if capability not declared. Generators already null-check bounded-context-sagas. 2 days Low — generators already cleanly gated
4. Event sourcing → capability Register event-sourcing in capability registry. Validate: EVENT_SOURCED / SOURCED_EVENT rejected if capability not declared. Generators already check aggregate-event-sourced?. 2 days Low — same pattern as saga

Total estimated effort: ~7 working days.

Parser Strategy (Grammar Purity)

Per ADR 20260211 (grammar-first), the PEG grammar will not change. SAGA, SOURCED_EVENT, and EVENT_SOURCED remain valid grammar productions. Gating happens at the validation layer (dslgen/validation/):

  1. Parser accepts all constructs unconditionally (grammar stays pure)
  2. Validation pass checks: "context uses SAGA but project lacks saga capability" → error with actionable message
  3. Generators remain data-driven (null checks already in place)

This is the same pattern as type validation: the parser accepts any identifier as a type, then the type registry rejects unknown types.

Frontends: Explicitly NOT Capabilities

Frontend generation remains CLI-flag gated:

Flag Purpose
--qt-frontend widgets|qml|both Build-time: which Qt dialect to emit
--cpp-dba, --grpc Build-time: which backend layers to emit

Rationale: A project model is the same regardless of whether you build Qt Widgets or QML from it. The .ddd file describes the domain; CLI flags describe the build. Mixing these concerns in CAPABILITIES would conflate "what the system is" with "how we build it."

Update to Example Projects

-- examples/fhir-health/project.ddd
PROJECT "FHIR Health" {
    CAPABILITIES { security event-sourcing }
    -- spatial NOT declared: no PostGIS, no spatial RPCs
    -- saga NOT declared: SAGA blocks in contexts would be validation errors
}

-- examples/geosciences/project.ddd
PROJECT "Geosciences" {
    CAPABILITIES { spatial saga }
    -- security NOT declared: no auth infrastructure
    -- event-sourcing NOT declared: EVENT_SOURCED on aggregates would be validation errors
}

Positive Consequences

  • Explicit project manifest: CAPABILITIES block is a complete declaration of infrastructure dependencies
  • Simpler hello-world: Minimal projects get no saga/event-sourcing/spatial/security overhead
  • Consistent gating: All four subsystems use the same mechanism (capability registry + validation pass + data-driven generators)
  • Safer evolution: New subsystems (e.g., scheduling, workflow, audit-trail) follow an established pattern
  • Better error messages: "SAGA block found but project does not declare saga capability" vs silent generation of unused tables

Negative Consequences

  • Migration burden: Existing .ddd projects must add capabilities to their project.ddd (one-time, trivial)
  • Two gating layers: Capability check (validation) + data-driven generation (generators). But this is defense-in-depth, not complexity
  • Grammar disconnect: Grammar accepts constructs that validation may reject. Acceptable — same as type validation today

Prerequisites

  • v1.10.0 TDD foundation (FTR-030) should land first — provides test coverage to validate the refactoring safely
  • Capability registry (dslgen/builtin/) already exists for security/spatial — extend, don't rebuild
