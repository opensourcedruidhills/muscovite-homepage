Grammar-Parser Parity Enforcement and Drift Detection

Grammar-Parser Parity Enforcement and Drift Detection

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-13
  • Tags: grammar, parser, testing, drift-detection, architecture-health, process

Context and Problem Statement

Muscovite has 32 documented rules and conventions across ADRs, AGENTS.md, and copilot-instructions.md. Of these, only 4 are enforced by automated tooling (relationship validation, event sourcing validation, reserved keyword checking, cross-layer verification). The remaining 28 are documentation-only — guidance for AI agents and human reviewers that can be silently violated.

The Project DSL Dark Corner (Root Cause Analysis)

The project DSL grew organically over multiple sessions: CAPABILITIES, OAUTH2, and FRONTENDS blocks were added to the parser in generate.rkt without corresponding PEG grammar updates, without tests, and without documentation. This violated ADR 20260211 (grammar-first), but nothing detected the violation because:

  1. Grammar conformance test only checks 24 modeling constructs — ProjectBlock is not in expected-rules
  2. No stub/opaque rule detection — the grammar declares ProjectBlock <- 'PROJECT' STRING? BracedBlock? which matches any braced content. The test doesn't flag this as "grammar says nothing about what's inside"
  3. No parser-grammar parity check — nothing compares "keywords the parser handles" vs "keywords the grammar specifies"
  4. No architecture health metrics — CONTEXT.md reports line counts but not specification coverage

Systemic Pattern

Every ADR containing MUST, FORBIDDEN, or MANDATORY without a corresponding automated check is a future dark corner. Current gap inventory estimate: ~28 unenforced rules.

Decision Drivers

  • Automation over discipline: Conventions enforced by tooling cannot be silently violated
  • Prompter awareness: The project owner supervises AI agents but cannot review every parser change for grammar compliance — tooling must surface drift
  • Compounding defense: Each check prevents an entire class of errors, not just one instance
  • Minimal overhead: Health checks should run as part of existing test infrastructure, not require new CI

Considered Options

Option 1: Drift Detection + Health Metrics + ADR Compliance Audit

Three-pronged approach:

  • Extend grammar conformance test with drift detection (opaque rule flagging, parser keyword counting)
  • Add specification coverage metrics to CONTEXT.md
  • Systematic one-time audit of all ADRs to inventory enforcement gaps, then backlog ideas for each

Option 2: CI Pipeline Only

Build a GitHub Actions CI pipeline that runs all tests on push. Catches violations after commit but doesn't help the prompter during interactive sessions.

Option 3: Documentation-Only (Status Quo)

Keep adding rules to copilot-instructions.md and trust AI agents to follow them. Proven insufficient by the project DSL drift.

Decision Outcome

Chosen option: Option 1 — Drift Detection + Health Metrics + ADR Compliance Audit, because it addresses the root cause (silent rule violation) at three levels: test-time detection, session-time visibility, and one-time gap inventory.

Component 1: Grammar Conformance Drift Detection

Extend dslgen/tests/test-grammar-conformance.rkt to:

  1. Detect opaque/stub PEG rules — Rules whose body is only BracedBlock or ( !'}' . )* are flagged as ⚠️ OPAQUE: RuleName — they accept any content without validation
  2. Count parser-accepted keywords — Scan parser.rkt and project-parser.rkt for match-keyword? calls, extract the keyword set
  3. Count grammar-specified keywords — Scan muscovite.peg for quoted terminal strings (e.g., 'ENTITY', 'CAPABILITIES'), extract the keyword set
  4. Report delta — If parser accepts keywords not in grammar: ⚠️ DRIFT: keyword X accepted by parser but not specified in grammar
  5. Verdict — PASS (zero drift), WARN (opaque rules exist), FAIL (parser-grammar keyword mismatch)

Component 2: Architecture Health in CONTEXT.md

Extend tools/generate-context.sh to emit a "Specification Coverage" section:

## Specification Coverage

| Metric | Value |
|--------|-------|
| PEG grammar rules (formal) | 250 |
| Parser keyword handlers | 248 |
| Delta (drift) | 2 ⚠️ |
| Opaque/stub rules | 0 ✅ |
| Grammar conformance verdict | PASS |
| ADR compliance (enforced/total) | 8/32 |

This surfaces in every AI session — the prompter sees drift immediately.

Component 3: ADR Compliance Audit

One-time systematic audit:

  1. Read all ~129 ADRs
  2. Extract every MUST, FORBIDDEN, MANDATORY, NO EXCEPTIONS rule
  3. Classify each as enforced (has automated check in dslgen/tests/ or dslgen/validation/) or unenforced (documentation only)
  4. Output: doc/ADR-COMPLIANCE-AUDIT.md with gap inventory table
  5. For each high-impact unenforced rule: create a workflow idea describing the enforcement mechanism

Slash Command: /health-check

Add to copilot-instructions.md:

/health-check — Run `racket dslgen/tests/run-all-tests.rkt` + grammar conformance.
                Report any DRIFT or WARN verdicts. Run at session start.

Critical Rule: §12 Grammar Parity

Add to copilot-instructions.md:

Every syntax the parser accepts MUST have a corresponding non-opaque PEG rule. If you add a keyword to any parser, update the grammar first (per ADR 20260211). The conformance test MUST report zero DRIFT warnings. Run /health-check to verify.

Positive Consequences

  • Silent drift impossible: Parser changes without grammar updates trigger DRIFT verdict in conformance test
  • Prompter awareness: CONTEXT.md health metrics visible at every session start
  • Systematic gap inventory: One-time audit surfaces all unenforced rules, creating a prioritized backlog
  • Defense in depth: Three independent detection mechanisms (test, metrics, audit) — any one catches the problem class

Negative Consequences

  • Test runtime increase: Keyword scanning adds ~2s to conformance test — acceptable
  • CONTEXT.md complexity: One more section — but high-signal, directly actionable
  • Audit effort: One-time ~0.5–1 day investment to read and classify 129 ADRs
  • Backlog growth: Audit will generate 5–10 new ideas — intentional, not overhead

AI Agent Guidance

When This Decision Applies

  • Adding any new keyword to any parser (modeling or project)
  • Modifying PEG grammar rules
  • Adding new ADRs with MUST/FORBIDDEN rules — consider whether enforcement tooling is needed
  • Starting a new session — run /health-check first

Enforcement

  • Conformance test: racket dslgen/tests/test-grammar-conformance.rkt must report PASS (zero DRIFT, zero opaque rules)
  • CONTEXT.md: Specification coverage section must show zero delta
  • New ADRs: When writing an ADR with a MUST rule, document the enforcement mechanism or create a follow-up idea
