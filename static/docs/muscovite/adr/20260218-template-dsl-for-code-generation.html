ADR-20260218: Template DSL for Code Generation

ADR-20260218: Template DSL for Code Generation

Status: Accepted Date: 2026-02-18 Feature: FTR-069

Context

Muscovite generators currently produce code using Racket format strings and string-append calls. This approach has served well for the initial implementation but presents scaling challenges:

  1. Positional placeholders: ~a placeholders in format strings are error-prone when templates grow large (counting errors, off-by-one mistakes)
  2. No structural validation: Format strings are opaque—there's no compile-time check that generated code structure is valid
  3. Template duplication: Similar patterns (e.g., Qt Widgets and QML validation logic) are copy-pasted across generators with minor variations
  4. Indentation fragility: Manual indentation via string concatenation ((indent 1 ...)) is scattered without consistent abstraction

As Muscovite adds more targets (Qt QML, TypeScript, Rust), the cost of duplicating format-string logic escalates.

Decision

Introduce an s-expression template DSL for code generation that replaces raw format/string-append with structured, composable templates.

Design Principles

  1. S-expression based: Templates are Racket data structures, not strings. This enables:

    • Syntax checking at template definition time
    • Programmatic composition (templates can reference other templates)
    • Target-agnostic structure (same template can emit C++, SQL, or TypeScript with different emitters)

  2. Explicit indentation: No auto-indent magic. Templates use explicit indent calls, preventing nested block double-indentation bugs.

  3. Composable building blocks: Templates are small, reusable functions (e.g., cpp-class, cpp-method, cpp-field) that combine to form complete files.

  4. Performance acceptable: Template evaluation should be within 10× of format-string baseline. Measured at 4× in initial implementation (acceptable, can optimize if needed).

API

;; Define a reusable template
(define-template cpp-class
  (name doc sections)
  (block 0
    (cpp-doc-comment doc)
    (line (format "class ~a final {" name))
    (map-items
     (λ (section) (emit-section section))
     sections)
    (line "};")))

;; Use it
(emit (cpp-class "Person" "User entity" sections) #:target 'cpp)

Core Template Nodes

Node Purpose Example
text Raw text (no newline) (text "hello") → "hello"
line Text with newline (line "int x;") → "int x;\n"
block Container (no auto-indent) (block 0 ...)
join Join with separator (join ", " items) → "a, b, c"
when-true Conditional inclusion (when-true cond tmpl)
map-items Map function over list (map-items fn lst)

Proof-of-Concept Scope

v1.21.0 converts value object generation only. This proves the DSL works end-to-end:

  • Template modules: dslgen/generators/template-dsl.rkt, dslgen/generators/cpp/emitters.rkt
  • Tests: dslgen/tests/test-template-dsl.rkt (all passing, 4× performance ratio)
  • Integration: generate-value-object in dslgen/generators/cpp/domain.rkt now calls generate-value-object-template
  • Output: Byte-identical to format-string version (verified via incremental generation SHA matching)

Remaining generators (entities, domain events, repositories, services, Qt, gRPC, SQL) migrate incrementally in future releases.

Consequences

Positive

  • Reduced duplication: Cross-target patterns (validation, getters, constructors) can be shared via template composition
  • Easier maintenance: Changing a template updates all uses automatically (vs. hunting down format-string duplicates)
  • Better error messages: Template evaluation errors can report source location (vs. cryptic format-string arity mismatches)
  • Incremental migration: Proof-of-concept (value objects) validates approach without forcing a big-bang rewrite

Negative

  • Learning curve: Contributors must learn the template DSL API (mitigated by comprehensive tests and examples)
  • Performance overhead: 4× slower than format strings (acceptable for generator code; can optimize if needed)
  • Two codepaths during transition: Some generators use templates, others use format strings (temporary, resolves as migration completes)

Neutral

  • Line count: Template definitions are slightly more verbose than equivalent format strings, but improved readability offsets this

Alternatives Considered

1. Stick with format strings

Rejected: Duplication and fragility already causing maintenance burden. Adding more targets (TypeScript, Rust) would exacerbate this.

2. Use external templating engine (Mustache, Jinja2)

Rejected: External tools add dependency and force context out of Racket. S-expressions keep everything in-language with full Racket expressiveness.

3. Macro-based code generation

Rejected: Racket macros operate at Racket syntax level. Generating C++/SQL/TypeScript requires string output, making macros awkward for this use case.

Implementation Notes

  • File locations:

    • Core DSL: dslgen/generators/template-dsl.rkt
    • C++ emitters: dslgen/generators/cpp/emitters.rkt
    • Tests: dslgen/tests/test-template-dsl.rkt

  • Migration strategy:

    1. ✅ Value objects (v1.21.0)
    2. Entities (v1.22.0 candidate)
    3. Domain events, repositories, services (v1.23.0+)
    4. Cross-target templates (Qt Widgets/QML shared logic) (v1.24.0+)

  • Performance: Measured at 4× slower than format strings (1000 iterations: template 4ms, format 1ms). This is acceptable for generator code which runs offline. If performance becomes critical, can optimize template-eval with caching or lazy evaluation.
