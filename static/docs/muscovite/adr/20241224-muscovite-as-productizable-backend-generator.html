Muscovite as Productizable Backend Generator

Muscovite as Productizable Backend Generator

  • Status: accepted
  • Deciders: jolo
  • Date: 2024-12-24
  • Tags: architecture, product-identity, code-generation

Context and Problem Statement

Muscovite has evolved from a DSL experimentation tool into a comprehensive code generator. We need to clarify its product identity and distinguish between core capabilities (shipped with Muscovite) and example projects (proof-of-concept demonstrations).

Key questions:

  1. What is Muscovite's primary purpose?
  2. What capabilities should be built-in vs. external/optional?
  3. How do example projects (like geosciences) relate to the core product?

Decision Drivers

  • Need for a self-contained, productizable code generator
  • Desire to ship common infrastructure domains (security, audit) as reusable built-ins
  • Clear separation between "Muscovite core" and "example projects"
  • All generated projects need security/auth integration

Considered Options

  1. External Module Registry: Allow importing modules from external sources
  2. Built-in Capabilities: Ship infrastructure domains as part of Muscovite distribution
  3. Copy-Paste Templates: Provide templates that users copy into their projects

Decision Outcome

Chosen option: Built-in Capabilities, because it keeps Muscovite self-contained, ensures consistency across all generated projects, and eliminates external dependencies.

Muscovite Product Definition

Muscovite is a productizable backend code generator with focus on:

  • PostgreSQL database (schemas, migrations via Sqitch)
  • PL/pgSQL stored procedure APIs
  • C++ server with gRPC services
  • Qt/QML client with QtGRPC integration

Core Components:

  1. DSL Parser: Racket-based parser for .ddd domain models
  2. Generators: Code generators for all target platforms
  3. Built-in Capabilities: Infrastructure domains (security, audit, storage) that are opt-in via project configuration

Example Projects:

  • Geosciences: Full-fledged experimental proof-of-concept (19 bounded contexts, ~8000 lines of DSL)
  • Located in src/dslgen/examples/geosciences/ (NOT part of core distribution)
  • Demonstrates all Muscovite features but is NOT shipped as a reusable module

Built-in Capabilities Architecture

Built-in capabilities are infrastructure domains that:

  1. Ship with Muscovite distribution (in src/dslgen/builtin/)
  2. Are enabled via CAPABILITIES { ... } block in project.ddd
  3. Follow the SAME generation patterns as user-defined contexts
  4. Generate <context> + <context>_api schemas like all other contexts

Positive Consequences

  • Self-contained product with no external dependencies
  • Consistent API patterns across all generated code
  • Security domain available to all projects by default
  • Clear separation of concerns between core and examples

Negative Consequences

  • All capabilities must be maintained as part of Muscovite
  • Users cannot add custom built-in capabilities without modifying Muscovite

AI Agent Guidance

When This Decision Applies

  • When determining where to place new infrastructure domains (security, audit, storage)
  • When deciding whether a feature should be built-in or example code
  • When designing the project.ddd configuration syntax
  • When implementing capability loading in generators

When This Decision Does NOT Apply

  • User-specific domain models (these go in user projects, not Muscovite)
  • One-off integrations or customizations
  • Example code demonstrating DSL features (these go in examples/)

Enforcement

  • Built-in capabilities MUST live in src/dslgen/builtin/<capability>/
  • Example projects MUST live in src/dslgen/examples/<project>/
  • All built-in capabilities MUST be opt-in via CAPABILITIES block
  • No external registry or import mechanism for capabilities

Key Files

  • src/dslgen/builtin/ - Built-in capability definitions
  • src/dslgen/builtin/registry.rkt - Capability registration and loading
  • src/dslgen/parser/project-dsl.rkt - CAPABILITIES parsing
  • src/dslgen/generate.rkt - Capability loading during generation
