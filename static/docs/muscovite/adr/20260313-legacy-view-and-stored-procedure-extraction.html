ADR: Legacy View and Stored Procedure Extraction for Brownfield Migration

ADR: Legacy View and Stored Procedure Extraction for Brownfield Migration

Date: 2026-03-13 Status: accepted Supersedes: — Extends: ADR 20260310-brownfield-extraction-architecture, ADR 20260217-mapping-dsl-block-brownfield-migration

Context

The brownfield extraction pipeline (ADR 20260310) captures entities, relationships, commands, events, sagas, and field-level transformations from legacy systems. Two critical categories of legacy database artifacts are not captured:

  1. Database views — named queries that represent derived/computed read models. In legacy PostgreSQL systems, views frequently encode business logic: reporting aggregations, denormalized read models, access-control filtered projections, and cross-context joins. Materialized views add caching semantics.

  2. Stored procedures and functions — imperative SQL logic encapsulated in the database. These range from simple utility functions to complex business rule engines. In systems like Gitea (229 tables, 0 FK constraints), much of the integrity and workflow logic lives in application code rather than stored procedures. But in enterprise PostgreSQL systems (ERP, EHR, financial), stored procedures are the primary location of business logic.

The gap matters because:

  • Views reveal read models: A legacy view joining 5 tables across 3 schemas exposes an integration concern that entity extraction alone cannot detect.
  • Procedures reveal commands: A stored procedure that mutates multiple tables in a transaction is a candidate COMMAND or SAGA step.
  • Cross-dependencies exist: Views can call stored functions (SELECT calculate_risk(id) FROM patient). Procedures can reference views. This dependency chain must be captured for correct bounded context assignment.
  • Human curation requires the body: Without seeing the SQL, a domain expert cannot decide whether a legacy view maps to a PROJECTION, a SPECIFICATION, or an entirely new aggregate.

Decision

1. Two New DSL Constructs: LEGACY_VIEW and LEGACY_PROCEDURE

Both are extraction-phase artifacts, not generation targets. They capture legacy database objects as-is, preserving metadata and body text for human curation. The LEGACY_ prefix explicitly marks them as pre-curation artifacts — greppable for "how much legacy remains to be modeled?"

LEGACY_VIEW

Captures the shape (columns + types), source tables, materialization, and SQL body:

LEGACY_VIEW MonthlyRevenueSummary @uuid "..." (
    customer_id   UUID        NOT_NULL,
    customer_name TEXT        NOT_NULL,
    month         DATE        NOT_NULL,
    total_revenue DECIMAL     NOT_NULL,
    order_count   INTEGER     NOT_NULL
)
SOURCE_TABLES [customer, "order", order_line]
CALLS [calculate_loyalty_tier]
MATERIALIZED false
DOC "Aggregates monthly revenue per customer with order count"
BODY """
    SELECT c.id, c.name, date_trunc('month', o.created_at),
           SUM(ol.price * ol.quantity), COUNT(DISTINCT o.id)
    FROM customer c
    JOIN "order" o ON o.customer_id = c.id
    JOIN order_line ol ON ol.order_id = o.id
    GROUP BY 1, 2, 3
""";

Subclauses:

Clause Required Type Purpose
Column list yes (name TYPE [NOT_NULL]) View output shape — the contract
SOURCE_TABLES yes [table, ...] Tables referenced — for context assignment
CALLS no [function, ...] Stored functions invoked — dependency chain
MATERIALIZED no true/false Default false. Affects refresh strategy
DOC no string Human description of purpose
BODY no triple-quoted string Original SQL definition for reference

LEGACY_PROCEDURE

Captures the signature, referenced objects, classification hint, and SQL body:

LEGACY_PROCEDURE ApplyTieredDiscount @uuid "..."
    PARAMS (order_id UUID IN, discount_pct DECIMAL OUT)
    RETURNS VOID
    REFERENCES [order, order_line, customer_loyalty]
    CLASSIFICATION COMMAND
    DOC "Tiered discount: 5% over $100, 10% over $500, 15% over $1000"
    BODY """
        DECLARE total DECIMAL;
        BEGIN
            SELECT SUM(price * quantity) INTO total
            FROM order_line WHERE order_id = $1;
            discount_pct := CASE
                WHEN total > 1000 THEN 0.15
                WHEN total > 500  THEN 0.10
                WHEN total > 100  THEN 0.05
                ELSE 0.0
            END;
        END;
    """;

Subclauses:

Clause Required Type Purpose
PARAMS yes (name TYPE direction) Parameters with IN/OUT/INOUT direction
RETURNS yes type or VOID or SETOF type Return type
REFERENCES no [table, ...] Tables read/written — for context assignment
CALLS no [function, ...] Other procedures/functions called
CLASSIFICATION no COMMAND/QUERY/TRIGGER/MIGRATION/UTILITY Extractor's best guess — human overrides
DOC no string Human description
BODY no triple-quoted string Original PL/pgSQL or SQL body

2. Triple-Quoted String for BODY

The SQL body is preserved as a triple-quoted string ("""..."""), not as DSL comments. Rationale:

  • Parseable: The body is structured content. Future tooling can analyze it (extract table references, detect mutation patterns, identify join complexity).
  • Round-trippable: Comments are lossy — they don't survive regeneration. Triple-quoted strings are first-class grammar elements.
  • Precedent: Python, Kotlin, and Scala use triple-quotes for multi-line strings. The Muscovite grammar already supports quoted strings; extending to triple-quote is a minimal grammar change.

The body is not validated as SQL — it is opaque content. The grammar accepts any text between the triple-quote delimiters.

3. LEGACY_ Prefix Semantics

The LEGACY_ prefix is not decorative — it carries semantic weight:

  • No code generation: Generators MUST skip LEGACY_VIEW and LEGACY_PROCEDURE constructs. They produce no SQL, no C++, no gRPC, no QML.
  • Curation workflow: Human curation replaces LEGACY_VIEW → PROJECTION or SPECIFICATION, and LEGACY_PROCEDURE → COMMAND, SAGA step, or custom DOMAIN_SERVICE logic. The LEGACY_ construct is then deleted.
  • Progress metric: grep -c LEGACY_ *.ddd measures remaining extraction debt.
  • Lint rule: --lint-only can warn on LEGACY_ constructs in non-brownfield projects (they should only appear in migration contexts).

4. Cross-Dependency Tracking

Both constructs support CALLS [...] and SOURCE_TABLES [...] / REFERENCES [...] clauses. These enable a dependency graph:

LEGACY_VIEW A → CALLS → LEGACY_PROCEDURE F
LEGACY_VIEW A → SOURCE_TABLES → ENTITY Customer, ENTITY Order
LEGACY_PROCEDURE F → REFERENCES → ENTITY Order, ENTITY OrderLine
LEGACY_VIEW B → SOURCE_TABLES → LEGACY_VIEW A  (view-on-view)

The extractor resolves these from the PostgreSQL catalog:

-- View → table dependencies
SELECT DISTINCT d.refobjid::regclass AS referenced
FROM pg_depend d
JOIN pg_rewrite r ON d.objid = r.oid
WHERE r.ev_class = 'view_name'::regclass
  AND d.refobjid != r.ev_class;

-- Function → table dependencies
SELECT DISTINCT dep.refobjid::regclass AS referenced
FROM pg_depend dep
WHERE dep.objid = 'function_name'::regproc
  AND dep.deptype = 'n';

-- View → function dependencies
SELECT DISTINCT p.proname
FROM pg_depend d
JOIN pg_proc p ON d.refobjid = p.oid
JOIN pg_rewrite r ON d.objid = r.oid
WHERE r.ev_class = 'view_name'::regclass;

5. Classification Taxonomy

The CLASSIFICATION clause uses a fixed vocabulary. The extractor assigns based on heuristics; humans override during curation.

Classification Heuristic Maps To
COMMAND Contains INSERT/UPDATE/DELETE, takes entity PK as input COMMAND in target model
QUERY SELECT only, returns SETOF or TABLE SPECIFICATION or PROJECTION
TRIGGER Referenced by pg_trigger, called on row events EVENT handler or POLICY
MIGRATION Contains DDL (ALTER, CREATE), typically one-shot MAPPING or discard
UTILITY Type coercion, formatting, no table references Helper — may not need modeling

6. Extraction Source: PostgreSQL Catalog

The extractor queries the PostgreSQL system catalog. No pg_dump parsing — direct catalog access provides structured metadata.

Catalog Source Provides
pg_views View name, schema, SQL definition
pg_matviews Materialized view name, schema, SQL definition
information_schema.columns View column names and types
pg_proc + pg_namespace Procedure/function name, args, return type, body, kind
pg_depend + pg_rewrite View→table, view→function, function→table dependencies
pg_trigger Trigger→function associations (for CLASSIFICATION)

7. No Generation — Extraction Only

These constructs are input artifacts for the curation workflow. They MUST NOT produce generated output in any target:

Target Behavior
PostgreSQL/Sqitch Skip — do not emit CREATE VIEW or CREATE FUNCTION
C++ DBA Skip — no entity struct, no repository
gRPC/Proto Skip — no service method
Qt Widgets/QML Skip — no UI form
Documentation Include — list in system manual as "legacy artifacts pending curation"
Fact Base JSON Include — with "legacy": true flag for analysis tooling

Consequences

Positive

  • Brownfield extraction captures the complete database surface: tables, views, and procedures — not just entities.
  • Human curators see the actual SQL body alongside the extracted metadata, enabling informed decisions about DDD classification.
  • Cross-dependency tracking enables automated context boundary suggestions.
  • The LEGACY_ prefix creates a measurable curation backlog.
  • No impact on existing generation — generators skip these constructs.

Negative

  • Grammar grows by ~2 constructs and ~8 keywords. Parser complexity increases.
  • Triple-quoted strings are a new lexer feature (currently only single-quoted and double-quoted strings exist).
  • The BODY clause may contain large SQL blocks (100+ lines), inflating .ddd file sizes during extraction. Acceptable for brownfield — not for target models.
  • CLASSIFICATION heuristics will have false positives. The taxonomy may need extension as more legacy systems are extracted.

Neutral

  • No existing .ddd files are affected — these are new constructs.
  • No migration needed — purely additive grammar change.
  • The LEGACY_ prefix convention can be applied retroactively to other extraction artifacts if needed (e.g., LEGACY_TRIGGER).
