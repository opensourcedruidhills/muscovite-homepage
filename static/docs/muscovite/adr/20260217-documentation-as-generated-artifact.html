Documentation as Generated Artifact

Documentation as Generated Artifact

  • Status: accepted
  • Date: 2026-02-17
  • Tags: documentation, doxygen, asciidoc, plantuml, sql-comments, grounding
  • Deciders: Johannes Lochmann

Technical Story: All DSL metadata (DOC, GROUNDING, @SOURCE, SYNONYMS) must propagate to every generated layer in the format canonical to that layer. Generated systems must be self-documenting.

Context and Problem Statement

Muscovite's IR is rich in documentation metadata — every struct has a doc field, GROUNDING is parsed and stored, SYNONYMS exist on entities and value objects. However, most generators ignore this metadata:

  • SQL generators emit COMMENT ON DOMAIN for strong types only — no table or column comments for entities/fields
  • C++ generators use hardcoded @brief Entity: ClassName instead of entity-doc
  • Proto generators emit hardcoded section banners, ignoring entity-doc and field-doc entirely
  • bounded-context-grounding is parsed but consumed by zero generators
  • No standalone system manual beyond the existing AsciiDoc glossary

This means generated systems lose the domain knowledge encoded in .ddd files. BI tools see undocumented tables, IDEs show generic class names instead of domain descriptions, and there's no human-readable system manual.

Decision Drivers

  • .ddd files contain rich DOC strings and GROUNDING blocks already (patient.ddd has DOC on every entity)
  • IR already stores all documentation metadata — it's parsed but not consumed
  • Existing AsciiDoc glossary generator (generators/docs/glossary-adoc.rkt, 564 lines) is a proven foundation
  • Database self-documentation enables BI tool discovery and compliance audits
  • C++ IDE tooltips should show domain-meaningful descriptions, not class name echoes
  • System manuals are required for client handoffs and enterprise compliance
  • Single source of truth (DSL) guarantees consistency across all documentation layers

Considered Options

  1. Enrich existing generators + extend AsciiDoc generator — propagate doc/grounding/@SOURCE to existing SQL, C++, proto generators; extend glossary-adoc for system manual
  2. New standalone documentation generators — separate sql-comments.rkt, doxygen.rkt, proto-comments.rkt, asciidoc-manual.rkt that run independently
  3. External documentation tooling — write docs manually outside the DSL pipeline

Decision Outcome

Chosen option: Option 1 — Enrich existing generators + extend AsciiDoc generator, because inline documentation (SQL comments, Doxygen headers, proto comments) cannot be generated as standalone files — it must be woven into the same output that the existing generators produce. A standalone approach would require either post-processing merges or duplicated template logic.

Positive Consequences

  • Database is self-documenting for BI tools, pg_admin, and compliance audits
  • Generated C++ code has IDE-visible domain documentation
  • Proto files carry domain semantics for API consumers
  • Complete system manual generated from DSL with architecture diagrams
  • Single source of truth maintained across all layers
  • No new generator registration overhead — enrichment happens within existing generators

Negative Consequences

  • Increased complexity in existing generator files (~15 files need edits)
  • Additional test surface for documentation content verification
  • PlantUML + asciidoctor become build-time dependencies for manual generation
  • Grammar extension required for @SOURCE annotation

Detailed Decisions

1. SQL: PostgreSQL COMMENT ON Statements — Complete Coverage

Every PostgreSQL object that supports COMMENT ON MUST have a generated comment if we can derive sensible content. This is not limited to tables and columns.

Object Type Comment Source
COMMENT ON SCHEMA bounded-context-grounding (domain, dialect, key concepts) + bounded-context-doc
COMMENT ON TABLE entity-doc + event-sourcing note + aggregate membership
COMMENT ON COLUMN field-doc + type semantics + @SOURCE + constraint summary
COMMENT ON CONSTRAINT (FK) Relationship name, direction, cardinality, target entity, CASCADE behavior
COMMENT ON CONSTRAINT (CHECK) invariant-description — the human-readable invariant text
COMMENT ON CONSTRAINT (UNIQUE) Which field(s), business rationale from field-doc if available
COMMENT ON INDEX Purpose: FK lookup, unique enforcement, spatial query, event store access pattern
COMMENT ON DOMAIN Already implemented — strong-type-doc + check constraints
COMMENT ON FUNCTION / TRIGGER Framework purpose: audit trigger, UUID generation, entity code auto-generation
COMMENT ON MATERIALIZED VIEW projection-doc + refresh mode + source entities

Comments are emitted in Sqitch deploy scripts immediately after the CREATE statement for each object. For constraints defined inline in CREATE TABLE, comments are emitted after the table creation.

2. C++: Doxygen Comments from IR

Replace hardcoded @brief strings in domain.rkt, repository.rkt, specifications.rkt, and DBA generators with actual entity-doc, field-doc, value-object-doc values. Add @see for @SOURCE references. Add file-level GROUNDING context.

3. Proto: Documentation Comments from IR

Add // documentation comments before proto message and field declarations using entity-doc and field-doc. Proto convention is // line comments.

4. System Manual: AsciiDoc + PlantUML

Extend existing glossary-adoc.rkt with GROUNDING sections, event sourcing flow diagrams, saga workflow diagrams, NATS topology diagrams, and API reference sections. All diagrams in PlantUML (standardize on one tool).

5. Diagram Format: PlantUML Only

Standardize on PlantUML. Convert existing Mermaid generation in context-map.rkt to PlantUML. The existing glossary-adoc.rkt already generates PlantUML ER diagrams. One diagram syntax, one dependency, consistent rendering.

6. CMake Build Integration

Add Documentation.cmake module with:

  • find_package(Doxygen) → add_custom_target(doxygen-docs)
  • find_program(asciidoctor) → add_custom_target(system-manual-html) and add_custom_target(system-manual-pdf)
  • Umbrella add_custom_target(docs) depending on both

7. @SOURCE Annotation

New grammar construct: @SOURCE STRING on fields and entities. Multiple @SOURCE annotations per element allowed. Stored as (Listof String) in IR. Propagated to SQL comments, Doxygen @see, proto comments, and AsciiDoc source reference tables.

Pros and Cons of the Options

Option 1: Enrich existing generators (chosen)

  • Good, because documentation is emitted in the same pass as code — no merge step
  • Good, because leverages existing generator infrastructure and testing
  • Good, because incremental — each generator can be enriched independently
  • Bad, because touches ~15 existing generator files (higher change surface)

Option 2: Standalone documentation generators

  • Good, because isolated from existing generators — lower regression risk
  • Bad, because C++ Doxygen and proto comments cannot be standalone files — they must be embedded in the generated output
  • Bad, because SQL COMMENT ON must appear in Sqitch deploy scripts, not separate files
  • Bad, because duplicates template logic (entity iteration, naming conventions)

Option 3: External documentation tooling

  • Good, because no generator changes needed
  • Bad, because violates single source of truth — documentation drifts from DSL
  • Bad, because manual maintenance defeats the purpose of code generation

Mandatory Documentation Maintenance (NO EXCEPTIONS)

All changes to Racket generator code and muscovite-cpp framework code MUST update existing documentation in the same commit.

This applies to:

  • Racket generators: If you change a generator's output format, update the corresponding Doxygen/SQL-comment/proto-comment generation in the same change
  • Racket source: If you change a function's behavior or purpose, update its ;; WHY: / ;; ADR: comments
  • C++ framework headers: If you change a public API, update the Doxygen @brief, @param, @return, @see comments
  • C++ framework implementation: If you change behavior, update @note and rationale comments

FORBIDDEN:

  • Submitting code changes that make existing documentation stale
  • Adding new public functions/methods without Doxygen documentation
  • Changing Racket function signatures without updating the doc comment
  • "Will document later" — documentation is part of the change, not a follow-up

This rule is enforced in copilot-instructions.md Rule 14 and AGENTS.md.

AI Agent Guidance

When This Decision Applies

  • When modifying any generator that emits C++ headers — include Doxygen comments from IR doc fields
  • When modifying SQL/Sqitch generators — include COMMENT ON for ALL generated objects (tables, columns, constraints, indexes, functions, triggers, views)
  • When modifying proto generators — include // documentation comments
  • When extending the AsciiDoc glossary generator — use PlantUML for all diagrams
  • When adding new DSL constructs — ensure doc field exists on IR struct and is consumed by all generators

When This Decision Does NOT Apply

  • Hand-authored muscovite-cpp framework documentation (covered by separate ADR 20260218)
  • Racket source code comments (governed by comment discipline instructions, not generators)

Enforcement

  • Rackunit tests verify documentation content appears in generated output
  • Grammar conformance test verifies @SOURCE is accepted
  • CMake targets (doxygen-docs, system-manual-html, system-manual-pdf) verified in CI
