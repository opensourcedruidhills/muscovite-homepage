Behavioral Sovereignty Roadmap — From Anemic to Rich Domain Model

Behavioral Sovereignty Roadmap — From Anemic to Rich Domain Model

  • Status: accepted
  • Date: 2026-03-07
  • Deciders: Johannes Lochmann
  • Tags: roadmap, behavioral-sovereignty, rich-domain-model

Context

Muscovite currently delivers Structural Sovereignty (types, relations, persistence) and State Sovereignty (bitemporality, sagas, state machines). Business logic remains in manually-implemented sidecar stubs or opaque DomainService methods.

This ADR establishes the roadmap for Behavioral Sovereignty — moving business logic into the DSL so it can be:

  • Parsed and validated at compile time
  • Generated across all targets (C++, SQL, docs, tests)
  • Visualized in system documentation
  • Fuzz-tested via property-based test generation
  • Reasoned about by AI agents working with the codebase

Decision

Behavioral Sovereignty is delivered across four iterations, each independently releasable:

v1.60.0 — Collection Predicates (Tier 1 Expression Language)

Extends the predicate language with collection operations. Every REQUIRES and WHERE clause becomes dramatically more expressive.

New expression nodes:

  • NOT IN, IS EMPTY, IS NOT EMPTY
  • INTERSECTS, SUBSET OF, CONTAINS ALL
  • CONTAINS(list, value)

Depends on: v1.55.0 (DomainService exists as generation target)

v1.61.0 — EFFECT Catalog + EMIT/PLAN + Computation Language (Tier 2)

Introduces behavioral constructs and the computation language needed to express them.

New DSL constructs:

  • EFFECT top-level construct (side effect declarations)
  • EMIT/PLAN intent separation in COMMAND
  • DECIDE block on COMMAND (optional, pure logic body)
  • ENSURES postcondition clause

New expression nodes:

  • IF cond THEN expr ELSE expr
  • LET name = expr IN body
  • MATCH field { CASE val => expr }

Depends on: v1.60.0 (collection predicates for REQUIRES/ENSURES)

v1.62.0 — List Comprehensions + Complex Business Logic (Tier 3)

Completes the functional expression language. Enables complex calculations like booking price computation, dynamic packaging, amendment/cancellation logic.

New expression nodes:

  • MAP(list, var -> expr)
  • FILTER(list, var -> expr)

v1.62.0 — List Comprehensions + Property-Based Test Generation + Doc Visualization

Completes the functional expression language AND exploits the rich domain model for automated quality assurance and documentation. Consolidated from originally separate v1.62.0 + v1.63.0.

Tier 3 — List Comprehensions:

  • MAP(list, var -> expr), FILTER(list, var -> predicate)
  • REDUCE(list, init, acc var -> expr)
  • Built-in aggregations: SUM, COUNT, MIN, MAX, AVG

Tier 4 — Testing & Documentation:

  • Property-based test generator: fuzz DECIDE inputs, verify ENSURES
  • Mermaid/PlantUML decision flow diagrams from DECIDE blocks
  • System manual "Business Logic" chapter generation
  • Test coverage reporting for DECIDE branch paths

Depends on: v1.61.0 (LET bindings for intermediate results)

Versioning Note

These versions follow v1.55.0–v1.59.0 (Domain Layer initiative + Stabilization sprints). The Domain Layer provides the DomainService generation target and three-tier validation pipeline that Behavioral Sovereignty builds upon. Stabilization sprints (v1.55.1–v1.55.2) ensure test infrastructure is solid before new feature development.

v1.54.0  Field constraints, strong type annotations
v1.55.0  DomainService, EntityValidator, PluginValidator, thin controllers
v1.55.1  STABILIZATION: Governance gates + golden-file regression tests
v1.55.2  STABILIZATION: Retroactive test coverage for shipped generators
v1.58.0  Cross-field invariants → EntityValidator codegen
v1.59.0  Frontend validation symmetry (Qt Widgets + QML)
v1.60.0  Collection predicates (Tier 1 expressions)          ← Behavioral Sovereignty starts
v1.61.0  EFFECT/DECIDE/ENSURES + computation language (Tier 2)
v1.62.0  List comprehensions + tests + doc visualization (Tiers 3+4, consolidated)

Consequences

  • .ddd amplification ratio will increase significantly (behavioral logic → generated C++ + SQL + tests + docs)
  • Commands with DECIDE blocks eliminate the need for sidecar stubs (fewer manual files)
  • The expression language becomes Muscovite's most complex subsystem — needs thorough testing
  • AI agents can reason about and modify business logic at the DSL level
  • Backward compatibility preserved: DECIDE is optional, existing commands unchanged
  • Each iteration is independently valuable — partial delivery still improves the system
