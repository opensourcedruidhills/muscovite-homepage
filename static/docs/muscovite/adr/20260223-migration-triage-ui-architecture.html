Migration Triage UI Architecture (Qt Widgets + QML)

Migration Triage UI Architecture (Qt Widgets + QML)

  • Status: accepted
  • Date: 2026-02-23
  • Tags: brownfield, migration, triage, qt-widgets, qml, dlq, generator
  • Deciders: Johannes Lochmann

Technical Story: Data quality failures during brownfield migration are inevitable. Records that fail strong-type validation or entity invariants are routed to a DLQ NATS subject. Without a generated triage UI, engineers must query DLQ contents in SQL, make corrections manually, and re-submit via curl/NATS CLI. This is unscalable for migrations involving thousands of dirty records.

Context and Problem Statement

The DLQ (Dead Letter Queue) for migration is a NATS subject (migration.dlq.{context}.{entity}) populated by the transformer whenever validation fails (ADR 20260223-migration-transformer). A human reviewer must:

  1. See all failed records — what field failed, why, what the original value was
  2. Correct the problematic field value in-place
  3. Approve (re-submit the corrected record to the transformer) or Reject (mark as permanently skipped)
  4. Export the DLQ contents as CSV for batch review or compliance audit

Nothing in the existing Muscovite Qt generator addresses DLQ/triage workflows. The standard {Entity}EditDialog is designed for normal CRUD — it has no concept of source fields, error annotations, approval/rejection actions, or re-submission to NATS.

Decision Drivers

  • Multi-frontend architecture (ADR 20260206): Qt Widgets + QML from one MAPPING block
  • Triage UI is generated from MAPPING block structure — field layouts mirror source→target field mapping
  • DLQ record structure is defined by the typed JSON envelope (ADR 20260217-typed-json) — triage UI reads those fields
  • The triage action (approve corrected record) publishes back to the transformer's NATS input subject — NatsMessageBroker is already in scope
  • Actions (Approve, Reject, Export CSV) are uniform across all triage dialogs — no per-entity customization needed

Considered Options

  1. Generated Qt Widgets {Context}{Entity}TriageDialog + QML {Context}{Entity}TriageView — master-detail layout per MAPPING block; all actions generated; driven by DLQ NATS subject
  2. Generic triage shell + generated JSON config — one shared widget parameterized by a generated descriptor
  3. Web-based triage UI — generated React/Vue component, served from a bundled HTTP server in the extractor binary
  4. No generated triage UI — rely on DLQ database table + psql for manual review

Decision Outcome

Chosen option: Option 1 — Generated per-MAPPING Qt Widgets dialog + QML view, because:

  • Consistent with the multi-frontend generation model (ADR 20260206)
  • MAPPING block fields map directly to dialog rows — no generic renderer needed
  • Approval re-submits to transformer input subject — reuses NatsMessageBroker already in the application
  • QML counterpart follows {Entity}ViewModel pattern (ADR 20260206) — no new patterns required

Qt Widgets: {Context}{Entity}TriageDialog

Layout: QSplitter horizontal.

  • Left panel (Master): QTableWidget showing DLQ records with columns: sequence, source_key, error_field, error_message, status (Pending / Approved / Rejected).
  • Right panel (Detail): QFormLayout with one row per MAPPING field, showing QLabel (source_value) + QLineEdit (corrected_value, pre-filled with source) + QLabel (error_annotation, red if this field failed).
  • Action bar: [Approve] [Reject] [Export CSV] buttons. Approve re-publishes corrected envelope to transformer input subject; Reject marks status in the DLQ record and acknowledges the NATS message.

QML: {Context}{Entity}TriageView.qml + {Context}{Entity}TriageViewModel

Follows the ViewModel pattern from ADR 20260206:

  • {Context}{Entity}TriageViewModel C++ class with Q_PROPERTY for each MAPPING field (source + corrected) and Q_PROPERTY for error, status
  • {Context}{Entity}TriageView.qml: SplitView horizontal — left ListView (master), right ColumnLayout (detail form + action buttons)
  • approve() and reject() as Q_INVOKABLE methods on the ViewModel

Approve / Reject Action Flow

Approve:
  1. Read corrected field values from QLineEdit widgets
  2. Reconstruct envelope with corrected values + "$resubmit": true
  3. NatsMessageBroker::publish("migration.extract.{context}.{entity}", corrected_json)
  4. Mark DLQ record status = Approved in local model + ACK NATS message

Reject:
  1. Mark DLQ record status = Rejected in local model + ACK NATS message
  2. Append to persistent rejection log in {project}_migration_rejections table

Positive Consequences

  • Human-in-the-loop triage has a generated UI — no bespoke tooling per migration project
  • Field-by-field error annotation makes the source of validation failure immediately visible
  • Approve action re-uses transformer input subject — no special approval code path
  • Export CSV enables batch offline review and compliance audit
  • QML counterpart enables triage on tablet/touch form factors (useful for warehouse/field migrations)

Negative Consequences

  • One dialog class per MAPPING block — larger generated codebase for projects with many mappings
  • Triage UI is DLQ consumer — it competes with other consumers if multiple triage windows open simultaneously (last-ack wins; acceptable for human-paced triage)
  • Export CSV is synchronous in v1.30.0 — large DLQ exports may block the UI thread

Constraints

  • Approve action MUST publish to transformer input subject — never write directly to target DB
  • Reject MUST persist to the rejection log table — no silent discards
  • Qt Widgets and QML variants MUST be generated together from the same MAPPING block
  • Generated code MUST carry dual-license SPDX header

AI Agent Guidance

When implementing the triage UI generator:

  • New file: dslgen/generators/qt/migration-triage-widgets.rkt (Qt Widgets variant)
  • New file: dslgen/generators/qt/migration-triage-qml.rkt (QML variant)
  • Follow patterns from dslgen/generators/qt/widgets-views.rkt for master-detail dialog structure
  • Follow patterns from dslgen/generators/qt/quick-views.rkt for QML ViewModel
  • For each MAPPING block: one dialog class, one ViewModel class, one QML file
  • DLQ NATS subject: (format "migration.dlq.~a.~a" (symbol->sql-id ctx-name) (symbol->sql-id entity-name))
  • Transformer input subject (for Approve): (format "migration.extract.~a.~a" (symbol->sql-id ctx-name) (symbol->sql-id entity-name))
  • Per-field widget: one QLineEdit per mapping-field with objectName = source field name for testing
  • Error annotation: QLabel with setStyleSheet("color: red;") shown only when $error_field matches this field
  • Test: ctest --preset debug -R triage — dialog loads mock DLQ records, approve button re-publishes to mock NATS subject
