Team-Type Contexts with Ownership and Downstream Rights

Team-Type Contexts with Ownership and Downstream Rights

  • Status: draft
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-31
  • Tags: dsl, ddd, strategic, team, ownership, governance, conways-law, grammar, ir
  • Supersedes: none

Technical Story: RFC-001 Extension 6 — add TEAM ownership and DOWNSTREAM_RIGHTS to contexts and relationships for organizational modeling.

Context and Problem Statement

Conway's Law: system architecture mirrors organizational structure. Muscovite models technical relationships between contexts (CUSTOMER_SUPPLIER, ACL, etc.) but not organizational ownership. During consulting engagements, understanding who owns what — and what governance rights downstream consumers have — is critical for planning migration sequences, staffing, and communication.

Currently, team ownership lives in external project documentation. It should be part of the model so generators can produce team-scoped documentation, ownership heatmaps, and organizational risk analysis.

Decision Drivers

  • Consulting deliverables require org-chart overlays on context maps
  • Downstream governance rights affect API evolution strategy
  • Governance vocabularies differ per organization — must be user-definable
  • Must be optional — not all projects model organizational concerns
  • Grammar-first mandate (ADR 20260211)

Considered Options

Option 1: TEAM, OWNER, GOVERNANCE_SCHEME with DOWNSTREAM_RIGHTS (chosen)

Teams defined in .mus, contexts declare OWNER, relationships declare DOWNSTREAM_RIGHTS using a user-defined GOVERNANCE_SCHEME.

  • Good, because organizational modeling is explicit and structured
  • Good, because governance vocabulary is user-definable (same scheme pattern)
  • Good, because generators produce team-scoped documentation automatically
  • Bad, because adds organizational concerns to a technical DSL

Option 2: OWNER as free-text annotation

CONTEXT Foo @uuid "..." OWNER "Backend Platform Team"

  • Good, because simple, no TEAM construct needed
  • Bad, because no validation (typos in team names)
  • Bad, because no team metadata (size, location, contact)
  • Bad, because downstream rights have no structured vocabulary

Option 3: External RACI matrix

Side-by-side YAML mapping contexts to team responsibilities.

  • Good, because no grammar changes
  • Bad, because drifts from the model
  • Bad, because not visible in context map visualizations

Decision Outcome

Chosen option: "TEAM, OWNER, GOVERNANCE_SCHEME with DOWNSTREAM_RIGHTS", because structured team definitions with validated ownership references provide the organizational modeling that consulting engagements require.

DSL Syntax

.mus project file — team definitions

TEAM BackendPlatform @uuid "..."
    DOC "Core platform team — 5 engineers, Berlin office";

TEAM ClinicalApps @uuid "..."
    DOC "Clinical applications team — 8 engineers, remote";

TEAM ExternalPartner @uuid "..."
    DOC "Third-party integration partner — Acme Health Ltd.";

.mus project file — governance scheme

GOVERNANCE_SCHEME DownstreamRights {
    CONFORMIST
        DOC "Downstream accepts upstream model without influence";
    OPINION_LEADER
        DOC "Downstream can suggest changes; upstream decides";
    DECISION_MAKER
        DOC "Downstream drives API evolution for their use cases";
    VETO_RIGHT
        DOC "Downstream can block upstream changes that break their contract";
    MONOPOLIST
        DOC "Downstream is the sole consumer — effectively co-owns the API";
}

.ddd model — ownership on contexts

CONTEXT PatientManagement @uuid "..."
    OWNER BackendPlatform
{ ... }

.ddd model — downstream rights on relationships

MAP STATE TARGET {
    PatientManagement -> ClinicalWorkflow : CUSTOMER_SUPPLIER VIA GRPC
        DOWNSTREAM_RIGHTS DECISION_MAKER
        DOC "Clinical team drives API evolution for patient queries";

    PatientManagement -> Reporting : CUSTOMER_SUPPLIER VIA EVENTS
        DOWNSTREAM_RIGHTS CONFORMIST
        DOC "Reporting consumes events as-is, no influence on schema";
}

PEG Grammar Changes

TeamDecl           <- 'TEAM' IDENTIFIER UuidAnnotation DocClause?
OwnerClause        <- 'OWNER' IDENTIFIER
DownstreamRightsClause <- 'DOWNSTREAM_RIGHTS' IDENTIFIER

OwnerClause is added as optional to the Context rule. DownstreamRightsClause is added as optional to the ContextRelationship rule.

IR Implications

  • New IR node type: Team with name, UUID, documentation
  • Context gains optional owner field (UUID reference to team)
  • ContextRelationship gains optional downstream_rights field (scheme:tag)

Generator Implications

  • Documentation: org-chart overlay on context map. Team-scoped docs show "your team owns these contexts, depends on these contexts, with these governance rights." Manager-scoped docs show staffing implications. C-level docs show organizational risk.
  • Heatmap: ownership density — which team owns the most contexts? Which contexts have no owner (risk)?

Positive Consequences

  • Conway's Law becomes visible in the model
  • Team-scoped documentation generated automatically
  • Organizational risk (unowned contexts, single-team bottlenecks) is derivable

Negative Consequences

  • Organizational information changes more frequently than technical structure
  • Team definitions may need synchronization with external HR/org systems

AI Agent Guidance

When This Decision Applies

  • When adding TEAM definitions or OWNER clauses to contexts
  • When adding DOWNSTREAM_RIGHTS to context map relationships
  • When generating team-scoped or manager-scoped documentation

When This Decision Does NOT Apply

  • Projects without organizational modeling needs
  • Technical relationship types (CUSTOMER_SUPPLIER etc.) — unchanged

Enforcement

  • PEG grammar rules for TEAM, OWNER, DOWNSTREAM_RIGHTS
  • Parser validation: OWNER references must resolve to defined TEAMs
  • DOWNSTREAM_RIGHTS references must resolve to GOVERNANCE_SCHEME tags
