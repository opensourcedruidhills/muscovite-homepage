InProcessDynamic Sidecar Transport

InProcessDynamic Sidecar Transport

  • Status: accepted
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-30
  • Tags: plugins, sidecar, transport, dlopen, muscomp
  • Supersedes: Extends Plugin Transport Selection

Technical Story: Add a fourth sidecar transport type that loads a shared library at runtime via dlopen, enabling dynamic behavioral customization without process spawn overhead. This transport is the bridge between the shared library extraction (migration path) and the full 2.0 plugin architecture.

Context and Problem Statement

ADR 20260223 defined three sidecar transport types: SyncVirtual (compile-time vtable), LocalGRPC (localhost gRPC), and QtRemoteObjects (Qt frontend). These serve generated applications well, but the muscomp 2.0 architecture (ADR 20260330-muscovite-2) creates a new need:

Built-in generators that ship with muscomp benefit from in-process execution (no serialization overhead, no process spawn), but the monolithic static library prevents independent compilation and selective loading. A fourth transport type — runtime-loaded shared libraries — provides near-zero overhead while enabling:

  • Independent compilation per generator (improved build times)
  • Selective loading (only load generators the project needs)
  • Same interface contract as other transports (no special-casing)
  • A natural migration path from the current monolith

Decision Drivers

  • Built-in generators should have near-zero invocation overhead
  • Generators should compile independently for build time isolation
  • The same EXTENSION_POINT interface must work across all transports
  • The shared libraries extracted during migration must become the final artifact, not throwaway scaffolding
  • muscomp should dogfood its own sidecar transport system

Considered Options

  1. InProcessDynamic — dlopen a .so at runtime, resolve factory function
  2. Always standalone executables — all generators, including built-in, spawn as separate processes reading the IR file
  3. Static linking only — keep built-in generators compiled into muscomp binary

Decision Outcome

Chosen option: "InProcessDynamic", because it provides near-zero overhead for built-in generators while maintaining the same EXTENSION_POINT contract used by all other transports. Combined with Option 2 (standalone executables for third-party generators per ADR 20260330-muscovite-2), this gives a complete spectrum of deployment options per extension point.

Transport Matrix (Updated)

Transport Mechanism Overhead Use Case
SyncVirtual Compile-time vtable Zero Validators in generated apps
InProcessDynamic dlopen + factory function ~1ms load, then vtable Built-in muscomp generators
LocalGRPC gRPC over 127.0.0.1 Process + serialization Side-effect plugins

Interface

Each InProcessDynamic plugin exports a C factory function:

// In each generator .so
extern "C" {
    const char* muscomp_plugin_api_version();    // ABI compatibility guard
    const char* muscomp_plugin_name();           // Human-readable name
    std::unique_ptr<AbstractGenerator> muscomp_create_generator();
}

The AbstractGenerator interface:

class AbstractGenerator {
public:
    virtual ~AbstractGenerator() = default;
    virtual std::string name() const = 0;
    virtual std::vector<GeneratedFile> generate(
        const DddModel& model,
        const ProjectConfig& config,
        const Options& opts) const = 0;
};

Equivalent interfaces exist for AbstractParser and AbstractValidator.

.mus Project File Integration

PLUGIN GeneratePostgres {
    EXTENSION_POINT GeneratePostgres
    TRANSPORT InProcessDynamic
    LIBRARY "muscomp-gen-postgres"      -- resolves to libmuscomp-gen-postgres.so
}

The transport is a deployment decision in the project file, not an architectural constraint. The same extension point implementation could be loaded as InProcessDynamic in one project and invoked as a standalone executable in another.

Shared Libraries as Stepping Stone

The shared library extraction from the monolithic muscomp_lib (Phase A of the migration path, see ADR 20260330-library-extraction) produces .so files that directly become InProcessDynamic sidecar implementations:

Phase A: muscomp_lib (monolith) → libmuscomp-gen-postgres.so (shared library)
    ↓ (no additional work)
Phase B: libmuscomp-gen-postgres.so is the InProcessDynamic sidecar

The migration stepping stone IS the final artifact.

Positive Consequences

  • Near-zero invocation overhead for built-in generators (~1ms dlopen, then vtable)
  • Independent compilation — change one generator, rebuild one .so
  • Selective loading — muscomp generate --target=postgres loads only that .so
  • Same EXTENSION_POINT interface across all transports
  • Shared library extraction work is not throwaway — it produces the final plugins
  • muscomp dogfoods its own sidecar transport system

Negative Consequences

  • dlopen/dlsym platform differences (mitigated by CMake MODULE library type)
  • ABI compatibility between muscomp and plugins (mitigated by api_version() guard and shipping built-in plugins from the same build)
  • Slightly more complex build system (per-library CMake targets)

AI Agent Guidance

When This Decision Applies

  • When implementing the muscomp 2.0 generator loading mechanism
  • When extracting shared libraries from the muscomp monolith
  • When adding a new built-in generator to muscomp
  • When configuring PLUGIN blocks in .mus project files with transport selection

When This Decision Does NOT Apply

  • Third-party generators — these use standalone executables reading the IR file
  • Generated application plugins — these continue to use SyncVirtual, LocalGRPC, or QtRemoteObjects as appropriate
  • muscomp v1.78.x work — the monolith is acceptable until the migration path begins

Enforcement

  • Each .so plugin must export the three C factory functions
  • api_version() is checked at load time; mismatch produces a clear error
  • CMake builds each generator as a MODULE library target
  • Integration tests verify that InProcessDynamic loading produces identical output to direct function calls

Key Files

  • toolchain/muscomp/CMakeLists.txt — build targets (to be split per generator)
