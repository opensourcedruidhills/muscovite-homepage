Defer Redis Caching to v1.x

Defer Redis Caching to v1.x

  • Status: accepted
  • Date: 2026-02-03
  • Deciders: User (jolo)
  • Tags: caching, redis, type-safety, serialization

Context and Problem Statement

Redis caching was added in v0.9.0 Phase 3 to provide lookup table caching for Basedata entities. However, the implementation was introduced without:

  1. User consultation or explicit approval
  2. An Architecture Decision Record
  3. Consideration of type safety implications

The implementation used nlohmann::json for serialization, which violates the project's type safety principles (no JSON in domain layer per ADR 20260120).

Decision Drivers

  • Type Safety: All data crossing boundaries should use strongly typed serialization
  • Consultation: Architectural additions require user review
  • Simplicity: PostgreSQL-native read models (VIEWs, MATERIALIZED VIEWs) handle most caching needs
  • Consistency: If Redis is added, it must use Protobuf serialization like the outbox pattern

Considered Options

  1. Keep Redis with JSON - Fast but violates type safety
  2. Refactor Redis to use Protobuf - Type-safe but requires significant work
  3. Remove Redis, defer to v1.x - Simple, PostgreSQL handles read models
  4. Remove Redis permanently - Closes option for future optimization

Decision Outcome

Chosen option: Remove Redis, defer to v1.x

Redis caching is removed from v0.9.0. Read model needs are addressed by:

  • PostgreSQL VIEWs (REFRESH LIVE projections)
  • PostgreSQL MATERIALIZED VIEWs (REFRESH ON_DEMAND, SCHEDULED)
  • EventConsumer-driven tables (REFRESH ON_EVENT)

If Redis is reconsidered for v1.x, it MUST use Protobuf serialization for cache values.

Consequences

Good:

  • Simpler architecture with fewer external dependencies
  • No Docker/infrastructure complexity for Redis
  • Type safety preserved throughout the stack
  • PostgreSQL-native approach aligns with existing patterns

Bad:

  • No distributed caching for horizontal scaling
  • Slightly higher latency for repeated lookup reads

Neutral:

  • ICache interface and MemoryCache remain for testing
  • CachedRepository template retained but unused in production

Future Considerations

If Redis is implemented in v1.x:

  1. Serialization: Use Protobuf, not JSON
  2. ADR: Document decision before implementation
  3. User approval: Explicit sign-off required
  4. Integration: Consider NATS KV as alternative to Redis

AI Agent Guidance

When This Applies

  • Any caching layer additions
  • External service dependencies (Redis, Memcached, etc.)
  • Serialization format decisions

What To Do

  • Propose ADR before implementing external dependencies
  • Use Protobuf for cross-boundary serialization
  • Get explicit user approval for infrastructure additions
  • Prefer PostgreSQL-native solutions when sufficient

What Not To Do

  • Add infrastructure without ADR and user consultation
  • Use JSON for domain data serialization
  • Assume caching is needed without profiling
