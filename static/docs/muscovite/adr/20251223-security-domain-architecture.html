Security Domain Architecture

Security Domain Architecture

Technical Story: Implementing user authentication, authorization, and session management for Muscovite server and clients.

Context and Problem Statement

Muscovite needs a comprehensive security layer to protect gRPC services. We need to decide on authentication mechanism, password storage, token strategy, and authorization model. The solution must integrate with both the server library (muscoviteserver) and Qt frontend (muscoviteqt).

Decision Drivers

  • Security best practices (OWASP guidelines)
  • Self-hosted deployment (no external identity provider required)
  • String-based usernames (not email)
  • Group-based access control
  • Token-based authentication for stateless API calls
  • Session management with revocation capability
  • Password security (hashing, not encryption)

Considered Options

  1. JWT with Refresh Tokens - Stateless access tokens, stateful refresh tokens
  2. Session-Only Authentication - Server-side sessions with session cookies
  3. OAuth2 with External Provider - Delegate to Keycloak/Auth0
  4. API Keys Only - Static keys per user

Decision Outcome

Chosen option: "JWT with Refresh Tokens", because it provides stateless API authentication while maintaining server-side session control for revocation and security.

Positive Consequences

  • Stateless API validation (scalable)
  • Session revocation via token blacklist
  • Refresh token rotation detects theft
  • Group-based privileges are flexible
  • Self-contained (no external dependencies)

Negative Consequences

  • Must manage token blacklist (short TTL helps)
  • JWT secret/key must be securely managed
  • Complexity of refresh token rotation

Key Decisions

Password Hashing: Argon2id

Decision: Use Argon2id for password hashing.

Rationale: Winner of Password Hashing Competition, recommended by OWASP, resistant to GPU/ASIC attacks, memory-hard.

Parameters (OWASP recommendations):

  • Time cost: 3 iterations
  • Memory cost: 64 MB
  • Parallelism: 4 threads
  • Hash length: 32 bytes

Username Format: String (not email)

Decision: Use simple string usernames, not email addresses.

Rationale:

  • Simpler validation
  • No email verification complexity
  • Enterprise environments often use non-email usernames
  • Email can be stored separately if needed later

Format: ^[a-zA-Z][a-zA-Z0-9_.-]*$ (3-64 chars)

Token Strategy: Short-Lived Access + Rotating Refresh

Decision:

  • Access tokens: 15-minute JWT
  • Refresh tokens: 7-day, rotated on each use, stored hashed

Rationale: Short access tokens limit exposure window. Refresh token rotation allows detection of token theft (both old and new tokens get invalidated).

Authorization Model: Group-Based Privileges

Decision: Users belong to groups; groups have privileges.

Rationale:

  • Simpler than per-user permissions
  • Matches enterprise patterns (RBAC)
  • Easy to audit and manage
  • Privileges auto-generated from bounded contexts

Privilege Structure

Auto-generated from DSL bounded contexts:

{context}.read   - Read entities in context
{context}.write  - Create/update entities
{context}.delete - Delete entities
{context}.admin  - Full access to context

security.users.read   - View users
security.users.write  - Manage users
security.groups.write - Manage groups
security.admin        - Full security admin

AI Agent Guidance

When This Decision Applies

  • Implementing authentication in server or client
  • Adding new protected gRPC services
  • Implementing user/group management
  • Generating privilege seeds from DSL

When This Decision Does NOT Apply

  • Service-to-service authentication (different pattern)
  • External API integrations
  • Development/testing with auth disabled

Enforcement

  • All gRPC services use AuthInterceptor (except public methods)
  • Password hashing via PasswordService only
  • Token generation via JwtService only
  • Never store plaintext passwords
