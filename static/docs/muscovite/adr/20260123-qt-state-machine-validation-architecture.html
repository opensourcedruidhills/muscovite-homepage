Qt State Machine Validation Architecture

Qt State Machine Validation Architecture

  • Status: accepted
  • Date: 2026-01-23
  • Tags: qt, validation, state-machine, ui, ddd

Context and Problem Statement

Entity edit dialogs require complex validation workflows:

  • Field-level validation on change (strong types like Email, Phone)
  • Required field validation on blur (NOT NULL constraints)
  • Async validation on blur (uniqueness checks via gRPC)
  • Form-level validation on save (arc constraints, cross-field rules)
  • State tracking (dirty detection, save enablement)
  • Error display with i18n support

Ad-hoc validation code leads to inconsistent behavior and difficult maintenance. How do we provide a structured, DDD-aligned validation framework for generated Qt dialogs?

Decision Drivers

  • Consistency: All generated dialogs behave the same way
  • DDD Alignment: Validation rules derive from domain model constraints
  • User Experience: Immediate feedback, clear error messages
  • Testability: Validation logic can be unit tested
  • Extensibility: Custom validations can be added without modifying generated code
  • I18n Ready: Error codes and messages support translation

Considered Options

  1. Inline validation code: Each dialog has custom validation logic
  2. Validator pattern (Qt): Use QValidator subclasses only
  3. State machine with validator composition: QStateMachine + EntityValidator (chosen)

Decision Outcome

Chosen option: "State machine with validator composition", because:

  • QStateMachine provides explicit, debuggable state transitions
  • Validators can be composed from DSL constraints
  • Separation of concerns: state tracking vs. validation logic
  • Framework can be extended for custom application rules

Validation Timing

Timing Trigger What's Validated DSL Source
OnChange textChanged Strong types (Email, Phone) STRONG_TYPE CHECK
OnBlur editingFinished Required fields NOT NULL
OnBlurAsync editingFinished Uniqueness @unique annotation
OnSave validate() Arc constraints, cross-field CHECK constraints

Validation Error Codes

namespace muscovite::qt::validation {

enum class ValidationCode {
    // VAL001-009: Required field errors
    REQUIRED_FIELD_EMPTY = 1,
    
    // VAL010-049: Format errors
    INVALID_EMAIL_FORMAT = 10,
    INVALID_PHONE_FORMAT = 11,
    INVALID_POSTAL_CODE = 12,
    INVALID_URL_FORMAT = 13,
    
    // VAL050-079: Range errors
    VALUE_BELOW_MINIMUM = 50,
    VALUE_ABOVE_MAXIMUM = 51,
    DATE_IN_FUTURE = 52,
    DATE_IN_PAST = 53,
    
    // VAL080-089: Uniqueness errors
    VALUE_NOT_UNIQUE = 80,
    
    // VAL090-099: Arc constraint errors
    ARC_CONSTRAINT_VIOLATION = 90,
    TOO_FEW_FIELDS_SET = 91,
    TOO_MANY_FIELDS_SET = 92,
    
    // VAL100+: Custom application errors
    CUSTOM_VALIDATION_BASE = 100
};

} // namespace

Generated Validator Example

// Generated from Patient entity with Email and Phone strong types
class PatientValidator : public muscovite::qt::EntityValidator {
public:
    explicit PatientValidator(PatientEditDialog* dialog);
    
    QList<FieldValidationRule> buildFieldRules() const override {
        return {
            // Strong type validation on change
            {dialog_->email_, ValidationTiming::OnChange, 
             ValidationCode::INVALID_EMAIL_FORMAT, emailPattern_},
            {dialog_->phone_, ValidationTiming::OnChange,
             ValidationCode::INVALID_PHONE_FORMAT, phonePattern_},
            
            // Required field validation on blur
            {dialog_->mrn_, ValidationTiming::OnBlur,
             ValidationCode::REQUIRED_FIELD_EMPTY, nullptr},
            {dialog_->familyName_, ValidationTiming::OnBlur,
             ValidationCode::REQUIRED_FIELD_EMPTY, nullptr},
             
            // Async uniqueness check
            {dialog_->mrn_, ValidationTiming::OnBlurAsync,
             ValidationCode::VALUE_NOT_UNIQUE, &checkUniqueMrn}
        };
    }
    
protected:
    // Hook for application-specific validations
    void addCustomValidations() override {
        // Subclass can add custom rules here
    }
    
private:
    static const QRegularExpression emailPattern_;
    static const QRegularExpression phonePattern_;
};

Positive Consequences

  • Explicit state machine makes dialog behavior predictable
  • Validation rules automatically derived from DSL constraints
  • Error codes enable i18n without changing logic
  • Custom validations via virtual hook, not generator modification
  • ValidationDisplayWidget provides consistent error display

Negative Consequences

  • Additional framework code to maintain
  • State machine adds complexity vs. simple boolean flags
  • Async validation requires careful handling of races
