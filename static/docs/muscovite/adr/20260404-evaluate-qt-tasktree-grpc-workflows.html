Evaluate Qt TaskTree for gRPC Async Workflows

Evaluate Qt TaskTree for gRPC Async Workflows

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04
  • Tags: qt, grpc, async, architecture

Technical Story: FTR-534

Context and Problem Statement

Current Muscovite gRPC workflows use GrpcCallTask + QtConcurrent::run() with manual signal/slot wiring for each call. This pattern produces ~40 lines of boilerplate per gRPC method (connect signals, handle errors, marshal response). Qt 6.11 ships TaskTree (Tasking framework) as a declarative async workflow engine. Should Muscovite adopt TaskTree for generated gRPC binding code?

Decision Drivers

  • Each generated gRPC binding has 4 methods (load/save/create/remove) × ~40 LOC = ~160 LOC boilerplate
  • TaskTree provides declarative task composition with automatic cancellation and error propagation
  • Qt 6.11 promotes TaskTree from tech preview to stable API
  • Generated code is fully under our control — migration risk is low

Considered Options

  1. Adopt TaskTree now — replace GrpcCallTask + QtConcurrent with TaskTree recipes
  2. Adopt incrementally — use TaskTree for new generators, keep existing pattern
  3. Do not adopt — current pattern is functional, changing adds risk

Decision Outcome

Chosen option: "Adopt incrementally", because:

  1. Stable API in 6.11: Unlike qmltc, TaskTree is promoted to stable in Qt 6.11. The TaskTree, Group, and custom TaskAdapter<T> APIs are documented and tested.

  2. Significant boilerplate reduction: A proof-of-concept shows that a 4-method gRPC binding can be reduced from ~160 LOC to ~60 LOC using TaskTree recipes:

    // Before (QtConcurrent):
void PatientGrpcBinding::load(const QString& id) {
    auto* task = new GrpcCallTask(m_channel);
    connect(task, &GrpcCallTask::finished, this, [this, task]() {
        if (task->hasError()) { Q_EMIT error(task->errorMessage()); }
        else { updateFromResponse(task->response()); Q_EMIT loaded(); }
        task->deleteLater();
    });
    task->call(makeGetRequest(id));
}

// After (TaskTree):
using namespace Tasking;
const auto loadRecipe = Group {
    GrpcCallTaskItem(makeGetRequest(id)),
    onGroupDone([this](const GrpcCallTask& t) {
        updateFromResponse(t.response()); Q_EMIT loaded();
    }),
    onGroupError([this](const GrpcCallTask& t) {
        Q_EMIT error(t.errorMessage());
    })
};
m_taskTree.start(loadRecipe);

  3. Incremental adoption is safe: New generators can emit TaskTree patterns while existing generated code keeps the QtConcurrent pattern. Both can coexist in the same project. Migration of existing generators can happen in a future iteration.

Consequences

  • Good: ~60% reduction in generated gRPC binding boilerplate for new code.
  • Good: Automatic cancellation semantics (TaskTree cancels in-flight tasks on destruction).
  • Good: Composable — saga workflows can chain multiple gRPC calls declaratively.
  • Bad: Requires find_package(Qt6 COMPONENTS Core) with Tasking module (already available).
  • Neutral: Existing generated code continues to work unchanged.

Next Steps

  1. Add TaskTree-based gRPC binding generator as alternative to current pattern
  2. Gate behind Qt version check (if(Qt6_VERSION VERSION_GREATER_EQUAL 6.11))
  3. Migrate saga view generators to use TaskTree for multi-step workflows
  4. Consider filing FTR for full migration of existing generators in a future iteration
