Global Event Log with UUIDv7 Ordering

Global Event Log with UUIDv7 Ordering

  • Status: proposed
  • Date: 2026-02-04
  • Deciders: jolo, Muscovite AI
  • Tags: event-sourcing, event-store, uuidv7, projections, scaling

Context and Problem Statement

The current event sourcing implementation (ADR 20260204-event-sourcing-typed-event-store) uses per-aggregate event stores. While this works for single-aggregate operations, it creates challenges for:

  1. Cross-aggregate projections: Must poll multiple tables
  2. NATS relay: Needs a single cursor to follow all events
  3. Audit queries: "Find all events involving Ward X" requires scanning multiple tables

Standard event store implementations use a global_position BIGINT sequence, but this creates contention under high-concurrency writes.

Decision Drivers

  • Type Safety: No JSONB; all event payloads in typed tables
  • Scalability: Minimize lock contention on writes
  • Simplicity: Reduce moving parts (no sequence batching)
  • Ordering: Sufficient precision for event replay and projections

Solution: UUIDv7 as Natural Ordering

UUIDv7 (RFC 9562) embeds a millisecond-precision Unix timestamp in the first 48 bits, followed by random bits. This provides:

  • Chronological sorting: UUIDs sort by creation time
  • No contention: Each writer generates independently
  • Index locality: B-tree indexes cluster by time
  • Uniqueness: Random bits prevent collisions

UUIDv7 Structure

flowchart LR
    TS["48 bits:\nUnix timestamp (ms)"] --> VER["4 bits:\nversion (7)"]
    VER --> R1["12 bits:\nrandom"]
    R1 --> VAR["2 bits:\nvariant"]
    VAR --> R2["62 bits:\nrandom"]

Proposed Architecture

Schema Design

-- Global event log - UUIDv7 provides ordering
CREATE TABLE event_store.event_log (
    event_id        UUID PRIMARY KEY,  -- UUIDv7 (C++ generated)
    stream_type     TEXT NOT NULL,     -- 'Patient', 'Observation'
    stream_id       UUID NOT NULL,     -- Aggregate ID
    stream_version  INTEGER NOT NULL,  -- Per-stream OCC
    event_type      TEXT NOT NULL,     -- 'PatientRegistered'
    event_version   INTEGER NOT NULL DEFAULT 1,  -- V1, V2, etc.
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    
    UNIQUE (stream_id, stream_version)
);

-- Indexes for common access patterns
CREATE INDEX idx_event_log_stream ON event_store.event_log (stream_id, stream_version);
CREATE INDEX idx_event_log_type ON event_store.event_log (stream_type, event_id);

-- Typed payload tables (per event type, per version)
CREATE TABLE patient.event_patientregistered_v1 (
    event_id    UUID PRIMARY KEY REFERENCES event_store.event_log(event_id),
    patient_id  UUID NOT NULL,
    active      BOOLEAN NOT NULL,
    gender_id   UUID,
    birth_date  DATE
);

CREATE TABLE patient.event_patientregistered_v2 (
    event_id          UUID PRIMARY KEY REFERENCES event_store.event_log(event_id),
    patient_id        UUID NOT NULL,
    active            BOOLEAN NOT NULL,
    gender_id         UUID,
    birth_date        DATE,
    registration_type TEXT NOT NULL  -- New in V2
);

Aspect global_position (BIGINT) UUIDv7 Only
Contention High (sequence lock) None (distributed)

NATS Relay Query

-- Single cursor across all aggregates
SELECT event_id, stream_type, stream_id, event_type, event_version
FROM event_store.event_log
WHERE event_id > $1  -- Last processed UUIDv7
ORDER BY event_id
LIMIT 1000;

DSL Syntax (Explicit Versioning)

CONTEXT Patient {
    EVENT PatientRegistered V1 (
        patient_id  UUID NOT NULL,
        active      BOOLEAN NOT NULL,
        gender_id   UUID,
        birth_date  DATE
    )
    APPLIES_TO Patient;

    EVENT PatientRegistered V2 (
        patient_id        UUID NOT NULL,
        active            BOOLEAN NOT NULL,
        gender_id         UUID,
        birth_date        DATE,
        registration_type TEXT NOT NULL
    )
    APPLIES_TO Patient
    UPCASTS FROM V1 {
        registration_type = 'standard'
    };
}

C++ UUIDv7 Generation

namespace muscovite::types {

boost::uuids::uuid generate_uuid_v7() {
    using namespace std::chrono;
    
    auto now = system_clock::now();
    auto ms = duration_cast<milliseconds>(now.time_since_epoch()).count();
    
    boost::uuids::uuid uuid{};
    
    // First 6 bytes: timestamp (big-endian)
    uuid.data[0] = static_cast<uint8_t>((ms >> 40) & 0xFF);
    uuid.data[1] = static_cast<uint8_t>((ms >> 32) & 0xFF);
    uuid.data[2] = static_cast<uint8_t>((ms >> 24) & 0xFF);
    uuid.data[3] = static_cast<uint8_t>((ms >> 16) & 0xFF);
    uuid.data[4] = static_cast<uint8_t>((ms >> 8) & 0xFF);
    uuid.data[5] = static_cast<uint8_t>(ms & 0xFF);
    
    // Random bits for remaining bytes
    static thread_local std::mt19937_64 rng{std::random_device{}()};
    auto rand1 = rng();
    auto rand2 = rng();
    
    uuid.data[6] = 0x70 | static_cast<uint8_t>((rand1 >> 8) & 0x0F);  // Version 7
    uuid.data[7] = static_cast<uint8_t>(rand1 & 0xFF);
    uuid.data[8] = 0x80 | static_cast<uint8_t>((rand1 >> 16) & 0x3F); // Variant
    uuid.data[9] = static_cast<uint8_t>((rand1 >> 24) & 0xFF);
    uuid.data[10] = static_cast<uint8_t>((rand2) & 0xFF);
    uuid.data[11] = static_cast<uint8_t>((rand2 >> 8) & 0xFF);
    uuid.data[12] = static_cast<uint8_t>((rand2 >> 16) & 0xFF);
    uuid.data[13] = static_cast<uint8_t>((rand2 >> 24) & 0xFF);
    uuid.data[14] = static_cast<uint8_t>((rand2 >> 32) & 0xFF);
    uuid.data[15] = static_cast<uint8_t>((rand2 >> 40) & 0xFF);
    
    return uuid;
}

} // namespace muscovite::types

Edge Cases

Same-Millisecond Ordering

Events created in the same millisecond from different writers have non-deterministic ordering (random bits decide). This is acceptable because:

  1. Different aggregates: Independent causality—order doesn't matter
  2. Same aggregate: stream_version enforces causal ordering
  3. Sub-millisecond precision: If needed, use created_at with microsecond precision as tiebreaker

Clock Skew

If a node's clock drifts backward, it could generate UUIDs that sort before recently-inserted events. Mitigations:

  1. NTP synchronization: Standard practice for distributed systems
  2. Monotonic counter: Track last-used timestamp, never go backward
  3. Hybrid logical clocks: Overkill for most use cases

Consequences

Positive

  • Zero contention: Each writer generates UUIDs independently
  • Simpler schema: No sequence management or batching
  • Natural sharding: Partition by time ranges if needed
  • Cross-aggregate queries: Single table scan for projections
  • NATS relay: Simple cursor-based polling

Negative

  • C++ dependency: Must generate UUIDv7 in application (not PostgreSQL)
  • Sub-ms ordering: Cannot guarantee strict ordering within same millisecond
  • Table proliferation: Each event version = new table (mitigated by generators)

Decision Outcome

Proposed for v1.1.0. Current per-aggregate event store (v1.0.0) is sufficient for initial release. This architecture becomes necessary when:

  • Cross-aggregate projections are needed
  • NATS relay requires single-cursor polling
  • Write throughput exceeds sequence lock capacity
