M-Coordinate Linear Referencing Deferred

M-Coordinate Linear Referencing Deferred

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-01-26

Context and Problem Statement

PostGIS supports M (measure) coordinates for linear referencing systems (LRS). This is used in transportation networks, pipelines, and utilities to measure positions along linear features (e.g., "mile marker 42.5 on Interstate 90").

Should Muscovite support M coordinates in its initial spatial implementation?

Decision Drivers

  • Complexity vs immediate need
  • Focus on core spatial features first
  • M-coordinate use cases are specialized
  • PostGIS LRS functions are complex

Considered Options

  1. Full M-coordinate support - POINTM, LINESTRINGM, POLYGONM types
  2. Deferred - Not in initial release, evaluate later
  3. Never support - Declare out of scope permanently

Decision Outcome

Chosen option: Deferred

M-coordinate support is deferred to a future release. Initial spatial capability focuses on 2D and 3D geometries without measure values.

Rationale

  1. Core features first: XY (2D) and XYZ (3D) cover 95%+ of use cases
  2. Specialized domain: LRS primarily matters for transportation/utility GIS
  3. Complexity budget: M adds another dimension to all geometry operations
  4. Can add later: Nothing in current design prevents future M support

What This Means

Not supported in initial release:

  • POINTM, LINESTRINGM, POLYGONM, MULTIPOINTM, etc.
  • ST_LocateAlong(), ST_LocateBetween() functions
  • Linear referencing queries

Supported:

  • 2D: POINT, LINESTRING, POLYGON (XY)
  • 3D: POINTZ, LINESTRINGZ, POLYGONZ (XYZ)
  • Standard spatial queries (within, near, intersects)

Future Evaluation Criteria

M-coordinate support should be re-evaluated when:

  • A project specifically needs linear referencing
  • Transportation/pipeline domain model is added
  • Users request LRS functionality

AI Agent Guidance

When this applies:

  • User asks about M coordinates or linear referencing
  • Parser encounters POINTM, LINESTRINGM, etc.
  • Generator needs to decide on M-coordinate handling

What to do:

  • Reject M-coordinate types in parser with clear error message
  • Point users to this ADR when LRS is requested
  • Do not implement M-coordinate support without new ADR

Error message:

Error: M-coordinate types (POINTM, LINESTRINGM, POLYGONM) are not supported.
Linear referencing (LRS) is deferred to a future release.
See ADR: 20260126-m-coordinate-deferred.md

Use 2D types (POINT, LINESTRING, POLYGON) or 
3D types (POINTZ, LINESTRINGZ, POLYGONZ) instead.
