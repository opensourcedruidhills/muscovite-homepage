NATS Migration Topology Documentation in System Manual

NATS Migration Topology Documentation in System Manual

  • Status: superseded by 20260318-nats-agent-coordination-abandoned
  • Date: 2026-02-23
  • Tags: documentation, nats, migration, plantuml, asciidoc, system-manual
  • Deciders: Johannes Lochmann

Technical Story: The brownfield migration factory (ADR 20260217) introduces NATS-coupled runtime components (Extractor, Transformer, DLQ, Triage UI). The generated AsciiDoc system manual has no diagram showing the NATS data-flow topology, leaving operators and new developers without a generated artifact that describes the runtime pipeline.

Context and Problem Statement

When a Muscovite project enables migration (any bounded context contains a MAPPING block), the generated application acquires a new runtime topology:

Legacy DB ──→ Extractor ──→ NATS JetStream ──→ Transformer ──→ CommandRepository ──→ Target DB
                                                      │
                                                      └──→ DLQ Subject ──→ Triage UI

The existing glossary-adoc.rkt generator emits domain glossaries, ER diagrams, and (for sagas) workflow diagrams. It does not generate any network/component diagram. NATS subjects, JetStream streams, consumer group names, and DLQ routing rules are opaque to the human reader unless they open C++ source files.

Other generators in the chain (Context Map → PlantUML) set the precedent: topology visible from .ddd sources. The migration factory must follow the same convention.

Decision Drivers

  • ADR 20260217 (Documentation as Generated Artifact): all generated systems must be self-documenting
  • ADR 20260217 (Brownfield Migration Factory Architecture): pipeline topology is determined by MAPPING blocks
  • PlantUML standardization (ADR 20260217): all diagrams in PlantUML — no Mermaid, no hand-drawn
  • The system manual is the handoff artifact to operations; missing topology = undocumented production dependency
  • NATS subject names and stream configurations are derived from generator templates — only the generator can accurately document them

Considered Options

  1. Extend glossary-adoc.rkt to emit PlantUML component diagram per project — diagram driven by presence of MAPPING blocks; subject names computed by same naming logic used by extractor/transformer generators
  2. Separate nats-topology.rkt generator — standalone generator invoked only for migration-capable projects
  3. Static template — a hand-written diagram appended to the manual template

Decision Outcome

Chosen option: Option 1 — extend glossary-adoc.rkt, because the naming logic for NATS subjects (migration.extract.{context}.{entity}, migration.dlq.{context}.{entity}) is already available in the generator environment and placing the diagram in the same pass as the glossary ensures consistency with entity names.

Generated Diagram Structure

For each MAPPING block M in context C targeting entity E, the generator emits:

@startuml nats-migration-topology
!pragma layout smetana
skinparam componentStyle rectangle

component "Legacy DB\n({M.source})" as legacy_C_E
component "Extractor\n({C}Extractor)" as ext_C_E
queue "NATS JetStream\nmigration.extract.{c}.{e}" as nats_C_E
component "Transformer\n({C}{E}Transformer)" as xfm_C_E
database "Target DB\n{c}.{e}" as db_C_E
queue "NATS DLQ\nmigration.dlq.{c}.{e}" as dlq_C_E
component "Triage UI\n({C}{E}TriageWidget)" as triage_C_E

legacy_C_E --> ext_C_E : cursor batch\n(batch_size)
ext_C_E --> nats_C_E : typed JSON envelope
nats_C_E --> xfm_C_E : consume
xfm_C_E --> db_C_E : CommandRepository::create()
xfm_C_E --> dlq_C_E : validation failure
dlq_C_E --> triage_C_E : poll

@enduml

The diagram is emitted as a standalone .puml file and included via AsciiDoc plantuml:: block macro in the system manual's "Infrastructure & Data Flow" section.

Naming Conventions for Subjects

Component NATS subject / stream name
Extractor publish migration.extract.{sql_id(context)}.{sql_id(entity)}
Transformer subscribe migration.extract.{sql_id(context)}.{sql_id(entity)}
DLQ subject migration.dlq.{sql_id(context)}.{sql_id(entity)}
JetStream stream MIGRATION_{SQL_ID(context)}

These are computed by symbol->sql-id — the same function used by extractor/transformer generators — ensuring the diagram is always accurate.

Positive Consequences

  • Operations receives an accurate, auto-updated topology diagram with every DSL regeneration
  • Subject names in the diagram are guaranteed correct — derived from same naming logic as code
  • Adding a MAPPING block automatically updates the diagram
  • Consistent with PlantUML-only diagram policy (ADR 20260217)

Negative Consequences

  • glossary-adoc.rkt grows; migration-specific section is conditionally emitted only when MAPPING blocks exist
  • plantuml:: macro requires Asciidoctor diagram extension — already a dependency for ER diagrams

AI Agent Guidance

When implementing NATS topology documentation:

  • File to modify: dslgen/generators/docs/glossary-adoc.rkt
  • Add a new section generate-nats-topology-diagram that takes the list of all bounded contexts, filters for those with mapping structs, and emits the PlantUML block
  • Emit the PlantUML as a standalone .puml file under generated/doc/diagrams/nats-migration-topology.puml
  • Inline via plantuml::../diagrams/nats-migration-topology.puml[] in the main AsciiDoc
  • Subject naming: use (symbol->sql-id (bounded-context-name ctx)) and (symbol->sql-id (mapping-target-entity m)) — these match what extractor/transformer generators emit
  • Guard the entire section: emit only if (any (map (λ (ctx) (not (null? (bounded-context-mappings ctx)))) contexts))
  • Test: generator test checks that a project with MAPPING blocks produces a .puml file referencing correct NATS subject names
