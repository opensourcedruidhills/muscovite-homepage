Remove pgTAP Tests in Favor of Sqitch Verify and Catch2

Remove pgTAP Tests in Favor of Sqitch Verify and Catch2

  • Status: Accepted
  • Date: 2026-01-26
  • Deciders: Development team
  • Technical Story: Streamline testing architecture by removing redundant pgTAP tests

Context and Problem Statement

The Muscovite project generates pgTAP tests that verify database schema structure (table existence, column presence, FK constraints, etc.). However, these tests provide minimal value beyond what Sqitch's verify scripts already accomplish. Meanwhile, Catch2 integration tests provide comprehensive coverage of domain services, repositories, and constraint behavior.

The question is: should we continue maintaining pgTAP test generation when other tools cover the same ground more effectively?

Decision Drivers

  1. Redundancy: pgTAP tests for schema structure duplicate Sqitch verify functionality
  2. Maintenance burden: Generator code for pgTAP adds complexity without proportional value
  3. CI/CD overhead: Running pg_prove adds test execution time with limited unique coverage
  4. Tooling dependencies: pgTAP requires manual installation from source in PostgreSQL containers
  5. Value proposition: Catch2 tests validate actual business logic; Sqitch verifies schema deployment

Considered Options

  1. Keep pgTAP for structure tests - Continue generating has_table(), has_column(), has_fk() tests
  2. Remove pgTAP entirely - Rely on Sqitch verify + Catch2 integration tests
  3. Convert pgTAP to Catch2 SQL assertions - Reimplement schema checks in Catch2

Decision Outcome

Chosen option: Remove pgTAP entirely

Rationale

Test Concern Previous: pgTAP Now: Alternative
Table exists has_table('schema', 'table') Sqitch verify/*.sql
Column exists has_column('schema', 'table', 'col') Sqitch verify
FK constraint has_fk('schema', 'table') Catch2: FK violation test
UNIQUE constraint has_unique('schema', 'table', 'col') Catch2: unique violation test
CHECK constraint regex extraction from pg_catalog Catch2: CHECK violation test
CRUD operations N/A (was stored procedure API) Catch2: CommandService/QueryService
Domain events N/A Catch2: event emission verification

Sqitch verify scripts already test schema existence:

-- verify/basedata-table-patient.sql
SELECT id, mrn, active, created_at, updated_at
FROM basedata.patient
WHERE FALSE;

This SELECT will fail if any column is missing or has incompatible type.

Catch2 integration tests validate constraint behavior through actual operations:

SECTION("rejects duplicate email") {
    auto entity1 = service.create({.email = "test@example.com"});
    REQUIRE_THROWS_MATCHES(
        service.create({.email = "test@example.com"}),
        pqxx::unique_violation,
        Catch::Message::Contains("unique")
    );
}

Benefits

  1. Simpler tooling: No pgTAP installation required in CI/CD
  2. Faster tests: No pg_prove overhead
  3. Less generated code: Remove ~1500 lines from pgtap.rkt
  4. Clearer separation: Sqitch = deployment, Catch2 = behavior
  5. Better constraint testing: Catch2 tests actual violation behavior, not just metadata

Consequences

  • Good: Reduced maintenance burden
  • Good: Faster CI/CD pipelines
  • Good: Single test framework (Catch2) for all behavior tests
  • Neutral: Some schema introspection tests removed (covered by Sqitch)
  • Bad: No automated CHECK constraint content verification (low value)

Implementation

  1. Remove --test flag from generate.rkt (or repurpose for Catch2)
  2. Remove/archive src/dslgen/generators/postgres/pgtap.rkt
  3. Remove pgtap-* CMake targets from DatabaseProject.cmake
  4. Update documentation (ROADMAP.md, pitch.md, AGENTS.md, copilot-instructions.md)
  5. Remove pgTAP prerequisites from container setup instructions
