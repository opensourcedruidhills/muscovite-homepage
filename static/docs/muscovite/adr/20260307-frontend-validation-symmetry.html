Frontend Validation Symmetry for Qt Widgets and QML

Frontend Validation Symmetry for Qt Widgets and QML

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-07
  • Tags: qt, qml, widgets, validation, frontend, symmetry

Technical Story: The backend three-tier validation architecture needs symmetric representation in both frontend frameworks. Qt Widgets already has an EntityValidator with a rule system but it is not populated from DSL constraints. QML has no validation infrastructure.

Context and Problem Statement

Muscovite generates two frontend frameworks from the same DSL: Qt Widgets and Qt QML (ADR 20260206). The backend introduces a three-tier validation architecture (ADR 20260307-three-tier-validation-architecture) with strong types (Tier 1), cross-field invariants (Tier 2), and plugin business rules (Tier 3).

Without frontend validation symmetry, users experience:

  1. Delayed feedback — invalid data is submitted to the backend, rejected, and the error must be displayed retroactively
  2. Inconsistent behavior — some constraints are enforced in the UI (e.g., spin box ranges) while others are not
  3. Duplicated effort — developers manually add frontend validation that already exists in the DSL

The DSL already declares all the information needed to generate frontend validation. The question is how to map the three backend tiers to idiomatic Qt Widgets and QML patterns.

Decision Drivers

  • Both frontends must validate the same constraints as the backend (consistency)
  • Frontend validation is optimistic — backend validation is authoritative
  • Qt Widgets uses QValidator subclasses and widget configuration APIs
  • QML uses declarative Validator components and JavaScript expressions
  • Tier 3 (plugin) must be extensible without editing generated code in both frameworks
  • All validation must report field-level errors for UI binding

Considered Options

  1. Mirror three tiers in both frontends — each tier maps to idiomatic Qt Widgets and QML patterns
  2. Backend-only validation — frontends submit to backend and display returned errors
  3. Shared C++ validation library — frontends call the same C++ validators via bindings
  4. Schema-driven validation — backend sends validation schema, frontends interpret it at runtime

Decision Outcome

Chosen option: Option 1 — Mirror three tiers in both frontends, because it provides immediate user feedback, leverages native Qt/QML validation patterns, and maintains the DSL as single source of truth.

Qt Widgets Tier Mapping

Tier Backend Qt Widgets
Tier 1 Entity::try_from() with strong types Widget configuration: QSpinBox::setMinimum/setMaximum, QLineEdit + QRegularExpressionValidator, QDateEdit::setMaximumDate(QDate::currentDate())
Tier 2 EntityValidator::validate() EntityValidator rules generated from cross-field constraints, called on form submission
Tier 3 PluginValidator::validate() Plugin hooks in EntityValidator — virtual method in sidecar file

Tier 1 (Widget Configuration):

// Generated from: age Integer @min(0) @max(150)
age_spin_box_->setMinimum(0);
age_spin_box_->setMaximum(150);

// Generated from: name String @not_empty
name_line_edit_->setValidator(new QRegularExpressionValidator(QRegularExpression(".+"), this));

// Generated from: birth_date Date @not_in_future
birth_date_edit_->setMaximumDate(QDate::currentDate());

Tier 2 (EntityValidator):

// Generated from: end_date @after(start_date)
ValidationResult EntityValidator::validate() const {
    ValidationResult result;
    if (end_date_edit_->date() <= start_date_edit_->date()) {
        result.addError("end_date", tr("must be after start date"));
    }
    return result;
}

QML Tier Mapping

Tier Backend QML
Tier 1 Entity::try_from() with strong types IntValidator, DoubleValidator, RegularExpressionValidator on TextFields
Tier 2 EntityValidator::validate() ViewModel::validate() method with cross-field comparison logic
Tier 3 PluginValidator::validate() Virtual validatePlugin() method in ViewModel sidecar

Tier 1 (QML Validators):

// Generated from: age Integer @min(0) @max(150)
TextField {
    id: ageField
    validator: IntValidator { bottom: 0; top: 150 }
}

// Generated from: email String @pattern("[^@]+@[^@]+\\.[^@]+")
TextField {
    id: emailField
    validator: RegularExpressionValidator {
        regularExpression: /[^@]+@[^@]+\.[^@]+/
    }
}

Tier 2 (ViewModel Validation):

// Generated ViewModel method from: end_date @after(start_date)
QVariantMap ViewModel::validate() const {
    QVariantMap errors;
    if (endDate() <= startDate()) {
        errors.insert("end_date", tr("must be after start date"));
    }
    return errors;
}

ValidationResult (Shared)

Both frontends use a ValidationResult that maps field names to error messages, enabling UI binding:

// Qt Widgets: highlight field with red border, show tooltip
// QML: bind to error text below field, conditional styling

Positive Consequences

  • Immediate feedback — users see validation errors as they type or when fields lose focus
  • Single source of truth — DSL constraint annotations generate validation in backend + 2 frontends + SQL
  • DSL amplification increases further — one @min(0) generates 4+ validation sites
  • Native patterns — Qt Widgets uses QValidator, QML uses declarative Validators
  • Plugin extensibility — Tier 3 sidecar pattern works identically in both frontends

Negative Consequences

  • Frontend validation is optimistic — backend remains authoritative, so validation logic exists in two places
  • Widget configuration APIs differ between Qt Widgets and QML — generator must handle both idioms
  • Temporal constraints (@not_in_future) in QML require JavaScript date comparison (less robust than C++)
  • Increased generated code volume per entity (~30% more lines in both frontend targets)

AI Agent Guidance

When This Decision Applies

  • When generating Qt Widgets form code — configure widgets from DSL constraint annotations
  • When generating QML form code — add Validator components from DSL constraint annotations
  • When generating EntityValidator for Qt Widgets — produce cross-field validation rules
  • When generating ViewModel::validate() for QML — produce cross-field validation logic
  • When generating sidecar stubs for frontend plugin validation (Tier 3)

When This Decision Does NOT Apply

  • Backend validation code — that follows the backend three-tier architecture ADR
  • List/table views — validation applies to form/detail views only
  • Read-only displays — no validation needed for display-only fields
  • Navigation and routing — UI structure decisions are separate from validation

Enforcement

  • Qt Widgets generator in dslgen/generators/qt/widgets-main.rkt produces widget configuration from constraints
  • QML generator in dslgen/generators/qt/qml-main.rkt produces Validator components from constraints
  • Integration tests verify frontend validators accept/reject the same values as backend validators
  • Generated sidecar stubs follow sentinel protocol (ADR 20260223)
