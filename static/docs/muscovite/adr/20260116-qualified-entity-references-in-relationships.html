Qualified Entity References in Relationships

Qualified Entity References in Relationships

  • Status: accepted
  • Date: 2026-01-16
  • Deciders: Architecture Team
  • Tags: dsl, parser, relationships, cross-context

Context and Problem Statement

When declaring relationships in .ddd files, the current syntax only allows unqualified entity names:

RELATIONSHIPS {
    status -> ObservationStatus REQUIRED ONE @uuid "...";
    patient -> Patient REQUIRED ONE @uuid "...";
}

This creates ambiguity for human readers:

  • Where is ObservationStatus defined? Is it in the current context or imported from elsewhere?
  • Which Patient entity? The context name and entity name might be the same (e.g., Patient.Patient)
  • Cross-context validation becomes harder without explicit context qualification

The semantic diff and migration tools need to resolve these references, and implicit resolution based on "search current context first, then all contexts" is error-prone.

Decision Drivers

  • Human readability: Developers should immediately know where a referenced entity lives
  • Explicit over implicit: Cross-context dependencies should be visible in the code
  • Tool support: Generators and validators need unambiguous reference resolution
  • DDD bounded context principles: Cross-context references represent anti-corruption layer boundaries

Considered Options

Option 1: Qualified References with Context.Entity Syntax

RELATIONSHIPS {
    status -> Basedata.ObservationStatus REQUIRED ONE @uuid "...";
    patient -> Patient.Patient REQUIRED ONE @uuid "...";
    observation -> Observation REQUIRED ONE @uuid "...";  -- local ref (same context)
}

Pros:

  • Unambiguous - always know where the entity lives
  • Consistent with many programming languages (C#, Java, etc.)
  • Easy to parse (split on .)
  • Self-documenting cross-context dependencies

Cons:

  • More verbose for local references
  • Requires updating existing models

Option 2: Import Statements with Aliases

IMPORT Basedata.ObservationStatus AS Status;
IMPORT Patient.Patient AS Patient;

RELATIONSHIPS {
    status -> Status REQUIRED ONE;
}

Pros:

  • Shorter references after import
  • Explicit import list at top of file

Cons:

  • Two places to look (imports + relationships)
  • More complex parser changes
  • Alias conflicts possible

Option 3: Namespace Prefixes with @from Annotation

RELATIONSHIPS {
    status -> ObservationStatus @from Basedata REQUIRED ONE;
}

Pros:

  • Annotation-based, consistent with @uuid pattern

Cons:

  • Unusual syntax, not familiar from other languages
  • Annotation ordering issues

Decision Outcome

Chosen option: Option 1 - Qualified References with Context.Entity Syntax

This provides the best balance of readability, familiarity, and implementation simplicity.

Syntax Specification

relationship-target := qualified-ref | unqualified-ref
qualified-ref       := context-name "." entity-name
unqualified-ref     := entity-name

context-name        := identifier  ; Must match a CONTEXT name in the project
entity-name         := identifier  ; Must match an ENTITY name in that context

Resolution Rules

  1. Qualified reference (Context.Entity): Look up entity in the specified context. Error if context or entity not found.

  2. Unqualified reference (Entity): Look up entity in the current context only. Error if not found locally.

  3. No implicit cross-context search: If an entity is in another context, you MUST use qualified syntax.

Examples

CONTEXT Observation {
    ENTITY Observation IDENTIFIED_BY id (
        id                    UUID NOT NULL,
        status_id             UUID NOT NULL,
        patient_id            UUID NOT NULL
    )
    RELATIONSHIPS {
        -- Cross-context: explicit qualification required
        status -> Basedata.ObservationStatus REQUIRED ONE @uuid "...";
        patient -> Patient.Patient REQUIRED ONE @uuid "...";
        
        -- Same context: unqualified is fine
        component <- ObservationComponent MANY @uuid "...";
    }
}

Data Model Changes

The relationship struct gains a new field:

(struct relationship
  (uuid
   name
   direction
   target-entity      ; Symbol: entity name
   target-context     ; Symbol or #f: context name (nil for local refs)
   required?
   multiplicity
   ...))

Backward Compatibility

Not maintained. Existing .ddd files with cross-context unqualified references will fail validation. This is intentional:

  • Forces explicit documentation of cross-context dependencies
  • Eliminates ambiguous references
  • One-time migration cost for existing models

Migration Path

  1. For each unqualified reference in RELATIONSHIPS block
  2. Determine which context contains the target entity
  3. Add context qualifier: Entity → Context.Entity

Consequences

Positive

  • Unambiguous references: No more guessing where entities come from
  • Self-documenting: Cross-context dependencies visible at point of use
  • Better tooling: IDEs can provide accurate go-to-definition
  • Easier validation: Fail-fast if target context/entity doesn't exist
  • DDD alignment: Makes bounded context boundaries explicit

Negative

  • More verbose: Patient.Patient is redundant when context = entity name
  • Migration effort: Existing models need updating
  • Learning curve: New syntax for existing users

Neutral

  • Parser complexity: Minimal (split identifier on .)
  • Generator impact: Must use target-context when generating FK references

Implementation Notes

Parser Changes

;; Parse qualified identifier: Context.Entity or just Entity
(define (parse-qualified-identifier ps)
  (let* ([first-part (expect-identifier! ps)])
    (if (match-punctuation? ps ".")
        (begin
          (advance! ps)
          (let ([second-part (expect-identifier! ps)])
            (values (string->symbol second-part)    ; entity
                    (string->symbol first-part))))  ; context
        (values (string->symbol first-part) #f))))  ; entity, no context

Validation

Cross-context reference validation in validate-relationships:

(define (validate-relationship rel current-context all-contexts)
  (define target-ctx-name (or (relationship-target-context rel)
                              (bounded-context-name current-context)))
  (define target-ctx (find-context-by-name all-contexts target-ctx-name))
  (unless target-ctx
    (error "Unknown context: ~a" target-ctx-name))
  (unless (find-entity-in-context target-ctx (relationship-target-entity rel))
    (error "Entity ~a not found in context ~a" 
           (relationship-target-entity rel) target-ctx-name)))

AI Agent Guidance

When This Decision Applies

  • Parsing RELATIONSHIPS blocks in .ddd files
  • Validating cross-context entity references
  • Generating FK constraints with cross-schema references
  • Computing semantic diff for relationship changes

When This Decision Does NOT Apply

  • Entity inheritance (EXTENDS clause) - uses same-context only
  • Field types - primitive types, not entity references
  • Aggregate membership - entities already in same context

Key Files Affected

  • src/dslgen/parser/parser.rkt - parse-qualified-identifier, parse-single-relationship
  • src/dslgen/model/ddd-types.rkt - relationship struct, make-relationship
  • src/dslgen/validation/validate-relationships.rkt - cross-context validation
  • src/dslgen/generators/postgres/*.rkt - FK generation with schema qualification

Enforcement

  • Parser extracts target-context from qualified syntax
  • Validation fails if cross-context reference uses unqualified name
  • Generator uses target-context to determine FK schema prefix
