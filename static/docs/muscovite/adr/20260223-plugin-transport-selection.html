Plugin Transport Selection — SyncVirtual, LocalGRPC, QtRemoteObjects

Plugin Transport Selection — SyncVirtual, LocalGRPC, QtRemoteObjects

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-23
  • Tags: plugins, transport, grpc, qt-remote-objects, sync-virtual, architecture

Technical Story: Extension point plugin classes (Validator, SideEffect, UIWidget) require different communication transports. Transport selection must be a per-plugin .mus configuration, not hardcoded in the DSL. This ADR defines the three approved transports and their generation rules.

Context and Problem Statement

The plugin system (ADR 20260223-plugin-system-extension-points-and-layer2-sidecar) defines three plugin classes. Each class has different performance and integration requirements:

  • Validator — synchronous, in-process, zero overhead. Network transport is wrong.
  • SideEffect — asynchronous, potentially slow, must not block the domain service. Needs process isolation. Network transport is correct.
  • UIWidget — frontend UI injection. Must integrate with Qt's object model. Qt-native transport is ergonomically correct.

The DSL declares the extension point with a recommended interface. The .mus file selects the actual transport for each plugin binding. Generator emits different base classes per transport.

Decision Drivers

  • Validators must have zero transport overhead — no serialization, no network
  • SideEffect plugins must be process-isolated — failures must not crash the domain service
  • UIWidget plugins must integrate with Qt signals/slots naturally
  • Transport switching must be a one-line .mus change + regeneration (no code changes)
  • Localhost-only for all non-SyncVirtual transports — plugins are local processes, not external network services
  • Qt Remote Objects approved by ADR 20260223-qt-remote-objects-lgpl-transport-approval

Considered Options

Transport per class:

Class Option A Option B
Validator Pure virtual C++ (SyncVirtual) CRTP zero-cost abstraction
SideEffect LocalGRPC (loopback) Qt Remote Objects (loopback)
UIWidget Qt Remote Objects LocalGRPC (loopback)

Decision Outcome

Transport 1: SyncVirtual (Validator class)

Pure virtual C++ function. No transport protocol. In-process call.

// Generated: password_validator_interface.hpp (Layer 2)
class PasswordValidatorBase {
public:
    virtual ~PasswordValidatorBase() = default;

    /// @brief Validate password value. Called before repository write.
    /// @param value  The raw password string to validate
    /// @return ValidationResult{.ok=true} on pass, {.ok=false, .message=...} on fail
    virtual ValidationResult validate(const std::string& value) const = 0;
};

Rationale for pure virtual over CRTP: Runtime pluggability is what the research thesis demonstrates. CRTP gives zero-cost abstraction but requires compile-time plugin selection (cannot load from shared library). Pure virtual enables runtime loading. CRTP documented as a future optimisation option when the compile-time variant is desired.

BE location: abstract base injected into command handler, called before repository write. FE location: abstract base injected into Qt dialog isValid() chain.

Transport 2: LocalGRPC (SideEffect class — backend)

gRPC over localhost loopback socket. Plugin runs as a separate process.

Localhost enforcement: Generated grpc::SslCredentials are rejected; generated server-side bind address is hardcoded to 127.0.0.1:PORT (not 0.0.0.0). This is enforced in generated code, not just convention.

Generator produces (per LocalGRPC extension point):

  • Proto service definition (Layer 2, always regenerated): {extension_point}_plugin.proto with input/output message types derived from TARGET
  • gRPC client call in generated domain service (Layer 2, always regenerated)
  • Sidecar server stub (Layer 2 Sidecar, generated once): {extension_point}_plugin_server.cpp — developer implements the service body
// Generated sidecar stub: audit_exporter_plugin_server.cpp
// MUSCOVITE:SIDECAR:UNMODIFIED — remove this line when editing this file
class AuditExporterServiceImpl final : public AuditExporter::Service {
public:
    grpc::Status Export(grpc::ServerContext* ctx,
                        const ExportRequest* req,
                        ExportResponse* resp) override {
        // TODO: implement audit export logic here
        return grpc::Status::OK;
    }
};

Transport 3: QtRemoteObjects (UIWidget class — frontend)

Qt Remote Objects proxy over a local domain socket. Plugin process exposes a QRemoteObjectHost; generated dialog connects via QRemoteObjectNode.

Approved by ADR 20260223-qt-remote-objects-lgpl-transport-approval.

Generator produces:

  • REPC source file (Qt Remote Object class spec) — Layer 2, regenerated
  • QRemoteObjectNode connection code in generated dialog — Layer 2, regenerated
  • Sidecar QRemoteObjectHost implementation stub — Layer 2 Sidecar, generated once

Fallback: if TRANSPORT: LocalGRPC is declared on a UIWidget-class extension point, the LocalGRPC transport is used instead. The generated base class differs but the extension point interface contract is identical.

Transport as .mus Configuration

Transport is declared in the .mus PLUGIN block:

PLUGIN MyPasswordPolicy
    IMPLEMENTS: PasswordValidator     # references EXTENSION_POINT in .ddd
    TRANSPORT: SyncVirtual            # overrides INTERFACE hint from .ddd if compatible

PLUGIN EnterpriseAudit
    IMPLEMENTS: AuditExporter
    TRANSPORT: LocalGRPC

Validation:

  • TRANSPORT must be compatible with the INTERFACE declared on the EXTENSION_POINT
  • Validator ↔︎ SyncVirtual always; SyncVirtual for SideEffect is rejected
  • SideEffect ↔︎ LocalGRPC or QtRemoteObjects; UIWidget ↔︎ QtRemoteObjects or LocalGRPC

Switching transport: change one .mus line, regenerate. No user code changes needed (the sidecar stub regenerates because the interface changed and the sentinel is still present).

Transport Compatibility Matrix

Class \ Transport SyncVirtual LocalGRPC QtRemoteObjects
Validator ✅ only ❌ ❌
SideEffect ❌ ✅ BE only ✅ FE only
UIWidget ❌ ✅ fallback ✅ primary

Positive Consequences

  • One-line transport switch in .mus without touching plugin implementation
  • Localhost enforcement prevents accidental network exposure
  • Each transport generates appropriate base classes and CMake targets
  • Plugin contract (abstract interface) is transport-independent

Negative Consequences

  • Three transport implementations to maintain in generators
  • Cross-validation between .ddd INTERFACE hint and .mus TRANSPORT required
  • LocalGRPC plugins require a running process at startup (CMake must generate a ExternalProcess launch target or documentation for plugin process management)
