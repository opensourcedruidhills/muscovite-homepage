Export Muscovite as CMake Package

Export Muscovite as CMake Package

  • Status: accepted
  • Date: 2026-01-12
  • Tags: cmake, build-system, packaging, generated-code

Context and Problem Statement

Generated projects (like FHIR-Health) produce C++ code that depends on Muscovite framework libraries (muscovite::dba, muscovite::server, muscovite::grpc). Currently, these generated projects cannot build standalone because the framework libraries are only available as CMake ALIAS targets within the Muscovite build tree.

How should generated projects discover and link against Muscovite framework libraries?

Decision Drivers

  • Generated projects should be self-contained and buildable outside Muscovite source tree
  • Standard CMake find_package() is the idiomatic way to discover dependencies
  • Generated projects may be deployed to different machines without Muscovite source
  • Conan packaging for generated projects requires proper CMake targets
  • Transitive dependencies (libpqxx, spdlog, fmt, Boost) must be correctly propagated

Considered Options

  1. Export CMake package - Install Muscovite with install(EXPORT) and config files
  2. Header-only framework - Remove .cpp files, make everything inline/templates
  3. Conan package for Muscovite - Publish muscovite to Conan Center
  4. Git submodule - Generated projects include Muscovite as submodule
  5. Copy framework sources - Generator copies framework code into generated project

Decision Outcome

Chosen option: Export CMake package, because it:

  • Uses standard CMake patterns that developers expect
  • Works with any build system that supports CMake
  • Enables future Conan packaging without additional work
  • Keeps generated projects clean (no framework source code)
  • Properly handles transitive dependencies via find_dependency()

Implementation

1. Package Config Template (cmake/muscoviteConfig.cmake.in)

@PACKAGE_INIT@

include(CMakeFindDependencyMacro)

# Required dependencies
find_dependency(libpqxx CONFIG)
find_dependency(spdlog CONFIG)
find_dependency(fmt CONFIG)
find_dependency(cnats CONFIG)
find_dependency(Boost COMPONENTS program_options headers)

# Optional gRPC dependencies
if(@MUSCOVITE_GRPC_AVAILABLE@)
    find_dependency(gRPC CONFIG)
    find_dependency(Protobuf CONFIG)
endif()

include("${CMAKE_CURRENT_LIST_DIR}/muscoviteTargets.cmake")

check_required_components(muscovite)

2. Library Install Rules

Each library target needs:

install(TARGETS muscovitedba
    EXPORT muscoviteTargets
    ARCHIVE DESTINATION lib
    LIBRARY DESTINATION lib
    RUNTIME DESTINATION bin
)

install(DIRECTORY include/muscovite
    DESTINATION include
)

2a. Proto File Installation

gRPC proto files are installed so standalone projects can compile generated services:

install(DIRECTORY proto/muscovite
    DESTINATION share/muscovite/proto
    FILES_MATCHING PATTERN "*.proto"
)

3. Export Generation (root CMakeLists.txt)

install(EXPORT muscoviteTargets
    FILE muscoviteTargets.cmake
    NAMESPACE muscovite::
    DESTINATION lib/cmake/muscovite
)

configure_package_config_file(...)
write_basic_package_version_file(...)

4. Generated Project Usage

Generated CMakeLists.txt files use:

find_package(muscovite REQUIRED COMPONENTS dba server)

target_link_libraries(${PROJECT_NAME}_dba
    PUBLIC
        muscovite::dba
        muscovite::server
)

Installation and Usage

# Build and install Muscovite
cd ${PROJECT_ROOT}
cmake --preset release
cmake --build --preset release
cmake --install build/release --prefix /opt/muscovite

# Build generated project (standalone)
cd examples/fhir-health/generated/dba
cmake -B build -DCMAKE_PREFIX_PATH=/opt/muscovite
cmake --build build

Positive Consequences

  • Generated projects build standalone without Muscovite source tree
  • Standard CMake workflow familiar to C++ developers
  • Enables CI/CD pipelines where generated code is built separately
  • Foundation for publishing Muscovite to Conan Center
  • Transitive dependencies automatically resolved

Negative Consequences

  • Requires explicit install step before building generated projects
  • Version compatibility between Muscovite install and generated project
  • Must keep muscoviteConfig.cmake.in in sync with actual dependencies

AI Agent Guidance

When This Applies

  • Building generated projects outside Muscovite source tree
  • Setting up CI/CD for generated projects
  • Distributing generated projects to other developers/machines

When This Doesn't Apply

  • Building within Muscovite source tree (use existing ALIAS targets)
  • Running code generators (they don't need installed Muscovite)

Key Files

  • cmake/muscoviteConfig.cmake.in - Package config template
  • framework/*/CMakeLists.txt - Install rules (dba, server, grpc)
  • framework/grpc/CMakeLists.txt - Proto file installation
  • dslgen/generators/grpc/main.rkt - Generated gRPC CMakeLists (proto dir resolution)
  • dslgen/generators/cpp/dba/cmake.rkt - Generated DBA CMakeLists (find_package)
