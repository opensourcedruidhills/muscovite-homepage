ADR-20251226: Use JSONB for Embedded Value Objects in Database API

ADR-20251226: Use JSONB for Embedded Value Objects in Database API

Status: Accepted

Date: 2025-12-26

Deciders: Development Team

Context

The Muscovite DSL generates PostgreSQL database APIs from domain models that include embedded value objects (VOs). Previously, the code generator took a column-flattening approach:

  • Each field from an embedded VO was generated as a separate column in the SQL table
  • Each VO field became a separate parameter in stored procedure signatures
  • Example: WaterSample aggregate with embedded FieldParameters VO resulted in:
    • Schema: 70+ individual columns (temperature_c, ph, dissolved_oxygen, ...)
    • API: watersample_insert(..., p_temperature_c, p_ph, p_dissolved_oxygen, ...) with 70+ parameters

Problem: This approach violated several key architectural principles:

  1. Parameter Explosion: Stored procedure signatures became unwieldy with 70+ parameters
  2. Type System Mismatch: Violated DDD principle that value objects are cohesive, reusable types
  3. API Instability: Changes to VO structure required changes to multiple column definitions and function signatures
  4. Impedance Mismatch: Database schema didn't reflect domain model structure (scattered VO fields across columns)
  5. Maintainability: Difficult to understand which columns belonged to which VO

Decision

Implement JSONB-based storage for embedded value objects:

  1. Schema Level:

    • Each embedded VO gets ONE JSONB column per VO type
    • Example: watersample table has fieldparameters JSONB, majorcations JSONB, etc. (7 columns instead of 70)
    • CHECK constraints still validate enum-like fields (e.g., sample_type IN ('grab', 'composite', ...))

  2. API Level:

    • Stored procedure parameters accept JSONB: p_fieldparameters JSONB, p_majorcations JSONB
    • Procedures parse JSONB and insert directly: INSERT INTO table (fieldparameters) VALUES (p_fieldparameters)
    • INSERT/UPDATE/DELETE functions work with JSONB natively

  3. Testing Level:

    • pgTAP tests serialize VO fields to JSON during test setup
    • Tests pass JSONB-formatted strings to API functions
    • Row deserialization extracts JSONB and returns complete rows

  4. Client Level (C++/Other Languages):

    • Repositories serialize domain value objects to JSONB using language-native JSON library
    • Example: serialize_vo_to_jsonb(entity.fieldparameters()) → "{\"temperature_c\": 20.5, ...}"
    • SELECT operations deserialize JSONB back to native VO objects

Rationale

Why JSONB Instead of Flattened Columns?

Aspect Flattened JSONB
Parameter Count 70+ per procedure 7 per procedure
Schema Stability High - VO changes = schema changes Low - VO changes = internal structure
Type Representation Scattered across columns Cohesive JSONB column
Domain Alignment Violates DDD (VO scattered) Respects DDD (VO as unit)
Query Complexity Simple SQL (column access) Requires JSONB operators (->>, ->)
Validation CHECK per column CHECK per JSONB path
Performance Fast (native columns) Slightly slower (JSONB parsing)

The performance cost is acceptable because:

  • VO access patterns are infrequent (insert/update per aggregate, not per query)
  • JSONB indexing available for filtering (if needed)
  • Consistency and maintainability gains outweigh performance impact

Why Not Use Custom PostgreSQL Types (CREATE TYPE)?

  • Custom types are PostgreSQL-specific, limits portability
  • JSONB is standard SQL-adjacent, widely understood
  • JSONB tools are mature and performant
  • Easier to introspect at runtime (API documentation)

Consequences

Positive

✅ API Clarity: Function signatures are concise and readable
✅ Schema Clarity: JSONB columns clearly indicate value objects
✅ DDD Alignment: Domain model structure reflected in database
✅ Stability: Adding/removing VO fields doesn't require schema migration
✅ Type Safety: Serialization/deserialization provides compile-time checking
✅ Testability: JSONB serialization can be tested independently

Negative

❌ Query Complexity: Accessing nested fields requires JSONB operators
❌ Performance: JSONB parsing overhead (minimal for typical workloads)
❌ Indexing: Can't create indexes on nested JSONB fields without functional indexes

Required Implementation Changes

  1. SQL Schema Generator (sqitch-unified.rkt):

    • CREATE JSONB columns instead of flattened columns
    • Status: ✅ Complete

  2. PL/pgSQL API Generator (plpgsql-api.rkt):

    • Accept JSONB parameters
    • Insert JSONB directly into columns
    • Status: ✅ Complete

  3. pgTAP Test Generator (pgtap.rkt):

    • Serialize VO fields to JSON during test data generation
    • Pass JSONB strings to API functions
    • Status: ✅ Complete

  4. C++ DBA Repository Generator (repository.rkt, jsonb_repository.rkt):

    • Separate entity fields into primitives vs. embedded VOs
    • Serialize VOs to JSONB before calling API functions
    • Deserialize JSONB back to VO objects when reading from database
    • Status: ✅ Complete (basic JSONB storage, advanced serialization pending)

  5. C++ Framework (type_converters.hpp, jsonb_converters.hpp):

    • Add to_jsonb_opt() for nullable JSONB deserialization
    • Add serialize_vo_to_jsonb<VO>(const VO&) -> std::string
    • Add deserialize_vo_from_jsonb<VO>(const std::string&) -> VO
    • Status: ✅ Complete (to_jsonb_opt implemented; helpers ready for VO code generation)

Examples

Before (Flattened)

-- Schema
CREATE TABLE watersample (
    id UUID PRIMARY KEY,
    fieldparameters_temperature_c NUMERIC,
    fieldparameters_ph NUMERIC,
    fieldparameters_dissolved_oxygen NUMERIC,
    majorcations_ca NUMERIC,
    majorcations_mg NUMERIC,
    majorcations_na NUMERIC,
    -- ... 70+ columns total
);

-- API
CREATE FUNCTION watersample_insert(
    p_sample_number TEXT,
    p_sample_type TEXT,
    p_fieldparameters_temperature_c NUMERIC,
    p_fieldparameters_ph NUMERIC,
    p_fieldparameters_dissolved_oxygen NUMERIC,
    p_majorcations_ca NUMERIC,
    p_majorcations_mg NUMERIC,
    p_majorcations_na NUMERIC,
    -- ... 70+ parameters
) RETURNS watersample AS $$
BEGIN
    INSERT INTO watersample (
        sample_number, sample_type,
        fieldparameters_temperature_c, fieldparameters_ph, fieldparameters_dissolved_oxygen,
        majorcations_ca, majorcations_mg, majorcations_na,
        -- ...
    ) VALUES (
        p_sample_number, p_sample_type,
        p_fieldparameters_temperature_c, p_fieldparameters_ph, p_fieldparameters_dissolved_oxygen,
        p_majorcations_ca, p_majorcations_mg, p_majorcations_na,
        -- ...
    ) RETURNING * INTO v_result;
    RETURN v_result;
END;
$$ LANGUAGE plpgsql;

After (JSONB)

-- Schema
CREATE TABLE watersample (
    id UUID PRIMARY KEY,
    sample_number TEXT NOT NULL,
    sample_type TEXT NOT NULL,
    source_type TEXT NOT NULL,
    fieldparameters JSONB,           -- VO as single column
    majorcations JSONB,               -- VO as single column
    majoranions JSONB,                -- VO as single column
    tracemetals JSONB,                -- VO as single column
    isotopesignature JSONB,           -- VO as single column
    watertype JSONB,                  -- VO as single column
    ionbalance JSONB                  -- VO as single column
    -- ... 7 columns total
);

-- API
CREATE FUNCTION watersample_insert(
    p_sample_number TEXT,
    p_sample_type TEXT,
    p_source_type TEXT,
    p_fieldparameters JSONB,          -- VO as single parameter
    p_majorcations JSONB,              -- VO as single parameter
    p_majoranions JSONB,               -- VO as single parameter
    p_tracemetals JSONB,               -- VO as single parameter
    p_isotopesignature JSONB,          -- VO as single parameter
    p_watertype JSONB,                 -- VO as single parameter
    p_ionbalance JSONB                 -- VO as single parameter
    -- ... 7 parameters total
) RETURNS watersample AS $$
BEGIN
    INSERT INTO watersample (
        sample_number, sample_type, source_type,
        fieldparameters, majorcations, majoranions, tracemetals, isotopesignature, watertype, ionbalance
    ) VALUES (
        p_sample_number, p_sample_type, p_source_type,
        p_fieldparameters, p_majorcations, p_majoranions, p_tracemetals, p_isotopesignature, p_watertype, p_ionbalance
    ) RETURNING * INTO v_result;
    RETURN v_result;
END;
$$ LANGUAGE plpgsql;

C++ Client Code (After)

// Serialize domain VO to JSONB
auto field_params = entity.fieldparameters();
auto field_params_json = serialize_vo_to_jsonb(field_params);

// Call API with JSONB parameter
auto result = repository.insert({
    entity.sample_number(),
    entity.sample_type(),
    field_params_json,      // Single JSONB param instead of 10+ individual params
    // ... other VOs
});

// SELECT returns full row with JSONB columns
auto row = repository.find_by_id(id);

// Deserialize JSONB back to domain VO
auto field_params = deserialize_vo_from_jsonb<FieldParameters>(
    row["fieldparameters"].as<std::string>()
);

Alternatives Considered

1. Keep Flattened Columns (Status: Rejected)

  • Pros: Simple SQL queries, no JSONB parsing overhead
  • Cons: Parameter explosion, poor DDD alignment, schema instability, maintainability nightmare
  • Verdict: Flattening violates domain modeling principles; unsustainable long-term

2. Use PostgreSQL RECORD Type (Status: Rejected)

  • Pros: Native PostgreSQL type
  • Cons: Limited by PostgreSQL version, difficult to serialize, schema-tied
  • Verdict: Less portable than JSONB, more complex than necessary

3. Use TEXT Columns with SQL Comments (Status: Rejected)

  • Pros: Would work, simpler than JSONB
  • Cons: No validation, no type safety, no indexing, harder to query
  • Verdict: Comments don't help at runtime; JSONB is better

4. Split Into Separate Tables (Status: Rejected)

  • Pros: Normalized, queryable VO fields, easier to index
  • Cons: Breaks aggregate boundaries, requires joins, contradicts DDD
  • Verdict: Violates aggregate root principle
