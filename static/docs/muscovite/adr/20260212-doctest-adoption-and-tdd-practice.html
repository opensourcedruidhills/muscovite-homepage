Adopt doctest and Test-Driven Development Practice

Adopt doctest and Test-Driven Development Practice

Context and Problem Statement

Muscovite's complexity is growing (24 contexts, 8 generator targets, 2,357 compile steps). The current testing strategy has structural gaps:

  1. Framework has zero direct tests — framework/tests/ is empty; framework code tested only indirectly through integration tests
  2. Generator tests are shallow — test-generator-suite.rkt checks file existence, not structural correctness
  3. Catch2 header cost — 380–470ms per include, parsed across thousands of TUs in generated code
  4. No TDD discipline — tests written after implementation, gaps discovered late
  5. AI agents lack executable usage examples — the PEG grammar anchors DSL syntax, but generated C++ API has no BDD-style executable documentation

Decision Drivers

  • Compile time: S3 (generated code) takes 14,050s CPU — test TU header costs compound
  • BDD parity: Must retain SCENARIO/GIVEN/WHEN/THEN macros for readable executable specifications
  • AI comprehension: BDD tests + formal grammar = complete anchor for AI agents (input syntax + output usage)
  • Migration cost: Must be gradual — Catch2 tests remain, doctest used for new tests
  • CMake integration: Must support doctest_discover_tests equivalent to catch_discover_tests

Considered Options

  1. Stay with Catch2, add more tests — no compile-time improvement
  2. Switch to doctest for new tests, keep Catch2 for existing — gradual, low risk (chosen)
  3. Switch everything to doctest at once — high risk, no benefit over gradual
  4. Use Google Test — heavier than both, no BDD macros

Decision Outcome

1. Adopt doctest for all new C++ tests

doctest (v2.4.12+, MIT license, single header) replaces Catch2 as the default test framework for:

  • New framework unit tests (framework/*/tests/)
  • New integration tests (tests/)
  • Generated tests (future: generators/cpp/dba/ctest.rkt update)

Catch2 remains for existing tests — migration is optional and opportunistic.

2. Mandate Test-Driven Development for AI agents

Rule: Every code change by an AI agent MUST follow test-first workflow:

Layer Test First Then Implement
Racket (Layer 0) Write rackunit test with expected output Write/modify generator
C++ Framework (Layer 1) Write doctest BDD test Write/modify framework code
Generated (Layer 2) Generated test IS the specification Generator produces code + test together

3. BDD style as default

All new tests use BDD macros for readability:

SCENARIO("ConnectionPool validates configuration", "[dba][pool]") {
    GIVEN("a PoolConfig with empty connection string") {
        PoolConfig config;
        config.connection_string = "";

        WHEN("creating a pool") {
            auto result = ConnectionPool::create(config);

            THEN("it fails with InvalidConfig error") {
                REQUIRE_FALSE(result.has_value());
                CHECK(result.error().code == PoolError::InvalidConfig);
            }
        }
    }
}

doctest vs Catch2 Comparison

Criterion Catch2 v3 doctest v2.4 Impact on Muscovite
Header include cost 380–470ms 11–23ms 36× lighter — compounds across 2,357 TUs
BDD macros ✅ ✅ No change needed
Tags [tag] ✅ native ❌ (use TEST_SUITE + name filter) Low — CTest labels are primary filter
Matchers ✅ rich ❌ (Approx/Contains only) Low — integration tests rarely use matchers
Generators GENERATE() ✅ ❌ Not used in current tests
CMake discovery catch_discover_tests doctest_discover_tests Identical API
Single header No (v3) Yes Simpler vendoring
DISABLE macro Partial Full elimination Production binary strips test overhead

Coexistence Strategy

# Testing.cmake — both frameworks discovered
find_package(Catch2 3 REQUIRED)      # Existing tests
find_package(doctest REQUIRED)        # New tests

# Existing: catch_discover_tests(target ...)
# New:      doctest_discover_tests(target ...)

No conflicts — different header includes, different CMake modules, same CTest integration.

Consequences

Positive

  • 36× lighter test headers for new/migrated tests
  • Executable specifications as AI agent anchors (grammar = input, BDD tests = output usage)
  • Framework test gap closed — direct unit tests for Repository, UnitOfWork, strong types
  • Test-first catches generator bugs before they reach generated code
  • Cultural shift — AI agents write test first, implementation second

Negative

  • Two test frameworks coexist (temporary, until optional migration)
  • No matchers in doctest (write explicit CHECKs instead)
  • doctest slower release cadence than Catch2 (mitigated: API is stable, no breaking changes expected)

Neutral

  • Existing Catch2 tests unchanged — no migration required
  • CTest presets work identically with both frameworks
  • Build acceleration (mold + ccache) benefits both

AI Agent Guidance

When this applies

  • Always: Every AI agent code change must follow TDD workflow
  • New C++ tests: Use doctest, not Catch2
  • Existing C++ tests: Keep Catch2, migrate opportunistically
  • Racket tests: Use rackunit with programmatic model construction

How it's enforced

  • Copilot instructions Rule 10: TDD Practice (MANDATORY)
  • Code review: test file must exist before implementation file in commit
  • Generated test templates updated to emit doctest (Phase 3)
