Git Flow Branching Strategy

Git Flow Branching Strategy

Context and Problem Statement

Muscovite already uses a branching pattern that closely resembles Git Flow (master, develop, feature/* branches, semantic version tags). However, this is implicit convention, not a formalized strategy. Key gaps exist: no release/* or hotfix/* branches, no CI on develop/PR, and the release script references main instead of master.

Should we adopt Git Flow formally and close these gaps?

Current State (as of v1.2.0)

Git Flow Concept Status
master branch (production) ✅ Exists
develop branch (integration) ✅ Exists, is the working branch
feature/* branches ✅ Already used (feature/schema-evolution, etc.)
Semantic version tags ✅ v0.7.0 → v1.2.0
Release script ✅ Exists (workflow/scripts/release.sh)
File-based feature workflow ✅ workflow/ directory
release/* branches ❌ Missing — tags created directly from develop
hotfix/* branches ❌ Missing — no convention for emergency fixes off master
CI on develop/PRs ❌ Missing — only tag-triggered release.yml exists
Branch protection ❌ Missing — no GitHub branch protection rules
Release script branch check ⚠️ References main, actual branch is master

Decision Drivers

  • Solo developer workflow — ceremony must not outweigh value
  • AI agents benefit from explicit, documented conventions
  • Production hotfix path must exist before it's needed
  • CI on develop catches regressions before release ceremony
  • Current approach already follows ~80% of Git Flow

Considered Options

Option 1: Formalize Git Flow (full ceremony)

Adopt the complete Git Flow model:

  • release/vX.Y.Z branches from develop for stabilization
  • hotfix/vX.Y.Z branches from master for emergency fixes
  • Back-merges from release/hotfix to develop
  • CI gates on develop, release/, hotfix/, and PRs

Option 2: Simplified Git Flow (current + fixes)

Keep current approach, add only:

  • CI on develop
  • Hotfix convention (documented, not enforced)
  • Fix release.sh main → master

Option 3: Trunk-Based Development

Abandon long-lived develop branch. All work on master with short-lived feature branches, feature flags, and continuous deployment.

Decision Outcome

Chosen option: "Option 1: Formalize Git Flow", because the project already follows 80% of it, the remaining 20% provides real value (hotfix path, CI gates, release stabilization), and formal documentation helps AI agents follow the conventions correctly.

Implementation Checklist

  1. Fix release.sh — Change main reference to master; add release branch workflow (release/vX.Y.Z from develop → merge to master + back-merge to develop)
  2. Add CI workflow — .github/workflows/ci.yml running Racket tests on push to develop, release/*, hotfix/*, and on PRs
  3. Add hotfix script — workflow/scripts/hotfix.sh vX.Y.Z creating branch from master, merging to both master and develop
  4. GitHub branch protection — Protect master (require PR, no force push) and develop (require CI pass)
  5. Document in CONTRIBUTING.md — Add branching model section with diagram

Branch Naming Conventions

Branch Source Merges To Purpose
master — — Production-ready code, tagged releases
develop master — Integration branch, next release
feature/FTR-NNN-* develop develop Feature work (matches workflow/ feature IDs)
release/vX.Y.Z develop master + develop Release stabilization
hotfix/vX.Y.Z master master + develop Emergency production fixes

Positive Consequences

  • Clear hotfix path exists before it's needed
  • CI catches regressions on develop before release ceremony
  • Release branches allow stabilization without blocking feature work
  • AI agents have unambiguous branch conventions to follow
  • Aligns with existing workflow/ feature lifecycle

Negative Consequences

  • Added ceremony for releases (create branch, merge twice, delete)
  • Hotfix back-merge to develop can conflict if develop has diverged
  • Solo developer overhead — most value comes with multiple contributors

Pros and Cons of the Options

Option 1: Formalize Git Flow

  • Good, because it builds on existing practice (minimal disruption)
  • Good, because hotfix path is pre-established
  • Good, because release branches decouple stabilization from feature work
  • Bad, because added ceremony for a solo developer
  • Bad, because release branch may be unnecessary if develop is always stable

Option 2: Simplified Git Flow

  • Good, because minimal overhead
  • Good, because CI on develop is the highest-value change
  • Bad, because hotfix path is documented but not tooled
  • Bad, because no release stabilization window

Option 3: Trunk-Based Development

  • Good, because simplest branching model
  • Good, because eliminates merge conflicts between long-lived branches
  • Bad, because requires feature flags (not natural for a code generator)
  • Bad, because discards existing develop/feature branch workflow
  • Bad, because no stabilization window for releases

AI Agent Guidance

When This Applies

  • Creating any new branch (use correct prefix and source)
  • Preparing a release (create release/vX.Y.Z from develop, not tag directly)
  • Fixing a production bug (create hotfix/vX.Y.Z from master)
  • Setting up CI workflows

When This Does NOT Apply

  • Experimental/throwaway branches (use any name, delete after)
  • Changes to the DSL, generators, or framework (those are feature content, not branching decisions)

Key Rules

  • Never commit directly to master — always via release or hotfix branch
  • Never commit directly to develop — always via feature branch or back-merge
  • Feature branches start from develop and merge back to develop
  • Release branches start from develop, merge to master AND back to develop
  • Hotfix branches start from master, merge to master AND back to develop
  • Tags are created on master only, after release branch merge
