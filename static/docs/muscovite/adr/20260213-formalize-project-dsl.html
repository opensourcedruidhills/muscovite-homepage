Formalize Project DSL with PEG Grammar, Dedicated Parser, and `.mus` Extension

Formalize Project DSL with PEG Grammar, Dedicated Parser, and .mus Extension

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-13
  • Tags: grammar, dsl, project-dsl, parser, file-extension, documentation, testing

Context and Problem Statement

Muscovite has two DSLs sharing the .ddd file extension:

  1. Modeling DSL — CONTEXT, ENTITY, AGGREGATE, RELATIONSHIPS, etc. Formally specified in dslgen/grammar/muscovite.peg (164 PEG rules), parsed by dslgen/parser/parser.rkt (~2,900 lines), auto-documented in doc/DSL.md, covered by grammar conformance tests (24 construct categories).

  2. Project DSL — PROJECT, OUTPUT, DATABASE, GENERATE, CAPABILITIES, OAUTH2, FRONTENDS, CONTEXTS, VALIDATION. Not formally specified. The PEG grammar declares an opaque stub: ProjectBlock <- 'PROJECT' STRING? BracedBlock? where BracedBlock is '{' ( BracedBlock / !'}' . )* '}' — balanced-brace matching with no interior validation. Parsed by an ad-hoc inline parser in dslgen/generate.rkt (~170 lines) with its own tokenizer, completely separate from the modeling DSL parser. No tests, no reference documentation, no grammar conformance coverage.

Additionally, a dead S-expression parser exists in dslgen/parser/project-dsl.rkt (323 lines) that expects (project name (version "1.0") ...) format — a format no project file uses. This module is unreferenced dead code.

How This Happened

The project DSL grew organically: CAPABILITIES was added for security/spatial gating, OAUTH2 for authentication configuration, FRONTENDS for Qt widget options — all without grammar updates, without tests, and without documentation. Each addition followed the pattern of extending the inline parser in generate.rkt rather than formalizing the syntax. The grammar-first ADR (20260211) was not applied because the project DSL was perceived as "just configuration" rather than a language requiring the same engineering discipline.

Concrete Problems

Problem Impact
No formal grammar for project DSL AI agents cannot validate generated .mus files; syntax is defined only by parser implementation
Same .ddd extension for both DSLs Scripts, tools, and AI agents cannot distinguish project files from model files
Inline parser in generate.rkt Parser and orchestrator are tightly coupled; impossible to test the parser in isolation
Dead S-expression parser 323 lines of confusion — appears to be an alternative format but is unreferenced
No project DSL tests CAPABILITIES, OAUTH2, FRONTENDS parsing has zero test coverage
No reference documentation No equivalent of doc/DSL.md for the project DSL
Grammar conformance gap The conformance test checks 24 modeling constructs but zero project constructs — ProjectBlock is not in expected-rules

Decision Drivers

  • Grammar-first discipline (ADR 20260211): Every accepted syntax must be formally specified in the PEG grammar
  • Separation of concerns: Parser and orchestrator should be independently testable modules
  • File extension clarity: AI agents, editors, and tooling should distinguish project configuration from domain models
  • Consistency: The project DSL should receive the same engineering treatment as the modeling DSL
  • Clean Racket API: The parser module should follow Racket conventions — opaque structs, provide'd accessors, contracts

Considered Options

Option 1: Formalize in PEG Grammar + Dedicated Parser + .mus Extension

Add sections 44–52 to muscovite.peg formalizing all project DSL constructs. Extract the inline parser from generate.rkt into a dedicated dslgen/parser/project-parser.rkt module with clean Racket API. Introduce .mus file extension for project files. Replace dead project-dsl.rkt with the new parser. Create test harness and reference doc generator.

Option 2: Keep Ad-Hoc Parser, Add Tests Only

Leave the inline parser in generate.rkt, add unit tests around it, but don't formalize the grammar. Lower effort but doesn't address the core problem of having unspecified syntax.

Option 3: Merge Into Modeling DSL Parser

Parse PROJECT blocks using the main modeling DSL parser in parser.rkt. Would require the modeling parser to handle configuration-style key-value syntax, mixing concerns.

Decision Outcome

Chosen option: Option 1 — Formalize in PEG Grammar + Dedicated Parser + .mus Extension, because it applies the same grammar-first discipline to both DSLs, cleanly separates the parser from the orchestrator, and eliminates the file extension ambiguity.

New File Extension: .mus

Extension Purpose Example
.ddd Domain model files (contexts, entities, aggregates) contexts/patient.ddd
.mus Project configuration (build targets, capabilities, context includes) project.mus

Rationale: .mus is short for "Muscovite", clearly distinguishes project files from model files, and is unused by other tools.

Parser Module API

;; dslgen/parser/project-parser.rkt
(provide
  parse-project-file          ; path? → project-config?
  project-config?             ; predicate
  project-config-name         ; project-config? → string?
  project-config-description  ; project-config? → string?
  project-config-version      ; project-config? → string?
  project-config-output-dir   ; project-config? → (or/c string? #f)
  project-config-sqitch-project ; project-config? → (or/c string? #f)
  project-config-databases    ; project-config? → (hash/c string? database-config?)
  project-config-generate-options ; project-config? → (hash/c string? boolean?)
  project-config-context-files ; project-config? → (listof string?)
  project-config-capabilities ; project-config? → (listof string?)
  project-config-oauth2-config ; project-config? → (hash/c string? string?)
  project-config-frontends-config ; project-config? → (or/c hash? #f))

The struct is opaque (no #:transparent) — consumers use accessor functions only. This allows internal representation changes without breaking callers.

FRONTENDS Block

Formalized structurally in the grammar. Generator support marked as planned — the parser accepts and stores the block, but generators do not yet consume it (same as current behavior where it's skipped with balanced-brace matching).

Migration

Rename all 4 project files: project.ddd → project.mus. Update all references (~100 occurrences across ~40 files). ADR references in historical decision records are left as-is (they document decisions made at the time).

Positive Consequences

  • Single grammar-first process for both DSLs — eliminates the "two parsers, one formal, one ad-hoc" inconsistency
  • .mus extension disambiguates project files from domain files — IDEs, scripts, and AI agents can distinguish them
  • Auto-generated doc/PROJECT-DSL.md eliminates documentation drift
  • Test harness catches regressions when project syntax evolves
  • Retires 323 lines of dead code (parser/project-dsl.rkt)
  • AI agents can generate valid .mus files from formal grammar
  • FTR-032 (underscore keywords, v1.13.0) benefits — project DSL keywords already use underscores, providing consistency

Negative Consequences

  • Grammar grows ~80–100 rules (from 164 → ~250) — manageable, clearly sectioned
  • 4 file renames + ~100 reference updates — small blast radius, sed-driven
  • generate.rkt refactoring risk — inline parser is coupled to orchestration; extraction requires care at the interface boundary

AI Agent Guidance

When This Decision Applies

  • Creating or modifying project configuration files (now .mus, not .ddd)
  • Adding new project-level configuration keywords
  • Parsing project files in any Racket module
  • Generating project files from brownfield systems

Enforcement

  • Grammar-first: New project DSL keywords MUST appear in PEG grammar sections 44–53 before parser implementation
  • File extension: Project files MUST use .mus extension. The --project CLI flag accepts .mus files.
  • Parser module: All project parsing MUST go through parser/project-parser.rkt — never inline in generate.rkt or other modules
  • Tests: New project DSL constructs MUST have parser test coverage in tests/test-project-parser.rkt
