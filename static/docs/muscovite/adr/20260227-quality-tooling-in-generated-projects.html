Quality Tooling in Generated Projects

Quality Tooling in Generated Projects

  • Status: accepted
  • Date: 2026-02-27
  • Tags: cmake, sanitizers, linters, clang-tidy, cppcheck, generated-code, layer2
  • Deciders: Johannes Lochmann

Technical Story: Generated standalone projects (Layer 2) lack the quality infrastructure that the framework (Layer 1) already has — sanitizer presets, clang-tidy integration, cppcheck, and coverage tooling. This means consumers of generated projects get a degraded development experience compared to the framework itself.

Context and Problem Statement

The Muscovite framework (Layer 1) has mature quality tooling:

  • clang-tidy integration via cmake/StaticAnalysis.cmake with 3 options (MUSCOVITE_ENABLE_CLANG_TIDY, MUSCOVITE_ENABLE_CPPCHECK, MUSCOVITE_ENABLE_IWYU)
  • Compiler warnings via cmake/CompilerWarnings.cmake (-Wall -Wextra -Wpedantic -Werror as an interface library)
  • AddressSanitizer + UBSan preset (asan) in CMakePresets.json
  • ThreadSanitizer preset (tsan) in CMakePresets.json
  • Code coverage preset (coverage) with lcov/genhtml targets
  • .clang-tidy configuration with 90+ checks tuned for C++23/Qt/Boost/pqxx

However, generated standalone projects (examples/*/generated/) receive none of this:

  • CMakePresets.json emitted by build-system-packaging.rkt only has debug, release, relwithdebinfo — no asan, tsan, or coverage presets
  • No .clang-tidy file is generated
  • No StaticAnalysis.cmake is generated
  • The generated CMakeLists.txt has inline -Wall -Wextra -Wpedantic -Werror but no clang-tidy/cppcheck option hooks
  • Framework sanitizer presets lack shared-library linker flags (BUILD_SHARED_LIBS=ON is the default)

This violates the project's principle that generated projects should be production-grade standalone applications — a consumer who git clones a generated project should get the same quality infrastructure as the framework developer.

Decision Drivers

  • Generated projects are the product — they must be self-contained with production-grade tooling
  • The framework already has working quality infrastructure; generated projects should inherit equivalent capability
  • Sanitizers catch real bugs: UB, use-after-free, data races, uninitialized reads — particularly important for generated pqxx repository code and gRPC service code
  • clang-tidy catches style regressions and common C++23 pitfalls in generated code
  • Coverage tooling enables consumers to measure their test coverage from day one
  • BUILD_SHARED_LIBS=ON requires sanitizer flags on both compile and link lines

Considered Options

  1. Emit full quality tooling into generated projects — CMake presets, .clang-tidy, StaticAnalysis.cmake, all wired into the generated build system
  2. Partial: presets only — add asan/tsan/coverage presets but skip linter integration
  3. No change — keep quality tooling in framework only, consumers add their own

Decision Outcome

Chosen option: Option 1 — Emit full quality tooling, because:

  • It aligns with the "one .ddd file generates a complete project" value proposition
  • The marginal generator complexity is low (CMake preset generation is templated)
  • Consumers get immediate value without manual CMake setup
  • The .clang-tidy configuration can be sensibly scoped for generated code patterns

Detailed Decisions

1. Generated CMakePresets.json: Add Sanitizer and Coverage Presets

The generate-cmake-presets function in build-system-packaging.rkt MUST emit:

Preset Type Flags
asan configure + build + test -fsanitize=address,undefined -fno-sanitize-recover=all -fno-omit-frame-pointer on compile + link
tsan configure + build + test -fsanitize=thread -g -O1 on compile + link
coverage configure + build + test -fprofile-arcs -ftest-coverage --coverage -g -O0 on compile + link

Test presets MUST include ASAN_OPTIONS and TSAN_OPTIONS environment variables.

2. Generated .clang-tidy

A .clang-tidy file MUST be emitted at the project root with checks appropriate for generated C++23 code. Derived from the framework's configuration but with relaxed naming checks (generated code uses its own conventions consistently).

3. Generated StaticAnalysis.cmake

A cmake/StaticAnalysis.cmake MUST be emitted with options for ENABLE_CLANG_TIDY and ENABLE_CPPCHECK. The generated CMakeLists.txt MUST include(StaticAnalysis).

4. Framework Preset Linker Flag Fix

The framework's own CMakePresets.json asan and tsan presets MUST add CMAKE_EXE_LINKER_FLAGS and CMAKE_SHARED_LINKER_FLAGS with the sanitizer flags, since BUILD_SHARED_LIBS=ON is the default.

5. Mold Linker Exclusion for Sanitizer Builds

BuildAcceleration.cmake MUST explicitly skip mold for sanitizer builds. Mold + sanitizers can produce false positives or miss instrumented symbols.

Positive Consequences

  • Generated projects are truly standalone with production-grade quality infrastructure
  • Consumers discover UB, memory errors, and data races from day one
  • clang-tidy enforces consistent code quality in generated output
  • Coverage measurement available without manual CMake setup
  • Framework sanitizer presets work correctly with shared libraries

Negative Consequences

  • Additional generator complexity (~100 lines in build-system-packaging.rkt)
  • Generated .clang-tidy needs maintenance when framework config evolves
  • Sanitizer builds are slower (expected, acceptable)

Compliance

This ADR does NOT change DSL syntax, IR structs, or parser behavior. It only affects:

  • CMake generator output (Layer 0 → Layer 2)
  • Framework build configuration (Layer 1)

AI Agent Guidance

When This Decision Applies

  • When modifying dslgen/generators/cmake/build-system-packaging.rkt — ensure sanitizer/coverage presets are emitted
  • When modifying dslgen/generators/cmake/build-system-generator.rkt — ensure include(StaticAnalysis) is in generated CMakeLists.txt
  • When modifying framework CMakePresets.json — ensure sanitizer presets have both compile AND link flags
  • When modifying cmake/BuildAcceleration.cmake — ensure mold is excluded for sanitizer builds

Key Files

  • dslgen/generators/cmake/build-system-packaging.rkt — CMakePresets.json generation
  • dslgen/generators/cmake/build-system-generator.rkt — CMakeLists.txt generation
  • CMakePresets.json — framework presets (fix linker flags)
  • cmake/BuildAcceleration.cmake — mold/ccache (sanitizer exclusion)
  • cmake/StaticAnalysis.cmake — template for generated version
  • .clang-tidy — template for generated version
