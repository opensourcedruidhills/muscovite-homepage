Extract C++ Framework to Separate Repository

Extract C++ Framework to Separate Repository

  • Status: accepted
  • Deciders: Jolo
  • Date: 2026-02-01

Context and Problem Statement

Muscovite generates full-stack applications from DDD models. The current monorepo contains:

  1. Racket DSL — Parser, IR, and code generators (language-agnostic)
  2. C++ Framework — Runtime libraries consumed by generated C++ code

We want to support multiple backend languages (Rust, Go, TypeScript) in the future. Each language ecosystem has its own package manager and community expectations. Keeping all runtimes in one repo creates friction.

Decision Drivers

  • Enable multi-language backend generation (Rust, Go, etc.)
  • Allow independent versioning of runtime libraries
  • Publish C++ framework to Conan Center for broader adoption
  • Simplify CI/CD — C++ builds don't need Racket, Racket tests don't need C++ toolchain
  • Lower barrier for language-specific contributors

Considered Options

  1. Keep monorepo — All languages in one repo with subdirectories
  2. Extract C++ framework only — Separate repo for C++ runtime, generators stay
  3. Full polyrepo — Each generator + runtime in its own repo

Decision Outcome

Chosen option: "Extract C++ framework only", because:

  • Generators share the same IR/AST and must evolve together
  • Runtime libraries are consumed differently per ecosystem (Conan vs Cargo vs go mod)
  • Keeps code generation as single source of truth
  • Avoids "generator drift" between languages

Implementation

Repository Structure After Extraction

muscovite/muscovite (this repo):

muscovite/
├── dslgen/                 # Racket: Parser → IR → ALL generators
│   ├── model/              # Language-agnostic IR (ddd-types.rkt)
│   ├── parser/             # DSL parser
│   ├── generators/
│   │   ├── postgres/       # Sqitch migrations (shared)
│   │   ├── grpc/           # Proto definitions (shared)
│   │   ├── cpp/            # C++ DBA, services (uses muscovite-cpp)
│   │   ├── qt/             # Qt Widgets (uses muscovite-cpp)
│   │   ├── rust/           # (future) Rust repos, services
│   │   └── go/             # (future) Go repos, services
│   └── migration/          # Schema evolution tools
├── examples/               # Sample DDD models
└── (no framework/)

muscovite/muscovite-cpp (new repo):

muscovite-cpp/
├── dba/                    # Connection pool, unit of work, type converters
├── server/                 # gRPC server utilities, security services
├── grpc/                   # Type mapping, interceptors, oauth2
├── qt/                     # Validation framework, widgets, auth
├── types/                  # Strong type validators (Email, Phone, etc.)
├── spatial/                # PostGIS geometry helpers
├── geo/                    # Geographic utilities
├── CMakeLists.txt
├── conanfile.py
└── README.md

Package Consumption

Generated C++ projects will consume via Conan:

# conanfile.py in generated project
def requirements(self):
    self.requires("muscovite-cpp/0.4.0")
# CMakeLists.txt in generated project
find_package(muscovite-cpp REQUIRED COMPONENTS dba server grpc qt)

Versioning Strategy

Component Version Notes
muscovite (generators) Semantic, tied to DSL Breaking DSL changes = major bump
muscovite-cpp Semantic, independent API changes = major bump
muscovite-rust (future) Semantic, independent Per Cargo conventions

Migration Steps

  1. Create muscovite/muscovite-cpp repository on GitHub
  2. Use git subtree split --prefix=framework to extract with history
  3. Push to new repository
  4. Update generators to reference external muscovite-cpp package
  5. Remove framework/ from main repo
  6. Update CI/CD pipelines

Consequences

Positive

  • Multi-language ready: Each runtime is first-class in its ecosystem
  • Conan Center publishing: conan install muscovite-cpp without Racket
  • Independent releases: Hotfix C++ runtime without touching generators
  • Cleaner CI: Separate pipelines for generators vs runtime libraries
  • Community contributions: Rust developers don't need to understand C++

Negative

  • Two repos to maintain: Coordination overhead for breaking changes
  • Version compatibility matrix: Must document which generator version works with which runtime
  • Initial setup complexity: Contributors need both repos for full-stack development

Mitigations

  • Compatibility table in README and generated code comments
  • CI integration tests that pull both repos and validate end-to-end
  • Monorepo development mode: Symlink framework/ for local iteration

AI Agent Guidance

When This Decision Applies

  • Adding a new backend language (create new muscovite-<lang> repo)
  • Modifying C++ framework code (work in muscovite-cpp repo)
  • Changing generated C++ code patterns (work in dslgen/generators/cpp/)

When This Decision Does NOT Apply

  • DSL syntax changes (dslgen/parser/ in main repo)
  • PostgreSQL/Sqitch generation (dslgen/generators/postgres/)
  • Proto/gRPC definitions (dslgen/generators/grpc/proto.rkt)

Key Files

  • dslgen/generators/cpp/dba/unified.rkt — References muscovite-cpp headers
  • dslgen/generators/grpc/main.rkt — References muscovite-cpp grpc utilities
  • dslgen/generators/qt/cmake.rkt — References muscovite-cpp qt components

Package Reference in Generated Code

# Generated CMakeLists.txt will contain:
find_package(muscovite-cpp REQUIRED COMPONENTS dba server)
target_link_libraries(myproject PRIVATE muscovite-cpp::dba muscovite-cpp::server)
