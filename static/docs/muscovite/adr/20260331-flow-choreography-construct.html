Flow Choreography Construct

Flow Choreography Construct

  • Status: draft
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-31
  • Tags: dsl, ddd, flow, choreography, events, commands, documentation, grammar, ir
  • Supersedes: none

Technical Story: RFC-001 Extension 7 — add FLOW as a declarative event/command choreography construct for process documentation and validation.

Context and Problem Statement

Muscovite expresses event-driven behavior through POLICY (event → action), SAGA (multi-step orchestration), and COMMAND (with EMITS clause). These are powerful but distributed — to understand the full event chain for a business process, you must read multiple constructs across the context and mentally reconstruct the flow.

There is no single construct that shows "when X happens, Y and Z follow, then W completes the process." This makes it hard for domain experts to review the event chain and for architects to validate that the declared behavior matches the intended process.

Decision Drivers

  • Domain experts need readable business process overviews
  • Architects need to validate that event chains are complete and consistent
  • Flows are documentation/validation constructs — they do NOT replace underlying declarations
  • Must support parallel (AND), exclusive (XOR), and inclusive (OR) branching
  • Grammar-first mandate (ADR 20260211)

Considered Options

Option 1: FLOW construct with branching operators (chosen)

A FLOW block inside a CONTEXT that chains events and commands with + (AND), X (XOR), and O (OR) operators.

  • Good, because readable overview of complete event chains
  • Good, because cross-validates against COMMAND/EVENT/POLICY/SAGA declarations
  • Good, because generates sequence diagrams directly
  • Good, because branching operators express parallelism and exclusive outcomes
  • Bad, because duplicate declaration (flow + underlying constructs)

Option 2: Auto-derive flows from existing constructs

Walk COMMAND → EMITS → POLICY → COMMAND chains to reconstruct flows automatically.

  • Good, because no additional DSL constructs
  • Bad, because implicit — the full chain is a derivation, not a declaration
  • Bad, because cannot express intended parallelism vs. incidental ordering
  • Bad, because errors in the chain are hard to diagnose without an explicit reference

Option 3: External BPMN/sequence diagram files

Side-by-side .bpmn or .puml files alongside .ddd models.

  • Good, because mature diagramming tooling
  • Bad, because drifts from the model
  • Bad, because not validated against actual COMMAND/EVENT declarations

Decision Outcome

Chosen option: "FLOW construct with branching operators", because explicit flow declarations serve as both readable documentation and validation baseline. The redundancy with underlying constructs is intentional — the flow is the "what should happen" against which the "what is declared" is validated.

DSL Syntax

FLOW OrderFulfillment @uuid "..." {

    COMMAND PlaceOrder
        DELEGATES_TO OrderAggregate
        EMITS OrderPlaced;

    EVENT OrderPlaced
        TRIGGERS ValidatePayment + CheckInventory
        DOC "Both validation and inventory check run in parallel";

    COMMAND ValidatePayment
        DELEGATES_TO PaymentAggregate
        EMITS PaymentApproved X PaymentDeclined
        DOC "Exactly one outcome";

    EVENT PaymentApproved
        TRIGGERS ReserveInventory;

    EVENT PaymentDeclined
        TRIGGERS CancelOrder;

    COMMAND ReserveInventory
        DELEGATES_TO InventoryAggregate
        EMITS InventoryReserved X InventoryUnavailable;

    EVENT InventoryReserved
        TRIGGERS ShipOrder;

    EVENT InventoryUnavailable
        TRIGGERS CancelOrder + RefundPayment
        DOC "Both cancellation and refund on stockout";

    COMMAND ShipOrder
        DELEGATES_TO FulfillmentAggregate
        EMITS OrderShipped;

    EVENT OrderShipped COMPLETES;

    DOC "End-to-end order fulfillment from placement to shipping";
}

Operators

Operator Meaning Example
+ AND — all targets triggered in parallel TRIGGERS A + B
X XOR — exactly one target triggered EMITS Success X Failure
O OR — one or more targets triggered TRIGGERS A O B

Rules

  • FLOW is declared inside a CONTEXT block
  • References to commands and events must resolve within the context or imported contexts
  • DELEGATES_TO references an aggregate within the context
  • EMITS references one or more events with optional branching operators
  • TRIGGERS references one or more commands with optional branching operators
  • COMPLETES marks terminal events in the flow
  • A flow does NOT replace underlying COMMAND/EVENT/POLICY/SAGA declarations — it is a documentation/validation overlay
  • Flows carry @uuid for semantic diff tracking
  • Flows can carry @compliance and @quality annotations

PEG Grammar

Flow             <- 'FLOW' IDENTIFIER UuidAnnotation ComplianceAnnotation*
                    QualityAnnotation*
                    '{' FlowStep+ DocClause? '}'

FlowStep         <- FlowCommand / FlowEvent / DocClause

FlowCommand      <- 'COMMAND' IDENTIFIER
                    DelegatesToClause?
                    EmitsClause?
                    DocClause? ';'?

FlowEvent        <- 'EVENT' IDENTIFIER
                    ( TriggersClause / CompletesClause )?
                    DocClause? ';'?

DelegatesToClause <- 'DELEGATES_TO' IDENTIFIER
EmitsClause      <- 'EMITS' FlowTargetList
TriggersClause   <- 'TRIGGERS' FlowTargetList
CompletesClause  <- 'COMPLETES'
FlowTargetList   <- IDENTIFIER ( FlowOperator IDENTIFIER )*
FlowOperator     <- '+' / 'X' / 'O'

IR Implications

  • New IR node type: Flow containing ordered FlowStep entries
  • Each step references commands/events by UUID
  • Operators stored as enum: AND, XOR, OR

Generator Implications

  • Documentation: sequence diagrams generated directly from flow declarations. Each flow produces a Mermaid sequence diagram showing the complete event chain with parallel/exclusive branching. Domain-expert docs show business process flows. Developer docs show the same flows with aggregate/service annotations.
  • Validation: cross-check flow declarations against actual COMMAND/EVENT/POLICY/SAGA definitions. Warn if a flow references an event that no command emits, or a command that no event triggers.
  • Test generator: flow declarations feed generated integration test skeletons that exercise the complete happy path and each branching alternative.

Positive Consequences

  • Business processes are readable as single constructs
  • Sequence diagrams generated from the model
  • Cross-validation catches inconsistencies between flow intent and actual declarations
  • Integration test skeletons derived from flows

Negative Consequences

  • Redundancy between flows and underlying COMMAND/EVENT/POLICY/SAGA declarations
  • Flows must be maintained alongside the constructs they describe
  • Branching operators (X, O) may confuse non-technical readers

AI Agent Guidance

When This Decision Applies

  • When adding FLOW choreography constructs to a bounded context
  • When generating sequence diagrams or process documentation
  • When validating that declared flows are consistent with COMMAND/EVENT declarations

When This Decision Does NOT Apply

  • Individual COMMAND/EVENT/POLICY/SAGA declarations (unchanged)
  • Infrastructure concerns (flow is a domain/documentation construct)

Enforcement

  • PEG grammar rules for Flow/FlowStep/FlowOperator
  • Parser validation: command/event references must resolve within context
  • Cross-validation: flow steps must match actual COMMAND/EVENT declarations
  • COMPLETES required: every flow must have at least one terminal event
