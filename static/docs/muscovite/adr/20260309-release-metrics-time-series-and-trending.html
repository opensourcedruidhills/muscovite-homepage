ADR: Release Metrics Time Series and Trending

ADR: Release Metrics Time Series and Trending

  • Status: accepted
  • Date: 2026-03-09
  • Supersedes: None
  • Extends: ADR 20260206-build-profiling-instrumentation, ADR 20260307-mandatory-multi-layer-test-gates, ADR 20260210-racket-generator-test-suite
  • Implements: Iteration v1.60.0 (metrics track)

Context

Muscovite has three categories of release-time metrics:

  1. Test timing — well-implemented (FTR-086). collect-release-metrics.sh runs 8 test layers, records per-layer wall-clock seconds, and saves versioned baselines to metrics/baselines/vX.Y.Z.json. The system brief (Section 11) displays these.

  2. Build timing — tools exist but are dormant. extract-ninja-metrics.py and merge-build-metrics.py (FTR-010, v1.6.0) parse Ninja logs and produce JSON, but release.sh never calls them. build-history.jsonl has exactly one entry from February 2026. The system brief Section 10 usually shows "Build metrics not available."

  3. Code coverage — infrastructure exists, zero execution. CMake coverage preset with gcov flags, coverage-report lcov target, and raco cover are all defined in ADRs and CMake files. No coverage data has ever been collected. Not mentioned in release.sh.

The original plan (ADR 20260206) explicitly called for time-series trending with regression detection at a 15% threshold. This was partially implemented for build timing but never activated. Test timing baselines are saved per release but no trend analysis exists — each release only compares against the previous baseline for pass/fail regressions, not for timing trends.

The gap: We collect pass/fail per release but not how fast things are getting or how much code is covered. Without trends, we cannot detect gradual performance degradation or coverage erosion.

Decision

1. Baseline Files Are the Time Series

Each metrics/baselines/vX.Y.Z.json becomes the canonical per-release metrics record. The schema is extended to include build timing and coverage data alongside the existing test layer data.

Extended baseline schema (new fields marked with +):

{
  "version": "vX.Y.Z",
  "timestamp": "ISO8601",
  "git_sha": "8-char",
  "all_green": true,
  "totals": { "layers_passed": 8, "layers_failed": 0, "layers_skipped": 0, "regressions": 0 },
  "racket_detail": { "suites": 93, "passed": 93, "failed": 0, "wall_time_ms": 439783 },
  "cpp_detail": { "tests": 28, "passed": 27, "failed": 1, "wall_time_s": 11.1 },
  "layers": { "layer-name": { "status": "pass|fail|skip", "wall_s": N } },
  "regressions_detail": [],
  "baseline_compared": "vX.Y.Z",
+ "build_detail": {
+   "s1_framework_wall_ms": N,
+   "s1_compile_ms": N,
+   "s1_link_ms": N,
+   "s2_generation_ms": N,
+   "s3_project_wall_ms": N,
+   "s3_compile_ms": N,
+   "s3_link_ms": N,
+   "total_build_wall_ms": N,
+   "nproc": N,
+   "ram_gb": N.N
+ },
+ "coverage_detail": {
+   "cpp_line_pct": N.N,
+   "cpp_branch_pct": N.N,
+   "cpp_function_pct": N.N,
+   "racket_module_pct": N.N,
+   "racket_expression_pct": N.N,
+   "racket_coverable_modules": N,
+   "racket_covered_modules": N
+ }
}

2. Build Profiling Wired into Release Ceremony

release.sh calls merge-build-metrics.py after the C++ build gate passes. This:

  • Appends to metrics/build-history.jsonl (high-frequency, non-release snapshots continue to work)
  • Feeds build_detail into the baseline via collect-release-metrics.sh

3. C++ Coverage Collected Every Release

After the standard test gates pass (using the debug preset), release.sh runs a second build+test cycle with the coverage preset, then extracts lcov summary percentages. This adds ~5 minutes to the release ceremony but provides line/branch/function coverage.

Sequence: cmake --preset coverage → cmake --build --preset coverage → ctest --preset coverage → cmake --build --preset coverage --target coverage-report → parse lcov summary.

4. Racket Coverage Best-Effort

raco cover is run on the subset of test suites that are rackunit-compatible. As tests are migrated (ADR 20260210 Phase 3), coverage increases. The metric tracks coverable_modules (denominator) and covered_modules (numerator) so the percentage is always meaningful even with partial compatibility.

5. Coverage Is Tracked, Not Gated

Coverage percentages are recorded in baselines and displayed in the system brief, but coverage thresholds do NOT block releases. Only the trend direction matters:

  • Improving or stable: ✅ no action
  • Declining >5% from previous release: ⚠️ warning in system brief
  • Declining >15% from previous release: 🔴 alert (still not blocking — investigation required)

This avoids the "coverage theater" antipattern where teams game percentages. The time series lets the project owner spot real trends.

6. Trend Analysis Tool

tools/metrics-trend.sh reads all metrics/baselines/v*.json files (sorted by version) and produces a trend report:

  • Last 10 releases in a table with ▲/▼ deltas for each metric
  • Regression flags for >15% build time increase or >5% coverage decrease
  • Output format: both human-readable (terminal table) and JSON (for system brief consumption)

7. System Brief Enhancement

generate-system-brief.sh Sections 10 and 11 are enhanced to:

  • Show build timing trends (last 5 releases)
  • Show coverage trends (last 5 releases)
  • Show timing trends per test layer (last 5 releases)
  • Flag regressions with ⚠️/🔴 indicators

Consequences

Positive

  • Every release produces a complete metrics snapshot: test pass/fail, test timing, build timing, code coverage
  • Time series enables trend analysis — gradual degradation becomes visible
  • System brief becomes a living dashboard, not a snapshot
  • build-history.jsonl continues for development-time profiling; baselines are the release-time authority

Negative

  • Release ceremony adds ~5 minutes for coverage build+test cycle
  • Racket coverage will be partial until test migration completes (ADR 20260210)
  • More data in baselines means more to maintain in schema evolution

Neutral

  • build-history.jsonl retains its role for high-frequency, non-release snapshots (manual merge-build-metrics.py calls during development)
  • FTR-010 is retroactively marked complete — tools shipped, wiring was the missing piece
  • FTR-304 (v1.62.0, DECIDE branch coverage) is unaffected — it measures a different thing (logic branch coverage of generated code, not source code coverage)
