Value Object Storage Modes with Composite Types

Value Object Storage Modes with Composite Types

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-01-17
  • Tags: ddd, value-objects, postgresql, composite-types, star-schema

Technical Story: Implement proper DDD Value Objects without JSON/JSONB while maintaining type safety and supporting collections.

Context and Problem Statement

Value Objects are immutable domain concepts defined by their attributes rather than identity. They are fundamental to DDD but our current implementation is incomplete. We need a storage strategy that:

  1. No JSON/JSONB: Maintains full relational integrity and type safety
  2. Supports Collections: Value Objects often appear as lists (phone numbers, addresses, measurements)
  3. Handles Defaults: Values must propagate through PostgreSQL → API → gRPC → Qt
  4. Star Schema Support: Reusable codes/concepts should be normalized

Decision Drivers

  • Type safety across all generated layers
  • PostgreSQL parameter limits (max 32,767 per query)
  • Performance for batch operations
  • FHIR-style CodeableConcept pattern requirements
  • Audit trail requirements (created_at/by, modified_at/by)

Considered Options

  1. JSONB storage - Rejected per ADR, violates type safety
  2. Separate table per usage - Duplicates structure, hard to query
  3. Composite types + storage modes - Three categories based on usage pattern
  4. EAV (Entity-Attribute-Value) - Anti-pattern, rejected

Decision Outcome

Chosen option: "Composite types + storage modes" with three categories:

Storage Mode Categories

Mode Annotation Use Case Storage Example
Inline @inline Single-value mixin Flattened columns AuditInfo
Owned @owned Per-entity collection Child table + FK Measurement[]
Shared @shared Reusable dimensions Star schema table CodeableConcept

DSL Syntax

-- INLINE: Fields merged into parent table
VALUE_OBJECT AuditInfo @inline (
    created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_by  UUID,
    modified_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    modified_by UUID
)
COMPUTED modified_at ON UPDATE: NOW()
IMMUTABLE created_at, created_by;

-- OWNED: Child table with FK, supports collections
VALUE_OBJECT Measurement @owned (
    value       DECIMAL     NOT NULL,
    unit        TEXT        NOT NULL,
    effective_at TIMESTAMPTZ DEFAULT NOW(),
    quality     TEXT        DEFAULT 'preliminary'
)
COMPUTED ordinal ON INSERT: auto_increment per parent;

-- SHARED: Dimension table, reusable across entities
VALUE_OBJECT CodeableConcept @shared (
    system      TEXT        NOT NULL,
    code        TEXT        NOT NULL,
    display     TEXT,
    version     TEXT        DEFAULT 'latest'
);

-- Usage in entities
ENTITY Observation IDENTIFIED_BY id (
    id          UUID        NOT NULL,
    subject_id  UUID        NOT NULL
)
USES AuditInfo                           -- @inline mixin
HAS measurements Measurement[]           -- @owned collection
HAS status CodeableConcept               -- @shared FK
RELATIONSHIPS { subject -> Patient REQUIRED ONE; }
DOC "Clinical observation with measurements";

-- Context-level defaults for shared dimensions
CONTEXT Clinical {
    DEFAULT observation_status CodeableConcept = (
        system  = 'http://hl7.org/fhir/observation-status',
        code    = 'preliminary',
        display = 'Preliminary'
    );
}

PostgreSQL Generation

Composite Types (@owned)

-- Composite type for array parameters
CREATE TYPE clinical.measurement_t AS (
    value       NUMERIC,
    unit        TEXT,
    effective_at TIMESTAMPTZ,
    quality     TEXT
);

-- Child table
CREATE TABLE clinical.observation_measurement (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    observation_id  UUID NOT NULL REFERENCES clinical.observation(id) ON DELETE CASCADE,
    ordinal         INTEGER NOT NULL,
    value           NUMERIC NOT NULL,
    unit            TEXT NOT NULL,
    effective_at    TIMESTAMPTZ DEFAULT NOW(),
    quality         TEXT DEFAULT 'preliminary',
    UNIQUE (observation_id, ordinal)
);

-- Index for FK lookups
CREATE INDEX idx_observation_measurement_observation_id 
    ON clinical.observation_measurement(observation_id);

Dimension Table (@shared)

-- Dimension table with natural key
CREATE TABLE basedata.codeable_concept (
    id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    system      TEXT NOT NULL,
    code        TEXT NOT NULL,
    display     TEXT,
    version     TEXT DEFAULT 'latest',
    UNIQUE (system, code, version)
);

-- FK in referencing table
ALTER TABLE clinical.observation 
    ADD COLUMN status_id UUID REFERENCES basedata.codeable_concept(id);

Flattened Columns (@inline)

-- AuditInfo fields added directly to table
CREATE TABLE clinical.observation (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    subject_id      UUID NOT NULL,
    status_id       UUID REFERENCES basedata.codeable_concept(id),
    -- @inline AuditInfo fields
    created_at      TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_by      UUID,
    modified_at     TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    modified_by     UUID,
    FOREIGN KEY (subject_id) REFERENCES clinical.patient(id)
);

-- Trigger for computed/immutable fields
CREATE OR REPLACE FUNCTION clinical.observation_audit_trigger()
RETURNS TRIGGER AS $$
BEGIN
    -- COMPUTED modified_at ON UPDATE
    IF TG_OP = 'UPDATE' THEN
        NEW.modified_at := NOW();
        -- IMMUTABLE fields
        NEW.created_at := OLD.created_at;
        NEW.created_by := OLD.created_by;
    END IF;
    RETURN NEW;
END;
$$ LANGUAGE plpgsql;

CREATE TRIGGER trg_observation_audit
    BEFORE UPDATE ON clinical.observation
    FOR EACH ROW EXECUTE FUNCTION clinical.observation_audit_trigger();

API Generation

Insert with Batch Helper (@owned)

CREATE OR REPLACE FUNCTION clinical_api.observation_insert(
    p_subject_id    UUID,
    p_status_id     UUID DEFAULT NULL,
    p_measurements  clinical.measurement_t[] DEFAULT NULL,
    p_created_by    UUID DEFAULT NULL
) RETURNS clinical.observation AS $$
DECLARE
    v_result clinical.observation%ROWTYPE;
    v_measurement clinical.measurement_t;
    v_ordinal INTEGER := 0;
BEGIN
    -- Insert main entity
    INSERT INTO clinical.observation (subject_id, status_id, created_by, modified_by)
    VALUES (p_subject_id, p_status_id, p_created_by, p_created_by)
    RETURNING * INTO v_result;
    
    -- Batch insert @owned measurements (1 parameter regardless of array size)
    IF p_measurements IS NOT NULL THEN
        FOREACH v_measurement IN ARRAY p_measurements LOOP
            v_ordinal := v_ordinal + 1;
            INSERT INTO clinical.observation_measurement 
                (observation_id, ordinal, value, unit, effective_at, quality)
            VALUES (
                v_result.id, 
                v_ordinal,
                v_measurement.value, 
                v_measurement.unit,
                COALESCE(v_measurement.effective_at, NOW()),
                COALESCE(v_measurement.quality, 'preliminary')
            );
        END LOOP;
    END IF;
    
    RETURN v_result;
END;
$$ LANGUAGE plpgsql SECURITY INVOKER;

Get-or-Create for @shared

CREATE OR REPLACE FUNCTION basedata_api.codeable_concept_upsert(
    p_system    TEXT,
    p_code      TEXT,
    p_display   TEXT DEFAULT NULL,
    p_version   TEXT DEFAULT 'latest'
) RETURNS basedata.codeable_concept AS $$
DECLARE
    v_result basedata.codeable_concept%ROWTYPE;
BEGIN
    -- Try to find existing
    SELECT * INTO v_result 
    FROM basedata.codeable_concept 
    WHERE system = p_system AND code = p_code AND version = p_version;
    
    IF FOUND THEN
        RETURN v_result;
    END IF;
    
    -- Insert new
    INSERT INTO basedata.codeable_concept (system, code, display, version)
    VALUES (p_system, p_code, p_display, p_version)
    ON CONFLICT (system, code, version) DO UPDATE SET display = EXCLUDED.display
    RETURNING * INTO v_result;
    
    RETURN v_result;
END;
$$ LANGUAGE plpgsql SECURITY INVOKER;

gRPC Generation

// Composite type becomes nested message
message Measurement {
    string value = 1;      // DECIMAL as string for precision
    string unit = 2;
    google.protobuf.Timestamp effective_at = 3;
    optional string quality = 4;  // Has DEFAULT
}

// Insert request with repeated for @owned
message CreateObservationRequest {
    string subject_id = 1;
    optional string status_id = 2;           // @shared FK
    repeated Measurement measurements = 3;    // @owned collection
    optional string created_by = 4;          // @inline with DEFAULT
    // modified_at omitted - COMPUTED
    // created_at omitted - COMPUTED
}

// Response includes computed fields
message Observation {
    string id = 1;
    string subject_id = 2;
    optional string status_id = 3;
    repeated Measurement measurements = 4;
    google.protobuf.Timestamp created_at = 5;
    optional string created_by = 6;
    google.protobuf.Timestamp modified_at = 7;
    optional string modified_by = 8;
}

Consequences

Positive

  • Type safety: All values are properly typed PostgreSQL columns
  • Query efficiency: Standard SQL joins, no JSON extraction functions
  • Indexable: All fields can be indexed normally
  • Migration friendly: Schema changes are standard ALTER TABLE
  • Parameter safe: Composite type arrays count as 1 parameter regardless of size
  • Audit ready: @inline AuditInfo with COMPUTED/IMMUTABLE provides standard audit trails

Negative

  • More tables: @owned creates additional child tables
  • Join complexity: @shared requires joins to fetch display values
  • Migration effort: Existing entities need updates to use USES/HAS syntax

Risks

  • Performance: Large @owned collections need ordinal index tuning
  • Circular references: @shared dimensions must be loaded before referencing entities

AI Agent Guidance

When This Decision Applies

  • Implementing Value Objects in DDD models
  • Any field that represents a concept without identity (measurements, codes, audit info)
  • When you need collections of values attached to an entity

When This Decision Does NOT Apply

  • Entities with their own lifecycle (use ENTITY, not VALUE_OBJECT)
  • Simple scalar fields (just use the field directly)
  • Reference data with its own CRUD operations (use ENTITY in Basedata)

How It's Enforced

  • Parser validates @inline/@owned/@shared annotations
  • Parser validates COMPUTED/IMMUTABLE field declarations
  • Generator emits appropriate PostgreSQL structures based on mode
  • Tests verify composite type array handling
