Specification WHERE Clause Hardening

Specification WHERE Clause Hardening

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-12
  • Tags: security, specification, sql-injection, grammar, defense-in-depth

Context and Problem Statement

The SPECIFICATION ... WHERE clause in the DSL accepts arbitrary SQL token sequences and interpolates them without validation or sanitization into:

  1. PostgreSQL VIEWs: CREATE VIEW ... AS SELECT * FROM ... WHERE {raw} (sqitch.rkt)
  2. PostgreSQL FUNCTIONs: CREATE FUNCTION ... $$ SELECT ... WHERE {raw} $$ (sqitch.rkt)
  3. C++ string literals: ntx.exec("SELECT * FROM ... WHERE {raw}") (unified.rkt)

The current PEG grammar rule SqlExpression accepts any token sequence (including DDL/DML keywords, semicolons, and subqueries) until hitting a terminating keyword. The parser reconstructs the token stream by concatenation with zero semantic validation.

Threat Model

.ddd files are author-controlled artifacts (like source code), not user input. However:

  1. AI-generated .ddd files — the brownfield extraction pipeline (VISION.md) will use LLMs to generate .ddd files from legacy systems. An LLM producing malicious SQL in a SPECIFICATION WHERE clause would pass through undetected.
  2. Malicious contributor — a DROP TABLE or data exfiltration query embedded in a specification WHERE clause survives generation and is deployed via sqitch deploy with full database privileges.
  3. No defense-in-depth — currently zero layers of protection between .ddd authoring and SQL execution.

Additional Bug: Parameterized Spec $N Placeholder Mismatch

Parameterized specifications have a name-to-positional mismatch in C++ generation:

  • PostgreSQL function: Parameters named p_threshold_m — WHERE clause references by name ✅
  • C++ pqxx: Parameters passed via pqxx::params{threshold_m} — but WHERE clause string contains bare name threshold instead of positional $1 ❌

This causes runtime query failures.

Decision Drivers

  • Defense-in-depth: Multiple independent layers of protection
  • Grammar-first evolution: Per ADR 20260211, grammar changes precede parser changes
  • Backward compatibility: 100+ existing SPECIFICATION declarations in geosciences must continue parsing
  • Fail-fast principle: Reject malicious input at the earliest possible point (parse time)
  • Parameterized query safety: Runtime user values must never be string-interpolated into SQL

Considered Options

  1. Grammar whitelist only — restrict SpecWhereExpression PEG rule to safe SQL subset
  2. Parser deny-list only — scan tokens after parsing for dangerous keywords
  3. Grammar whitelist + parser deny-list + pqxx::params enforcement (chosen)
  4. SQL AST parsing — full SQL parser for WHERE clauses (rejected — excessive complexity)

Decision Outcome

Chosen option: "Grammar whitelist + parser deny-list + pqxx::params enforcement" — three independent defense layers at build time and runtime.

Layer 1: PEG Grammar Whitelist (Build Time — Parse)

Replace the opaque SqlExpression rule for specification WHERE clauses with a restrictive SpecWhereExpression:

Allowed constructs:

  • Column references (identifiers, qualified table.column)
  • Comparison operators: =, !=, <>, <, >, <=, >=
  • Logical operators: AND, OR, NOT
  • Predicates: IN (...), BETWEEN ... AND ..., IS NULL, IS NOT NULL, LIKE, ILIKE
  • Literals: string ('...'), number, boolean (TRUE/FALSE/NULL)
  • Temporal functions: CURRENT_DATE, CURRENT_TIMESTAMP, NOW()
  • Interval expressions: INTERVAL '...'
  • Type casts: ::type (PostgreSQL cast syntax)
  • Arithmetic: +, -, *, /
  • Parenthesized subexpressions
  • Parameter references (bare names matching specification parameters)

Rejected constructs (parse error):

  • DDL: DROP, CREATE, ALTER
  • DML: INSERT, UPDATE, DELETE, TRUNCATE
  • Access control: GRANT, REVOKE, EXECUTE
  • Data movement: COPY
  • Statement terminators: ;
  • Subqueries: SELECT
  • CTEs: WITH
  • Comments: --, /*

Layer 2: Parser Deny-List (Build Time — Validation)

After parse-sql-expression, scan the token stream for deny-listed keywords. This is defense-in-depth — if a construct somehow passes the grammar, the parser catches it.

Fail fast with:

specification-parse-error: specification 'EvilSpec' contains forbidden SQL keyword 'DROP' in WHERE clause

Layer 3: pqxx::params for Runtime Values (Runtime — C++)

Parameterized specifications pass user-supplied values exclusively through pqxx::params (parameterized queries). The WHERE clause string uses positional $1, $2, ... placeholders — never string interpolation of user values.

Fix the C++ generator to rewrite parameter references in sql-where:

  • Scan specification-parameters list
  • For each parameter name found in sql-where, replace with $N (1-indexed)
  • Generate pqxx::params{param1, param2, ...} matching the positional order

Additional runtime defense layers for parameterized specs:

  • gRPC: protobuf type enforcement (typed request messages)
  • Qt UI: type-specific widgets with validation (QSpinBox for INT, QDateEdit for DATE, etc.)

Positive Consequences

  • Three independent defense layers — grammar, parser, and runtime
  • Malicious .ddd content rejected at parse time with clear error messages
  • AI-generated .ddd files validated before any SQL generation
  • Parameterized specs use proper prepared-statement semantics
  • All 100+ existing specs continue parsing (whitelist covers all patterns in use)

Negative Consequences

  • Grammar becomes more complex (restrictive rule vs. permissive rule)
  • Some advanced SQL patterns in WHERE clauses may be rejected (e.g., window functions, CTEs) — acceptable trade-off; these don't belong in specification predicates
  • $N rewriting adds generator complexity

AI Agent Guidance

When This Decision Applies

  • Modifying the SPECIFICATION ... WHERE parsing pipeline
  • Generating SQL from specification WHERE clauses (any generator)
  • Adding new SQL constructs to the specification whitelist
  • Implementing parameterized specification C++ generation
  • Generating gRPC handlers that pass user input to specification queries
  • Processing AI-generated .ddd files through the generation pipeline

When This Decision Does NOT Apply

  • Non-specification SQL generation (e.g., CHECK constraints, migration DDL)
  • SQL in COMMAND REQUIRES clauses (separate validation concern)
  • PostgreSQL DOMAIN definitions (type-registry controlled)

Enforcement

  • Grammar: SpecWhereExpression PEG rule rejects forbidden tokens at parse time
  • Parser: validate-specification-where! function scans for deny-listed keywords
  • C++ Generator: parameterized spec WHERE clauses must use $N positional placeholders with pqxx::params
  • Test: test-specification-generators.rkt includes injection rejection test cases
