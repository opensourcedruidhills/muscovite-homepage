ADR: Documentation Consolidation for AI Efficiency

ADR: Documentation Consolidation for AI Efficiency

  • Status: Accepted
  • Date: 2026-02-01
  • Deciders: Project maintainers
  • Tags: documentation, ai-agents, developer-experience

Context and Problem Statement

Muscovite has accumulated 253 markdown files totaling 70,346 lines of documentation. Every AI agent conversation re-parses the same large instruction files:

  • .github/copilot-instructions.md (1,887 lines)
  • AGENTS.md (610 lines)
  • doc/VISION.md (678 lines)
  • .github/USER_PROFILE.md (varies)

This creates ~4,000+ lines of context load before any actual work begins, consuming token budget that could be used for code analysis and generation.

Additionally, completed phase documentation and session summaries accumulate in the repository root, creating visual clutter and potential confusion about current project state.

Decision Drivers

  1. Token efficiency: AI agents have limited context windows; every line of instruction consumes budget
  2. Relevance: Not all documentation is relevant to every task
  3. Maintenance burden: Duplicate content across multiple docs diverges over time
  4. Developer experience: Clean repository structure aids navigation
  5. AI-first project: Muscovite is designed for AI-assisted development; docs should optimize for this

Considered Options

Option 1: Status Quo

Keep all documentation as-is. Accept the context overhead.

Pros: No migration effort Cons: Ongoing inefficiency, growing clutter

Option 2: Aggressive Deletion

Delete all "completed" documentation and session summaries.

Pros: Maximum reduction Cons: Loses historical context, may need to re-discover decisions

Option 3: Tiered Archive Strategy (Selected)

Move completed/historical docs to attic/, keep active docs lean, use ADRs as single source of truth.

Pros: Preserves history, reduces active context, improves navigation Cons: Requires ongoing discipline to archive completed work

Decision

Implement a tiered documentation strategy:

Tier 1: Always-Loaded (Minimize)

Files attached to every AI conversation. Target: <1,000 lines combined.

File Purpose Target Lines
.github/copilot-instructions.md Core rules, critical patterns <800
AGENTS.md Quick-start for AI agents <400

Tier 2: On-Demand Reference

Loaded when relevant to the task. Stored in doc/ or .github/copilot-details/.

Category Files When Loaded
Build workflow doc/DEVELOPER-GUIDE.md Building, CMake issues
DSL syntax doc/DSL-REFERENCE.md Parser/generator work
Qt patterns doc/qt-frontend-architecture-plan.md Qt client work
Migration ADRs 1-7 Schema evolution

Tier 3: Archived (Never Auto-Loaded)

Historical context preserved but not in active paths.

Content Location
Completed phase docs attic/phase-docs/
Session summaries attic/session-summaries/
Superseded plans attic/planning/

Implementation Rules

  1. Archive completed phases immediately: When PHASE*_COMPLETE.md is created, move related planning docs to attic
  2. Session summaries are ephemeral: Move to attic after 7 days or next session
  3. ADRs are canonical: Don't duplicate architectural decisions in other docs; reference ADRs
  4. Prompt files stay focused: Each .prompt.md file targets one specific task

Consequences

Positive

  • Reduced default context load by ~40-50%
  • Cleaner repository root (fewer PHASE*.md, SESSION_*.md files)
  • Single source of truth for decisions (ADRs)
  • Faster AI agent startup time

Negative

  • Requires discipline to archive completed work
  • Historical context requires explicit navigation to attic/
  • May need to update references in other docs

Neutral

  • Git history preserves all changes regardless of file location

AI Agent Guidance

When archiving documentation:

  1. Check if any other docs reference the file being archived
  2. Update references to point to new location or remove if obsolete
  3. Commit with message: chore(docs): Archive completed phase X documentation

When creating new phase docs:

  1. Use naming convention: PHASE{N}_{FEATURE}_PLAN.md for planning
  2. Create PHASE{N}_{FEATURE}_COMPLETE.md only when phase is done
  3. Archive both within 7 days of completion

When to load additional context:

  • Ask user: "Should I review the Qt architecture docs for this task?"
  • Or proactively: "I see this involves schema evolution. Loading ADRs 1-7 for context."
