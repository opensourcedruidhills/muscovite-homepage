Use TIMESTAMPTZ for All Date/Time Values

Use TIMESTAMPTZ for All Date/Time Values

  • Status: accepted
  • Deciders: Database architects
  • Date: 2024-06-01
  • Tags: postgresql, schema, types, datetime

Context and Problem Statement

When storing date/time values in PostgreSQL, we must choose between TIMESTAMP (without time zone) and TIMESTAMPTZ (with time zone). Additionally, we must decide whether to ever store dates as TEXT strings.

Decision Drivers

  • Geosciences data spans global locations with different time zones
  • Correct time zone handling prevents data corruption
  • TEXT storage loses semantic meaning and query capabilities
  • PostgreSQL's TIMESTAMPTZ stores UTC internally, displays in session TZ
  • Avoid ambiguity in temporal data

Considered Options

  • TIMESTAMPTZ for all datetime values
  • TIMESTAMP without time zone
  • TEXT with ISO 8601 format strings

Decision Outcome

Chosen option: "TIMESTAMPTZ for all datetime values", because it ensures unambiguous storage in UTC while allowing proper time zone conversion on retrieval, which is essential for a global geosciences dataset.

CRITICAL RULE: Never store dates or timestamps as TEXT.

Positive Consequences

  • All timestamps are unambiguous (stored as UTC)
  • Time zone conversions handled automatically by PostgreSQL
  • Full temporal query support (BETWEEN, date arithmetic)
  • Consistent behavior across different client time zones
  • Proper indexing and partitioning support

Negative Consequences

  • Requires awareness of session time zone settings
  • Slightly more complex display logic in clients
  • Historical dates before time zones existed may seem anachronistic

Pros and Cons of the Options

TIMESTAMPTZ (with time zone)

  • Good, because stores UTC, converts on display
  • Good, because unambiguous across time zones
  • Good, because supports all temporal operations
  • Good, because proper indexing
  • Bad, because requires session TZ awareness
  • Bad, because historical dates get modern TZ semantics

TIMESTAMP (without time zone)

  • Good, because simpler mental model
  • Good, because no TZ conversion overhead
  • Bad, because ambiguous if data crosses time zones
  • Bad, because can cause silent data corruption
  • Bad, because clients must manually handle TZ

TEXT with ISO 8601

  • Good, because human-readable storage
  • Good, because no TZ conversion surprises
  • Bad, because loses semantic type information
  • Bad, because no temporal query support
  • Bad, because requires parsing for any operation
  • Bad, because larger storage footprint

AI Agent Guidance

When This Decision Applies

  • ALL datetime/timestamp fields in DSL entities
  • DSL types: DateTime, Timestamp, TIMESTAMPTZ → PostgreSQL TIMESTAMPTZ
  • Event timestamps, audit fields (created_at, updated_at)
  • Field observation dates, sample collection times

When This Decision Does NOT Apply

  • Pure DATE values (use PostgreSQL DATE type)
  • Pure TIME values (use PostgreSQL TIME or TIMETZ)
  • Durations/intervals (use PostgreSQL INTERVAL)

Enforcement

  • Racket generators map DateTime to TIMESTAMPTZ
  • NEVER generate TEXT for temporal values
  • API functions accept/return proper temporal types
