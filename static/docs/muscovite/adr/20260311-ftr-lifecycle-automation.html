ADR: FTR Lifecycle Automation — Script-Driven Status Transitions

ADR: FTR Lifecycle Automation — Script-Driven Status Transitions

  • Status: accepted
  • Date: 2026-03-11
  • Deciders: Johannes Lochmann
  • Amends: ADR 20260306 (Multi-Agent Session Protocol)

Context

The governance model requires FTR files to track status (planned → in-progress → complete) and ITERATION.md to reflect per-FTR status in the scope table. Currently:

Transition Automated? Mechanism
→ planned ✅ Yes plan-feature.sh stamps Status + Updated
planned → in-progress ❌ No Social contract — agents forget
in-progress → complete ❌ No Social contract — agents forget
complete → released ⚠️ Partial release.sh stamps ITERATION.md only, not FTR files

Observed failure mode: After v1.61.0 and v1.62.0 iterations, agents consistently forgot to update FTR Status fields and ITERATION.md scope table rows. Files marked planned were shipped as complete — discovered only during release validation. The validate-release.sh script catches the discrepancy but cannot distinguish "forgot to mark" from "not yet done."

The root cause: agents coordinate via NATS/JSONL (ADR 20260309) but never propagate state back to the governance files. Two sources of truth diverge silently.

Decision

Create start-feature.sh and complete-feature.sh — mandatory lifecycle scripts that agents call when claiming and finishing FTRs.

workflow/scripts/start-feature.sh <ftr-number>

Called when an agent begins work on an FTR. Updates:

  1. FTR file **Status**: planned → **Status**: in-progress
  2. FTR file **Updated**: → current date
  3. ITERATION.md scope table: row for FTR-N status column → in-progress

Preconditions: FTR file must exist, Status must be planned.

workflow/scripts/complete-feature.sh <ftr-number>

Called when an agent finishes work (typically after PR created or merged). Updates:

  1. FTR file **Status**: in-progress → **Status**: complete
  2. FTR file **Updated**: → current date
  3. FTR file acceptance criteria: reports unchecked count (does NOT auto-tick)
  4. ITERATION.md scope table: row for FTR-N status column → complete

Acceptance criteria are never auto-ticked — agents must verify and tick each criterion individually. validate-release.sh catches unticked criteria before release.

Preconditions: FTR file must exist, Status must be in-progress.

Integration Points

Tool Hook
agents/nats-agent.sh agent_claim Calls start-feature.sh after successful KV claim
agents/nats-agent.sh agent_release Optionally calls complete-feature.sh if --complete flag
workflow/scripts/agent-pr.sh Calls complete-feature.sh before creating PR
Manual agent workflow Agent calls scripts directly if not using nats-agent.sh

Acceptance Criteria — Deliberate Verification Only

complete-feature.sh does NOT auto-tick acceptance criteria checkboxes. It reports the count of unchecked criteria as a warning. Each criterion must be individually verified and ticked by the agent or human who confirmed it was met.

Rationale: Bulk sed 's/- \[ \]/- [x]/' across all criteria provides zero assurance that the criteria were actually verified. It defeats the purpose of having acceptance criteria and bypasses validate-release.sh as a safety net. Lesson learned from v1.62.0 where bulk-ticking was used. Enforced from v1.63.0 onward.

FORBIDDEN: Bulk-ticking acceptance criteria via sed, script, or any automated means.

Idempotency

Both scripts are idempotent:

  • start-feature.sh on an already in-progress FTR → no-op with warning
  • complete-feature.sh on an already complete FTR → no-op with warning

Error Handling

  • FTR file not found → hard error (exit 1)
  • Wrong status for transition → warning + exit 0 (idempotent)
  • ITERATION.md not found → warning (FTR file still updated)
  • ITERATION.md has no matching row → warning (FTR file still updated)

Consequences

Positive

  • Single source of truth: FTR files and ITERATION.md always reflect actual state
  • No social contract needed: Scripts enforce transitions mechanically
  • Audit trail: Updated timestamp changes on every transition
  • Release validation simplified: validate-release.sh can trust Status fields
  • Agent integration: NATS claim/release automatically triggers file updates

Negative

  • Commit discipline: Scripts modify tracked files — agent must commit the changes
  • Merge conflicts: Two agents updating ITERATION.md simultaneously could conflict (mitigated: different rows, and develop-push lock serializes merges)

Neutral

  • plan-feature.sh remains the entry point for the planned state (unchanged)
  • release.sh still stamps released on ITERATION.md (could be extended to stamp FTR files too)
