Use API-Only Testing with DDD-Driven FK Seeding

Use API-Only Testing with DDD-Driven FK Seeding

  • Status: accepted
  • Deciders: Test architects, Database architects
  • Date: 2024-09-01
  • Tags: testing, pgtap, api, ddd

Context and Problem Statement

pgTAP tests need to insert test data to verify CRUD operations. How should tests create data and handle foreign key dependencies between entities?

Decision Drivers

  • Tests should exercise the same code path as production
  • FK dependencies must be satisfied in correct order
  • Test data setup should be maintainable
  • DDD model is the single source of truth for relationships
  • Avoid brittle SQL that breaks when schema changes

Considered Options

  • API-only testing with DDD-driven seeding
  • Direct SQL INSERT with manual FK ordering
  • Test fixtures loaded before test runs
  • Mock/stub database layer

Decision Outcome

Chosen option: "API-only testing with DDD-driven seeding", because it ensures tests exercise the actual API layer and automatically derives FK dependencies from the DDD model.

ABSOLUTE REQUIREMENT: All pgTAP tests MUST use ONLY stored procedures for data manipulation. NO direct SQL INSERT/UPDATE/DELETE/SELECT statements allowed.

Positive Consequences

  • Tests verify actual production code paths
  • FK seeding order derived from DDD relationships
  • Tests auto-update when relationships change
  • Single source of truth (DDD model)
  • Deterministic test execution order

Negative Consequences

  • Tests are slightly slower (API overhead)
  • Requires complete API implementation before testing
  • More complex test generation logic

API Contract

All {schema}_api.{entity}_insert() functions MUST return the complete table row via RETURNING * INTO v_result. This ensures:

  • Returned PKs can be used for subsequent FK references
  • All columns (including defaults) are available
  • Tests never need to query the table directly

Prohibited Patterns

  • ❌ Direct INSERT INTO table VALUES (...)
  • ❌ Direct UPDATE table SET ...
  • ❌ Direct DELETE FROM table WHERE ...
  • ❌ Direct SELECT * FROM table to extract IDs
  • ❌ Manual gen_random_uuid() calls
  • ❌ Inferring FKs from SQL migration files
  • ❌ Test-specific SQL fixtures

Seeding Algorithm

  1. Extract relationships from DDD metadata (not SQL parsing)
  2. Compute transitive closure (all ancestors via DFS + memoization)
  3. Topologically sort entities (parents always before children)
  4. Generate deterministic API insert calls (in dependency order)

AI Agent Guidance

When This Decision Applies

  • ALL pgTAP test generation
  • ALL test data setup in generated tests
  • Manual pgTAP tests for custom scenarios
  • Integration tests that touch the database

When This Decision Does NOT Apply

  • C++ unit tests (use mocks)
  • Performance benchmarks (may use direct SQL)
  • Data migration scripts (separate from tests)

Enforcement

  • Generated tests only call *_api.*_insert/select/update/delete()
  • pgTAP test generator uses topological sort
  • CI runs full pgTAP suite to verify
