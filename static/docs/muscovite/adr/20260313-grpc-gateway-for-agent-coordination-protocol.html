ADR: gRPC Gateway for Agent Coordination Protocol

ADR: gRPC Gateway for Agent Coordination Protocol

  • Date: 2026-03-13
  • Status: superseded by 20260318-nats-agent-coordination-abandoned
  • Related: ADR 20260313-muscovite-bridge-human-in-the-loop-command-center, ADR 20260312-push-based-coordinator-with-durable-nats, ADR 20260309-nats-jetstream-agent-coordination

Context

Current architecture

The Muscovite multi-agent coordinator communicates via NATS JetStream. Agents connect directly to a NATS server (port 4223), publish/subscribe on coord.* subjects, and exchange JSON-encoded messages. Authentication is token-based (UUID per session), with role enforcement in the coordinator process.

This architecture served the dogfood run (v1.65.0) well for development. However, it has structural limitations for commercial deployment:

Limitation Impact
NATS directly exposed Clients need NATS client libraries; attack surface is the full NATS protocol
JSON payloads Human-readable on the wire; no schema enforcement; easy to reverse-engineer
Token auth only No certificate-based identity; no revocation infrastructure
Python coordinator Source code ships to customers; trivial to inspect and replicate
Single-port lock-in Corporate firewalls often block non-HTTP ports

Commercial opportunity

The v1.65.0 dogfood validated the coordination model: TLA+-verified state machine, role-based governance, JetStream crash recovery, push-based assignment. This has standalone value as a product — formally verified AI agent orchestration.

Two deployment tiers are envisioned:

Tier Persistence Recovery Target
Basic Ephemeral NATS (no JetStream) None — restart loses state OSS / evaluation
Pro Durable JetStream on persistent storage Full crash recovery via event replay Production / SaaS

Decision

Introduce a gRPC gateway as the public API surface

flowchart LR
    CLIENTS["Agents / QML Cockpit\n(C++, Python, Go)"] <-->|"gRPC\nmTLS+JWT\nHTTP/2 port 443"| GW["muscovite-agent-gateway\n(C++ binary)"]
    GW <-->|"NATS\nlocalhost only"| JS["JetStream"]

NATS becomes an implementation detail — never exposed to clients.

Protocol design

A single coordinator.proto defines the entire public API:

syntax = "proto3";
package muscovite.agent.v1;

// Unary RPCs — command/response
service Coordinator {
  rpc RegisterWorker(RegisterWorkerRequest) returns (RegisterWorkerResponse);
  rpc UnregisterWorker(UnregisterWorkerRequest) returns (UnregisterWorkerResponse);
  rpc ReportProgress(ReportProgressRequest) returns (ReportProgressResponse);
  rpc SignalPRReady(SignalPRReadyRequest) returns (SignalPRReadyResponse);
  rpc ReviewFTR(ReviewFTRRequest) returns (ReviewFTRResponse);
  rpc ReleaseFTR(ReleaseFTRRequest) returns (ReleaseFTRResponse);
  rpc EscalateToHuman(EscalateRequest) returns (EscalateResponse);
  rpc HumanDecide(HumanDecisionRequest) returns (HumanDecisionResponse);
  rpc GetStatus(StatusRequest) returns (StatusResponse);
}

// Bidirectional streaming — real-time event feed
service EventStream {
  rpc Subscribe(SubscribeRequest) returns (stream CoordinatorEvent);
  rpc StreamCommands(stream AgentCommand) returns (stream CommandResult);
}

Key design choices:

  1. Unary RPCs for commands — register, progress, review, release. Maps 1:1 to current NATS coord.cmd.* subjects. Gateway translates gRPC→NATS internally.

  2. Bidirectional streaming for events — replaces NATS subscribe. The QML cockpit and monitoring tools use EventStream.Subscribe() for real-time updates. Gateway subscribes to coord.event.> internally and fans out to gRPC streams.

  3. StreamCommands for interactive agents — a single long-lived connection where agents send commands and receive assignments, progress updates, and escalation requests.

Security layers

Layer Technology Purpose
Transport mTLS (gRPC native) Wire encryption, mutual certificate verification
Identity JWT with NKey signing Per-agent identity, role claims, expiry, revocation
Payload Protobuf binary Typed schema, not human-readable on wire
Audit JetStream event stream Append-only, sequenced, tamper-evident event log
Firewall HTTP/2 on port 443 Passes through any corporate proxy/firewall

mTLS provides transport encryption and client authentication in one step. Each agent holds a client certificate signed by the coordinator's CA. Certificate revocation handles agent decommissioning without coordinator restart.

JWT tokens carry role claims (worker, leader, qa, human) that the gateway validates before forwarding to NATS. This moves role enforcement from Python coordinator code to the gateway's interceptor chain.

Implementation language: C++

The gateway and coordinator are implemented in C++ for:

  1. IP protection — binary distribution, no source exposure
  2. Performance — gRPC C++ is the reference implementation; lowest latency
  3. Unified stack — same language as the Muscovite framework, QML cockpit, and generated code
  4. License enforcement — binary can embed license-key validation

The Python coordinator (agents/coordinator.py) remains as a reference implementation and development tool. The C++ gateway is the production artifact.

NATS encapsulation

The gateway owns the NATS connection. Externally, clients see only gRPC. Internally:

gRPC call NATS subject Direction
RegisterWorker coord.cmd.worker.register request/reply
ReportProgress coord.cmd.worker.progress request/reply
ReviewFTR coord.cmd.qa.review request/reply
Subscribe coord.event.> JetStream subscribe → gRPC stream
StreamCommands coord.assign.<worker> + coord.cmd.* bidirectional

This means NATS can be replaced (Kafka, Redis Streams, in-process queue) without changing the client API. The gRPC contract is the stable interface; NATS is the swappable backend.

SaaS tier implementation

Tier NATS backend Gateway config
Basic Ephemeral NATS (no JetStream) persistence: false, event replay disabled
Pro Durable JetStream, persistent storage persistence: true, /var/lib/nats/ storage
Enterprise On-prem, customer-managed NATS cluster Gateway binary + config, customer infra

The gateway reads a tier field from the license and configures NATS accordingly.

Deployment topology — orchestration, not execution

A critical architectural constraint: the gateway orchestrates agents but does not execute them. Agents run on customer infrastructure with customer-owned AI credentials.

Why agents must run on customer infra:

  1. Code access — agents read/write the customer's codebase. We never host or see their source.
  2. AI billing — customers use their own API keys (OpenAI, Anthropic, Ollama, etc.). We are not an AI reseller; we orchestrate, not execute.
  3. Secrets isolation — GitHub tokens, DB credentials, API keys stay on customer machines.
  4. Liability boundary — agent actions (commits, PR merges, DB migrations) execute in the customer's environment under their credentials.
  5. GDPR/compliance — we only process event metadata (FTR states, timings, role transitions). No PII, no source code, no AI prompts/completions cross the boundary.

Proto registration carries capabilities, not credentials:

message RegisterWorkerRequest {
  string worker_id = 1;
  repeated string capabilities = 2;  // ["cpp", "racket", "sql", "qt"]
  string client_version = 3;         // SDK version for compatibility
  // NO ai_api_key — stays on customer side, never transmitted
}

The coordinator assigns FTRs based on declared capabilities (e.g., FTR-417 needs ["qt", "qml"], FTR-413 needs ["sql", "racket"]). The agent's AI provider and credentials are entirely opaque to the gateway.

QML cockpit integration

The QML cockpit becomes a gRPC client:

QML UI ◄──signals──► C++ ViewModel ◄──gRPC stream──► Gateway ◄──NATS──► JetStream

The ViewModel subscribes via EventStream.Subscribe() and emits Qt signals on each CoordinatorEvent. The cockpit never touches NATS directly — same binary works against local dev gateway or remote SaaS endpoint.

This makes the cockpit a portable sales demo: point it at any gateway URL and it visualizes the coordination in real time.

Consequences

Positive

  • Single public API — gRPC contract is versioned, documented, and SDK-generatable
  • Security-by-default — mTLS + JWT + binary protobuf eliminates the casual inspection problem
  • Firewall-friendly — HTTP/2 port 443 works everywhere
  • Backend-agnostic — NATS is an implementation detail, swappable without client changes
  • SDK for free — protoc generates C++, Python, Go, TypeScript, Java, C# clients
  • QML cockpit portability — works against local or remote gateway identically
  • IP protection — C++ binary distribution, no source code exposure
  • Tiered SaaS — same binary, different config, different price

Negative

  • Development overhead — C++ gateway is more code than the Python coordinator
  • Two implementations — Python reference + C++ production must stay in sync until cutover
  • Certificate management — mTLS requires a CA, cert issuance, revocation infrastructure
  • Migration path — existing Python-based dogfood workflows need a transition period

Migration strategy

  1. v1.65.1–v1.65.3: Stabilize Python coordinator (dogfood fixes, human role, TLA+ update)
  2. v1.66.x: Define coordinator.proto, implement gateway skeleton in C++
  3. v1.67.x: Feature parity — gateway handles all coordinator commands
  4. v1.68.x: QML cockpit connected via gRPC streams
  5. v1.69.x: Python coordinator deprecated; C++ gateway is production default

Risks

  • Premature optimization: the Python coordinator works. Don't port too early — stabilize first.
  • Proto schema evolution: once clients depend on coordinator.proto, breaking changes are expensive. Use proto3 field numbering discipline from day one.
  • mTLS complexity: consider starting with JWT-only auth and adding mTLS in a later tier.
