Name All Primary Keys "id"

Name All Primary Keys "id"

  • Status: accepted
  • Deciders: Database architects, API designers
  • Date: 2024-06-15
  • Tags: postgresql, schema, naming, api

Context and Problem Statement

Entity primary keys need consistent naming across the entire stack: DSL definitions, PostgreSQL schemas, PL/pgSQL APIs, C++ entities, gRPC protos, and Qt models. What naming convention should we use for primary keys?

Decision Drivers

  • Consistency across all generated code targets
  • Simplifies generic repository patterns
  • Reduces cognitive load when working across layers
  • FK naming becomes predictable (<table>_id)
  • Aligns with common ORM conventions

Considered Options

  • Always name primary key id
  • Use <entity>_id pattern (e.g., sample_id, mineral_id)
  • Allow custom names per entity

Decision Outcome

Chosen option: "Always name primary key id", because it provides maximum consistency across all targets and simplifies generic code patterns.

Strict Convention:

  • Entity declaration: ENTITY Sample IDENTIFIED_BY id
  • Field declaration: id UUID NOT NULL
  • PostgreSQL column: id UUID PRIMARY KEY DEFAULT gen_random_uuid()
  • API parameters: p_id UUID for select/update/delete operations
  • FK columns: <referenced_table>_id (e.g., sample_id references sample.id)

Positive Consequences

  • Completely predictable naming across all layers
  • Generic repository methods work without configuration
  • FK naming is unambiguous: locality_id → locality.id
  • Simplified code generation templates
  • Easy to teach and remember

Negative Consequences

  • Less descriptive in complex queries (must use table alias)
  • Self-referential FKs need special handling (parent_id)
  • TPT inheritance uses same id for shared PK

Pros and Cons of the Options

Always "id"

  • Good, because maximum consistency
  • Good, because simple generic patterns
  • Good, because FK naming is predictable
  • Good, because matches Rails/Django conventions
  • Bad, because ambiguous in JOIN queries
  • Bad, because self-referential FKs need exceptions

Entity-prefixed id (e.g., sample_id)

  • Good, because self-documenting in queries
  • Good, because no ambiguity in JOINs
  • Bad, because FK naming becomes sample_sample_id or inconsistent
  • Bad, because harder to write generic code
  • Bad, because longer names everywhere

Custom names per entity

  • Good, because maximum flexibility
  • Good, because can match legacy schemas
  • Bad, because no predictability
  • Bad, because generators need per-entity configuration
  • Bad, because cognitive overhead

AI Agent Guidance

When This Decision Applies

  • ALL entity primary key definitions in DSL
  • ALL generated PostgreSQL CREATE TABLE statements
  • ALL generated PL/pgSQL API function signatures
  • ALL generated C++ entity classes
  • ALL generated gRPC proto messages
  • ALL generated Qt model classes

When This Decision Does NOT Apply

  • Junction tables for M:N relationships (use composite PK)
  • Legacy schema integrations (may need mapping layer)
  • Third-party API integrations (map at boundary)

Enforcement

  • DSL parser validates IDENTIFIED_BY id pattern
  • Generators assume PK is named id
  • pgTAP tests verify column existence
