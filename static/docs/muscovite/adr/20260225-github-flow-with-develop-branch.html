GitHub Flow with Develop Branch for Multi-Agent Isolation

GitHub Flow with Develop Branch for Multi-Agent Isolation

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-25

Context and Problem Statement

Muscovite uses a single-branch model (main only) with direct commits. This works for solo development but breaks down when multiple AI agents operate concurrently:

  1. No isolation: Agent A performing a release blocks Agent B from committing feature work.
  2. No audit trail: Direct commits to main leave no review artifact showing what changed and why.
  3. No integration buffer: A broken commit on main immediately affects the release branch.

The project is hosted on GitHub (private repo, no paid org). Server-side branch protection is not available. All enforcement must be convention-based via agent instructions and scripts.

Decision Drivers

  • Multiple AI agents need concurrent, isolated workspaces
  • GitHub CLI (gh) is installed and authenticated — PR-based workflow is available
  • No server-side CI or branch protection (private repo, no paid org)
  • Release ceremony (release.sh) must remain the sole release path
  • Feature-complete PRs are the natural work unit (not per-commit PRs)
  • Gitea is a local Docker container for brownfield extraction, NOT the project host

Considered Options

Option 1: Full Git Flow (master/develop/release/hotfix)

Full Git Flow with release/* and hotfix/* branches, back-merges, CI gates.

Option 2: GitHub Flow + develop (chosen)

Add develop as the integration branch. Feature branches merge to develop via gh pr create. release.sh merges develop → main at release time. No release/* or hotfix/* branches.

Option 3: Keep single-branch main

Status quo. Accept agent conflicts as the cost of simplicity.

Decision Outcome

Chosen option: Option 2 — GitHub Flow + develop, because it provides agent isolation and audit trails with minimal ceremony. Full Git Flow (Option 1) adds release/* and hotfix/* branches that are unnecessary for a solo developer with AI agents. The single-branch model (Option 3) is what prompted this change.

Branch Model

Branch Source Merges To Purpose
main — — Release-only. Tagged releases. Never committed to directly.
develop main main (via release.sh) Integration branch. Default branch on GitHub.
feature/ftr-NNN-* develop develop (via gh pr create) Feature work by one agent.

Feature Workflow

1. git checkout develop && git pull
2. git checkout -b feature/ftr-NNN-short-desc
3. (do work, commit)
4. workflow/scripts/agent-pr.sh   # runs quality gates, creates PR
5. gh pr merge --merge             # merge to develop after review

Release Workflow

1. cd $REPO_ROOT && git checkout develop
2. workflow/scripts/release.sh vX.Y.Z
   - Merges develop → main (fast-forward-only)
   - Runs quality gates on main
   - Tags, pushes
   - Back-merges main → develop (keep in sync)

Quality Gate Enforcement

Since no server-side CI exists, quality gates are enforced by scripts:

  • agent-pr.sh: Runs parser tests, generator tests, integration tests, C++ build, C++ tests. Results documented in PR body. Non-zero exit on any failure = no PR created.
  • release.sh: Same quality gates plus effort estimation, changelog generation, muscovite-cpp sync.

The PR body IS the evidence artifact. No CI bot posts status — the creating agent documents test results.

Positive Consequences

  • Agents work in isolation on feature branches
  • develop is the integration point — broken features don't reach main
  • PRs create an audit trail per feature (visible on GitHub)
  • gh pr list shows what's in flight — agents check before starting work
  • main is always release-ready (only receives merged develop)
  • No new infrastructure required — gh CLI is the only addition

Negative Consequences

  • Convention-enforced only — no server-side block on direct push to main
  • Agents must remember to use agent-pr.sh, not git push develop directly
  • Fast-forward-only merge to main requires develop to be a superset of main

Key Rules

  • FORBIDDEN: Direct commits to main (except by release.sh)
  • FORBIDDEN: Direct push to develop (must go through PR via agent-pr.sh)
  • FORBIDDEN: Force-push to develop or main
  • REQUIRED: gh pr list --base develop before starting new feature work
  • REQUIRED: All quality gates pass before PR creation (enforced by agent-pr.sh)
  • REQUIRED: release.sh is the sole path from develop to main

AI Agent Guidance

When This Applies

  • Starting any new feature work (branch from develop, not main)
  • Completing a feature (use agent-pr.sh, not git push)
  • Preparing a release (run release.sh on develop)
  • Checking what other agents are working on (gh pr list)

When This Does NOT Apply

  • Experimental/throwaway branches (any name, delete after)
  • Hotfixes: create hotfix/description from main, merge to both main and develop via release.sh --hotfix (future extension)

Key Files

  • workflow/scripts/agent-pr.sh — Feature PR creation with quality gates
  • workflow/scripts/release.sh — Release ceremony (develop → main)
  • AGENTS.md — Branching model documentation
  • CLAUDE.md — Branching rules for Claude agents
  • .github/copilot-instructions.md — Branching rules for Copilot agents
