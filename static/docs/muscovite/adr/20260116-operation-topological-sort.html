Operation Topological Sort for Migration Execution

Operation Topological Sort for Migration Execution

  • Status: accepted
  • Deciders: Architecture Team
  • Date: 2026-01-16
  • Tags: migration, dependency, ordering

Context and Problem Statement

Migration operations are currently appended in detection order, not dependency order. This causes execution failures when operations have implicit dependencies:

Example problem:

1. change-type: clinical_status → UUID (destructive)
2. rename-field: clinical_status → clinical_status_id (safe)

Execution should be:

  1. rename-field first (column still exists with old name)
  2. Then change-type (column now has new name)

Similarly, foreign key constraints must be added after both source and target tables exist.

Decision Drivers

  • PostgreSQL DDL has strict ordering requirements
  • FK constraints require referenced tables to exist
  • Column operations must respect current column names
  • Automated migrations should "just work" without manual reordering

Considered Options

  1. Manual ordering by developers
  2. Priority-based sorting (assign numeric priorities)
  3. Topological sort based on operation dependencies

Decision Outcome

Chosen option: Topological sort based on operation dependencies

Operations form a directed acyclic graph (DAG) based on their dependencies. Topological sort produces an execution order that respects all dependencies.

Dependency Rules

Operation Type Must Come After
add-entity add-context (same context)
remove-entity remove all FKs pointing to it
rename-entity - (no dependencies)
add-field add-entity (same entity)
remove-field remove relationships using it
rename-field - (no dependencies)
change-type rename-field (same field)
change-nullable change-type (same field)
add-relationship add-entity (both source and target)
add-check add-entity (same entity)
remove-check - (no dependencies)

Execution Order Categories

  1. Schema operations (add-context, remove-context)
  2. Table creation (add-entity)
  3. Column additions (add-field with nullable or default)
  4. Column renames (rename-field)
  5. Type changes (change-type)
  6. Constraint additions (add-check, change-nullable to NOT NULL)
  7. Relationship additions (add-relationship / FK constraints)
  8. Removals (in reverse dependency order)

Consequences

Positive

  • Migrations execute correctly without manual intervention
  • Complex multi-operation migrations are handled automatically
  • Clear, deterministic execution order

Negative

  • Additional computation at migration generation time
  • Cycle detection needed (cycles indicate model errors)

AI Agent Guidance

  • Always sort operations before generating SQL
  • If topological sort fails with a cycle, report the cycle to the user
  • Group operations by schema after sorting for cleaner SQL output
