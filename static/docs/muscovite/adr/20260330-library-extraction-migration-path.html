20260330-library-extraction-migration-path

Library Extraction as Migration Path to Muscovite 2.0

  • Status: accepted
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-30
  • Tags: muscomp, migration, libraries, build-system, 2.0
  • Supersedes: none

Technical Story: Define the phased migration from the monolithic muscomp_lib static library to the Muscovite 2.0 architecture of standalone executables and InProcessDynamic sidecar plugins.

Context and Problem Statement

muscomp v1.78.0 delivered the Racket-to-C++ translation as a single muscomp_lib static library: 195 source files, 12K lines, all 10 generators compiled into one target. This was the correct approach for rapid delivery, but it creates:

  • All-or-nothing recompilation on header changes
  • No build parallelism across generators (single library target)
  • No path to runtime-loadable plugins
  • No independent testing per generator

The 2.0 architecture (ADR 20260330-muscovite-2) requires standalone executables and InProcessDynamic plugins. This ADR defines how to get there incrementally without breaking the working system.

Decision Drivers

  • v1.78.1 quality gates (golden diff, parser test coverage) must complete first — regression safety net is prerequisite for refactoring
  • Each step must produce a working, tested muscomp — no big-bang migration
  • Shared libraries extracted in early phases must be the final InProcessDynamic sidecar artifacts — no throwaway work
  • Build time improvements should be realized incrementally, not deferred to 2.0

Decision Outcome

A four-phase migration with each phase independently shippable.

Phase 1: Foundation Libraries

Extract the core infrastructure that all plugins will depend on:

Library Sources Depends On
libmuscomp-ir src/ir/, src/naming/ nlohmann_json, Boost
libmuscomp-tmpl src/tmpl/ (none)
libmuscomp-gen-core src/generators/shared/, src/manifest/ libmuscomp-ir, libmuscomp-tmpl

Acceptance criteria: muscomp builds and links against these three shared libraries instead of compiling their sources into the monolith. Golden diff tests pass. Build time for a single-generator change is measurably reduced.

What changes: CMakeLists.txt splits into per-library targets. Public headers separated from private implementation. Per-library test targets.

What doesn't change: muscomp binary still works identically. No runtime behavior change.

Phase 2: Generator Libraries

Extract each generator into its own shared library:

Library Sources Size
libmuscomp-gen-postgres src/generators/postgres/ 9 files
libmuscomp-gen-grpc src/generators/grpc/ 6 files
libmuscomp-gen-qt src/generators/qt/ 13 files
libmuscomp-gen-cmake src/generators/cmake/ ~3 files
libmuscomp-gen-docs src/generators/docs/ ~3 files
libmuscomp-gen-search src/generators/search/ ~3 files
libmuscomp-gen-messaging src/generators/messaging/ ~3 files
libmuscomp-gen-tlaplus src/generators/tlaplus/ ~3 files
libmuscomp-gen-cpp src/generators/cpp/ 72 files (largest)
libmuscomp-gen-migration src/migration/ 2 files

Order: Start with small generators (docs, tlaplus, search, messaging) to prove the pattern. Extract gen-cpp last — it's the largest and has internal sub-dependencies (entity/repository/service/saga/plugins/acl) that need an internal dependency audit.

Acceptance criteria: Each generator builds as an independent shared library. muscomp links against all of them. Golden diff passes after each extraction. CMake can build generators in parallel.

Phase 3: Parser and Validator Libraries

Extract parser and validators as independent libraries:

Library Sources Size
libmuscomp-parser src/parser/ 5 files
libmuscomp-validator-xref src/validation/cross_ref_validator.* 1 file
libmuscomp-validator-relationship src/validation/relationship_validator.* 1 file
libmuscomp-validator-type src/validation/type_validator.* 1 file
libmuscomp-validator-naming src/validation/naming_validator.* 1 file
libmuscomp-validator-statemachine src/validation/state_machine_validator.* 1 file
libmuscomp-validator-eventsourcing src/validation/event_sourcing_validator.* 1 file

Acceptance criteria: Parser and each validator build as independent libraries. Per-validator test targets. Golden diff passes.

Phase 4: InProcessDynamic + Standalone Executables

Convert shared libraries to runtime-loadable plugins and standalone executables:

  1. Add C factory functions to each generator library (per ADR 20260330-inprocess-dynamic)
  2. Add AbstractGenerator, AbstractParser, AbstractValidator abstract interfaces to libmuscomp-gen-core
  3. Replace generator_registry.cpp dispatch with dlopen loader
  4. Create standalone executable wrappers per generator (read model.ir, call generator)
  5. Implement muscomp parse and muscomp generate subcommands
  6. Implement protobuf IR serialization/deserialization

Acceptance criteria: muscomp parse produces model.ir. muscomp generate --target=postgres loads the InProcessDynamic plugin OR spawns the standalone executable (configurable). Golden diff passes for both paths. Third-party generator example (Python, ~50 lines) reads model.ir and produces output.

Positive Consequences

  • Each phase is independently shippable and valuable (build time improves incrementally)
  • Shared libraries extracted in Phases 1-3 become the final InProcessDynamic sidecar artifacts in Phase 4 — zero throwaway work
  • Golden diff regression safety net ensures no output changes at any step
  • Small generators prove the pattern before tackling gen-cpp (72 files)
  • Build parallelism increases with each phase

Negative Consequences

  • Migration effort spread across multiple iterations
  • CMakeLists.txt complexity increases during transition (multiple library targets)
  • Test infrastructure must be restructured per-library
  • gen-cpp extraction (Phase 2, final step) requires internal dependency audit of its 10 sub-modules

AI Agent Guidance

When This Decision Applies

  • When planning muscomp build system changes
  • When extracting a generator from the monolithic muscomp_lib
  • When creating CMake targets for individual generator libraries
  • When adding test infrastructure for a specific generator

When This Decision Does NOT Apply

  • v1.78.1 work (golden diff, parser coverage) — complete these first
  • Racket dslgen changes — the Racket codebase is unaffected
  • Layer 1/2 changes — this ADR concerns Layer 0 build structure only
  • New generator development — new generators should be created as standalone libraries from the start, following the Phase 2 pattern

Enforcement

  • Golden diff tests must pass after each extraction step
  • Each library must have its own CMake target and test target
  • No generator library may depend on another generator library (only on foundation libraries)
  • Phase ordering is mandatory: foundation → generators → parser/validators → 2.0

Key Files

  • toolchain/muscomp/CMakeLists.txt — primary file that changes in every phase
  • toolchain/muscomp/src/generators/generator_registry.cpp — replaced in Phase 4
  • toolchain/muscomp/src/ir/types.hpp — becomes part of libmuscomp-ir
  • toolchain/muscomp/src/tmpl/ — becomes libmuscomp-tmpl
  • toolchain/muscomp/tests/ — restructured into per-library test targets
