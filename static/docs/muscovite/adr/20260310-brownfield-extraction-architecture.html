ADR: Brownfield Extraction Architecture — Multi-Source Triangulation with Direct .ddd Output

ADR: Brownfield Extraction Architecture — Multi-Source Triangulation with Direct .ddd Output

Date: 2026-03-10 Status: Accepted (revised 2026-03-10) Deciders: Johannes Lochmann Supersedes: Original version (neutral IR + ir-to-ddd converter)

Context

Muscovite's brownfield modeling feature (MAPPING blocks, ADR 20260102) allows generated code to coexist with legacy databases. FTR-349 (Gitea extraction) revealed that real-world brownfield targets have:

  • No database-level foreign keys — relationships are enforced in application code (XORM, GORM, Django ORM, JPA, etc.)
  • ORM struct tags that encode constraints, indexes, nullability, and column mappings
  • Application-level documentation in source code comments that should become DSL DOC blocks
  • Integer enums, Unix timestamps, VARCHAR limits that need type transformation metadata
  • Service layers that encode commands, events, state transitions, and workflows
  • UI templates and API specs that reveal which fields and actions are domain-visible

Extracting this information requires parsing multiple source layers, not just the database schema or ORM models. Different brownfield targets use different languages and ORMs:

Target Language ORM Example
Gitea Go XORM 107 tables, 229 _id columns, 0 FKs
FHIR R4 N/A StructureDefinition JSON Schema specification, not application code
Future: Django apps Python Django ORM models.ForeignKey() declarations
Future: Rails apps Ruby ActiveRecord belongs_to/has_many declarations
Future: Spring apps Java JPA/Hibernate @ManyToOne/@JoinColumn annotations

Building one monolithic extractor for all languages is unmaintainable. Building per-system extractors (one for Gitea, one for another Go app) wastes effort on duplicate Go parsing logic.

Why Not a Neutral Intermediate Representation?

The original design proposed a three-layer pipeline: Extractor → Brownfield IR (JSON) → ir-to-ddd (Racket) → .ddd. Analysis revealed this adds complexity without value:

  1. The .ddd format IS the intermediate representation — it's human-readable, well-defined, stable (grammar-first mandate, ADR 20260211), and the only format with downstream consumers.
  2. A neutral IR has no standalone consumer — it exists only to be transformed into .ddd. That's pure overhead.
  3. Behavioral constructs (commands, events, sagas, state machines) would require extending the IR to essentially mirror the .ddd AST — reinventing a format that already exists.
  4. .ddd is already the lingua franca — every extractor (Go, Python, Java) speaks .ddd as output, every downstream tool (parser, generators) consumes .ddd as input.

Decision

Direct .ddd Output with Multi-Source Triangulation

flowchart TD
    DB["Database (schema)\ncolumns, types, indexes, sequences"]
    SC["Source Code (AST + types)\nservice funcs, event types\ninterfaces, state consts"]
    UI["User Interface (templates + API)\nform fields, buttons/actions\ndropdowns, navigation"]

    DB --> CS["Confidence scoring"]
    SC --> CS
    UI --> CS
    CS --> DDD[".ddd files (direct)"]
    DDD --> HC["Human curation"]

Language-Specific Extractors (each its own repository)

Repository Language Parses Uses
muscovite-go-extractor Go Go structs, services, templates, Swagger go/ast, go/types, go/parser stdlib
(future) muscovite-python-extractor Python Django/SQLAlchemy models + views ast stdlib
(future) muscovite-java-extractor Java JPA/Hibernate + Spring controllers javaparser

Each extractor:

  • Is a standalone CLI tool in its own repository
  • Integrated into Muscovite as a git subtree (same pattern as muscovite-cpp at framework/)
  • Takes a configuration file pointing at source paths, ORM type, context classification
  • Outputs .ddd files directly — conformance validated by dual PEG checking
  • Analyzes multiple source layers, not just ORM models

Five-Pass Extraction Pipeline

Each extractor performs five sequential passes:

Pass Source Layer Extracts Confidence
1. Schema ORM model structs (models/) Entities, fields, types, relationships, enums, indexes High
2. Behavioral Service layer (services/) Commands, state transitions, workflows, event emissions Medium–High
3. UI/API Templates + Swagger/OpenAPI (templates/, routers/) Domain-visible fields, user commands, enum values, navigation Medium
4. Triangulation Cross-reference all passes Merge evidence, compute confidence scores per construct —
5. Emit — Write .ddd files with confidence annotations as comments —

Three-Tier Behavioral Extraction

Behavioral constructs are extractable at three confidence tiers:

Tier 1 — High confidence (signatures, always extractable):

Muscovite Construct Source Pattern Example
COMMAND func Create*(ctx, opts), func Update*(...), func Delete*(...) services/issue/NewIssue()
DOMAIN EVENT Notification types, *Event structs, publisher calls modules/notification/NotifyNewIssue()
EXTENSION_POINT type FooInterface interface { ... } modules/markup/Renderer
STATE_MACHINE states const ( Open = iota; Closed; Merged ) Issue/PR status enums
SPECIFICATION type SearchOptions struct { ... } with filter fields models/issues/IssuesOptions

Tier 2 — Medium confidence (structure + heuristics):

Construct Source Pattern Strategy
STATE_MACHINE transitions func ChangeStatus(), func (i *Issue) Close() Method receivers that mutate status fields
POLICY Event handler registrations, func On*Event() Notification listeners with conditional logic
SAGA candidates Multi-step service functions with rollback/cleanup 3+ sequential service calls with error handling

Tier 3 — Low confidence (deep analysis, needs human review):

Construct Source Pattern What we CAN extract
DECIDE logic Conditional branches in command handlers Branch structure and conditions, not domain semantics
EFFECT catalog External calls (email, webhook, queue publish) Cross-package function calls
Compensations defer blocks, error-path cleanup Paired create/delete in error handlers

Where automated extraction fails, plugin interfaces are still extractable — Go interface definitions provide the contract name, method signatures, and parameters, mapping directly to EXTENSION_POINT constructs.

Confidence Scoring and .ddd Annotations

Each extracted construct receives a confidence score from 0.0 to 1.0 based on evidence triangulation:

confidence = 0.0
  + 0.2 if in database schema / ORM model
  + 0.2 if in Go struct with typed fields
  + 0.2 if rendered in template / present in API spec
  + 0.2 if present in service layer
  + 0.2 if validated or constrained

The score determines how the construct is emitted:

Confidence Emission
≥ 0.8 Full .ddd construct
0.4–0.8 .ddd construct with -- REVIEW: confidence <score> — <evidence> comment
< 0.4 -- TODO: confidence <score> — <description> comment block only

Example output:

COMMAND CreateIssue {            -- CONFIDENCE: 1.0 (schema + service + template + API)
  -- Source: services/issue/issue.go:142
  title TEXT NOT_NULL             -- form field: required input
  content TEXT                    -- form field: optional textarea
  milestone_id REFERENCE Milestone  -- dropdown in template
}

-- REVIEW: confidence 0.6 — state transitions inferred from ChangeIssueStatus()
STATE_MACHINE IssueLifecycle FOR Issue {
  STATES { Open, Closed }
  INITIAL Open
  TERMINAL Closed
  TRANSITION Open -> Closed ON CloseIssue
  TRANSITION Closed -> Open ON ReopenIssue
}

-- TODO: confidence 0.2 — multi-step pattern in services/pull/merge.go
-- Potential SAGA: MergePullRequest (3 sequential service calls with error rollback)
-- Steps: UpdatePullRequest → MergeGitBranch → ClosePullRequest
-- Compensation detected: defer branch cleanup on merge failure

Dual Conformance Validation (Belt and Suspenders)

All extractor output must be syntactically valid .ddd:

1. Go-side PEG validator (fast, self-contained):

  • The Muscovite PEG grammar (dslgen/grammar/muscovite.peg) is compiled into a Go parser using a PEG generator library (e.g., pointlander/peg or mna/pigeon)
  • The go-extractor validates its own output before writing — instant feedback during extraction
  • Grammar-first mandate (ADR 20260211) ensures both parsers accept the same language

2. Racket parse-back gate (authoritative, CI-enforced):

  • After extraction, racket parser.rkt --parse-only *.ddd validates all output files
  • The Racket parser is the conformance oracle — if it parses, the output is legal
  • This runs as a CI gate in the go-extractor repository and during agent-pr.sh

Subtree Integration Pattern

# Initial setup (once)
git remote add go-extractor git@github.com:opensourcedruidhills/muscovite-go-extractor.git
git subtree add --prefix=tools/go-extractor go-extractor main

# Sync during release (same as muscovite-cpp)
git subtree push --prefix=tools/go-extractor go-extractor main

Per-Project Configuration

Each brownfield sample project provides a configuration file:

# examples/gitea-brownfield/extractor-config.yaml
source:
  repository: vendor/gitea           # submodule path
  version: v1.21.0
  language: go
  orm: xorm
  model_paths:
    - models/
  service_paths:
    - services/
  template_paths:
    - templates/
  router_paths:
    - routers/
  notification_paths:
    - modules/notification/

contexts:
  Repository:
    packages:
      - code.gitea.io/gitea/models/repo
    doc: "Git repository management and metadata"
  IssueTracking:
    packages:
      - code.gitea.io/gitea/models/issues
    doc: "Issue lifecycle, labels, milestones, and time tracking"

type_overrides:
  timeutil.TimeStamp: DateTime       # Unix epoch → TIMESTAMPTZ
  api.GitServiceType: Integer        # Enum integer

extraction:
  behavioral: true                   # Enable service/template analysis
  min_confidence: 0.2                # Emit constructs above this threshold
  swagger_path: templates/swagger/v1_json.tmpl  # Optional: Swagger spec for API triangulation

FHIR Extractor Alignment

FTR-345 (FHIR extraction tool) parses FHIR R4 StructureDefinition JSON — not application source code. It outputs .ddd files directly (same principle: no intermediate IR). The FHIR extractor stays as a Python script in tools/ (not its own repo — FHIR is a schema format, not a programming language).

Consequences

Positive

  • One Go extractor serves all Go projects — Gitea, Gogs, Drone, any XORM/GORM application
  • Extractors are independently testable — Go tests parse Go, no Racket dependency
  • Direct .ddd output — no intermediate format to design, maintain, or version
  • .ddd is the lingua franca — every extractor outputs it, every tool consumes it
  • Multi-source triangulation — dramatically richer extraction than ORM-only analysis
  • Behavioral extraction — commands, events, state machines, interfaces — not just tables and columns
  • Confidence annotations — human curators see exactly what needs review and why
  • Dual conformance validation — malformed .ddd is caught at extraction time AND in CI
  • Subtree pattern is proven — same governance as muscovite-cpp (release.sh pushes subtree)

Negative

  • Extractors must know .ddd syntax — mitigated by the Go-side PEG validator compiled from the same grammar, and by the stability of .ddd syntax (grammar-first mandate)
  • New repository — go-extractor needs its own CI, releases, documentation
  • Multi-source analysis is complex — service layer, templates, and API add extraction scope

Neutral

  • FHIR extractor also outputs .ddd directly (consistent with architecture)
  • Future extractors follow an established pattern — reduced design effort
  • doc/GO-PATTERNS.md governs Go coding style for the extractor codebase
