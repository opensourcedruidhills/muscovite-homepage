ADR: TLA+ Formal Verification of Coordinator State Machine

ADR: TLA+ Formal Verification of Coordinator State Machine

  • Date: 2026-03-11
  • Status: accepted
  • Deciders: Johannes Lochmann
  • Category: verification, coordinator, tooling

Context

The multi-agent coordinator (agents/coordinator.py, ADR 20260311-leader-worker) manages FTR lifecycle, resource locking, sync barriers, and merge queuing for N concurrent AI agent workers. The state machine has 6 states, 9 transitions, and interacts with 3 subsystems (ConcurrencyGovernor, resource locks, MergeQueue) under concurrent access.

Three classes of bugs have already manifested during development:

  1. Missing transitions — REVIEWING → PR_READY was absent from the transition graph, discovered only when implementing merge failure recovery. A formal spec would have caught this as a deadlock state.
  2. Resource leak on worker death — compute semaphore slots were not released when a worker unregistered, causing permanent capacity reduction. The invariant "sum of held slots ≤ limit" held, but the liveness property "every queued request is eventually granted" was violated.
  3. Idempotency gaps — the command deduplication infrastructure existed but was not wired into handlers, allowing duplicate state transitions on NATS timeout retries.

These bugs are invisible to unit tests (which test single-actor paths) and emerge only under specific interleavings of concurrent workers — the domain of TLA+ model checking.

Decision

Write a hand-authored TLA+ specification of the coordinator state machine and verify it with TLC as a development-time check. Unlike the saga TLA+ generator (ADR 20260311-tlaplus- saga-formal-verification), this is a one-off specification — not generated from IR.

Scope

The spec models the coordinator's core state machine, NOT the NATS transport layer. Message delivery semantics (at-least-once, ordering) are modeled as nondeterministic choices in the environment process.

State Variables Modeled

Variable Type Maps to

Properties Verified

Property Type Formula
No two workers hold same FTR Safety (invariant) ∀ w1, w2 ∈ WORKERS: w1 ≠ w2 ⇒ worker_ftr[w1] = None ∨ worker_ftr[w1] ≠ worker_ftr[w2]
Worker holds at most 1 FTR Safety (invariant) Structural (single worker_ftr field)
Exclusive tags mutually exclusive Safety (invariant) `∀ t ∈ EXCLUSIVE_TAGS:
Semaphore limit respected Safety (invariant) ∀ r ∈ RESOURCES: Cardinality(semaphore_held[r]) ≤ limit[r]
Single merge writer Safety (invariant) merge_current ≠ None ⇒ merge_queue_not_dequeuing
Sync barrier blocks PR_READY Safety (invariant) ∀ f: ftr_state[f] = "pr_ready" ⇒ worker_synced(ftr_worker[f])
Every assigned FTR eventually merges or returns to pending Liveness ∀ f: ftr_state[f] = "assigned" ↝ ftr_state[f] ∈ {"merged", "pending"}
No permanent semaphore leak Liveness ∀ f, r: f ∈ semaphore_held[r] ↝ f ∉ semaphore_held[r]
Merge queue fairness Liveness ∀ f ∈ merge_queue: f ↝ merge_current = f
Idempotent command replay Safety ∀ cmd: cmd ∈ seen_commands ⇒ replay(cmd) = noop

Model Parameters (TLC)

CONSTANTS
  FTR_IDS = {"FTR-1", "FTR-2", "FTR-3"}
  WORKERS = {"w1", "w2"}
  EXCLUSIVE_TAGS = {"framework", "ir"}
  RESOURCES = {"build", "ctest"}
  RESOURCE_LIMITS = [build |-> 2, ctest |-> 1]

Small model (3 FTRs, 2 workers) keeps state space tractable while exercising all concurrency interleavings. Symmetry reduction on WORKERS and FTR_IDS.

NOT a Release Gate

Unlike saga TLA+ specs (which are generated and gated), this coordinator spec is a development-time verification tool. It is run manually when the state machine changes, not on every release. Rationale: the coordinator is infrastructure code that changes infrequently, and TLC runs take minutes for meaningful state spaces.

Consequences

  • State machine bugs that manifest only under concurrent interleaving are caught before they reach production
  • The spec serves as executable documentation of the coordinator's invariants
  • Adding new transitions or states requires updating the spec first — "spec-first" discipline
  • TLC counterexample traces provide concrete reproduction scenarios for bugs

Alternatives Considered

  1. Property-based testing (Hypothesis) — good for single-actor properties, poor at exploring concurrent interleavings across multiple actors
  2. Generated spec from Python — rejected; the coordinator is a one-off, and translating Python async patterns to TLA+ automatically adds complexity without value
  3. No formal verification — rejected; we already found 3 bugs that formal verification would have caught
