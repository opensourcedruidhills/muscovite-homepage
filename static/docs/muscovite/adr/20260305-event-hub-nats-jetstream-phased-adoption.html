ADR: Event Hub Integration — Phased NATS JetStream Adoption

ADR: Event Hub Integration — Phased NATS JetStream Adoption

  • Date: 2026-03-05
  • Status: accepted
  • Tags: infrastructure, messaging, events, nats

Context

Muscovite generates domain events unconditionally (ADR 20260124) and persists them in {schema}.domain_events tables within the same transaction as the write operation. Sagas declare event-driven semantics (TRIGGERED_BY, ON SUCCESS, ON FAILURE) but execute via DB-backed state machine polling. Context maps (MAP {} blocks) declare cross-context relationships with VIA grpc — synchronous point-to-point calls.

This works for single-instance desktop deployments today. However, three architectural pressures are building:

  1. Cross-context integration is synchronous — DDD literature (Vernon, Millett) consistently argues that bounded context integration should be event-driven, not synchronous RPC.
  2. Saga runtime doesn't match DSL semantics — sagas declare event triggers but execute via polling. An event hub would make the runtime match the declared model.
  3. Multi-instance scaling is blocked — competing consumers need coordination that DB polling cannot provide cleanly.

Decision Drivers

  • The domain_events table is already a Transactional Outbox (Microservices Patterns, Richardson)
  • The IR already models events fully: domain-event, sourced-event, saga structs carry event names, payloads, correlation keys, and step sequences
  • NATS JetStream is lightweight (single binary, no ZooKeeper), persistent, and Apache 2.0 licensed (compatible with AGPL-3.0 dual license)
  • Generator architecture means adding a target is ~1K–5K lines of Racket
  • Premature adoption adds operational burden without current consumers

Considered Options

  1. Kafka — industry standard, but operationally heavy (ZooKeeper/KRaft, JVM, multi-broker)
  2. RabbitMQ — mature, but Erlang runtime, less natural for event streaming patterns
  3. NATS JetStream — lightweight, single binary, built-in persistence, natural subject hierarchy ← chosen
  4. Redis Streams — capable but conflates cache + messaging roles; SSPL licensing concerns
  5. Defer entirely — miss the opportunity to align generated saga runtime with DSL semantics

Decision Outcome

Chosen option: NATS JetStream, phased adoption in three stages.

NATS subject hierarchy maps naturally to Muscovite's bounded context structure: muscovite.{project}.{context}.{aggregate}.{event} — generated directly from the IR.

Phase 1 — Near-Term: NATS Subject Schema Generation (v1.53.0)

Generate NATS subject/topic definitions from context maps + event declarations. No runtime yet — just the messaging schema as a generated artifact.

Deliverables:

  • NATS subject hierarchy generated from IR (generators/messaging/nats-subjects.rkt)
  • Subject namespace document (AsciiDoc) for each project
  • Docker Compose integration for NATS JetStream (alongside existing PostgreSQL + OpenSearch)
  • Stream/consumer configuration generated as NATS CLI setup scripts
  • Framework muscovite::messaging namespace with Publisher and Subscriber interfaces

Estimated effort: ~2K–3K lines of Racket generator + ~500 lines C++ framework interfaces

Phase 2 — Mid-Term: Outbox Relay + Event-Driven Sagas (v1.53.0)

Generate an outbox relay service that reads domain_events and publishes to NATS JetStream. Saga runtime switches from DB polling to NATS subscription.

Deliverables:

  • Outbox relay: generated C++ service polling domain_events, publishing to NATS
  • Saga runtime: generated NATS subscribers replacing DB polling for step transitions
  • VIA events transport option in context maps (alongside existing VIA grpc)
  • Event deduplication via domain_events.id as NATS message ID (exactly-once via JetStream)
  • Health check + metrics for relay lag (events pending publish)

Estimated effort: ~3K–4K lines of Racket generator + ~1K lines C++ framework

Phase 3 — Deferred: Full Event-Driven Context Integration

Replace synchronous gRPC cross-context calls with event-driven integration. Context map relationships using VIA events generate NATS pub/sub instead of gRPC client stubs.

This phase is explicitly deferred. Triggers to revisit:

  • Multi-instance deployment requiring competing consumers
  • Performance SLA requiring async cross-context communication
  • Context map complexity exceeding what synchronous RPC can maintain (> 10 cross-context edges)

Why deferral is safe:

  • Phase 1 + 2 deliver the infrastructure (NATS running, subjects defined, outbox active)
  • The VIA events DSL syntax can be added without breaking existing VIA grpc consumers
  • The IR already has context-map entries — adding a transport mode is a field addition, not a redesign

Positive Consequences

  • Generated saga runtime matches DSL event-driven semantics
  • domain_events table becomes the authoritative event source (outbox pattern)
  • NATS subject hierarchy provides a machine-readable messaging contract
  • Path to multi-instance scaling without architectural rework
  • Docker Compose already manages PostgreSQL + OpenSearch — adding NATS is incremental

Negative Consequences

  • NATS JetStream becomes a runtime dependency for event-driven features
  • Outbox relay introduces eventual consistency (events are not published synchronously)
  • Relay lag monitoring is needed to detect publishing delays
  • Phase 3 deferral means cross-context communication remains synchronous until revisited
