Technical Story: Muscovite roadmap v3.0 planning — when and how to adopt C++26.

Context and Problem Statement

C++26 has been finalized (ISO meeting, March 2026). GCC trunk already has reflection and contracts merged. Muscovite currently targets C++23 (gnu23) with GCC on Linux.

C++26 brings four major features relevant to Muscovite: reflection, contracts, hardened standard library, and std::execution. Adopting C++26 changes the minimum compiler requirement for both the framework and all consumer repos (fhir-health, geosciences, hello-world), making this a major version boundary.

The question is: when to adopt, and how to manage the transition for consumers.

Decision Drivers

  • GCC 16 (expected mid-2026) will ship with C++26 support including reflection and contracts
  • Hardened stdlib and no-UB-uninit are free safety wins requiring only a recompile
  • Consumer repos pin MUSCOVITE_CPP_VERSION via FetchContent — they must also upgrade compilers
  • Healthcare and geosciences domains benefit from demonstrable safety improvements
  • C++23 → C++26 is not backward compatible (contracts syntax, reflection API are new)
  • Clang C++26 support may lag GCC by 6-12 months (relevant for multi-compiler consumers)

Considered Options

  1. GCC 16 day-one adoption — Flip to gnu26 as soon as GCC 16 releases
  2. GCC 16 + 6 months — Wait for one point release (GCC 16.1) to stabilize
  3. Parallel support period — Maintain both gnu23 and gnu26 Conan profiles temporarily
  4. Wait for Clang parity — Adopt only when both GCC and Clang have full C++26

Decision Outcome

Chosen option: TBD — to be decided when GCC 16 release date is confirmed.

Leaning toward option 2 (GCC 16 + stabilization) with option 3 as transition mechanism:

  • v2.x remains gnu23 (no disruption to current iteration pipeline)
  • v3.0.0 targets gnu26, released after GCC 16.1 proves stable
  • A 1-release overlap period where muscovite-cpp v2.last supports both profiles
  • Consumer repos get a migration guide with the v3.0.0 release

Implementation Checklist (when decision is accepted)

Positive Consequences

  • Access to contracts, reflection, hardened stdlib, std::execution
  • Safety improvements for healthcare/geosciences domains (demonstrable, measurable)
  • Positions Muscovite at the leading edge of C++ practice

Negative Consequences

  • Forces consumer repos to upgrade compilers (breaking change)
  • Clang users may be temporarily blocked
  • CI infrastructure must upgrade to GCC 16+
  • Two-profile transition period adds maintenance overhead

Compiler Research Findings (2026-04-04)

GCC trunk has reflection (-freflection), contracts, and P2795 (no-UB uninit reads) merged and awaiting GCC 16 release (expected April–May 2026, following yearly cadence). Clang has none of these features — no timeline visible. std::execution is not implemented in either compiler.

Hardened stdlib (_GLIBCXX_ASSERTIONS) is available since GCC 12 and works with C++23. Adopted early as FTR-687 in v2.9.0 — no need to wait for C++26.

Feature GCC trunk GCC 16 Clang Action
Reflection ✅ -freflection ✅ ❌ v3.2+
Contracts ✅ -std=c++2c ✅ ❌ v3.0
Hardened stdlib ✅ NOW ✅ ✅ v2.9.0 (FTR-687)
Uninit UB removal ✅ ✅ ❌ v3.0 (free)
std::execution ❌ ❌ ❌ Defer

AI Agent Guidance

When This Decision Applies

  • When modifying Conan profiles, CMake minimum version, or compiler.cppstd settings
  • When planning iterations that depend on C++26 features
  • When updating consumer repo documentation or migration guides

When This Decision Does NOT Apply

  • v2.x development — remains on C++23 until this ADR is accepted
  • Racket generator code — unaffected by C++ standard version
  • Database/Sqitch layer — unaffected

Enforcement

  • Conan profile compiler.cppstd setting is the single source of truth
  • CI builds verify the compiler version matches the profile requirement

Key Files

  • conan_profiles/debug: compiler.cppstd=gnu23 → gnu26
  • conan_profiles/release: compiler.cppstd=gnu23 → gnu26
  • dslgen/generators/cmake/build-system-packaging.rkt: generated CMake C++ standard
