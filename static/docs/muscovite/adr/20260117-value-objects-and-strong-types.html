ADR: Value Objects and Strong Types

ADR: Value Objects and Strong Types

  • Status: Accepted
  • Date: 2026-01-17
  • Deciders: Architecture Team
  • Technical Story: Implement DDD Value Objects and constrained primitive types without JSONB

Context and Problem Statement

Domain-Driven Design relies heavily on Value Objects for modeling immutable, identity-less concepts. The current Muscovite implementation has placeholder support for VALUE_OBJECT but lacks:

  1. Storage strategy - How to persist multi-field value objects relationally
  2. Constrained primitives - Single-field types with domain-specific validation
  3. Default handling - Propagating defaults through all layers (PostgreSQL, API, gRPC, Qt)
  4. Collection support - Arrays of value objects without JSONB

JSONB is explicitly forbidden per project conventions (breaks type safety, relationships, migrations).

Decision Drivers

  • No JSONB or JSON types anywhere
  • Full type safety through all layers
  • PostgreSQL parameter limits (batch insert via composite arrays)
  • Support for both single-field (Email, Currency) and multi-field (AuditInfo, Measurement) patterns
  • Defaults must propagate correctly: column DEFAULT, trigger-computed, API parameter DEFAULT

Considered Options

  1. JSONB storage - ❌ Forbidden by project conventions
  2. EAV pattern - ❌ Loses type safety, poor query performance
  3. Relational patterns with PostgreSQL DOMAIN + composite TYPE - ✅ Selected

Decision Outcome

Implement two complementary constructs:

1. STRONG_TYPE → PostgreSQL DOMAIN

Single-field constrained primitives using PostgreSQL CREATE DOMAIN:

STRONG_TYPE EmailAddress TEXT
    CHECK email_format: VALUE ~ '^[^@]+@[^@]+\.[^@]+$'
    DOC "Valid email address";

STRONG_TYPE PositiveDecimal DECIMAL
    CHECK positive: VALUE > 0;

STRONG_TYPE Currency TEXT
    CHECK valid_currency: VALUE IN ('USD', 'EUR', 'GBP', 'CAD')
    DEFAULT 'USD';

Generates:

CREATE DOMAIN security.email_address AS TEXT
    CONSTRAINT email_format CHECK (VALUE ~ '^[^@]+@[^@]+\.[^@]+$');

CREATE DOMAIN security.positive_decimal AS DECIMAL
    CONSTRAINT positive CHECK (VALUE > 0);

CREATE DOMAIN security.currency AS TEXT
    DEFAULT 'USD'
    CONSTRAINT valid_currency CHECK (VALUE IN ('USD', 'EUR', 'GBP', 'CAD'));

Usage in entities:

ENTITY Invoice IDENTIFIED_BY id (
    id UUID NOT NULL,
    amount PositiveDecimal NOT NULL,
    currency Currency NOT NULL,
    contact_email EmailAddress
);

2. VALUE_OBJECT → Three Storage Modes

Multi-field value objects with explicit storage strategy:

2.1 @inline - Flatten fields into parent table

Best for: Audit trails, addresses, single-instance embedded data

VALUE_OBJECT AuditInfo @inline (
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_by UUID,
    modified_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    modified_by UUID
)
COMPUTED modified_at ON UPDATE: NOW()
IMMUTABLE created_at, created_by
DOC "Standard audit fields for all entities";

Usage:

ENTITY Patient IDENTIFIED_BY id (
    id UUID NOT NULL,
    name TEXT NOT NULL
)
USES AuditInfo;  -- Flattens all 4 fields into patient table

Generates:

CREATE TABLE security.patient (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name TEXT NOT NULL,
    -- Inlined from AuditInfo
    created_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    created_by UUID,
    modified_at TIMESTAMPTZ NOT NULL DEFAULT NOW(),
    modified_by UUID
);

CREATE TRIGGER patient_audit_trigger
    BEFORE UPDATE ON security.patient
    FOR EACH ROW EXECUTE FUNCTION security.update_modified_at();

CREATE TRIGGER patient_immutable_trigger
    BEFORE UPDATE ON security.patient
    FOR EACH ROW EXECUTE FUNCTION security.enforce_immutable('created_at', 'created_by');

2.2 @owned - Child table with lifecycle tied to parent

Best for: Collections with ordering, measurements, line items

VALUE_OBJECT Measurement @owned (
    value DECIMAL NOT NULL,
    unit TEXT NOT NULL,
    effective_at TIMESTAMPTZ DEFAULT NOW()
)
COMPUTED ordinal ON INSERT: auto_increment
DOC "Time-series measurement with automatic ordering";

Usage:

ENTITY Observation IDENTIFIED_BY id (
    id UUID NOT NULL,
    subject_id UUID NOT NULL
)
HAS measurements Measurement[];  -- Creates child table

Generates:

-- Composite type for batch insert parameter
CREATE TYPE fhir.measurement_t AS (
    value DECIMAL,
    unit TEXT,
    effective_at TIMESTAMPTZ
);

-- Child table
CREATE TABLE fhir.observation_measurement (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    observation_id UUID NOT NULL REFERENCES fhir.observation(id) ON DELETE CASCADE,
    ordinal INTEGER NOT NULL,
    value DECIMAL NOT NULL,
    unit TEXT NOT NULL,
    effective_at TIMESTAMPTZ DEFAULT NOW(),
    UNIQUE (observation_id, ordinal)
);

-- Ordinal trigger
CREATE TRIGGER observation_measurement_ordinal_trigger
    BEFORE INSERT ON fhir.observation_measurement
    FOR EACH ROW EXECUTE FUNCTION fhir.auto_ordinal('observation_id');

API with composite array parameter:

CREATE FUNCTION fhir_api.observation_insert(
    p_id UUID,
    p_subject_id UUID,
    p_measurements fhir.measurement_t[]  -- Single parameter for N measurements
) RETURNS fhir.observation AS $$
DECLARE
    v_result fhir.observation%ROWTYPE;
    v_measurement fhir.measurement_t;
    v_ordinal INTEGER := 0;
BEGIN
    INSERT INTO fhir.observation (id, subject_id)
    VALUES (COALESCE(p_id, gen_random_uuid()), p_subject_id)
    RETURNING * INTO v_result;
    
    FOREACH v_measurement IN ARRAY p_measurements LOOP
        v_ordinal := v_ordinal + 1;
        INSERT INTO fhir.observation_measurement (observation_id, ordinal, value, unit, effective_at)
        VALUES (v_result.id, v_ordinal, v_measurement.value, v_measurement.unit, 
                COALESCE(v_measurement.effective_at, NOW()));
    END LOOP;
    
    RETURN v_result;
END;
$$ LANGUAGE plpgsql;

2.3 @shared - Star schema dimension table

Best for: Reference data, CodeableConcept, reusable lookups

VALUE_OBJECT CodeableConcept @shared (
    system TEXT NOT NULL,
    code TEXT NOT NULL,
    display TEXT,
    version TEXT DEFAULT 'latest'
)
NATURAL_KEY (system, code, version)
DOC "FHIR CodeableConcept as reusable dimension";

Usage:

ENTITY Condition IDENTIFIED_BY id (
    id UUID NOT NULL,
    clinical_status CodeableConcept NOT NULL,  -- FK to dimension
    verification_status CodeableConcept
);

CONTEXT Fhir {
    DEFAULT active_status CodeableConcept = (
        system = 'http://hl7.org/fhir/condition-clinical',
        code = 'active',
        display = 'Active'
    );
}

Generates:

-- Dimension table
CREATE TABLE fhir.codeable_concept (
    id UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    system TEXT NOT NULL,
    code TEXT NOT NULL,
    display TEXT,
    version TEXT DEFAULT 'latest',
    UNIQUE (system, code, version)
);

-- Get-or-create function
CREATE FUNCTION fhir_api.codeable_concept_upsert(
    p_system TEXT,
    p_code TEXT,
    p_display TEXT DEFAULT NULL,
    p_version TEXT DEFAULT 'latest'
) RETURNS fhir.codeable_concept AS $$
DECLARE
    v_result fhir.codeable_concept%ROWTYPE;
BEGIN
    SELECT * INTO v_result
    FROM fhir.codeable_concept
    WHERE system = p_system AND code = p_code AND version = p_version;
    
    IF NOT FOUND THEN
        INSERT INTO fhir.codeable_concept (system, code, display, version)
        VALUES (p_system, p_code, p_display, p_version)
        RETURNING * INTO v_result;
    END IF;
    
    RETURN v_result;
END;
$$ LANGUAGE plpgsql;

-- Context default (seeded)
INSERT INTO fhir.codeable_concept (system, code, display, version)
VALUES ('http://hl7.org/fhir/condition-clinical', 'active', 'Active', 'latest')
ON CONFLICT (system, code, version) DO NOTHING;

DSL Syntax Summary

-- Strong Types (single-field, constrained → PostgreSQL DOMAIN)
STRONG_TYPE TypeName BaseType
    [CHECK constraint_name: expression]...
    [DEFAULT value]
    [DOC "description"];

-- Value Objects (multi-field → composite type / table patterns)
VALUE_OBJECT Name @inline|@owned|@shared (
    field1 Type [NOT NULL] [DEFAULT value],
    field2 Type [NOT NULL] [DEFAULT value],
    ...
)
[COMPUTED field ON INSERT|UPDATE: expression]...
[IMMUTABLE field1, field2, ...]
[NATURAL_KEY (field1, field2, ...)]  -- @shared only
[DOC "description"];

-- Usage in Entities
ENTITY Name IDENTIFIED_BY id (
    field1 StrongType,           -- Uses DOMAIN directly
    field2 SharedVO,             -- FK to dimension table
    ...
)
[USES InlineVO1, InlineVO2]      -- Flattens @inline fields
[HAS collection OwnedVO[]]       -- Creates child table
;

-- Context Defaults (for @shared value objects)
CONTEXT Name {
    DEFAULT alias SharedVO = (field1 = value1, field2 = value2);
}

Consequences

Positive

  • No JSONB - Full relational integrity maintained
  • Type safety - PostgreSQL DOMAINs enforce constraints at DB level
  • Batch-friendly - Composite type arrays avoid parameter limits
  • Star schema - @shared enables efficient dimension queries
  • Audit automation - @inline + COMPUTED/IMMUTABLE handles common patterns
  • Migration-safe - All patterns work with Sqitch semantic diff

Negative

  • More tables - @owned creates child tables (but properly indexed)
  • Complexity - Three storage modes require careful selection
  • Learning curve - Developers must understand when to use each pattern

Neutral

  • PostgreSQL-specific (DOMAINs, composite types) - acceptable for this project

Implementation Phases

  1. Phase 1: Type system (ddd-types.rkt) - Add strong-type, extend value-object
  2. Phase 2: Parser - STRONG_TYPE, @inline/@owned/@shared, COMPUTED, IMMUTABLE
  3. Phase 3: PostgreSQL generators - CREATE DOMAIN, CREATE TYPE, child tables
  4. Phase 4: Triggers - Audit, ordinal, computed, immutable enforcement
  5. Phase 5: API generators - Parameter defaults, composite arrays, upsert helpers
  6. Phase 6: gRPC/Qt - Optional fields, pre-populated defaults

AI Agent Guidance

When This Applies

  • User defines STRONG_TYPE → Generate PostgreSQL DOMAIN
  • User defines VALUE_OBJECT @inline → Flatten fields into parent table
  • User defines VALUE_OBJECT @owned → Create child table + composite type
  • User defines VALUE_OBJECT @shared → Create dimension table + upsert function
  • User uses COMPUTED → Generate BEFORE trigger
  • User uses IMMUTABLE → Generate UPDATE check trigger

When This Does NOT Apply

  • Regular entities with identity → Use standard ENTITY pattern
  • Lookup tables with user extensibility → Use LOOKUP_TABLE pattern
  • Cross-context references → Use standard FK relationships
