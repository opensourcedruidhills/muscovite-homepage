Multi-Agent Session Protocol

Multi-Agent Session Protocol

  • Status: superseded by 20260318-nats-agent-coordination-abandoned
  • Deciders: Johannes Lochmann
  • Date: 2026-03-06
  • Tags: agents, concurrency, governance, workspace

Technical Story: Multiple concurrent Copilot CLI / Claude Code sessions operating on the same checkout need coordination to prevent file clobbers, build directory corruption, database collisions, and FTR ownership conflicts.

Context and Problem Statement

Muscovite is developed AI-first: AI agents write 95%+ of the code, guided by ADRs and copilot instructions. As the project matures, concurrent agent sessions become necessary — a planning agent analyzing the codebase, an implementation agent executing the current iteration, and a hotfix agent resolving release issues, all running simultaneously.

The current workflow assumes a single agent at a time. Multiple concurrent sessions in the same checkout create race conditions:

  1. Shared working tree — two agents editing the same files produce untracked merge conflicts
  2. Shared build/debug/ — parallel CMake/Ninja builds corrupt each other
  3. Shared PostgreSQL on port 5555 — concurrent db-fresh-* destroys state
  4. Shared tmp/ — release.sh cleans tmp/ as part of the release ceremony, destroying any agent state stored there
  5. No FTR ownership tracking — two agents may claim the same feature number

How should concurrent agent sessions coordinate without introducing infrastructure complexity that agents cannot reliably operate?

Decision Drivers

  • Agents are LLMs — they can reliably create/check/delete files, but running lock daemons or maintaining database WAL files is fragile
  • File existence is the simplest mutex an LLM can reason about
  • The coordination layer must survive release.sh (which nukes tmp/)
  • Read-only sessions (planning, analysis) should require zero setup
  • Writing sessions must be fully isolated from each other and from the main checkout
  • The solution must be enforceable via copilot instructions (agent-readable rules)

Considered Options

  1. agents/ directory with file-based claims and git worktrees — dedicated gitignored workspace, POSIX file existence as mutex, worktrees for filesystem isolation
  2. SQLite-based session registry — agents/sessions.db with WAL mode for concurrent access
  3. Redis/external coordination service — distributed lock manager
  4. Convention-only (no enforcement) — document rules, trust agents to follow them

Decision Outcome

Chosen option: Option 1 — agents/ directory with file-based claims and git worktrees, because it uses primitives agents can reliably operate (file create/check/delete), survives release ceremonies, requires no external services, and is fully enforceable via copilot instructions.

Directory Layout

agents/                          ← gitignored, never cleaned by release.sh
├── .gitignore                   ← contains "*" — nothing committed
├── README.md                    ← human-readable protocol description (committed)
├── log.sh                       ← iteration log helper script (committed)
├── iteration-log.jsonl          ← append-only coordination log (gitignored, ephemeral)
├── claims/                      ← FTR/resource ownership registry
│   ├── ftr-232.lock             ← contains: role, PID, branch, timestamp
│   ├── release.lock             ← mutex: only one release ceremony at a time
│   └── develop-push.lock        ← mutex: only one push to develop at a time
├── planner/                     ← read-only analyst session
│   └── logs/
├── impl/                        ← implementation session
│   ├── logs/
│   └── worktree/                ← git worktree on feature branch
└── hotfix/                      ← release-fix session
    ├── logs/
    └── worktree/                ← git worktree on develop or hotfix branch

Claim File Format

role: impl
pid: 48291
branch: feature/ftr-232-saga-retry
started: 2026-03-06T10:31:00Z

Session Roles

Role Working Directory Branch Writes Code?
planner Main checkout (read-only) develop No
impl agents/impl/worktree/ feature/ftr-NNN-* Yes
hotfix agents/hotfix/worktree/ develop or hotfix/* Yes

Coordination Protocol

  1. Before starting work on an FTR: check agents/claims/ftr-NNN.lock — fail if exists
  2. Claim: write lock file with role, PID, branch, ISO 8601 timestamp
  3. Before pushing to develop: check agents/claims/develop-push.lock — fail if exists
  4. Before release ceremony: check agents/claims/release.lock — fail if exists
  5. On session end: remove all lock files owned by this session
  6. Stale lock detection: locks older than 4 hours are presumed stale (agent should verify PID before removing)

Positive Consequences

  • Zero external dependencies — works with POSIX filesystem primitives only
  • Survives release.sh — agents/ is not tmp/
  • Self-documenting — lock files contain metadata any agent can read and report
  • Scales to N concurrent sessions by adding roles
  • Worktree isolation eliminates git working tree conflicts entirely
  • Each worktree has its own relative build/ directory — no build system collisions

Negative Consequences

  • Stale locks if agent crashes without cleanup — mitigated by 4-hour TTL and PID check
  • Agents must follow the protocol — enforceable via copilot instructions but not mechanically enforced
  • Worktree creation/cleanup adds setup cost per writing session (~5 seconds)
  • agents/ README is the only committed file — all runtime state is ephemeral

Pros and Cons of the Options

Option 1: agents/ directory with file-based claims

  • Good, because file existence is the simplest check an LLM can perform
  • Good, because lock files are human-readable and self-documenting
  • Good, because git worktrees provide real filesystem isolation
  • Good, because no external services or background processes required
  • Bad, because stale locks require TTL-based heuristic cleanup
  • Bad, because protocol enforcement is rule-based, not mechanical

Option 2: SQLite-based session registry

  • Good, because SQL queries enable richer coordination (e.g., "who has FTR-232?")
  • Good, because WAL mode supports concurrent readers and one writer
  • Bad, because agents managing SQLite connections and WAL recovery is fragile
  • Bad, because a corrupted database file blocks all agents

Option 3: Redis/external coordination service

  • Good, because distributed locks are well-understood
  • Good, because TTL is built into Redis key expiry
  • Bad, because introduces an external dependency for development workflow
  • Bad, because requires running infrastructure that agents cannot self-manage
  • Bad, because violates project philosophy of zero-infrastructure local development

Option 4: Convention-only

  • Good, because zero implementation cost
  • Bad, because agents have no way to detect conflicts before they occur
  • Bad, because "trust agents to follow rules" has already failed (governance regressions)

Amendment: Iteration Log (2026-03-07)

Problem: Claim files (binary mutex) failed to prevent duplicate work. During v1.52.0, two agents independently implemented FTR-241 through FTR-245 because:

  • Claims are point-in-time snapshots, not progress streams
  • Agent A's claims existed but Agent B started before checking (or started a new session)
  • No shared timeline showing "FTR-241 is 60% done, PR #58 created"

Solution: Append-only iteration log at agents/iteration-log.jsonl.

Iteration Log Format

{"ts":"2026-03-07T07:00:00Z","agent":"hotfix","event":"start","ftr":257,"detail":"Starting FTR-257 on branch feature/ftr-257-framework-domainevent-proto-payload"}
{"ts":"2026-03-07T07:05:00Z","agent":"hotfix","event":"progress","ftr":257,"detail":"GROUND-TRUTH.md updated, ctest --preset unit 276/276 passing"}
{"ts":"2026-03-07T07:08:00Z","agent":"hotfix","event":"pr","ftr":257,"detail":"PR #63 created: feat(FTR-257): Document DomainEvent proto payload type"}
{"ts":"2026-03-07T07:10:00Z","agent":"impl","event":"merge","ftr":257,"detail":"PR #63 merged to develop"}
{"ts":"2026-03-07T08:00:00Z","agent":"hotfix","event":"done","ftr":257,"detail":"Claim released, worktree removed"}

Field Schema

Field Type Required Description
ts ISO 8601 yes Timestamp of the log entry
agent string yes Agent role: impl, hotfix, planner
event enum yes start, progress, pr, merge, done, blocked, release, info
ftr integer no FTR number (omit for non-FTR events like release)
detail string yes Human-readable description of what happened

Iteration Log Protocol (IRON RULE — NO EXCEPTIONS)

  1. Read the entire log on session start — understand what other agents have done
  2. Log start before claiming any FTR — even before creating the lock file
  3. Log progress at each significant milestone — test passing, file created, PR created
  4. Log pr when a PR is created — include PR number and title
  5. Log done when FTR work is complete — after claim is released
  6. Log blocked if stuck — include what is blocking and what help is needed
  7. Log info for non-FTR coordination — iteration-level decisions, dependency observations

FORBIDDEN: Starting FTR work without reading the log first | Failing to log start before claiming | Going silent (no log entries) for more than 30 minutes of active work.

Helper Script

Source agents/log.sh for the agent_log function:

source agents/log.sh
agent_log hotfix start 257 "Starting FTR-257 on branch feature/ftr-257-*"
agent_log hotfix progress 257 "Tests passing, PR ready"
agent_log hotfix pr 257 "PR #63 created"
agent_log hotfix done 257 "Claim released"
agent_log impl info "" "Reviewing iteration log before starting"

Planner Sessions on develop

Planning sessions (read-only analysis, iteration review, code review) MAY work directly on the develop branch in the main checkout without a worktree. They MUST still read and write the iteration log. They MUST NOT modify source files.

AI Agent Guidance

When This Decision Applies

  • When running multiple Copilot CLI or Claude Code sessions concurrently on the same repository
  • When an agent needs to claim exclusive ownership of an FTR number or shared resource
  • When setting up a writing session (implementation or hotfix) that will modify code

When This Decision Does NOT Apply

  • Single-agent operation — no claims or worktrees needed
  • Read-only sessions (planning, analysis, code review) — use main checkout directly
  • CI/CD pipelines — these run in isolated containers, not shared checkouts

Enforcement

  • Rule 26 in .github/copilot-instructions.md — agents read this every session
  • Lock file existence check before FTR work, develop push, or release ceremony
  • Stale lock TTL (4 hours) + PID verification for cleanup
