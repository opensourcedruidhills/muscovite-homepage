Code Generation Over C++ Templates for Qt Widgets

Code Generation Over C++ Templates for Qt Widgets

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: qt, widgets, code-generation, moc, type-safety

Context and Problem Statement

Muscovite generates Qt Widget classes (edit dialogs, list views, reference widgets) for each entity in the domain model. Two approaches exist for generating typed widgets like FK reference comboboxes and strong-type input fields:

  1. C++ templates: EntityReferenceWidget<T> parameterized by entity type
  2. Code generation: ClinicReferenceWidget, PatientReferenceWidget — one specialized class per entity

Qt's Meta-Object Compiler (MOC) cannot process C++ templates that introduce new Q_OBJECT macros. Templates inheriting from QObject subclasses cannot define new signals or slots with entity-specific types.

Question: Should Muscovite use C++ templates with type erasure, or generate specialized widget classes per entity?

Decision Drivers

  • Qt MOC compatibility — generated code must compile without MOC workarounds
  • Type safety — signals should carry actual entity types, not erased QString/QVariant
  • Muscovite's philosophy — we are already a code generator; generating more code is free
  • Customizability — users may need to subclass or extend generated widgets
  • Compile-time safety — errors caught at compile time, not runtime

Considered Options

Option A: Runtime Polymorphism (Templates with Type Erasure)

template<typename T>
class EntityReferenceWidget : public QComboBox {
    Q_OBJECT  // Inherits QComboBox's Q_OBJECT — no new signals
signals:
    void entityIdChanged(QString id);  // Type-erased to QString
};

Option B: Code Generation (Specialized Classes)

class ClinicReferenceWidget : public QComboBox {
    Q_OBJECT
signals:
    void clinicSelected(clinic::Clinic::id_type clinic_id);  // Fully typed
};

Decision Outcome

Chosen option: Option B — Code Generation, because:

  • MOC compatible: Each generated class is a plain QObject subclass — no template/MOC interaction
  • Full type safety: Signals carry actual entity types (clinic::Clinic::id_type), not type-erased strings
  • Zero runtime overhead: All specialization happens at compile time via code generation
  • Aligns with Muscovite: We already generate ~100K lines of C++; generating widget classes is incremental
  • Customizable: Users can subclass ClinicReferenceWidget to add project-specific behavior
  • Debuggable: Generated code is readable, not template instantiation noise

Generated Widget Structure

Entity Reference Widget (per FK relationship):

  • Extends QComboBox
  • Constructor takes std::shared_ptr<EntityService> for data loading
  • selectedId() → std::optional<Entity::id_type> (typed)
  • setSelectedId(const Entity::id_type&) (typed)
  • refresh() — reloads from service
  • Signal: entitySelected(Entity::id_type) — fully typed

Strong Type Input Widget (per STRONG_TYPE with CHECK):

  • Extends QLineEdit (or QSpinBox, QDoubleSpinBox depending on base type)
  • Q_PROPERTY(bool valid READ isValid NOTIFY validChanged)
  • Constructor installs QRegularExpressionValidator from DSL CHECK constraint
  • value() → std::optional<StrongType> (typed)
  • setValue(const StrongType&) (typed)
  • Visual feedback (red border) on validation failure

Positive Consequences

  • No MOC issues — ever
  • Full IntelliSense/autocomplete for generated widget APIs
  • Signals carry domain-meaningful types
  • Users can subclass to customize
  • Generated code is the documentation

Negative Consequences

  • More generated code volume (one .h/.cpp pair per FK reference, per strong type)
  • Longer compilation times for projects with many entities
  • If widget pattern changes, all generated widgets must be regenerated (but that's what racket generate.rkt does)

AI Agent Guidance

When This Decision Applies

  • Generating Qt Widget classes for entities, FK references, or strong types
  • Adding new widget types to the Qt Widgets generator
  • Discussing whether to use C++ templates in generated Qt code

When This Decision Does NOT Apply

  • Qt QML generation (QML has no MOC constraint — ViewModels are already generated per-entity)
  • C++ DBA layer (no Qt dependency)
  • gRPC service generation
  • Non-Qt targets (future mobile frontends)

Enforcement

  • Never generate template<typename T> class ... { Q_OBJECT in Qt Widgets output
  • Every entity with FK relationships gets a dedicated *ReferenceWidget class
  • Every STRONG_TYPE with CHECK constraints gets a dedicated *Input class
  • Widget classes are generated in dslgen/generators/qt/widgets-views.rkt
