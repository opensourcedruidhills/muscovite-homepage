Racket Idiomatic Style and Byte Compilation

Racket Idiomatic Style and Byte Compilation

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-02-26
  • Tags: racket, performance, idioms, byte-compilation, code-quality
  • Relates to: 20260218-template-dsl-for-code-generation.md
  • Relates to: 20260223-racket-generator-unit-testing-strategy.md

Technical Story: Muscovite's 98K-line Racket codebase operates without byte compilation (zero .zo files) and uses non-idiomatic patterns — 382 set! mutations, zero contract-out enforcement, zero in-list annotations. Module load time for generate.rkt is 71 seconds before any generation work begins. AI agents working on the codebase are hampered by unfamiliar idioms because the code does not match standard Racket patterns found in training corpora.

Context and Problem Statement

Performance Problem

Every racket generate.rkt invocation re-expands and re-compiles all 349 .rkt files from source. There are zero .zo bytecode files, zero compiled/ directories, no raco make step, and no package installation despite a valid info.rkt existing.

Measured cold module load times (Racket v9.0 [cs], no bytecache):

Module Load Time
model/ddd-types.rkt 5.2s
parser/parser.rkt 10.5s
generators/cpp/dba/unified.rkt 22.3s
generate.rkt (all deps) 71.1s

This 71-second overhead applies to every generation run, every test invocation, and every REPL session. The actual generation work (which is fast) is dominated by module expansion.

Idiom Problem

Quantitative analysis (Feb 2026) of the generator codebase reveals patterns that deviate from idiomatic Racket:

Metric Current Idiomatic Racket
set! mutations in generators 111 Replaced by for/fold, let-values
set! across entire dslgen/ 382 Near zero in functional Racket
make-hash (mutable) in generators 102 Use immutable hash + hash-set
match expressions in generators 23 Primary dispatch mechanism
for/fold in generators 14 Should replace most set! loops
in-list annotations 0 On every for over a list
contract-out in provides 0 On all public module boundaries
define/contract 0 On critical internal functions
module+ test submodules 20 (of 349) In every module with testable logic
Struct #:guard / #:property 1 For invariant enforcement

The central IR module (ddd-types.rkt) imports racket/contract but uses (provide (all-defined-out)) — a blanket export with zero contract enforcement.

AI Agent Effectiveness Problem

AI models (Claude, GPT, Copilot) are trained on vastly more idiomatic Racket (from The Racket Guide, Racket documentation, and open-source Racket projects) than on the imperative-style patterns used in Muscovite. When the codebase uses set! accumulator loops instead of for/fold, or bare struct accessors instead of match, the models generate code that doesn't stylistically match. This is not a Racket-vs-Python problem — it's a non-idiomatic-vs-idiomatic problem.

Decision Drivers

  • 71-second cold start is the primary developer pain point, blocking rapid iteration
  • Byte compilation is a zero-risk, zero-cost fix available today
  • Contract enforcement catches cross-module type errors at runtime — critical for a 70-struct IR
  • in-list annotations are a mechanical fix with measurable performance impact
  • for/fold replacing set! eliminates an entire class of mutation bugs
  • Immutable hashes remove shared-state surprises in generators
  • module+ test enables targeted unit testing (per ADR 20260223)
  • AI agents produce better code when the existing codebase matches standard Racket patterns

Considered Options

  1. Status quo — continue with no byte compilation and current coding style
  2. Byte compilation only — add raco make but don't change coding style
  3. Byte compilation + incremental idiomatic migration — enforce new standards for new code, migrate existing code file-by-file when touched
  4. Rewrite in Python — abandon Racket entirely

Decision Outcome

Chosen option: Option 3 — Byte compilation + incremental idiomatic migration, because byte compilation eliminates 90% of the performance pain with zero risk, and idiomatic migration improves both code quality and AI agent effectiveness without a disruptive rewrite.

Phase 1: Byte Compilation (Week 1 — Zero Risk)

1a. Add raco make to generation workflow

# Before generation
cd dslgen && raco make generate.rkt

Expected: Module load drops from 71s → 3–8s. Incremental recompilation only rebuilds changed files and their dependents.

1b. Add compiled/ to .gitignore

# Racket bytecode (generated by raco make)
**/compiled/

1c. Add CMake integration target

add_custom_target(racket-compile
    COMMAND raco make generate.rkt
    WORKING_DIRECTORY ${CMAKE_SOURCE_DIR}/dslgen
    COMMENT "Compiling Racket modules to bytecode"
)

All generate-* CMake targets depend on racket-compile.

1d. Add raco make before test runs

cd dslgen && raco make tests/run-all-tests.rkt
racket tests/run-all-tests.rkt  # now fast

Phase 2: Idiomatic Standards for New Code (Week 2 — Enforce on Write)

All new and modified Racket files MUST follow these rules:

2a. in-list on all for comprehensions over lists

;; FORBIDDEN
(for ([x xs]) ...)
(for/list ([x xs]) ...)

;; REQUIRED
(for ([x (in-list xs)]) ...)
(for/list ([x (in-list xs)]) ...)

Why: in-list eliminates generic sequence dispatch. Measurable 5–10% speedup on iteration-heavy generators.

2b. for/fold instead of set! accumulator loops

;; FORBIDDEN (mutable accumulator)
(define result '())
(for ([ctx contexts])
  (set! result (append result (generate-context ctx))))
result

;; REQUIRED (functional fold)
(for/fold ([acc '()])
          ([ctx (in-list contexts)])
  (append acc (generate-context ctx)))

;; Or when building a list:
(for/append ([ctx (in-list contexts)])
  (generate-context ctx))

2c. Immutable hashes for lookup tables

;; FORBIDDEN (mutable hash)
(define h (make-hash))
(hash-set! h 'key val)

;; REQUIRED (immutable hash)
(define h (hash 'key val))
(define h2 (hash-set h 'key2 val2))

Exception: Performance-critical paths with >1000 insertions may use make-hash with a WHY: performance comment.

2d. contract-out on public module boundaries

;; FORBIDDEN (blanket export)
(provide (all-defined-out))

;; REQUIRED (explicit contracts)
(provide
 (contract-out
  [struct entity ([uuid string?]
                  [name symbol?]
                  [fields (listof field?)]
                  ...)]
  [make-entity (->* (symbol?)
                    (#:uuid string? #:fields (listof field?))
                    entity?)]))

Priority targets for contract enforcement:

  1. model/ddd-types.rkt — the IR core (70 structs)
  2. model/type-registry.rkt — type mapping functions
  3. generators/headers.rkt — license header generation
  4. parser/parser.rkt — parser entry points

2e. module+ test for unit-testable logic

;; At bottom of every module with testable functions
(module+ test
  (require rackunit)
  (check-equal? (symbol->sql-id 'PatientContext) "patient_context")
  (check-equal? (symbol->cpp-class 'order-item) "OrderItem"))

Phase 3: Incremental Migration of Existing Code (Ongoing)

When a generator file is touched for any reason (feature, bugfix, refactor), apply the Phase 2 standards to the entire file. Do not create dedicated "style migration" commits — fold style improvements into functional changes.

Track migration progress:

# Audit commands (add to tools/)
grep -rc "set!" dslgen/generators/ --include="*.rkt" | awk -F: '{s+=$2} END {print "set! count:", s}'
grep -rL "in-list" dslgen/generators/ --include="*.rkt" | wc -l  # files missing in-list
grep -rL "contract-out" dslgen/ --include="*.rkt" | wc -l        # files without contracts

Positive Consequences

  • 71s → 3–8s module load time (byte compilation alone)
  • Contract violations caught at module boundaries, not deep in generator logic
  • for/fold eliminates mutation bugs in list accumulation
  • AI agents produce more consistent code matching idiomatic patterns
  • module+ test enables the per-construct testing strategy (ADR 20260223)
  • in-list provides measurable iteration speedup across 646 for/list expressions

Negative Consequences

  • compiled/ directories appear alongside source (managed by .gitignore)
  • Contract checking adds ~5–15% runtime overhead (acceptable for offline generators)
  • Two coding styles coexist during migration period (old files vs new/touched files)
  • Initial raco make adds ~2–3 minutes to first build (subsequent builds are incremental)

Enforcement

AI Agent Instructions (copilot-instructions.md, CLAUDE.md, AGENTS.md)

Add rule: Racket Idiomatic Style (ADR 20260226)

### N. Racket Idiomatic Style (MANDATORY for new/modified code)

Per ADR 20260226:
- `in-list` on ALL `for` comprehensions over lists
- `for/fold` instead of `set!` accumulator loops
- Immutable `hash` instead of `make-hash` + `hash-set!`
- `contract-out` on public module provides (not `all-defined-out`)
- `module+ test` in every module with testable logic
- `raco make generate.rkt` before any generation or test run

**FORBIDDEN**: `set!` for accumulation | `(provide (all-defined-out))` in new files |
bare `for` without `in-list` | mutable hashes without `WHY: performance` comment

Grammar Conformance Parity

Extend test-grammar-conformance.rkt (or create test-racket-style.rkt) to check:

  • Every provide form in model/ files uses contract-out
  • Every for in generators/ uses in-list/in-hash/in-range/in-naturals

CMake Gate

The racket-compile target runs before all generation. If bytecache is stale, regeneration is safe but slow (falls back to source expansion).

AI Agent Guidance

When This Decision Applies

  • When writing any new .rkt file — follow all Phase 2 rules from the start
  • When modifying an existing .rkt file — apply Phase 2 rules to the whole file
  • When running generation or tests — ensure raco make has been run
  • When reviewing AI-generated Racket code — verify idiomatic patterns

When This Decision Does NOT Apply

  • Quick REPL experiments (no byte compilation needed)
  • One-off scripts in tools/ (style rules relaxed for throwaway code)

Quick Reference

;; ✅ Idiomatic
(for/list ([x (in-list xs)]) (f x))
(for/fold ([acc '()]) ([x (in-list xs)]) (cons (f x) acc))
(define h (hash 'a 1 'b 2))
(provide (contract-out [f (-> symbol? string?)]))

;; ❌ Non-idiomatic
(for/list ([x xs]) (f x))
(define result '()) (for ([x xs]) (set! result (cons (f x) result)))
(define h (make-hash)) (hash-set! h 'a 1)
(provide (all-defined-out))
