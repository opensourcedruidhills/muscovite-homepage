Deployment Infrastructure: Hetzner Cloud over AWS for Development and Non-Compliance Workloads

Deployment Infrastructure: Hetzner Cloud over AWS for Development and Non-Compliance Workloads

  • Status: accepted
  • Date: 2026-02-28
  • Tags: deployment, infrastructure, cloud, aws, hetzner, cost, grpc, ec2, lambda
  • Deciders: Johannes Lochmann

Technical Story: Evaluated cloud deployment options for generated gRPC services (e.g., fhirhealth-server) covering AWS Lambda, AWS EC2 (eu-south-1 Milan), and Hetzner Cloud. Selected Hetzner for development and non-regulated workloads.

Context and Problem Statement

Generated Muscovite applications produce a monolithic gRPC server binary ({project}-server) via server_main.cpp. This binary registers all bounded-context service implementations against a single grpc::ServerBuilder, calls BuildAndStart(), and blocks in server->Wait(). It needs to run somewhere.

Three candidate deployment models were evaluated against cost, complexity, and compliance requirements:

  1. AWS Lambda — serverless, per-invocation billing, HTTP API Gateway frontend
  2. AWS EC2 (eu-south-1 Milan) — traditional VM, on-demand Linux
  3. Hetzner Cloud — European VPS/cloud, hourly billing, no managed-services ecosystem

The generated server is a long-running blocking process — the exact workload traditional VMs are designed for. The question is which VM provider best fits development-phase and non-regulated production workloads.

Decision Drivers

  • Cost for dev/prototype phase (primary driver)
  • Zero architectural changes required to generated code
  • GDPR compliance for EU data
  • HIPAA / data residency requirements for regulated health data (future production concern)
  • Operational simplicity (no DevOps team)
  • Egress pricing (C++ gRPC generates non-trivial binary payloads)

Considered Options

  1. AWS Lambda — new --aws-lambda generator target, per-context Lambda, API Gateway HTTP API, aws-lambda-cpp, JSON adapter layer
  2. AWS EC2 eu-south-1 (Milan) — t3.medium (2 vCPU, 4GB), RDS PostgreSQL, no code changes
  3. Hetzner Cloud CX33 — 4 vCPU, 8GB, 80GB NVMe SSD, PostgreSQL on same host or Hetzner Managed DB

Decision Outcome

Chosen option: Hetzner Cloud, because it requires zero code changes to the generated server, costs 3–6× less than AWS EC2 for equivalent specs, includes 20 TB egress, and is fully GDPR-compliant as a German provider operating under EU data protection law.

AWS Lambda was ruled out for this workload: the generated server_main.cpp uses a blocking grpc::ServerBuilder::Wait() loop and a shared ConnectionPool — both are architectural incompatibilities with Lambda's handler model. Supporting Lambda would require a new generator target (~2,500–3,500 lines of Racket) producing a per-context JSON adapter, aws-lambda-cpp runtime, and SAM/Terraform templates. This is a valid future investment but not appropriate for dev-phase deployment.

AWS EC2 Milan is reserved for workloads where Italian data residency or HIPAA BAA is required.

Positive Consequences

  • docker run fhirhealth-server — zero deployment friction; no generator changes
  • €5.99/month total for dev (VM + Postgres on same CX33 host + IPv4) vs ~$38/month on AWS EC2
  • 20 TB egress included — gRPC binary traffic is irrelevant at dev load
  • GDPR-compliant by default (German provider, EU data protection law)
  • Hetzner Cloud API (hcloud CLI) is mature; infrastructure scripting is straightforward

Negative Consequences

  • No HIPAA BAA available — Hetzner cannot be used for FHIR production workloads with real patient data
  • eu-south-1 (Italy) data residency is unavailable — nearest Hetzner location is Nuremberg (~840km) or Falkenstein
  • No managed services ecosystem (no equivalent of RDS Multi-AZ, IAM, CloudWatch, Secrets Manager)
  • If AWS is needed for production, dev/prod parity requires care (same Docker image, different infra)

Pros and Cons of the Options

AWS Lambda

  • Good, because per-invocation billing is near-zero at dev load
  • Good, because scales to zero — no idle cost
  • Bad, because grpc::ServerBuilder::Wait() is fundamentally incompatible — Lambda requires a handler function, not a blocking process
  • Bad, because gRPC frames are incompatible with API Gateway HTTP API — requires full JSON adapter layer
  • Bad, because requires ~2,500–3,500 new Racket generator lines before any deployment is possible
  • Bad, because ConnectionPool(min=2, max=10) × 9 contexts × warm instances exhausts RDS connection limits on small instances without RDS Proxy (+$10.80/month)
  • Bad, because BUILD_SHARED_LIBS ON requires Docker container packaging anyway, removing the "no EC2" advantage

AWS EC2 eu-south-1 (Milan)

  • Good, because no code changes required — docker run works immediately
  • Good, because HIPAA BAA available — mandatory for FHIR production with real patient data
  • Good, because Italian data residency — meets strictest interpretation of EU/NIS2 for health data
  • Good, because formal per-instance SLA (99.5%)
  • Bad, because ~$38/month for t3.medium + 20GB EBS + optional RDS — expensive for dev
  • Bad, because egress pricing ($0.09/GB after 100GB free tier) becomes significant at production scale
  • Bad, because Reserved Instance commitment required to achieve reasonable pricing (~$23/month at 1yr)

Hetzner Cloud CX33

  • Good, because €5.49/month for 4 vCPU / 8GB / 80GB NVMe — 6× cheaper than equivalent EC2
  • Good, because 20 TB egress included at all EU locations
  • Good, because GDPR-compliant (German provider, ISO 27001, EU data protection)
  • Good, because docker run works identically — no deployment-specific code changes
  • Good, because Hetzner Cloud Firewall, private Networks, load balancer all available at low cost
  • Good, because ARM64 available (CAX31: 8 vCPU / 16GB / €12.49/month) — matches generated project's CMake arm64 support
  • Bad, because no HIPAA BAA — cannot host real patient health data
  • Bad, because no Italian data residency — disqualifies it for strictest FHIR production in Italy
  • Bad, because no managed secrets, IAM, CloudWatch equivalents — operational tooling must be self-hosted or third-party

Production Decision Rule

Scenario Provider
Development / prototyping / demos Hetzner Cloud
Production without HIPAA / Italian residency requirement Hetzner Cloud (dedicated CCX33: €16.49/month)
FHIR health data production (Italy, real patients) AWS eu-south-1 (HIPAA BAA, Italian data residency mandatory)
High-traffic production, multi-AZ HA AWS (use Reserved Instances to offset cost)

AI Agent Guidance

When This Decision Applies

  • When generating deployment documentation or docker-compose.yml for generated projects
  • When advising on infrastructure setup for running generated gRPC servers
  • When estimating hosting costs for generated applications

When This Decision Does NOT Apply

  • FHIR applications with real patient data — AWS eu-south-1 is mandatory (HIPAA BAA)
  • Workloads requiring Italian data residency under NIS2

Key Architecture Fact

The generated fhirhealth-server (and all Muscovite-generated gRPC servers) is a blocking long-running process. It is not refactorable to Lambda without a dedicated generator target. Do not suggest Lambda as a "quick option" — it requires ~2,500–3,500 new Racket generator lines as a prerequisite.
