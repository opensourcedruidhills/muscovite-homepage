ADR: gRPC-Backed Qt Item Models (Widgets + QML)

ADR: gRPC-Backed Qt Item Models (Widgets + QML)

  • Status: Accepted
  • Date: 2025-12-30
  • Deciders: Muscovite Team

Context

We need a reusable UI data layer for generated frontends. The first frontend is Qt Widgets, with Qt QML to follow. Lists must be gRPC-backed, support auth header injection, be sortable/searchable, and be shared between Qt Widgets and Qt QML without duplication. QTableWidget is too view-centric and cannot be reused in QML. We also want per-entity models generated from .ddd via Racket.

Decision

Use a gRPC-backed QAbstractItemModel as the core data source for all entities. Provide a base class in a shared library (e.g., muscoviteqt-models) that handles:

  • gRPC calls (fetch/filter/paginate) with JWT metadata injection (reuse ServiceClientBase)
  • Async loading with signals for success/failure
  • Column metadata derived from DSL (headers, roles, display/tooltip data)
  • Client-side sorting/filtering; server-side pagination hooks
  • Error propagation suitable for both Qt Widgets and Qt QML

Generated per-entity models will subclass this base model and map DSL fields to Qt roles. Views are thin wrappers:

  • Qt Widgets: QTableView/QListView bound to the model
  • Qt QML: ListView/TableView bound via roles

Consequences

Positive

  • Single data layer reused by Qt Widgets and Qt QML
  • Clear separation of model (data) and view (UI), easier to test
  • Supports future frontends (QML) without rework
  • Aligns with Qt best practices (model/view)
  • Keeps auth and gRPC concerns centralized

Negative / Mitigations

  • Slightly higher upfront complexity vs QTableWidget → mitigated by providing a robust base class
  • Need to design Qt roles and column metadata carefully → derive from DSL with sensible defaults
  • Pagination and filtering APIs must be stable for both Widgets and QML → document in generator contract

Notes

  • QTableWidget is explicitly rejected for data binding; we will use QAbstractItemModel-derived classes.
  • The base model lives in a separate shared library so that QML can link without widget dependencies.
