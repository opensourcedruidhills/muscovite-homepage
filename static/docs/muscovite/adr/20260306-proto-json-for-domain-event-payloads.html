ADR: Proto-JSON Serialization for Domain Event Payloads

ADR: Proto-JSON Serialization for Domain Event Payloads

  • Date: 2026-03-06
  • Status: accepted
  • Tags: events, serialization, outbox, governance, protobuf
  • Supersedes: implicit nlohmann::json usage in event pipeline (no prior ADR)

Technical Story: Governance audit found nlohmann::json payloads in generated CommandService, EventPublisher, and domain_events SQL — contradicting ADR 20260204 spirit ("no JSON") and the type-safety mandate.

Context and Problem Statement

Every generated CommandService and EventPublisher constructs event payloads using nlohmann::json, serializes them to a string, stores them in {schema}.domain_events.payload (TEXT column), and roundtrips through nlohmann::json::parse() when filling proto wire messages. Yet the project already generates typed protobuf event messages per context (events.proto) that are only used on the gRPC wire — never for persistence.

This creates three problems:

  1. Governance contradiction: ADR 20260204 chose "strongly typed per-event payload tables (no JSON)" for SOURCED_EVENT. The regular domain_events outbox uses the same untyped JSON pattern that ADR 20260204 explicitly rejected — just in a different table.

  2. Unnecessary dependency: nlohmann_json is linked into every generated DBA and gRPC library solely for event payload serialization. Removing it from the event hot path simplifies the dependency graph.

  3. Type duplication: Proto event messages are already generated with typed fields. Constructing a parallel nlohmann::json object with the same fields is redundant work that can diverge silently.

Current Data Flow

CommandService → nlohmann::json{...} → .dump() → TEXT column
TEXT column → nlohmann::json::parse() → fill proto message → gRPC wire

Desired Data Flow

CommandService → proto event message → MessageToJsonString() → TEXT column
TEXT column → JsonStringToMessage() → proto message → gRPC wire (or NATS)

Decision Drivers

  • Typed proto event messages (events.proto) already exist per context
  • ADR 20260305 Phase 2 (outbox relay → NATS) will read domain_events and publish to NATS — proto is the natural NATS wire format
  • domain_events.payload is already TEXT NOT NULL in sqitch-events.rkt (no DDL change needed)
  • Proto canonical JSON is deterministic and human-readable in psql
  • nlohmann_json remains needed for other uses (VO JSONB columns, OpenSearch document mapping, metadata) — this ADR scopes only the event payload path

Considered Options

Option A: Proto-binary payload (base64 in TEXT column)

  • Serialize event proto → SerializeToString() → base64-encode → TEXT column
  • Pro: Smallest payload, fastest (de)serialization, no JSON parse at all
  • Con: Payload is opaque in SQL — no SELECT payload FROM domain_events debugging without a proto decoder tool. Breaks DBA ergonomics.

Option B: Proto-JSON payload (canonical JSON via protobuf API)

  • Serialize event proto → google::protobuf::util::MessageToJsonString() → TEXT column
  • Deserialize via google::protobuf::util::JsonStringToMessage()
  • Pro: Human-readable in SQL, deterministic serialization, typed at C++, drop-in for NATS relay (Phase 2 can deserialize to proto and publish binary)
  • Con: Still JSON content in TEXT — not a schema-level type safety win. Slightly larger than binary. Adds protobuf::libprotobuf link to DBA layer.

Option C: Typed payload columns per event (extend ADR 20260204 to outbox)

  • Each EVENT declaration generates its own outbox table with typed columns
  • Pro: Full type safety at DB level. Zero JSON. Maximum ADR 20260204 alignment.
  • Con: Major DDL explosion (N tables per context), saga orchestration and outbox relay must dispatch per event type, multi-iteration migration scope.

Decision Outcome

Chosen option: Option B — Proto-JSON payload via protobuf canonical JSON.

This is the pragmatic middle ground that:

  • Removes nlohmann::json from the event serialization hot path
  • Uses the typed proto messages that are already generated
  • Keeps domain_events.payload human-readable in SQL
  • Aligns with NATS Phase 2 (FTR-250): the relay reads TEXT, deserializes to proto, publishes proto-binary to NATS JetStream
  • Requires no DDL change (payload stays TEXT NOT NULL)

What Does NOT Change (Out of Scope)

  • metadata JSONB column: Stays JSONB — metadata is schemaless by design (correlation IDs, trace context, custom headers). Separate ADR if needed.
  • Value Object JSONB columns: @owned VO storage as JSONB is a separate concern (sqitch-entity.rkt). Separate ADR needed.
  • OpenSearch document mapping: Uses nlohmann::json for document construction — this is the correct tool for interacting with the OpenSearch REST API.
  • sqitch-event-changes.rkt (event store DDL): Uses payload JSONB for the event store variant. ADR 20260204 already mandates typed tables for SOURCED_EVENT — the event store generator should move to typed columns per ADR 20260204 (separate work, already covered by existing ADR).

Positive Consequences

  • Event payload serialization uses the same proto types as the gRPC wire
  • Clean pipeline for NATS Phase 2: TEXT → proto → NATS binary
  • nlohmann_json removed from event hot path (fewer transitive dependencies)
  • Proto canonical JSON is deterministic — byte-equal payloads for identical events
  • psql users can still SELECT payload FROM domain_events and read JSON

Negative Consequences

  • protobuf::libprotobuf becomes a DBA layer link dependency (currently only gRPC layer)
  • Slightly larger payloads than binary proto (canonical JSON includes field names)
  • Generated CommandService code changes from nlohmann::json construction to proto message construction — affects ~213 generated files (all generator-driven)

Implementation Plan

  1. Generator: Modify unified-repository-mutations.rkt and command-methods.rkt to emit proto event construction instead of nlohmann::json construction
  2. Generator: Modify events.rkt (gRPC events generator) to use MessageToJsonString / JsonStringToMessage instead of nlohmann::json
  3. Framework: Update DomainEvent struct in unit_of_work.hpp to accept const google::protobuf::Message& for payload
  4. Generator: Update CMake/Conan generators to link protobuf::libprotobuf on DBA targets and remove nlohmann_json from event-only link targets
  5. Test: Verify roundtrip: proto → canonical JSON → TEXT → canonical JSON → proto
  6. Regenerate: All sample projects
