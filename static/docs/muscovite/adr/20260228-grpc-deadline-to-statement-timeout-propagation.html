gRPC Deadline to PostgreSQL Statement Timeout Propagation

gRPC Deadline to PostgreSQL Statement Timeout Propagation

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-28
  • Tags: grpc, postgresql, resilience, timeout, deadline, dba, code-generation
  • Relates: 20260120-domain-services-as-aggregate-boundary.md, 20260120-cqrs-ready-architecture.md

Technical Story: Generated gRPC services have no deadline awareness. Server handlers mark ServerContext* as [[maybe_unused]]. Client stubs set no deadlines. A slow PostgreSQL query can block indefinitely, and the client has no timeout protection. This ADR addresses the first concrete step: propagating the gRPC deadline as a PostgreSQL statement_timeout on every transaction.

Context and Problem Statement

Currently, generated gRPC service methods have zero deadline/timeout infrastructure:

  1. Server-side: grpc::ServerContext* is received but [[maybe_unused]]. Even if a client sets a deadline, the handler runs the full query and commits regardless of whether the client has given up.
  2. Client-side: Generated e2e test stubs create bare grpc::ClientContext with infinite timeout (no set_deadline()).
  3. DBA layer: libpqxx has no concept of gRPC. A PostgreSQL query will execute to completion even if the gRPC deadline expired seconds ago.

The consequence: a single slow query (table scan, lock contention, unindexed join) blocks a gRPC thread indefinitely with no backpressure to the client. In production under load, this causes thread pool starvation and cascading failures.

Scope of this ADR: Specifically the mechanism by which a gRPC deadline is converted to a PostgreSQL statement_timeout (or transaction_timeout on PostgreSQL 17+) per unit of work. The broader resilience picture (client-side deadlines, retry coordination, idempotency, circuit breaking) is deferred to a separate ADR.

Decision Drivers

  • gRPC C++ does not automatically propagate deadlines to downstream calls (unlike Go's context.Context)
  • PostgreSQL supports per-transaction SET LOCAL statement_timeout = 'Nms' which aborts queries exceeding the limit and auto-rollbacks the transaction
  • PostgreSQL 17+ introduces transaction_timeout which is a stricter per-transaction wall-clock limit
  • The framework already has muscovite::dba::execute() in transactor.hpp — a natural choke point for deadline injection
  • Generated services currently bypass the transactor (see "Negative Consequences")
  • The statement timeout must be shorter than the gRPC deadline to allow time for serialization, network transit, and error response formatting

Considered Options

  1. Framework utility apply_deadline() + generator injection — Layer 1 provides a apply_deadline(tx, grpc_context, buffer) helper; Layer 0 generators emit calls to it in every RPC handler
  2. Transactor-level integration — Extend muscovite::dba::execute() to accept an optional grpc::ServerContext* and internally set statement_timeout before invoking the user callable
  3. gRPC server interceptor — Use gRPC's ServerInterceptor to extract deadlines globally and store in thread-local, making it transparent to handlers
  4. No framework support — inline in generated code — Generator emits raw SET LOCAL statement_timeout directly in each handler without framework abstraction

Decision Outcome

Chosen option: Option 2 — Transactor-level integration, because:

  • The transactor is already the choke point for all transaction lifecycle (begin, commit, retry, error mapping)
  • It naturally owns the SET LOCAL statement_timeout call — this is a transaction-level concern
  • Generated services should migrate to use the transactor (they currently use raw UnitOfWork) — this gives a forcing function
  • Retry logic in the transactor can respect the remaining deadline: stop retrying if deadline is about to expire
  • The buffer calculation (deadline minus overhead) is centralized in one place, not scattered across generated code

Mechanism

Layer 1: Framework (transactor.hpp)

New overload of execute() accepting a deadline:

namespace muscovite::dba {

/// Execute with gRPC deadline propagation.
/// Converts remaining gRPC deadline to PostgreSQL statement_timeout.
/// Returns DEADLINE_EXCEEDED if deadline already expired before DB call.
/// Buffer (default 500ms) is reserved for serialization + network overhead.
template <typename F>
auto execute(ConnectionPool& pool, F&& fn,
             std::chrono::system_clock::time_point deadline,
             RetryPolicy policy = {},
             std::chrono::milliseconds buffer = std::chrono::milliseconds{500})
    -> std::expected<std::invoke_result_t<F, pqxx::work&>, Error<DbError>>;
}

Inside the implementation:

auto remaining = deadline - std::chrono::system_clock::now();
if (remaining <= std::chrono::milliseconds::zero()) {
    return std::unexpected(Error<DbError>{
        .code = DbError::DeadlineExceeded,
        .message = "gRPC deadline already expired before database call"});
}

auto timeout_ms = std::chrono::duration_cast<std::chrono::milliseconds>(remaining) - buffer;
if (timeout_ms <= std::chrono::milliseconds::zero()) {
    timeout_ms = std::chrono::milliseconds{1}; // PostgreSQL minimum
}

pqxx::work tx{conn};
tx.exec0(std::format("SET LOCAL statement_timeout = '{}'", timeout_ms.count()));
// ... invoke user callable, commit ...

On retry iterations, the remaining time is recalculated. Retries are skipped if remaining time is insufficient.

Layer 0: Generator changes (cpp-service-v2-crud.rkt)

Generated RPC handlers pass context->deadline() to the transactor:

// Generated — before (current):
dba_ctx::EntityCommandService cmd_service{pool_};
auto result = cmd_service.create(entity);

// Generated — after:
auto result = muscovite::dba::execute(pool_, [&](pqxx::work& tx) {
    // ... business logic using tx ...
}, context->deadline());

The [[maybe_unused]] annotation on context is removed.

Layer 0: Error mapping (cpp-service-errors.rkt)

New enum value and gRPC mapping:

DbErrorCode::DeadlineExceeded → grpc::StatusCode::DEADLINE_EXCEEDED

Also fix: SerializationFailure → grpc::StatusCode::ABORTED (current INTERNAL is incorrect per gRPC convention).

Override Hierarchy for Default Deadline

This ADR establishes that the server side honors whatever deadline the client sends. The server's only job is to convert the remaining time to statement_timeout. The DSL-driven override hierarchy for setting client-side deadlines (global default → per-context → per-aggregate → per-RPC) is deferred to the companion ADR on full resilience.

However, a server-side maximum is desirable as a safety net (even if the client sends no deadline or an absurd 10-minute deadline):

Level Configuration Default
Server global muscovite::server::Config::max_statement_timeout 30s
Per-aggregate (future) DSL GRPC_DEADLINE clause on AGGREGATE inherits server

The server picks min(grpc_remaining - buffer, max_statement_timeout).

Positive Consequences

  • A slow query is aborted by PostgreSQL at statement_timeout, rolling back the transaction automatically — no orphaned long-running queries
  • The gRPC thread is freed promptly — prevents thread pool starvation
  • Client gets DEADLINE_EXCEEDED with a meaningful message instead of hanging
  • Retry logic becomes deadline-aware — no wasted retries when the deadline is about to expire
  • Foundation for the full resilience pattern (companion ADR)
  • Transactor adoption in generated services is a forcing function for consistency with framework patterns

Negative Consequences

  • Generated service pattern changes significantly — CommandService.create() must accept a transaction or deadline parameter
  • All existing generated code is affected (breaking regeneration change)
  • DeadlineExceeded must be added to the generated DbErrorCode enum
  • Requires PostgreSQL 9.3+ for SET LOCAL statement_timeout (universally available)
  • Buffer calculation (500ms default) may need tuning per deployment — too small risks premature timeout, too large wastes available time

Prerequisite Work

Before this ADR can be implemented:

  1. Unify error enums: Generated DbErrorCode (9 variants) must align with framework DbError (16 variants). Add DeadlineExceeded to both.
  2. Add missing pqxx exception catches: Generated map_error() in repository_base.hpp is missing broken_connection, check_violation, not_null_violation, deadlock_detected.
  3. Fix SerializationFailure mapping: Change from INTERNAL to ABORTED in cpp-service-errors.rkt.

AI Agent Guidance

When This Decision Applies

  • When generating any gRPC service method that performs database operations
  • When modifying transactor.hpp or execute() function
  • When modifying cpp-service-v2-crud.rkt, cpp-service-errors.rkt, or cpp-service-v2-source.rkt
  • When adding new RPC types (saga, state machine, event sourcing) — they must also propagate deadlines

When This Decision Does NOT Apply

  • Client-side deadline defaults (covered by companion ADR)
  • DSL syntax for deadline configuration (covered by companion ADR)
  • Non-database RPCs (pure computation) — no statement_timeout needed, but IsCancelled() checks may still be appropriate
