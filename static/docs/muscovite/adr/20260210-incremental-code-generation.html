Incremental Code Generation via UUID-Based Change Tracking

Incremental Code Generation via UUID-Based Change Tracking

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-10
  • Tags: generator, performance, incremental, uuid, manifest

Technical Story: Full code generation runs all generators on every invocation (~17 files per entity × 6 layers), even when only one entity changed. With 26 contexts in geosciences (~8,000 lines of DSL), regeneration is wasteful. The existing @uuid identity system, diff-model.rkt semantic diff engine, and scaffolded manifest/dependency-graph infrastructure provide the foundation for selective regeneration.

Context and Problem Statement

Muscovite regenerates the entire output tree on every racket generate.rkt invocation. While write-if-changed (content-hash comparison at write time) prevents unnecessary disk writes and CMake rebuilds, all generators still execute for all contexts. This means:

  • Parse time is unavoidable (all .ddd files must be read)
  • Generator execution time grows linearly with model size
  • Every entity contributes to 17+ files across PostgreSQL, C++ DBA, gRPC, Qt Widgets, and Qt Quick layers
  • Cross-context dependencies (e.g., Basedata.Status references) make naïve file-timestamp approaches unreliable

The @uuid annotation on every DSL element provides stable identity across versions, enabling semantic change detection rather than textual diffing.

Decision Drivers

  • Generation time grows linearly with number of contexts and entities
  • write-if-changed prevents downstream rebuilds but doesn't skip generator execution
  • @uuid on every DSL element already provides stable identity for semantic diff
  • diff-model.rkt (685 lines) already computes 22+ operation types between model versions
  • dslgen/incremental/ and dslgen/manifest/ have scaffolded but disconnected infrastructure
  • Cross-context FK relationships create transitive invalidation chains
  • Sqitch greenfield generator rewrites sqitch.plan completely — must not corrupt deployed plans
  • The migration pipeline (migrate.rkt) is already inherently incremental (appends to plan)

Considered Options

  1. Context-level incremental generation with model serialization
  2. Entity-level incremental generation (fine-grained per-file skip)
  3. File-timestamp-based incremental generation (like Make)
  4. Status quo — rely on write-if-changed only

Decision Outcome

Chosen option: Option 1 — Context-level incremental generation, because it matches the existing generator granularity (most generators operate per-context), leverages existing diff-model.rkt infrastructure, and avoids refactoring generator internals. Entity-level granularity (Option 2) is a future optimization once context-level is proven.

Architecture

Model Serialization

After each generation, the fully parsed ddd-model is serialized to .muscovite-model.sexp alongside the artifact manifest (.artifact-manifest.json). On the next invocation:

  1. Parse current .ddd files → new model
  2. Load .muscovite-model.sexp → old model
  3. Run diff-model → list of semantic operations
  4. Convert operations → changed-uuids set
  5. Resolve affected contexts via dependency closure (BFS on reverse edges)
  6. Skip generators for unaffected contexts
  7. Serialize new model + update manifest

If .muscovite-model.sexp is missing or corrupt, fall back to full generation.

Change-Set Translation

diff-model.rkt returns SQL-oriented migration operations (ALTER TABLE, etc.). A new dslgen/incremental/change-set.rkt module translates these into a simple changed-uuids set:

Migration Operation Extracted UUIDs
op-add-entity entity UUID + context UUID
op-remove-entity entity UUID + context UUID
op-rename-entity entity UUID + context UUID
op-add-field field UUID + entity UUID + context UUID
op-remove-field field UUID + entity UUID + context UUID
op-change-field-type field UUID + entity UUID + context UUID
op-add-relationship relationship UUID + both entity UUIDs + both context UUIDs
op-remove-relationship relationship UUID + both entity UUIDs + both context UUIDs
op-add-index entity UUID + context UUID
op-add-invariant entity UUID + context UUID
Any aggregate-level op aggregate UUID + root entity UUID + context UUID

Dependency Graph (Cross-Context Invalidation)

The dependency graph in dslgen/incremental/dependency-graph.rkt uses BFS transitive closure on reverse edges. Edges include:

  • Entity → field types: If a field references a strong type from another context, changing that type invalidates the entity
  • Entity → relationship targets: If entity A has RELATIONSHIPS { x -> B.Entity REQUIRED ONE }, changing B.Entity invalidates A's context
  • Aggregate → contained entities: Changing any child entity invalidates the aggregate root's context
  • Value object → using entities: Changing a shared/inline value object invalidates all entities that USES or HAS it

Cross-context FK references (e.g., Basedata.ConditionClinicalStatus referenced by Condition in PatientContext) create hard dependency edges. A change to any Basedata entity propagates to every context that references it.

Sqitch Lifecycle Guard

The Sqitch greenfield generator (sqitch-unified.rkt) and the migration pipeline (migrate.rkt) are two completely separate code paths designed for different lifecycle stages:

Aspect Greenfield (sqitch-unified.rkt) Migration (migrate.rkt)
Plan management Full rewrite (preserves timestamps) Append only
SQL style CREATE TABLE ALTER TABLE
Target Test/ephemeral databases Production databases
Safety write-if-changed prevents unnecessary writes Semantic diff + safety classification

Rules:

  • Greenfield generator: always runs during generation (fast, write-if-changed prevents disk writes, only used for --fresh/test databases)
  • Migration pipeline: never invoked by generate.rkt — separate CLI tool (migrate.rkt) for production schema evolution
  • A new --sqitch-mode greenfield|migration flag (default: greenfield) guards against accidental plan corruption. When migration, the greenfield Sqitch generator is skipped entirely.
  • The greenfield generator must never be run against a Sqitch project whose plan has been deployed to a production database. The full-rewrite approach would produce a plan that conflicts with Sqitch's internal registry of already-deployed changes.

Positive Consequences

  • Generation time scales with change size, not model size
  • No behavioral change for users who don't use @uuid (falls back to full generation)
  • write-if-changed remains as safety net even when incremental is active
  • Sqitch lifecycle explicitly guarded against corruption
  • Existing diff-model.rkt, dependency-graph, and manifest infrastructure gets wired into production
  • Foundation for entity-level granularity in future versions

Negative Consequences

  • Model serialization adds disk I/O (~100KB for large models)
  • First run after any infrastructure change requires --full
  • Cross-context dependency edges may cause over-invalidation (conservative but safe)
  • Two Sqitch code paths remain separate (no shared SQL generation logic)

Pros and Cons of the Options

Option 1: Context-Level Incremental Generation (Chosen)

  • ✅ Matches existing generator granularity (per-context invocation)
  • ✅ Leverages existing diff-model.rkt (685 lines) and dependency-graph (208 lines)
  • ✅ No refactoring of generator internals required
  • ✅ Conservative — over-invalidation is safe (regenerates too much, never too little)
  • ❌ Coarse granularity — changing one entity regenerates all 17+ files for every entity in that context
  • ❌ Basedata changes invalidate many/all contexts (common cross-cutting dependency)

Option 2: Entity-Level Incremental Generation

  • ✅ Maximum skip efficiency — only affected entity files regenerated
  • ✅ Basedata changes only propagate to specific referencing entities
  • ❌ Requires refactoring generators to accept single-entity invocation (major effort)
  • ❌ Proto files contain all entities per context — can't skip individual entities
  • ❌ sqitch.plan is per-project — entity-level skip doesn't help
  • ❌ Higher risk of missing transitive dependencies

Option 3: File-Timestamp-Based (Make-style)

  • ✅ Simple implementation — compare .ddd mtime vs output mtime
  • ❌ Doesn't handle cross-context dependencies (Basedata change doesn't touch other .ddd files)
  • ❌ Doesn't handle semantic renames (same UUID, different name = change, but file content differs)
  • ❌ Git operations (checkout, rebase) change mtimes unpredictably
  • ❌ Ignores the @uuid infrastructure entirely

Option 4: Status Quo (write-if-changed only)

  • ✅ Zero implementation effort
  • ✅ Already prevents unnecessary CMake rebuilds
  • ❌ Generator execution time still grows with model size
  • ❌ All generators run for all contexts on every invocation

AI Agent Guidance

When This Decision Applies

  • Any work on generation performance or skipping unchanged artifacts
  • Adding new generators (must register output paths in element-mapping)
  • Modifying generate.rkt pipeline flow
  • Working with dslgen/incremental/ or dslgen/manifest/ modules
  • Discussions about Sqitch plan management and production deployment

When This Decision Does NOT Apply

  • Parser changes (parsing always runs on all files)
  • Migration pipeline (migrate.rkt) — already incremental by design
  • Runtime C++ framework changes (Layer 1)
  • Manual testing with --fresh flag (always full regeneration)

Enforcement

  • --sqitch-mode flag guards against greenfield Sqitch on deployed databases
  • write-if-changed remains as safety net regardless of incremental mode
  • Missing .muscovite-model.sexp triggers automatic --full fallback
  • Cross-context dependency edges ensure conservative invalidation

Key Files

  • dslgen/generate.rkt — Main pipeline, integration point
  • dslgen/incremental/dependency-graph.rkt — BFS transitive closure on reverse edges
  • dslgen/incremental/incremental-gen.rkt — Orchestrator with stubs to wire
  • dslgen/manifest/artifact-manifest.rkt — Artifact tracking with content hashes
  • dslgen/manifest/element-mapping.rkt — UUID → file path mappings (STALE — must update)
  • dslgen/migration/diff-model.rkt — Semantic diff engine (22+ operation types)
