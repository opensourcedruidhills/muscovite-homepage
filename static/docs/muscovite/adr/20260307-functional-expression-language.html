Functional Expression Language for Behavioral Sovereignty

Functional Expression Language for Behavioral Sovereignty

  • Status: accepted
  • Date: 2026-03-07
  • Deciders: Johannes Lochmann
  • Tags: expression-ast, dsl, behavioral-sovereignty

Context

Muscovite's expression AST (9 node types in ddd-types.rkt:804-852) is a predicate language — it answers yes/no questions via comparisons, boolean logic, and null checks. Behavioral Sovereignty requires a functional expression language that computes values, transforms collections, and binds intermediate results.

Use cases like booking price calculation with dynamic packaging, amendments, and cancellations require:

  • Collection operations: containment, emptiness, intersection, subset
  • Conditional expressions: branching without side effects
  • Variable binding: intermediate results in multi-step calculations
  • List comprehensions: map, filter, reduce over typed collections
  • Decimal arithmetic with rounding (already partially supported)

Google CEL was evaluated and rejected: it introduces opaque strings that bypass the PEG grammar (violating ADR 20260211), prevents Racket generators from walking the AST for documentation and test generation, and adds a runtime C++ dependency.

Decision

Extend the existing expression AST in three tiers, each independently useful:

Tier 1 — Collection Predicates (extends predicate language):

  • NOT IN, IS EMPTY, IS NOT EMPTY
  • INTERSECTS, SUBSET OF, CONTAINS ALL
  • New IR nodes: not-in-expr, empty-check-expr, set-op-expr

Tier 2 — Computation Language (adds value-producing expressions):

  • IF cond THEN expr ELSE expr → if-expr(cond, then-branch, else-branch)
  • LET name = expr IN body → let-expr(bindings, body)
  • MATCH field { CASE val => expr, ... } → match-case-expr(field, cases, default)
  • These are expressions (produce values), not statements

Tier 3 — List Comprehensions (functional collection processing):

  • MAP(list, var -> expr) → map-expr(list, var, body)
  • FILTER(list, var -> expr) → filter-expr(list, var, body)
  • REDUCE(list, init, acc var -> expr) → reduce-expr(list, init, acc, var, body)
  • SUM(list), COUNT(list), MIN(list), MAX(list) → built-in aggregations via call-expr
  • Lambda expressions are single-expression, no closures over mutable state

Non-Turing completeness guarantee: No general recursion, no mutable state, no unbounded loops. MAP/FILTER/REDUCE are bounded by collection length. This is the same computational model as Excel formulas or SQL SELECT expressions.

Compilation targets per tier:

Target Tier 1 Tier 2 Tier 3
PostgreSQL NOT IN, array_length = 0 CASE WHEN SELECT ... FROM unnest()
C++ std::ranges::contains, .empty() Ternary / if-else std::ranges::transform, fold_left
Docs/Mermaid Predicate annotations Decision diamond nodes Data flow boxes
Property tests Predicate oracle Branch coverage Collection boundary

Consequences

  • Expression grammar (muscovite.peg §37) grows from 30 to ~80 rules
  • expression->sql and expression->cpp emitters must handle all new node types
  • IR struct count in expression AST grows from 9 to ~18 nodes
  • Every generator that walks expressions must be updated per tier
  • Tier 3 introduces variable scoping — parser must track lambda parameter scope
  • CEL is explicitly rejected; no external expression language dependency
