Mandatory Multi-Layer Test Gates

Mandatory Multi-Layer Test Gates

  • Status: accepted
  • Date: 2026-03-07
  • ADR: 20260307-mandatory-multi-layer-test-gates
  • Supersedes: None
  • Context: Governance audit of v1.0.0–v1.54.0 revealed 73% of shipped code-generating FTRs had test coverage gaps. 34 FTRs shipped with zero tests. 52% of SQL-generating FTRs never verified against PostgreSQL. C++ build gate in release.sh was non-blocking (warn). ctest was entirely absent from release ceremony. DB deploy was never tested.

Decision

1. Every Code-Generating FTR MUST Specify Tests Per Affected Layer

Layer Gate Required When Command
Racket (Layer 0) rackunit Always (parser/IR/generator change) racket tests/run-all-tests.rkt
C++ build (Layer 1+2) cmake build FTR generates or modifies C++/Proto/CMake cmake --build --preset debug
C++ tests (Layer 1+2) ctest FTR generates or modifies C++ ctest --preset unit --output-on-failure
DB deploy (Layer 2) Sqitch deploy FTR generates or modifies SQL cmake --build --preset debug --target db-fresh-fhir
clang-tidy Static analysis FTR generates or modifies C++ cmake --build --preset clang-tidy

FTR acceptance criteria MUST include the specific test commands for every applicable layer. "Racket tests pass" is insufficient — the exact test file or suite must be named.

2. All Gates Are BLOCKING in Both Merge and Release Scripts

agent-pr.sh (merge to develop):

  • Racket parser + generator + integration tests → FAIL on error
  • C++ build → FAIL on error (was: not present for all)
  • ctest --preset unit → FAIL on error
  • clang-tidy → FAIL on error

release.sh (tag and merge to main):

  • All of the above, PLUS:
  • Full Racket test suite (run-all-tests.rkt) → FAIL on error
  • C++ build → FAIL on error (was: warn, non-blocking)
  • ctest --preset debug → FAIL on error (was: absent)
  • DB fresh deploy → FAIL on error (was: absent)
  • clang-tidy → FAIL on error (already was)

No gate may use warn. Every gate uses fail.

3. FTR Template Enforces Layer Checklist

The FTR template (workflow/templates/feature.md) includes a mandatory "Test Gates" section with per-layer checkboxes. The accepting agent must fill in which gates apply and add the specific commands to Acceptance Criteria.

4. Retroactive Stabilization Required

All shipped versions (v1.0.0–v1.54.0) have test debt. A stabilization phase must:

  • Verify all generated C++ compiles (likely already true)
  • Verify all generated SQL deploys (likely already true)
  • Add missing regression tests for generator targets that lack them
  • Add golden-file snapshot tests for generator output stability

Consequences

  • Positive: No more silent test gaps. Every FTR is provably tested at every layer it touches.
  • Positive: Release ceremony catches regressions that were previously invisible.
  • Positive: DB deploy failures caught before shipping — prevents broken Sqitch plans reaching production.
  • Negative: Release ceremony takes longer (adds ctest + DB deploy + full Racket suite).
  • Negative: Stabilization iterations delay new feature development.
  • Risk: Existing tests may have false passes (test exists but doesn't actually verify the right thing). Stabilization must verify test quality, not just test existence.
