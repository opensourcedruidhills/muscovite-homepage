ADR-20260206: QML ViewModel Binding Pattern (All Fields as Q_PROPERTY)

ADR-20260206: QML ViewModel Binding Pattern (All Fields as Q_PROPERTY)

Status: Accepted
Date: 2026-02-06
Deciders: Jolo (Staff Engineer)

Context

Qt Quick (QML) requires C++ objects to expose state via Q_PROPERTY for binding to work. When an entity has 20+ fields (common in geosciences domain), the question is: how many become properties?

Design Space:

  • Option A (Selective): Only fields marked with @property DSL hint become Q_PROPERTY (less boilerplate, but requires DSL syntax decision)
  • Option B (All Fields): Every entity field → Q_PROPERTY with NOTIFY signal (consistent, verbose ViewModel)

This ADR codifies Option B for v0.10.0 as the SoC validation baseline. Future v1.x ADR will refine to selective approach with visibility/read-only hints.

Problem Statement

Current (Widgets-only):

  • No Q_PROPERTY generation; dialogs are pure C++
  • UI state is implicit in QWidget field values

Multi-Frontend Challenge:

  • QML binds to C++ properties: TextField { text: patientViewModel.name }
  • Every bound field needs a Q_PROPERTY
  • Entity might have: id, name, mrn, gender_id, birth_date, deceased, created_at, updated_at, ... (8-40 fields)
  • Do we generate Q_PROPERTY for all, or be selective?

Simplicity vs. Completeness Trade-Off:

  • All Fields: Boilerplate, but:
    • No DSL design needed (v0.10.0 focus)
    • Guarantees all data is bindable
    • Future v1.x ADR can hide/mark read-only without regeneration
  • Selective: Cleaner, but:
    • Requires DSL extension (@property or @ui hints)
    • Missed fields become tedious to retrofit
    • Adds complexity to parser/generator

Decision

ViewModel Pattern: All Entity Fields as Q_PROPERTY

For v0.10.0, every entity field generates a Q_PROPERTY with NOTIFY signal.

Per-Entity ViewModel C++ Class:

// Generated: PatientViewModel.hpp
class PatientViewModel : public QObject {
    Q_OBJECT
    Q_PROPERTY(QUuid id READ id CONSTANT)
    Q_PROPERTY(QString mrn READ mrn WRITE setMrn NOTIFY mrnChanged)
    Q_PROPERTY(QString givenName READ givenName WRITE setGivenName NOTIFY givenNameChanged)
    Q_PROPERTY(QString familyName READ familyName WRITE setFamilyName NOTIFY familyNameChanged)
    Q_PROPERTY(QUuid genderId READ genderId WRITE setGenderId NOTIFY genderIdChanged)
    Q_PROPERTY(QDate birthDate READ birthDate WRITE setBirthDate NOTIFY birthDateChanged)
    Q_PROPERTY(bool deceased READ deceased WRITE setDeceased NOTIFY deceasedChanged)
    Q_PROPERTY(QDateTime createdAt READ createdAt CONSTANT)
    Q_PROPERTY(QDateTime updatedAt READ updatedAt CONSTANT)
    // ... 12 more fields for a typical domain entity
    
    Q_PROPERTY(QString validationError READ validationError NOTIFY validationErrorChanged)
    Q_PROPERTY(bool isValid READ isValid NOTIFY isValidChanged)

public:
    // Getters
    QUuid id() const { return data_.id; }
    QString mrn() const { return data_.mrn; }
    // ... etc
    
    // Setters (only for writable fields)
    void setMrn(const QString& value) {
        if (data_.mrn != value) {
            data_.mrn = value;
            emit mrnChanged();
            validateField("mrn");
        }
    }
    
    // Validation delegation
    QString validationError() const { return validation_error_; }
    bool isValid() const { return validation_error_.isEmpty(); }
    
    // gRPC service delegation
    Q_INVOKABLE void save();  // Calls PatientCommandService::UpdatePatient

signals:
    void mrnChanged();
    void givenNameChanged();
    void familyNameChanged();
    void genderIdChanged();
    void birthDateChanged();
    void deceasedChanged();
    void validationErrorChanged();
    void isValidChanged();

private:
    void validateField(const std::string& field);
    
    Patient data_;  // Entity struct from DBA layer
    std::shared_ptr<PatientValidator> validator_;  // Shared validator
    std::shared_ptr<PatientCommandService> service_;  // Shared gRPC service
    QString validation_error_;
};

Key Principles

  1. No Duplication of Business Logic

    • Validation delegates to PatientValidator (from DBA layer) ✅ Shared
    • gRPC calls delegate to PatientCommandService (from gRPC layer) ✅ Shared
    • Entity struct is from DBA layer ✅ Shared

  2. CONSTANT vs. Writable

    • id, created_at, updated_at → CONSTANT (set at construction, never change)
    • Mutable fields → READ/WRITE with NOTIFY signal

  3. Validation Synchronization

    • Property setter calls validateField() → triggers validation error property update
    • QML binds to validationError property: Label { color: "red"; text: viewModel.validationError }

  4. gRPC Service Integration

    • Q_INVOKABLE void save() calls CommandService::UpdatePatient()
    • Async handling deferred to v1.0.0 (ADR 20260206-async-viewmodel-services)
    • For now: blocking calls acceptable (UI will freeze, but iteration speed still faster than C++ recompilation)

Code Generation Template

Racket generator produces ViewModel class from entity definition:

;; dslgen/generators/qt/qml.rkt
(define (generate-viewmodel entity output-dir)
  (define hpp-content
    (string-append
      "#include <QObject>\n"
      "#include <QString>\n"
      "// ... includes from DBA, gRPC layers ...\n\n"
      "class " (entity-name entity) "ViewModel : public QObject {\n"
      "    Q_OBJECT\n"
      ;; Generate Q_PROPERTY for each field
      (string-join (map (lambda (field) 
        (format "    Q_PROPERTY(~a ~a READ ~a ~a)\n"
          (field-qml-type field)
          (field-name field)
          (field-getter-name field)
          (if (field-writable? field)
            (format "WRITE ~a NOTIFY ~aChanged" 
              (field-setter-name field)
              (field-name field))
            "CONSTANT")))
        (entity-fields entity)))
      "// ... getters, setters, signals, private methods ...\n"
      "};\n"))
  ;; Write to file
  )

Grpc Service Bindings in ViewModel

Q_INVOKABLE void PatientViewModel::save() {
    // Validation already checked (NOTIFY on validationError)
    if (!isValid()) return;
    
    // Call shared service (from gRPC layer)
    try {
        auto response = service_->updatePatient(data_);
        // Update local data_ from response
        emit idChanged();  // Re-emit to update UI
    } catch (const std::exception& e) {
        validation_error_ = QString::fromStdString(e.what());
        emit validationErrorChanged();
    }
}

Consequences

Positive

  1. Simplicity: No DSL extensions needed; all fields are properties by default
  2. Completeness: Every data field is bindable to QML, no surprises at iteration time
  3. SoC Proven: ViewModel is pure binding bridge; validation and services come from shared layers
  4. Future-Proof: v1.x ADR can add @readonly, @hidden hints without breaking existing QML
  5. Generator Clarity: Single code path per entity (not multiple branches for "is this property?")

Negative

  1. Boilerplate: ViewModel can be large (50-200 lines per entity for 20+ fields)
  2. Memory: Each ViewModel instance holds full entity struct + property cache
  3. Signal Spam: Many NOTIFY signals for large entities; could impact performance if all observed simultaneously
  4. Maintenance: Future changes to entity require ViewModel regeneration

Neutral

  1. QML Binding Performance: No difference between 5 properties and 25 properties (Qt engine handles)
  2. Testing: ViewModel unit tests become comprehensive (all properties must be validated)

Implementation Plan

v0.10.0: All Fields as Q_PROPERTY

Generator tasks:

  1. dslgen/generators/qt/qml.rkt: Generate .hpp + .cpp for ViewModel classes
  2. dslgen/generators/qt/common.rkt: ViewModel base class in framework
  3. Framework: Add ViewModel base interface with property binding hooks
  4. Generate Q_INVOKABLE methods for commands (create, update, delete)

v1.x: Selective Properties + Visibility Hints

Future DSL extension (new ADR):

ENTITY Patient (
    id UUID @readonly @hidden,
    mrn TEXT @property,
    givenName TEXT @property,
    status TEXT @readonly,
    notes TEXT @hidden   -- Never shown in UI
)

Mapping:

  • @property → explicit Q_PROPERTY
  • @readonly → Q_PROPERTY READ only (no WRITE/NOTIFY)
  • @hidden → skip Q_PROPERTY entirely

(Current: all fields treated as @property implicitly)

AI Agent Guidance

When is this decision applicable?

  • Any QML frontend generation
  • Any ViewModel generation
  • Any C++/QML binding question

How it's enforced:

  • Racket generator checks: every entity field → ViewModel Q_PROPERTY
  • Tests verify: all properties are readable and (if writable) settable
  • QML bindings will fail at runtime if property missing (Qt error messaging)

Key files affected:

  • dslgen/generators/qt/qml.rkt → new ViewModel generation
  • framework/qt/include/muscovite/qt/viewmodel.hpp → base ViewModel interface
  • framework/qt/src/viewmodel.cpp → ViewModel implementations
  • dslgen/tests/test-qt-qml-generation.rkt → new tests
