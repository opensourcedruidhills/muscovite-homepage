Semantic Model Delta Computation

Semantic Model Delta Computation

Technical Story: Compute structural differences between two DDD model versions to generate precise migration operations.

Context and Problem Statement

Muscovite currently regenerates all artifacts from scratch on every run. This works for greenfield development but fails for schema evolution:

  1. Full regeneration destroys data: DROP TABLE + CREATE TABLE loses production data
  2. No change detection: Cannot determine what actually changed between versions
  3. No rename support: Entity rename looks like DELETE + ADD
  4. No safe/unsafe classification: All changes treated equally

The migration system needs a semantic diff engine that:

  • Compares two IR graphs by UUID identity (not name)
  • Produces a list of atomic change operations
  • Classifies each operation as safe, unsafe, or destructive

Decision Drivers

  • Detect adds, deletes, renames, moves, type changes, cardinality changes
  • Generate a lossless Migration IR that can be reviewed before execution
  • Distinguish safe vs unsafe operations for policy enforcement
  • Support complex cross-domain references
  • Deterministic output (same inputs → same diff)

Considered Options

  • Diff raw SQL (post-generation): Rejected - loses semantic intent, hard to reverse
  • Diff DSL text: Rejected - whitespace-sensitive, doesn't handle renames
  • Diff database schema (introspection): Rejected - only works after deployment
  • Diff IR using UUID identity: Accepted - semantic comparison, rename-aware

Decision Outcome

Chosen option: "Diff IR using UUID identity", implemented as a pure function that takes two model versions and produces migration operations.

Positive Consequences

  • Safe rename detection (same UUID, different name → RENAME op)
  • Deterministic migration generation
  • Supports complex cross-context FK changes
  • Can be validated before execution
  • Enables review workflow for destructive changes

Negative Consequences

  • Cannot infer human intent for ambiguous changes
  • Requires strict UUID enforcement from ADR 0002
  • Complex dependency tracking for cross-entity changes
  • May produce suboptimal migrations for radical restructures

Implementation

Phase 1: Core Diff Engine

Create src/dslgen/migration/diff-model.rkt:

#lang racket/base

(provide diff-model)

;; Main diff function
;; Returns list of migration-operation structs
(define (diff-model ir-old ir-new)
  (append
   (diff-contexts ir-old ir-new)
   (diff-aggregates ir-old ir-new)
   (diff-entities ir-old ir-new)
   (diff-fields ir-old ir-new)
   (diff-relationships ir-old ir-new)
   (diff-indexes ir-old ir-new)))

;; Compare entities by UUID
(define (diff-entities ir-old ir-new)
  (define old-entities (model->entity-map ir-old))  ;; UUID → entity
  (define new-entities (model->entity-map ir-new))
  
  (define operations '())
  
  ;; Find additions (in new, not in old)
  (for ([(uuid ent) (in-hash new-entities)])
    (unless (hash-has-key? old-entities uuid)
      (set! operations (cons (add-entity-op uuid ent) operations))))
  
  ;; Find deletions (in old, not in new)
  (for ([(uuid ent) (in-hash old-entities)])
    (unless (hash-has-key? new-entities uuid)
      (set! operations (cons (remove-entity-op uuid ent) operations))))
  
  ;; Find modifications (in both, but different)
  (for ([(uuid new-ent) (in-hash new-entities)])
    (when (hash-has-key? old-entities uuid)
      (define old-ent (hash-ref old-entities uuid))
      (set! operations 
            (append (diff-single-entity old-ent new-ent) operations))))
  
  operations)

Phase 2: Change Detection Functions

;; Detect specific changes within an entity
(define (diff-single-entity old-ent new-ent)
  (define operations '())
  
  ;; Name change = RENAME
  (unless (equal? (entity-name old-ent) (entity-name new-ent))
    (set! operations 
          (cons (rename-entity-op (entity-uuid old-ent)
                                  (entity-name old-ent)
                                  (entity-name new-ent))
                operations)))
  
  ;; Parent change = MOVE (in inheritance hierarchy)
  (unless (equal? (entity-parent-name old-ent) (entity-parent-name new-ent))
    (set! operations
          (cons (change-parent-op (entity-uuid old-ent)
                                  (entity-parent-name old-ent)
                                  (entity-parent-name new-ent))
                operations)))
  
  ;; Field-level changes
  (set! operations 
        (append (diff-entity-fields old-ent new-ent) operations))
  
  operations)

;; Detect field changes
(define (diff-entity-fields old-ent new-ent)
  (define old-fields (entity->field-map old-ent))
  (define new-fields (entity->field-map new-ent))
  
  (define operations '())
  
  ;; Added fields
  (for ([(uuid fld) (in-hash new-fields)])
    (unless (hash-has-key? old-fields uuid)
      (set! operations (cons (add-field-op (entity-uuid new-ent) uuid fld) operations))))
  
  ;; Removed fields
  (for ([(uuid fld) (in-hash old-fields)])
    (unless (hash-has-key? new-fields uuid)
      (set! operations (cons (remove-field-op (entity-uuid old-ent) uuid fld) operations))))
  
  ;; Modified fields
  (for ([(uuid new-fld) (in-hash new-fields)])
    (when (hash-has-key? old-fields uuid)
      (define old-fld (hash-ref old-fields uuid))
      (set! operations (append (diff-single-field old-fld new-fld) operations))))
  
  operations)

Phase 3: Field-Level Change Detection

;; Detect changes to a single field
(define (diff-single-field old-fld new-fld)
  (define uuid (field-uuid old-fld))
  (define operations '())
  
  ;; Name change
  (unless (equal? (field-name old-fld) (field-name new-fld))
    (set! operations
          (cons (rename-field-op uuid (field-name old-fld) (field-name new-fld))
                operations)))
  
  ;; Type change
  (unless (equal? (field-type old-fld) (field-type new-fld))
    (set! operations
          (cons (change-type-op uuid 
                                (field-type old-fld) 
                                (field-type new-fld)
                                (classify-type-change (field-type old-fld) (field-type new-fld)))
                operations)))
  
  ;; Nullability change
  (unless (equal? (field-nullable? old-fld) (field-nullable? new-fld))
    (set! operations
          (cons (change-nullable-op uuid
                                    (field-nullable? old-fld)
                                    (field-nullable? new-fld))
                operations)))
  
  operations)

;; Classify type changes as safe/unsafe
(define (classify-type-change old-type new-type)
  (cond
    ;; Widening is safe
    [(and (string-type? old-type) (string-type? new-type)) 'safe]
    [(and (integer-type? old-type) (bigint-type? new-type)) 'safe]
    ;; Narrowing is unsafe
    [(and (text-type? old-type) (varchar-type? new-type)) 'unsafe]
    [(and (bigint-type? old-type) (integer-type? new-type)) 'unsafe]
    ;; Type family change is destructive
    [(not (same-type-family? old-type new-type)) 'destructive]
    [else 'review]))

Operation Types

Operation Description Safety
add-context New bounded context Safe
remove-context Delete context (cascade) Destructive
add-entity New table Safe
remove-entity Drop table Destructive
rename-entity Rename table Safe
move-entity Change aggregate/context Review
add-field Add nullable column Safe
add-field Add NOT NULL column Unsafe (needs default)
remove-field Drop column Destructive
rename-field Rename column Safe
change-type Alter column type Varies
change-nullable NOT NULL ↔︎ NULL Varies
add-relationship Add FK constraint Safe
remove-relationship Drop FK Safe
add-index Create index Safe
remove-index Drop index Safe

AI Agent Guidance

When This Decision Applies

  • When implementing model version comparison
  • When generating migration scripts
  • When detecting schema changes for CI/CD

When This Decision Does NOT Apply

  • Runtime data comparisons
  • Generated code diffs (use git for that)
  • SQL schema introspection (that's verification, not diff)

Enforcement

  • Diff engine MUST use UUID for identity matching
  • Diff engine MUST classify all operations by safety
  • Diff engine MUST be a pure function (no side effects)
  • Output MUST be deterministic (same inputs → same output)

Acceptance Criteria

  1. (diff-model ir-v1 ir-v2) returns list of operations
  2. Entity rename (same UUID, different name) produces rename-entity-op
  3. New entity (UUID not in old) produces add-entity-op
  4. Removed entity (UUID not in new) produces remove-entity-op
  5. Type widening classified as 'safe
  6. Type narrowing classified as 'unsafe
  7. Cross-type-family change classified as 'destructive
  8. Diff of identical models returns empty list
