Typed Proto Quantity Messages

Typed Proto Quantity Messages

  • Status: proposed
  • Deciders: Johannes Lochmann
  • Date: 2026-04-04
  • Tags: units, proto, grpc, fhir, type-safety

Technical Story: IDEA-134 — DSL-Native Units of Measure, Decision 2

Context and Problem Statement

When a DSL entity declares systolic QUANTITY<Pressure, mmHg>, the protobuf generator must decide how to represent that field in the .proto file. The two extremes are:

  1. Typed messages — one proto message per dimension (Pressure, Length, Mass), each with a dimension-specific unit enum and a ucum_code string field
  2. Generic Quantity — a single Quantity { double value; string unit; } message reused for all fields (FHIR-style)

This decision affects compile-time safety, proto surface area, FHIR interoperability, and the complexity of generated C++ converters.

Decision Drivers

  • Type safety: prevent accidentally passing a Length where a Pressure is expected
  • FHIR interoperability: healthcare systems exchange quantities as UCUM-coded strings
  • Proto surface area: number of generated .proto files and messages
  • Converter complexity: C++ from_proto / to_proto code per dimension
  • IDE support: proto enums provide auto-completion for unit selection
  • Backward compatibility: adding new units to an enum is backward-compatible in proto3

Considered Options

  1. Typed messages with ucum_code field — Per-dimension messages with enum + UCUM string
  2. Generic FHIR-style Quantity — Single Quantity { double value; string unit; }
  3. Typed wrapper around generic — message Pressure { Quantity quantity = 1; }

Decision Outcome

Chosen option: "Typed messages with ucum_code field" (option 1), because it provides compile-time type safety through per-dimension messages and enums, while the ucum_code string field preserves FHIR wire compatibility.

The key insight: enums give compile-time safety, the UCUM field gives interop — best of both worlds.

Each DIMENSION in the DSL generates:

// muscovite/quantities/pressure.proto
enum PressureUnit {
    PRESSURE_UNIT_UNSPECIFIED = 0;
    PRESSURE_UNIT_PASCAL = 1;
    PRESSURE_UNIT_KILOPASCAL = 2;
    PRESSURE_UNIT_MMHG = 6;
    // ... auto-generated from QUDT standard library
}

message Pressure {
    double value = 1;
    PressureUnit unit = 2;
    string ucum_code = 3;  // FHIR wire encoding: "mm[Hg]"
}

The ucum_code field is:

  • Set by the C++ converter when serializing to proto (from the unit enum's UCUM mapping)
  • Used by FHIR-facing services as the canonical wire encoding
  • Informational when the enum is present (enum is authoritative, UCUM is for interop)

Positive Consequences

  • Compile-time type safety: gRPC service accepting Pressure cannot receive Length
  • IDE auto-completion: proto enums list available units per dimension
  • FHIR compatibility: ucum_code field carries UCUM encoding for healthcare interop
  • Backward-compatible evolution: new enum values can be added without breaking clients
  • Generated converters are per-dimension, making switch statements exhaustive

Negative Consequences

  • Larger proto surface: one .proto file per dimension (mitigated: only dimensions used in the project are generated)
  • More generated code: per-dimension converter functions in C++
  • Proto enum names are verbose (PRESSURE_UNIT_KILOPASCAL), required by proto3 naming

Pros and Cons of the Options

Option 1: Typed Messages with ucum_code Field

Per-dimension message with unit enum and UCUM string for FHIR interop.

  • Good, because compile-time type safety at the proto/gRPC level
  • Good, because enum provides IDE auto-completion for unit selection
  • Good, because ucum_code preserves FHIR wire compatibility
  • Good, because switch on enum is exhaustive — compiler catches missing cases
  • Good, because backward-compatible: adding enum values is a non-breaking change
  • Bad, because larger proto surface (one file per dimension)
  • Bad, because verbose enum names required by proto3 scoping rules
  • Bad, because dual representation (enum + UCUM string) requires consistency enforcement

Option 2: Generic FHIR-Style Quantity

Single reusable message: Quantity { double value; string unit; }.

  • Good, because minimal proto surface (one message for all quantities)
  • Good, because directly FHIR-compatible wire format
  • Good, because simpler generated code (one converter handles all)
  • Bad, because stringly-typed — no compile-time safety
  • Bad, because typos in unit strings ("mmhg" vs "mm[Hg]") are runtime errors
  • Bad, because no IDE auto-completion for valid units
  • Bad, because dimension violations (Length value in Pressure field) undetectable at compile time

Option 3: Typed Wrapper Around Generic

message Pressure { Quantity quantity = 1; } — type name gives safety, generic inside.

  • Good, because type name prevents dimension confusion at the service boundary
  • Good, because reuses single Quantity definition
  • Bad, because extra nesting level in proto messages
  • Bad, because still stringly-typed inside the wrapper (unit is a string)
  • Bad, because no enum auto-completion for units
  • Bad, because awkward access pattern: request.systolic().quantity().value()

AI Agent Guidance

When This Decision Applies

  • When generating .proto files for entities that contain QUANTITY fields
  • When generating C++ proto converter functions (from_proto / to_proto) for quantities
  • When designing gRPC service interfaces that accept or return quantity values
  • When mapping DSL QUANTITY<Dimension, unit> fields to protobuf representations

When This Decision Does NOT Apply

  • When working with non-quantity fields (TEXT, DECIMAL, DATE, etc.)
  • When generating PostgreSQL DDL (see PostgreSQL storage strategy)
  • When generating QML display code (QML uses string-based unit display)
  • When working on the QUDT-to-DDD generator (that produces the standard library)

Enforcement

  • Proto generator (dslgen/generators/proto/) emits per-dimension messages with enums
  • C++ converter generator emits from_proto/to_proto with exhaustive switch on enum
  • clang-tidy: -Wswitch ensures all enum cases handled in converters
  • Proto lint: ucum_code field must be present on every quantity message
