gRPC as Primary Transport for Non-C++ Clients

gRPC as Primary Transport for Non-C++ Clients

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: grpc, transport, integration, architecture, frontend

Context and Problem Statement

Muscovite's Qt frontends (Widgets, QML) link directly against the C++ DBA layer — they share the same process and call repositories in-process. This works because Qt is C++.

Non-C++ frontends (React/TypeScript, Kotlin, Swift, etc.) cannot link C++ directly. They need a network transport to the backend. The C++ gRPC server already exists and exposes all aggregate operations. The question is: should non-C++ clients use gRPC directly, or should we introduce additional transport layers?

Decision Drivers

  • gRPC server already exists and is generated from .proto contracts
  • .proto files are the canonical service definition (generated from IR)
  • Streaming support needed for domain events (server → client push)
  • Minimize infrastructure (no additional gateways if avoidable)
  • Strong typing end-to-end (proto → generated client stubs)
  • Multi-platform: mobile, desktop, web all have different transport constraints

Considered Options

Option 1: Direct gRPC (native HTTP/2)

Client uses official gRPC library for its language (e.g., grpc-web, grpc-kotlin).

Non-C++ Client (mobile/desktop)
        ↓ gRPC (HTTP/2)
C++ gRPC Server
        ↓
PostgreSQL

Option 2: gRPC-Web via Proxy

Browser-based clients use gRPC-Web protocol through a proxy (Envoy or grpcwebproxy) that translates to native gRPC.

Browser Client
     ↓ gRPC-Web (HTTP/1.1 or HTTP/2)
Proxy (Envoy / grpcwebproxy)
     ↓ gRPC (HTTP/2)
C++ gRPC Server

Option 3: REST/JSON Gateway

Introduce a REST gateway (e.g., grpc-gateway) that translates HTTP/JSON to gRPC.

Client (HTTP/JSON)
        ↓ REST
REST Gateway
        ↓ gRPC
C++ gRPC Server

Option 4: Native C++ Bridge via FFI

Embed a native C++ gRPC client into the frontend via platform channels or FFI.

Frontend UI
   ↓ FFI / platform channel
Native C++ Client Library
   ↓ gRPC
C++ gRPC Server

Decision Outcome

Chosen option: Option 1 (direct gRPC) as primary, with Option 2 (gRPC-Web proxy) as conditional add-on for browser targets.

Options 3 and 4 rejected.

Rationale

Criterion Direct gRPC gRPC-Web REST Gateway Native Bridge
Reuses existing .proto ✅ ✅ ⚠️ needs annotations ✅
Streaming support ✅ full ⚠️ server-stream only ❌ polling ✅ full
Infrastructure overhead None Proxy process Gateway process FFI complexity
Type safety ✅ generated stubs ✅ generated stubs ⚠️ JSON serialization ✅
Mobile/desktop ✅ native N/A ✅ ⚠️ per-platform
Browser ❌ ✅ ✅ ❌
Generator complexity Low Low (proxy config) Medium (OpenAPI) High (FFI bindings)

Why not REST? Introduces a translation layer that loses streaming, weakens typing, and requires maintaining OpenAPI annotations alongside proto. Contradicts "single contract" principle.

Why not FFI? Per-platform FFI binding is high maintenance. The gRPC server exists precisely to decouple clients from the C++ process.

Architecture Rule

C++ frontends (Qt Widgets, QML)     → Direct C++ (in-process DBA)
Non-C++ frontends (future targets)  → gRPC (network, HTTP/2)
Browser frontends (SPA)             → gRPC-Web (via proxy)

Positive Consequences

  • Single API contract (.proto) for all clients regardless of language
  • Streaming domain events work natively over gRPC
  • No additional gateway or translation layer for mobile/desktop
  • Clean separation: C++ server is a network service, clients are independent processes
  • Each language ecosystem uses its own idiomatic gRPC library
  • Generator only needs to produce proto stubs + thin service wrappers

Negative Consequences

  • Browser targets require an Envoy proxy (additional deployment component)
  • gRPC-Web supports only server-streaming (no bidirectional) — sufficient for domain events, but limits future use cases
  • Network latency vs. in-process calls (acceptable for frontend use cases)
  • TLS configuration needed for production deployments

AI Agent Guidance

When This Decision Applies

  • Integrating any non-C++ frontend with the Muscovite backend
  • Choosing transport protocol for new client platforms
  • Configuring deployment for browser-based clients
  • Generating client stubs for non-C++ languages

When This Decision Does NOT Apply

  • Qt Widgets or QML frontends (these use in-process C++ — no network transport)
  • C++ test infrastructure (uses DBA directly)
  • PostgreSQL schema or migration generation
  • Server-to-server communication within the same backend process

Enforcement

  • All non-C++ frontends MUST use gRPC as primary transport
  • REST gateways are FORBIDDEN unless explicitly justified by a new ADR
  • gRPC-Web proxy is REQUIRED for browser targets, OPTIONAL otherwise
  • Generated proxy configuration goes to deploy/envoy.yaml

Key Files

  • proto/*.proto — canonical service definitions (source of truth)
  • dslgen/generators/grpc/proto.rkt — proto file generation
  • deploy/envoy.yaml — generated gRPC-Web proxy configuration (conditional)
