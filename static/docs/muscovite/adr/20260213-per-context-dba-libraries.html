ADR: Per-Context Libraries for Change Isolation (All Layers)

ADR: Per-Context Libraries for Change Isolation (All Layers)

Date: 2026-02-13 Status: accepted Deciders: jolo Supersedes: — Related: ADR 20260113 (shared libraries only), IDEA-018 (one shared library per context)

Context

Two generated layers still produce monolithic shared libraries that aggregate all bounded contexts into a single build target:

  1. DBA layer: {project}_dba — all contexts' entities, repositories, and domain services
  2. Qt Quick/QML layer: single qt_add_qml_module() — all contexts' ViewModels and QML files

In a project like Geosciences (26 contexts), a source change in one context (e.g., Mineralogy) causes recompilation and relinking of all 25 unrelated contexts in these layers.

The gRPC and Qt Widgets generators already produce per-context libraries with INTERFACE aggregators. The monolithic layers are inconsistent with this pattern.

IDEA-018 originally proposed a 3-way per-context split (core/svc/ui) for build-time optimization, but was shelved because mold (FTR-011) solved the link-time problem sufficiently. This ADR revives the full approach under a different rationale: architectural hygiene.

Decision

Generate one shared library per bounded context per layer, with INTERFACE aggregator libraries for backward-compatible linking. Every layer uses the same per-context pattern.

For each bounded context, three independent libraries exist:

Library Layer Contents
{context}_dba DBA Entities, repositories, domain services
{context}_grpc gRPC Proto compilation + service stubs (already per-context)
{context}_ui Qt Widgets / QML Models, widgets/ViewModels, QML files

Generated Structure

# === DBA layer (currently monolithic → per-context) ===
add_library(mineralogy_dba SHARED
    mineralogy/src/dba/entities/mineral.cpp
    mineralogy/src/dba/repositories/mineral_command_repository.cpp
    mineralogy/src/dba/repositories/mineral_query_repository.cpp
    mineralogy/src/dba/services/mineral_command_service.cpp
    mineralogy/src/dba/services/mineral_query_service.cpp
)
target_link_libraries(mineralogy_dba PUBLIC muscovite::dba)

# Cross-context link edges (from explicit RELATIONSHIPS only)
# e.g., if Mineralogy has: sample -> SampleTracking.Sample REQUIRED ONE;
# then: target_link_libraries(mineralogy_dba PUBLIC sample_tracking_dba)

# INTERFACE aggregator (backward-compatible)
add_library(geosciences_dba INTERFACE)
target_link_libraries(geosciences_dba INTERFACE basedata_dba mineralogy_dba ...)
add_library(Geosciences::dba ALIAS geosciences_dba)

# === gRPC layer (already per-context — no change needed) ===
# mineralogy_grpc, petrology_grpc, etc. + geosciences_grpc INTERFACE

# === Qt Quick/QML layer (currently monolithic → per-context) ===
# Per-context QML module
qt_add_qml_module(mineralogy_qml
    URI Mineralogy
    VERSION 1.0
    QML_FILES qml/MineralForm.qml qml/MineralListView.qml ...
    SOURCES src/models/mineralmodel.cpp src/viewmodels/mineralviewmodel.cpp ...
)
target_link_libraries(mineralogy_qml PRIVATE mineralogy_dba Qt6::Quick)

# Qt Widgets (already per-context — no change needed)
# mineralogy-qt-widgets, petrology-qt-widgets, etc.

Cross-Context Dependencies

Derived exclusively from RELATIONSHIPS blocks with qualified target-context references. Empirical analysis of both example projects:

Project Contexts Cross-Context Edges Topology
Geosciences 26 0 Fully isolated — every context is an island
FHIR Health 9 ~55 Star — Basedata is hub, domain contexts are spokes
Hello World 2 2 Star — Todo → Basedata

In all three projects, cross-context coupling is minimal and forms a star topology (infrastructure context at center, domain contexts at periphery). Non-infrastructure contexts do not depend on each other.

Rationale

Primary: Change Isolation (Locality of Change)

Bounded contexts ARE isolation boundaries by definition. The build system should reflect that semantic boundary in its physical structure.

Property Monolithic DBA Per-Context DBA
Files recompiled per single-context change All contexts Only changed context
.so files relinked per change 1 large 1 small
Blast radius of a breaking header change All contexts Only dependent contexts
Test scope per context change Entire DBA test suite Per-context tests

Secondary: Consistency Across All Layers

After this change, all four C++ layers use the same per-context pattern:

Layer Before After
DBA Monolithic {project}_dba Per-context {context}_dba + INTERFACE aggregator
gRPC Per-context {context}_grpc + INTERFACE aggregator (already done)
Qt Widgets Per-context {context}-qt-widgets (already done)
Qt Quick/QML Monolithic qt_add_qml_module() Per-context {context}_qml modules

Not a Build-Time Decision

This ADR does NOT claim build-time improvement as justification. With mold + ccache (ADR 20260113, FTR-011), rebuild times are already acceptable. The motivation is purely structural: keep stuff separate, minimize blast radius, enforce context boundaries in the build graph.

Objective Criteria

Measurable properties to evaluate the decision:

  1. Recompilation scope: ninja -n -d explain after touching one context — should show only that context's sources, not the full project
  2. Relink scope: Only the changed context's .so relinks, not a project-wide .so
  3. Dependency fan-out: Cross-context target_link_libraries edges should match the relationship graph from the IR (no hidden coupling)
  4. CMake configure overhead: time cmake --preset debug should not regress significantly despite increased target count (N contexts × 1 library vs 1 monolithic library)
  5. Backward compatibility: Existing code linking {Project}::dba continues to work via the INTERFACE aggregator

Consequences

Positive

  • Source changes in one context cannot affect compilation of another (unless explicitly linked)
  • Per-context test targets become natural (test only what changed)
  • Deploy granularity: individual context .so files can be updated independently (given binary/API backward compatibility)
  • Consistent per-context pattern across all C++ layers
  • Aligns build structure with DDD bounded context semantics

Negative

  • Increased CMake target count (~1 → ~N per project)
  • Cross-context dependencies require explicit target_link_libraries edges (derived from IR — no manual maintenance)
  • Slightly more complex generated CMakeLists.txt

Neutral

  • No expected build-time change (mold already fast, ccache handles recompilation)
  • The standalone per-context CMake generator (dba-cmake.rkt) already produces per-context libraries — this unifies the two code paths

Implementation

Phase 1: DBA per-context split

  1. Modify dslgen/generators/cpp/dba/unified-build.rkt to emit per-context add_library() calls instead of one monolithic library
  2. Generate cross-context target_link_libraries() edges from relationship.target-context
  3. Generate INTERFACE aggregator {project}_dba for backward compatibility
  4. Mirror the pattern already used by dslgen/generators/grpc/grpc-cmake.rkt
  5. Unify with standalone per-context generator (dba-cmake.rkt) — eliminate dual code paths

Phase 2: QML per-context split

  1. Modify dslgen/generators/qt/cmake.rkt to emit per-context qt_add_qml_module() calls instead of one monolithic module
  2. Each context gets its own QML module URI ({Context})
  3. Per-context QML modules link their own {context}_dba — no monolithic DBA dependency
  4. Top-level QML application imports per-context modules

Verification

  1. ninja -n -d explain after touching one context shows only that context's sources
