Domain Service Layer Between gRPC and Persistence

Domain Service Layer Between gRPC and Persistence

  • Status: accepted
  • Deciders: Johannes Lochmann
  • Date: 2026-03-07
  • Tags: architecture, domain-services, validation, grpc, ddd

Technical Story: gRPC handlers currently act as both controller and business logic host, calling CommandService/QueryService directly. This creates an anemic domain model where DSL-declared constraints are discarded at the C++ boundary. There is no orchestration layer for the three-tier validation pipeline.

Context and Problem Statement

The current architecture has gRPC service handlers performing double duty: they act as transport-layer controllers (extracting auth context, deserializing protobuf messages, serializing responses) AND as business logic hosts (calling CommandService/QueryService directly, performing ad-hoc validation). This violates separation of concerns and produces an anemic domain model.

DSL-declared constraints (field validations, cross-field invariants, business rules) have no C++ enforcement point. The parser captures them, the IR represents them, but the generated C++ code discards them at the gRPC boundary. There is no orchestration layer that composes field-level validation (Tier 1), entity-level invariants (Tier 2), and user-extensible business rules (Tier 3) into a coherent pipeline.

How should we introduce a domain service layer that orchestrates validation across all three tiers while keeping gRPC handlers thin?

Decision Drivers

  • gRPC handlers must remain thin transport controllers — no business logic
  • All aggregate access must go through CommandService/QueryService (ADR 20260120)
  • Plugin extensibility via sidecar pattern (ADR 20260223) requires a composition point
  • Validation errors must accumulate (not fail-fast) for client UX
  • The layer must be fully generated from DSL declarations
  • All 8 generator targets must remain consistent

Considered Options

  1. DomainService per aggregate — dedicated orchestration class between gRPC and CommandService/QueryService
  2. Validation middleware in gRPC interceptors — gRPC-specific interceptor chain for validation
  3. Fat CommandService — extend CommandService to include validation orchestration
  4. No domain layer — keep validation in gRPC handlers with helper functions

Decision Outcome

Chosen option: Option 1 — DomainService per aggregate, because it provides a clean, testable orchestration point that is independent of the transport layer and respects the existing CommandService/QueryService boundary.

DomainService orchestrates the three-tier validation pipeline:

  1. Tier 1 (Strong Types): Entity::try_from(Command) for per-field construction validation
  2. Tier 2 (EntityValidator): EntityValidator::validate() for cross-field invariants
  3. Tier 3 (PluginValidator): PluginValidator::validate() for user-owned business rules via sidecar pattern

gRPC handlers become thin controllers: extract auth context, deserialize protobuf to command, delegate to DomainService, serialize result to protobuf response. DomainService returns std::expected<Result, ValidationErrors>.

Positive Consequences

  • Clean separation of concerns — gRPC handles transport, DomainService handles domain logic
  • Business logic is testable without gRPC infrastructure (pure C++ unit tests)
  • Plugin extensibility via sidecar pattern composes naturally at the DomainService level
  • Error accumulation across all three tiers gives clients comprehensive validation feedback
  • Generated code — DomainService is fully generated from DSL, no manual wiring
  • Consistent pattern across all aggregates in all bounded contexts

Negative Consequences

  • Additional generated class per aggregate increases code volume
  • All 8 generator targets need updates (~22 FTRs across 4 iterations)
  • Migration path for existing gRPC handlers requires moving logic out incrementally
  • One more layer of indirection for developers to understand

Pros and Cons of the Options

Option 1: DomainService per aggregate

  • Good, because it creates a transport-independent business logic layer
  • Good, because it provides a natural composition point for the three validation tiers
  • Good, because it is fully generated from DSL declarations
  • Good, because it is independently testable with doctest (no gRPC dependency)
  • Bad, because it adds one more generated class per aggregate
  • Bad, because it requires updating all 8 generator targets

Option 2: Validation middleware in gRPC interceptors

  • Good, because gRPC interceptors are a standard pattern for cross-cutting concerns
  • Bad, because it couples validation to the gRPC transport layer
  • Bad, because validation logic becomes untestable without gRPC infrastructure
  • Bad, because Qt frontends cannot reuse the same validation pipeline

Option 3: Fat CommandService

  • Good, because it requires no new class — validation is added to existing services
  • Bad, because it violates single responsibility (CommandService is for persistence orchestration)
  • Bad, because it conflates aggregate access (ADR 20260120) with domain validation
  • Bad, because plugin extensibility would need to be wired into an already complex class

Option 4: No domain layer

  • Good, because zero implementation cost
  • Bad, because business logic stays scattered across gRPC handlers
  • Bad, because validation is duplicated across handlers for the same aggregate
  • Bad, because DSL-declared constraints remain unenforced at the C++ boundary

AI Agent Guidance

When This Decision Applies

  • When generating C++ service code for any aggregate declared in a .ddd file
  • When generating gRPC handler implementations — handlers MUST delegate to DomainService
  • When adding validation logic — validation belongs in DomainService, not gRPC handlers
  • When generating test scaffolding — DomainService tests use doctest, not gRPC test fixtures

When This Decision Does NOT Apply

  • Query-only operations that do not mutate state (pure QueryService calls may bypass DomainService)
  • Infrastructure concerns (auth, rate limiting, logging) which remain in gRPC interceptors
  • Lookup table access — compile-time constants do not require domain service orchestration

Enforcement

  • Generator in dslgen/generators/ produces DomainService class per aggregate
  • gRPC handler generator emits thin delegation code only
  • Integration tests verify DomainService is callable without gRPC infrastructure

Key Files

  • dslgen/generators/cpp-service-v2.rkt — gRPC service generation (to be refactored)
  • dslgen/generators/dba/unified.rkt — CommandService/QueryService generation
  • dslgen/model/ddd-types.rkt — aggregate IR struct definitions
