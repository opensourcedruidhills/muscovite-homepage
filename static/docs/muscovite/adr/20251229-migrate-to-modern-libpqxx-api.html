ADR: Migrate to Modern libpqxx API (exec + pqxx::params)

ADR: Migrate to Modern libpqxx API (exec + pqxx::params)

Date: 2025-12-29
Status: Accepted
Deciders: Architecture Team
Affected Components: Database Access Layer (all C++ code using PostgreSQL)

Context

The project uses libpqxx 7.10.3 for PostgreSQL connectivity from C++. libpqxx provides two APIs for executing parameterized queries:

Deprecated API (pre-7.x):

txn.exec_params("SELECT * FROM users WHERE id = $1", user_id);
txn.exec_params("INSERT INTO logs (msg) VALUES ($1)", message);

Modern API (7.x+):

txn.exec("SELECT * FROM users WHERE id = $1", pqxx::params{user_id});
txn.exec("INSERT INTO logs (msg) VALUES ($1)", pqxx::params{message});

Key differences:

  • exec_params: Variadic; uses C++ type inference; overload resolution can be ambiguous
  • exec + pqxx::params: Explicit parameter wrapper; clear intent; better error messages
  • Type handling: exec_params accepts raw pointers (&value, nullptr); exec + params requires actual values or std::optional

The project currently has mixed usage:

  • Some modules use modern exec API
  • Security module previously used deprecated exec_params (now migrated)
  • Legacy code may still use exec_params

Decision

Migrate all libpqxx usage to the modern API: txn.exec(query, pqxx::params{...})

This applies to:

  • All new code must use exec + pqxx::params
  • All existing code using exec_params must be migrated (via refactoring or regeneration)
  • Null handling: use std::optional<T>{} (empty) or std::optional<T>{value} instead of nullptr

Rationale

Criterion exec_params (deprecated) exec + pqxx::params (modern)
Status Deprecated in libpqxx 7.x Recommended, maintained
Type safety Variadic; relies on inference Explicit param wrapper
Error messages Ambiguous overload errors Clear parameter type errors
Null handling Raw pointers (nullptr, &value) std::optional<T>
Future-proof Will be removed in libpqxx 8.x Supported long-term
Compiler compatibility Works with older C++ Requires C++17+ (we use C++23)
Code clarity Intent unclear from call site Intent explicit at call site

Selected: exec + pqxx::params because:

  1. Futures-proof: exec_params is deprecated and will be removed in libpqxx 8.x
  2. Type clarity: pqxx::params{} makes parameterization explicit
  3. Error messages: Compiler can distinguish parameter type errors
  4. Null safety: std::optional is idiomatic C++17+
  5. Team readability: Consistent across codebase

Consequences

Positive

  • ✅ Future-proof: Compatible with libpqxx 8.x and beyond
  • ✅ Clear intent: pqxx::params{a, b, c} is unambiguous
  • ✅ Better errors: Type mismatch errors point directly to the parameter
  • ✅ Null safety: std::optional encourages explicit nullable handling
  • ✅ Consistency: Uniform API across all database access code

Negative

  • ⚠️ Refactoring cost: All exec_params calls must be migrated
  • ⚠️ Null handling syntax: Slightly more verbose (std::optional<std::string>{} vs. nullptr)
  • ⚠️ Optional chaining: Need to handle optional values in some contexts

Migration Pattern

// Before (deprecated)
txn.exec_params("SELECT * FROM users WHERE id = $1", user_id);
txn.exec_params("INSERT INTO audit (msg) VALUES ($1)", 
                msg.empty() ? nullptr : &msg);

// After (modern)
txn.exec("SELECT * FROM users WHERE id = $1", pqxx::params{user_id});
txn.exec("INSERT INTO audit (msg) VALUES ($1)", 
         pqxx::params{msg.empty() ? std::optional<std::string>{} : msg});

Special Cases

1. Optional String Parameters

// Before:
txn.exec_params("CALL proc($1, $2)", value, opt_msg.empty() ? nullptr : &opt_msg);

// After:
txn.exec("CALL proc($1, $2)", pqxx::params{
    value,
    opt_msg.empty() ? std::optional<std::string>{} : opt_msg
});

2. Null Literals

// Before:
txn.exec_params("UPDATE table SET col = NULL WHERE id = $1", id);

// After (no change, NULL in SQL):
txn.exec("UPDATE table SET col = NULL WHERE id = $1", pqxx::params{id});

3. DEFAULT Values

// Before:
txn.exec_params("INSERT INTO table (name) VALUES (DEFAULT)");

// After (no parameters):
txn.exec("INSERT INTO table (name) VALUES (DEFAULT)");

AI Agent Guidance

When this decision applies:

  • Writing new C++ code that accesses PostgreSQL
  • Reviewing database access code
  • Migrating legacy code to modern libpqxx

How it's enforced:

  • Code review: No exec_params in pull requests
  • Static analysis: Could add clang-tidy rule to flag exec_params
  • CMakeLists.txt: Require libpqxx 7.10.3+ (enforced via Conan)
  • Compiler warnings: Deprecation warnings on exec_params (if enabled in libpqxx)

Key files:

  • All files in src/muscframework/server/src/security/ - Database access
  • All files in src/dslgen/generators/cpp/ - Emitted C++ code (must use modern API)
  • CMakeLists.txt - Enforce libpqxx >= 7.10.3
  • Code style guide / CONTRIBUTING.md - Add migration guidance

Alternatives Considered

A. Keep exec_params for backward compatibility (rejected)

  • Reason: libpqxx 8.x will remove exec_params; delaying the migration just defers the work
  • Cost: Technical debt, harder migration later

B. Create a wrapper layer (rejected)

  • Reason: Unnecessary indirection; exec + pqxx::params is already idiomatic
  • Cost: Maintenance burden, harder to debug

C. Use an ORM (e.g., sqlpp11) (rejected)

  • Reason: Out of scope for this ADR; would require separate architecture decision
  • Cost: Large integration effort

Migration Roadmap

  1. Phase 1: New code uses exec + pqxx::params immediately
  2. Phase 2: Racket generators emit exec + pqxx::params (not exec_params)
  3. Phase 3: Audit existing code for exec_params usage
  4. Phase 4: Migrate remaining exec_params calls (batch by module)
  5. Phase 5: Update CONTRIBUTING.md with migration pattern as reference
