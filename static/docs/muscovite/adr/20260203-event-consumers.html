Event Consumers

Event Consumers

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-03

Technical Story: Complete the event-driven architecture by implementing consumption of events published via the Outbox pattern (v0.8.0).

Context and Problem Statement

v0.8.0 implemented the Outbox pattern for reliable event publishing to NATS JetStream. Applications now need to consume these events to:

  1. Trigger side effects (send notifications, update external systems)
  2. Build projections (read-optimized views, materialized aggregations)
  3. Synchronize bounded contexts (eventual consistency between microservices)
  4. Enable event sourcing patterns (replay events to rebuild state)

We need a consistent framework for consuming events with proper error handling, replay capability, and monitoring.

Decision Drivers

  • Reliability: At-least-once delivery with idempotent handlers
  • Scalability: Horizontal scaling via consumer groups
  • Observability: Metrics and tracing for debugging
  • Simplicity: Easy handler registration and routing
  • Consistency: Match the OutboxPublisher patterns from v0.8.0

Considered Options

  1. Pull-based consumers (application polls JetStream)
  2. Push-based consumers (NATS pushes to callback)
  3. Hybrid approach (push with acknowledgment control)

Decision Outcome

Chosen option: Push-based consumers with explicit acknowledgment, because:

  • JetStream's push subscriptions handle flow control automatically
  • Explicit ack/nak allows retry control
  • Consumer groups provide horizontal scaling via durable subscriptions

Positive Consequences

  • Simpler implementation (no polling loop needed)
  • Built-in load balancing across consumer instances
  • JetStream handles message redelivery

Negative Consequences

  • Requires message handlers to be idempotent (cannot prevent redelivery)
  • Backpressure managed by NATS, not application

Handler Signature

using EventHandler = std::function<HandlerResult(const EventContext& ctx)>;

struct EventContext {
    std::string event_id;              // UUID from domain_events.id
    std::string event_type;            // e.g., "PatientCreated"
    std::string aggregate_id;          // e.g., patient UUID
    std::string schema;                // Bounded context schema name
    nlohmann::json payload;            // Parsed JSON payload
    std::chrono::system_clock::time_point occurred_at;
    std::uint64_t sequence;            // JetStream sequence number
    std::optional<std::string> reply_to; // For request-reply patterns
};

enum class HandlerResult {
    Ack,        // Successfully processed, acknowledge
    Nak,        // Temporary failure, retry later (exponential backoff)
    Term,       // Permanent failure, do not retry (move to DLQ)
    InProgress  // Long-running operation, defer ack (for async handlers)
};

Consumer Configuration

struct EventConsumerConfig {
    /// Consumer group name (durable subscription)
    std::string consumer_name{"default"};

    /// Subjects to subscribe to (e.g., "muscovite.patient.>")
    std::vector<std::string> subjects;

    /// Maximum messages to buffer before backpressure
    std::size_t max_pending{1000};

    /// Ack wait before redelivery (default 30s)
    std::chrono::seconds ack_wait{30};

    /// Maximum redelivery attempts
    std::size_t max_deliver{5};

    /// Enable message deduplication window
    std::chrono::seconds dedupe_window{120};
};

Registration API

class EventConsumer {
public:
    // Register handler for specific event type
    void on(const std::string& event_type, EventHandler handler);

    // Register catch-all handler (for unmatched events)
    void on_any(EventHandler handler);

    // Register error handler (called on handler exceptions)
    void on_error(ErrorHandler handler);

    // Start consuming events
    auto start() -> bool;

    // Stop consuming (waits for in-flight handlers)
    void stop();
};

Usage Example

NatsConfig nats_config;
nats_config.url = "nats://localhost:4222";

EventConsumerConfig consumer_config;
consumer_config.consumer_name = "patient-notifications";
consumer_config.subjects = {"muscovite.patient.>"};

NatsEventConsumer consumer(nats_config, consumer_config);

// Register handlers
consumer.on("PatientCreated", [&smtp](const EventContext& ctx) {
    auto email = ctx.payload["contact_email"].get<std::string>();
    smtp.send_welcome_email(email);
    return HandlerResult::Ack;
});

consumer.on("PatientDeceasedFlagged", [&audit](const EventContext& ctx) {
    audit.log_sensitive_access(ctx.event_id, ctx.aggregate_id);
    return HandlerResult::Ack;
});

// Catch-all for unknown events
consumer.on_any([](const EventContext& ctx) {
    spdlog::debug("Ignoring event: {}", ctx.event_type);
    return HandlerResult::Ack;  // Acknowledge to avoid redelivery
});

// Start consuming
consumer.start();

// ... application runs ...

// Graceful shutdown
consumer.stop();

Idempotency

Handlers MUST be idempotent because:

  1. JetStream delivers at-least-once (not exactly-once)
  2. Consumer restarts may redeliver in-flight messages
  3. Network issues may cause duplicate deliveries

Recommended idempotency strategies:

  • Database constraints: UNIQUE on event_id in processed_events table
  • Conditional updates: Only apply if current state matches expected
  • Deduplication cache: In-memory or Redis cache of recent event IDs

Consumer Groups (Horizontal Scaling)

// Multiple instances share the same consumer_name for load balancing
consumer_config.consumer_name = "patient-notifications-group";

// JetStream distributes messages across active consumers
// Each message delivered to exactly one consumer in the group

When This Decision Applies

  • Implementing event consumers in framework/dba/
  • Adding event-driven projections
  • Building cross-context synchronization
  • Implementing saga coordinators

When It Does NOT Apply

  • Direct database queries (use repositories)
  • Synchronous command handling (use gRPC services)
