Context

From February through March 2026 (v1.63.0–v1.67.2), Muscovite built a NATS JetStream-based multi-agent coordination system (agents/coordinator.py, ~3700 LOC) intended to orchestrate parallel sub-agent workers during releases and feature iterations. The system included:

  • A Python coordinator with FTR state machines, work orders, and durable queues
  • A NATS JetStream server (nats-muscovite-agent systemd service, port 4223)
  • Leader/worker agent roles with NATS-enforced separation of duties
  • Hook-based safety gates (.copilot/hooks/safety-gates.js) intercepting risky tool calls
  • TLA+ formal verification of the coordinator state machine
  • A 30-minute blocking QA sign-off gate in release.sh via request/reply polling
  • ~12 supporting ADRs and 2500+ lines of coordination documentation

Decision

Abandon the entire NATS-based coordination infrastructure. Remove all coordinator code, documentation, and governance artifacts. Write off the investment as learning.

Rationale

The coordination system failed in practice for several compounding reasons:

  1. Infrastructure dependency brittleness: Every release required a running NATS server. When the server was unreachable, the release was blocked entirely (FTR-394 made it mandatory with no fallback). This added an environmental prerequisite that exceeded the value delivered.

  2. Overhead exceeded benefit: The coordinator.py codebase (~3700 LOC Python) grew larger than the value it provided. Setting up a release required: start NATS, register workers, assign FTRs, wait for QA sign-off via polling. Human-in-the-loop with a single Copilot CLI session achieved the same outcomes faster and with less failure surface.

  3. The right tool already existed: The Copilot CLI task tool natively supports dispatching work to sub-agents with the correct model for each capability level (haiku for exploration, sonnet for bounded changes, opus for architecture). This is exactly what the coordinator was trying to provide — without any infrastructure.

  4. Parallelism without the machinery: Independent explore agents can be launched in parallel within a single CLI session. The main session (premium model) acts as coordinator natively. No state machine, no NATS, no Python daemon needed.

  5. Work order ceremony: The work order format (ADR 20260313, ADR 20260315) became a bureaucratic overhead. A simple delegation model (see ADR 20260318-delegate-via-task-tool) captures the same guardrails without the ceremony.

Consequences

Removed

  • agents/ directory (coordinator.py, depgraph.py, TLA+, NATS config, worker worktrees)
  • doc/COORDINATOR.md, doc/AGENTROLES.md
  • .copilot/hooks/safety-gates.js
  • products/muscovite-agent-coordination submodule
  • NATS telemetry from release.sh (90+ lines); replaced with gate timing only
  • Blocking QA sign-off gate in release.sh; replaced with interactive y/N prompt
  • Coordinator handshake and work-order validation from agent-pr.sh
  • skill-create-work-order prompt

Preserved

  • cmake/DockerNats.cmake — NATS for product domain events (port 4222) is unaffected
  • ADR 20260305 — event-hub NATS for product is unaffected
  • All quality gates in release.sh and agent-pr.sh — the gates themselves were sound
  • Gate timing data lake (FTR-426) — _record_gate wrapper keeps emit_gate_timing calls
  • doc/AGENT_COORDINATION_LEARNINGS.md — failure record preserved as institutional memory

Replaced

  • Rule 28 (mandatory skill-create-work-order) → skill-delegate-work (lightweight delegation model using Copilot CLI's native task tool with model dispatch matrix)
  • Multi-agent protocol documentation → simple delegation section in AGENTS.md

Not Changed

  • Retrospective practice (Rule 29) — still valuable, NATS-independent
  • Improvement FTR loop (Rule 30) — still valuable
  • All DSL, C++, generator, and database features — completely unaffected
