S-Expression Template DSL for Code Generation (Future Direction)

S-Expression Template DSL for Code Generation (Future Direction)

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-11
  • Tags: templates, s-expression, racket, code-generation, future

Context and Problem Statement

Muscovite's generators produce code using Racket's format strings and string-append concatenation:

;; Current approach (widgets-views.rkt, qml-main.rkt)
(format "class ~aEditDialog : public QDialog {\n    Q_OBJECT\npublic:\n    explicit ~aEditDialog(QWidget* parent = nullptr);\n};\n"
        class-name class-name)

This works but has drawbacks:

  • ~a placeholders are positional — counting placeholders in long format strings is error-prone
  • Template structure is invisible — no distinction between "boilerplate" and "interpolated value"
  • No composability — templates are flat strings, not structured data
  • Validation is runtime-only — malformed templates produce bad output silently

An alternative approach uses s-expression templates — structured Racket data that describes the code to generate:

;; Proposed future approach
(define-template entity-reference-widget
  (λ (entity context)
    `(cpp-class
      (name ,(format "~aReferenceWidget" (entity-name entity)))
      (base "QComboBox")
      (qobject #t)
      (signals
       ,(format "void ~aSelected(~a::~a::id_type id)"
               (to-camel-case (entity-name entity))
               (context-name context)
               (entity-name entity))))))

Question: Should Muscovite adopt s-expression templates now, or continue with format-based generation?

Decision Drivers

  • Existing generators work — ~5,500 lines of format-based generation across widgets-views.rkt and qml-main.rkt
  • Risk of rewrite — converting existing generators to s-expression templates is a large refactor
  • Benefit scales with target count — more valuable when generating for 5+ targets, marginal for 2
  • Racket's macro system is naturally suited to s-expression templates
  • No external template engine dependency

Considered Options

Option 1: Adopt S-Expression Templates Now

Introduce define-template, template-file, and a template evaluation engine. Convert generators incrementally.

Option 2: Defer to Future (Continue format-based)

Document the approach as a future direction. Continue using format/string-append in current generators. Revisit when adding the 3rd or 4th UI target.

Decision Outcome

Chosen option: Option 2 — Defer, because:

  • Current format-based generators work and are tested
  • Only 2 UI targets exist — the composability benefit doesn't justify the refactor cost
  • Layout-spec integration (ADR abstract-layout-vocabulary) can be implemented with existing format patterns
  • The s-expression approach is well-understood and can be adopted later without breaking changes
  • Rewriting 5,500 lines of working generator code carries regression risk

When to Revisit

This ADR should be revisited when:

  1. A 3rd UI target is added — template duplication becomes painful
  2. Layout-spec generators reach 1,000+ lines of format strings — maintainability threshold
  3. Cross-target template sharing is needed (e.g., both Qt Widgets and QML emit similar validation code)

S-Expression Template Design (For Future Reference)

The template system would consist of:

  1. define-template — macro for declaring named templates
  2. Template evaluation — walks s-expression tree, emits target-language code
  3. Target-specific emitters — cpp-class → C++ header/source, qml-item → QML component
  4. Composition — templates reference other templates by name
;; Template definition
(define-template entity-edit-dialog
  (λ (entity layout-spec target-config)
    `(cpp-class
      (name ,(format "~aEditDialog" (entity-name entity)))
      (base "QDialog")
      (qobject #t)
      (constructor ...)
      (private-fields
       ,@(for/list ([field (entity-fields entity)])
           (generate-field-widget field target-config))))))

;; Template evaluation
(define (emit-cpp-class template-output)
  (match template-output
    [`(cpp-class (name ,name) (base ,base) (qobject ,qobj) . ,rest)
     (string-append
      (format "class ~a : public ~a {\n" name base)
      (if qobj "    Q_OBJECT\n" "")
      ;; ... emit rest
      )]))

Positive Consequences

  • Documents the design direction for future contributors
  • Current generators are unaffected — no regression risk
  • Decision is reversible — can adopt templates incrementally later

Negative Consequences

  • Generators continue using format strings (harder to maintain at scale)
  • Adding new targets requires duplicating format-string patterns
  • No structured validation of generated code structure

AI Agent Guidance

When This Decision Applies

  • When proposing to replace format-based generation with structured templates
  • When discussing code generation architecture for multiple targets
  • When evaluating template engines or DSLs for code generation

When This Decision Does NOT Apply

  • Current generator work — use format/string-append as today
  • Adding LAYOUT support to existing generators
  • Non-template-related generator changes

Enforcement

  • Do not introduce define-template or s-expression template infrastructure until this ADR is superseded
  • Continue using format/string-append in generators
  • If proposing template adoption, reference this ADR and provide migration plan
