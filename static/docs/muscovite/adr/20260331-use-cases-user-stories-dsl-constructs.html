Use Cases and User Stories as DSL Constructs

Use Cases and User Stories as DSL Constructs

  • Status: draft
  • Deciders: Johannes Lochmann (Architect)
  • Date: 2026-03-31
  • Tags: dsl, ddd, requirements, traceability, use-case, user-story, grammar, ir
  • Supersedes: none

Technical Story: RFC-001 Extension 5 — add USE_CASE and USER_STORY as first-class constructs for requirements traceability.

Context and Problem Statement

Muscovite models what the system is (entities, aggregates, events) but not what the system does for users (use cases, user stories). There is no formal link between user requirements and domain constructs. During brownfield extraction, AI agents discover entities and relationships from code/schema, but user-facing behavior is inferred, not declared.

For consulting deliverables, traceability from "what the user wants" to "which aggregates serve that need" is essential — and currently maintained in external documents that drift.

Decision Drivers

  • Requirements traceability is a consulting deliverable
  • Use cases reference domain constructs (entities, events) — must be validated
  • Must be declared inside CONTEXT blocks (they belong to a bounded context)
  • Actor modeling is organizational, not structural — free-text is sufficient
  • Grammar-first mandate (ADR 20260211)

Considered Options

Option 1: USE_CASE and USER_STORY as DSL constructs (chosen)

First-class constructs in .ddd files with entity/event references.

  • Good, because traceability is validated by the parser (references must resolve)
  • Good, because generators produce requirements traceability matrix automatically
  • Good, because dual format (USE_CASE for formal, USER_STORY for agile teams)
  • Bad, because adds DSL complexity for a non-technical concern

Option 2: External requirements file (.requirements.yaml)

Side-by-side YAML mapping requirements to entity UUIDs.

  • Good, because no grammar changes
  • Bad, because drifts from the model
  • Bad, because references are not parser-validated

Option 3: DOC annotations with structured text

ENTITY Patient @uuid "..." DOC "USE_CASE: RegisterPatient (Admissions Clerk)" — embed requirements in DOC strings.

  • Good, because no grammar changes
  • Bad, because unstructured — no validation, no reference resolution
  • Bad, because DOC strings are documentation, not model constructs

Decision Outcome

Chosen option: "USE_CASE and USER_STORY as DSL constructs", because formal constructs with validated entity/event references provide the traceability that consulting deliverables require.

DSL Syntax

USE_CASE RegisterPatient @uuid "..." {
    ACTOR "Admissions Clerk"
    CREATES Patient
    UPDATES Encounter
    READS AdministrativeGender, InsuranceProvider
    TRIGGERS PatientRegistered
    BENEFIT "New patient record created with insurance linkage"
    DOC "Primary admission workflow — 80% of all patient registrations";
}

USER_STORY ManageAllergies @uuid "..." {
    AS "Nurse"
    I_WANT_TO UPDATE Allergy FOR Patient
    SO_THAT "Critical allergy information is available before medication administration"
}

Rules

  • Use cases and user stories are declared inside a CONTEXT block
  • Entity references (CREATES, UPDATES, READS, FOR) must resolve to entities within the same context or imported contexts
  • TRIGGERS references domain events
  • ACTOR / AS are free-text strings
  • Both carry @uuid for semantic diff tracking
  • Both can carry @compliance and @quality annotations

PEG Grammar

UseCase     <- 'USE_CASE' IDENTIFIER UuidAnnotation ComplianceAnnotation*
               QualityAnnotation*
               '{' UseCaseBody '}'
UseCaseBody <- (ActorClause / CreatesClause / UpdatesClause / ReadsClause /
                TriggersClause / BenefitClause / DocClause)*

UserStory   <- 'USER_STORY' IDENTIFIER UuidAnnotation ComplianceAnnotation*
               QualityAnnotation*
               '{' UserStoryBody '}'
UserStoryBody <- AsClause IWantToClause SoThatClause? DocClause?

IR Implications

  • New IR node types: UseCase, UserStory with entity/event references (UUIDs)
  • Stored within the context they belong to

Generator Implications

  • Documentation: requirements traceability matrix — which use cases touch which entities, which events they trigger. Domain-expert docs show use cases in natural language. Developer docs show entity/event linkage.
  • Test generator: use case declarations feed generated acceptance test skeletons.
  • Extraction pipeline: AI agents emit use cases alongside entities when extracting from legacy systems.

Positive Consequences

  • Requirements traceability formalized in the model
  • Parser validates entity/event references — broken links caught at compile time
  • Acceptance test skeletons generated from use case declarations

Negative Consequences

  • DSL becomes more verbose for teams that don't use formal use cases
  • Actor modeling is limited to free-text (no organizational hierarchy)

AI Agent Guidance

When This Decision Applies

  • When adding USE_CASE or USER_STORY constructs to a bounded context
  • When generating requirements traceability documentation
  • When extracting requirements from legacy systems

When This Decision Does NOT Apply

  • Technical use cases that don't involve user-facing behavior
  • Process flows (use FLOW construct instead — separate ADR)

Enforcement

  • PEG grammar rules for UseCase/UserStory
  • Parser validation: entity/event references must resolve within context or imports
  • Generators produce traceability matrix only when use cases are present
