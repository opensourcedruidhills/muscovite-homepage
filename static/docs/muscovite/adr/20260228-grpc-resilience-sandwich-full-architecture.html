gRPC Resilience Sandwich — Full Deadline and Retry Architecture

gRPC Resilience Sandwich — Full Deadline and Retry Architecture

  • Status: accepted
  • Deciders: jolo
  • Date: 2026-02-28
  • Tags: grpc, resilience, retry, deadline, circuit-breaker, idempotency, dsl, code-generation
  • Relates: 20260228-grpc-deadline-to-statement-timeout-propagation.md, 20260120-domain-services-as-aggregate-boundary.md, 20260120-cqrs-ready-architecture.md, 20260212-compile-time-lookup-constants.md

Technical Story: The initial ADR (20260228-grpc-deadline-to-statement-timeout-propagation) addresses server-side deadline-to-timeout conversion. This companion ADR proposes the full resilience architecture: DSL-configured client-side deadlines with override hierarchy, retry coordination between gRPC and pqxx layers, client-side graceful handling of deadline failures, and the "Resilience Sandwich" pattern catalog entry.

Context and Problem Statement

The initial deadline ADR solves the server-side plumbing: gRPC remaining time becomes PostgreSQL statement_timeout. But a production-grade system needs the full picture:

  1. Who sets the deadline? Currently no client code sets set_deadline(). A client-side default must exist, configurable from the DSL with an override hierarchy (global → context → aggregate → RPC).
  2. How do retries interact with deadlines? The framework transactor retries serialization failures and deadlocks with exponential backoff. But retries must respect the remaining gRPC deadline — no point retrying if 4.8s of a 5s deadline are already spent.
  3. How do C++ gRPC clients handle DEADLINE_EXCEEDED? Generated client stubs (e2e tests, inter-service calls) must distinguish DEADLINE_EXCEEDED from UNAVAILABLE and react appropriately (no retry for the former, retry for the latter).
  4. How is the pattern documented? The "Resilience Sandwich" mental model (gRPC deadline / business logic / pqxx retry) needs a pattern catalog entry for developers and AI agents.

The Resilience Sandwich

┌─────────────────────────────────────────────────┐
│  TOP SOURDOUGH: gRPC set_deadline()             │
│  Circuit breaker behavior — client gives up      │
│  after N seconds. Prevents cascade starvation.   │
├─────────────────────────────────────────────────┤
│  THE FILLING: Business Logic                     │
│  Idempotent operations. Transactional writes.    │
│  Domain services as aggregate boundary.          │
│  Single UoW per command — commit or rollback.    │
├─────────────────────────────────────────────────┤
│  BOTTOM SOURDOUGH: libpqxx Transactor            │
│  Exponential backoff retry for transient errors. │
│  statement_timeout from remaining gRPC deadline. │
│  Retry budget = remaining_deadline - buffer.     │
└─────────────────────────────────────────────────┘

Key insight: The two "breads" operate on different failure modes:

  • Top: Protects the caller from slow downstream (wall-clock timeout)
  • Bottom: Protects the operation from transient infrastructure glitches (retry with backoff)

The filling must be idempotent because the bottom bread may replay it. The top bread must be strict because once the client has moved on, server-side work is wasted.

Decision Drivers

  • DSL-first: deadline configuration must be expressible in .ddd / .mus and propagate to all generated layers
  • Override hierarchy must be intuitive: global default → context → aggregate → specific RPC
  • Retry budget must be coordinated: pqxx retries must not exceed the gRPC deadline
  • Generated C++ clients must handle DEADLINE_EXCEEDED distinctly from other errors
  • The pattern must compose with existing features: sagas (multi-step, longer deadlines), event sourcing (dual-write), specifications (read-only, shorter deadlines)
  • Framework code must be testable without a running gRPC server

Proposed Architecture

1. DSL Syntax: Deadline Override Hierarchy

Project-level default (.mus):

SERVER {
    GRPC_DEFAULT_DEADLINE "5s";
    GRPC_MAX_DEADLINE "120s";    -- safety cap, server-side
}

Context-level override (.ddd):

CONTEXT HeavyComputation @uuid "..." {
    GRPC { DEADLINE "30s"; }
    ...
}

Aggregate-level override (.ddd):

AGGREGATE DataImport @uuid "..." ROOT DataImport
    GRPC_DEADLINE "60s"
    DOC "Long-running batch import";

RPC-level override (via command annotation):

COMMAND RunAnalysis ON DataSet (
    dataset_id UUID NOT_NULL @uuid "..."
)
GRPC_DEADLINE "120s"
DOC "CPU-intensive analysis, needs extended deadline";

Resolution order: RPC → Aggregate → Context → Project → hardcoded 5s default.

2. Retry Budget Coordination

The transactor's RetryPolicy gains a deadline-aware mode:

struct RetryPolicy {
    // ... existing fields ...

    /// When set, retries stop when remaining time < min_remaining.
    /// Prevents wasting the last fraction of the deadline on a retry
    /// that will almost certainly be aborted by statement_timeout anyway.
    std::optional<std::chrono::system_clock::time_point> deadline{};
    std::chrono::milliseconds min_remaining_for_retry{1000};
};

Retry loop logic:

for each attempt:
    remaining = deadline - now
    if remaining < min_remaining_for_retry:
        return DeadlineExceeded("insufficient time for retry")
    statement_timeout = remaining - buffer
    execute(fn, statement_timeout)
    if transient_error and should_retry:
        backoff_delay = min(calculated_backoff, remaining - min_remaining_for_retry)
        sleep(backoff_delay)

3. Client-Side Deadline Injection (Generated)

e2e test stubs (e2e-grpc-test.rkt):

grpc::ClientContext ctx;
ctx.set_deadline(std::chrono::system_clock::now() +
    std::chrono::milliseconds{5000});  // from DSL or default
auto status = stub->CreateEntity(&ctx, request, &response);
if (status.error_code() == grpc::StatusCode::DEADLINE_EXCEEDED) {
    // Log, metric, do NOT retry — the server may have committed
    spdlog::warn("CreateEntity: deadline exceeded");
}

Inter-service calls (future: context map gRPC clients):

// Deadline shrinkage: each hop reduces the deadline by a buffer
auto remaining = incoming_ctx->deadline() - std::chrono::system_clock::now();
auto downstream_deadline = remaining - hop_buffer;
outbound_ctx.set_deadline(std::chrono::system_clock::now() + downstream_deadline);

4. Generated Error Handling for Clients

New generated client-side error classification:

gRPC Status Client Action Retry Safe?
OK Success —
DEADLINE_EXCEEDED Log warning. Do NOT retry — server may have committed. No
UNAVAILABLE Retry with backoff (server down or overloaded) Yes
ABORTED Retry with backoff (serialization conflict) Yes
ALREADY_EXISTS Idempotency hit — treat as success —
NOT_FOUND Entity does not exist — domain error No
FAILED_PRECONDITION FK/constraint error — domain error No
INVALID_ARGUMENT Validation error — fix request No
INTERNAL Bug — log error, alert No

5. Idempotency (Future Phase)

For write RPCs, an optional idempotency key prevents duplicate creation when the client didn't receive the response (timeout between commit and response):

message CreatePatientRequest {
    string idempotency_key = 100;  // Client-generated UUID
    // ... fields ...
}

Server checks idempotency_keys table before INSERT. If key exists, returns the existing entity. This makes DEADLINE_EXCEEDED on writes safe to retry.

6. Pattern Catalog: The Resilience Sandwich

To be added to doc/RESILIENCE-PATTERNS.md (new document, parallel to CPP-PATTERNS.md and POSTGRES-PATTERNS.md):

The Resilience Sandwich describes the three-layer timeout and retry architecture for any gRPC + PostgreSQL service:

Layer Component Failure Mode Mechanism Configured By
Top Bread gRPC set_deadline() Slow downstream Wall-clock timeout, DEADLINE_EXCEEDED DSL override hierarchy
Filling Business logic (CommandService) Domain errors Transactional UoW, single commit point Generated code
Bottom Bread muscovite::dba::execute() Transient DB errors statement_timeout + exponential backoff retry Framework RetryPolicy + remaining deadline

Invariants:

  1. statement_timeout < grpc_remaining - buffer (always)
  2. Retry budget ≤ remaining deadline (stop retrying when nearly expired)
  3. Write operations idempotent (when idempotency keys enabled)
  4. DEADLINE_EXCEEDED is never retried by the client (server may have committed)

Positive Consequences (Expected)

  • Complete end-to-end deadline awareness from client to PostgreSQL
  • DSL-configured deadlines — domain experts express timing requirements declaratively
  • No more infinite-timeout gRPC calls or unbounded PostgreSQL queries
  • Generated clients handle timeout gracefully instead of hanging
  • Retry budget is coordinated — no wasted retries near deadline expiry
  • Pattern catalog gives developers and AI agents a mental model for the architecture

Negative Consequences (Expected)

  • Significant generated code change — all existing services regenerated
  • Grammar change required for DSL deadline syntax (grammar-first rule)
  • Complexity increase in transactor (deadline + retry interaction)
  • Idempotency adds storage overhead (dedup table per aggregate)
  • Testing complexity — need to simulate deadline expiry and verify rollback

Open Questions

  1. Should the SERVER block live in .mus (project config) or .ddd (domain model)? .mus seems right — it's deployment configuration, not domain modeling.
  2. Should GRPC_DEADLINE on COMMAND also affect the server-side max_statement_timeout (i.e., the server enforces the DSL-declared deadline even if the client sends a higher one)?
  3. How do saga step timeouts (already in DSL) interact with gRPC deadlines? A saga orchestrator calling a step service should propagate the saga step timeout as the gRPC deadline for that call.
  4. For read-only RPCs (Get, List, specifications), should the default deadline be lower than for writes? e.g., 2s for reads, 5s for writes.
