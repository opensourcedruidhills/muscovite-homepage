Qt TaskTree Adoption for Async Operations

Qt TaskTree Adoption for Async Operations

  • Status: accepted
  • Date: 2026-03-03
  • Tags: qt, async, tasktree, grpc, framework
  • Deciders: Johannes Lochmann
  • Updated: 2026-03-03 — Qt 6.11.0 available; vendor strategy replaced with native Qt6::TaskTree

Note (2026-04): openSUSE Tumbleweed now ships Qt 6.11.0 as system packages. The custom ~/opt/Qt installation documented below is no longer required. Install via sudo zypper install qt6-base-devel qt6-tasktree-devel ... instead.

Technical Story: FTR-189 — Adopt Qt's TaskTree library as the async orchestration layer for all gRPC calls in generated Qt clients. Originally planned as vendored from qt-creator; now uses native Qt6::TaskTree (Qt 6.11.0+).

Context and Problem Statement

Every generated Qt dialog and ViewModel currently makes synchronous gRPC calls on the UI thread (Grade F in QT-WIDGETS-PATTERNS.md §1). This causes:

  1. UI freezes — any network latency directly blocks the event loop
  2. No timeout/cancellation — long-running RPCs cannot be aborted
  3. No sequential composition — multi-step workflows (load → validate → save) are nested callback pyramids

The framework needs a structured async primitive that:

  • Integrates with Qt's event loop (signals, QObject lifecycle)
  • Supports timeout, cancellation, sequential and parallel composition
  • Is testable with mock stubs (no real gRPC server needed)
  • Has a clear migration path to Qt's native solution

Decision Drivers

  • License compatibility: Must be compatible with AGPL-3.0 + Commercial dual licensing
  • Qt event loop integration: Must work with QCoreApplication::processEvents(), signals, and object lifecycle
  • Testability: Must support mock gRPC stubs for unit testing without a server
  • Native availability: Qt 6.11.0 ships Qt6::TaskTree module — no vendoring needed
  • Code generation simplicity: The generator must emit a clean, uniform async pattern

Considered Options

  1. Qt TaskTree (native Qt6::TaskTree from Qt 6.11.0)
  2. QtConcurrent + QFutureWatcher (Qt built-in)
  3. Hand-rolled AsyncGrpcCall template
  4. Boost.Asio with Qt event loop bridge

Decision Outcome

Chosen option: Option 1 — Qt TaskTree (native), because it provides structured composition (sequential, parallel, barrier), built-in timeout, cancellation propagation, and is available as a first-party Qt module in 6.11.0.

Qt 6.11 Native Integration

Qt 6.11.0 (available as system packages on openSUSE Tumbleweed) ships Qt6::TaskTree as a first-party module. No vendoring required.

Namespace: QtTaskTree:: (not Tasking:: as in qt-creator)

CMake:

find_package(Qt6 REQUIRED COMPONENTS TaskTree)
target_link_libraries(mylib PUBLIC Qt6::TaskTree)

Headers: #include <QtTaskTree/QCustomTask>, #include <QtTaskTree/QTaskInterface>, etc.

Key types:

  • QtTaskTree::QCustomTask<Task, Adapter> — extension point for custom task types
  • QtTaskTree::QDefaultTaskAdapter<Task> — default adapter for QObject tasks with start() + done(DoneResult)
  • QtTaskTree::QTaskInterface — reports done(DoneResult) to the tree engine
  • QtTaskTree::QTaskTree — the tree execution engine
  • QtTaskTree::DoneResult — Success | Error
  • QtTaskTree::DoneWith — Success | Error | Cancel
  • QtTaskTree::SetupResult — Continue | StopWithSuccess | StopWithError

Historical Note — Vendor Strategy (Superseded)

The original plan was to vendor TaskTree from qt-creator (src/libs/solutions/tasking/, BSD-3-Clause) into framework/qt/tasking/ until Qt 6.11 shipped. With Qt 6.11.0 now available, the vendor strategy was never implemented. This section is preserved for context only.

GrpcCallTask — Framework Bridge

The framework provides a non-template GrpcCallTask (QObject) as the bridge between gRPC blocking stubs and TaskTree. It uses QDefaultTaskAdapter via QCustomTask.

namespace muscovite::qt {

/// @brief QObject task that runs a blocking gRPC call off the main thread.
/// Compatible with QtTaskTree::QDefaultTaskAdapter<GrpcCallTask>.
class GrpcCallTask : public QObject {
    Q_OBJECT
public:
    using CallFunction = std::function<::grpc::Status()>;

    void setCallFunction(CallFunction fn);
    void start();  // Runs call_fn_ on worker thread via QtConcurrent

    [[nodiscard]] const ::grpc::Status& status() const;

signals:
    void done(QtTaskTree::DoneResult result);
};

/// TaskTree task type alias for gRPC unary calls.
using GrpcUnaryTask = QtTaskTree::QCustomTask<GrpcCallTask>;

} // namespace muscovite::qt

Usage pattern (generated code):

auto resp = std::make_shared<GetPatientResponse>();
GrpcUnaryTask task(
    [stub, req, resp](GrpcCallTask& t) {
        t.setCallFunction([=]() {
            grpc::ClientContext ctx;
            ctx.set_deadline(std::chrono::system_clock::now() + std::chrono::seconds{10});
            return stub->GetPatient(&ctx, req, resp.get());
        });
        return SetupResult::Continue;
    },
    [this, resp](const GrpcCallTask& t, DoneWith dw) {
        if (dw == DoneWith::Success)
            updateFromResponse(*resp);
        else
            handleGrpcError(t.status());
        return DoneResult::Success;
    });

Positive Consequences

  • All gRPC calls become non-blocking — zero UI freezes
  • Uniform async pattern simplifies generator output (~57 files currently sync)
  • Testable with mock stubs (FTR-192 provides MockGrpcStub)
  • No vendoring — uses native Qt 6.11 Qt6::TaskTree module
  • Future-proof — TaskTree is a first-party Qt module with active development

Negative Consequences

  • Requires Qt 6.11.0+ (older Qt versions not supported for async gRPC)
  • Generator output changes significantly (FTR-194) — one-time migration
  • QtTaskTree:: namespace differs from qt-creator's Tasking:: — docs/examples referencing qt-creator patterns need translation
