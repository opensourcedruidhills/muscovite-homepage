Racket Generator Unit Testing Strategy

Racket Generator Unit Testing Strategy

  • Status: accepted
  • Date: 2026-02-23
  • Tags: testing, racket, generators, quality, amplification
  • Relates to: 20260212-doctest-adoption-and-tdd-practice.md
  • Relates to: 20260211-grammar-first-language-evolution.md

Context and Problem Statement

Muscovite's Racket generators (88,556 lines) transform DSL models into 8 distinct target layers. As of v1.26.0 the DSL amplification ratio is 19× (11,737 DSL lines → 229,577 generated lines). This ratio will grow as more constructs are added.

The existing test suite (13,656 lines) is weighted heavily towards:

  • Integration tests (test-dsl-integration.rkt) — parse a full .ddd example, run all generators, verify the output directory is non-empty
  • Grammar conformance (test-grammar-conformance.rkt) — verify parser/grammar parity
  • Coverage patterns (test-generator-coverage.rkt) — spot-check that specific strings appear somewhere in generated output

Code review of FTR-106 (state machine generators, 8 files, ~3,000 lines) found zero dedicated Racket tests. BUG-003 (TransitionState lost-update) was discovered only by manual code review, not by the test suite. At a 19× amplification ratio, a structural defect in a generator is silently multiplied across every entity that carries that construct.

The problem: integration tests verify that generation completes; they do not verify that the generated structure is semantically correct.

Decision Drivers

  • At 19× amplification, a generator producing subtly wrong SQL is a correctness multiplier
  • test-generator-coverage.rkt (55 tests) checks for string presence but not structural correctness (e.g., it cannot verify that an UPDATE includes a WHERE guard)
  • BUG-003 demonstrates the failure mode: a complete and compilable generated file that is nonetheless logically wrong
  • ADR 20260212 mandates test-first for C++ (doctest) but does not define the parallel discipline for Racket generators
  • The cmake-emitter has 33 dedicated unit tests (test-cmake-emitter.rkt) proving the pattern works; it needs to become the mandatory standard, not an exception
  • Saga generators have test-saga-generator.rkt — also a proof the pattern works

Considered Options

  1. Status quo — integration tests + string presence checks
  2. Per-construct unit tests — one test-{construct}-generator.rkt per new generator family, testing against minimal IR fixtures
  3. Full output snapshot tests — golden-file comparisons of complete generated output
  4. Property-based tests — randomized IR structs via rackcheck

Decision Outcome

Chosen option: Option 2 — per-construct unit tests with minimal IR fixtures, because:

  • It is already proven by test-cmake-emitter.rkt, test-saga-generator.rkt, and test-specification-generators.rkt — no new infrastructure needed
  • Minimal IR fixtures exercise the generator directly without a full parse cycle (fast: <100ms per test file)
  • Tests assert on structural patterns, not just string presence — can verify SQL correctness, missing guards, ordering, namespacing
  • Snapshot tests are brittle under intentional output changes and require manual golden-file maintenance
  • Property-based testing has high setup cost; reserved for type registry / naming convention validation

How: Mandatory Unit Test Contract

Every new generator file MUST have a companion test file following this pattern:

dslgen/generators/<layer>/<construct>.rkt
dslgen/tests/test-<construct>-generator.rkt   ← companion, mandatory

Minimum test coverage per generator (enforced in PR/commit review):

Generator area Minimum assertions
SQL generation Correct table/column names; constraint names; SPDX header present
C++ generation Correct namespace; correct class name; method signatures; SPDX header
gRPC / proto Correct message names; field types; RPC names; correctness of SQL in handlers
Qt/QML Q_PROPERTY names; signal names; correct camelCase/PascalCase
CMake Library names; target linkage; file glob patterns

Fixture pattern (from test-cmake-emitter.rkt, test-saga-generator.rkt):

;; Build a minimal IR struct — do NOT parse a .ddd file
(define test-entity
  (make-entity 'Order
    #:uuid "00000001-0001-4000-8000-000000000001"
    #:id-field (make-field 'id (make-type-spec 'UUID #f #f #f #f #f)
                 #:uuid "00000001-0001-4000-8000-000000000002"
                 #:required? #t)
    #:fields (list ...)))

(define test-sm
  (make-state-machine 'OrderLifecycle 'Order
    #:uuid "..."
    #:states (list ...)
    #:transitions (list ...)
    #:initial-state 'Draft
    #:terminal-states '(Fulfilled Cancelled)))

;; Run generator into a temp dir
(define outdir (make-temporary-file "muscovite-test-~a" 'directory))
(generate-state-machine-service-stubs test-ctx outdir "testproject")

;; Assert on structural patterns
(define source-content
  (file->string (build-path outdir "src" "grpc" "order_state_machine_service.cpp")))

(check-true (string-contains? source-content "FOR UPDATE"))
(check-true (string-contains? source-content "AND current_state_id ="))
(check-false (string-contains? source-content "FAILED_PRECONDITION") ; not present before CAS check)

Scope of Backfill

Existing generators currently lacking unit tests require backfill as part of the next planned feature that touches that generator. Backfill is not a separate feature — it is the prerequisite for any change to the generator.

Generator family Test file to create Priority
State machine (all 8 files) test-state-machine-generator.rkt immediate (BUG-003 fix)
Search indexing test-search-generator.rkt high (FTR-112 followup)
Projection (CQRS views) test-projection-generator.rkt medium
Event sourcing store test-event-sourcing-generator.rkt medium
Qt/QML (widgets+qml) test-qt-generator.rkt medium
Proto definitions test-proto-generator.rkt medium

Integration Tests Remain Mandatory

Unit tests replace nothing. The three-tier test pyramid is:

flowchart BT
    UNIT["Unit · test-construct-generator.rkt (NEW mandatory)\nIR→output · fast, per-generator"]
    STRUCT["Structural · test-generator-coverage.rkt (existing)\npresence checks · medium speed"]
    INTEG["Integration · test-dsl-integration.rkt\nparse→all · slow, full-stack"]

    UNIT --> STRUCT --> INTEG

Positive Consequences

  • Generator bugs caught in O(milliseconds) rather than end-to-end regeneration
  • BUG-class defects (wrong SQL, missing guards, wrong namespacing) caught before they reach Layer 2
  • Enables TDD for generator changes: write failing test → fix generator → green
  • Prevents silent regression when a generator is modified for unrelated reasons

Negative Consequences

  • Backfill effort for existing untested generators (~8 files × ~30 tests = ~240 tests)
  • Dual-maintenance: both generator and companion test must be updated together
  • No escape hatch: "I'll add tests later" explicitly violates this ADR

Enforcement

Per ADR 20260212, implementation before test is FORBIDDEN. This ADR extends that to Racket generators explicitly:

FORBIDDEN:

  • Merging a new generators/<layer>/<construct>.rkt without a companion test file
  • Modifying an existing untested generator without first adding the missing tests
  • Using test-generator-coverage.rkt string-presence checks as a substitute for structural correctness assertions

ALLOWED backfill path: When a generator is modified (bug fix or feature), the minimum viable test file covering the changed code must be created in the same commit.
