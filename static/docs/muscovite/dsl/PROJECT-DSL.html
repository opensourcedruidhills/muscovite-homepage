The Muscovite Project DSL (.mus files) is a declarative language for configuring project build targets, output directories, databases, capabilities, and context file includes. The formal PEG grammar sections 44–55 are the authoritative syntax specification. This document is auto-generated from the grammar and example files.

Two DSLs

Muscovite has two distinct DSLs, both formally specified in the PEG grammar:

DSL Extension Grammar Sections Parser Purpose
Project DSL .mus 44–55 parser/project-parser.rkt Build config: output dirs, databases, capabilities, contexts, frontends
Modeling DSL .ddd 1–43 parser/parser.rkt Domain model: entities, aggregates, events, sagas, relationships

.mus files configure what to generate and where. .ddd files declare the domain model.

44. Lifecycle Hooks (Reserved)

PROJECT DSL — Sections 45–53 The Project DSL (.mus files) configures build targets, output directories, databases, capabilities, and context file includes. It is a separate language from the Modeling DSL (.ddd files, sections 1–43). Grammar-first: ADR 20260211 + ADR 20260213

LifecycleHook        <- ( 'ON_INSERT' / 'ON_UPDATE' / 'ON_DELETE' )

45. Project Body

A PROJECT block contains metadata (description, version) followed by zero or more configuration sections in any order.

ProjectBody          <- ProjectItem*
ProjectItem          <- ProjectDescription / ProjectVersion / OutputBlock / DatabaseBlock / GenerateBlock / ContextsBlock / CapabilitiesBlock / OAuth2Block / FrontendsBlock / PluginBlock / ServerBlock / TelemetryBlock
ProjectDescription   <- 'DESCRIPTION' STRING
ProjectVersion       <- 'VERSION' STRING

46. Output Configuration

Specifies where generated artifacts are written and the Sqitch project name. Example: OUTPUT { directory "generated" sqitch_project "database" }

OutputBlock          <- 'OUTPUT' '{' OutputDirective* '}'
OutputDirective      <- OutputDirectory / OutputSqitchProject
OutputDirectory      <- 'DIRECTORY' STRING
OutputSqitchProject  <- 'SQITCH_PROJECT' STRING

47. Database Targets

Named database connection targets (development, test, production, etc.). Each target has a Sqitch-format URI and optional description. Example: DATABASE development { uri "db:pg://postgres:postgres@localhost:5555/mydb" description "Local Docker PostgreSQL" }

DatabaseBlock        <- 'DATABASE' IDENTIFIER '{' DatabaseDirective* '}'
DatabaseDirective    <- DatabaseUri / DatabaseDescription
DatabaseUri          <- 'URI' STRING
DatabaseDescription  <- 'DESCRIPTION' STRING

48. Generation Options

Boolean flags controlling which generator targets to run. Keys are generator target identifiers; values are true/false. Example: GENERATE { sqitch true cpp_dba true grpc true qt_widgets false }

GenerateBlock        <- 'GENERATE' '{' GenerateDirective* '}'
GenerateDirective    <- GenerateTarget BooleanLiteral
GenerateTarget       <- 'SQITCH' / 'CPP_DBA' / 'CPP_DBA_TEST' / 'GRPC' / 'QT_WIDGETS' / 'QT'
BooleanLiteral       <- 'TRUE' / 'FALSE'

49. Context File Includes

Ordered list of .ddd context files to load and generate from. Order matters: dependency contexts must be listed before dependents. Example: CONTEXTS { include "contexts/basedata.ddd" include "contexts/patient.ddd" }

ContextsBlock        <- 'CONTEXTS' '{' ContextInclude* '}'
ContextInclude       <- 'INCLUDE' STRING

50. Capabilities

Declares built-in infrastructure modules to enable. Each capability loads framework-provided bounded contexts and configures generator behavior. Example: CAPABILITIES { security spatial }

CapabilitiesBlock    <- 'CAPABILITIES' '{' CapabilityName+ '}'
CapabilityName       <- 'SECURITY' / 'SPATIAL' / 'SAGA' / 'EVENT_SOURCING' / 'STATE_MACHINE' / 'SEARCH' / 'TELEMETRY'

51. OAuth2 Configuration

Configures OAuth2 authentication for the gRPC server. Mode "builtin" uses the muscovite security module; mode "external" delegates to an external provider. Example: OAUTH2 { enabled true mode "builtin" provider "keycloak" realm "https://keycloak.example.com/realms/myapp" client_id "muscovite-server" }

OAuth2Block          <- 'OAUTH2' '{' OAuth2Directive* '}'
OAuth2Directive      <- OAuth2Enabled / OAuth2Mode / OAuth2Provider / OAuth2Realm / OAuth2ClientId
OAuth2Enabled        <- 'ENABLED' BooleanLiteral
OAuth2Mode           <- 'MODE' STRING
OAuth2Provider       <- 'PROVIDER' STRING
OAuth2Realm          <- 'REALM' STRING
OAuth2ClientId       <- 'CLIENT_ID' STRING

52. Frontend Configuration

Configures UI frontend generation targets with target-specific settings. Each frontend block contains key-value directives. Example: FRONTENDS { qt_widgets { enabled true app_name "My Desktop App" style "Fusion" theme "Light" mdi true startup_context "Dashboard" } }

FrontendsBlock       <- 'FRONTENDS' '{' FrontendTarget* '}'
FrontendTarget       <- FrontendTargetName '{' FrontendDirective* '}'
FrontendTargetName   <- 'QT_WIDGETS' / 'QML'
FrontendDirective    <- FrontendEnabled / FrontendAppName / FrontendStyle / FrontendTheme / FrontendMdi / FrontendStartupContext
FrontendEnabled      <- 'ENABLED' BooleanLiteral
FrontendAppName      <- 'APP_NAME' STRING
FrontendStyle        <- 'STYLE' STRING
FrontendTheme        <- 'THEME' STRING
FrontendMdi          <- 'MDI' BooleanLiteral
FrontendStartupContext<- 'STARTUP_CONTEXT' STRING

53. Plugin System — Project-Level Wiring (FTR-137, ADR 20260223)

PLUGIN blocks in .mus connect implementation code to EXTENSION_POINT contracts declared in .ddd files. Each plugin names an extension point via IMPLEMENTS, declares its transport mechanism, and optionally configures a circuit breaker for resilience. Cross-reference validation ensures IMPLEMENTS targets resolve to declared EXTENSION_POINTs. TRANSPORT compatibility is checked against the extension point's CLASS/LAYER/INTERFACE matrix. Example: PLUGIN AuditTrailPlugin { IMPLEMENTS Security.AuditLogger TRANSPORT SyncVirtual CIRCUIT_BREAKER { FAILURE_THRESHOLD 5 TIMEOUT "30s" BACKOFF Exponential BASE "100ms" MAX "10s" JITTER 15 ON_OPEN Warn ON_HALF_OPEN Probe ON_TRIP Emit } }

PluginBlock          <- 'PLUGIN' IDENTIFIER '{' PluginDirective+ '}'
PluginDirective      <- PluginImplements / PluginTransport / PluginCircuitBreaker
PluginImplements     <- 'IMPLEMENTS' QualifiedOrSimpleId
PluginTransport      <- 'TRANSPORT' TransportType
TransportType        <- 'SYNCVIRTUAL' / 'LOCALGRPC' / 'QTREMOTEOBJECTS'
PluginCircuitBreaker <- 'CIRCUIT_BREAKER' '{' CbDirective+ '}'
CbDirective          <- CbFailureThreshold / CbTimeout / CbBackoff / CbOnOpen / CbOnHalfOpen / CbOnTrip
CbFailureThreshold   <- 'FAILURE_THRESHOLD' NUMBER
CbTimeout            <- 'TIMEOUT' STRING
CbBackoff            <- 'BACKOFF' BackoffAlgorithm BackoffParams*
BackoffAlgorithm     <- 'Exponential' / 'Linear' / 'Fixed'
BackoffParams        <- 'BASE' STRING / 'MAX' STRING / 'INTERVAL' STRING / 'JITTER' NUMBER
CbOnOpen             <- 'ON_OPEN' FailureModeAction
CbOnHalfOpen         <- 'ON_HALF_OPEN' FailureModeAction
CbOnTrip             <- 'ON_TRIP' FailureModeAction
FailureModeAction    <- 'Warn' / 'Block' / 'Ignore' / 'Probe' / 'Emit'
QualifiedOrSimpleId  <- IDENTIFIER ( '.' IDENTIFIER )?

54. Server Configuration (FTR-232, ADR 20260228)

PROJECT-level gRPC/database timeout defaults. Per-aggregate overrides via GRPC_DEADLINE on AGGREGATE (FTR-231). Duration format: "Ns" (seconds) or "Nms" (milliseconds). Example: SERVER { GRPC_DEFAULT_DEADLINE "5s" GRPC_MAX_DEADLINE "30s" DEADLINE_BUFFER "100ms" }

ServerBlock          <- 'SERVER' '{' ServerDirective* '}'
ServerDirective      <- ServerGrpcDefaultDeadline / ServerGrpcMaxDeadline / ServerDeadlineBuffer
ServerGrpcDefaultDeadline<- 'GRPC_DEFAULT_DEADLINE' STRING
ServerGrpcMaxDeadline<- 'GRPC_MAX_DEADLINE' STRING
ServerDeadlineBuffer <- 'DEADLINE_BUFFER' STRING

55. Telemetry Configuration (FTR-239)

OTLP (OpenTelemetry Protocol) observability configuration. Requires TELEMETRY capability in CAPABILITIES block. Example: TELEMETRY { OTLP_ENDPOINT "http://collector:4317" SERVICE_NAME "fhir-health-server" SAMPLING_RATE 1.0 EXPORT_INTERVAL "30s" HEADERS { Authorization "Bearer token123" X-Custom "value" } }

TelemetryBlock       <- 'TELEMETRY' '{' TelemetryItem* '}'
TelemetryItem        <- TelemetryEndpoint / TelemetryServiceName / TelemetrySampling / TelemetryInterval / TelemetryHeaders
TelemetryEndpoint    <- 'OTLP_ENDPOINT' STRING ';'?
TelemetryServiceName <- 'SERVICE_NAME' STRING ';'?
TelemetrySampling    <- 'SAMPLING_RATE' NUMBER ';'?
TelemetryInterval    <- 'EXPORT_INTERVAL' STRING ';'?
TelemetryHeaders     <- 'HEADERS' '{' HeaderItem* '}'
HeaderItem           <- IDENTIFIER STRING ';'?

Project File Structure

A typical Muscovite project:

my-project/
├── project.mus              # Project configuration (this DSL)
├── contexts/
│   ├── basedata.ddd         # Shared lookup tables
│   ├── patient.ddd          # Domain bounded context
│   └── encounter.ddd        # Another bounded context
├── seeds/
│   └── basedata/            # JSON seed data for lookup tables
└── generated/               # 100% generated output (never edit)
    ├── dba/                  # C++ DBA layer
    ├── grpc/                 # gRPC services
    └── fhirhealth/           # Sqitch migrations
