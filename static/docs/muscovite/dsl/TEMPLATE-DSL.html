The Muscovite Template DSL is a Racket-embedded S-expression language for structured code generation. It replaces format/string-append patterns with composable, type-checked template nodes.

Overview

Templates are trees of template nodes. Each node represents a structural element of generated code. The emit function evaluates a template tree into a string, optionally scoped to a #:target language.

(emit (block 0
        (line "class Person {")
        (line (indent 1 "int age;"))
        (line "};")
      #:target 'cpp))
;; → "class Person {\n    int age;\n};\n"

Type Hierarchy

A Template is one of:

  • A template-node struct (kind + data)
  • A plain String (treated as raw text)
  • A (Listof Template) (concatenated)

The template? predicate checks all three cases.

Nesting Patterns

;; ✅ block containing lines
(block 0
  (line "first")
  (line "second"))

;; ✅ indent-block for uniform nesting
(block 0
  (line "namespace foo {")
  (indent-block 1
    (line "class Bar {")
    (indent-block 1
      (line "int x;"))
    (line "};")
  (line "}"))

;; ✅ guard for C++ headers
(guard "MY_HPP"
  (blank-line)
  (line "#include <string>"))

;; ✅ comment + line composition
(line (comment 'cpp "Entity definition"))
(line (comment 'sql "Schema migration"))

;; ✅ choose for type dispatch
(choose
  (cons (eq? type 'String) (text "std::string"))
  (cons (eq? type 'Integer) (text "std::int32_t"))
  (cons #t (text "auto")))

;; ✅ include-template for composition
(define-template license-header ()
  (block 0 (line "// SPDX-License-Identifier: AGPL-3.0-or-later")))
(block 0
  (include-template 'license-header)
  (blank-line)
  (line "#pragma once"))

;; ✅ conditional sections
(block 0
  (line "class Foo {")
  (when-true has-docs?
    (line (indent 1 "/// Documentation")))
  (when-false read-only?
    (line (indent 1 "void update();")))
  (line "};")

define-template Macro

Creates a named, registrable template function with source-location error reporting.

(define-template template-name
  (param1 param2 ...)
  body-expr)

Behavior

  1. Defines a function template-name that takes the specified parameters
  2. Wraps the body in error handling that reports the source file, line, and column
  3. Registers the template in template-registry (a hash from symbol to function)

Example

(define-template cpp-struct
  (name fields)
  (block 0
    (line (format "struct ~a {" name))
    (map-items
     (λ (f) (line (indent 1 (format "~a ~a;" (car f) (cdr f)))))
     fields)
    (line "};")))

;; Usage:
(emit (cpp-struct "Point" '(("double" . "x") ("double" . "y"))))
;; → struct Point {
;;       double x;
;;       double y;
;;   };

Error Reporting

If a template body raises an exception, the error message includes:

  • Template name
  • Source file path
  • Line and column numbers
Template error in cpp-struct at generators/foo.rkt:42:3 - format: ...

emit Function

(emit template #:target [target 'cpp]) → String

Evaluates a template tree to a string. The #:target keyword is reserved for future target-specific formatting (e.g., different indentation for Python vs C++).

Currently, all targets use the same evaluator (template-eval).

Migration Patterns: format → Template DSL

Recipe 1: Simple String Concatenation

Before:

(string-append
  "struct " name " {\n"
  "    int " field ";\n"
  "};\n")

After:

(block 0
  (line (format "struct ~a {" name))
  (line (indent 1 (format "int ~a;" field)))
  (line "};")

Recipe 2: Conditional Sections

Before:

(string-append
  header
  (if has-doc? (format "/// ~a\n" doc) "")
  body)

After:

(block 0
  (text header)
  (when-true has-doc?
    (line (format "/// ~a" doc)))
  (text body))

Recipe 3: List Iteration

Before:

(string-join
  (map (λ (f) (format "    ~a ~a;" (car f) (cdr f))) fields)
  "\n")

After:

(map-items
  (λ (f) (line (indent 1 (format "~a ~a;" (car f) (cdr f)))))
  fields)

Recipe 4: Separator-Joined Values

Before:

(string-join (map symbol->string enums) ", ")

After:

(join ", " (map (λ (e) (text (symbol->string e))) enums))

Recipe 5: Include Guards

Before:

(string-append
  "#pragma once\n\n"
  "#include <string>\n"
  body)

After:

(guard "MY_HPP"
  (blank-line)
  (line "#include <string>")
  body)

Recipe 6: Nested Indentation

Before:

(string-append
  "namespace foo {\n"
  "    struct Bar {\n"
  "        int x;\n"
  "    };\n"
  "}\n")

After:

(block 0
  (line "namespace foo {")
  (indent-block 1
    (line "struct Bar {")
    (indent-block 1
      (line "int x;"))
    (line "};")
  (line "}"))

Recipe 7: Language-Specific Comments

Before:

(format "// ~a entity\n" name)  ; C++
(format "-- ~a table\n" name)  ; SQL

After:

(line (comment 'cpp (format "~a entity" name)))
(line (comment 'sql (format "~a table" name)))

Recipe 8: Multi-way Type Dispatch

Before:

(cond
  [(eq? type 'String) "std::string"]
  [(eq? type 'Integer) "std::int32_t"]
  [else "auto"])

After:

(choose
  (cons (eq? type 'String) (text "std::string"))
  (cons (eq? type 'Integer) (text "std::int32_t"))
  (cons #t (text "auto")))

Relationship to CMake IR

The CMake generator uses a typed IR (cmake-ir.rkt, 22 struct types) rather than the general template DSL. Use the right tool:

Scenario Use
CMake generation CMake IR structs → cmake-emitter.rkt
C++/SQL/Proto/QML generation Template DSL (text, line, block, ...)
Simple one-off strings Plain format/string-append

The template DSL and CMake IR serve the same goal (structured code generation) at different specificity levels. CMake IR is domain-specific; the template DSL is general-purpose.
